## 本篇目标

读完本篇，你应该能够：

- 用“查资料”和“训练做事方式”解释 RAG 与微调；
- 区分知识缺失、行为不稳定和任务能力不足三类问题；
- 说清 RAG 与微调各自的收益、成本和失败方式；
- 设计基础模型、RAG、微调以及两者组合的对照实验；
- 避免把两种技术讲成非此即彼。

## 60 秒回答

RAG 和微调改变的是不同环节。RAG 不修改基础模型参数，而是在推理时检索外部证据，适合频繁变化、需要权限控制、引用追溯和按文档删除的知识；微调使用训练样本更新模型参数或附加参数，适合让模型更稳定地遵循某种任务、格式、语气和领域表达，也可以改善特定能力。

RAG 的优势是知识更新和回滚较灵活，来源可追溯；代价是在线多了检索、重排和长上下文，检索失败会直接影响答案。微调的优势是推理流程可能更简洁，行为可以更稳定；代价是需要训练数据、训练与回归评测，事实难以按文档精确更新和删除，也不天然产生引用。

两者不是二选一。常见做法是用 RAG 提供“本次该知道什么”，再用提示词或微调让模型学会“拿到证据后该怎样回答”。选型应先根据错误来源判断，再比较质量、延迟、成本和治理要求。

## 详细解析

### 先说人话：一本手册和一次岗位培训

仍然使用企业制度助手的例子。

公司需要一名客服回答员工问题。现在有两个需求：

1. 客服必须知道 2026 年最新差旅标准；
2. 客服回答时必须使用统一格式，先给结论，再列适用条件和来源。

第一项更像“给客服一本最新手册，并让他回答前查阅”，对应 RAG。

第二项更像“训练客服按照公司的标准话术和流程工作”，对应提示词或微调。

可以记成：

> **RAG 主要提供本次回答需要的外部知识，微调主要改变模型长期稳定的行为或任务能力。**

这是一条实用的工程边界，不是数学上的完全隔离。RAG 上下文也会影响行为，微调样本也可能让模型记住部分事实；但如果要做可更新、可删除和可引用的知识管理，外部知识层通常更合适。

### RAG 到底改变了什么

RAG 改变的是推理时的输入上下文。

模型原本看到：

> 用户问题：“上海出差住宿上限是多少？”

加入 RAG 后，模型看到：

> 系统规则 + 用户问题 + 当前生效制度中的相关证据 + 来源信息

基础模型参数没有因这次查询而改变。新制度发布后，应用更新文档和索引，下一次请求就可以检索新证据。

RAG 的知识位于外部系统，所以可以：

- 绑定文档 ID 和版本；
- 按部门或租户过滤；
- 展示引用；
- 删除某份资料的派生内容；
- 切换索引版本进行回滚。

### 微调到底改变了什么

广义的模型适配包含多种方式，面试中应先说明自己说的是哪一种。

#### 监督微调

监督微调，简称 SFT，使用“输入 -> 期望输出”样本训练模型。例如：

| 输入 | 期望输出 |
| --- | --- |
| 员工询问报销政策并提供证据 | 先给结论，再列适用范围，最后给来源 |
| 证据不足 | 明确说明缺少地区或职级，不猜金额 |
| 两份制度冲突 | 列出冲突并请求人工确认 |

它可以帮助模型更稳定地学习任务格式和响应习惯。

#### 参数高效微调

LoRA 等参数高效微调方法只训练一小部分附加参数，减少训练和存储成本。它改变了训练方式与资源需求，但没有把模型变成可精确修改的知识数据库。

#### 继续预训练

继续预训练通常使用大量领域文本，让模型进一步学习某个领域的语言和分布。它可能改善专业术语理解，但训练与评测成本更高，也不天然提供来源。

所以“微调”不是一个完全统一的方案。回答时要说明训练目标、数据类型和验证方式。

### 先判断错误属于哪一类

选型最实用的方法，是先看错误来自哪里。

| 错误类型 | 例子 | 优先方向 |
| --- | --- | --- |
| 知识缺失 | 模型不知道公司 2026 年制度 | RAG 或结构化数据查询 |
| 知识过期 | 模型仍回答旧金额 | 更新外部知识与版本过滤 |
| 检索失败 | 正确制度存在但没召回 | 改解析、切块、检索或重排 |
| 行为不稳 | 证据完整但经常不按格式输出 | 提示、结构化输出或微调 |
| 领域表达差 | 不理解内部缩写和专业术语 | 提示、词典、检索或领域适配 |
| 推理或计算错 | 规则和金额都找到，但合计错误 | 确定性程序、工具或能力适配 |
| 需要执行动作 | 用户要求真实提交报销 | 受控 API、Workflow 或 Agent |

不要用微调修复检索漏召回，也不要用扩大知识库解决输出格式不稳定。

### RAG 的优势

#### 知识更新更灵活

新制度发布后，可以只处理受影响文档并构建新索引。无需等待基础模型重新训练。

#### 容易追溯来源

Chunk 可以保存文档、章节、页码和版本映射，答案可以展示引用。

#### 支持细粒度权限

检索请求可以携带租户、部门和角色条件，让不同用户获得不同证据集合。

#### 便于删除和回滚

外部知识可以按文档 ID 删除，索引可以使用版本和别名切换。这里仍需验证缓存、日志和副本是否同步处理。

#### 可观察中间过程

可以检查召回了哪些候选、过滤了哪些文档、最终上下文使用了什么证据。这有助于归因。

### RAG 的劣势

#### 链路更长

在线多了查询理解、检索、过滤、融合、重排和上下文组装，每一层都会增加延迟和故障点。

#### 严重依赖数据与检索质量

知识库缺失、解析错误、Chunk 不合理或召回失败时，模型看不到正确证据。

#### 增加上下文成本

检索内容会占用输入 Token。重复和无关证据还可能干扰模型。

#### 仍需处理生成忠实度

正确证据进入上下文后，模型仍可能误读、漏读或错误组合。

#### 运维负担不低

需要管理采集、索引、权限、更新、缓存、监控、评测和回滚。

### 微调的优势

#### 行为更稳定

高质量训练样本可以让模型更稳定地遵循特定格式、术语和任务步骤。

#### 减少重复提示内容

部分稳定规则通过训练进入模型行为后，请求中可能不再需要放入大量示例。是否真的降低成本，要结合微调模型价格和效果测量。

#### 改善特定任务能力

在数据质量足够且任务清晰时，微调可以改善分类、抽取、结构转换或领域响应。

#### 推理链路可能更简单

如果任务不需要外部动态知识，微调模型可直接完成任务，不必增加检索服务。

### 微调的劣势

#### 需要高质量训练数据

样本中的错误、偏差和冲突会被模型学习。构造、清洗、去重和审核数据通常比启动训练更耗时。

#### 需要完整回归评测

某类任务变好后，其他能力可能退化。需要同时评测目标能力、安全性、格式遵循和通用能力。

#### 事实更新与删除困难

参数中的知识是分布式表示，无法像数据库记录一样按文档执行精确增删。

#### 不天然提供引用

即使模型记住了某个结论，也很难证明具体来自哪份材料。

#### 训练与部署增加版本管理

需要记录基础模型、数据集、训练配置、检查点和服务版本，并设计回滚。

### 一张完整对比表

| 维度 | RAG | 微调 |
| --- | --- | --- |
| 主要改变 | 推理时上下文 | 模型参数或附加参数 |
| 主要目标 | 接入外部、可更新知识 | 调整行为、格式或任务能力 |
| 知识更新 | 更新文档与索引 | 准备数据并重新训练或切换模型 |
| 引用追溯 | 可保留来源映射 | 不天然支持 |
| 权限控制 | 可在检索层按请求过滤 | 很难在参数中做文档级权限 |
| 删除与回滚 | 可按文档或索引版本处理 | 通常依赖模型版本切换或重训 |
| 在线延迟 | 增加检索、重排和上下文开销 | 取决于模型与部署方式 |
| 主要失败点 | 数据、解析、召回、重排、生成 | 训练数据、过拟合、能力退化、部署 |
| 评测重点 | 检索质量、忠实度、引用、端到端效果 | 任务质量、行为稳定性、安全与回归 |
| 适合变化速度 | 高频变化的外部事实 | 相对稳定的行为与能力 |

### 两者怎样组合

企业制度助手可以这样设计：

1. RAG 检索当前生效且用户有权查看的制度；
2. 模型读取证据；
3. 经过 SFT 的模型按照统一模板回答；
4. 证据不足时，模型按训练样例提出澄清问题；
5. 服务端校验引用和敏感字段。

RAG 负责“拿到哪份知识”，微调负责“拿到知识后怎样稳定处理”。

组合会增加模型、索引和数据集三套版本管理，不应默认两者叠加后效果就更好。需要通过对照实验验证每一层收益。

### 一个实用选型顺序

#### 第一步：先做可工作的基础版本

使用合适的基础模型和简洁提示，建立质量、延迟和成本基线。

#### 第二步：检查是否缺少外部知识

如果正确答案不在模型输入中，先接入权威数据源。结构化实时数据可直接查询 API，文档知识可考虑 RAG。

#### 第三步：检查检索是否找到证据

如果正确文档存在但没进入上下文，优化解析、Chunk、Embedding、混合检索、过滤和重排。

#### 第四步：检查模型如何使用证据

如果证据完整但模型不按格式、拒答规则或任务步骤执行，先尝试更明确提示、结构化输出和少量示例。

#### 第五步：评估微调

当错误模式稳定、样本足够、提示已变得冗长或不稳定时，再评估微调是否能带来可重复收益。

### 怎样做公平实验

至少比较四组：

| 组别 | 作用 |
| --- | --- |
| 基础模型 | 建立无 RAG、无微调基线 |
| 基础模型 + RAG | 测量外部知识带来的收益 |
| 微调模型 | 测量行为或能力适配收益 |
| 微调模型 + RAG | 验证组合是否有额外价值 |

控制变量包括：

- 相同评测集；
- 相同生成参数；
- 可比的提示规则；
- 相同 RAG 索引与检索配置；
- 记录模型、数据和代码版本。

同时报告：

- 答案正确性；
- 证据忠实度；
- 引用正确率；
- 无答案处理；
- P95 延迟；
- 单次请求成本；
- 训练和索引维护成本。

## 工程实践与边界

### 把知识版本和模型版本分开

部署配置应明确绑定：

- 生成模型版本；
- 提示词版本；
- Embedding 模型版本；
- 索引版本；
- 微调数据集和训练版本。

当线上回归出现时，可以分别切换模型或索引，避免把两个变量混在一起。

### 不用微调承担文档级权限

把多个租户资料混入模型参数后，很难在请求时保证某个用户只能访问其中一部分。需要细粒度权限的知识应保留在可过滤的外部数据层。

### 不用 RAG 代替确定性规则

例如报销金额需要依据票据数量与固定公式计算时，应由程序计算。RAG 提供制度条款，大模型解释条款，确定性程序执行计算，各组件各负其责。

### 计算总拥有成本

不能只比较一次训练费用和一次检索费用。应包括：

- 数据标注和审核；
- 训练、部署和回归评测；
- 文档采集、解析和索引；
- 在线检索、重排和 Token；
- 监控、故障处理和人员维护；
- 安全、合规与删除成本。

不同流量、模型、硬件和托管方式会得到不同结论。

### 设计回滚

微调模型异常时切回上一个模型版本；索引异常时切回旧读别名。若两者一起上线，应支持独立回滚，否则难以定位回归来源。

## 常见误区

- **“微调就是把企业知识灌进模型”**：微调可能让模型记住部分模式或事实，但不具备数据库式的精确更新、权限、删除和引用能力。
- **“RAG 只管知识，微调只管语气”**：这是便于理解的主线，不是严格边界。微调也能影响任务能力，RAG 上下文也会影响行为。
- **“微调后就不需要 RAG”**：只要知识会变化、需要引用或按用户过滤，仍可能需要外部检索。
- **“用了 RAG 就不用训练数据”**：RAG 也需要查询、相关文档、无答案和边界样本来评测与优化。
- **“两者一起用效果就更好”**：组合增加复杂度，收益需要用消融实验证明。
- **“RAG 总比微调便宜”**：结论依赖查询量、上下文长度、模型价格、索引规模、部署和维护成本。
- **“格式错误就扩大 Top-K”**：Top-K 主要影响检索候选，行为和格式问题应从提示、约束或微调解决。

## 面试追问

1. **资料已在上下文中，但模型持续答错，应该先做什么？**

   先检查证据是否完整、冲突和位置合理，再改提示或结构化输出。如果错误模式稳定且有足够样本，再评估微调；不要先扩大召回范围。

2. **能不能用微调记住产品手册？**

   技术上模型可能学习部分内容，但产品手册频繁更新、需要引用和删除时，RAG 更便于治理。微调可用于让模型更会解释手册。

3. **怎样判断微调数据是否够用？**

   不只看样本数量，还要看任务覆盖、标签一致性、困难样本、训练验证隔离和学习曲线。先做小规模实验并观察新增数据是否继续带来稳定收益。

4. **RAG 与微调怎样独立回滚？**

   服务配置分别引用模型版本和索引别名，发布时记录组合矩阵。出现回归后可以保持一项不变，只回退另一项。

5. **什么时候优先选择结构化查询而不是 RAG？**

   当答案来自订单状态、库存、余额或固定聚合时，直接查询数据库或 API 更可靠。可以把结构化结果提供给模型解释，但无需先把它变成文档向量。

6. **如何向面试官证明选型不是凭感觉？**

   展示错误分类、四组对照实验，以及质量、延迟、成本和治理指标。说明每项改动解决哪类错误，并给出未覆盖的边界。
