## 本篇目标

学完这一篇，你应该能够：

1. 用“按字面找”和“按含义找”的类比解释关键词检索与向量检索。
2. 说清倒排索引、BM25、稀疏向量、稠密向量、Embedding 和 ANN 的基本概念。
3. 根据错误码、型号、自然语言问答等不同 Query 选择检索方式。
4. 解释为什么生产 RAG 常使用混合检索，以及怎样融合两路结果。
5. 用 Recall@K、MRR、nDCG、延迟和成本做消融评测，而不是凭感觉选型。

## 60 秒回答

> 关键词检索主要按字面匹配，通常使用倒排索引和 BM25。它擅长错误码、产品型号、人名、政策编号和用户原话，速度快、可解释性强，但当用户和文档用不同措辞表达同一含义时可能漏召回。
>
> 向量检索先用 Embedding 模型把 Query 和文档编码成稠密向量，再按余弦相似度、点积或欧氏距离找近邻。它擅长语义匹配，比如“怎么退钱”可以找到“退款流程”，但对罕见编号、数字和领域外语义可能不稳定，还会引入模型推理、向量存储和 ANN 近似误差。
>
> 两者不是替代关系。生产 RAG 常并行做稀疏和稠密召回，再用 RRF 或经校准的分数融合，最后交给 Reranker 精排。我会按 Query 类型做消融实验，对比 sparse-only、dense-only、hybrid 和 hybrid+rerank 的 Recall@K、MRR/nDCG、P95 和成本，确认混合方案是否真的有增益。

## 详细解析

### 先说人话：一个按“原词”找，一个按“意思”找

假设知识库里有一句：

```text
订单付款后 24 小时内可以申请原路退款。
```

用户可能这样问：

```text
我刚买的东西不想要了，钱怎么退回来？
```

- 关键词检索会关注两边是否出现相同或相关词，例如“退款”。
- 向量检索会尝试理解“钱怎么退回来”和“原路退款”语义相近。

但如果用户问：

```text
报错 E-1047 怎么处理？
```

精确的 `E-1047` 往往比“语义相似”更重要，这时关键词检索更有优势。

### 关键词检索也叫稀疏检索

#### 什么是倒排索引

书末索引会告诉你“退款”出现在哪几页。搜索引擎的倒排索引类似：

```text
退款 -> 文档 2、文档 8、文档 31
耳机 -> 文档 4、文档 8
E-1047 -> 文档 17
```

查询时无需扫描全部文档，只需要从词项找到包含它的文档列表。

#### 为什么叫“稀疏”

假设词表有几十万个词，一段文档只包含其中很少一部分。将它表示成向量时，绝大多数维度都是 0，所以叫**稀疏向量**。

#### BM25 在做什么

**BM25**是一种经典的概率检索排序函数。零基础可以记住三个直觉：

1. 查询词在某篇文档中出现，通常增加相关分；
2. 越少见的词越有区分力，例如 `E-1047` 比“问题”更重要；
3. 很长的文档天然包含更多词，因此需要做长度归一化。

BM25 不是简单数词频。随着词频继续增加，新增词频带来的边际收益会逐渐变小。

#### 中文关键词检索的额外问题

英文有空格分隔词，中文通常需要分词或使用字符级 n-gram。下面这些配置都会影响结果：

- 分词器是否理解领域术语；
- “星云X2”和“星云 X2”能否归一；
- 同义词“退钱/退款/退货退款”是否扩展；
- 大小写、全半角、连字符和繁简体如何处理；
- 标题、正文、标签等字段是否使用不同权重；
- 停用词是否误删领域内重要词。

### 向量检索也叫稠密检索

#### 什么是 Embedding

**Embedding（嵌入）**是把文本转换为一串数字，使语义接近的文本在向量空间中更接近。

```text
“怎么退款” -> [0.12, -0.08, 0.31, ...]
“退钱流程” -> [0.10, -0.06, 0.29, ...]
```

实际向量可能有几百到几千维，且大部分维度不是 0，因此叫**稠密向量**。

#### 怎样比较两个向量

常见距离或相似度包括：

- **余弦相似度**：比较两个向量方向是否接近；
- **点积**：把对应维度相乘后求和；
- **欧氏距离**：比较空间中的直线距离。

选择哪一种应遵循 Embedding 模型的训练方式和建议。不能在没有归一化等前提时随意互换。

#### 为什么需要 ANN

如果有几百万条向量，每次都和 Query 精确比较会很贵。向量数据库通常使用 HNSW、IVF 等 ANN 索引缩小搜索范围。

ANN 用速度换取近似结果，因此存在两层质量问题：

1. Embedding 是否把相关文本放得足够近；
2. ANN 是否在近似搜索中找到了本应返回的近邻。

### 两种检索的能力对比

| 维度 | 关键词/稀疏检索 | 向量/稠密检索 |
| --- | --- | --- |
| 核心依据 | 词项重合和统计权重 | 学习到的语义空间 |
| 擅长 | 型号、错误码、人名、原文短语 | 同义表达、自然语言问题、概念匹配 |
| 主要索引 | 倒排索引 | HNSW、IVF 等 ANN 索引 |
| 可解释性 | 可看到命中词和词项权重 | 相似度较难直接解释 |
| 领域适配 | 分词、同义词和字段权重很重要 | Embedding 模型与领域语料很重要 |
| 成本 | 通常计算和存储较低 | 有 Embedding、向量存储与 ANN 成本 |
| 更新 | 倒排索引增量更新成熟 | 取决于向量生成和索引维护方式 |
| 典型失败 | 用户换一种说法就漏召回 | 精确编号被“语义稀释”或领域漂移 |

### 贯穿案例：四类售后问题分别怎么搜

#### 类型一：精确编号

```text
E-1047 是什么错误？
```

优先关键词或精确字段检索，并保留向量路处理描述性补充。

#### 类型二：同义表达

```text
耳机只有右边响，该怎么办？
```

文档写的是“单侧无声故障排查”。向量检索更容易跨措辞召回。

#### 类型三：混合问题

```text
星云 X2 出现单侧无声，保修期怎么处理？
```

`星云 X2` 需要精确词项，“单侧无声”和自然语言表达需要语义匹配，适合混合检索。

#### 类型四：带结构化限制

```text
华东区 2026 年 6 月生效的 X2 保修政策
```

文本检索之外，还需要地区、产品和有效时间过滤。过滤不是关键词与向量检索的替代品。

### 什么是混合检索

混合检索的基本结构是：

```text
用户 Query
   |-------------------|
   v                   v
BM25 稀疏召回       向量稠密召回
   |                   |
   |------- 去重/融合 --|
              |
              v
           Rerank
              |
              v
          最终候选
```

它利用两路的互补性，但不代表“两路都开就会更好”。一条弱召回路也可能把噪声推到前面。

### 融合方式一：RRF

RRF 只看名次，不直接比较原始分数：

```text
score(d) = Σ 1 / (k + rank_i(d))
```

某文档在多路都排得靠前，会获得更高融合分。参数 `k` 控制头部名次的相对影响，需要在目标数据上验证。

优点：

- 不要求 BM25 和向量分数同尺度；
- 实现简单，通常是稳健基线；
- 便于解释每路贡献。

限制：

- 只利用名次，忽略原始分数差距；
- 某一路质量很差时仍会引入噪声；
- 各路候选数量和权重仍需调节。

### 融合方式二：分数融合

可以先对两路分数归一化，再做加权：

```text
最终分数 = α × 稀疏分数 + (1 - α) × 稠密分数
```

但要注意：

- 不同 Query 下分数分布可能变化；
- min-max 归一化会受极端值影响；
- 更换模型或索引后需要重新校准；
- 权重可以按 Query 类型动态选择，而不必全局固定。

### Reranker 为什么放在最后

第一阶段检索要从全库快速找出可能相关的几十或几百条。Reranker 再同时读取 Query 与候选文本，做更精细的相关性判断。

```text
高召回、低成本候选生成 -> 低数量、高精度重排
```

它能修正部分稀疏/稠密融合顺序，但无法找回第一阶段完全漏掉的文档。

### 不要把稀疏检索等同于“老式关键词”

除了 BM25，还有 **SPLADE** 等学习式稀疏检索方法。它们用神经网络学习词项权重和扩展词，但结果仍可放入稀疏倒排结构。

这说明检索不是简单的“传统关键词 vs 先进向量”。真正要比较的是：

- 表示方法；
- 索引结构；
- 训练数据；
- 目标语料；
- 延迟与成本；
- 失败类型。

## 工程实践与边界

### 评测集应该覆盖什么

至少按 Query 类型分层：

1. 错误码、型号、政策编号；
2. 同义改写和口语表达；
3. 长问题和多条件问题；
4. 拼写错误、全半角和中英文混写；
5. 带日期、租户和 ACL 过滤；
6. 知识库中没有答案的问题；
7. 热门问题和长尾问题。

只用自然语言问答评测，会高估向量检索；只用编号查找，又会高估关键词检索。

### 做一组完整的消融实验

**消融实验**是一次移除或增加一个组件，观察它带来的真实变化。

| 实验组 | 用途 |
| --- | --- |
| BM25 only | 稀疏基线 |
| Dense only | 稠密基线 |
| BM25 + Dense + RRF | 验证混合召回收益 |
| Hybrid + Rerank | 验证精排收益 |
| Hybrid - 某一路 | 检查该路是否只在制造噪声 |

每组同时比较：

- Recall@K：相关文档是否进入前 K；
- MRR：第一个相关结果出现得多靠前；
- nDCG：多个不同相关等级结果的整体排序质量；
- P50/P95/P99、吞吐和错误率；
- 单次查询成本、索引大小和更新延迟；
- 不同 Query 类型的分项结果。

### Query 路由

如果混合检索成本过高，可以先判断 Query 类型：

- 含明确错误码：提高关键词权重；
- 纯自然语言描述：提高向量权重；
- 两者混合：并行召回；
- 低置信度：扩大召回或转入 Rerank；
- 结构化查询：直接走数据库/API。

路由器也会出错，因此要保留回退策略，并监控每类 Query 的收益。

### 索引一致性

同一文档更新时，稀疏索引和向量索引可能不是同时完成。工程上应：

- 使用稳定的 document_id/chunk_id；
- 记录文档和索引版本；
- 新版本双写或在 staging 索引完成后切换；
- 删除时同时处理两路索引和缓存；
- 融合前排除版本不一致的候选。

### 权限边界

每一路召回都要受到相同的租户、ACL 和有效期约束。不能让 BM25 路正确过滤，向量路却从全库返回敏感候选。

## 常见误区

### 误区一：BM25 只会机械匹配，已经没有价值

它对精确标识、罕见词、专业术语和短查询仍然很强，也是低成本、可解释的基线。

### 误区二：向量检索理解了语义，所以结果都相关

Embedding 会受训练数据、领域和文本截断影响；相似不等于能回答问题，仍需业务评测和重排。

### 误区三：混合检索就是分数直接相加

两路分数可能不在同一尺度。应使用排名融合或经过验证的分数校准。

### 误区四：RRF 的 `k` 固定为 60

论文或实现中的常用值不是所有数据的通用答案。候选深度、检索路数和业务分布改变后都应验证。

### 误区五：Recall 越高，最终答案越好

更多候选也可能增加噪声和上下文竞争。还需观察 Rerank、最终上下文和答案质量。

### 误区六：关键词检索不需要数据治理

分词、字段权重、同义词、停用词和文档结构会显著影响效果。倒排索引同样依赖高质量数据。

## 面试追问

### 1. 为什么错误码更适合关键词检索？

错误码的字符本身就是关键身份，语义模型可能把罕见字符串切分或弱化。精确词项和规范化字段通常更可靠。

### 2. 向量检索为什么能找到没有共同词的文档？

Embedding 模型在训练中学习了文本之间的语义关系，使相似含义在向量空间中接近，检索按向量距离而不是字面重合排序。

### 3. 混合检索反而变差，可能是什么原因？

弱召回路引入噪声、融合权重不合适、候选深度不足、分数未校准、权限/版本不一致，或 Reranker 对候选类型存在偏差。

### 4. 如何判断两路是否互补？

统计每一路独有的相关文档、两路交集和 union recall；按 Query 类型分析谁解决了对方的失败，而不是只看总平均分。

### 5. BM25 的 `k1` 和 `b` 是什么？

`k1` 控制词频饱和速度，`b` 控制文档长度归一化强度。无需死背默认值，应在目标语料和评测集上验证。

### 6. 向量相似度阈值怎么定？

按具体 Embedding、语料和查询类型，在有答案与无答案验证集上校准。相似度不是跨模型通用的置信概率。

### 7. 什么情况下只用关键词检索就够了？

查询以精确编号和规范术语为主、语料规模较小、同义表达少、低延迟和可解释性优先，且离线评测已满足质量要求时。

### 8. 什么情况下只用向量检索可能更合适？

用户表达高度自然化、文档措辞多样、缺少稳定关键词，并且目标 Embedding 在领域数据上验证良好时。但仍应保留精确字段查询处理编号和过滤。
