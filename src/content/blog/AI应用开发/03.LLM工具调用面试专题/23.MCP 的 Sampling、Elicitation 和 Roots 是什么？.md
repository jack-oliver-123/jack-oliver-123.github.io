## 本篇目标

学完这一篇，你应该能够：

- 解释为什么 Sampling、Elicitation、Roots 都属于 Client 侧能力；
- 说清 Server 发起、Client 控制、用户可拒绝的调用方向；
- 理解 Sampling 的模型选择、Human-in-the-loop 和工具循环；
- 区分 Elicitation Form mode、URL mode 与 MCP OAuth 授权；
- 描述 Roots 的 file URI、列表变化和路径安全；
- 区分协议能力、用户同意、操作系统权限和业务授权。

## 60 秒回答

Sampling、Elicitation 和 Roots 是 MCP 2025-11-25 的 Client 侧能力：由 Client 在 initialize 中声明，Server 只有在相应能力已协商后才能发请求。Sampling 让 Server 请求 Client 代表它调用模型，模型访问、实际选择、成本策略和用户审核仍在 Client；若使用 Sampling Tools，Client 还必须声明 sampling.tools，工具循环由 Server执行并受迭代上限控制。

Elicitation 让 Server 通过 Client 向用户补充信息。Form mode 使用受限、扁平的 JSON Schema 收集普通结构化数据，禁止索要密码、API key、access token 和支付凭证；敏感交互使用 2025-11-25 引入的 URL mode，让用户在外部安全页面输入，数据不经过 MCP Client。URL mode 不是 Client 访问 MCP Server 的 OAuth 授权替代方案。

Roots 让 Client 通过 roots/list 暴露当前允许的文件系统根，当前规范要求 file:// URI；声明 listChanged 后，根变化会通知 Server 重新获取。Roots 表达工作区边界，但真正文件访问仍需 Client 文件工具和操作系统权限强制执行。三种能力都不自动授予权限，Client 和用户可以拒绝，并应设置超时、限流、来源提示、审计和最小暴露。

## 详细解析

### 先说人话：Server 提需求，Client 保留钥匙和用户界面

想象一个外部顾问在公司里协助办事：

- 顾问需要分析时，请公司电脑调用内部模型；
- 顾问信息不够时，请前台向员工提问；
- 顾问要处理文件时，先问公司当前开放哪些工作区。

顾问可以提出请求，但：

- 模型账号和费用由公司电脑控制；
- 员工可以拒绝回答；
- 文件目录由公司选择；
- 顾问不能因为“协议支持”就获得全部权限。

MCP 中 Server 像顾问，Client/Host 像掌握用户、模型和本地资源的公司电脑。

### 贯穿案例：代码分析 MCP Server

一个代码分析 Server 希望：

1. 知道用户当前开放的仓库；
2. 读取代码后让 Client 模型总结；
3. 发现缺少部署环境信息，向用户提问；
4. 如果需要登录第三方扫描平台，引导用户到安全页面。

对应：

| 需求 | MCP 能力 |
| --- | --- |
| 获取开放仓库 | Roots |
| 请求 Client 调用模型 | Sampling |
| 询问普通结构化信息 | Elicitation Form mode |
| 处理凭证或第三方授权 | Elicitation URL mode |

### 最低前置概念：能力方向

常见 Server 能力：

~~~text
Client → Server
tools/list
tools/call
resources/read
prompts/get
~~~

本篇三种 Client 能力：

~~~text
Server → Client
sampling/createMessage
elicitation/create
roots/list
~~~

方向相反，所以 Server 处理 tools/call 时，可以在内部再嵌套请求 Client Sampling 或 Elicitation。

### 初始化中的能力声明

Client 示例：

~~~json
{
  "capabilities": {
    "roots": {
      "listChanged": true
    },
    "sampling": {
      "tools": {}
    },
    "elicitation": {
      "form": {},
      "url": {}
    }
  }
}
~~~

规则：

- Client 不声明，Server 不能发送对应请求；
- 声明基础 sampling 不自动表示支持 sampling.tools；
- Elicitation 至少声明 form 或 url；
- 为向后兼容，空 elicitation 对象等价于只支持 form；
- 能力声明不代表每个具体请求都会接受。

### 一、Sampling 是什么

Sampling 让 MCP Server 请求 Client 调用大模型。

~~~text
MCP Server
→ sampling/createMessage
→ MCP Client / Host
→ 用户审核与策略
→ Client 选择并调用模型
→ 返回采样结果
→ MCP Server 继续处理
~~~

### 为什么不是 Server 自己直接持有模型 key

Sampling 允许：

- 模型 API key 留在 Client；
- Client 统一模型选择；
- 用户控制成本和隐私；
- Host 应用执行内容策略；
- Server 不绑定某一家模型供应商；
- 用户查看和拒绝请求。

Server 可以表达偏好，但最终选择权在 Client。

### sampling/createMessage

概念示例：

~~~json
{
  "jsonrpc": "2.0",
  "id": 20,
  "method": "sampling/createMessage",
  "params": {
    "messages": [
      {
        "role": "user",
        "content": {
          "type": "text",
          "text": "请总结当前仓库的认证流程"
        }
      }
    ],
    "modelPreferences": {
      "costPriority": 0.3,
      "speedPriority": 0.5,
      "intelligencePriority": 0.9
    },
    "maxTokens": 800
  }
}
~~~

Client 可以：

- 展示给用户；
- 修改或裁剪上下文；
- 选择可用模型；
- 降低 maxTokens；
- 拒绝；
- 记录审计；
- 返回安全错误。

### Model Preferences 只是偏好

Server 可以提供：

- costPriority；
- speedPriority；
- intelligencePriority；
- 可选模型 hints。

这些是偏好，不是命令。Client 可能：

- 没有指定模型；
- 因数据驻留选择另一模型；
- 因预算降级；
- 因安全策略拒绝；
- 把 hint 映射到等价能力。

文章和代码不应把某个易过时模型名写成协议要求。

### Sampling 支持的内容

当前规范描述 Sampling 消息可包含：

- text；
- image；
- audio。

Client 仍要按自身能力和数据政策处理。Server 发送图片不代表 Client 必须把它交给任意外部模型。

### Human-in-the-loop

规范建议始终让用户能够拒绝 Sampling：

- 查看请求来源；
- 查看和编辑 Prompt；
- 查看拟发送上下文；
- 审核模型结果；
- 决定是否返回给 Server。

生产界面至少说明：

- 哪个 Server 请求；
- 请求目的；
- 会发送哪些数据；
- 预计影响；
- 使用什么策略；
- 用户如何拒绝。

低风险批量自动批准也必须有明确范围和撤销方式。

### Sampling with Tools

2025-11-25 允许 Server 在 Sampling 请求中提供 tools 和可选 toolChoice。

Client 必须声明：

~~~json
{
  "sampling": {
    "tools": {}
  }
}
~~~

未声明时，Server 不得发送 tool-enabled sampling。

### Sampling 工具循环

~~~text
Server → Client：Sampling 请求，附工具
Client → LLM：经审核的请求
LLM → Client：ToolUseContent
Client → Server：返回 tool use
Server：执行工具
Server → Client：新的 Sampling，请求中带 tool result
Client → LLM：继续
LLM → Client：最终文本
Client → Server：返回最终结果
~~~

工具实际执行在 Server 一侧，Client 负责模型调用与用户控制。

### 工具结果配对规则

当前 Sampling 规范要求：

- assistant 消息中的每个 ToolUseContent 都有匹配 toolUseId 的 ToolResultContent；
- 工具结果消息只能包含 tool_result，不能和 text、image、audio 混在同一用户消息；
- 所有工具使用得到结果后，才能继续下一条普通消息。

这有助于跨不同模型供应商 API 保持一致。

### Sampling 工具循环要有硬上限

- 最大迭代数；
- 最大工具调用数；
- 最大 token；
- 最大墙钟时间；
- 最大费用；
- 连续无新增证据停止；
- 相同调用去重；
- 最后一轮可禁用工具并要求最终回答。

Human-in-the-loop 不能代替资源预算。

### includeContext 的当前状态

2025-11-25 将 includeContext 的 thisServer 和 allServers 设为 soft-deprecated：

- 默认 none；
- Server 应避免依赖它们；
- 只有 Client 声明 sampling.context 时才可使用相关值；
- 未来版本可能移除。

这说明协议能力也会演进，代码要按协商版本处理。

### Sampling 的失败分支

| 失败 | 处理 |
| --- | --- |
| Client 未声明 sampling | Server 不发送 |
| 用户拒绝 | 返回明确错误，不反复弹窗 |
| Client 无可用模型 | 能力或服务错误 |
| Prompt 含敏感数据 | 裁剪、拒绝或本地模型 |
| 工具结果缺失 | Invalid params |
| 工具循环超限 | 停止并返回结构化错误 |
| 模型超时 | 遵守 deadline，不无限重试 |
| Server 提供危险工具 | Client 策略可拒绝 |

### 二、Elicitation 是什么

Elicitation 让 Server 通过 Client 向用户请求补充信息。

~~~text
Server
→ elicitation/create
→ Client 展示可信 UI
→ 用户 accept / decline / cancel
→ Client 返回结果
→ Server 继续流程
~~~

它不是让 Server 随意弹窗，而是受 Client 能力、界面和用户决定控制。

### Elicitation 的两种模式

| 模式 | 数据路径 | 适合 |
| --- | --- | --- |
| Form | 用户 → Client → Server | 普通结构化信息 |
| URL | 用户在外部页面 → Server/第三方 | 密码、API key、支付、第三方授权 |

### Form mode

请求示例：

~~~json
{
  "jsonrpc": "2.0",
  "id": 30,
  "method": "elicitation/create",
  "params": {
    "mode": "form",
    "message": "请选择部署环境",
    "requestedSchema": {
      "type": "object",
      "properties": {
        "environment": {
          "type": "string",
          "enum": ["development", "staging", "production"]
        }
      },
      "required": ["environment"]
    }
  }
}
~~~

### Form Schema 是受限子集

为了让 Client 生成一致表单，当前规范限制为扁平对象和原始属性，主要包括：

- string；
- number / integer；
- boolean；
- 单选 enum；
- 多选 enum；
- 部分 format 和边界。

不支持任意复杂嵌套对象和对象数组。Server 与 Client 都要按当前规范子集实现。

### Form mode 禁止敏感凭证

Server 不得用 Form mode 请求：

- 密码；
- API key；
- access token；
- 支付凭证；
- 能直接授予访问或交易权限的秘密。

一般联系信息并非协议上统一禁止，但仍要说明目的、最小化收集，并让用户查看和拒绝。

### Response Actions

三种结果：

| action | 含义 |
| --- | --- |
| accept | 用户明确提交 |
| decline | 用户明确拒绝 |
| cancel | 用户关闭、退出或未明确选择 |

Server 应区分：

- decline：尊重明确拒绝，提供替代；
- cancel：可以稍后恢复，但不要反复骚扰；
- accept：仍要校验 content，不因用户提交就完全可信。

### URL mode

URL mode 是 2025-11-25 引入的新能力，适合敏感交互不经过 MCP Client。

~~~json
{
  "jsonrpc": "2.0",
  "id": 31,
  "method": "elicitation/create",
  "params": {
    "mode": "url",
    "elicitationId": "elicit-7",
    "url": "https://scanner.example.com/connect",
    "message": "请连接代码扫描服务"
  }
}
~~~

必须包含：

- mode=url；
- message；
- url；
- 唯一 elicitationId。

### accept 不代表 URL 流程已完成

Client 返回 accept 只表示：

> 用户同意打开这个 URL。

不代表：

- 已登录；
- 已支付；
- 已授权第三方；
- Server 已取得凭证；
- 原工具调用可以立即成功。

Server 完成外部流程后，可以向发起该请求的 Client 发送 notifications/elicitation/complete。Client 也要保留手动重试或取消，不能无限等待通知。

### URLElicitationRequiredError

如果原请求必须先完成 URL 交互，Server 可以返回错误码 -32042，并在 data 中列出所需 URL elicitations。Client 在用户完成后可以重试原请求。

重试仍需：

- 原调用幂等；
- 绑定同一用户；
- 检查状态；
- 不重复副作用。

### URL mode 不是 MCP OAuth 替代

两条不同链路：

~~~text
MCP Client → MCP Server
使用 MCP Authorization 规范
~~~

~~~text
MCP Server → 第三方服务
可通过 URL Elicitation 让用户完成外部授权
~~~

MCP Server 不能用 URL mode 绕过 Client 到 Server 的正式授权。Client 的 Bearer token 也不会因此改变。

### URL mode 的安全要求

Client：

- 不自动预取 URL 或元数据；
- 未经明确同意不打开；
- 先展示完整 URL；
- 突出目标域；
- 对 Punycode 和可疑 URI 警告；
- 用不让 Client 或 LLM 检查用户输入的安全浏览器；
- 不把页面凭证带回模型。

Server：

- URL 中不放密码、token、PII 等敏感信息；
- 不提供可冒充用户的预认证 URL；
- 非开发环境使用 HTTPS；
- 把 elicitation 绑定到 Client 与用户；
- 验证打开页面的人正是发起请求的人；
- 安全存储第三方凭证；
- 不把第三方 token 回传 Client。

### 防 URL Elicitation 钓鱼

攻击者可能把自己的 elicitation URL 发给受害者，让受害者授权后，Server 错绑到攻击者账户。

Server 要：

- 由 MCP Authorization 的可信 subject 识别发起用户；
- 打开 connect 页面时重新验证用户会话；
- 比较发起 subject 与完成 subject；
- 绑定 state / elicitationId；
- 防 URL 参数被篡改；
- 不只相信 Client 传来的用户名。

### 三、Roots 是什么

Roots 让 Client 告诉 Server：

> 当前工作区允许你在这些文件系统根范围内工作。

请求：

~~~json
{
  "jsonrpc": "2.0",
  "id": 40,
  "method": "roots/list"
}
~~~

响应：

~~~json
{
  "jsonrpc": "2.0",
  "id": 40,
  "result": {
    "roots": [
      {
        "uri": "file:///workspace/frontend",
        "name": "Frontend"
      },
      {
        "uri": "file:///workspace/backend",
        "name": "Backend"
      }
    ]
  }
}
~~~

当前 2025-11-25 规范要求 Root URI 使用 file://。name 是可选的人类可读名称。

### Roots 的来源

Client 可以通过：

- 用户工作区选择器；
- 当前项目；
- 版本控制仓库；
- 配置文件；
- 用户临时授权

生成 roots。协议不强制具体 UI，但建议在暴露前获得用户同意。

### Root 列表变化

Client 声明：

~~~json
{
  "roots": {
    "listChanged": true
  }
}
~~~

变化后发送：

~~~json
{
  "jsonrpc": "2.0",
  "method": "notifications/roots/list_changed"
}
~~~

Server 收到后重新 roots/list，并：

- 停止使用已移除 root；
- 取消或拒绝新操作；
- 失效路径缓存；
- 检查进行中的任务；
- 更新索引或文件监听。

### Roots 不是自动文件权限

Root 表达协议级工作范围，但安全仍要由 Client 和文件工具执行：

- 只暴露用户允许目录；
- 路径规范化；
- 防 ../；
- 防符号链接和 junction 逃逸；
- 检查最终真实路径；
- 挂载和操作系统权限；
- 只读与可写分离；
- 每次调用授权。

Server 应尊重 roots，但不能仅靠自己的自觉形成安全边界。

### 三种能力对比

| 维度 | Sampling | Elicitation | Roots |
| --- | --- | --- | --- |
| Server 想要 | 模型生成 | 用户补充信息/外部交互 | 文件工作范围 |
| 请求方向 | Server → Client | Server → Client | Server → Client |
| Client 控制 | 模型、Prompt、批准 | UI、同意、数据发送 | 暴露哪些目录 |
| 用户可拒绝 | 是 | 是 | 是 |
| 是否自动授权 | 否 | 否 | 否 |
| 主要风险 | 成本、数据、工具循环 | 钓鱼、秘密收集 | 路径越界、文件泄露 |

### 嵌套请求与死锁

一个 tools/call 处理中，Server 发出 Elicitation：

~~~text
Client 等 tools/call 完成
Server 等用户 Elicitation
UI 又因 tools/call 占用而无法展示弹窗
→ 死锁
~~~

实现要：

- 支持双向并发 Request；
- 每个 Request 独立 id；
- Elicitation UI 不阻塞消息泵；
- 设置嵌套深度；
- 独立 timeout；
- 取消能传播；
- 不持有数据库锁等待用户。

Sampling 工具循环也要防递归：Sampling 的工具又触发同一个 Server Sampling，可能无限嵌套。

## 工程实践与边界

### Client 统一策略

~~~yaml
sampling:
  require_user_review: true
  max_tokens: configured
  max_tool_iterations: 5
  allowed_data_classes:
    - public
    - internal

elicitation:
  form:
    allow_contact_data: false
    max_fields: 10
  url:
    require_explicit_navigation_consent: true
    allow_https_only: true

roots:
  user_selected_only: true
  default_mode: read_only
  allow_symlinks_outside_root: false
~~~

数值和策略只是结构示例，应按产品和风险确定。

### Sampling 执行伪代码

~~~python
def handle_sampling(request, server_identity, user):
    require_capability("sampling")
    validate_request(request)
    enforce_rate_limit(server_identity, user)

    if request.tools:
        require_sub_capability("sampling.tools")

    reviewed = user_review(request)
    if not reviewed.approved:
        return error("USER_REJECTED")

    sanitized = apply_data_policy(reviewed.request)
    model = choose_model(
        preferences=sanitized.model_preferences,
        user_policy=user.policy,
        data_class=sanitized.data_class,
    )

    return call_model_with_budget(model, sanitized)
~~~

### Elicitation Form 处理

~~~text
检查 elicitation.form
→ 验证 requestedSchema 属于允许子集
→ 检测敏感字段意图
→ 显示 Server 身份和原因
→ 用户填写、审查
→ Client Schema 校验
→ accept / decline / cancel
→ Server 再次校验
~~~

### URL 处理

~~~text
检查 elicitation.url
→ 校验 URL 格式和策略
→ 不预取
→ 展示完整 URL 与域名
→ 用户明确同意
→ 在安全浏览器打开
→ Client 不读取页面凭证
→ 等可选 complete 通知或用户手动恢复
~~~

### Roots 文件访问伪代码

~~~python
requested = normalize_path(tool_arguments.path)
root = find_matching_user_root(requested)

if root is None:
    deny("PATH_OUTSIDE_ROOT")

real_path = resolve_symlinks_safely(requested)

if not is_within(real_path, root.real_path):
    deny("PATH_ESCAPE")

authorize_file_action(user, action, real_path)
~~~

文件系统沙箱仍应只挂载 roots。

### 监控指标

#### Sampling

- 请求数与来源 Server；
- 用户批准、修改、拒绝；
- 模型选择与偏好偏差；
- token、费用、延迟；
- Sampling tool 迭代；
- 循环和超限；
- 敏感数据阻断。

#### Elicitation

- form / url 数量；
- accept / decline / cancel；
- 不支持 mode 错误；
- 敏感字段阻断；
- URL 域分布和风险警告；
- completion 通知延迟；
- 身份绑定失败。

#### Roots

- 暴露 root 数；
- list_changed；
- 越界路径拦截；
- symlink 逃逸拦截；
- root 移除后仍访问；
- 用户授权变化。

### 审计

记录：

- Client 与 Server 身份和版本；
- 协商 capability；
- 请求类型和 id；
- 用户决定；
- 数据分类；
- 模型和预算策略；
- Elicitation URL 域；
- Roots 名称和安全摘要；
- 错误、超时和取消。

不记录：

- 密码；
- API key；
- access token；
- 支付凭证；
- 没必要的完整 Prompt；
- 用户在外部 URL 页面输入的敏感数据。

### 安全边界

- Server 提供的 Prompt、message、URL、tool 和 root 名称都不可信；
- Client 声明 capability 不代表自动同意；
- Sampling hint 不授予模型选择权；
- Elicitation accept 不代表外部流程完成；
- URL mode 不授权 Client 访问 MCP Server；
- Roots 不授予 Server 任意 OS 文件权限；
- User approval 不代替 Schema 和授权；
- 嵌套请求不应持有长事务或锁。

### 协议边界

Sampling 不定义：

- 具体模型供应商；
- 固定模型名；
- 价格；
- Client 必须如何展示 UI；
- 工具业务授权。

Elicitation 不定义：

- 所有表单组件；
- 第三方 OAuth 具体流程；
- 支付服务实现；
- 自动批准策略。

Roots 不定义：

- 通用云对象存储根；
- 完整文件沙箱；
- 读写权限模型；
- Server 可直接访问文件的保证。

## 常见误区

- **“Sampling 是 Server 直接调用 Host 模型 API”**：Server 发请求，Client 保留模型与权限控制。
- **“Server 指定 model hint，Client 必须照做”**：hint 是建议，Client 最终选择。
- **“声明 sampling 就支持 Sampling Tools”**：还要声明 sampling.tools。
- **“Elicitation 可以用表单收密码”**：敏感凭证必须走 URL mode。
- **“URL mode 就是 MCP OAuth”**：它用于 Server 的外部敏感交互，不替代 Client 到 Server 授权。
- **“URL accept 就表示授权完成”**：只表示同意打开，完成要由外部状态确认。
- **“Roots 等于操作系统权限”**：实际文件访问仍需 Client 工具和 OS 强制。
- **“能力声明后 Server 的每个请求都必须接受”**：Client 策略和用户仍可拒绝。

## 面试追问

### 1. 为什么三者都是 Client capability？

因为实际模型访问、用户交互和本地工作区掌握在 Client/Host 手中，Server 只能向 Client 请求这些能力。

### 2. Sampling 为什么不把 API key 给 Server？

模型调用由 Client 执行，Server 只发 createMessage 请求。这样 Client 可以控制凭证、模型选择、成本、数据政策和用户审核。

### 3. sampling.tools 如何工作？

Client 声明子能力后，Server 可在 Sampling 请求中提供 tools。模型返回 tool use，Server 执行并把 tool result 放入下一次 Sampling，直到最终回答或达到上限。

### 4. Form 与 URL Elicitation 怎么选？

普通、可审查的结构化信息用 Form；密码、API key、支付和第三方授权等敏感交互用 URL，让数据不经过 MCP Client。

### 5. URL mode 与 MCP Authorization 有何不同？

MCP Authorization 保护 Client 对 MCP Server 的 HTTP 访问；URL mode 可让 MCP Server 获取第三方凭证或完成外部敏感流程，两套 token 和角色边界不同。

### 6. URL accept 后为什么不能立刻重试写操作？

accept 只表示用户同意导航，不证明外部流程完成。应等待 Server 状态或 completion 通知，并让原写操作具备幂等和状态检查。

### 7. Roots 变化后 Server 怎么办？

收到 list_changed 后重新 roots/list，停止使用已移除目录，失效缓存和索引，并处理正在运行的文件任务。

### 8. Roots 如何防路径逃逸？

Client 文件执行层规范化并解析真实路径，检查仍位于用户授权 root，处理符号链接与大小写，并用文件系统挂载提供第二层限制。
