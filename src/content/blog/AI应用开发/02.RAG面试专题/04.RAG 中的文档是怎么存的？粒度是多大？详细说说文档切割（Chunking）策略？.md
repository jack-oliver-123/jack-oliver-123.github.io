## 本篇目标

读完本篇，你应该能够：

- 区分原始文档、解析结果、Chunk、Embedding 和索引；
- 解释为什么向量不能替代原文与元数据；
- 说清 Chunk 的长度为什么没有通用固定答案；
- 对比固定长度、结构、递归、重叠、语义和父子切块；
- 设计一套可以更新、删除、回滚和评测的文档数据模型。

## 60 秒回答

生产 RAG 通常不会只保存向量，而是分层保存：权威原始文档、解析后的结构化内容、用于检索的 Chunk、Chunk 对应的 Embedding、关键词或向量索引，以及来源、版本、有效期和 ACL 等元数据。向量只是用于相似性检索的派生表示，答案生成、引用追溯、权限过滤和删除更新仍然需要原文与元数据。

Chunking 没有通用固定长度。我会先按标题、段落、列表、表格和代码等结构边界切分，块过长再按句子或 Token 递归切分；根据任务选择受控重叠、父子切块或多表示检索。用于召回的粒度可以较小，命中后送给模型的上下文可以扩展到父章节或相邻段落。

具体长度、重叠和 Top-K 要在目标查询集上比较 Recall@K、命中证据完整性、重复率、上下文 Token、答案质量、延迟和成本，同时受 Embedding 最大输入长度与生成模型上下文预算约束。

## 详细解析

### 先说人话：不能把一本书只存成“坐标”

假设图书馆把每本书变成一个数字坐标，然后把原书全部扔掉。

以后读者虽然可能根据坐标找到“与差旅有关的书”，却看不到正文，也无法确认版本、页码、作者和借阅权限，更无法在制度废止后准确删除其中某个章节。

RAG 中的向量也类似。它便于比较语义，却不是原文的可逆压缩，更不是完整的数据管理方案。

一套可维护的知识库通常保留：

> 原始资料 -> 可解析的结构 -> 可检索 Chunk -> 向量与索引 -> 来源与治理元数据。

### 文档存储的五个层次

#### 第一层：权威原始文档

原始 PDF、Word、网页快照、数据库记录或 API 数据通常保存在业务系统、文档平台或对象存储中。

这一层负责回答：

- 原始内容到底是什么；
- 谁拥有这份资料；
- 当前权威版本在哪里；
- 发生争议时怎样核对；
- 删除或更新由哪个系统发起。

向量库中的文本副本通常不是权威源。它是为了检索生成的派生数据。

#### 第二层：解析后的结构

解析器把原文转换成便于处理的结构，例如：

- 文档标题；
- 章节树；
- 段落；
- 列表；
- 表格；
- 图片与说明；
- 页码和坐标；
- 阅读顺序。

保留结构非常重要。只得到一大串纯文本，会丢失“这一段属于哪个标题”“这个金额对应哪一列”等信息。

例如，差旅标准表格包含：

| 城市级别 | 普通员工 | 经理 | 备注 |
| --- | --- | --- | --- |
| 一线城市 | 某金额 | 某金额 | 旺季需审批 |

如果解析后只剩几个金额，模型无法知道每个数字对应哪个职级。

#### 第三层：可检索 Chunk

Chunk 是为检索构造的内容单元。它通常包含正文和元数据，而不是只有一段字符串。

一个 Chunk 可能有以下字段：

| 字段 | 作用 | 示例 |
| --- | --- | --- |
| document_id | 关联原文 | travel-policy-2026 |
| chunk_id | 稳定标识片段 | travel-policy-2026-4-2-01 |
| parent_id | 关联父章节 | travel-policy-2026-4-2 |
| content | 用于检索和生成的文本 | 上海住宿标准与审批条件 |
| section_path | 提供层级语境 | 差旅 / 国内 / 住宿 |
| source_uri | 回到原文 | 内部文档地址 |
| page_or_offset | 定位原始位置 | 第 8 页或字符区间 |
| version | 区分版本 | 2026-v1 |
| valid_from | 生效时间 | 2026-01-01 |
| valid_to | 失效时间 | 空，表示尚未失效 |
| acl | 访问控制 | 中国区正式员工 |
| content_hash | 判断内容变化 | 正文哈希 |
| parser_version | 复现解析结果 | parser-3.2 |
| chunker_version | 复现切块结果 | policy-structure-v4 |

稳定 ID 很重要。更新、删除、引用和问题回放都需要找到同一个数据单元。

#### 第四层：Embedding 与索引

Embedding 是由 Chunk 内容生成的向量。索引是为了快速搜索这些向量或关键词而建立的数据结构。

可以同时拥有：

- 向量字段；
- 关键词倒排索引；
- 文档 ID、租户、权限、日期等过滤字段；
- 用于重排的原文；
- 用于展示的标题和来源。

Embedding 模型变化后，新旧向量空间通常不可直接混用，因此模型版本也要保存在索引配置中。

#### 第五层：版本、发布和血缘

数据血缘是“某个结果由哪些输入和处理步骤生成”的关系。

一次回答最好能够追溯：

> 答案 -> 最终上下文 -> Chunk -> 切块配置 -> 解析结果 -> 原始文档版本。

这样才能回答“为什么昨天能搜到，今天搜不到”。

### 什么是 Chunking

Chunking 就是把长文档切成更适合检索与使用的片段。

为什么要切？

1. Embedding 模型有最大输入长度；
2. 整篇文档主题太多，一个向量难以准确代表所有细节；
3. 返回整篇文档会占用大量上下文；
4. 用户问题通常只对应其中一小部分；
5. 细粒度 Chunk 更容易绑定引用位置。

但是切得过小也有问题：

- 片段失去标题和适用范围；
- 代词找不到指代对象；
- 定义与例外被分开；
- 召回大量相似碎片；
- 重复元数据和索引开销增加。

所以 Chunking 本质上是在“检索精度”和“语义完整性”之间做取舍。

### Token、字符和词有什么区别

Token 是模型分词器处理文本时使用的单位。一个中文字符可能对应一个或多个 Token，英文单词也可能被拆成多个词片段。

字符数不能稳定代表 Token 数。具体关系取决于模型和 tokenizer。

切块时应以目标 Embedding 模型的 tokenizer 测量输入长度，同时还要考虑生成模型最终能接收多少上下文。

### 两种粒度不要混为一谈

这是面试中很加分的一点：

> **用于检索的 Chunk 粒度，与送给生成模型的上下文粒度，可以不同。**

例如：

1. 把“上海住宿标准”切成一个 150 Token 的子块，用它生成 Embedding；
2. 子块被命中后，取回包含它的 600 Token 父章节；
3. 父章节带有标题、适用职级和例外条件；
4. 最终把完整证据送给模型。

小块利于精确定位，大块利于完整理解。这就是父子检索的基本思路。

### 常见 Chunking 策略

#### 固定长度切块

按固定字符数或 Token 数切分。

优点：

- 实现简单；
- 结果可重复；
- 处理速度快；
- 适合作为基线。

缺点：

- 容易在句子或表格中间切断；
- 不理解文档结构；
- 相同长度不代表相同信息密度。

适合作为最初对照组，不应未经评测就当成最终策略。

#### 固定长度加重叠

相邻 Chunk 共享一部分内容。例如前一块结尾的几句话会出现在后一块开头。

重叠可以缓解局部边界断裂，但会：

- 增加 Chunk 数量和索引体积；
- 让同一证据重复进入 Top-K；
- 浪费生成上下文；
- 增加更新与去重工作。

重叠应该根据边界样本评测，而不是直接套用固定比例。

#### 结构切块

按文档天然结构切分，例如：

- Markdown 标题；
- HTML DOM 节点；
- Word 段落和样式；
- PDF 版面区域；
- 列表和表格；
- 代码函数或类。

结构切块通常更容易保留语义边界，也便于记录章节路径。

前提是解析器正确恢复了结构。解析错误时，结构切块会把错误继续传下去。

#### 递归切块

先尝试按较大的语义边界切分；块仍然过长时，再依次按段落、句子、词或 Token 切分。

例如：

> 章节 -> 小节 -> 段落 -> 句子 -> Token

递归切块兼顾结构与长度限制，适合作为通用起点。

#### 句子切块与句子窗口

用句子作为最小检索单元，命中后返回前后若干句。

它适合事实定位，但单个句子可能缺少标题、主语或限定条件。窗口扩展后需要重新做权限、版本和 Token 检查。

#### 语义切块

先为句子或段落生成表示，再根据相邻内容的主题变化寻找切分点。

优点是有机会识别没有显式标题的主题边界。

代价包括：

- 需要额外 Embedding 计算；
- 依赖相似度阈值；
- 结果随模型和版本变化；
- 不如结构规则容易解释和复现。

它是否优于结构切块，要在目标文档上验证。

#### 父子切块

建立父章节和子 Chunk 两种层级：

- 子块用于检索；
- 父块用于给模型提供完整上下文。

它适合制度、手册和技术文档。父块过大时，可以只返回父标题、命中子块和必要邻居，而不是把整个章节全部送入模型。

#### 多表示或摘要检索

同一原文可以建立多个检索表示：

- 原文 Chunk；
- 标题加正文；
- 为 Chunk 生成的摘要；
- 为 Chunk 生成的可能问题；
- 表格的结构化描述。

检索命中某个表示后，返回它所指向的权威原文。

这种方法可能提升难以直接匹配的查询，但增加生成成本、索引规模和数据血缘复杂度。生成的摘要或问题也可能引入错误，不能替代原文。

### 不同文档需要不同策略

| 文档类型 | 主要问题 | 建议起点 |
| --- | --- | --- |
| 员工制度 | 标题层级、例外条款 | 结构 + 递归 + 父子 |
| FAQ | 一问一答边界清楚 | 每组问答为一个 Chunk |
| 新闻文章 | 段落连续叙事 | 标题 + 段落 + 受控重叠 |
| 合同 | 条款编号与引用关系 | 按条款切分并保留层级 |
| 表格 | 表头与单元格关系 | 按逻辑区域，重复必要表头 |
| 代码 | 函数、类和依赖 | 语法树或符号结构 |
| 聊天记录 | 说话人和时间顺序 | 按会话窗口并保留角色 |
| 扫描 PDF | OCR 与阅读顺序 | 先做版面和 OCR 质量处理 |

### Chunk 应该多大

没有脱离场景的通用数字。需要考虑：

#### 问题需要多大证据范围

“报销发票抬头是什么”可能只需一两句话；“比较国内与海外差旅政策”需要多个章节。

#### 文档信息密度

法规和代码每句话信息密度高；叙述性文章可能需要更长片段才有完整语境。

#### Embedding 模型输入限制

超过最大输入长度的部分可能被截断或拒绝。还要确认模型建议的查询与文档编码方式。

#### 生成上下文预算

最终上下文还要容纳系统规则、用户问题、历史对话和模型输出预算。

#### 检索方式

小块通常更精确，但缺上下文；大块更完整，但向量主题更混杂。父子检索可以分开处理这两个目标。

### 怎样选择长度和重叠

可以建立实验矩阵：

| 实验变量 | 示例候选 |
| --- | --- |
| 切块方法 | 固定、结构、递归、父子 |
| 子块长度 | 短、中、长三档 |
| 重叠 | 无、句子级、较小 Token 窗口 |
| 返回上下文 | 子块、父块、相邻扩展 |
| 召回 K | 多个候选值 |

在同一评测集、Embedding 模型和索引参数下比较：

- Recall@K；
- MRR 或 nDCG；
- 最小充分证据是否完整；
- 重复片段比例；
- 平均上下文 Token；
- 端到端答案正确性与忠实度；
- P95 延迟和成本。

只比较最终答案分数，可能看不出切块究竟改善了召回还是只是让模型猜得更好。

### 用员工制度做一次完整切块

原始章节：

> 4.2 国内住宿标准
>
> 普通员工在一线城市住宿标准为……
>
> 经理级员工标准为……
>
> 每年 4 月和 10 月展会期间，上海标准经审批后可上浮……
>
> 申请人应在出发前完成审批。

不好的切法可能是：

- Chunk A：普通员工金额；
- Chunk B：旺季上浮条件；
- Chunk C：需要提前审批。

用户只命中 Chunk A 时，会漏掉审批条件。

更合理的设计可以是：

- 子块 1：普通员工基础标准；
- 子块 2：经理级基础标准；
- 子块 3：上海旺季例外；
- 父块：完整的“4.2 国内住宿标准”；
- 每个子块都带标题路径和适用范围；
- 命中子块后，根据问题扩展父块或相邻例外条款。

这不是说父块要原样全部返回，而是让检索层有能力恢复完整证据。

### 更新与删除怎样处理

文档更新时不要只追加新 Chunk。常见流程是：

1. 根据文档 ID 找到旧版本；
2. 计算新内容哈希；
3. 重新解析发生变化的内容；
4. 生成新 Chunk 与向量；
5. 写入影子索引；
6. 校验数量、权限和检索结果；
7. 原子切换新版本；
8. 让旧版本退出查询；
9. 按保留策略清理旧数据与缓存。

如果只新增新版而不处理旧版，检索可能同时返回两个冲突金额。

### 向量存储开销怎样估算

仅看原始浮点向量，粗略大小可以用：

> 向量数量 × 维度 × 每个数值的字节数

例如 float32 每个数值通常占 4 字节。但真实存储还包含：

- ANN 索引结构；
- 主键和元数据；
- 原文；
- 副本；
- 写前日志或持久化开销；
- 压缩或量化。

所以公式只能做容量起点，生产容量应以具体数据库版本和真实数据压测。

## 工程实践与边界

### 建议的数据契约

每个 Chunk 至少应有：

- 稳定 document_id 和 chunk_id；
- 原文或可安全回源的地址；
- 章节路径与原始偏移；
- 内容哈希；
- 文档、解析器、切块器和 Embedding 版本；
- 租户、ACL、有效期和数据分类；
- 创建、更新与删除状态。

### 发布切块策略要像发布代码

切块配置变化会改变整个索引，不应直接覆盖：

1. 固定评测集；
2. 构建新索引；
3. 做数据完整性检查；
4. 比较新旧检索和端到端指标；
5. 灰度；
6. 观察线上失败率与延迟；
7. 保留快速回滚能力。

### 权限继承

子 Chunk 默认继承原文权限。父子扩展、相邻窗口和多路召回的每个结果都要重新验证权限，不能因为“父块已经命中”就跳过检查。

### 解析失败要显式处理

不要把空文本、乱码或表格错位静默写入索引。应记录：

- 解析成功率；
- OCR 置信度；
- 空 Chunk；
- 异常长度；
- 重复内容；
- 不支持的文件类型。

低质量文档可以进入隔离区等待人工复核。

### 一份切块检查清单

- 是否保留标题和层级；
- 是否在句子、表格或代码中间硬切；
- 是否保存原文偏移；
- 是否存在大量重复 Chunk；
- 是否能按 document_id 删除全部派生内容；
- 是否区分检索粒度和生成上下文粒度；
- 是否覆盖跨边界问题；
- 是否记录 Embedding 最大输入限制；
- 是否在真实问题集上评测；
- 是否能独立回滚索引版本。

## 常见误区

- **“固定 500 Token 是通用方案”**：Chunk 长度取决于文档、问题、模型和上下文预算，需要实验验证。
- **“重叠越大，语义越完整”**：过多重叠会造成重复召回、索引膨胀和 Token 浪费。
- **“向量维度决定 Chunk 长度”**：向量维度是输出表示大小，输入长度由模型与 tokenizer 配置决定。
- **“向量就是压缩后的原文”**：Embedding 不是可逆文本存储，不能替代正文。
- **“检索块和生成块必须相同”**：可以小块检索、父块或邻居扩展后生成。
- **“语义切块比结构切块更高级”**：两者适用条件和成本不同，效果要按目标数据验证。
- **“只要保存 content 和 vector 就够了”**：缺少来源、版本、ACL 和血缘会导致无法治理与回放。
- **“文档更新只需要新增新版”**：旧版、缓存和派生数据不退出查询，会产生冲突和陈旧答案。

## 面试追问

1. **如何评估 Chunking 策略？**

   固定查询集、Embedding、索引和重排设置，对比 Recall@K、排序、充分证据完整率、重复率、上下文 Token、答案质量、延迟和成本。

2. **为什么小块检索、大块生成可能更好？**

   小块主题集中，利于精确匹配；较大的父上下文可以补回标题、定义和例外。两层分别优化召回与理解。

3. **表格应该怎样切？**

   保留表名、表头、行列关系和适用说明。可按逻辑行组生成文本表示，同时保存原表位置；不要把单元格打散成无上下文数字。

4. **如何避免孤儿向量？**

   所有派生记录绑定 document_id 与版本，更新删除通过事件或事务化作业级联处理，并用对账任务检查原文、Chunk 和索引数量。

5. **修改 Chunk 长度后为什么要重建索引？**

   Chunk 的内容、数量、ID 和 Embedding 都变化了。新旧结果混用会造成重复、引用错位和不可复现。

6. **Overlap 应该怎样选？**

   先收集答案跨边界的查询，再比较无重叠、句子重叠和不同小窗口；同时观察边界召回、重复率与 Token 成本。
