## 本篇目标

用同一个任务对比三种范式，掌握“边走边判断、先计划再执行、根据反馈再修改”的区别。读完后，你应该能够给出有依据的选型，而不是回答“看情况”。

## 60 秒回答

ReAct 是“每获得一个新观察，就决定下一步动作”，适合路径较短、环境反馈频繁且步骤无法提前确定的任务；Plan-and-Execute 是“先生成带依赖和验收条件的高层计划，再逐步执行并在必要时重规划”，适合有长依赖链的复杂任务；Reflection 是“根据测试、规则或评价证据进行有限次修正”，适合结果可验证、重复修改有收益的任务。

三者不是互斥的。常见组合是由 Planner 生成少量高层步骤，每一步用受限 ReAct 执行，关键产物再经过一次基于明确标准的评价。选型要看路径长度、反馈频率、结果可验证性、副作用风险和预算，并通过基线评测证明新增循环带来了净收益。

## 详细解析

### 先用一句话区分

- **ReAct**：下一步去哪，走到这里再决定；
- **Plan-and-Execute**：先规划整条路线，再按计划走；
- **Reflection**：走完后检查哪里不对，再有针对性地修正。

它们分别解决三个不同问题：

| 问题 | 更适合的范式 |
| --- | --- |
| 现在不知道下一步要调用什么工具 | ReAct |
| 任务很长，步骤之间有依赖 | Plan-and-Execute |
| 已经有结果，但质量可能不达标 | Reflection |

### 用同一个“制作竞品分析报告”案例理解

用户目标：

> 调研三款客服 Agent 产品，给出适合中小企业的选型报告，所有关键结论都要有来源。

#### 如果使用 ReAct

Agent 可能这样运行：

1. 搜索产品 A；
2. 观察结果，发现缺少价格；
3. 改用官方定价页；
4. 搜索产品 B；
5. 发现资料存在冲突，再查发布日期；
6. 判断三款产品信息已经足够；
7. 生成报告并结束。

它不要求一开始就知道要搜索多少次，而是根据每次结果继续判断。

优势：

- 适应网页变化和信息缺失；
- 能根据新证据调整关键词；
- 每一步都能使用最新观察。

风险：

- 搜着搜着偏离“中小企业选型”这个目标；
- 重复搜索相同内容；
- 忘记第三款产品；
- 上下文越来越长。

#### 如果使用 Plan-and-Execute

系统先生成高层计划：

1. 确定统一评价维度；
2. 收集三款产品的官方资料；
3. 按相同维度结构化对比；
4. 根据中小企业约束给出建议；
5. 检查所有关键结论的来源。

每一步还有输入、输出、依赖和完成标准。例如第 3 步必须等第 2 步收集完三款产品资料。

优势：

- 始终保留全局目标；
- 依赖关系清楚；
- 可以从失败步骤恢复；
- 方便并行执行独立子任务。

风险：

- 初始计划可能遗漏重要维度；
- 网站变化会让计划过期；
- 计划写得太细，执行中频繁返工；
- 只顾完成步骤，却没有真正满足用户目标。

#### 如果使用 Reflection

系统先生成初稿，再用验证器检查：

- 是否覆盖三款产品；
- 价格是否来自官方页面；
- 每个关键结论是否有引用；
- 建议是否结合中小企业预算和运维能力；
- 表格字段是否完整。

如果验证器发现产品 B 的价格缺少来源，它应返回具体问题：

> 失败项：产品 B 价格无官方引用；允许修改范围：价格段落和对比表。

优化器只补充相关信息，再重新验证。

优势：

- 能利用真实失败证据改进；
- 适合代码测试、格式校验和引用检查；
- 可在交付前建立质量门禁。

风险：

- 评价器和生成器可能共享偏差；
- 评价标准不清时只会反复润色；
- 模型可能迎合评分器，而不是真正提高质量；
- 多轮修改会增加延迟和费用。

### ReAct：用局部反馈换灵活性

ReAct 的决策节奏最细。每轮只需要回答：

> 根据当前状态，下一步最有价值的动作是什么？

适合：

- 搜索与调查；
- 故障诊断；
- 客服信息收集；
- 工具可用性不断变化；
- 步数无法提前确定。

不适合：

- 路径完全固定；
- 每一步都有重大副作用；
- 很长的依赖关系需要全局协调；
- 极低延迟的任务。

ReAct 的关键保护是最大轮数、重复动作检测、无进展检测和目标状态摘要。

### Plan-and-Execute：把全局规划和局部执行分开

一个真正可执行的计划不是普通待办清单。每一步至少应包含：

| 字段 | 含义 |
| --- | --- |
| 步骤 ID | 稳定标识 |
| 目标 | 这一步要解决什么 |
| 依赖 | 哪些步骤完成后才能开始 |
| 输入 | 执行所需信息 |
| 预期输出 | 应产出什么结构 |
| 工具请求 | 可能需要哪些能力 |
| 验收条件 | 怎样证明这一步完成 |
| 状态 | 待执行、进行中、成功、失败或跳过 |

Planner 只负责提出计划，不能授予工具权限。调度器仍要根据真实用户身份和策略计算可执行工具。

#### 什么情况下重规划

不是每做一步都重新规划。常见触发条件是：

- 前置条件不成立；
- 关键工具不可用；
- 执行结果没有达到验收条件；
- 用户目标或硬约束发生变化；
- 发现原计划缺少必要步骤。

已经完成并产生副作用的步骤不能简单改回“未执行”。例如邮件已经发送，重规划只能补发更正或走补偿流程。

### Reflection：反馈质量决定上限

Reflection 的基本结构是：

> 生成 -> 验证 -> 反馈 -> 定向修改 -> 再验证。

评价信号按可靠程度可以大致排序：

1. 可执行测试；
2. Schema 和确定性规则；
3. 权威数据或引用核验；
4. 人工反馈；
5. 经校准的模型评分；
6. 同一模型无标准的自我评价。

越靠后，越不能直接当作事实。

有效反馈应说明：

- 哪一条标准没有满足；
- 证据在哪里；
- 严重程度是什么；
- 允许修改哪些内容；
- 修改后怎样复验。

“答案不够好，请再想想”只是随机重采样，不是可靠的反思机制。

### 三种范式的核心对比

| 维度 | ReAct | Plan-and-Execute | Reflection |
| --- | --- | --- | --- |
| 核心问题 | 下一步做什么 | 整体怎样完成 | 当前结果哪里需要修 |
| 决策节奏 | 每次观察后 | 先规划，按触发器重规划 | 结果或步骤验证后 |
| 最适合 | 路径未知、反馈频繁 | 长任务、有依赖 | 结果可验证 |
| 状态重点 | 最新观察和下一动作 | 任务图、依赖和进度 | 候选结果、问题和版本 |
| 主要风险 | 漂移、重复、上下文膨胀 | 错误计划、计划过期 | 评价偏差、无效循环 |
| 主要停止条件 | 完成、预算、无进展 | 计划完成、失败或重规划 | 通过、停滞或达到轮次 |

### 实际项目通常怎样组合

竞品报告可以采用：

1. Planner 生成五个高层步骤；
2. “收集每款产品资料”由三个 Worker 并行执行；
3. 每个 Worker 内部用 ReAct 搜索和补充证据；
4. 汇总后用规则检查字段完整性；
5. 对缺失引用进行一次 Reflection 修正；
6. 达到验证标准后结束。

但这只是复杂任务的一种方案。如果单次模型加检索已经达到要求，增加 Planner、Worker 和 Evaluator 只会扩大故障面。

### 用五个维度做选型

#### 1. 路径长度

- 一两步：固定 Workflow 或受限 ReAct；
- 长依赖链：考虑 Plan-and-Execute。

#### 2. 反馈频率

- 几乎没有外部反馈：不必使用 ReAct；
- 每个工具结果都会改变下一步：适合 ReAct。

#### 3. 可验证性

- 有测试、规则或权威证据：Reflection 可能有效；
- 只有主观感觉：谨慎使用。

#### 4. 副作用风险

- 只读搜索：允许更多动态尝试；
- 付款、发信、删除：固定流程、审批和幂等优先。

#### 5. 预算与延迟

每多一个 Planner、Evaluator 或循环，就会增加模型调用和状态空间。必须比较：

- 端到端成功率；
- P95 延迟；
- 平均 Token 和工具调用；
- 费用；
- 人工接管率；
- 安全失败。

### 一个简单的选型顺序

> 路径固定吗？

- 是：先用 Workflow。
- 否：环境反馈频繁吗？

> 反馈会持续改变下一步吗？

- 是：考虑 ReAct。
- 否，但任务有长依赖：考虑 Plan-and-Execute。

> 结果能可靠验证且可以修改吗？

- 是：在交付前增加有限 Reflection。
- 否：不要为了“看起来更智能”增加反思循环。

## 工程实践与边界

- **计划结构化**：保存步骤、依赖、输入、预期输出、状态和验收条件，不只保存自然语言清单。
- **重规划有触发器**：仅在前提失效、工具不可用、验收失败或约束变化时触发。
- **反思引用证据**：反馈要指向测试错误、缺失字段或权威来源，限制修改范围和最大轮次。
- **不同角色分别评测**：Planner、Executor 和 Evaluator 即使使用不同模型，也要单独建立基线。
- **硬约束外置**：预算、权限、数据地域和审批规则不能被重规划或反思改掉。
- **写操作冻结范围**：执行前审批动作目标和关键参数，重规划超出范围时重新审批。
- **保留简单基线**：始终和单次调用或固定 Workflow 比较，防止把复杂度误当成能力。

## 常见误区

- **“ReAct 等于边展示思维链边调用工具”**：核心是观察驱动的动作循环，不需要公开私有思维链。
- **“Plan-and-Execute 就是先列一张清单”**：可靠计划还需要依赖、契约、验收和重规划条件。
- **“Reflection 就是让模型再想一次”**：没有失败证据和评价标准，只是重新采样。
- **“长任务都要使用 Planner”**：如果路径稳定，固定 Workflow 可能更可靠。
- **“三个范式叠加就会更强”**：每层循环都会增加调用、状态和故障处理。
- **“换不同模型评价就代表相互独立”**：模型仍可能依赖相同数据和偏差，外部验证更重要。

## 面试追问

> **面试官：** 什么时候不需要 Planner？
>
> **候选人：** 一两步即可完成、路径可以可靠枚举，或者规划成本高于执行成本时，用固定 Workflow 或受限 ReAct 更合适。

> **面试官：** 如何避免错误计划一路执行？
>
> **候选人：** 每步设置前置条件和验收条件，关键节点使用规则、测试或人工验证；第一次偏离时就暂停、重规划或转人工，而不是等最后失败。

> **面试官：** Reflection 使用同一个模型可以吗？
>
> **候选人：** 可以，但生成和评价容易共享偏差。应尽量引入隐藏测试、确定性规则、权威证据或人工标注，而不是把模型意见直接当作真值。

> **面试官：** 三种范式如何组合？
>
> **候选人：** 常见组合是 Planner 生成少量高层步骤，每一步由受约束的 ReAct 执行，关键产物再通过有明确标准的 Evaluator，只有失败项进入有限 Reflection。

> **面试官：** 如何证明组合方案值得？
>
> **候选人：** 在同一数据集上与最简单基线比较端到端成功率、关键错误、延迟、成本和安全指标，只有净收益稳定存在才保留复杂结构。
