## 本篇目标

学完这一篇，你应该能够：

- 区分“上下文窗口装得下”和“模型能稳定利用”；
- 解释长上下文与 RAG 各自解决的主要问题；
- 从语料规模、更新、ACL、任务类型、引用、延迟和成本做选型；
- 设计“先检索文档，再长上下文精读”等混合方案；
- 在真实数据上公平比较 full-context、RAG 和 hybrid；
- 回答“模型上下文越来越长，向量数据库是否没用了”这类追问。

## 60 秒回答

长上下文模型不会普遍替代 RAG，两者解决的问题不同。长上下文适合一次输入规模可控、需要通读全文结构或跨段关系、文档集合相对稳定的任务，它能减少切块造成的信息损失；但标称窗口长度只表示最多可容纳多少 token，不代表模型能在每个位置稳定找到并正确使用信息，输入越长通常还会增加预填充延迟、token 成本和权限管理难度。

RAG 适合大规模、持续变化的语料，能在模型调用前做 ACL、版本、时间和来源过滤，便于删除、更新和引用，也减少平均输入量；但它会引入解析、切块、漏召回、排序和索引运维问题。实际项目常用混合方案：先检索到相关文档或章节，再把完整候选文档放入长上下文模型精读，或者按问题复杂度在 full-context、RAG 和 hybrid 之间路由。

选型不能只看窗口大小。我会在真实查询和文档长度上比较三种路径，测答案正确性、忠实度、引用、多跳与全局问题、关键信息在开头中间结尾时的稳定性，以及 P95 延迟、输入 token、缓存命中、新鲜度和 ACL 风险。哪个方案满足业务质量、安全和预算，就用哪个，也可以按查询类型组合。

## 详细解析

### 先说人话：把所有书搬上桌，不等于能更快找到答案

你要回答一个问题，有两种办法：

1. 把图书馆所有书堆到桌上，从头阅读；
2. 先让目录系统找出最相关的几本，再认真读。

第一种避免目录漏书，但书太多时阅读慢，也容易忽略藏在中间的一句话。
第二种更快，但如果目录选错书，后面再聪明也找不到答案。

长上下文更接近第一种；RAG 更接近第二种。实际方案常常是先选几本，再把这几本完整阅读。

### 贯穿案例：30 页制度、5 万份文档和一个复杂问题

考虑三个任务：

#### 任务 A：阅读一份 30 页合同

用户问：

> 请总结合同中所有违约条件，并比较前后条款有没有冲突。

需要理解全文结构和跨章节关系。如果模型能安全容纳完整合同，长上下文可能很合适。

#### 任务 B：在 5 万份制度中查深圳住宿标准

用户只需要其中一份现行制度的一行表格。把全部资料放入上下文既不现实，也会增加噪声。RAG 更合适。

#### 任务 C：先找制度，再综合整份附件

用户问：

> 找到适用于上海研发中心的海外出差规则，并分析附件中所有例外情况。

可以先用 RAG 根据组织、地区和时间找到正确制度，再把该制度及附件完整放入长上下文模型。这就是混合方案。

### 先认识五个基础术语

| 术语 | 通俗解释 |
| --- | --- |
| Context Window | 一次模型请求能够容纳的最大 token 范围 |
| Long Context | 使用较大上下文窗口处理长文档或多文档 |
| Full-context | 把当前任务相关的全部材料直接放进模型 |
| RAG | 先检索少量相关证据，再交给模型生成 |
| Effective Context | 模型在特定任务上能可靠利用的实际范围 |

标称窗口是输入容量上限；有效上下文是任务表现。二者不能直接画等号。

### 为什么“装得下”不等于“用得好”

#### 1. 信息位置影响

Lost in the Middle 等研究观察到，模型对长上下文中不同位置的信息利用可能不同，关键信息在中间时表现可能下降。不同模型、任务和版本差异很大，因此要实测。

#### 2. 干扰和噪声增加

长上下文中可能同时有：

- 旧版本；
- 相似但不适用的规则；
- 冲突观点；
- 无关附件；
- 恶意指令；
- 大量重复。

模型要在更多干扰项中识别真正证据。

#### 3. 任务需要检索还是推理

模型可能能理解一段已知位置的文字，但未必能从海量输入中稳定找到它。
容量、检索能力和推理能力是不同维度。

#### 4. 输入成本和预填充延迟

模型先处理全部输入 token，这一阶段常称为 **prefill（预填充）**。输入更长通常意味着：

- 更多 token 费用；
- 更长的传输和预填充时间；
- 更大缓存和内存压力；
- 更低吞吐或更复杂的容量规划。

具体成本取决于模型服务、硬件、缓存和计费方式，不能只用理论长度判断。

### 长上下文的优势

#### 1. 保留全文结构

不需要把完整合同硬切成彼此独立的 chunk，标题、定义、前后引用和章节关系更容易保留。

#### 2. 支持全局阅读

适合：

- 全文总结；
- 比较多个章节；
- 发现前后矛盾；
- 代码仓库局部子集分析；
- 会议记录的时间线；
- 多份候选文档综合。

#### 3. 减少检索漏召回

如果所有相关材料已经确定并放入上下文，就不再依赖 chunk 级 top K 召回关键句。

#### 4. 架构可能更简单

对小规模、一次性分析，无需长期维护向量索引、增量同步和切块评测。

### 长上下文的限制

#### 1. 语料规模仍有上限

企业知识库可能有数百万页，不可能每次请求全部输入。

#### 2. 更新和删除

每次都拼接最新材料，需要可靠的数据选择与版本管理。Prompt 缓存若包含旧文档，还要处理失效。

#### 3. 细粒度权限

在发送给模型之前，仍要过滤所有无权文档。
不能把整家公司资料送入模型，再要求它只回答用户有权看的部分。

#### 4. 引用定位

即使模型读了全文，仍要知道结论来自哪一页哪一段，并验证引用支持陈述。

#### 5. 中间信息与噪声

上下文越长，关键信息可能更难被稳定利用，冲突和注入面也会增加。

#### 6. 成本方差

不同请求带入文档量差异很大，P95/P99 延迟和费用可能比平均值更重要。

### RAG 的优势

#### 1. 面向大规模语料

通过索引从大量文档中选出少量候选，不让每个问题读取全部知识库。

#### 2. 更新与删除更明确

文档变更后更新索引，查询可以使用最新发布版本；删除和权限收紧可以在数据与检索层阻断。

#### 3. ACL 和元数据过滤

在模型调用前按：

- 租户；
- 用户组；
- 时间；
- 地区；
- 文档类型；
- 正式状态；
- 数据分类

筛选候选。

#### 4. 来源和引用

检索结果天然携带文档 ID、页码、chunk 和分数，便于建立证据链。

#### 5. 平均输入可控

只放 top N 证据，token、延迟和成本更容易预算。

### RAG 的限制

#### 1. 解析和切块会丢结构

表格、图片、跨段定义和章节引用可能被破坏。

#### 2. 检索可能漏掉证据

查询表达、Embedding、关键词、K、过滤和重排任一环节都可能造成漏召回。

#### 3. 多跳与全局问题较难

只检索局部 chunk，可能看不到多个文档之间的整体关系。

#### 4. 系统组件更多

需要维护摄取、解析、索引、更新、评测、缓存和版本发布。

#### 5. 检索到证据也不保证正确生成

模型仍可能忽略、误读或错误组合证据。RAG 降低幻觉风险，但不能消除。

### 两者对比

| 维度 | 长上下文 / Full-context | RAG |
| --- | --- | --- |
| 语料规模 | 一次输入可控 | 可扩展到较大语料 |
| 全局结构 | 更容易保留 | 可能被切块破坏 |
| 局部事实定位 | 可能受长输入干扰 | 检索后更聚焦 |
| 更新 | 重新组织输入和缓存 | 更新索引与版本 |
| ACL | 输入前过滤全部材料 | 检索层细粒度过滤 |
| 引用 | 需额外定位与校验 | 候选通常自带来源 |
| 延迟 | 随输入长度增长 | 有检索开销，输入较短 |
| 成本 | 大量输入 token | 索引运维加较少输入 token |
| 漏召回 | 材料已全放入时较少 | 可能漏掉正确证据 |
| 噪声 | 长材料可能包含更多干扰 | top K 可聚焦，也可能选错 |

这张表没有“固定胜出”的一列。

### 三类常见方案

#### 方案一：Full-context

~~~text
授权材料集合
→ 结构化拼接
→ 长上下文模型
→ 答案和引用
~~~

适合：

- 一次性分析；
- 文档数量少；
- 全文结构重要；
- 文档集合已经确定；
- 输入、权限和成本可控。

#### 方案二：标准 RAG

~~~text
查询
→ ACL 与元数据过滤
→ 检索和重排
→ top K chunk
→ 模型生成与引用
~~~

适合：

- 语料大；
- 文档经常更新；
- 用户权限复杂；
- 多数问题是局部事实；
- 要求可追踪来源。

#### 方案三：Hybrid 混合

~~~text
查询
→ RAG 找到相关文档或章节
→ 读取候选的完整章节或全文
→ 长上下文模型精读
→ 答案与细粒度引用
~~~

它把 RAG 当“图书管理员”，长上下文模型当“精读者”。

### 混合方案的几种设计

#### 1. 文档级检索 + 全文阅读

先检索文档，不直接用碎片回答；选出少量文档后放入全文。适合合同、报告和制度。

#### 2. 章节级检索 + 扩展上下文

命中一个 chunk 后，读取它所属的完整章节、父标题、前后段或表格，减少切块断裂。

#### 3. 层级检索

~~~text
先找文档
→ 再找章节
→ 再找细粒度证据
→ 必要时加载完整候选
~~~

兼顾规模与结构。

#### 4. 路由

~~~text
单文档全文分析 → full-context
大知识库局部事实 → RAG
先选文档再综合 → hybrid
复杂跨源调查 → 可选 Agentic RAG
~~~

路由可以先用确定性规则，例如文档数量、总 token、任务类型和权限，再用轻量分类器补充。

#### 5. 摘要用于导航，原文用于证据

长文档可建立章节摘要帮助检索，但最终关键事实要回到原文引用，避免摘要遗漏或幻觉成为唯一证据。

### Prompt Cache 会改变结论吗

缓存重复长前缀可能降低一部分延迟或费用，但不会消除：

- 首次处理成本；
- 文档更新后的缓存失效；
- ACL 不同导致前缀不能安全共享；
- 长上下文中的信息利用问题；
- 引用与事实校验；
- 大量不同文档组合造成的低命中。

缓存是系统优化手段，不是 RAG 的全面替代。

### 如何做公平的选型实验

#### 准备三条路径

1. Full-context；
2. RAG；
3. RAG 检索文档 + 长上下文精读。

#### 控制变量

- 同一生成模型和版本；
- 同一问题集；
- 同一权威语料快照；
- 同一用户权限；
- 同一答案和引用评分；
- 合理调优每种方案，而不是故意给某条路径差参数。

#### 查询集要分层

| 切片 | 示例 |
| --- | --- |
| 单一局部事实 | 某城市住宿标准 |
| 全文总结 | 合同全部义务 |
| 跨段关系 | 定义与例外相隔很远 |
| 多文档 | 比较三份制度 |
| 多跳 | 身份、地区和时间共同决定 |
| 无答案 | 材料中没有对应规定 |
| 冲突版本 | 新旧制度并存 |
| 权限 | 有无权访问的材料混合 |

#### 专门测试位置

把关键证据分别放在：

- 开头；
- 中间；
- 结尾；
- 多个远距离位置。

这样才能测有效上下文，而不是只测模型能否接收长输入。

#### 质量指标

- 答案正确性；
- 对输入证据的忠实度；
- 关键条件完整性；
- 引用准确率和覆盖率；
- 无答案错误作答率；
- 多跳证据覆盖；
- 不同位置和长度切片的稳定性。

#### 系统指标

- 输入和输出 token；
- 首 token 延迟；
- 端到端 P50/P95/P99；
- 吞吐；
- 缓存命中与失效；
- 检索、重排和模型费用；
- 索引构建和更新成本；
- 单位成功任务成本。

### 一个实用的选择框架

依次问：

1. 授权材料能否在预算内放进一次请求？
2. 任务需要全文结构，还是只需局部事实？
3. 文档是否持续更新和删除？
4. 是否需要细粒度 ACL？
5. 是否要求精确引用和审计？
6. 能否接受检索漏召回？
7. 长输入的 P95 延迟和成本是否可接受？
8. 真实评测中哪条路径更稳定？

#### 倾向长上下文

- 文档少；
- 全文阅读重要；
- 一次性任务；
- 权限简单；
- 成本可控；
- 评测显示关键信息利用稳定。

#### 倾向 RAG

- 语料大；
- 更新频繁；
- ACL 复杂；
- 局部事实居多；
- 需要新鲜度、删除和来源追踪；
- 输入预算有限。

#### 倾向混合

- 语料大，但每次只需精读少量完整文档；
- chunk 检索能找到文档，却会丢全文结构；
- 任务既需要候选筛选，又需要跨章节综合。

## 工程实践与边界

### Full-context 也要有数据治理

不能直接遍历文件夹，把所有内容拼接。仍要：

- 选择权威版本；
- 执行 ACL；
- 去重和排除废止文档；
- 保留标题、章节和页码；
- 标记来源边界；
- 防间接提示注入；
- 控制总 token 和截断策略；
- 保存请求使用的数据版本。

### 不要静默截断

如果材料超过窗口，静默砍掉尾部可能刚好删除关键附件。应：

- 预先计算 token；
- 按文档和章节设优先级；
- 明确记录哪些内容未包含；
- 选择 RAG 或分批处理；
- 证据不足时提示用户，而不是假装读完。

### ACL 和缓存

无论 full-context 还是 RAG：

- 权限在模型调用前执行；
- 缓存键绑定租户、权限和数据版本；
- 权限收紧主动失效；
- 第三方模型只接收允许外发的数据；
- 预生成摘要也继承来源权限。

### 降级设计

| 故障 | 安全降级 |
| --- | --- |
| 检索服务不可用 | 对已授权小材料走 full-context，或返回不可用 |
| 长上下文模型超时 | 回退到已验证的 RAG 快速路径 |
| 输入超预算 | 层级检索或请求用户缩小范围 |
| 证据不足 | 明确拒答或请求补充 |
| 缓存版本不明 | 不使用缓存，重新授权和构建 |
| 权限服务不可用 | 默认拒绝受限数据 |

### 可观测性

记录：

- 使用 full-context、RAG 还是 hybrid；
- 候选文档和权限过滤；
- 输入文档数、token 和位置；
- 是否截断；
- 缓存命中和版本；
- 检索与模型各阶段延迟；
- 答案、引用和评测结果；
- 单位成功成本。

这样才能判断路由是否正确。

### 长上下文不是安全沙箱

输入越多，潜在恶意内容也越多。文档中的指令不能：

- 修改系统规则；
- 要求访问其他数据；
- 调用高权限工具；
- 外发内容；
- 关闭引用或审计。

工具仍需最小权限网关和审批。

### 技术边界

- 长上下文模型能力随模型和版本变化，需持续复测；
- RAG 检索指标高不保证最终答案好；
- full-context 不代表没有检索，选择哪些材料本身就是检索问题；
- 混合方案也会增加编排和评测复杂度；
- 公开 Benchmark 不能替代真实语料、长度、权限和硬件测试；
- 三种方案都不能消除幻觉。

## 常见误区

- **“窗口装得下，模型就能准确利用全部内容”**：容量与有效利用能力不同。
- **“长上下文完全不需要 RAG”**：大规模、更新、ACL、引用和成本仍需要检索与治理。
- **“RAG 总是更便宜”**：索引、重排、更新和运维也有成本，要按真实流量测量。
- **“长上下文没有漏召回”**：材料选择、截断和模型注意仍可能漏信息。
- **“把更多文档放进去，答案只会更好”**：噪声、冲突和注入面也会增加。
- **“Prompt Cache 让长上下文免费”**：命中、失效、权限和首次处理仍有成本。
- **“长上下文不需要引用”**：高风险结论仍要定位原文。
- **“只能在长上下文和 RAG 中二选一”**：检索后精读完整候选是常见且有效的候选方案。

## 面试追问

### 1. 上下文窗口和有效上下文有什么区别？

窗口是模型最多能接收的 token 数；有效上下文是模型在具体任务中能稳定找到、理解和使用的信息范围，需要通过位置、长度和任务实验测量。

### 2. 什么场景优先 full-context？

文档少、一次性分析、全文结构与跨段关系重要、权限和成本可控，而且目标模型在真实长度上评测稳定时。

### 3. 什么场景应保留 RAG？

语料大且持续更新、需要细粒度 ACL、删除、新鲜度、引用和低平均输入预算时。局部事实查询也通常更适合先检索聚焦。

### 4. 混合方案具体怎么做？

先在授权范围内检索到少量文档或章节，再读取这些候选的完整内容交给长上下文模型；最终答案仍引用原始页码和片段。

### 5. 如何避免 RAG 切块丢失全文结构？

使用父子 chunk、章节级检索、邻居扩展、文档级候选和层级检索。命中局部片段后，可以加载完整章节或全文精读。

### 6. 如何设计选型实验？

固定模型、语料、权限和问题集，对 full-context、RAG、hybrid 做配对比较；覆盖局部、全局、多跳、无答案和不同证据位置，并报告质量、引用、P95、token、更新和成本。

### 7. 长上下文如何做权限？

在拼接输入前逐份文档鉴权，只把当前用户可见内容发送给模型；缓存与摘要绑定权限域。不能输入全部内容后靠 Prompt 过滤。

### 8. 模型窗口继续变长，向量数据库会消失吗？

窗口增长会改变部分场景的架构，但大型、动态、多租户知识库仍需要搜索、过滤、版本、删除和引用。检索实现可能演化，检索这一能力不会仅因窗口变长而失去价值。
