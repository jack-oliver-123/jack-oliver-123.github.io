## 项目背景

要开发的是一个企业级 AI 实战项目：**基于 Spring AI Alibaba 的一键应用生成平台**。

这个项目也有一个更简洁的英文名：**Auto Code**。后续在课程和文章中，都会统一使用 Auto Code 来指代它。

## Auto Code 是什么

Auto Code 是一个面向 AI 原生开发的一键应用生成平台。

它参考了市面上一些成熟的低代码、无代码和 AI 应用生成产品，例如美团 NoCode、飞书妙搭、百度秒搭等。这类产品最核心的能力，就是让用户通过一句自然语言描述，快速生成一个完整可用的应用。

Auto Code 的核心目标也是如此：

> 用户只需要输入一句话，就能生成一个可以预览、编辑、发布的完整应用。

这意味着它不是单纯生成几段代码片段，而是要完成从需求理解、应用生成、在线编辑到最终部署的完整闭环。

## 核心能力概览

Auto Code 主要提供以下能力：

- 邮箱验证码登录；
- 通过自然语言生成应用需求文档；
- 根据需求文档一键生成应用；
- 支持单 HTML 页面、Vue3 前端应用、Vue3 + Spring Boot 前后端应用三种生成类型；
- 生成后实时预览应用效果；
- 支持 AI 可视化编辑；
- 支持类 VS Code 的高级代码编辑；
- 支持应用独立发布和访问；
- 提供“我的应用”和“案例广场”两个应用列表；
- 监控大模型调用情况和 Token 消耗；
- 内部集成 Agent 工作流、RAG、多层上下文记忆、MCP、Docker 等工程能力。

从用户视角看，它是一个可以快速生成应用的平台；从开发者视角看，它是一个覆盖 AI 应用开发、部署、监控和工程化落地的综合项目。

## 登录功能

使用 Auto Code 前，用户需要先登录。

项目提供的是**邮箱验证码登录**。用户输入邮箱后，系统会向该邮箱发送验证码，用户复制验证码并提交后即可完成登录。

之所以选择邮箱验证码登录，是因为很多个人开发者在申请短信服务时会遇到限制。相比短信验证码，邮箱验证码的接入成本更低，也更适合课程项目和个人练习环境。

这个功能是在原有脚手架短信验证码登录能力的基础上扩展出来的，既保留了验证码登录的交互方式，也解决了个人开发者接入短信服务不方便的问题。

## 一句话生成应用

登录后，用户可以在首页的对话框中输入想要生成的应用描述，例如：

```text
生成一个技术博客
```

Auto Code 会先根据这句话生成一份简短的需求文档。需求文档会包含应用名称、核心功能、页面结构等信息。

这里不会生成特别长的需求文档，因为和大模型交互会消耗 Token。需求文档越长，后续应用生成时需要处理的上下文就越多，成本也会更高。因此项目会在信息完整性和 Token 成本之间做一个平衡。

当用户确认需求文档符合预期后，可以点击“立即生成应用”。系统随后会调用大模型，根据需求文档生成完整应用代码，并在页面左侧实时展示预览效果。

## 三种应用类型

Auto Code 支持生成三种类型的应用。

### 单 HTML 页面

这是最轻量的一种应用类型，整个应用只包含一个 HTML 文件。

它适合生成展示页、简单工具页、个人介绍页、活动页等内容。因为文件结构简单，所以生成速度相对更快，也适合用来演示整个生成流程。

### Vue3 前端应用

第二种是基于 Vue3 的纯前端应用。

相比单 HTML 页面，Vue3 应用具备更好的组件化结构，适合承载更复杂的前端交互。它仍然不包含后端服务，但已经更接近真实前端项目的组织方式。

### Vue3 + Spring Boot 应用

第三种是 Vue3 + Spring Boot 的前后端应用。

这种类型会同时生成前端和后端代码：前端基于 Vue3，后端基于 Spring Boot。它的复杂度最高，生成内容最多，因此生成耗时也会更长。

课程中选择这三类应用，是为了兼顾学习价值和实现难度：既能覆盖从简单页面到前后端项目的完整形态，又不会让项目范围无限扩大。

## 实时预览

应用生成完成后，Auto Code 会直接在页面中展示生成结果。

这个预览能力非常重要。用户不需要先下载代码、手动运行项目，再回到浏览器查看效果，而是可以在生成后立即看到页面长什么样。

实时预览让整个应用生成流程更接近真实产品体验：输入需求、确认文档、生成应用、查看效果，都可以在同一个页面中完成。

## AI 可视化编辑

生成出来的应用不一定第一次就完全符合用户预期，因此 Auto Code 提供了编辑能力。

第一种编辑方式是 **AI 可视化编辑**。

用户点击编辑按钮后，可以直接选中页面上的某个元素，然后在对话框中描述想要修改的内容。例如：

```text
把标题改为黑色
```

系统会将用户选中的元素和修改要求提交给后端，再由后端调用大模型完成对应代码修改。修改完成后，预览区域会自动展示新的效果。

这种方式对不熟悉代码的用户比较友好，因为用户不需要知道具体的 HTML、CSS 或 JavaScript 写法，只需要描述自己想要的结果。

## 高级代码编辑

除了可视化编辑，Auto Code 还提供了面向开发者的高级编辑功能。

点击“高级编辑”后，系统会打开一个类似 VS Code 的在线编辑器。这里需要注意，它并不是 VS Code 官方编辑器，而是一个体验上接近 VS Code 的代码编辑界面。

通过高级编辑，用户可以直接查看和修改生成应用的源码。例如给标题追加文字、修改样式、调整结构等。

这种方式适合有开发基础的用户。相比自然语言编辑，直接改代码会更精确，也更符合开发者的使用习惯。

Auto Code 同时提供可视化编辑和高级代码编辑，是为了覆盖两类用户：

- 不熟悉代码的用户，可以通过自然语言描述修改应用；
- 熟悉代码的用户，可以直接进入编辑器修改源码。

## 应用发布

当应用修改到满意状态后，用户可以点击“发布”。

发布后，Auto Code 会为当前应用进行独立部署，并返回一个新的访问地址。用户可以把这个地址分享给其他人，对方在浏览器中打开后，就能访问这个生成好的应用。

这一步让 Auto Code 不只是一个“生成代码”的工具，而是一个真正完成交付闭环的平台。

完整流程可以总结为：

1. 输入一句话描述需求；
2. 生成并确认需求文档；
3. 生成应用；
4. 在线预览；
5. 可视化编辑或代码编辑；
6. 发布应用；
7. 通过独立地址访问。

## 我的应用与案例广场

首页中有两个重要列表：**我的应用**和**案例广场**。

“我的应用”展示当前用户创建过的所有应用。无论应用是否发布，只要是用户自己生成的，都可以在这里看到。

“案例广场”只展示已经发布的应用。一个应用被发布后，说明它已经具备对外访问的条件，因此会进入案例广场，供其他用户查看和参考。

这样的设计既方便用户管理自己的应用，也能让公开应用形成一个展示区。用户可以从其他人生成的应用中获得灵感，了解不同提示词和不同应用类型能生成怎样的效果。

## 大模型调用监控

Auto Code 是一个 AI 应用，大量核心功能都依赖大模型调用，例如：

- 生成需求文档；
- 生成应用代码；
- 执行 AI 可视化编辑；
- 辅助修改应用内容。

只要调用大模型，就会涉及 Token 消耗和调用成本。因此项目中引入了可观测性设计，用来监控大模型调用情况。

项目基于 Prometheus 和 Grafana 构建监控体系，可以观察单个用户的 Token 消耗、单次对话的上下文占比、单个应用的 Token 消耗等指标。

传统项目通常关注 CPU、内存、磁盘、接口耗时等指标。AI 应用则需要额外关注模型调用次数、Token 用量、上下文长度和调用成本。这也是 AI 应用工程化落地时必须考虑的问题。

## Agent 工作流

Auto Code 的应用生成不是简单调用一次大模型完成的。

在底层实现中，项目基于 Spring AI Alibaba Graph 构建 Agent 工作流，通过多个 Agent 协作完成应用生成过程。

这些 Agent 会分别承担不同职责，例如需求理解、代码生成、文件组织、结果检查等。多个 Agent 组成工作流后，可以让应用生成过程更加结构化，而不是把所有任务都塞进一次提示词里。

这种设计也是当前 AI 应用开发中的重要方向：让大模型不只是回答问题，而是作为一个工作流中的执行节点，和业务系统、工具调用、状态管理结合起来。

## 多层上下文记忆

在和大模型交互时，系统需要保存对话历史和上下文信息。

Auto Code 通过 MySQL 和 Redis 构建了多层上下文记忆管理能力，用来存储和组织用户与 AI 的对话历史。

这样做的好处是：

- 短期上下文可以快速读取；
- 长期历史可以持久化保存；
- 应用生成和编辑过程可以追踪；
- 后续对话能够基于之前的信息继续展开。

对于 AI 应用来说，上下文管理会直接影响用户体验。如果系统无法记住前面的需求和修改过程，后续编辑就很难保持一致。

## RAG 与 Milvus

为了提升 AI 回复和生成结果的准确性，Auto Code 还引入了 RAG 技术。

RAG 的核心思路是：在调用大模型前，先从知识库中检索相关资料，再把检索结果作为上下文提供给大模型。这样可以减少模型胡编乱造的概率，也能让回答更贴近项目知识和业务规则。

在向量数据库选型上，项目使用 Milvus。Milvus 是当前比较常见的向量数据库之一，在 AI 应用、知识库问答和语义检索场景中都有较多使用。

通过 RAG + Milvus，Auto Code 可以把外部知识、项目规则和业务说明引入到大模型调用过程中，让生成结果更稳定、更可控。

## MCP 与 Gitee 代码托管

应用生成后，代码需要被保存下来。

Flash Code 中封装了一个 Gitee MCP Server，用来完成与 Gitee 相关的代码提交和仓库操作。

之所以自己封装 MCP Server，是因为官方或平台侧能力在个人用户使用时可能存在限制。自己实现一套 MCP 服务，一方面可以绕开这些接入限制，另一方面也能帮助学习 MCP Server 的设计方式。

通过这个模块，项目可以把生成出来的应用代码提交到代码仓库中，让应用不只是临时存在于页面里，而是拥有可追踪、可管理的代码资产。

## Docker 预览与部署

Flash Code 的实时预览和独立发布都和 Docker 有关。

在预览阶段，项目会借助 Docker 为生成应用提供运行环境，让用户能够直接看到应用效果。

在发布阶段，项目同样基于 Docker 完成独立部署。发布后的应用会获得一个独立访问地址，用户可以将它分享给真实使用者。

Docker 的引入让应用生成不只停留在“代码生成”，而是进一步走向“可运行、可访问、可部署”。

## 为什么要设计这些功能

设计 Flash Code 时，不能只是为了堆技术，而是要回答三个问题。

### 第一，项目是否解决实际问题

Flash Code 解决的是用户想快速生成应用的问题。尤其是对于不会写代码的用户来说，一句话生成应用可以显著降低创建工具页、展示页和小型应用的门槛。

如果一个项目能在现实中找到参照产品，也能解释清楚它解决了什么问题，那么这个项目就更有说服力。

### 第二，技术是否贴近行业趋势

Flash Code 聚焦 AI 应用开发，因此会使用 Agent、RAG、MCP、向量数据库、可观测性、Docker 部署等当前比较热门的技术。

这些技术不是孤立存在的，而是围绕“一键应用生成”这个目标组合在一起。这样既能体现项目的技术含量，也能避免为了使用技术而使用技术。

### 第三，是否贴近企业实战场景

一个好的实战项目，最好能够和企业真实开发场景接轨。

Flash Code 中既有前后端开发，也有 AI 能力集成；既有用户登录、应用管理，也有部署、监控和代码托管。这些能力能够覆盖一个 AI 应用从开发到上线的完整链路，也更贴近企业对工程能力的要求。

## 项目学习价值

通过 Flash Code，可以系统学习以下内容：

- Spring AI Alibaba 在实际项目中的使用方式；
- AI 应用如何进行需求理解和代码生成；
- Agent 工作流如何组织复杂任务；
- RAG 如何提升生成准确性；
- Milvus 向量数据库如何接入项目；
- MCP Server 如何封装外部工具能力；
- Docker 如何支持预览和部署；
- Prometheus 与 Grafana 如何监控 AI 调用成本；
- 一个完整 AI 应用平台如何从功能设计走向工程落地。

这些内容结合起来，构成了 Flash Code 的核心价值。它不仅是一个能演示效果的应用生成平台，也是一个适合系统学习 AI 工程化开发的实战项目。

## 小结

Flash Code 是一个基于 Spring AI Alibaba 的一键应用生成平台。

从用户角度看，它可以通过一句话生成应用，并支持预览、编辑和发布；从开发者角度看，它融合了 Agent 工作流、RAG、MCP、Docker、可观测性和上下文记忆等多个 AI 工程化能力。

本节主要介绍了项目的整体功能和设计思路。理解这些内容之后，后续再深入学习具体模块实现时，就能更清楚每一项技术为什么要引入，以及它在整个项目链路中承担什么角色。
