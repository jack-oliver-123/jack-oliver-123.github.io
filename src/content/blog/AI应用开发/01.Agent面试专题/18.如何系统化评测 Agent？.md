## 本篇目标

学完后，你应该能解释为什么 Agent 不能只评最终答案，认识样本、数据集、Grader、Trace 和切片等术语，并设计一套从离线回归到线上监控的完整评测流程。

## 60 秒回答

Agent 评测不能只看最终回答。我会从四个维度评：第一，任务结果是否满足业务验收；第二，执行轨迹中的工具选择、参数、顺序和故障恢复是否合理；第三，延迟、Token、工具次数和费用是否在预算内；第四，是否出现越权、Prompt Injection、敏感数据泄露或绕过审批等安全问题。

方法上，先从真实业务、边界情况和历史失败中建立分层数据集。能用 Schema、测试、数据库对账和权限规则判断的，优先使用确定性 Grader；开放式质量再使用有明确评分量表、且经过人工校准的模型 Grader。评轨迹时只看可观察的动作、状态和证据，不要求模型输出私有思维链。

离线评测用于开发和发布门禁，影子流量与小比例上线用于发现真实分布问题，生产 Trace 用于把失败定位到模型、上下文、工具或编排。不能只看平均分，高风险切片必须单独设门槛。

## 详细解析

### 先说人话：评 Agent 像考“会办事的人”，不只是考一道问答题

普通问答模型可能只需要检查最后一句答案是否正确。Agent 会查资料、调用工具、修改系统，还可能花钱，因此至少要问：

- 事情最后办成了吗？
- 中间走的路正确吗？
- 花了多少时间和费用？
- 有没有做不该做的事？

例如两个 Agent 都回复“退款已完成”：

- Agent A 查询订单、核对政策、请求审批、退款一次并验证状态。
- Agent B 试了三个订单号、绕过审批、重复调用退款接口，最后碰巧成功。

只看最终文字，两者都“答对了”；从系统角度看，Agent B 明显不能上线。

### 先认识五个基础术语

| 术语 | 通俗解释 | 退款案例 |
| --- | --- | --- |
| 样本（Case） | 一道测试题 | 一个具体退款请求 |
| 数据集（Dataset） | 一组有代表性的测试题 | 正常、超期、重复、越权等请求集合 |
| Grader | 判卷规则或判卷器 | 检查金额、状态、审批和工具轨迹 |
| Trace | Agent 执行过程的可观察记录 | 模型、检索、工具、审批和错误记录 |
| 切片（Slice） | 按特征分出的样本子集 | 高金额退款、英文请求、工具超时 |

评测不是“跑个分数”就结束，而是先定义什么叫成功，再收集代表性样本，用可靠的判卷方式检查，最后把失败变成可修复的问题。

### 贯穿案例：评测一个客服退款 Agent

假设业务验收标准是：

1. 必须找到属于当前用户的订单；
2. 必须使用当前版本退款政策；
3. 金额超过 500 元必须人工审批；
4. 同一个退款只能执行一次；
5. 最终回复中的金额和真实退款记录一致。

我们可以构造这些样本：

| 样本 | 目的 |
| --- | --- |
| 普通且符合政策的退款 | 验证主流程 |
| 已超过退款期 | 验证拒绝与解释 |
| 金额 501 元 | 验证审批边界 |
| 同一请求重复提交 | 验证幂等 |
| 退款接口超时但实际成功 | 验证状态对账，防止重复退款 |
| 用户尝试查询别人的订单 | 验证授权 |
| 政策文档中藏有恶意指令 | 验证间接 Prompt Injection 防护 |

这比随机找几十条“正常问题”更能发现真实风险。

### 第一层：任务结果评测

结果评测要从业务目标出发，不要只问“语言是否流畅”。

常见结果指标包括：

- 任务是否真正完成；
- 关键字段是否正确；
- 结果是否有证据支持；
- 外部系统状态是否符合预期；
- 用户要求和业务约束是否同时满足。

代码 Agent 可以看测试是否通过；检索 Agent 可以看引用是否支持结论；客服 Agent 可以查工单状态和必要字段是否完整。

### 第二层：执行轨迹评测

**轨迹（Trajectory）**是 Agent 可观察的动作序列，例如：

```text
读取用户身份
→ 查询订单
→ 查询退款政策
→ 请求人工审批
→ 调用退款工具
→ 查询退款状态
→ 回复用户
```

轨迹评测不是要求所有正确任务都走同一条固定路径。更合理的做法是定义：

- **必须发生的属性**：例如执行退款前必须有有效审批；
- **禁止发生的属性**：例如不能访问其他租户订单；
- **允许变化的部分**：例如先查订单还是先向用户补充询问。

这样既保留 Agent 的灵活性，又能守住业务边界。

轨迹评测只需要模型调用、工具调用、参数摘要、状态转换和证据引用，不需要获取模型的私有思维链。

### 第三层：效率评测

成功但成本失控的 Agent 也不能稳定上线。应记录：

| 指标 | 为什么重要 |
| --- | --- |
| 端到端延迟 | 用户等了多久 |
| 模型调用次数 | 是否存在无效反复思考 |
| 工具调用次数 | 是否重复查询或重复写入 |
| 输入与输出 Token | 估算模型成本和上下文膨胀 |
| 工具费用 | 搜索、浏览器、第三方 API 可能单独收费 |
| 人工介入率 | 自动化是否真正减少工作量 |
| P50、P95、P99 | 平均值之外，慢请求有多严重 |

不要为了省一次模型调用而牺牲安全校验，也不要只追求成功率而忽略每个成功任务的成本。

### 第四层：安全与鲁棒性评测

安全集至少应覆盖：

- 直接和间接 Prompt Injection；
- 越权访问与跨租户访问；
- 敏感信息被写入日志、记忆或外发；
- 未审批的付款、发信、删除等副作用；
- 工具参数注入、路径穿越和不安全对象引用；
- 超时、限流、错误数据和部分成功；
- 无限循环和资源耗尽；
- 审批过期或关键参数变化。

这里的“鲁棒性”是指环境出错或输入变化时，系统仍能按预期降级、停止或转人工，而不是勉强给出一个看似完整的答案。

### 数据集应该怎么建立

#### 1. 从业务分布开始

收集真实任务类型、语言、用户群、工具和风险等级。线上数据必须脱敏，并遵守用途和保留限制。

#### 2. 加入边界与历史失败

每次生产事故、用户投诉、工具故障和错误轨迹，都应在修复后转成回归样本，防止同类问题再次出现。

#### 3. 加入对抗样本

主动构造越权、注入、模糊指令、冲突政策和恶意工具输出，而不是等攻击发生。

#### 4. 做分层和切片

按任务类型、难度、语言、工具、风险和用户群分别统计。总体 95% 的成功率，可能掩盖“高金额退款只有 60%”的严重问题。

### Grader 应该怎么选

推荐优先级如下：

```text
确定性规则或可执行测试
       ↓
权威数据和系统状态对账
       ↓
人工评分
       ↓
经过人工校准的模型 Grader
```

#### 确定性 Grader

适合检查 JSON Schema、字段值、测试结果、权限、引用链接、工具调用次数和审批顺序。优点是稳定、可重复、便宜。

#### 人工 Grader

适合复杂语义、专业判断和建立“标准答案”。缺点是成本高，而且不同标注人也会不一致，因此需要评分说明和一致性检查。

#### 模型 Grader

适合大规模评价开放式回答，但必须：

- 给出明确评分量表和示例；
- 隐藏与任务无关的信息，减少偏差；
- 定期与人工标注比较；
- 不把单一模型分数作为高风险发布依据；
- 版本化评分模型和 Prompt。

### 从离线到线上的评测闭环

```text
开发阶段：小型快速评测，定位问题
   ↓
合并前：完整离线回归，阻止明显退化
   ↓
上线前：影子流量，不执行真实副作用
   ↓
发布时：小比例流量，设置回滚门槛
   ↓
生产中：监控指标和 Trace，抽样人工复核
   ↓
新失败加入回归数据集
```

**影子流量**是把真实请求复制给新版本观察结果，但不让新版本影响真实用户或执行真实写操作。

## 工程实践与边界

### 让评测结果可复现

每个样本应保存：

- 输入和必要的环境夹具；
- 期望结果或必须满足的属性；
- 允许和禁止的轨迹条件；
- Grader 及其版本；
- 模型、Prompt、工具 Schema、检索索引版本；
- 随机参数和运行时间；
- 失败证据与 Trace 引用。

第三方 API 会变化，必要时使用模拟器或固定快照，避免今天与明天的分数无法比较。

### 安全地测试工具故障

在隔离环境主动注入：

| 故障 | 要检查的行为 |
| --- | --- |
| 超时 | 是否先查询状态再决定重试 |
| 限流 | 是否退避、降级或排队 |
| 返回错误数据 | 是否校验 Schema 和业务约束 |
| 部分成功 | 是否对账、补记或补偿 |
| 服务不可用 | 是否停止或转人工 |

付款、邮件、删除等工具必须使用沙箱或模拟器，不能让离线评测产生真实副作用。

### 发布门禁怎么设

不能只规定“总分超过 90”。更合理的是：

- 主任务成功率不得低于基线；
- 高风险安全样本必须达到业务规定的逐项门槛，越权和真实副作用等场景可设为零容忍；
- P95 延迟和单任务成本不超过上限；
- 关键用户与语言切片不得明显退化；
- 发现越权、真实副作用或数据泄露时直接阻止发布。

### 生产日志的隐私边界

Trace 进入评测系统前要脱敏。包含个人可识别信息（PII）的记录应限制访问和保留时间，密钥、密码和完整支付信息不应进入评测数据。评测方便不能成为扩大数据收集的理由。

## 常见误区

- **“最终答案正确就算通过”**：轨迹可能越权、重复写入或成本失控。
- **“模型裁判就是客观评分”**：模型 Grader 也有偏差，需要人工校准和确定性证据。
- **“公开 Benchmark 分高就能上线”**：公开任务不包含你的工具、权限、数据和业务失败模式。
- **“平均成功率提高就可以发布”**：高风险小切片的退化可能被大量简单样本掩盖。
- **“评测集越大越好”**：代表性、可解释性和失败覆盖比单纯数量更重要。
- **“线上监控可以替代离线评测”**：高风险问题不能等到影响真实用户后才发现。

## 面试追问

> **面试官：为什么 Agent 不能只评最终答案？**
>
> **候选人：** 因为同一正确答案可能来自越权、重复调用或偶然成功的危险轨迹。还需要评工具行为、成本、延迟、安全和故障恢复。

> **面试官：任务有多种正确路径，怎么评轨迹？**
>
> **候选人：** 不匹配唯一动作序列，而是定义必须满足和禁止出现的属性。例如必须有有效审批、不得调用越权工具、最终证据要一致。

> **面试官：模型 Grader 怎么保证可靠？**
>
> **候选人：** 使用清晰评分量表和示例，定期与人工标注校准，监控不同切片的一致性，并让测试、规则和权威数据优先于模型分数。

> **面试官：如何评估工具故障恢复？**
>
> **候选人：** 在沙箱注入超时、限流、部分成功和错误数据，检查重试、幂等、对账、降级和人工接管是否符合策略。

> **面试官：线上指标下降怎么定位？**
>
> **候选人：** 先按模型、Prompt、工具、索引版本和任务切片分解，再用 Trace 找到第一个偏离节点，判断问题在上下文、决策、工具还是编排，最后把新失败加入回归集。
