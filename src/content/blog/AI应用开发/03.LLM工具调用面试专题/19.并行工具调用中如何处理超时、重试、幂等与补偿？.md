## 本篇目标

学完这一篇，你应该能够：

- 判断多个工具调用是否真的可以并行；
- 区分任务截止时间、单工具超时、取消与远端执行状态；
- 为只读、幂等写和非幂等写设计不同重试策略；
- 解释为什么模型 call ID 不能直接替代业务幂等键；
- 处理并行调用的部分成功、乱序结果和状态未知；
- 用 Saga、补偿动作、状态机和人工队列治理跨系统副作用。

## 60 秒回答

并行前先把工具调用建成依赖图。无数据依赖、无共享资源冲突的只读调用可以并行；后一步依赖前一步结果、修改同一资源或要求顺序的写操作必须串行或使用并发控制。每次调用用 call ID 关联，结果乱序返回也不能混淆；整次任务设置 deadline，再为连接、读取和单工具分配更短预算。

重试按操作语义分类。只读请求在瞬时网络错误、限流或服务不可用时可以有界重试；写操作只有执行端支持业务幂等键或能先查询提交状态时才安全重放。超时只是调用方停止等待，不代表远端没有提交。参数错误、权限拒绝和业务冲突通常不自动重试，并使用指数退避、抖动、Retry-After、并发上限和熔断避免重试风暴。

跨系统流程使用持久化状态机或 Saga。每个正向步骤和补偿步骤都幂等，记录意图、尝试、结果和状态。补偿是新的业务动作，例如释放库存或退款，不是数据库回滚，也可能失败，所以要支持重试、对账和人工介入。部分失败时按调用逐项返回状态，只重试安全的失败项，不能盲目重放整批写操作。

## 详细解析

### 先说人话：多人同时办事，先看是否互相依赖

准备一次旅行：

- 查天气；
- 查汇率；
- 订酒店；
- 支付酒店；
- 发送行程。

查天气和查汇率可以同时进行。
支付必须等酒店订单创建；发送行程应等预订状态确定。把所有步骤同时启动，可能先付款却没有订单。

模型一次返回多个工具调用，只表示“它提出了多个调用意图”，不代表宿主应无条件并行。

### 贯穿案例：退款、库存和通知

客服 Agent 处理“退货退款”：

1. 查询订单；
2. 查询退款政策；
3. 查询仓库是否已收货；
4. 创建退款；
5. 恢复库存；
6. 发送通知。

前三个读操作可能并行。
后面存在依赖：

~~~text
查询订单 ─┐
查询政策 ─┼→ 判断可退款 → 创建退款 → 恢复库存 → 发送通知
查询收货 ─┘
~~~

如果创建退款成功、恢复库存失败，就出现部分成功，需要后续修复，而不是把前面所有步骤重跑一遍。

### 先认识九个术语

| 术语 | 通俗解释 |
| --- | --- |
| DAG | 有向无环图，用来表达步骤依赖 |
| Deadline | 整个任务最晚完成时间 |
| Timeout | 某个等待阶段最多等多久 |
| Retry | 失败后再次尝试 |
| Backoff | 每次重试前逐渐增加等待 |
| Jitter | 在等待时间加入随机抖动，避免同时重试 |
| Idempotency | 重复请求不会重复产生业务效果 |
| Saga | 多个本地事务组成的长业务流程 |
| Compensation | 用新业务动作抵消已完成步骤的影响 |

### 第一步：判断能否并行

可以用四个问题判断：

1. B 是否需要 A 的输出？
2. A 和 B 是否修改同一资源？
3. 业务是否要求固定顺序？
4. 同时执行是否会违反不变量？

### 并行决策表

| A | B | 是否可并行 | 原因 |
| --- | --- | --- | --- |
| 查天气 | 查汇率 | 通常可以 | 独立只读 |
| 查订单 | 查退款政策 | 通常可以 | 独立只读 |
| 创建订单 | 支付订单 | 不可以 | 支付依赖订单 ID |
| 两次扣减同一库存 | 谨慎或不可以 | 共享资源冲突 |
| 给两个不同用户发独立通知 | 可在授权后并行 | 资源独立，但有副作用 |
| 创建退款 | 查询同一退款状态 | 应按顺序 | 先有退款记录 |

### 工具元数据帮助调度

宿主可以维护：

~~~yaml
tool: create_refund
operation: write
resource_key: order_id
idempotency: required
retryable_errors:
  - RATE_LIMITED
  - UNAVAILABLE
default_timeout_ms: 5000
requires_confirmation: true
~~~

模型不负责声明操作安全性。元数据由工具所有者和平台维护。

### 第二步：构建依赖图，而不是相信数组顺序

模型可能一次返回：

~~~json
[
  {
    "call_id": "call_a",
    "name": "get_order",
    "arguments": {"order_id": "O-42"}
  },
  {
    "call_id": "call_b",
    "name": "get_refund_policy",
    "arguments": {"region": "CN"}
  }
]
~~~

宿主结合工具元数据和参数计算：

- 两者都是只读；
- 资源独立；
- 可以放入同一并发批次；
- 每个结果仍按 call_id 关联。

不要依赖返回数组第一个最先完成。网络调用会乱序。

### 第三步：设计截止时间和分层超时

#### Deadline

用户愿意等待的端到端上限，例如：

~~~text
任务 deadline：10 秒
  ├─ 规划：1 秒
  ├─ 读取工具并行批次：3 秒
  ├─ 写工具：3 秒
  ├─ 最终生成：2 秒
  └─ 预留：1 秒
~~~

数字只是示意。真正预算来自业务 SLO 和压测。

#### 分层超时

- DNS / 建连；
- TLS；
- 首字节；
- 读取响应；
- 工具内部执行；
- 总等待；
- 队列等待。

只设置一个很大的 HTTP timeout，无法快速区分连接故障、服务慢还是队列积压。

#### 传播剩余预算

调用下游时传递剩余 deadline：

~~~python
remaining = task_deadline - now()
tool_timeout = min(tool_default_timeout, remaining - reserve_for_final_answer)
~~~

如果只剩 100 毫秒，却仍启动一个通常需要 3 秒的工具，最终只会浪费资源。

### 超时、取消和失败不是同义词

调用方超时后：

- 本地停止等待；
- 可以发送取消信号；
- 远端可能已经提交；
- 取消可能来不及；
- 响应可能只是丢失；
- 外部副作用可能不可撤销。

所以：

~~~text
timeout ≠ not executed
cancel requested ≠ cancelled
connection closed ≠ transaction rolled back
~~~

写操作超时后，优先查询业务状态，而不是马上重试。

### 第四步：把错误分成可重试与不可重试

| 错误 | 自动重试倾向 | 原因 |
| --- | --- | --- |
| 短暂网络中断 | 可能 | 瞬时错误 |
| 服务不可用 | 可能 | 等待恢复，但需熔断 |
| 429 限流 | 可能 | 遵守 Retry-After |
| 读取请求超时 | 视语义 | 只读通常更安全 |
| 参数格式错误 | 不重试 | 同样参数仍会失败 |
| 认证失败 | 不重试 | 需要重新认证 |
| 权限拒绝 | 不重试 | 不能靠重试绕过 |
| 业务冲突 | 通常不盲重试 | 先刷新状态 |
| 写入超时且状态未知 | 先查状态 | 可能已提交 |
| 内容安全拒绝 | 不以同参数重试 | 不是瞬时故障 |

### 读操作怎么重试

只读工具例如 get_order：

1. 参数与授权已通过；
2. 错误被识别为瞬时；
3. deadline 仍有余量；
4. 没有超过次数；
5. 使用指数退避和抖动；
6. 遵守服务端 Retry-After；
7. 必要时在断路器打开后快速失败。

只读也不等于没有风险：

- 可能泄露敏感数据；
- 大查询可能消耗资源；
- 结果会随时间变化；
- 下游可能错误地把 GET 实现成副作用。

工具契约与实际实现要一致。

### 写操作怎么重试

分三类。

#### 天然幂等写

例如“把资源状态设为 DISABLED”，重复设置得到相同状态。
仍要考虑版本冲突和审计次数。

#### 带业务幂等键的写

例如创建退款：

~~~json
{
  "idempotency_key": "tenant-a:refund:order-O42:user-request-7",
  "order_id": "O-42",
  "amount_minor": 29900
}
~~~

服务端持久化：

~~~text
幂等键
→ 请求参数摘要
→ 执行状态
→ 最终结果或错误
~~~

同一键重复到达时：

- 参数一致且已成功：返回原结果；
- 参数一致且处理中：返回处理中或等待；
- 参数不一致：拒绝为幂等冲突；
- 上次失败：按错误语义决定是否允许重试。

#### 非幂等且不可查询状态的写

例如某些旧系统“发送一次不可追踪指令”。自动重试风险高，应：

- 避免自动重试；
- 尽量改造下游；
- 使用中间状态和人工确认；
- 保留审计；
- 失败后人工对账。

### 为什么 call ID 不是业务幂等键

模型调用 ID 用于关联：

~~~text
模型提出的某次调用 ↔ 对应工具结果
~~~

但用户重试、模型重新规划或供应商切换后，可能生成新的 call ID，却表达同一个退款意图。

业务幂等键应绑定稳定业务语义，例如：

~~~text
租户 + 业务对象 + 动作 + 用户请求实例
~~~

不能只用参数哈希，因为用户可能合法地连续执行两个参数相同的动作。需要一个由宿主生成并持久化的请求实例标识。

### 幂等实现的并发细节

两个相同幂等键同时到达时，要防止同时执行。可使用：

- 数据库唯一约束；
- 原子 compare-and-set；
- 分布式锁；
- 状态行事务；
- 执行租约。

示意状态：

~~~text
RECEIVED
→ RUNNING
→ SUCCEEDED
→ 返回已存结果

RUNNING 超过租约
→ 查询下游状态
→ 继续、恢复或人工处理
~~~

幂等记录保留多久取决于业务重试窗口和合规，不能请求一结束就删除。

### 指数退避与抖动

简单示意：

~~~text
base = 200ms
attempt 1：随机等待 0 ~ 200ms
attempt 2：随机等待 0 ~ 400ms
attempt 3：随机等待 0 ~ 800ms
并设置最大等待与最大次数
~~~

随机抖动避免大量请求在同一时刻重新打到恢复中的服务。

### 只在一层主导重试

危险组合：

~~~text
Agent 重试 3 次
× 网关重试 3 次
× SDK 重试 3 次
= 最多 27 次下游尝试
~~~

应该：

- 明确哪层拥有重试策略；
- 其他层关闭或限制；
- 传递 attempt 和 deadline；
- 记录总尝试次数；
- 对下游实施并发与预算上限。

### 第五步：处理并行调用的部分成功

假设：

| call_id | 工具 | 结果 |
| --- | --- | --- |
| call_a | get_order | 成功 |
| call_b | get_policy | 超时 |
| call_c | get_return_status | 权限拒绝 |

返回给模型或工作流：

~~~json
[
  {
    "call_id": "call_a",
    "status": "succeeded",
    "data": {"order_status": "DELIVERED"}
  },
  {
    "call_id": "call_b",
    "status": "failed",
    "error": {"code": "TIMEOUT", "retryable": true}
  },
  {
    "call_id": "call_c",
    "status": "failed",
    "error": {"code": "PERMISSION_DENIED", "retryable": false}
  }
]
~~~

不要把整批压成“工具失败”。宿主只对 call_b 做有界重试；call_c 不应让模型换参数绕过权限。

### 是否允许返回部分答案

取决于业务：

- 天气查询：可以返回已成功城市，并说明缺失；
- 财务汇总：缺一项可能不能发布；
- 批量通知：要逐个记录成功和失败；
- 退款流程：关键前置查询失败时不能执行写操作。

这应由确定性业务规则决定，不由模型临时猜。

### 第六步：并发写的资源冲突

两个调用同时修改订单 O-42：

- create_refund；
- close_order。

即使调用数组彼此独立，业务资源相同。可以：

- 按 resource_key 串行；
- 使用乐观锁和版本号；
- 数据库条件更新；
- 明确冲突错误；
- 重新读取状态后规划。

示例：

~~~sql
UPDATE orders
SET status = 'REFUNDING', version = version + 1
WHERE id = :id
  AND status = 'DELIVERED'
  AND version = :expected_version;
~~~

更新行数为 0 表示状态已变化，不应盲重试原写。

### 第七步：Saga 与补偿

跨系统流程：

~~~text
创建退款
→ 恢复库存
→ 创建退货单
→ 发送通知
~~~

很难放进一个数据库事务。Saga 把流程拆成多个本地事务，并为可补偿步骤定义反向业务动作。

| 正向动作 | 补偿动作 |
| --- | --- |
| 预留库存 | 释放预留 |
| 创建草稿订单 | 取消草稿 |
| 扣减积分 | 返还积分 |
| 收款 | 退款 |
| 创建配送任务 | 取消未开始配送 |

### 补偿不是 rollback

数据库 rollback 可以让未提交事务像没发生。补偿是一个新动作：

- 退款会留下资金流水；
- 发出的邮件无法收回；
- 用户可能已经看到通知；
- 汇率和价格已经变化；
- 取消配送可能失败；
- 外部系统状态可能继续演化。

所以补偿目标通常是恢复业务可接受状态，而不是恢复物理世界的每个细节。

### Saga 的两种组织方式

#### Orchestration

中央协调器决定下一步：

~~~text
Saga Orchestrator
→ Step A
→ Step B
→ Step C
→ 失败则按逆序触发补偿
~~~

优点是流程清楚、容易审计；缺点是协调器复杂。

#### Choreography

各服务通过事件响应：

~~~text
RefundCreated
→ Inventory 服务恢复库存
→ Shipping 服务创建退货任务
~~~

耦合较松，但全局流程、重复事件和故障定位更难。

Agent 工具流程常更适合有显式协调状态机，因为模型决策本身已经有不确定性。

### Saga 状态机示例

~~~text
PENDING
→ REFUND_SUCCEEDED
→ INVENTORY_SUCCEEDED
→ NOTIFICATION_SUCCEEDED
→ COMPLETED

若 INVENTORY_FAILED：
→ NEEDS_RETRY
→ 超过门槛
→ MANUAL_REVIEW
~~~

不是每个失败都要撤销退款。如果退款是业务主目标，库存恢复可以异步修复。补偿策略要由业务定义。

### 补偿也需要幂等

释放库存可能被重复触发。每个补偿动作：

- 独立幂等键；
- 明确前置状态；
- 有界重试；
- 结构化结果；
- 审计；
- 失败进入人工队列。

### 第八步：熔断、舱壁和背压

- **熔断**：持续失败时快速阻断；
- **舱壁**：为租户、工具或下游分隔并发池；
- **背压**：下游处理不过来时让上游减速；
- **队列**：可延迟任务异步处理；
- **负载削减**：在过载时拒绝低优先级请求。

不能靠不断重试解决容量不足。

## 工程实践与边界

### 可靠执行器伪代码

~~~python
async def execute_call(call, task_deadline, trusted_context):
    spec = registry.get(call.name)
    validate_schema(spec.input_schema, call.arguments)
    authorize(trusted_context.subject, spec, call.arguments)

    timeout = compute_timeout(task_deadline, spec.default_timeout)

    if spec.operation == "read":
        return await retry_with_backoff(
            lambda: spec.execute(call.arguments, trusted_context),
            retryable=spec.retryable_errors,
            max_attempts=spec.max_attempts,
            timeout=timeout,
        )

    idempotency_key = trusted_context.business_request_id
    return await execute_idempotently(
        key=idempotency_key,
        normalized_arguments=call.arguments,
        operation=spec.execute,
        timeout=timeout,
    )
~~~

伪代码省略了状态查询、租约和审计，生产实现要补齐。

### 并行批次伪代码

~~~python
graph = build_dependency_graph(validated_calls, tool_registry)

for ready_batch in graph.ready_batches():
    safe_calls = enforce_resource_conflicts(ready_batch)
    results = await gather_with_concurrency_limit(
        [execute_call(call) for call in safe_calls]
    )
    graph.record(results)

    if critical_dependency_failed(results):
        stop_downstream_writes()
        break
~~~

### 监控指标

| 类别 | 指标 |
| --- | --- |
| 并行 | 批次大小、并发度、排队时间 |
| 延迟 | 单工具和端到端 P50/P95/P99 |
| 超时 | 按阶段和工具的超时率 |
| 重试 | 尝试次数、成功恢复率、放大倍数 |
| 幂等 | 重复键命中、参数冲突、重复副作用 |
| 部分成功 | 批次部分失败率、关键步骤失败 |
| Saga | 卡住状态、补偿成功率、人工介入 |
| 资源 | 熔断、队列积压、舱壁耗尽 |

最重要的安全指标之一是真实重复写入数，而不是“幂等代码存在”。

### Trace 至少记录

- 原始用户意图；
- 模型 call ID；
- 业务请求 ID 和幂等键摘要；
- 工具与版本；
- 规范化参数摘要；
- 依赖与批次；
- 每次 attempt；
- deadline 和 timeout；
- 授权与确认；
- 下游 request ID；
- 结果、状态查询和补偿；
- 最终业务状态。

敏感字段脱敏，幂等键也可能包含业务标识，不应原样无范围传播。

### 安全边界

- 模型不能决定某错误可安全重试；
- 工具错误文本是不可信输入；
- 权限拒绝不能通过换参数反复试探；
- 重试时复用已校验的规范化参数；
- 写操作确认绑定参数，参数变化后确认失效；
- 补偿工具权限不应自动授予所有 Agent；
- 人工队列也要租户隔离和审计。

### 哪些副作用难以补偿

- 已发送的外部邮件；
- 公布的消息；
- 物理设备动作；
- 已下载的数据；
- 第三方不可撤销交易；
- 已触发的人工流程。

对这类工具，更重要的是执行前预览、审批、最小权限和限额，而不是事后补偿。

## 常见误区

- **“模型返回多个调用就应全部并行”**：依赖、共享资源和副作用决定调度。
- **“超时等于远端没有执行”**：请求可能已提交，只是响应丢失。
- **“只读操作可以无限重试”**：仍会放大负载、成本和数据泄露面。
- **“call ID 就是幂等键”**：重新规划后 call ID 会变，业务意图可能相同。
- **“相同参数哈希就是幂等键”**：用户可能合法重复做两个同参数动作。
- **“写操作都不能重试”**：有执行端业务幂等和状态查询时可以安全设计。
- **“补偿等于事务回滚”**：补偿是可能失败的新业务动作。
- **“部分失败就重跑整批”**：会重复已成功写操作，应逐项处理。

## 面试追问

### 1. 哪些工具调用可以并行？

无数据依赖、无共享资源冲突、业务不要求顺序的调用。只读更容易并行，但仍要考虑权限、容量和数据一致性。

### 2. 读操作与写操作的重试差异是什么？

只读在瞬时错误下通常可有界重试；写操作要先确认天然幂等、支持业务幂等键，或能查询上次提交状态，否则可能重复副作用。

### 3. 写操作超时后第一步做什么？

保留原幂等键，查询业务状态或幂等记录，判断成功、失败还是仍处理中。不要立即用新键重发。

### 4. 怎样设计幂等键？

由可信宿主生成，绑定租户、业务对象、动作和用户请求实例；执行端持久化参数摘要、状态和结果，并拒绝同键不同参数。

### 5. 怎样避免重试风暴？

一层主导重试，限制次数，指数退避加抖动，遵守 Retry-After，传播 deadline，配合并发上限、舱壁、熔断和背压。

### 6. Saga 和数据库事务有什么区别？

数据库事务在一个事务边界内原子提交或回滚；Saga 由多个本地事务组成，失败后使用补偿动作恢复业务可接受状态。

### 7. 补偿失败怎么办？

补偿本身幂等并可有界重试；持续失败时进入持久状态和人工处理队列，通过对账确认最终业务状态，不能假装原流程已回滚。

### 8. 并行批次部分失败，怎么给模型结果？

按 call ID 返回每项 succeeded、failed 或 unknown，并给安全的结构化错误。只对满足策略的失败项重试，关键依赖失败时阻止后续写步骤。
