## 本篇目标

学完这一篇，你应该能够：

1. 解释“多路召回”是多个互补检索通道共同找候选，不等于简单把 Top-K 调大。
2. 设计关键词、向量、结构化、规则、图和实时 API 等召回路。
3. 讲清统一候选格式、权限过滤、去重、融合和 Rerank 的执行顺序。
4. 比较 RRF、加权分数、学习排序和 Query 路由的优缺点。
5. 用每路独有贡献、Oracle Recall、候选污染率、延迟和成本判断某一路是否值得保留。

## 60 秒回答

> 多路召回是用多个互补检索通道并行或按路由找候选。例如 BM25 负责型号和错误码，稠密向量负责语义表达，规则或结构化查询负责产品、时间和状态，必要时再接图查询或实时 API。它的目标是降低单一检索器的盲区，而不是重复搜很多遍。
>
> 具体流程是：先根据用户身份生成统一的 ACL 和业务过滤条件，再为每一路生成适合的 Query；各路受控并行返回带稳定 document_id、chunk_id、来源、版本和分数的候选；然后做权限复核、版本过滤、去重和配额控制；由于各路分数不可直接比较，可以用 RRF 按名次融合，或在有标注数据时做分数校准和学习融合；最后对有限候选做 Rerank，再组装上下文。
>
> 上线前我会做消融实验，比较每一路的 Recall@K、独有相关文档、Oracle Union Recall、MRR/nDCG、噪声、P95 和成本。多一路不代表更好，如果它几乎没有独有贡献却增加噪声和尾延迟，就应该降权、只对特定 Query 启用，甚至移除。

## 详细解析

### 先说人话：像同时派不同专长的人去找资料

用户问：

```text
星云 X2 报 E-1047，更新后还是连不上，华东区能换货吗？
```

这个问题同时包含：

- 产品型号 `星云 X2`；
- 精确错误码 `E-1047`；
- 自然语言故障“更新后连不上”；
- 地区 `华东区`；
- 动作意图“换货”；
- 可能变化的订单与售后状态。

只用一种检索方法，很容易只解决其中一部分。多路召回像让不同专长的人并行找资料：

- 关键词专家查错误码；
- 语义专家查“连接失败”相关故障；
- 规则专家查华东区换货政策；
- 订单系统查购买时间和订单状态。

最后不能把所有资料乱堆在一起，还要统一、去重、融合和精排。

### 什么叫“召回”

信息检索中，**召回阶段**负责从大规模数据里快速找出一批“可能相关”的候选。第一阶段宁可保留一些待判断内容，也要避免把正确证据完全漏掉。

随后由 Reranker 和上下文选择缩小范围：

```text
全量知识库
   -> 多路召回：几百/几十个候选
   -> 融合与 Rerank：少量高相关候选
   -> 上下文组装：最终证据集合
```

召回、排序、上下文选择是不同阶段，指标也应分开。

### 多路召回不等于这些概念

| 容易混淆的做法 | 为什么不是完整的多路召回 |
| --- | --- |
| 把向量 Top-K 从 10 调到 100 | 仍然只有一个召回器，只是取更多结果 |
| 同一个 Query 搜三遍同一索引 | 若配置和表示相同，结果缺少互补性 |
| 只做多个 Query | 这是多查询扩展，可成为一类召回路，但不是全部 |
| 检索后让模型挑答案 | 这是候选选择，不解决第一阶段的覆盖盲区 |
| 把所有数据都放进长上下文 | 这是跳过检索，成本和噪声边界不同 |

### 常见的召回通道

#### 1. 关键词/BM25 路

适合：

- 错误码、型号、SKU、政策编号；
- 人名、地名和专业名词；
- 引号中的原文短语；
- 用户记得文档中的部分原话。

#### 2. 稠密向量路

适合：

- 用户口语与文档术语不同；
- 同义表达和概念匹配；
- 描述较长、含义比精确词更重要的问题。

#### 3. 规则与精确字段路

适合：

- 订单号、工单号和设备序列号；
- 已知字段上的精确查询；
- 热门高价值 Query 的人工运营规则；
- 法规、政策等必须命中指定权威源的场景。

规则路要有版本、负责人和过期机制，防止历史规则长期污染。

#### 4. 结构化数据库/API 路

适合：

- 当前订单状态、库存、价格；
- 需要范围、聚合或精确计算的数据；
- 事务性和实时性要求高的结果。

结构化数据不应为了“统一”全部转成文本向量后再猜。

#### 5. 多索引或多知识库路

例如：

- 产品手册索引；
- 售后政策索引；
- 历史工单索引；
- 用户可见帮助中心；
- 内部客服 SOP。

不同知识库有不同权威等级、权限和时效，不能只把结果分数混在一起。

#### 6. 图检索路

适合需要实体关系、依赖路径和跨文档聚合的问题。图检索不是默认升级项，只有关系本身对答案关键、图质量可控时才值得加入。

#### 7. 父子与层级召回路

一种做法是用小 chunk 做精确匹配，命中后返回较大的父段落；另一种是同时检索标题摘要和细节正文。它解决“检索粒度与阅读粒度不同”的问题。

### 多路召回的标准流程

```text
                         -> BM25 ---------
用户 Query -> 理解/路由 -> Dense --------|
                         -> SQL/API ------|-> 统一候选
                         -> Graph --------|      |
                         -> Rules ---------      v
                                            权限/版本复核
                                                  |
                                                  v
                                           去重与配额控制
                                                  |
                                                  v
                                              融合排序
                                                  |
                                                  v
                                               Rerank
                                                  |
                                                  v
                                              最终证据
```

### 第一步：生成统一权限与过滤条件

可信服务根据登录身份生成：

```json
{
  "tenant_id": "tenant_a",
  "allowed_departments": ["support"],
  "region": "east_china",
  "effective_at": "2026-07-15T10:00:00+08:00"
}
```

每一路都要执行等价权限约束。模型可以提取“华东区”这个业务条件，但不能自己扩大 `allowed_departments`。

### 第二步：为每一路准备合适的 Query

```text
原始 Query：星云 X2 报 E-1047，更新后还是连不上，华东区能换货吗？

BM25："星云 X2" "E-1047" 更新 连接失败 换货
Dense：星云 X2 更新后仍无法连接，应如何排查以及是否符合换货条件？
结构化：product=X2, region=east_china, action=exchange
订单 API：user_id + product_id -> purchase_date/status
```

一条通用 Query 不一定适合所有检索器。

### 第三步：受控并行执行

各路设置：

- 独立超时；
- 最大候选数；
- 并发和速率限制；
- 权限过滤；
- 熔断与降级；
- 来源和版本标记。

不是所有路都必须等待。若一个辅助召回路超时，可以使用已完成的合法候选降级；权限服务失败则应拒绝继续。

### 第四步：统一候选格式

```json
{
  "document_id": "doc_1024",
  "chunk_id": "doc_1024#chunk_07",
  "source": "product_manual",
  "version": "2026-06",
  "retriever": "bm25",
  "rank": 2,
  "raw_score": 13.8,
  "tenant_id": "tenant_a",
  "acl_tags": ["support"],
  "effective_from": "2026-06-01"
}
```

稳定 ID 是去重、引用、版本治理和追踪的基础。

### 第五步：权限和有效性复核

即使各路已经过滤，汇聚层仍应检查：

- 租户和 ACL；
- 文档是否删除或撤销；
- 版本是否为当前有效版本；
- 来源是否允许在当前回答中引用；
- 缓存结果是否来自相同权限版本。

### 第六步：去重与聚合

候选可能有三种重复：

1. 同一个 chunk 被多路召回；
2. 同一文档的相邻 chunk 内容重叠；
3. 不同文档实际上是同一政策的多个副本。

处理方式：

- 先按 `chunk_id` 精确去重，同时保留命中的召回路列表；
- 再按文档 ID、内容哈希或近重复模型合并；
- 相邻片段在上下文组装阶段按原顺序合并；
- 不要因为去重丢失“多路共同命中”这个有价值的信号。

### 第七步：融合排序

#### RRF：稳健的起点

RRF 按每一路的名次累加：

```text
RRF(d) = Σ 1 / (k + rank_i(d))
```

示例：

| 文档 | BM25 名次 | Dense 名次 | 直觉 |
| --- | ---: | ---: | --- |
| A | 1 | 3 | 两路都认可，融合后通常靠前 |
| B | 2 | 未命中 | 关键词路独有候选 |
| C | 未命中 | 1 | 向量路独有候选 |

RRF 不要求原始分数同尺度，适合先建立可靠基线。

#### 加权 RRF

当某一路在特定 Query 上更可靠时：

```text
score(d) = Σ weight_i / (k + rank_i(d))
```

权重应由分类型评测决定。例如错误码 Query 可以提高 BM25 权重，开放式故障描述可以提高 Dense 权重。

#### 分数校准与加权

如果能把各路分数校准成可比较的相关概率，可以加权融合。但模型、语料或查询分布变化后，校准可能失效。

#### 学习排序

有足够标注或用户反馈时，可训练模型利用：

- 各路原始分数与名次；
- 命中路数量；
- Query 类型；
- 文档权威性、时效和点击信号；
- 文档与用户上下文特征。

学习排序更灵活，也会带来训练偏差、特征泄漏、解释和维护成本。

### 第八步：候选配额与来源多样性

如果 BM25 返回 100 条而图检索只有 5 条，简单拼接会让 BM25 淹没其他路。可以设置：

- 每路最大候选数；
- 每个来源或文档的上限；
- 权威来源的保底配额；
- 去重后的动态补位；
- Rerank 前的总候选上限。

配额不是越平均越好，应围绕相关性和权威性评测。

### 第九步：Rerank

融合只利用分数、名次和少量特征。Reranker 会更深入地比较 Query 与候选正文。

正确顺序通常是：

```text
多路召回 -> 去重/融合 -> Rerank -> 上下文选择
```

Rerank 不能救回所有召回路都漏掉的文档，因此先看召回覆盖，再看精排。

## 工程实践与边界

### 怎样证明多路真的互补

除了整体 Recall@K，还要看：

| 指标 | 含义 |
| --- | --- |
| 每路 Recall@K | 该路单独覆盖多少相关证据 |
| 独有相关文档 | 只有该路召回的正确证据数量 |
| 候选交集率 | 各路是否高度重复 |
| Oracle Union Recall | 只要任一路召回就算命中的理论上限 |
| 融合后 Recall/MRR/nDCG | 融合有没有把正确内容保留下来并排好 |
| 候选污染率 | 无关、过期、越权候选的比例 |
| 增量 P95 与成本 | 新增一路带来的系统代价 |

如果 Oracle Union Recall 提升很大，但融合后收益很小，问题多半在去重、配额、融合或 Rerank，而不是继续增加召回路。

### 消融矩阵

```text
BM25
Dense
BM25 + Dense
BM25 + Dense + 规则路
BM25 + Dense + 结构化路
全部召回 + Rerank
全部召回 - 某一路
```

按 Query 类型报告结果。总平均值可能掩盖错误码类的明显退化。

### 动态路由与全量并行怎么选

| 方式 | 优点 | 风险 |
| --- | --- | --- |
| 所有路全量并行 | 覆盖稳定、实现直观 | 成本和尾延迟较高 |
| 规则路由 | 快、可解释 | 规则覆盖有限 |
| 模型路由 | 能处理复杂语义 | 路由错误、成本与漂移 |
| 先轻量召回再按置信度补路 | 平均成本较低 | 控制逻辑和评测更复杂 |

路由系统要有默认安全路径和失败回退，不能让路由器决定用户权限。

### 故障与降级

- 某个非关键检索器超时：使用其他已完成候选，记录降级。
- 实时 API 超时：明确状态暂不可确认，不用旧文档猜实时结果。
- Reranker 超时：退回融合排序，并监控该降级组的质量。
- 权限过滤异常：失败关闭，不返回候选。
- 新旧索引不一致：选择一个已发布版本，不混合未经验证的数据。
- 全路无可靠候选：澄清、拒答或转人工。

### 可观测性

每次请求记录：

- 启用了哪些路，为什么启用；
- 每路耗时、返回数、过滤数和超时；
- 每个候选来自哪些路及原始名次；
- 去重、融合、Rerank 后的名次变化；
- 索引、模型、规则和权限版本；
- 最终使用的证据与拒答原因。

这使“答案错了”可以追到具体召回路，而不是只看最终文本。

## 常见误区

### 误区一：多路召回就是多生成几个 Query

多 Query 只是一个维度。多路还可以来自不同检索算法、数据源、索引粒度和结构化能力。

### 误区二：路数越多，Recall 越高，系统就越好

弱路会增加噪声、成本和尾延迟，还可能挤掉正确候选。应看端到端增量收益。

### 误区三：把候选列表直接拼起来

不同路会重复、分数不同尺度、权限与版本状态也可能不同。必须统一、过滤、去重和融合。

### 误区四：所有路的 Top-K 都设成相同值

不同路的候选质量和成本不同。候选深度应按独有贡献、融合效果和预算选择。

### 误区五：Reranker 能修复所有问题

Reranker 只能对已有候选排序，无法找回第一阶段漏掉的证据，也不能替代权限和版本控制。

### 误区六：结构化数据也转文本做向量搜索

精确状态、金额和时间范围应由数据库/API 查询。把它们文本化后让模型推断，会损失精确性和实时性。

## 面试追问

### 1. 为什么 BM25 和 Dense 适合组合？

BM25 擅长字面精确和罕见词，Dense 擅长跨措辞语义匹配。二者错误模式不同，因此可能互补。

### 2. 为什么不直接相加 BM25 分数和余弦相似度？

它们的范围、分布和含义不同，且会随 Query 变化。可先用 RRF，或在标注数据上校准后加权。

### 3. RRF 后还需要 Rerank 吗？

常常需要。RRF 主要利用排名信号，不深入理解候选正文。Reranker 可以进一步判断候选是否真正回答 Query，但要权衡成本。

### 4. 什么是 Oracle Union Recall？

把所有路候选取并集，只要正确证据在任一路出现就算命中。它代表现有召回路理论上能提供的覆盖上限，不代表当前融合已经实现该上限。

### 5. 如何发现某一路在制造噪声？

看独有正确证据与独有错误候选、移除该路后的 nDCG/答案质量，以及它对 Rerank 候选和上下文预算的挤占。

### 6. 某一路很慢怎么办？

设置独立超时；只在高收益 Query 类型启用；做缓存或轻量预筛；异步补充非关键结果。实时关键数据不能用过期缓存静默替代。

### 7. 去重后应该保留哪个分数？

保留各路原始分数和名次作为特征，而不是只留一个最大值。融合层再计算统一分数，便于解释和回放。

### 8. 多路召回如何保证权限一致？

权限由可信身份系统生成并下发到每一路；汇聚层再次复核；缓存键包含权限版本；测试覆盖跨租户、权限撤销和服务失败场景。
