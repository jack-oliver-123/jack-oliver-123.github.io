小朋友第一次使用自动售货机时，并不是一出生就会操作。
他需要认识商品名称，理解按钮与商品的对应关系，知道要先付款再选择商品，还要能处理“余额不足”和“商品售罄”。
大模型学习工具调用也类似。
它不是突然获得了一只可以点击按钮的手。
它学到的是：
- 在什么情况下需要借助外部能力；
- 哪个工具最适合当前任务；
- 参数应该怎样填写；
- 工具返回后该怎样继续；
- 什么时候应该停止或向用户追问。
真实按钮仍然由宿主应用来按。

## 本篇目标
读完本篇，你应该能够：
- 区分“模型会生成工具调用”和“系统能执行工具”；
- 理解预训练、示例学习、监督微调和反馈优化各自的作用；
- 拆解工具调用需要的六项子能力；
- 解释模型怎样读懂从未见过的工具 Schema；
- 理解为什么工具名称和描述会影响准确率；
- 说明执行反馈怎样帮助模型恢复；
- 识别训练能力、API 能力和 Host 能力三个独立条件；
- 回答“模型变大后会不会自然学会工具调用”等追问。
先记住一句话：
> 模型学会的是预测合适的调用表示，宿主应用才把这种表示变成真实动作。

## 60 秒回答
LLM 学会工具调用，本质上是学会在“用户请求、对话历史和工具说明”这个条件下，生成合适的下一步。
这个下一步可能是直接回答、澄清、拒绝，也可能是某个工具名称和结构化参数。
预训练让模型获得语言、代码、API 文档和常见调用模式的基础知识。
监督微调通过高质量轨迹示范何时调用、选哪个工具、怎样填参数、怎样读取工具结果。
偏好优化、强化学习或基于执行结果的反馈，可以进一步减少错误调用、无效参数和过度调用。
推理时，Host 把当前可用工具及其 Schema 放入上下文。
模型不一定在训练时见过这个具体工具，也可以根据名称、描述和字段语义进行组合泛化。
但系统要真正可用，还必须同时满足三个条件：模型具备工具使用能力，模型 API 能传递结构化调用，Host 实现了校验、执行和结果回传循环。
缺少任何一个条件，都不能完成真实工具调用。

## 详细解析

### 1. “学会调用”到底学会了什么
不要把工具调用能力看成一个单一开关。
它至少包含六项子能力。

| 子能力 | 模型要解决的问题 | 常见错误 |
| --- | --- | --- |
| 调用判断 | 这次需不需要工具 | 该查不查、不该查却查 |
| 工具选择 | 应该用哪个工具 | 选到名字相近但用途不同的工具 |
| 参数抽取 | 用户提供了哪些参数 | 漏字段、类型错误 |
| 参数推断 | 哪些参数可由上下文确定 | 把“明天”换错日期 |
| 结果理解 | 工具返回说明了什么 | 忽略错误码、读错单位 |
| 恢复与停止 | 下一步是重试、追问还是结束 | 无限循环、重复写操作 |
训练集只覆盖“选对工具”，并不能保证其他五项能力。
生产评测也应该分别测量。

### 2. 预训练提供基础积木
预训练阶段，模型会接触大量自然语言、代码、函数签名、API 文档和调用示例。
它可能学到：
- 函数名通常表达用途；
- 参数有名称和类型；
- JSON 使用键值结构；
- 日期、城市和金额有常见表达；
- API 调用后会返回结果或错误；
- 代码常需要按特定依赖顺序组合函数。
这些知识是工具调用的基础。
但基础知识不等于稳定的产品能力。
一个模型能解释 REST API，不代表它能在严格接口格式中持续输出正确调用。

### 3. 监督微调教会行为格式
监督微调（Supervised Fine-Tuning，SFT）可以简单理解为：
> 给模型大量“输入与标准输出”的示范，让它模仿正确行为。
一条最小训练样本可能包含：
~~~text
用户：上海明天天气怎么样？

可用工具：
- get_weather(city, date)

标准输出：
调用 get_weather
city = 上海
date = 2026-07-16
~~~
完整训练轨迹还可以包含工具结果：
~~~text
工具结果：
condition = rain
temperature = 27

标准最终回答：
上海明天有雨，约 27 摄氏度，建议带伞。
~~~
模型由此学习工具调用前后的消息角色和格式。

### 4. 负样本同样重要
如果所有训练问题都要求调用工具，模型容易形成“见问题就调用”的习惯。
高质量数据要包含：
- 应该调用；
- 不应该调用；
- 信息不足，需要追问；
- 请求越权，需要拒绝；
- 没有合适工具；
- 工具返回失败；
- 工具结果相互冲突；
- 写操作需要确认；
- 调用一次后就应停止。
例如：
~~~text
用户：什么是天气预报？
正确行为：直接解释，不调用实时天气工具。
~~~
~~~text
用户：帮我订明天的票。
正确行为：追问出发地、目的地和乘客，而不是猜参数。
~~~
“不调用”也是工具使用能力的一部分。

### 5. 模型怎样理解新工具
推理时，Host 会提供当前工具说明。
模型可以利用自然语言语义理解一个训练时没见过的具体工具。
例如：
~~~json
{
  "name": "search_company_flights",
  "description": "查询符合企业差旅政策的航班，不创建订单",
  "parameters": {
    "from": "出发城市",
    "to": "到达城市",
    "departure_date": "出发日期"
  }
}
~~~
模型会把：
- 用户的“找航班”；
- 描述中的“查询航班”；
- 字段中的“出发城市”和“到达城市”
建立语义对应。
这是一种基于上下文的组合泛化。
它不是保证。
名称含糊、描述冲突、字段缩写或工具过多都会降低准确率。

### 6. 名称与描述为什么影响结果
比较两个工具：
~~~text
工具 A：do_action
描述：处理请求
~~~
~~~text
工具 B：search_flights
描述：只查询航班与票价，不创建订单
~~~
工具 B 的选择边界明显更清楚。
再比较：
~~~text
get_order：获取订单
query_order：查询订单
fetch_order：读取订单
~~~
如果三者实际功能相似，模型很难知道区别。
这不是单纯的模型问题，而是工具设计问题。
好的工具目录要做到：
- 名称稳定且语义明确；
- 描述说明“何时使用”；
- 描述也说明“何时不要使用”；
- 相近工具边界互斥；
- 参数名贴近业务语言；
- 枚举值有解释；
- 避免要求模型拼接复杂内部 ID。

### 7. Toolformer 展示了什么思路
Toolformer 研究了一种自监督思路。
核心直觉是：
1. 让语言模型在普通文本中提出候选 API 调用；
2. 执行这些调用；
3. 比较加入工具结果后，模型预测文本是否变得更好；
4. 保留有帮助的调用样本；
5. 用筛选后的样本继续训练。
这项研究的重要意义不是提供唯一训练配方。
它展示了模型可以从数据中学习：
- 在哪里插入工具调用；
- 使用什么参数；
- 怎样利用结果继续生成。
它也提醒我们，工具调用训练可以利用执行结果自动筛选数据。

### 8. ToolLLM、Gorilla 与 API-Bank 分别关注什么
这些研究从不同角度探索工具学习。

| 工作 | 主要关注点 | 可以得到的启发 |
| --- | --- | --- |
| ToolLLM | 大规模真实 API 的指令构造与工具使用 | 工具覆盖和多步轨迹很重要 |
| Gorilla | API 文档检索与准确调用 | 文档检索可以缓解工具知识变化 |
| API-Bank | 工具增强模型的训练与评测环境 | 要同时评估调用和对话结果 |
| Toolformer | 自监督产生并筛选调用 | 执行反馈可用于数据构造 |
这些论文的实验设置和产品 API 不完全相同。
面试时不要把论文中的某种格式说成行业统一标准。
应该提炼共同问题：
- 工具数量大；
- 文档会更新；
- 需要多步规划；
- 需要可执行反馈；
- 需要系统化评测。

### 9. 偏好优化在改什么
监督数据告诉模型“正确答案长什么样”。
偏好优化则可以告诉模型“两种行为中哪种更好”。
例如：

| 较差行为 | 较好行为 |
| --- | --- |
| 缺日期时猜一个日期 | 向用户追问日期 |
| 同时调用十个无关工具 | 只选择最相关工具 |
| 工具超时后无限重试 | 说明失败并给恢复选项 |
| 预订前直接执行 | 先展示关键参数并确认 |
| 工具报权限错误后换工具绕过 | 停止并说明权限不足 |
反馈来源可以是人工偏好、规则校验、执行结果或自动评审。
具体训练算法很多。
面试回答不应把这些常见方法都说成某个模型已经使用过。
除非厂商公开了训练细节，否则应明确说这是常见方法，而非产品事实。

### 10. 执行反馈为什么有价值
只有文本标签时，模型可能生成“看起来合理”的调用。
真正执行后才能知道：
- API 是否存在；
- 参数是否被接受；
- 资源是否可用；
- 调用顺序是否正确；
- 最终任务是否完成。
例如模型输出：
~~~json
{
  "name": "book_flight",
  "arguments": {
    "flight_id": "MU0001"
  }
}
~~~
执行返回：
~~~json
{
  "ok": false,
  "error": {
    "code": "MISSING_PASSENGER_ID",
    "retryable": false
  }
}
~~~
好的后续行为是向用户或档案工具获取乘客信息。
不是原样重复调用。
执行反馈能训练和评测这种恢复能力。

### 11. 推理时发生了什么
可以把模型推理理解成条件预测：
~~~text
下一个输出
= f(
  用户问题,
  对话历史,
  系统规则,
  当前可用工具,
  工具 Schema,
  已有工具结果
)
~~~
模型不是把所有工具永久写进参数后才会调用。
很多时候，当前工具说明就是上下文的一部分。
这也是为什么：
- 改工具描述会影响结果；
- 删除某项工具后模型不应再调用；
- 工具列表过大会挤占上下文；
- 动态加载工具是重要工程能力。

### 12. 训练能力、API 能力和 Host 能力
完成真实调用必须同时满足三层条件。
#### 条件一：模型能力
模型能理解工具说明并生成合适调用。
#### 条件二：API 通道
模型接口能把工具定义传进去，并把结构化调用可靠传出来。
即使没有原生通道，也可能用约束输出和解析器实现。
但可靠性与开发成本会不同。
#### 条件三：Host 执行循环
Host 能校验、授权、执行、回传并继续对话。
缺少这一层，调用只是一段结构化文本。
可以记成：
~~~text
会选择
+ 能表达
+ 有人执行
= 完整工具调用系统
~~~

### 13. “模型支持 MCP”为什么不是严谨说法
MCP 是 Host 与能力提供方之间的协议。
模型通常不直接建立 MCP 连接。
更严谨的问题是：
- 模型是否擅长工具调用；
- 模型 API 是否暴露结构化工具通道；
- Host 是否实现 MCP Client；
- Host 是否能把 MCP Tools 映射给模型；
- Host 是否实现调用结果回传。
一个模型即使不知道 MCP，也能在支持它的 Host 中使用 MCP 工具。
反过来，Host 能连接 MCP Server，也不能保证模型会正确选择工具。

### 14. 为什么模型规模不是唯一决定因素
更大的模型通常拥有更强的理解和组合能力。
但工具调用表现还受到以下因素影响：
- 是否有相关训练数据；
- 接口格式是否清晰；
- Schema 是否友好；
- 工具数量；
- 工具描述质量；
- 解码约束；
- 对话上下文；
- 评测分布；
- Host 的恢复策略。
因此不能说“参数量够大就自然支持所有工具”。
也不能只靠换大模型解决糟糕的工具设计。

## 工程实践与边界

### 1. 先设计任务分布，再谈模型
建立真实调用样本集。
至少分为：
- 必须调用；
- 可以直接回答；
- 必须澄清；
- 必须拒绝；
- 单工具；
- 多工具；
- 并行工具；
- 工具失败；
- 高风险写操作；
- 无匹配工具。
如果评测集全是“问题对应一个明显工具”，得分会虚高。

### 2. 分层测量
不要只看最终回答。
可以分别测：

| 层次 | 指标示例 |
| --- | --- |
| 调用判断 | 应调用召回率、不应调用误触发率 |
| 工具选择 | 工具名准确率、Top-K 召回 |
| 参数 | 字段级准确率、Schema 合法率 |
| 执行 | 成功率、超时率、重试率 |
| 恢复 | 错误后正确追问或停止的比例 |
| 最终结果 | 任务完成率、答案忠实度 |
分层以后，才能知道应该改模型、Schema、路由还是执行器。

### 3. 用简单接口降低学习难度
不要让模型调用：
~~~text
execute(operation_type, payload, options, metadata)
~~~
其中每个字段都可以塞任意对象。
优先拆成语义明确的小工具：
~~~text
search_flights(...)
create_travel_request(...)
book_flight(...)
~~~
但也不能拆得过细。
如果一次简单任务必须让模型正确组合二十个底层 CRUD，错误会累积。
工具粒度要围绕稳定业务动作。

### 4. 对动态信息使用检索
工具文档、版本和可用范围会变化。
把所有工具知识永久依赖模型参数会过时。
工具很多时，可以：
1. 维护版本化工具目录；
2. 根据用户请求检索候选工具；
3. 只把候选 Schema 放入上下文；
4. 执行前再按注册表校验版本；
5. 工具变更后重新跑回归集。

### 5. 不要让模型学习绕过安全
训练数据中如果包含“遇到权限错误就换另一个接口”，可能教出危险策略。
权限拒绝通常应该：
- 停止；
- 说明原因；
- 引导用户走正规授权；
- 记录审计。
恢复能力不等于绕过控制。

### 6. 处理分布外请求
分布外请求是训练和评测中没有覆盖的新情况。
安全默认策略应该是：
- 不确定时缩小动作；
- 缺信息时追问；
- 高风险时确认；
- 无匹配工具时明确说明；
- 不编造工具；
- 不把自由文本当可执行代码。

### 7. 版本变化要回归
工具增加、删除、改名或修改 Schema 后，都可能改变模型行为。
即使模型版本不变，也要重新测试：
- 工具选择冲突；
- 历史提示兼容；
- 参数默认值；
- 错误处理；
- 旧会话中的工具版本；
- 缓存是否刷新。

### 8. 不推测封闭模型的训练秘密
面试中可以说：
> 常见做法包括预训练基础、监督工具轨迹、偏好优化和执行反馈。
不要说：
> 某闭源模型采用了某篇论文中的具体算法。
如果官方没有公开，就应保留边界。

## 常见误区

### 误区一：模型调用工具是因为内部保存了所有 API
不准确。
模型可以根据推理时提供的工具说明理解新工具。

### 误区二：模型变大后自然就会稳定调用
不一定。
训练数据、接口通道、Schema 和评测同样重要。

### 误区三：只训练正确调用即可
错误。
还要训练不调用、澄清、拒绝、恢复和停止。

### 误区四：工具调用正确就代表任务完成
错误。
执行可能失败，结果可能被误读，多步顺序也可能错误。

### 误区五：JSON 合法率就是工具能力
错误。
JSON 合法只是参数评测的一小部分。

### 误区六：Host 只负责转发
错误。
Host 是授权、执行、状态和审计的核心边界。

### 误区七：模型能解释 MCP 就能连接 MCP
错误。
连接由 Host 的 MCP Client 实现。

### 误区八：工具越多，模型能力越强
错误。
无关工具会造成选择冲突和上下文膨胀。

## 面试追问

### 追问一：模型没见过这个工具，为什么还能调用
因为模型可以从名称、自然语言描述和字段语义进行上下文学习与组合泛化。
但效果依赖工具设计和模型能力，必须实测。

### 追问二：怎样训练“不要调用”
加入直接回答、澄清、拒绝、无匹配工具等负样本。
并在评测中测量误触发率。

### 追问三：执行反馈与普通监督数据有什么不同
执行反馈能验证 API 是否真实接受调用，以及任务是否真正完成。
它能发现“格式看起来正确但运行失败”的样本。

### 追问四：为什么 API 文档检索有帮助
工具多且版本变化时，检索当前文档可以减少依赖参数记忆。
也能只向模型展示本次相关工具。

### 追问五：怎样判断问题出在模型还是系统
按层排查：
1. 候选工具中是否有正确工具；
2. 描述是否清楚；
3. 模型是否选对；
4. 参数是否正确；
5. 执行器是否成功；
6. 结果是否正确回传；
7. 模型是否正确理解结果。

### 追问六：工具调用能完全靠 Prompt 教会吗
少量工具和简单任务可能通过提示与示例获得可用效果。
稳定泛化、复杂多步和错误恢复通常还依赖模型已有训练能力、结构化接口与系统控制。

### 追问七：模型为什么会过度调用
常见原因包括训练数据偏向调用、提示要求含糊、工具描述过宽、缺少直接回答样本，以及评测只奖励调用成功。

### 可背诵完整版
> LLM 学会工具调用，本质上是学会在用户问题、对话历史和工具 Schema 条件下预测合适行为。预训练提供语言、代码和 API 语义基础；监督微调用轨迹教会是否调用、工具选择、参数生成和结果使用；偏好优化或执行反馈进一步改善澄清、错误恢复与停止。推理时 Host 提供当前候选工具，模型可以根据描述理解新工具。但会生成调用、API 能传递调用、Host 能校验执行，是三个独立条件。模型只学会表达和决策，不会因此自动获得外部系统权限。
