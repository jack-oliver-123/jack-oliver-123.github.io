## 本篇目标

学完后，你应该能判断什么时候需要多 Agent，说明 Orchestrator、Worker 和 Handoff 的区别，并设计一套包含路由、任务交接、状态同步、失败回退和权限控制的协作流程。

## 60 秒回答

多 Agent 不是简单地让多个模型互相聊天，而是把一个复杂任务拆给多个职责明确、权限受限的 Agent。设计时要先确定控制拓扑，再定义通信契约。生产中常用中心 Orchestrator 维护任务图和全局状态，Worker 只处理有边界的子任务；需要把后续对话责任转给另一个专业 Agent 时，可以使用 Handoff。

动态切换不能让模型任意点名下一个 Agent。路由器应先按权限、租户、数据域、地区和健康状态过滤，再在合格候选中按能力、质量、成本和负载选择。转交时只传递完成任务必需的目标、约束、状态、证据和审批信息，并要求接收方确认。

全链路还要有任务 ID、状态版本、超时、跳数限制、访问链、幂等、审计和人工接管。A2A 等协议可以统一发现、任务和消息格式，但协议兼容不等于可信，身份、授权和业务语义仍然要由应用负责。

## 详细解析

### 先说人话：把多 Agent 想成一个有项目经理的小组

假设要处理一个复杂客户投诉：既要查订单，又要解释合同，还可能退款。

- **Orchestrator（编排器）**像项目经理：拆任务、分配任务、记录进度、验收结果。
- **Order Agent**像订单专员：只能读取订单相关数据。
- **Policy Agent**像政策专员：查找并解释公司政策。
- **Refund Agent**像退款专员：能准备退款，但真正执行可能需要人工审批。

如果只是把四个人拉进群，让大家自由聊天，没有负责人、交付格式和权限边界，信息越多反而越乱。多 Agent 系统也是一样。

### 先问：真的需要多个 Agent 吗

多 Agent 会增加模型调用、延迟、状态同步和排错难度。以下情况才比较值得使用：

- 子任务需要不同的专业上下文或工具；
- 不同数据域必须使用不同权限；
- 多个独立子任务可以并行；
- 单个 Agent 的工具太多、上下文太杂；
- 需要把责任明确转给另一个长期对话角色。

如果一个 Agent 加几个工具就能清楚完成任务，通常先用单 Agent。面试时说出这一点很重要：**多 Agent 是复杂度换能力，不是数量越多越智能。**

### 三种常见协作方式

| 方式 | 谁控制下一步 | 适合场景 | 主要风险 |
| --- | --- | --- | --- |
| Orchestrator-Worker | 中心编排器 | 可拆分、可验收的复杂任务 | 编排器成为瓶颈或单点 |
| Handoff | 当前 Agent 提议，运行时校验后转交 | 客服分流、专家接管后续对话 | 来回转交、上下文泄露 |
| 对等协作或群聊 | 多个 Agent 共同决定或轮流发言 | 探索、辩论、研究实验 | 难停止、难归责、成本高 |

生产环境通常优先选择 Orchestrator-Worker，因为控制权和全局状态更清楚。Handoff 适合“后续由谁负责”真的需要改变的场景。对等群聊更适合研究或边界很小的任务，不应默认用于高风险流程。

### 贯穿案例：处理一次退款投诉

用户说：“我昨天买的课程打不开，帮我解决，解决不了就退款。”

一个合理流程如下：

```text
用户请求
   ↓
Orchestrator 创建 task_id，识别需要订单与技术排查
   ↓
规则过滤出有权访问当前租户订单的 Order Agent
   ↓
Order Agent 返回订单状态和证据引用
   ↓
Support Agent 检查访问故障，返回“无法立即修复”
   ↓
Policy Agent 判断是否符合退款政策
   ↓
Refund Agent 生成退款预览
   ↓
人工确认金额和订单
   ↓
运行时执行退款，Orchestrator 验收并回复用户
```

这里每个 Worker 只返回结构化产物，不直接覆盖全局状态。最终是否完成，由 Orchestrator 按业务标准验收。

### 四类必须提前定义的协议

#### 1. 发现与能力协议

系统需要知道有哪些 Agent，以及它们能处理什么：

```json
{
  "agent_id": "refund-agent-v3",
  "capabilities": ["prepare_refund", "query_refund_status"],
  "input_schema": "refund-task-v2",
  "data_domains": ["orders-cn"],
  "health": "ready"
}
```

但能力声明只是一张“简历”，不是授权证明。运行时仍要验证发布者身份和真实权限。

#### 2. 任务协议

至少定义：任务 ID、父任务 ID、输入输出 Schema、状态、截止时间、取消方式、进度、完成条件和错误类型。

常见生命周期可以是：

```text
created → accepted → working → completed
                    ↘ failed
                    ↘ canceled
                    ↘ waiting_for_human
```

#### 3. 状态协议

必须回答：

- 全局状态由谁拥有？
- Worker 能修改什么？
- 两个 Worker 同时返回时如何合并？
- 状态版本冲突怎么办？
- 哪些结果是事实，哪些只是建议？

一个稳妥做法是：Orchestrator 拥有全局状态，Worker 提交带版本和证据的产物，由 Orchestrator 校验后合并。

#### 4. 故障协议

要提前规定超时、重试、替代 Agent、部分结果、补偿和人工升级。不能等故障发生后再让模型临时决定。

### 动态路由应该怎么做

动态路由可以分成两层：

```text
第一层：确定性过滤
权限、租户、地区、数据域、工具范围、健康状态、风险等级
   ↓
第二层：在合格候选中选择
语义能力匹配、历史质量、成本、延迟、当前负载
```

第一层是安全边界，不能交给模型随意突破。第二层可以使用规则、打分模型或 LLM。

例如 Refund Agent 的质量最高，但没有当前地区的退款权限，它就不应进入候选集。不能因为“它更聪明”而把任务交给它。

### Handoff 到底传什么

Handoff 是把后续处理责任转交给另一个 Agent。最小任务包通常包括：

| 内容 | 例子 |
| --- | --- |
| 用户目标 | 课程无法使用，无法修复则退款 |
| 已确认约束 | 只能退回原支付方式 |
| 当前状态 | 订单已查到，技术排查失败 |
| 待办事项 | 判断退款资格并生成预览 |
| 证据引用 | order:123、diagnosis:456 |
| 审批状态 | 尚未批准执行退款 |
| 截止时间 | 15 分钟内给出处理结果 |

不应默认复制完整聊天记录。完整复制会放大个人信息泄露、Prompt Injection 传播和无关上下文干扰。

交接还需要一个“握手”：发送方提出转交，运行时校验，接收方确认能够接单，Orchestrator 更新所有者。否则任务可能处于“双方都以为对方在处理”的状态。

### A2A 解决什么，不解决什么

Agent2Agent（A2A）是一套 Agent 之间互操作的开放协议，覆盖能力发现、任务、消息和产物等通信问题。可以把它理解成统一“怎么介绍自己、怎么收发任务”的语言。

它不自动解决：

- 这个 Agent 是否可信；
- 用户是否授权它读取某份数据；
- 两家公司对“退款完成”的业务定义是否相同；
- 错误时谁负责补偿；
- 敏感数据是否允许跨边界传递。

因此，协议兼容只是通信起点，身份、授权、信任和业务治理仍要在网关与应用层完成。

## 工程实践与边界

### 防止 Agent 互相踢皮球

每次转交记录：

```json
{
  "hop_count": 2,
  "max_hops": 5,
  "visited_agents": ["support", "policy"],
  "progress_marker": "refund_eligibility_confirmed"
}
```

达到最大跳数、再次进入相同 Agent 且状态没有进展，或连续两次被拒绝时，应回到 Orchestrator 或转人工。

### 并发与状态冲突

如果订单 Agent 和政策 Agent 并行工作，它们可能基于同一个旧版本状态返回结果。Worker 的结果应带 `state_version`，合并时检查版本；冲突时由 Orchestrator 重算或请求补充，不能让最后写入者随意覆盖前面的结果。

### 副作用与幂等

网络超时不代表退款没有执行。对于付款、退款、发信等写操作：

- 使用稳定的幂等键；
- 重试前查询外部状态；
- 明确唯一执行者；
- 无法确认时暂停并转人工；
- 不让多个 Agent 同时尝试同一个副作用。

### 权限和凭据

每个 Agent 使用独立的最小权限身份。跨域访问通过授权网关获取短期凭据，不能把一个“万能密钥”共享给所有 Agent。高风险任务没有合格候选时，应拒绝或转人工，不能降级给权限更宽的通用 Agent。

### 常见失败场景

| 失败场景 | 后果 | 处理方式 |
| --- | --- | --- |
| 接收方不可用 | 任务悬空 | 超时、健康检查、授权替代或返回部分结果 |
| 相互 Handoff | 无限循环和费用上涨 | 访问链、跳数上限、无进展检测 |
| 全量复制上下文 | 数据泄露、注入传播 | 最小任务包、来源和敏感级别标记 |
| 两个 Worker 同时写状态 | 结果互相覆盖 | 版本控制、单一状态所有者 |
| 重复提交写操作 | 重复付款或发信 | 幂等、状态对账、唯一执行者 |
| Agent 能力声明造假 | 越权或错误处理 | 验证身份、授权和实际能力，不只信声明 |

## 常见误区

- **“多个 Agent 天然比一个 Agent 强”**：协作会增加成本、延迟和故障点，应先证明拆分有收益。
- **“让 LLM 选一个 Agent 就完成路由”**：还缺权限过滤、健康检查、能力验证和失败回退。
- **“共享全部上下文协作最好”**：会增加泄露、注入传播和注意力干扰。
- **“Agent Card 可以直接信任”**：它是能力描述，不是身份和授权证明。
- **“任务转出去，原系统就没责任了”**：Orchestrator 仍要追踪生命周期、验收产物和处理失败。
- **“A2A 能自动解决安全问题”**：协议统一通信格式，安全仍依赖双方的身份、授权和数据政策。

## 面试追问

> **面试官：什么时候应该用多 Agent？**
>
> **候选人：** 当子任务需要不同工具、权限或专业上下文，或者可以安全并行时才值得拆分。如果一个 Agent 能清楚完成，优先保持简单，因为多 Agent 会引入通信、状态和故障复杂度。

> **面试官：Orchestrator 和 Handoff 有什么区别？**
>
> **候选人：** Orchestrator 始终拥有全局任务，只把有界子任务交给 Worker；Handoff 会改变后续处理责任。前者适合任务分解，后者适合客服分流和专家接管。

> **面试官：如何避免两个 Agent 相互转交？**
>
> **候选人：** 记录访问链和跳数，要求每次转交产生明确状态进展；重复访问且无进展或超过上限时，回到 Orchestrator、失败关闭或转人工。

> **面试官：下游 Agent 超时怎么办？**
>
> **候选人：** 先判断任务是否有副作用和能否幂等重试，再根据截止时间选择重试、授权替代、返回部分结果或转人工。不能无条件重复写操作。

> **面试官：跨厂商 Agent 怎么通信？**
>
> **候选人：** 可以用 A2A 等开放协议统一发现、任务、消息和产物格式；同时在网关完成身份验证、授权、限流、审计和数据策略，协议兼容不代表可以互相信任。
