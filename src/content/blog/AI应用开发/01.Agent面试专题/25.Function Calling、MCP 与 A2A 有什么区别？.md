## 本篇目标

不再把 Function Calling、MCP 和 A2A 当作三个互相替代的“工具调用方案”；理解它们分别解决模型表达动作、应用连接工具、Agent 协作任务的问题，并能画出三者组合使用的完整链路。

## 60 秒回答

Function Calling，也常叫 Tool Calling，是模型 API 提供的一种结构化输出机制。开发者先告诉模型有哪些函数及参数 Schema，模型在需要时返回“函数名 + 参数”。它只是提出调用请求，真正的函数仍由应用程序校验并执行。它主要解决“模型怎样表达要调用哪个能力”。

MCP 是 Model Context Protocol，主要标准化 AI 应用怎样发现并连接外部工具和上下文。MCP 采用 Host、Client、Server 架构，Server 可以暴露 Tools、Resources 和 Prompts，双方还会进行版本与能力协商。它主要解决“不同 AI 应用怎样用统一协议接入不同工具和数据源”。

A2A 是 Agent2Agent Protocol，主要让独立、可能属于不同团队或厂商的 Agent 互相发现能力并协作。它围绕 Agent Card、Message、Task、Task 状态和 Artifact 等对象，支持多轮、长时运行、流式和异步任务。它主要解决“一个 Agent 怎样把复杂任务交给另一个 Agent，并持续跟踪协作结果”。

三者可以一起使用：Agent A 的模型通过 Function Calling 决定使用某个能力；运行时通过 MCP 调用工具；遇到需要另一个专业 Agent 长时间处理的任务时，再通过 A2A 委派给 Agent B。它们是不同层次的互补能力，不是三选一。

一句话记忆：

> **Function Calling 管模型怎样“提出调用”，MCP 管 Agent 怎样“使用工具和数据”，A2A 管 Agent 怎样“与另一个 Agent 合作”。**

## 详细解析

### 先用公司协作理解三者

想象一名客服专员正在处理用户退款：

- 他决定填写一张“查询订单”的标准申请表，这像 Function Calling；
- 公司有统一系统，能把这张申请表送到订单、物流或支付服务，这像 MCP；
- 如果问题需要法务团队调查并多轮补充材料，客服把完整任务交给法务团队，持续查看进度并接收报告，这像 A2A。

这个比喻对应三个不同问题：

| 问题 | 对应技术 |
| --- | --- |
| 模型如何把自然语言意图变成结构化动作请求 | Function Calling |
| AI 应用如何统一连接工具、数据和提示模板 | MCP |
| 独立 Agent 如何发现、委派和协作长任务 | A2A |

只要先问“通信双方是谁、传递的基本单位是什么”，大部分混淆就会消失。

### Function Calling：模型返回调用意图

Function Calling 的基本参与者是：

1. 开发者的应用程序；
2. 大模型；
3. 应用程序能够执行的函数或工具。

应用先把工具定义发给模型：

```json
{
  "name": "get_order",
  "description": "根据当前用户的订单 ID 查询订单状态",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string"
      }
    },
    "required": ["order_id"],
    "additionalProperties": false
  }
}
```

当用户问“订单 ord_123 到哪了”，模型可能返回：

```json
{
  "name": "get_order",
  "arguments": {
    "order_id": "ord_123"
  }
}
```

然后应用程序才会：

```text
解析参数
  -> 校验 Schema
  -> 检查当前用户是否有权查看订单
  -> 调用真实订单服务
  -> 把结果交回模型
  -> 让模型生成最终回答或继续调用
```

Function Calling 最容易被误解的地方是：

> **模型没有直接执行函数，它只是生成一个结构化的调用请求。**

Function Calling 通常是某个模型平台 API 的能力。不同平台对工具字段、严格模式、并行调用和结果回传格式可能不同，因此它不是一个统一的跨厂商工具网络协议。

### Function Calling 解决什么，不解决什么

它擅长解决：

- 从多个已提供的函数中选择一个；
- 把自然语言信息提取成结构化参数；
- 根据工具结果继续生成或发起下一次调用；
- 让模型输出比自由文本更容易被代码处理的动作请求。

它不自动解决：

- 函数从哪里发现；
- 远程工具如何连接和传输；
- 用户身份、OAuth 和资源授权；
- 工具版本与能力协商；
- 高风险动作审批；
- 跨厂商 Agent 的长时任务协作；
- 调用超时、重试、幂等和审计。

这些仍然需要应用运行时、业务系统或更上层协议完成。

### MCP：为 AI 应用统一连接工具和上下文

MCP 的全称是 Model Context Protocol。可以把它理解为 AI 应用和外部能力之间的一套标准连接方式。

MCP 中常见的三个角色是：

| 角色 | 通俗理解 | 主要职责 |
| --- | --- | --- |
| Host | 承载 AI 体验的主应用 | 管理模型、权限、用户交互和多个连接 |
| Client | Host 内连接某个 Server 的协议客户端 | 建立会话、协商能力、发送请求 |
| Server | 对外提供能力的服务 | 暴露工具、资源和提示模板 |

一个 Host 可以连接多个 MCP Server。例如一个桌面助手同时连接文件系统 Server、代码仓库 Server 和数据库 Server。Host 通常为每个 Server 管理独立的 Client 连接和信任边界。

### MCP Server 能提供什么

MCP 的核心能力不只有 Tool：

| 原语 | 用途 | 例子 |
| --- | --- | --- |
| Tools | 执行计算、查询或动作 | 查询天气、创建工单 |
| Resources | 提供可读取的上下文数据 | 文件、数据库记录、API 内容 |
| Prompts | 提供可复用的提示模板 | 代码审查模板、报告模板 |

当前规范中，客户端和服务器会先初始化，协商协议版本与可用能力，然后再进入正常操作。工具通常通过 `tools/list` 发现，通过 `tools/call` 调用；工具输入和可选输出可以用 JSON Schema 描述。

MCP 使用 JSON-RPC 2.0 消息，并定义本地标准输入输出和远程 Streamable HTTP 等传输方式。具体版本会演进，生产系统应按协商结果使用能力，而不是假设所有客户端都实现了相同特性。

### MCP 工具调用与 Function Calling 怎样配合

两者经常出现在同一条链路，但负责不同部分：

```text
MCP Client 向 Server 获取工具清单
  ↓
Host 把相关工具定义提供给模型
  ↓
模型通过 Function Calling 选择工具并生成参数
  ↓
Host 校验调用请求和用户权限
  ↓
MCP Client 发送 tools/call
  ↓
MCP Server 执行并返回结果
  ↓
Host 把安全整理后的结果交回模型
```

Function Calling 是模型与 Host 之间的“动作表达”；MCP 是 Host / Client 与 Server 之间的“能力发现和调用”。

但这不是唯一组合：

- Function Calling 可以直接调用应用本地函数，完全不用 MCP；
- MCP Client 可以由确定性程序发起调用，不一定由模型选择；
- 一个应用也可以把 MCP 工具转换成某个平台的 Tool 定义再交给模型。

### A2A：让独立 Agent 围绕任务协作

A2A 的全称是 Agent2Agent Protocol。它关注的不是一个短小函数，而是一个可能会思考、补问、使用自己的工具、维护任务状态的远程 Agent。

A2A 的典型参与者是：

| 角色 | 说明 |
| --- | --- |
| User | 提出目标的人或服务 |
| A2A Client / Client Agent | 代表用户发起协作的一方 |
| A2A Server / Remote Agent | 接收任务并返回状态、消息和产物的一方 |

远程 Agent 对调用方通常是“黑盒”。调用方不需要知道它内部用了哪个模型、提示、记忆或工具，只需要知道它声明了哪些能力、怎样认证、任务当前进行到哪里，以及最终产出了什么。

### A2A 的几个核心对象

| 对象 | 通俗解释 | 作用 |
| --- | --- | --- |
| Agent Card | Agent 的能力名片 | 描述身份、接口、技能、输入输出形式和安全要求 |
| Message | 一次交流消息 | 传递请求、追问或补充信息 |
| Task | 一项有状态的工作 | 保存任务 ID、状态、历史和产物 |
| Task State | 任务当前阶段 | 例如已提交、处理中、完成、失败、需输入 |
| Artifact | 任务产生的可交付结果 | 报告、结构化数据、图片或其他内容 |

这使 A2A 能表达比一次函数调用更丰富的过程：

```text
Agent A 发现 Agent B 的能力
  ↓
Agent A 发送“调查退款争议”任务
  ↓
Agent B 返回“处理中”
  ↓
Agent B 请求补充订单凭证
  ↓
Agent A 提交材料
  ↓
Agent B 持续发送状态或产物更新
  ↓
任务完成，返回调查报告
```

A2A 官方规范支持任务状态、多轮消息、流式更新和异步处理，适合耗时较长、需要协商或中途补充信息的 Agent 协作。

### A2A 与 MCP 的本质边界

官方 A2A 文档用一个很实用的角度区分二者：看交互对象是工具，还是 Agent。

**MCP 一侧的能力通常：**

- 输入输出相对明确；
- 执行一个具体功能；
- 常见形式是查询、计算或短动作；
- 调用方把它当作可使用的能力。

**A2A 一侧的 Agent 通常：**

- 可以自行推理、规划和使用多个工具；
- 可能维护更长时间的任务状态；
- 会多轮补问、协商和产生产物；
- 调用方把它当作合作的独立执行者。

例如“查询某航班价格”很适合 MCP Tool；“综合航班、酒店、签证要求并反复确认偏好，最后提交旅行方案”更像一个 A2A Remote Agent 的任务。

工程边界允许折中。一个 Agent 的简单技能也可以包装成 MCP Tool；一个 MCP Tool 后面也可能代理复杂服务。选型时要看你是否真的需要任务状态、多轮协作、异步更新和 Agent 身份，而不是看服务内部有没有大模型。

### 三者完整对比

| 对比项 | Function Calling | MCP | A2A |
| --- | --- | --- | --- |
| 核心问题 | 模型怎样表达结构化调用意图 | AI 应用怎样连接工具与上下文 | 独立 Agent 怎样协作任务 |
| 主要通信双方 | 模型与应用运行时 | MCP Client 与 MCP Server | Client Agent 与 Remote Agent |
| 基本单位 | 函数名、参数、调用结果 | Tool、Resource、Prompt 和协议消息 | Message、Task、状态和 Artifact |
| 能力发现 | 通常由应用把工具列表发给模型 | 支持列出能力和协商变更 | 通过 Agent Card 描述能力与接口 |
| 状态模型 | 通常由应用自己维护 | 以连接和请求为主，也支持扩展任务能力 | 原生围绕有状态任务和多轮协作 |
| 是否定义传输 | 通常依赖模型提供商 API | 定义协议消息与传输 | 定义 Agent 通信接口与多种绑定 |
| 是否负责执行 | 否，应用执行 | Server 执行相应能力 | Remote Agent 自主处理任务 |
| 典型时长 | 一次或少量调用 | 短调用为主，也可支持长任务扩展 | 可长时运行、流式或异步 |
| 典型场景 | 提取参数、选择本地函数 | 接文件、数据库、业务工具 | 跨团队专业 Agent 委派与协作 |
| 是否互斥 | 否 | 否 | 否 |

### 三者一起工作的完整例子

用户说：

> “帮我判断这笔企业采购是否符合合同，并提交退款申请。”

系统可能这样工作：

```text
用户
  ↓
采购 Agent A
  ├─ Function Calling：模型决定先查订单
  ├─ MCP：调用订单 Tool 和合同 Resource
  ├─ A2A：把复杂合同解释任务交给法务 Agent B
  │    └─ 法务 Agent B 内部也可通过 MCP 查询法规库
  ├─ Agent A 汇总订单与法务结论
  ├─ 运行时展示退款金额并请求人工审批
  └─ MCP：审批后调用退款 Tool，验证结果
```

这个例子也说明：协议只负责互操作，不会自动替你做好业务编排、风险判断和审批。

### 怎样判断应该用哪个

可以按下面顺序提问：

1. **只需要让模型选择应用中已有的几个函数吗？** 先用 Function Calling。
2. **需要让多个 AI 应用以统一方式连接同一批工具或数据吗？** 考虑 MCP。
3. **交互对象是否是独立 Agent，并且任务需要多轮、状态、产物或异步更新？** 考虑 A2A。
4. **三个需求同时存在吗？** 可以组合，而不是强行只选一个。

如果只有一个应用和三个稳定本地函数，引入远程 MCP Server 可能增加不必要的部署、认证和运维成本。如果两个“Agent”之间只是一次确定性查询，把它建模为普通 Tool 可能更简单。

## 工程实践与边界

### 三层都要独立做安全控制

| 层次 | 主要风险 | 必要控制 |
| --- | --- | --- |
| Function Calling | 模型选错工具、伪造参数、越权意图 | Schema 校验、服务端鉴权、审批、幂等 |
| MCP | 恶意或被攻陷的 Server、过度授权、数据外流 | Server 信任、最小 OAuth Scope、工具确认、输出校验、网络限制 |
| A2A | 伪造 Agent、任务越权、恶意产物、回调滥用 | Agent 身份认证、任务级授权、租户隔离、产物扫描、回调验证 |

协议能定义认证和消息格式，但“对方值得信任吗”“当前用户能操作哪条订单”仍是业务系统必须回答的问题。

### 不要信任模型参数和远程元数据

- 模型生成的 `tenant_id`、`user_id`、`role` 不能作为可信授权依据；
- MCP Tool annotations 等描述性元数据只有来自可信 Server 时才可参考；
- A2A Agent Card 是发现入口，不等于自动建立业务信任；
- Remote Agent 返回的文本、文件和 URL 都应作为外部输入处理；
- 权限上下文应在每一跳明确传递、缩减并重新校验，不能默认继承全部权限。

### 让用户知道系统要做什么

对发送、付款、删除、上传文件等敏感工具调用，Host 应展示工具、关键参数和影响，并允许用户拒绝。A2A 任务如果会进一步委派或向外部组织传数据，也需要明确的数据使用边界。

批准应绑定到确定的动作和参数。审批“退款 100 元给订单 A”，不能被复用于“退款 1000 元给订单 B”。

### 版本、能力与兼容性

MCP 和 A2A 都在演进，客户端不能只根据协议名称假设每项可选能力都已经实现。应在运行时：

- 协商或读取对方声明的协议版本；
- 检查具体能力和输入输出模式；
- 对未知字段保持兼容，但拒绝未知高风险行为；
- 对 Tool Schema、Agent Card 和任务对象做契约测试；
- 记录协议、Server、Agent 和工具版本；
- 升级前跑互操作与端到端回归。

Function Calling 的不同模型和提供商也可能支持不同 Schema 子集、并行调用与严格模式，需要对真实目标模型做评测。

### 可观测性要贯穿三层

一次用户任务可能跨越模型调用、MCP 工具和 A2A 远程任务。应使用统一的 trace 或关联 ID 串起来，并记录：

- 谁发起了任务；
- 模型提出了什么调用；
- 运行时批准或拒绝了什么；
- 调用了哪个 MCP Server 和工具版本；
- 创建了哪个 A2A Task；
- 状态如何变化；
- 最终产物、错误、延迟和费用是什么。

日志要脱敏，不能把访问令牌、密码或完整个人数据作为普通 trace 内容保存。

### 测试应按层分开，再做组合测试

1. Function Calling 层测试工具选择、参数和不调用反例；
2. MCP 层测试初始化、发现、Schema、传输、错误、超时和权限；
3. A2A 层测试 Agent 发现、任务状态、多轮补充、流式或异步恢复；
4. 组合测试关联 ID、权限缩减、取消、超时和错误传播；
5. 在沙箱测试远程 Server 或 Agent 被攻陷、返回恶意内容的情况。

不要只用一个“正常天气查询”证明协议集成可靠。生产测试要包含断连、限流、重复消息、部分成功、任务取消、身份过期和对抗内容。

## 常见误区

- **“Function Calling 会替我执行函数”**：模型只返回调用请求，应用运行时才负责校验和执行。
- **“MCP 就是统一版 Function Calling”**：MCP 还定义客户端与服务器的发现、能力、消息、传输和上下文原语；Function Calling 处在模型交互层。
- **“有 MCP 就不需要 Function Calling”**：MCP 解决能力连接，模型仍可能通过 Function Calling 或其他机制选择某个 MCP Tool。
- **“A2A 是 MCP 的新版本”**：二者面向不同交互对象，官方定位是互补而非替代。
- **“用了 A2A 就自动成为 Multi-Agent 系统”**：A2A 规定通信，不替你设计角色、路由、共识、失败恢复和业务目标。
- **“任何远程服务都应该包装成 Agent”**：确定性、输入输出明确的能力通常更适合作为 Tool。
- **“Agent Card 或 Tool 描述就是可信承诺”**：发现元数据不能替代身份认证、授权和运行时验证。
- **“协议支持 OAuth 就等于权限安全”**：仍要正确设置 Scope、用户同意、资源级权限、令牌受众和租户隔离。
- **“协议统一后不需要端到端测试”**：格式兼容不代表业务语义、权限和故障恢复正确。

## 面试追问

> **面试官：** Function Calling、MCP 和 A2A 最简洁的区别是什么？
>
> **候选人：** Function Calling 是模型表达“调用哪个函数、参数是什么”的机制；MCP 是 Host / Client 连接 Tool、Resource 和 Prompt Server 的协议；A2A 是独立 Agent 围绕有状态任务进行多轮、流式或异步协作的协议。它们分属不同层次，可以组合。

> **面试官：** Function Calling 返回函数参数后，能直接执行吗？
>
> **候选人：** 不能把模型输出当可信命令。运行时还要做 Schema、身份、资源权限、业务规则和审批校验；高风险写操作还要处理幂等、超时和结果验证。

> **面试官：** MCP 和 Function Calling 是怎样配合的？
>
> **候选人：** MCP Client 可以先从 Server 发现工具，Host 再把相关定义提供给模型。模型通过 Function Calling 选择工具和参数，Host 校验后通过 MCP `tools/call` 调用 Server，再把结果交回模型。前者负责模型决策表达，后者负责能力连接和协议调用。

> **面试官：** MCP 为什么不只有 Tools？
>
> **候选人：** MCP 还定义 Resources 和 Prompts。Resource 更像可读取的上下文，Prompt 是可复用模板，Tool 是可调用能力。具体交互控制方式不同，不能把所有上下文都强行建成函数。

> **面试官：** 什么时候应该把能力设计成 A2A Agent，而不是 MCP Tool？
>
> **候选人：** 当远端需要自主规划、多轮补问、维护长任务状态、流式返回进度或产出复杂 Artifact，而且希望隐藏内部模型与工具时，更适合 A2A。短小、确定、结构化的查询或动作通常更适合 Tool。

> **面试官：** A2A 解决了 Multi-Agent 的所有问题吗？
>
> **候选人：** 没有。A2A 解决互操作通信和任务对象，角色分工、Agent 选择、冲突处理、权限、成本、质量评测与业务停止条件仍由系统架构负责。

> **面试官：** 三者能否出现在一个系统里？
>
> **候选人：** 可以。模型用 Function Calling 提出动作，运行时通过 MCP 使用内部工具；当需要外部专业 Agent 处理复杂长任务时，通过 A2A 委派。远程 Agent 内部还可以继续使用自己的 Function Calling 和 MCP 工具。

> **面试官：** MCP Server 返回的工具注解能直接决定是否跳过审批吗？
>
> **候选人：** 不能。MCP 规范明确要求客户端把来自不可信 Server 的 annotations 视为不可信。审批应基于 Host 自己的信任策略、用户身份、具体参数和业务风险。

> **面试官：** A2A Agent Card 的作用是什么？
>
> **候选人：** 它是远程 Agent 的能力名片，描述身份、接口、技能、输入输出形式和安全要求，帮助客户端发现并选择交互方式。但它不是自动信任证明，后续仍要认证、授权和验证结果。

> **面试官：** 如何追踪跨 Function Calling、MCP 和 A2A 的一次失败？
>
> **候选人：** 用统一 trace 或关联 ID 连接模型 Tool Call、MCP 请求和 A2A Task，记录各层版本、权限决策、状态变化和错误。然后找第一个偏离预期的节点，区分是模型选错、Host 拒绝、工具失败还是远程任务失败。
