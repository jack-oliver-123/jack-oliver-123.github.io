## 本篇目标

学完这一篇，你应该能够：

- 用生活化例子解释 LLM 网关位于哪里、为什么需要它；
- 区分反向代理、API 网关、LLM 网关和 Agent 编排器；
- 说清路由、限流、配额、重试、熔断、降级、成本与审计的区别；
- 解释流式响应和工具调用为什么让网关比普通 HTTP 代理更复杂；
- 用伪配置设计多租户、多供应商的网关策略；
- 回答“用了某个框架就算有网关能力吗”和“网关不能解决什么”。

## 60 秒回答

LLM 网关位于业务应用与模型服务之间，提供稳定的统一入口。它把不同供应商的认证、请求格式、流式事件、工具调用和错误码适配为内部契约，再集中实施模型与地域路由、每秒限流、周期配额、超时、有限重试、熔断、降级、成本预算、内容策略、审计和可观测性。与普通反向代理相比，它还理解模型能力、token 用量、首 token 延迟、上下文限制和工具调用事件。

我会把网关、Agent 编排和工具执行分开：网关治理“如何安全可靠地访问模型”；编排器决定“任务下一步做什么”；工具执行器负责参数校验、业务授权、幂等与补偿。网关不能提高模型本身的推理质量，也不能替应用判断一个退款调用是否获批。

框架只是实现选择，不是架构答案。Kong、Envoy AI Gateway、LiteLLM、云厂商 API 管理或自研层各有取舍。选型看协议覆盖、部署方式、流式与工具事件兼容、策略扩展、可观测性、数据驻留和运维能力。上线前要用影子流量验证语义兼容，避免把能力不同的模型当作透明备份。

## 详细解析

### 先说人话：网关像机场的统一塔台

一家航空公司可能连接多个机场、不同机型和不同地勤系统。如果每个机组自己记：

- 每个机场的联系办法；
- 不同跑道的规则；
- 哪里拥堵；
- 谁今天停运；
- 每条航线预算；
- 事故如何记录，

管理会很混乱。塔台提供统一入口和规则，但塔台不会替机长决定乘客的医疗处置，也不会替维修工程师修发动机。

LLM 网关也是这样：

- 上游是客服、搜索、代码助手、Agent 等应用；
- 下游是不同模型服务或自托管推理集群；
- 网关负责连接与治理；
- 业务决策仍留在应用和工具服务中。

### 贯穿案例：客服 Agent 接入多个模型服务

假设公司有一个客服 Agent：

1. 总结用户问题；
2. 查询订单；
3. 根据退款政策判断；
4. 必要时调用退款工具；
5. 回复用户。

公司同时使用：

- 一个适合复杂推理的模型服务；
- 一个低延迟模型服务；
- 一个部署在指定地域的内部模型；
- 一个备用供应商。

如果没有网关，每个业务服务都要维护多套 SDK、密钥、错误处理、用量统计和切换逻辑。久而久之：

- 同一种错误在不同应用里被重试不同次数；
- 密钥散落在多个服务；
- 某个团队没有预算限制；
- 日志字段不统一；
- 供应商故障时，各应用同时重试形成流量风暴；
- 工具调用事件在协议转换中丢失。

网关的价值是把这些横切能力集中治理。

### 先认识八个基础术语

| 术语 | 通俗解释 |
| --- | --- |
| Provider | 提供模型 API 或推理服务的一方 |
| Route | 根据条件把请求送到某个目标 |
| Rate Limit | 限制短时间请求速度，例如每秒请求数 |
| Quota | 限制较长周期总量，例如每天 token 或预算 |
| Retry | 同一个请求在满足条件时再次尝试 |
| Circuit Breaker | 下游持续故障时暂时停止发送，防止拖垮系统 |
| Fallback | 主路径不可用时切换到经过验证的备选路径 |
| Shadow Traffic | 复制真实请求给新路径比较，但不让其结果影响用户 |

限流和配额常被混淆：限流保护瞬时容量，配额控制长期使用和预算。

### 网关在架构中的位置

~~~text
业务应用 / Agent
        ↓
企业身份与调用方凭证
        ↓
LLM Gateway
  ├─ 统一协议
  ├─ 路由和健康判断
  ├─ 限流、配额、预算
  ├─ 超时、重试、熔断、降级
  ├─ 内容与数据策略
  └─ 指标、Trace、审计
        ↓
模型服务 A / 模型服务 B / 自托管集群
~~~

工具通常不应因为“经过了模型网关”就获得执行权限：

~~~text
模型返回工具调用意图
        ↓
Agent 编排器
        ↓
工具授权与执行网关
        ↓
订单、支付、文件、搜索等业务系统
~~~

模型网关和工具网关可以由同一平台团队建设，但信任边界与职责要分清。

### 普通反向代理、API 网关、LLM 网关有什么区别

| 能力 | 普通反向代理 | 通用 API 网关 | LLM 网关 |
| --- | --- | --- | --- |
| TLS、转发、负载均衡 | 有 | 有 | 有 |
| 调用方认证、API 策略 | 部分 | 有 | 有 |
| 理解 token 用量 | 通常没有 | 需扩展 | 核心能力之一 |
| 理解首 token 与流式事件 | 通常没有 | 需扩展 | 应支持 |
| 模型能力路由 | 没有 | 需自定义 | 应支持 |
| 工具调用事件适配 | 没有 | 需自定义 | 常见需求 |
| 提示与响应策略 | 没有 | 需扩展 | 常见需求 |
| 模型成本归集 | 没有 | 需扩展 | 常见需求 |

LLM 网关不是一个完全不同的物种，它通常建立在 API 网关或代理能力之上，再加入生成式 AI 的语义。

### 职责一：统一协议，但不要抹平语义差异

不同模型服务可能在这些方面不同：

- 消息角色与内容块；
- 工具定义与调用 ID；
- 并行工具调用；
- 结构化输出能力；
- 流式事件格式；
- 用量统计；
- 错误码与限流响应；
- 内容安全拒绝；
- 上下文和输入模态；
- 是否支持取消。

网关可以给上游一个稳定内部协议，但必须保留重要差异。例如：

~~~json
{
  "status": "provider_error",
  "gateway_code": "UPSTREAM_RATE_LIMITED",
  "provider": "provider_b",
  "provider_request_id": "req_xxx",
  "retryable": true,
  "retry_after_ms": 1200
}
~~~

如果只返回“500 模型失败”，应用无法判断限流、参数错误还是内容拒绝。

### 职责二：模型路由

#### 静态路由

按应用、租户或环境固定目标：

~~~text
生产客服 → 经过批准的主模型
开发测试 → 低成本测试模型
受监管租户 → 指定地域的部署
~~~

#### 能力路由

根据请求需求选择：

- 是否支持工具调用；
- 是否支持严格结构化输出；
- 是否支持图像或音频；
- 上下文容量是否满足；
- 是否符合数据驻留；
- 是否通过该业务评测。

#### 健康路由

综合：

- 实时错误率；
- P95/P99 延迟；
- 限流状态；
- 当前并发；
- 熔断状态；
- 地域健康。

#### 成本与预算路由

在满足质量门槛后，再按预算选择。不能只因某条路径便宜，就把高风险任务交给未经评测的模型。

#### 语义路由

根据任务类型路由，例如简单分类、复杂推理、代码或多模态。语义路由器本身也会误判，要有置信阈值、回退和独立评测。

### 路由规则的伪配置

~~~yaml
routes:
  - name: regulated-tenant
    when:
      tenant_tier: regulated
      data_region: cn
    require:
      - tool_calling
      - structured_output
    targets:
      - internal-region-cluster

  - name: customer-support
    when:
      application: support-agent
    targets:
      - provider-a-primary
      - provider-b-fallback
    fallback_requires:
      schema_compatible: true
      tool_protocol_compatible: true
      eval_gate: support-regression-v12

budgets:
  per_tenant:
    requests_per_second: 20
    tokens_per_day: 5000000
    monthly_cost_limit: configured-by-finance
~~~

这里的数字只是结构示例，不是通用推荐值。

### 职责三：限流、配额和并发控制

至少可以按以下维度控制：

- 用户；
- 租户；
- 应用；
- API key 或服务身份；
- 模型；
- 地域；
- 请求数；
- 输入和输出 token；
- 并发流；
- 预算。

#### 为什么只限制 QPS 不够

一个短分类请求和一个超长生成请求对资源的影响不同。还要考虑：

- 输入 token；
- 最大输出；
- 流式连接占用时间；
- 并行工具循环带来的后续模型请求；
- 每个租户的公平性。

网关应明确返回限流类型、重试时间和配额信息，便于应用安全降级。

### 职责四：超时、重试、熔断和降级

#### 超时

至少区分：

- 连接超时；
- 首 token 超时；
- 流中断或空闲超时；
- 总请求截止时间。

首 token 超时不等于模型没有开始处理；总超时也不代表供应商没有计费。

#### 重试

只对可重试、尚未产生不可安全重放结果的错误做有限重试，例如短暂网络错误或明确的服务不可用。

不能盲目重试：

- 无效参数；
- 认证与授权失败；
- 内容安全拒绝；
- 已经向客户端发送部分流；
- 完整 Agent 请求中可能已触发写工具的路径。

多层系统选择一层主导重试，避免应用重试 3 次、网关重试 3 次、SDK 又重试 3 次。

#### 熔断

当某个目标持续失败：

~~~text
Closed：正常请求
→ 失败超过门槛
Open：暂时拒绝或走备选
→ 等待探测窗口
Half-open：放少量探测请求
→ 成功则恢复，失败则继续熔断
~~~

#### 降级

可选方式：

- 切到经过评测的兼容模型；
- 降低非关键功能；
- 对低风险任务排队；
- 返回检索结果而不生成；
- 明确告知暂时不可用。

降级不能绕过数据驻留、权限和安全策略。

### 工具调用让故障转移更危险

模型 A 返回：

~~~json
{
  "call_id": "call_123",
  "name": "create_refund",
  "arguments": {
    "order_id": "o_42",
    "amount": 100
  }
}
~~~

网关必须保留：

- 调用 ID；
- 工具名；
- 参数内容；
- 流式事件顺序；
- 多个并行调用的关联；
- 最终停止原因。

如果流中断后透明切到模型 B，模型 B 可能再次产生退款调用。网关不知道第一次是否已经被应用执行，因此不能把完整 Agent 回合当普通只读 HTTP 请求自动重放。

### 职责五：成本与预算治理

网关可以归集：

- 输入、缓存输入和输出 token；
- 模型调用次数；
- 每个应用、租户、用户和功能的用量；
- 重试和回退产生的额外用量；
- 失败请求成本；
- 预算消耗速度。

有价值的指标不是只有“每次请求成本”，还包括：

~~~text
单位成功任务成本
= 模型、检索、工具和人工总成本
  ÷ 通过业务验收的任务数
~~~

更便宜但经常失败的路径，业务总成本可能更高。

### 职责六：可观测性和审计

#### 常见指标

| 类别 | 指标 |
| --- | --- |
| 流量 | 请求数、并发、输入输出 token |
| 延迟 | 首 token、总延迟、P50/P95/P99 |
| 稳定性 | 错误率、超时、重试、熔断、回退 |
| 路由 | 各目标占比、路由原因、误路由 |
| 成本 | 租户、应用、模型、功能的用量与预算 |
| 工具事件 | 工具调用解析失败、调用数、事件乱序 |
| 安全 | 策略拒绝、敏感数据检测、异常租户访问 |

#### Trace 应该串起什么

~~~text
业务请求
→ 网关路由决策
→ 供应商请求
→ 流式响应
→ 模型工具调用
→ 工具执行 Trace
→ 后续模型请求
→ 最终回答
~~~

OpenTelemetry 的生成式 AI 语义约定可以帮助统一 span、事件和指标字段，但处于不同稳定级别的字段要按官方版本治理，不能把实验字段当长期不变接口。

#### 审计与普通日志不同

普通日志用于排障；审计记录用于回答：

- 谁在什么时间调用了哪类模型；
- 网关为何选择该目标；
- 使用了什么策略版本；
- 是否触发敏感内容或预算规则；
- 哪个供应商 request ID 对应；
- 谁修改了路由和配额配置。

Prompt、工具参数和模型输出可能含敏感数据，默认脱敏和最小记录。

### 职责七：缓存

#### 精确缓存

只有请求关键因素相同才可复用，例如：

- 规范化输入；
- 模型与参数；
- 系统提示版本；
- 工具定义版本；
- 用户和权限域；
- 数据版本；
- 安全策略版本。

#### 语义缓存

相似问题返回历史答案，命中率高，但风险也更大：

- 语义相似不代表业务条件相同；
- 旧答案可能过期；
- 跨租户或权限泄露；
- 随机生成被错误复用；
- 工具调用结果不可重复使用。

高风险、实时、带写操作和权限敏感请求通常不应直接语义缓存。

### 框架怎么选

| 方案 | 更适合 | 主要取舍 |
| --- | --- | --- |
| Kong AI Gateway | 已使用 Kong，希望统一 API 与 AI 策略 | 关注插件能力和部署复杂度 |
| Envoy AI Gateway | 云原生、Envoy/Gateway API 体系 | 关注成熟度和团队运维经验 |
| LiteLLM Proxy | 需要快速统一多模型接口和预算 | 关注高可用、策略与版本治理 |
| 云厂商 API 管理 | 已深度使用对应云治理体系 | 关注供应商绑定和跨云能力 |
| 自研轻量网关 | 协议和策略非常特定 | 维护、安全和兼容成本更高 |

这张表表达实现方向，不是排名。
“使用过 LiteLLM”只是实现经历；面试官更关心你是否理解路由、故障、预算、安全和观测。

## 工程实践与边界

### 从直连迁移到网关

推荐步骤：

1. 盘点现有模型、SDK、密钥、重试和错误语义；
2. 定义内部请求、流式事件和工具调用契约；
3. 先让网关只做透传和观测；
4. 用回放或影子流量比较输出与延迟；
5. 逐步开启认证、限流和预算；
6. 再开启路由、重试、熔断和回退；
7. 每项策略设置独立门禁和回滚开关。

不要一次启用全部策略，否则回归难以归因。

### 影子流量的边界

影子请求：

- 不向真实用户返回结果；
- 不允许执行真实写工具；
- 敏感数据必须获准复制；
- 仍会产生模型成本；
- 要绑定相同输入与版本；
- 结果用于离线比较质量、协议和延迟。

### 防止网关成为单点

- 无状态数据面多副本；
- 配置和配额状态使用高可用存储；
- 按地域和故障域部署；
- 健康探测与容量保护；
- 配置变更灰度；
- 保留经过审批的应急路径；
- 故障演练覆盖配置中心、供应商和网络。

所谓应急直连也要审计，不能成为绕过治理的长期后门。

### 配置治理

- 路由规则版本化；
- 变更需要评审；
- 语法校验加语义校验；
- 在影子环境验证；
- 小流量发布；
- 自动监控错误、延迟和成本；
- 一键回滚到已知版本；
- 记录操作者和变更原因。

### 安全边界

网关集中持有供应商凭证，因此是高价值目标：

- 应用使用服务身份访问网关；
- 每个应用只允许经过批准的模型和能力；
- 供应商密钥放在密钥管理系统并轮换；
- 出站目标使用 allowlist；
- Prompt 和工具参数按数据分类处理；
- 管理面与数据面隔离；
- 审计配置与密钥访问；
- 防止用户伪造租户和预算标签。

### 网关不能解决什么

- 不能让弱模型自动变强；
- 不能代替 Prompt 和上下文设计；
- 不能判断业务事实是否正确；
- 不能代替工具执行授权；
- 不能让能力不同的模型无损互换；
- 不能消除提示注入；
- 不能替代端到端 Agent 评测；
- 不能保证外部供应商永不故障。

## 常见误区

- **“LLM 网关只是负载均衡”**：它还负责协议、策略、成本和生成式 AI 可观测性。
- **“统一接口后模型可以随便互换”**：工具、Schema、模态、安全和质量能力不同。
- **“接入某个网关框架就完成治理”**：框架只是载体，策略、SLO、评测和运维仍需设计。
- **“重试越多，可用性越高”**：会放大延迟、成本和下游压力，还可能重复副作用。
- **“流式请求失败可以透明换模型续写”**：部分内容和工具事件已经发出，语义难以保持。
- **“网关缓存只看 Prompt”**：身份、权限、模型、数据和策略版本也影响结果。
- **“网关统一错误码越简单越好”**：过度抹平会丢失可重试性和供应商定位信息。
- **“有网关后应用不用记录 Trace”**：应用仍需任务、工具和业务结果级可观测性。

## 面试追问

### 1. LLM 网关与 Agent 编排器有什么区别？

网关治理模型访问、路由、可靠性、预算和观测；编排器保存任务状态并决定下一步模型或工具动作。网关可以提供底层能力，但不应替业务决定是否退款。

### 2. 如何判断两个模型能否做故障切换？

比较工具调用、结构化输出、输入模态、上下文、数据地域和安全策略，再在真实评测集上验证语义质量。协议能转换不代表业务兼容。

### 3. 已经流式输出一半后故障，怎么办？

通常不能无痕重放。应终止或明确标记中断，根据业务决定让用户重试；如果已经出现工具调用，要先确认是否执行，避免重新生成重复写操作。

### 4. 限流和配额有什么区别？

限流保护短时间容量，例如每秒请求和并发；配额控制较长周期总量，例如每天 token、月度预算。生产中两者通常一起使用。

### 5. 网关层应该自动重试吗？

只对明确瞬时、尚未向上游交付部分结果、且请求可安全重放的故障有限重试。参数错误、拒绝、流式半成功和可能含写工具的完整 Agent 回合不应盲目重试。

### 6. 如何监控 LLM 网关？

按租户、应用、路由和目标统计请求、token、首 token、总延迟、错误、限流、重试、熔断、回退和成本；用 Trace 关联业务请求、供应商 request ID 与工具调用。

### 7. 为什么不能让每个应用直接保存供应商密钥？

密钥分散会增加泄露、轮换和审计难度。网关集中管理可以缩小分发范围，但网关自身因此需要更强隔离、密钥管理和访问审计。

### 8. 你会怎么回答“用过哪个框架”？

先如实说明实现，例如在哪种部署中使用过某框架；再讲具体解决了哪些协议、配额、路由和观测问题、遇到什么限制、如何评测。不要把产品名当作架构能力的替代。
