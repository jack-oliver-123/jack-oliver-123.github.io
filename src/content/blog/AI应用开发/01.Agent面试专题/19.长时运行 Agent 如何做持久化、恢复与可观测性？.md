## 本篇目标

学完后，你应该能解释为什么长任务不能只保存在内存中，知道检查点、状态机、租约、关联 ID、幂等键和 Trace 分别解决什么问题，并能设计一条不会盲目重复副作用的恢复流程。

## 60 秒回答

长时运行 Agent 应建模为可持久化状态机，而不是一个一直占用内存的循环。系统给每次任务分配稳定的 `run_id`，每一步记录当前节点、结构化状态、已完成动作、待处理动作、审批状态、预算和版本，并在关键状态变化以及副作用前后写检查点。进程重启后，先获取任务租约，再从最后一个已确认状态恢复。

恢复最难的是外部副作用，因为超时只表示没有收到结果，不代表动作没有执行。所有模型、检索和工具调用都应携带关联 ID，方便追踪；可能重试且有副作用的写操作，在下游支持时还应使用稳定幂等键。遇到“已发出但结果未知”的动作，要先查询外部状态，确认未执行且允许重试才重放，无法确认时暂停转人工或执行业务补偿。

可观测性方面，用 Trace 串联 Run、Step、模型、检索、工具、审批和错误；用指标监控延迟、重试、成本和人工介入；用审计记录谁在什么授权下执行了什么副作用。同时对 Prompt、响应、个人信息和密钥做最小化、脱敏和访问控制。

## 详细解析

### 先说人话：长任务要像游戏存档，也要像银行记账

一个任务如果只运行 2 秒，进程崩溃后重新开始也许问题不大。但 Agent 任务可能持续几小时或几天：

- 等用户补充资料；
- 等人工审批；
- 等外部系统完成处理；
- 中间经历多次模型与工具调用；
- 服务可能升级、重启或迁移。

这时只把状态放在内存里，就像玩了 10 小时游戏却从不存档，断电后全部丢失。

但 Agent 又不能简单地“从最近存档再点一次按钮”。如果上一次动作是银行转账，按钮可能已经点成功，只是页面没有返回。恢复设计既要像游戏存档，知道做到哪里；也要像银行记账，知道哪些动作可能已经产生真实后果。

### 先认识七个核心术语

| 术语 | 通俗解释 | 解决的问题 |
| --- | --- | --- |
| Run | 一次完整任务运行 | 区分不同任务实例 |
| Step | Run 中的一个可观察步骤 | 定位做到哪一步 |
| 状态机 | 明确定义状态和允许的迁移 | 防止流程随意跳转 |
| Checkpoint（检查点） | 持久化保存的可恢复状态 | 服务重启后继续 |
| Lease（租约） | 限时拥有任务执行权 | 防止两个 Worker 同时恢复 |
| Correlation ID（关联 ID） | 串联同一任务中不同系统的调用标识 | 跨服务排查一次请求 |
| Idempotency Key（幂等键） | 让同一个写请求重复提交时只产生一次业务效果 | 防止重复付款、退款、发信 |

关联 ID 和幂等键不是一回事：

- 关联 ID 解决“这些日志是不是同一次任务”，**所有调用都应该使用**；
- 幂等键解决“同一个副作用请求重复到达时，业务效果是否只发生一次”，主要用于可能重试的写操作。

### 贯穿案例：等待审批的退款 Agent

退款 Agent 的流程如下：

```text
查询订单
→ 判断退款资格
→ 生成退款预览
→ 暂停等待人工审批
→ 调用退款接口
→ 查询退款状态
→ 通知用户
```

任务可能在三个位置崩溃：

1. **等待审批时崩溃**：重启后应继续等待，不能重新创建多个审批请求。
2. **发起退款前崩溃**：确认状态显示尚未发起，可以安全继续。
3. **退款请求已发出、响应前崩溃**：结果未知，绝不能直接再退一次。

因此，系统必须区分“准备执行”“已经发出”“已经确认成功”，不能只保存一句自然语言摘要：“正在处理退款”。

### 为什么要使用状态机

状态机把模糊的聊天过程变成明确状态：

```text
created
  ↓
checking_order
  ↓
waiting_for_approval
  ↓
refund_prepared
  ↓
refund_in_flight
  ↓
refund_confirmed
  ↓
completed
```

每个状态只允许有限的下一步。例如没有有效审批时，不能从 `waiting_for_approval` 跳到 `refund_in_flight`。即使模型提出了这个动作，运行时也应拒绝。

### 检查点应该保存什么

一个可恢复检查点至少包含：

```json
{
  "run_id": "run-20260715-001",
  "workflow_version": "refund-v4",
  "state_version": 12,
  "current_node": "refund_in_flight",
  "task_state": {
    "order_id": "order-123",
    "amount": 299,
    "currency": "CNY"
  },
  "pending_action": {
    "tool": "create_refund",
    "tool_call_id": "call-789",
    "idempotency_key": "refund-order-123-v1",
    "status": "sent_unconfirmed"
  },
  "approval_id": "approval-456",
  "budget_remaining": 4,
  "updated_at": "2026-07-15T12:00:00+08:00"
}
```

大型文档和工具响应不必全部塞进状态表，可以保存不可变对象引用、版本和校验值。这样状态小而明确，同时能验证恢复时引用内容是否被改变。

### 检查点什么时候写

不是简单地“每 10 秒保存一次”，而要按状态和副作用边界保存：

- 关键任务状态发生变化后；
- 昂贵步骤完成后；
- 发起有副作用动作之前；
- 收到副作用确认之后；
- 进入人工等待之前；
- 暂停、取消或交接之前；
- 即将超过运行环境生命周期之前。

写得太少会丢失进度，写得太频繁会增加存储和锁竞争。最重要的是保证恢复点的语义清楚。

### 一条安全的恢复流程

```text
1. 读取检查点
2. 获取带过期时间的任务租约
3. 校验状态版本、工作流版本和工具 Schema
4. 找出已发出但没有确认结果的动作
5. 向外部系统查询真实状态
6. 成功：补记结果并继续
7. 明确失败且允许重试：使用同一幂等键重试
8. 无法确认：暂停、转人工或进入补偿流程
9. 恢复预算、截止时间和审批有效性
10. 从下一个允许状态继续
```

先获取租约，是为了避免原进程其实还活着，而恢复进程又同时推进任务。租约必须会过期，也要配合数据库版本检查，防止旧 Worker 在租约失效后继续写入。

### 为什么“超时后重试”很危险

一次工具调用可能有三种结果：

| Agent 看到的结果 | 外部系统真实状态 | 能否直接重试 |
| --- | --- | --- |
| 明确返回未执行 | 未执行 | 满足策略时可以 |
| 明确返回成功 | 已执行 | 不应重试 |
| 超时或连接中断 | 可能执行，也可能没执行 | 不能直接重试，先对账 |

这叫**不确定结果**。网络只告诉你没有收到响应，不能证明服务器没有完成操作。

### 幂等并不等于“请求绝不会重复”

幂等的目标是：同一个业务意图即使被发送多次，也只产生一次效果。

```text
第一次请求：idempotency_key=refund-order-123-v1 → 创建退款 R001
第二次请求：idempotency_key=refund-order-123-v1 → 返回同一个结果 R001
```

注意：

- 幂等键必须稳定，重试时不能重新生成；
- 不同业务意图不能复用同一个键；
- 下游系统必须真的支持去重，并定义保存多久；
- 如果下游不支持幂等，应先查询状态，无法确认时转人工或补偿。

所谓“Exactly Once（恰好一次）”在跨系统中通常是通过幂等、去重、事务和对账共同近似实现，不能只依赖消息队列重投。

### Outbox 模式解决什么

假设系统要同时完成两件事：

1. 把 Agent 状态更新为“退款已准备”；
2. 发布一条“执行退款”的消息。

如果数据库更新成功但消息没发出去，任务会卡住；消息发出但数据库更新失败，恢复后可能再次发送。

**Transactional Outbox（事务发件箱）**是在同一个数据库事务里写入业务状态和“待发布消息”，再由后台进程投递消息。它能避免“状态写入和消息意图二选一成功”，但后续投递通常仍是至少一次，因此消费者仍需幂等、去重和状态对账。它也不能自动让任意外部副作用变成数据库事务。

### 可观测性到底观察什么

| 能力 | 回答的问题 | 例子 |
| --- | --- | --- |
| 日志（Logs） | 某个时刻发生了什么 | 工具返回超时 |
| 指标（Metrics） | 系统整体是否异常 | P95 延迟、失败率、重试率 |
| 追踪（Traces） | 一次任务经过了哪些服务 | 模型 → 检索 → 工具 → 审批 |
| 状态历史 | 任务如何迁移 | waiting → in_flight → confirmed |
| 审计（Audit） | 谁在什么授权下做了什么 | 审批人、金额、执行结果 |

一条 Trace 可以按层级组织：

```text
Run span
├── Step: retrieve_policy
│   └── Retrieval span
├── Step: decide_refund
│   └── Model span
├── Step: wait_approval
│   └── Approval span
└── Step: execute_refund
    ├── Tool span
    └── Reconciliation span
```

Trace 记录可观察的输入摘要、输出摘要、版本、耗时、状态和错误即可，不需要记录模型的私有思维链。

## 工程实践与边界

### 必备指标

- 任务完成时间和队列等待时间；
- 每个步骤的 P50、P95、P99 延迟；
- 模型与工具调用次数；
- 重试率、不确定结果率和补偿率；
- 无进展轮次和循环次数；
- Token、工具费用和单任务总成本；
- 人工介入率、审批等待时间；
- 恢复成功率和重复副作用事件数。

### Pause、Resume、Cancel 和 Deadline

长任务必须支持暂停、恢复、取消和截止时间。取消不是把数据库状态改成 `canceled` 就结束，还要把信号传播给仍在运行的模型和工具。已经提交的外部操作可能无法取消，此时要记录结果并按业务规则补偿。

### 版本升级

检查点可能由旧代码写入。升级时有三种常见选择：

- 提供状态迁移程序；
- 让旧 Run 继续由旧版本 Worker 完成；
- 对无法兼容的任务暂停并人工处理。

不能假设新代码能读懂所有旧状态，也不能在恢复时悄悄改变已审批动作的含义。

### 隐私和安全

- 所有调用使用关联 ID，但关联 ID 本身不要包含用户身份证号等敏感信息；
- Trace 默认不保存完整 Prompt、响应、密码、密钥和支付信息；
- 记录内容按租户、角色和用途控制访问；
- 设置保留期限和自动删除；
- 审计记录需要防篡改，但展示时仍应最小化敏感字段。

### 常见失败场景及处理

| 失败场景 | 根因 | 处理方式 |
| --- | --- | --- |
| 两个 Worker 同时恢复 | 没有执行权协调 | 租约、状态版本和条件更新 |
| 超时后重复退款 | 把未知当失败 | 同一幂等键、状态查询、人工对账 |
| 检查点恢复后读不懂 | 版本不兼容 | 状态迁移或旧 Worker |
| 日志很多但无法定位 | 没有结构和关联 ID | 统一 Run、Step、Call 标识和字段 |
| Trace 泄露客户数据 | 默认保存完整上下文 | 最小化、脱敏、访问控制和保留期 |
| 取消后工具仍在执行 | 信号未传播 | 协作式取消、结果对账与补偿 |

## 常见误区

- **“有数据库就能恢复”**：还需要状态机、检查点语义、并发控制、版本兼容和副作用协调。
- **“超时等于执行失败”**：超时表示结果未知，外部动作可能已经成功。
- **“有幂等键就可以无限重试”**：还要有重试策略、有效期、状态查询和业务上限。
- **“Outbox 能保证所有事情恰好一次”**：它原子保存状态与消息意图，消费者和外部副作用仍要幂等与对账。
- **“日志越全越可观测”**：无结构、无关联并含敏感内容的日志既难查询又危险。
- **“重启后把聊天记录重新喂给模型即可”**：聊天摘要无法精确恢复工具提交、审批、并发和外部状态。

## 面试追问

> **面试官：长时 Agent 为什么要建模为状态机？**
>
> **候选人：** 状态机把当前节点、允许迁移和副作用边界变成程序可验证的状态，服务重启后能精确恢复，也能阻止模型跳过审批或重复执行。

> **面试官：检查点应该多久写一次？**
>
> **候选人：** 按状态和副作用边界写，而不是只按固定时间。关键状态变化、昂贵步骤完成、人工暂停以及副作用前后通常都要写。

> **面试官：关联 ID 和幂等键有什么区别？**
>
> **候选人：** 所有调用都用关联 ID 来串联追踪；可重试且有副作用的写操作还用幂等键，确保同一业务意图重复到达时不产生多次效果。

> **面试官：退款请求超时了，如何恢复？**
>
> **候选人：** 先把它当成结果未知，使用原工具调用 ID 或业务键查询外部状态。已成功就补记结果，明确未执行且允许重试才用同一幂等键重放，无法确认则暂停转人工。

> **面试官：如何避免两个 Worker 同时恢复？**
>
> **候选人：** 使用带过期时间的租约获取执行权，状态更新再校验版本和租约所有者，防止租约失效的旧 Worker 覆盖新状态。

> **面试官：Trace 可以记录完整上下文吗？**
>
> **候选人：** 默认不应。应记录来源、版本和脱敏摘要；只有经授权的排障场景临时采集必要内容，并限制访问、用途和保留时间。
