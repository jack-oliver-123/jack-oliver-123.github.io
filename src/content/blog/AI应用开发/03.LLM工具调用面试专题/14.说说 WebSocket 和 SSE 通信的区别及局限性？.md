## 本篇目标

学完这一篇，你应该能够：

1. 解释 SSE 是 HTTP 事件流，WebSocket 是建立后的双向帧协议。
2. 区分单向与双向、文本与二进制、自动重连与应用自建恢复。
3. 说明二者都没有自动提供完整业务可靠性、幂等和消息持久化。
4. 判断 LLM 文本流、工具事件、实时协作和控制信令分别适合什么方案。
5. 排查代理缓冲、空闲超时、慢消费者、断线重复和鉴权过期。
6. 说明为什么“WebSocket 功能更多”不等于所有场景都应该选择它。

## 60 秒回答

> SSE，也就是 Server-Sent Events，使用 HTTP 的 `text/event-stream` 让 Server 持续向 Client 发送 UTF-8 文本事件。浏览器 `EventSource` 是单向的 Server 到 Client，内建断线重连、`retry` 和 `Last-Event-ID` 语义；Client 若要发送数据，继续使用普通 HTTP 请求。WebSocket 则建立一条持久双向连接，双方可随时发送文本或二进制帧，并提供消息、Ping/Pong 和关闭帧，但重连、会话恢复、业务确认和消息重放需要应用自行设计。
>
> LLM 常见的“一次请求、持续返回文本和状态事件”通常用 SSE 更简单，也更容易复用 HTTP 鉴权、网关和观测。实时协作、频繁双向控制、在线状态或二进制消息更适合 WebSocket。不过 SSE 可能被代理缓冲或空闲超时，浏览器原生接口自定义请求能力有限；WebSocket 则增加长连接治理、负载均衡、部署排空、背压和协议设计成本。
>
> 两者都不等于消息队列，也都不保证业务恰好一次。断线重连后要用事件 ID 或业务序号去重，写操作使用幂等键并先查状态。选型应看数据方向、频率、媒体类型、恢复要求和现有基础设施，而不是看哪个协议“更高级”。

## 详细解析

### 先说人话：SSE 像广播，WebSocket 像对讲机

SSE 像车站广播：

- 广播站持续向旅客发送消息；
- 旅客若要反馈，需要走另一个窗口；
- 断线后可以告诉广播站“我最后听到第 42 条”。

WebSocket 像一条双向对讲机：

- 双方随时可以讲话；
- 可以传文字，也可以传二进制；
- 但“刚才断线前那句话对方听到了没有”，要由双方的业务协议确认。

这两个类比能建立直觉，但不能替代真实协议细节。

### 什么是 SSE

SSE 全称 **Server-Sent Events**。

服务端返回：

```http
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache

id: 101
event: token
data: 你好

id: 102
event: token
data: ，世界

event: done
data: {}

```

事件之间使用空行分隔。

常见字段：

| 字段 | 作用 |
| --- | --- |
| `data` | 事件数据，可以出现多行 |
| `event` | 自定义事件类型 |
| `id` | 事件标识，可用于恢复 |
| `retry` | 建议客户端重连等待时间，单位毫秒 |
| 以 `:` 开头 | 注释，可用于保持活动 |

事件流使用 UTF-8 文本。

### 浏览器 `EventSource`

浏览器提供 `EventSource` API：

```javascript
const source = new EventSource('/api/events');

source.addEventListener('token', (event) => {
  renderToken(event.data);
});

source.addEventListener('done', () => {
  source.close();
});
```

它的特点：

- 使用 HTTP 获取事件流；
- 连接异常时默认自动重连；
- 可以处理 `id` 和 `Last-Event-ID`；
- 原生方向是 Server 到 Client；
- 只承载文本事件；
- API 简单。

### “SSE 只能 GET”这句话要怎样说才准确

浏览器原生 `EventSource` 构造连接时使用 GET，且可自定义请求头的能力有限。

但 `text/event-stream` 是一种 HTTP 响应格式。

一个 API 完全可以：

1. Client 使用 POST 提交 Prompt；
2. Server 在该 POST 响应中返回 SSE 编码事件；
3. Client 使用 Fetch 流式读取并解析。

因此更准确的说法是：

```text
EventSource API 主要使用 GET；
SSE 编码也可以出现在其他 HTTP 方法的响应中，前提是客户端自行解析。
```

### 什么是 WebSocket

WebSocket 是一套双向通信协议。

经典方式先通过 HTTP/1.1 Upgrade 完成握手：

```http
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
```

Server 接受后返回 `101 Switching Protocols`。

随后双方使用 WebSocket 帧通信，不再把每条业务消息当普通 HTTP 请求和响应。

标准还定义了在 HTTP/2 和 HTTP/3 上引导 WebSocket 的机制，但实际 Client、代理和服务端支持需要单独验证。

### WebSocket 帧和消息

WebSocket 支持：

- 文本帧；
- 二进制帧；
- Continuation 帧；
- Ping；
- Pong；
- Close。

一条应用消息可以由一个或多个帧组成。

浏览器示意：

```javascript
const socket = new WebSocket('wss://ai.company.test/chat');

socket.addEventListener('open', () => {
  socket.send(JSON.stringify({
    type: 'user_message',
    text: '你好'
  }));
});

socket.addEventListener('message', (event) => {
  handleMessage(event.data);
});
```

协议有帧边界，但不知道你的业务字段是什么意思。

应用仍要定义：

- 消息类型；
- 请求 ID；
- 顺序号；
- 错误对象；
- 认证刷新；
- 确认与重放；
- 版本；
- 兼容性。

### 核心对比表

| 维度 | SSE | WebSocket |
| --- | --- | --- |
| 通信方向 | 原生 Server -> Client | Client <-> Server |
| 基础 | HTTP 响应事件流 | 握手后使用 WebSocket 帧 |
| 数据类型 | UTF-8 文本 | 文本和二进制 |
| 浏览器 API | `EventSource`，也可 Fetch 解析 | `WebSocket` |
| 自动重连 | `EventSource` 内建基础重连 | 应用自行实现 |
| 恢复位置 | `id` / `Last-Event-ID` 语义 | 应用自行定义序号或游标 |
| 消息边界 | 空行分隔事件 | 协议提供消息/帧边界 |
| Ping/Pong | 无 WebSocket 控制帧，可用注释心跳 | 协议定义 Ping/Pong |
| 复用 HTTP 语义 | 强 | 握手后不再是普通请求/响应 |
| 双向高频事件 | 需另发 HTTP 请求 | 原生适合 |
| 二进制 | 不直接支持 | 支持 |
| 典型 AI 场景 | 文本生成、任务进度 | 实时协作、频繁控制、双向事件 |

### 方向：最先考虑的维度

如果通信形态是：

```text
Client 发一次请求
Server 持续返回 token、引用、工具状态和完成事件
```

SSE 很自然。

如果通信形态是：

```text
双方在一条会话内持续高频发送
用户输入、打断、光标、控制、确认和状态
```

WebSocket 更自然。

不要因为未来“可能有双向需求”就立即选择最复杂方案。

普通 HTTP 请求与 SSE 可以很好地组成非对称双向系统：

```text
Client -> Server：POST 请求
Server -> Client：SSE 流
```

### 文本与二进制

SSE 数据是 UTF-8 文本。

二进制必须：

- Base64 编码；
- 转成 URL；
- 或通过单独文件通道传输。

Base64 会增加体积和编解码成本。

WebSocket 可以直接发送二进制帧，更适合：

- 自定义二进制协议；
- 小型音频块；
- 游戏或协作状态；
- 已有二进制消息格式。

但实时音视频通常还要考虑 WebRTC，而不是因为 WebSocket 能传二进制就直接自己重建媒体栈。

### 连接与基础设施

#### SSE

SSE 保留 HTTP 语义，通常更容易复用：

- Cookie 或标准 HTTP 认证；
- 反向代理；
- API 网关；
- 限流；
- HTTP 日志；
- 追踪；
- HTTP/2 多路复用。

但代理可能：

- 缓冲响应；
- 压缩后延迟刷新；
- 在空闲时关闭；
- 错误缓存；
- 不支持足够长的响应时间。

#### WebSocket

WebSocket 需要基础设施正确支持握手和持久连接。

还要考虑：

- 长连接数量；
- 每实例连接上限；
- 连接与用户的路由；
- 会话状态；
- 扩缩容；
- 服务发布时的连接排空；
- 负载均衡健康检查；
- 代理空闲超时。

一条连接保持数小时，会让部署模型与普通短 HTTP 请求明显不同。

### 自动重连不等于可靠消息

`EventSource` 会自动尝试重连，并可以发送 `Last-Event-ID`。

但 Server 是否：

- 保存事件历史；
- 支持从该 ID 重放；
- 保留多久；
- 如何处理过期 ID；
- 是否去重；

都需要应用设计。

WebSocket 甚至连基础重连都要应用实现：

```text
断线
  -> 指数退避
  -> 重新认证
  -> 发送最后确认序号
  -> Server 补发可恢复消息
  -> Client 去重
```

所以两者都不天然提供“恰好一次”。

### TCP 与队头阻塞

经典 WebSocket 通常运行在可靠、有序的字节流上。

SSE 的 HTTP 流最终也依赖可靠传输。

如果底层某段数据丢失，后续数据可能等待重传，这叫 **队头阻塞**。

它保证顺序和完整性，但对实时媒体可能不理想。

不要把 WebSocket 说成“UDP 一样低延迟”，也不要把 SSE 说成“天然不会丢”。

### 背压：消费者读不动怎么办

**背压**指生产速度高于消费速度时，系统怎样限制积压。

#### SSE 的问题

浏览器 `EventSource` 接口没有提供很精细的应用级流控。

Server 如果不断写，而 Client 或网络很慢，缓冲区可能增长。

#### WebSocket 的问题

浏览器 WebSocket 也不是完整的流式背压 API。

可以观察 `bufferedAmount`，但应用仍需设计发送上限、丢弃、合并或断开策略。

通用做法：

- 限制单消息大小；
- 限制每连接队列；
- 合并低价值进度；
- 不逐 token 永久落队列；
- 达到水位时暂停上游；
- 过慢 Client 明确断开；
- 大产物使用对象存储和链接。

### 鉴权差异

#### SSE / EventSource

浏览器原生 `EventSource` 自定义请求头能力有限。

常见方式：

- 同源 Cookie；
- 短期、一次性流票据；
- 先通过认证 POST 创建任务，再用不可猜测但非长期密钥的 ID 订阅；
- 使用 Fetch 自行读取事件流，以便发送所需 Header。

不要把长期访问令牌放进 URL query，因为 URL 可能进入日志、历史和监控。

#### WebSocket

浏览器 WebSocket API 同样不能任意设置所有握手 Header。

常见方式：

- Cookie；
- 短期握手票据；
- 通过受控子协议；
- 建连后立即发送认证消息。

无论哪种方式，都要处理凭证过期、重新认证和权限变化。

### 心跳

SSE 可以发送注释：

```text
: keep-alive

```

用于防止中间层将连接判断为空闲。

WebSocket 协议定义 Ping/Pong 控制帧，但浏览器 API 不一定把底层 Ping/Pong 暴露给 JavaScript。

应用还可能定义业务心跳。

心跳只能说明某一层连接仍有响应，不能证明：

- 后端模型正常；
- 任务仍在推进；
- 数据已经持久化；
- 写操作已经成功。

### 关闭与取消

用户点击“停止生成”时：

```text
关闭浏览器流
        ≠
后端任务一定取消
```

系统应传播独立取消信号：

- 标记任务已取消；
- 停止尚未开始的工具；
- 尽力中断可取消的模型或工具；
- 不回滚已经完成且不可逆的副作用；
- 记录最终状态。

WebSocket Close 和 SSE 连接断开都不能独自承担完整业务取消语义。

### AI 文本流中的事件设计

不要把所有数据都塞成一串无类型字符串。

可以定义：

```text
response.started
text.delta
citation.added
tool_call.started
tool_call.completed
warning
response.completed
response.failed
```

SSE 示例：

```text
id: 201
event: text.delta
data: {"text":"你好"}

id: 202
event: tool_call.started
data: {"call_id":"c1","name":"search"}

id: 203
event: response.completed
data: {"response_id":"r1"}

```

WebSocket 也可以使用相同应用消息模型：

```json
{
  "seq": 203,
  "type": "response.completed",
  "response_id": "r1"
}
```

传输可以不同，业务事件模型应保持清晰。

### 什么时候 SSE 更合适

- LLM 文本流；
- 后端任务进度；
- 通知流；
- 股票或监控看板的单向更新；
- Client 偶尔通过普通 HTTP 发控制请求；
- 希望尽量复用 HTTP 基础设施；
- 主要数据是文本或 JSON。

### 什么时候 WebSocket 更合适

- 实时协作编辑；
- 高频双向控制；
- 聊天双方持续主动发送；
- 在线状态和房间事件；
- 小型二进制消息；
- 一条长会话内有大量交错请求和响应；
- 已有成熟 WebSocket 网关和会话治理。

### 什么时候两者都不合适

- 只返回一个短结果：普通 HTTP 即可；
- 需要持久队列和离线消费：使用消息队列；
- 大文件交付：对象存储或文件下载；
- 超长后台任务：任务 API + 轮询/回调；
- 实时双向音视频：优先评估 WebRTC；
- 强事务命令：同步 API + 幂等与状态查询。

## 工程实践与边界

### 选型五问

1. 数据是单向还是持续双向？
2. 是文本/JSON 还是二进制？
3. 断线后要恢复哪些事件？
4. 现有网关和客户端支持什么？
5. 一条连接承载多少并发业务会话？

先回答这些问题，再选传输。

### SSE 上线检查单

- `Content-Type: text/event-stream`；
- 禁止不必要缓存；
- 关闭代理缓冲；
- 每个事件后及时 flush；
- 定义 `event`、`id` 和完成事件；
- 设置心跳；
- 设计 `Last-Event-ID` 恢复；
- 限制事件大小和队列；
- 区分连接断开与任务取消；
- 测试认证过期；
- 测试 HTTP/1.1、HTTP/2 和真实代理链路；
- 日志对 Prompt 与结果脱敏。

### WebSocket 上线检查单

- 使用 `wss`；
- 校验 `Origin`；
- 建连认证与授权；
- 定义消息版本和 Schema；
- 请求 ID、顺序号和确认语义；
- 最大帧/消息大小；
- 心跳与空闲超时；
- 指数退避重连；
- 恢复游标和去重；
- 慢消费者与队列上限；
- 连接排空和滚动发布；
- 每条消息重新检查必要权限。

### 断线恢复设计

统一思路：

```text
Server 为业务事件分配单调序号或唯一 ID
  -> Client 处理后记录最后位置
  -> 重连时提交最后位置
  -> Server 重放保留窗口内事件
  -> Client 按 ID 去重
  -> 超出窗口则获取完整快照
```

不要无限保存所有 token。

可以只恢复：

- 最终响应；
- 工具状态；
- 关键任务事件；
- 当前文本快照。

### 写操作与连接状态解耦

错误逻辑：

```text
连接断了 -> 再发一次退款
```

正确逻辑：

```text
连接断了
  -> 用业务幂等键查询退款状态
  -> 已完成：返回已有结果
  -> 处理中：继续等待
  -> 明确未创建：才允许重试
```

传输事件 ID 不应直接替代业务幂等键。

### 安全边界

两种长连接都要处理：

- 跨站连接与 Origin；
- 身份认证；
- 每条消息授权；
- 租户隔离；
- 消息大小与频率限制；
- JSON 解析和压缩炸弹；
- 敏感数据日志；
- 提示注入；
- 连接耗尽；
- DDoS 和慢连接。

连接已认证不代表权限永不变化。

长连接期间也要处理用户登出、角色变更和令牌撤销。

### 观测指标

共同指标：

- 活跃连接数；
- 建连成功率；
- 首事件延迟；
- 断线率；
- 重连次数；
- 恢复成功率；
- 每连接发送队列；
- 慢消费者数；
- 每消息大小；
- 任务完成率。

SSE 额外关注：

- 代理缓冲导致的事件批量到达；
- `Last-Event-ID` 命中率；
- 心跳间隔；
- HTTP 响应持续时间。

WebSocket 额外关注：

- 握手失败原因；
- Close code；
- Ping/Pong 延迟；
- `bufferedAmount` 或服务端队列水位；
- 每实例连接数；
- 发布排空时长。

### 测试场景

至少测试：

1. 正常持续流；
2. 首事件前断线；
3. 收到一半后断线；
4. 重连重复事件；
5. 恢复窗口过期；
6. Client 极慢；
7. Server 重启；
8. 代理空闲超时；
9. 凭证中途过期；
10. 用户主动取消；
11. 大消息和恶意消息；
12. 滚动发布与连接排空。

## 常见误区

### 误区一：WebSocket 功能更多，所以选择它更好

功能越多，连接和应用协议治理成本也越高。单向文本流通常 SSE 更简单。

### 误区二：SSE 完全不能双向交互

SSE 通道本身单向，但 Client 可以通过普通 HTTP POST 发送输入和控制。

### 误区三：SSE 只能用于 GET

`EventSource` 使用 GET；SSE 编码也可用于 POST 响应，由 Fetch 等客户端解析。

### 误区四：SSE 自动重连，所以不会丢消息

Server 必须保存可恢复事件并处理 `Last-Event-ID`，Client 还要去重。

### 误区五：WebSocket 保证消息永久可靠

它保证当前连接上的帧传输语义，不提供跨断线持久化、重放和业务确认。

### 误区六：WebSocket 天然比 HTTP 延迟低

减少重复请求开销可能有优势，但真实延迟取决于网络、代理、排队、应用处理和消息设计。

### 误区七：连接断开就是任务取消

任务可能继续运行。取消需要独立业务状态和传播机制。

### 误区八：二者选一个就不需要消息队列

长连接负责在线传输，消息队列负责持久化、削峰和异步消费，职责不同。

### 误区九：WebSocket 已鉴权后，每条消息无需授权

用户权限可能变化，不同消息也可能访问不同资源；服务端必须执行必要的逐消息授权。

## 面试追问

### 追问一：一句话区分 SSE 与 WebSocket

> SSE 是基于 HTTP 的服务端单向 UTF-8 事件流，WebSocket 是建立后可双向传输文本和二进制帧的持久连接协议。

### 追问二：LLM 文本流为什么常选 SSE

数据主要从 Server 单向返回，事件是文本或 JSON，而且 SSE 易复用 HTTP 请求、鉴权、网关和观测，不必为简单流维护完整双向协议。

### 追问三：SSE 如何恢复

Server 给事件分配 `id`，Client 保存最后处理位置，重连时用 `Last-Event-ID` 告知 Server；Server 从保留窗口重放，Client 按 ID 去重，过期时获取快照。

### 追问四：WebSocket 如何恢复

应用自行定义会话 ID、序号、确认和恢复游标；重连后重新认证并提交最后确认位置，Server 重放或返回快照。

### 追问五：二者如何处理背压

都需要应用限制队列和消息大小、合并低价值事件、暂停上游或断开慢消费者。WebSocket 可观察发送缓冲，但也不是自动业务流控。

### 追问六：SSE 可以传二进制吗

不能直接传原始二进制，只能编码成文本或发送资源链接。二进制为主时 WebSocket 或专用媒体/文件协议更合适。

### 追问七：为什么不把访问令牌放 SSE URL

URL 可能进入浏览器历史、代理日志和监控。应优先 Cookie、短期一次性票据或使用能发送 Header 的 Fetch 流方案。

### 追问八：WebSocket 的 Ping/Pong 是否等于业务健康

不是。它最多说明连接对端能响应控制帧，不证明模型任务、数据库或工具执行正常。还需要业务级状态和截止时间。

### 可直接背诵的完整答案

> SSE 使用 HTTP `text/event-stream` 从 Server 向 Client 持续发送 UTF-8 文本事件。浏览器 `EventSource` 是单向的，并内建重连、`retry` 与 `Last-Event-ID` 基础语义；Client 发数据时继续用普通 HTTP。WebSocket 建立持久双向连接，可传文本和二进制帧，也有 Ping/Pong 和 Close，但应用要自己定义消息 Schema、重连、恢复、确认和去重。
>
> 一次请求持续返回 LLM token、引用和工具状态时，SSE 通常简单且易复用 HTTP 基础设施；高频双向控制、协作状态和二进制消息更适合 WebSocket。SSE 的问题包括代理缓冲、空闲超时和浏览器请求头限制；WebSocket 的问题包括长连接治理、会话路由、部署排空、慢消费者和协议维护。两者都不保证业务恰好一次，连接断开也不等于任务取消；写操作必须使用幂等键和状态查询。
