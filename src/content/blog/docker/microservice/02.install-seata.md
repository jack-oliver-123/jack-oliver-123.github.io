## 适用场景

Seata 用于处理分布式事务，常见于多个服务或多个数据库之间需要保证一致性的场景。本文记录如何使用 Docker 快速启动 Seata Server。

## 准备工作

确认 Docker 可用：

```bash
docker --version
```

创建工作目录：

```bash
# Linux
mkdir -p /opt/seata/seata-data /opt/seata/logs

# Windows（PowerShell）
mkdir D:/seata-env/seata-data, D:/seata-env/logs
```

## 编写 docker-compose

新建 `docker-compose.yml`：

```yaml
services:
  seata-server:
    image: seataio/seata-server:1.8.0
    container_name: seata-local
    restart: always
    ports:
      - "8099:8099"
      - "7091:7091"
    volumes:
      - ./seata-data:/seata-server/sessionStore
      - ./logs:/root/logs/seata
    environment:
      - SEATA_IP=127.0.0.1
      - STORE_MODE=file
      - JAVA_OPTS=-Xms256m -Xmx256m
      - TZ=Asia/Shanghai
    deploy:
      resources:
        limits:
          memory: 512M
```

端口说明：

- `8099`：Seata Server RPC 通信端口；
- `7091`：Seata Web 控制台端口。

> `SEATA_IP=127.0.0.1` 适用于客户端运行在宿主机的场景。如果客户端也在 Docker 网络中，需要改为宿主机实际 IP 或使用自定义 Docker 网络 + 容器名通信。

## 启动 Seata

```bash
docker compose up -d
```

查看日志：

```bash
docker logs -f seata-local
```

## 登录控制台

浏览器访问：

```text
http://localhost:7091
```

默认账号密码：

```text
seata / seata
```

## 创建 undo_log 表

如果业务数据库使用 AT 模式，需要在每个参与分布式事务的业务库中创建 `undo_log` 表（MySQL 语法）：

```sql
CREATE TABLE `undo_log` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `branch_id` bigint(20) NOT NULL,
  `xid` varchar(100) NOT NULL,
  `context` varchar(128) NOT NULL,
  `rollback_info` longblob NOT NULL,
  `log_status` int(11) NOT NULL,
  `log_created` datetime NOT NULL,
  `log_modified` datetime NOT NULL,
  `ext` varchar(100) DEFAULT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `ux_undo_log` (`xid`,`branch_id`)
) ENGINE=InnoDB AUTO_INCREMENT=1 DEFAULT CHARSET=utf8;
```

> 如果使用 PostgreSQL，需要调整 DDL 语法（去掉 `ENGINE=InnoDB`，`bigint(20)` 改为 `bigint`）。

## 应用接入示例

Spring Cloud Alibaba 项目引入依赖：

```xml
<dependency>
  <groupId>com.alibaba.cloud</groupId>
  <artifactId>spring-cloud-starter-alibaba-seata</artifactId>
</dependency>
```

配置示例（`application.yml`）：

```yaml
seata:
  tx-service-group: app-tx-group
  service:
    vgroup-mapping:
      app-tx-group: default
    grouplist:
      default: 127.0.0.1:8099
```

> 注意：Seata 配置使用顶层 `seata:` 前缀，不要放在 `spring.cloud` 下面。

代码中在需要全局事务的方法上添加注解：

```java
@GlobalTransactional(name = "app-global-tx", rollbackFor = Exception.class)
public void executeBusiness() {
    // 跨服务或跨库的业务操作
}
```

## 常见问题

### 服务连接不上 Seata

检查：
- `SEATA_IP` 是否配置为客户端可访问的地址；
- `8099` 端口是否开放；
- 应用中的事务组名称是否和配置一致。

### 控制台能打开但应用报 no available service

确认应用配置中的 `tx-service-group` 与 `vgroup-mapping` 中的 key 完全匹配。

### undo_log 表不存在

AT 模式要求每个参与事务的业务数据库都有 `undo_log` 表，否则会报错。

## 停止和清理

```bash
docker compose down
```

## 后续建议

- 本地开发可以使用 `file` 模式快速验证，生产环境建议改为 `db` 模式或接入 Nacos 做注册中心。
- `file` 模式仅适合单机开发，事务日志保存在本地文件中。
- 生产部署请参考 Seata 官方的 [版本兼容矩阵](https://github.com/seata/seata/wiki)，确认与 Spring Cloud Alibaba 版本匹配。
