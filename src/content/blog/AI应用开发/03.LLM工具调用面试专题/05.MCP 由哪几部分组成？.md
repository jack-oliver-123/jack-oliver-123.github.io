第一次看一辆汽车，零基础学习者可能只看到“一个能开的整体”。
机械师却会分层观察：
- 谁在驾驶；
- 发动机怎样提供能力；
- 控制信号怎样传递；
- 燃油、电力和制动系统分别做什么；
- 故障时怎样告警和停止。
学习 MCP 也不能只记“Client 和 Server”。
如果只记两个角色，面试官再问 Tools、Resources、Prompts、生命周期和传输，你就会觉得它们是互不相关的名词。
更好的方法是从四个维度拆：
1. 架构角色；
2. 数据层；
3. 传输层；
4. 能力原语。

## 本篇目标
读完本篇，你应该能够：
- 从四个维度完整拆解 MCP；
- 区分 Host、Client、Server 的责任；
- 解释请求、响应、通知和错误；
- 说明初始化、版本和能力协商的位置；
- 对比 stdio 与 Streamable HTTP；
- 区分 Server 侧 Tools、Resources、Prompts；
- 认识 Client 侧 Sampling、Elicitation、Roots；
- 了解进度、取消、日志等辅助能力；
- 说明哪些组件是必需概念，哪些能力需要协商；
- 画出一条从模型选择到业务执行的完整链路。

## 60 秒回答
MCP 可以从四个维度理解。
第一是架构角色：Host 承载模型、用户和安全策略；Host 内的 MCP Client 与某个 MCP Server 维护协议会话；Server 提供外部上下文或操作能力。
第二是数据层：基于 JSON-RPC 2.0 的请求、响应和通知，并包含初始化、协议版本、能力协商、取消、进度和日志等语义。
第三是传输层：2025-11-25 规范的标准传输包括本地常用的 stdio 和远程常用的 Streamable HTTP。
第四是能力原语：Server 侧主要有 Tools、Resources、Prompts；Client 侧可协商 Sampling、Elicitation、Roots。
这些部分不是平铺的零件。
Host 通过 Client 建立连接，完成初始化和能力协商，再通过传输发送数据层消息，调用已协商的原语。
最后仍由 Host 和 Server 共同落实授权、安全、超时、审计与业务规则。

## 详细解析

### 1. 一张总览表

| 维度 | 组成 | 回答的问题 |
| --- | --- | --- |
| 角色 | Host、Client、Server | 谁与谁协作 |
| 数据层 | JSON-RPC、生命周期、能力协商、辅助消息 | 消息表达什么 |
| 传输层 | stdio、Streamable HTTP | 消息怎样移动 |
| 原语 | Tools、Resources、Prompts、Sampling、Elicitation、Roots | 双方能提供什么 |
可以把一次 MCP 连接记成：
~~~text
角色建立关系
    ↓
传输打通消息通道
    ↓
数据层完成初始化与协商
    ↓
双方使用已协商原语
    ↓
Host 应用安全与产品策略
~~~

### 2. 角色一：Host
Host 是用户真正使用的应用。
例如：
- 桌面 AI 助手；
- IDE；
- 企业知识助手；
- 客服 Agent 平台；
- 自动化工作台。
Host 不是一个被动转发器。
它承担产品级职责：
- 选择模型；
- 管理对话；
- 维护用户身份；
- 决定连接哪些 Server；
- 为每个 Server 创建 Client；
- 决定向模型暴露哪些能力；
- 展示用户确认；
- 控制敏感上下文；
- 记录审计；
- 中断危险调用。
Host 是 MCP 系统中最重要的信任边界。

### 3. 角色二：MCP Client
Client 位于 Host 内。
它面向一个 Server 维护协议会话。
Client 的职责包括：
- 建立传输；
- 发送 initialize；
- 声明支持的能力；
- 接收协商后的版本；
- 发送请求；
- 匹配响应；
- 接收通知；
- 维护会话状态；
- 关闭连接。
一个 Host 可以有多个 Client：
~~~text
Host
 ├─ Client A ↔ 文件 Server
 ├─ Client B ↔ Git Server
 └─ Client C ↔ 工单 Server
~~~
这样能隔离每个 Server 的能力、认证和故障。

### 4. 角色三：MCP Server
Server 向 Client 暴露能力。
Server 可能是：
- 本地 Node.js/Python 进程；
- 企业内部服务；
- SaaS 提供的远程服务；
- 现有 REST API 的适配层；
- 数据库或文件系统的受控代理。
Server 负责：
- 声明自身能力；
- 列出与执行 Tools；
- 列出与读取 Resources；
- 列出与获取 Prompts；
- 返回结构化结果或错误；
- 在支持时发送列表变化通知；
- 遵守协议生命周期。
Server 仍要执行服务端授权。
不能把安全责任全推给 Host。

### 5. 数据层基础：JSON-RPC 2.0
MCP 使用 JSON-RPC 2.0 消息结构。
JSON-RPC 提供三种核心消息。
#### Request
请求需要对方处理并返回响应。
概念示例：
~~~json
{
  "jsonrpc": "2.0",
  "id": 17,
  "method": "tools/list",
  "params": {}
}
~~~
#### Response
响应使用同一个 ID 对应请求。
成功时有结果：
~~~json
{
  "jsonrpc": "2.0",
  "id": 17,
  "result": {
    "tools": []
  }
}
~~~
失败时返回协议错误对象。
#### Notification
通知没有请求 ID，也不期待响应。
它适合表达状态变化或事件。
例如初始化完成通知。

### 6. 请求 ID 与工具调用 ID 不要混淆
系统里可能同时出现：
- 模型 API 的响应 ID；
- 模型生成的 tool call ID；
- MCP JSON-RPC request ID；
- 业务系统的 request ID；
- 幂等键；
- trace ID。
它们不是同一个东西。

| 标识 | 作用 |
| --- | --- |
| trace ID | 串起整条链路 |
| 模型 tool call ID | 关联模型调用意图与工具结果 |
| MCP request ID | 关联 MCP 请求与响应 |
| 幂等键 | 防止业务写操作重复 |
| 业务 request ID | 业务服务自身追踪 |
工程上可以建立映射，但不能随意互换。

### 7. 生命周期
生命周期描述连接从建立到结束的过程。
最关键的是初始化阶段。
~~~text
Client → initialize
Server → initialize result
Client → notifications/initialized
~~~
initialize 交换：
- 协议版本；
- Client 能力；
- Server 能力；
- 实现名称与版本等信息。
初始化完成前，不应执行普通操作。
关闭阶段要根据传输和实现正确清理。
本地子进程尤其要避免残留。

### 8. 协议版本
MCP 使用日期形式的协议版本。
连接时双方要协商一个可用版本。
不能简单假设：
- 最新 Client 会自然兼容旧 Server；
- SDK 升级不会改变行为；
- Server 声明的所有字段都被 Client 理解。
Host 应记录：
- Client 支持的版本；
- Server 返回的版本；
- 实际协商版本；
- SDK 版本；
- 能力集合。
这些信息对于排查兼容问题很重要。

### 9. 能力协商
能力协商让双方声明可选功能。
例如 Server 可以声明：
- Tools；
- Resources；
- Prompts；
- 相关列表变化通知能力。
Client 可以声明：
- Sampling；
- Elicitation；
- Roots。
只使用协商成功的能力。
如果对方没有声明，不能因为某个 SDK 恰好有方法就直接调用。

### 10. 传输一：stdio
stdio 常用于本地 Server。
关系通常是：
~~~text
Host 启动子进程
Client 写 stdin
Server 读 stdin
Server 写 stdout
Client 读 stdout
~~~
优点：
- 本地接入直接；
- 不需要开放网络端口；
- 与命令行工具容易集成；
- 生命周期可由 Host 管理。
边界：
- 进程继承环境权限；
- stdout 不能混普通日志；
- 崩溃会中断连接；
- 需要管理启动和退出；
- 不天然等于沙箱。

### 11. 传输二：Streamable HTTP
Streamable HTTP 面向 HTTP 部署。
它通过 MCP 端点处理消息。
在需要时可利用 SSE 承载服务器到客户端的流式消息。
优点：
- 适合远程服务；
- 可利用 HTTP 基础设施；
- 便于统一网关、TLS 和监控；
- 可支持流式交互。
边界：
- 需要认证和授权；
- 存在网络超时与重试；
- 需要会话和负载均衡设计；
- 必须防请求伪造与来源风险；
- 远程失败语义更复杂。

### 12. Server 原语：Tools
Tools 面向可执行操作。
典型请求语义包括：
- 列出工具；
- 调用工具。
一项 Tool 通常有：
- name；
- description；
- inputSchema；
- 可选 outputSchema；
- annotations 或其他元数据；
- 调用结果。
Tool 可以只读，也可以有副作用。
协议中的提示性元数据不能替代真实安全策略。
Host 不能只因为 Tool 标为只读就无条件信任。

### 13. Tool 的执行链
模型驱动时：
~~~text
Host 从 Server 获取 Tools
        ↓
Host 根据权限筛选
        ↓
Host 转成模型工具格式
        ↓
模型选择 Tool 并给参数
        ↓
Host 再次校验
        ↓
MCP Client 发送 tools/call
        ↓
Server 执行真实逻辑
        ↓
结果返回 Host
        ↓
Host 交回模型
~~~
这里至少发生了两次格式转换：
1. MCP Tool 定义到模型供应商格式；
2. MCP Tool 结果到模型工具结果消息。
适配层要保留调用关联。

### 14. Server 原语：Resources
Resources 面向可寻址上下文。
核心操作是发现和读取。
Resource 使用 URI。
它可以表示：
- 文本文件；
- 数据库 Schema；
- 项目配置；
- 日志快照；
- 业务文档；
- 二进制内容。
Resources 还可以有模板，用参数化 URI 描述一类资源。
部分实现可支持订阅变化。
是否可用取决于能力协商。

### 15. Server 原语：Prompts
Prompts 是可发现、可参数化的消息模板。
它们适合把 Server 所属领域的操作方法提供给用户或 Host。
例如：
~~~text
review_pull_request(repository, pull_request)
~~~
Host 获取模板后，可以：
- 显示给用户选择；
- 填入参数；
- 合并到会话；
- 根据自身规则调整。
Prompt 内容仍是不可信外部内容。
它不能覆盖 Host 的更高优先级安全策略。

### 16. Client 能力：Sampling
Sampling 是 Server 到 Client 的反向请求。
Server 可以请求 Host 侧模型完成生成。
这样 Server 无需持有模型密钥。
Host 可以：
- 检查请求；
- 选择模型；
- 限制上下文；
- 请求用户同意；
- 过滤结果；
- 拒绝请求。
Sampling 不是 Server 取得 Host 模型的无限使用权。

### 17. Client 能力：Elicitation
Elicitation 用于请求用户补充信息。
例如 Server 执行差旅申请时发现缺少成本中心。
它可以经 Client 请求用户输入。
Host 决定：
- 怎样展示；
- 哪些字段允许收集；
- 是否包含敏感信息；
- 用户能否取消；
- 返回什么内容。

### 18. Client 能力：Roots
Roots 让 Client 声明相关文件系统根。
用途包括：
- 告诉 Server 当前工作区；
- 限定用户期望的项目范围；
- 在根目录变化时通知 Server。
但 Roots 是协议层信息。
强制隔离需要：
- 文件权限；
- 容器；
- 沙箱；
- 进程身份；
- Host 的路径校验。

### 19. 辅助能力：取消
长时间操作应支持取消语义。
取消可以来自：
- 用户主动停止；
- Host 总超时；
- 上游会话结束；
- 任务已经不再需要。
收到取消后，Server 应尽快停止可停止的工作。
已经发生的副作用不一定能自动撤回。
取消与补偿是两件事。

### 20. 辅助能力：进度
长任务可以报告进度。
进度有助于：
- 给用户反馈；
- 监控卡住的请求；
- 调整超时；
- 诊断性能。
进度值应保持可解释。
不要用“99%”长期停住。

### 21. 辅助能力：日志
协议可支持日志消息。
日志要区分：
- 面向开发者的诊断；
- 面向用户的状态；
- 安全审计；
- 协议消息。
stdio 模式尤其不能把普通日志写到 stdout。
否则 Client 可能把日志当 JSON-RPC 消息解析。

### 22. 辅助能力：Ping
Ping 可用于确认对端是否仍响应。
但 Ping 成功不代表业务依赖健康。
Server 进程活着，数据库可能已经不可用。
生产健康检查还要覆盖真实依赖。

### 23. Experimental 能力
规范可能包含实验性能力。
实验性意味着：
- 接口可能变化；
- SDK 支持不一致；
- 不应默认所有实现可用；
- 生产采用前需明确兼容策略。
不要把实验性能力说成所有 MCP 系统的稳定必备组成。

### 24. 怎样回答“由哪几部分组成”
不要只报一串名词。
推荐按这个顺序：
1. 先说三角色；
2. 再说 JSON-RPC 数据层；
3. 再说标准传输；
4. 再说 Server 与 Client 原语；
5. 最后补生命周期、协商和安全边界。
这样有结构，也能自然应对追问。

## 工程实践与边界

### 1. 为每个组件设置责任人
生产系统中应明确：

| 组件 | 责任团队 |
| --- | --- |
| Host | AI 应用团队 |
| Client 适配 | 平台或应用团队 |
| MCP Server | 能力所属业务团队 |
| 真实 API | 业务系统团队 |
| 身份与授权 | 安全/身份平台 |
| 可观测性 | 平台团队 |
否则故障容易互相推诿。

### 2. 对连接建立状态机
例如：
~~~text
DISCONNECTED
  ↓
CONNECTING
  ↓
INITIALIZING
  ↓
READY
  ↓
CLOSING
  ↓
CLOSED
~~~
异常状态可以有：
- AUTH_FAILED；
- VERSION_MISMATCH；
- SERVER_CRASHED；
- TIMEOUT；
- PROTOCOL_ERROR。
只有 READY 状态允许普通调用。

### 3. 工具目录要经过 Host 筛选
不要直接把 tools/list 返回的全部内容传给模型。
先检查：
- Server 是否可信；
- 工具是否允许；
- 当前用户是否有权；
- 工具版本是否批准；
- 风险等级；
- 是否健康；
- 是否与任务相关。

### 4. 结果要做统一封装
内部可以使用稳定结果结构：
~~~json
{
  "status": "success",
  "data": {},
  "error": null,
  "metadata": {
    "server": "travel",
    "toolVersion": "2"
  }
}
~~~
再适配到模型需要的内容。
不要把供应商字段渗透到所有业务代码。

### 5. 对失败进行分层

| 层 | 例子 | 处理者 |
| --- | --- | --- |
| 传输 | 连接断开 | Client |
| 协议 | 非法 JSON-RPC | Client/Server |
| 能力 | 方法未协商 | 协议层 |
| 授权 | 用户无权 | Host/Server |
| 业务 | 航班售罄 | Tool |
| 模型 | 选错工具 | Host/模型策略 |
“MCP 调用失败”太笼统。
必须先定位层级。

### 6. 防止跨 Server 数据泄露
Host 管理多个 Server 时，不应默认把 A Server 的敏感结果传给 B Server。
需要上下文分区和数据流策略。
例如工资 Server 的结果不能自动进入公开搜索工具参数。

### 7. 统一超时预算
总请求 10 秒时，不能给模型、MCP 和业务 API 各自 10 秒。
应拆分预算：
~~~text
模型决策 2 秒
MCP 连接与请求 1 秒
业务工具 4 秒
模型总结 2 秒
缓冲 1 秒
~~~
实际数值按业务实测。

### 8. 做契约测试
Server 更新后，自动验证：
- initialize；
- 版本协商；
- tools/list；
- Schema；
- 合法调用；
- 非法参数；
- 权限拒绝；
- 超时与取消；
- 列表变化；
- 关闭清理。

### 9. 不把协议注解当安全事实
Tool 元数据可以帮助 Host 做展示和初步策略。
但 Server 可能配置错误或恶意声明。
最终授权要基于受信策略和实际操作。

### 10. 逐层做可观测性
分别统计：
- 连接成功率；
- 初始化耗时；
- 版本不匹配率；
- 列表刷新次数；
- tools/call 延迟；
- 协议错误；
- 业务错误；
- 用户拒绝确认率；
- Server 崩溃率；
- 结果大小。

## 常见误区

### 误区一：MCP 只有 Client 和 Server
不完整。
Host 承担用户、模型和安全策略。

### 误区二：Tools 是 MCP 的全部
错误。
还有 Resources、Prompts 及 Client 侧能力。

### 误区三：传输层就是 MCP 的全部协议
错误。
stdio 或 HTTP 只负责搬运消息。

### 误区四：支持 JSON-RPC 就支持 MCP
错误。
还必须实现 MCP 的生命周期、方法和能力语义。

### 误区五：协商了 Tools 就可以执行所有工具
错误。
还要做业务授权和 Host 策略。

### 误区六：Roots 是操作系统沙箱
错误。
它是协议声明，不自动形成强制隔离。

### 误区七：取消请求会自动回滚副作用
错误。
已经创建的订单仍可能需要补偿。

### 误区八：把本地 stdio 当作天然安全边界
错误。
子进程仍可能读取文件、环境变量或主动联网。

## 面试追问

### 追问一：为什么 Host 通常为每个 Server 建一个 Client
这样可以隔离会话、版本、能力、认证和故障。
Host 再统一聚合多个连接。

### 追问二：Tools、Resources、Prompts 怎样快速区分
Tool 面向动作。
Resource 面向 URI 可寻址上下文。
Prompt 面向可复用消息模板。

### 追问三：Sampling 为什么放在 Client 侧能力
因为它是 Server 反向请求 Host/Client 侧模型生成。
模型凭据和策略仍由 Host 控制。

### 追问四：能力协商有什么价值
允许实现只支持需要的可选能力，并在运行前明确双方共同能力。

### 追问五：Request 和 Notification 有什么区别
Request 有 ID，需要响应。
Notification 不期待响应。

### 追问六：stdio 模式日志写哪里
普通日志应写 stderr。
stdout 保留给协议消息。

### 追问七：MCP request ID 能直接作为幂等键吗
不应默认这样做。
请求 ID 主要用于协议关联。
业务幂等键需要跨重试稳定，并满足业务去重语义。

### 可背诵完整版
> MCP 可以从四个维度拆解。角色层有 Host、Client、Server：Host 管用户、模型和策略，Client 管一条协议会话，Server 提供能力。数据层基于 JSON-RPC，包含请求、响应、通知、生命周期、版本和能力协商。传输层的标准方式有 stdio 与 Streamable HTTP。能力层中，Server 侧主要是 Tools、Resources、Prompts，Client 侧可协商 Sampling、Elicitation、Roots，另外还有取消、进度和日志等辅助语义。运行时先建立传输并初始化，再使用协商成功的能力。协议组件不替代业务授权、安全隔离、幂等和审计。
