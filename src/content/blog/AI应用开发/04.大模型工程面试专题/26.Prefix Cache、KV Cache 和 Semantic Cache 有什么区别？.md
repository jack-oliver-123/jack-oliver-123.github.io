用户连续问模型两句话：

```text
第一轮：请阅读这份 50 页合同，总结付款条款。
第二轮：其中逾期付款怎么处理？
```

第二轮如果把整份合同再次从头计算，显然很浪费。

另一个用户问：

```text
“公司报销出租车的上限是多少？”
```

系统昨天已经回答过一个意思几乎相同的问题：

```text
“打车费最多能报多少钱？”
```

这时似乎也可以复用结果。

但这两种“复用”并不是同一种缓存：

- 前者可能复用模型中间计算；
- 后者可能直接复用最终答案。

理解 KV Cache、Prefix Cache 和 Semantic Cache 的边界，是做大模型推理优化的基础。

## 本篇目标

学完本篇，你应该能够：

1. 解释注意力里的 Key、Value 为什么可以缓存；
2. 说明单次请求 KV Cache 如何避免重复计算历史 Token；
3. 说明 Prefix Cache 如何在多个请求间复用相同前缀；
4. 说明 Semantic Cache 为什么复用的是近似问题的结果；
5. 区分精确 Token 命中和向量相似命中；
6. 用公式估算 KV Cache 显存量；
7. 设计包含模型、Tokenizer、租户和权限的缓存键；
8. 处理淘汰、失效、版本升级和多租户隔离；
9. 判断哪些场景不应该使用语义答案缓存；
10. 在面试中完整比较三类缓存的收益和风险。

先记住一句话：

> KV Cache 复用同一生成过程中的历史注意力状态；Prefix Cache 把可复用的前缀 KV 跨请求保存；Semantic Cache 根据语义相似度复用结果。前两者主要减少模型计算，后者可能直接跳过模型，因此正确性和安全风险更高。

## 60 秒回答

KV Cache 是自回归生成的基础优化。Transformer 每生成一个新 Token，都需要让新 Query 关注历史 Key 和 Value。历史 Token 的 K/V 在参数和前缀不变时无需重复计算，所以按层保存，之后每步只计算新 Token 的 K/V。它主要服务当前序列，会随上下文增长并占用大量显存。

Prefix Cache，也叫 Prompt Cache 或 Automatic Prefix Caching，是把某段已经算好的前缀 KV 跨请求复用。命中要求 Token 序列和影响计算的模型配置一致，通常按 Token block 的哈希索引。它能减少共享系统提示词、公共文档和多轮前缀的 Prefill，但不会替用户生成新的 Decode Token。

Semantic Cache 则先把新问题编码成向量，在历史问题中找相似项，超过阈值时直接返回或改写缓存答案。它允许文字不同但意思相近，因此命中率可能更高，但“语义相似”不代表答案仍适用。用户身份、时间、权限、个性化和数据版本都可能让相似问题需要不同答案。

工程上我会把三者分层管理：KV/Prefix Cache 的键包含模型、权重、Tokenizer、适配器、Token ID 和租户隔离信息；Semantic Cache 还要包含知识版本、权限、语言、工具状态和有效期。上线时同时监控命中率、节省的 Prefill Token、显存占用、错误命中、陈旧答案和跨租户泄露，并为高风险或强时效任务禁用结果复用。

## 详细解析

### 一、先回顾注意力中的 Q、K、V

单头缩放点积注意力常写为：

$$
\operatorname{Attention}(Q,K,V)
=\operatorname{softmax}\left(\frac{QK^\top}{\sqrt{d_k}}+M\right)V
$$

逐个符号解释：

- $Q$：Query，表示当前位置要查找什么；
- $K$：Key，表示各位置可以按什么特征被匹配；
- $V$：Value，表示匹配后要取回的内容表示；
- $K^\top$：Key 矩阵转置；
- $d_k$：Key 向量维度；
- $\sqrt{d_k}$：缩放项，避免点积过大；
- $M$：因果掩码，让当前位置看不到未来 Token；
- $\operatorname{softmax}$：把分数转成权重；
- 最后乘 $V$：按注意力权重汇总 Value。

每层会从隐藏状态计算：

$$
Q=XW_Q,\quad K=XW_K,\quad V=XW_V
$$

其中：

- $X$：这一层输入的隐藏状态；
- $W_Q,W_K,W_V$：训练得到的投影矩阵。

### 二、没有 KV Cache 会重复做什么

假设已经生成：

```text
我 喜欢 学习
```

现在要生成下一个 Token。

如果没有缓存，模型可能重新对“我、喜欢、学习”计算每一层的 Key 和 Value。

生成下一个 Token 后，又会对更长前缀重新计算。

历史 Token 在每一步都不变，这部分计算是重复的。

### 三、KV Cache 如何工作

第一次处理 Prompt 时，模型计算每层所有输入 Token 的 K/V 并保存。

生成新 Token 时：

1. 只为新 Token 计算新的 Q/K/V；
2. 把新 K/V 追加到缓存；
3. 新 Query 与缓存中的全部历史 Key 做注意力；
4. 使用历史 Value 汇总结果；
5. 继续下一步。

```text
第 1 步缓存：[K1,V1] ... [Kn,Vn]
第 2 步追加：[Kn+1,Vn+1]
第 3 步追加：[Kn+2,Vn+2]
```

它避免重复计算历史 K/V，但新 Token 仍需关注越来越长的历史。

因此 KV Cache 降低了重复计算，却不会让长上下文注意力成本消失。

### 四、KV Cache 的生命周期

最常见的 KV Cache 生命周期是单个活跃序列：

```text
请求进入
  -> Prefill 创建 KV
  -> Decode 持续追加
  -> 请求结束
  -> 释放或转入可复用缓存
```

多轮对话如果服务端保留会话状态，也可能在后续请求继续使用同一前缀 KV。

但只要 Prompt 中间发生变化，变化位置之后的 KV 通常需要重新计算。

### 五、KV Cache 为什么很占显存

一个简化估算公式是：

$$
M_{KV}\approx
2\times L\times B\times S\times H_{kv}\times D_h\times b
$$

其中：

- $M_{KV}$：KV Cache 字节数；
- 前面的 $2$：Key 和 Value 各一份；
- $L$：Transformer 层数；
- $B$：批量中的序列数；
- $S$：每个序列缓存的 Token 数；
- $H_{kv}$：KV Head 数量；
- $D_h$：每个 Head 的维度；
- $b$：每个元素的字节数，例如 FP16/BF16 常为 2 字节。

假设：

```text
层数 L = 32
批量 B = 1
上下文 S = 8192
KV Head H_kv = 8
Head 维度 D_h = 128
每元素 b = 2 字节
```

则：

$$
M_{KV}\approx2\times32\times1\times8192\times8\times128\times2
$$

约为 1 GiB。

这只是简化的原始张量估算。

实际框架还会有 block 元数据、对齐、碎片、临时张量和并行分片差异。

### 六、MHA、GQA 和 MQA 对 KV 大小的影响

在多头注意力 MHA 中，Query Head 和 KV Head 数量通常相同。

GQA 让多组 Query Head 共享较少的 KV Head。

MQA 则让所有 Query Head 共享一组 K/V。

在其他条件相同时，减少 $H_{kv}$ 会按比例减少 KV Cache。

这也是 GQA/MQA 对推理服务很重要的原因之一。

但它们是模型架构选择，不是部署时随意把已有 MHA 模型的 KV Head 删除。

### 七、什么是 Prefix Cache

Prefix Cache 把某个已计算前缀的 KV 状态保存下来，让后续请求复用。

假设很多请求共享：

```text
[系统提示词 2000 Token]
[公司政策 10000 Token]
[每位用户自己的问题]
```

第一次请求计算前 12,000 Token 的 KV。

后续请求如果前缀 Token 完全相同，可以从缓存接着处理用户问题，而不重新 Prefill 公共部分。

```text
共享前缀 KV（命中） + 新后缀（计算） -> Decode
```

### 八、Prefix Cache 为什么常按 Block 管理

如果必须整段 Prompt 一字不差才命中，复用粒度会很粗。

PagedAttention 风格的引擎通常把 KV 分成固定大小的 Token block。

例如每块 16 Token：

```text
Block 1: token 1-16
Block 2: token 17-32
Block 3: token 33-48
```

缓存键可以递归包含：

```text
hash(父 block hash + 当前 block token IDs + 额外上下文)
```

这样两个请求共享前 32 Token、从第 33 Token 开始不同，仍可命中前两个完整 block。

最后一个不完整 block 是否缓存，取决于具体引擎实现。

### 九、Prefix Cache 的“相同”到底多严格

对模型计算而言，字符串看起来相同还不够。

通常需要影响隐藏状态的条件一致：

- 模型权重版本；
- Tokenizer 版本；
- Token ID 序列；
- 位置编码和位置索引；
- LoRA 或其他 Adapter；
- 多模态输入哈希；
- 推理精度和部分引擎配置；
- 租户或缓存隔离域。

例如字符串空格、模板换行或消息角色变化，都可能改变 Token ID。

不同 LoRA Adapter 即使 Prompt 相同，K/V 也可能不同。

错误复用会让后续计算建立在不属于当前模型状态的 K/V 上。

### 十、Prefix Cache 节省什么、不节省什么

它节省的是命中前缀的 Prefill 计算。

它不直接节省：

- 新后缀的 Prefill；
- 新答案的 Decode；
- 新输出 Token；
- 不同前缀部分；
- 网络和业务工具时间。

端到端收益可以粗略看成：

$$
\Delta T\approx T_{prefill,hit}-T_{lookup}-T_{load}
$$

其中：

- $T_{prefill,hit}$：原本需要计算的命中前缀 Prefill 时间；
- $T_{lookup}$：查找缓存时间；
- $T_{load}$：把缓存块准备到可用设备的时间。

如果前缀很短，查找和管理开销可能接近节省的计算。

如果缓存放在远端存储，加载网络成本也要计算。

### 十一、哪些场景适合 Prefix Cache

典型场景：

- 大量请求共享长系统提示词；
- 同一份长文档被连续问答；
- 多轮对话保留完整历史；
- Few-shot 示例固定；
- Agent 多条分支共享同一前缀；
- 批量评测使用相同说明和示例。

不太有利的场景：

- 每个 Prompt 都完全不同；
- 共享前缀很短；
- 模板在开头插入随机时间戳；
- 每个用户使用不同 Adapter；
- 缓存容量太小且请求复用间隔很长。

### 十二、Prompt 设计会影响 Prefix 命中

假设模板是：

```text
当前时间：每次变化
固定系统规则：很长
固定示例：很长
用户问题：变化
```

时间戳放在最前面，会让后面所有 Token 的前缀都不同。

更有利的顺序通常是：

```text
固定系统规则
固定示例
变化较慢的上下文
动态时间和用户问题
```

但不能为了缓存命中而改变语义、权限或消息优先级。

模板设计首先要正确，其次才是可复用性。

### 十三、什么是 Semantic Cache

Semantic Cache 不要求 Token 一样。

它通常把问题编码为向量：

$$
e=f_{embed}(x)
$$

其中：

- $x$：当前问题；
- $f_{embed}$：嵌入模型；
- $e$：语义向量。

再与历史问题向量比较余弦相似度：

$$
\operatorname{sim}(e_q,e_c)=
\frac{e_q\cdot e_c}{\|e_q\|\|e_c\|}
$$

其中：

- $e_q$：新问题向量；
- $e_c$：缓存问题向量；
- $e_q\cdot e_c$：点积；
- $\|e\|$：向量长度；
- 相似度越接近 1，通常表示方向越接近。

如果最高相似度超过阈值，系统可能直接返回缓存答案。

### 十四、语义相似为什么不等于可复用

下面两句话很相似：

```text
“我的余额是多少？”
“他的余额是多少？”
```

但答案不能复用。

下面两句话也很相似：

```text
“今天美元汇率是多少？”
“昨天美元汇率是多少？”
```

时间不同，答案也不同。

再看：

```text
“我能退款吗？”
```

不同用户、订单、地区和商品可能有不同结论。

所以 Semantic Cache 的键不能只有问题向量。

### 十五、Semantic Cache 的复合键

可以把候选限制在同一个命名空间：

```json
{
  "tenant_id": "tenant_42",
  "user_scope": "public-policy-only",
  "language": "zh-CN",
  "model_version": "model-v8",
  "prompt_version": "support-v12",
  "knowledge_version": "policy-2026-07",
  "tool_state_version": "pricing-188",
  "risk_class": "low",
  "semantic_vector": "..."
}
```

先按确定性字段过滤，再做向量相似检索。

不能先跨所有用户检索答案，再期待模型自行删除敏感内容。

### 十六、相似度阈值怎么定

阈值太低：

- 命中率高；
- 错误命中也多；
- 不相关答案可能被直接返回。

阈值太高：

- 错误命中少；
- 但很多可复用问题无法命中；
- 省下的成本有限。

需要构造配对数据：

| 新问题 | 缓存问题 | 是否允许复用 |
| --- | --- | --- |
| 同义改写 | 同一公开政策 | 是 |
| 时间不同 | 实时价格 | 否 |
| 用户不同 | 个人余额 | 否 |
| 语言不同 | 同一静态 FAQ | 视翻译策略而定 |
| 条件少一个 | 资格判断 | 否或需要澄清 |

对不同业务类别分别画 Precision—Recall 曲线。

高风险任务应更重视错误命中的代价，甚至关闭结果缓存。

### 十七、三类缓存放在一张表里

| 维度 | KV Cache | Prefix Cache | Semantic Cache |
| --- | --- | --- | --- |
| 复用对象 | 历史 K/V 张量 | 跨请求前缀 K/V block | 过去的答案或中间结果 |
| 匹配方式 | 当前序列位置 | 精确 Token 前缀和配置 | 语义相似 + 业务条件 |
| 生命周期 | 通常是一次活跃生成 | 跨请求、可持久或分层 | 跨请求、按 TTL/版本失效 |
| 主要收益 | 避免重复算历史 K/V | 减少共享前缀 Prefill | 可能跳过模型调用 |
| 主要资源 | GPU/CPU KV 内存 | GPU/CPU/远端 KV 存储 | 向量索引、答案存储 |
| 正确性风险 | 状态管理错误 | 错版本、错租户 KV 复用 | 近似误命中、答案陈旧 |
| 是否生成新答案 | 是 | 是 | 命中时可能不生成 |

### 十八、缓存命中率不是唯一指标

普通命中率：

$$
H=\frac{N_{hit}}{N_{request}}
$$

其中：

- $H$：请求命中率；
- $N_{hit}$：命中请求数；
- $N_{request}$：总请求数。

但两个 Prefix Cache 都是一次命中：

- 请求 A 命中 16 Token；
- 请求 B 命中 20,000 Token。

价值显然不同。

还应记录 Token 命中率：

$$
H_{token}=\frac{\sum T_{cached}}{\sum T_{prefill}}
$$

以及：

- 节省的 Prefill 时间；
- 每次命中加载开销；
- 显存占用；
- 淘汰率；
- Semantic Cache 错误命中率；
- 陈旧答案率；
- 跨租户安全事件。

### 十九、缓存淘汰策略

容量有限时，需要选择淘汰项。

常见策略：

- LRU：优先淘汰最久未使用；
- LFU：优先淘汰使用频率最低；
- TTL：超过有效期失效；
- Size-aware：考虑对象大小；
- Cost-aware：优先保留重算昂贵的前缀；
- Tenant quota：每租户设置配额；
- Priority：关键公共前缀优先保留。

对 Prefix Cache，价值可以粗略估计为：

$$
V_i\approx P_i\times C_i-S_i
$$

其中：

- $V_i$：缓存项价值；
- $P_i$：未来再次命中的概率；
- $C_i$：命中时节省的计算成本；
- $S_i$：存储、加载和管理成本。

精确预测并不容易，但这个公式提醒我们：大对象不一定最值得保留，高频小前缀也可能更有价值。

### 二十、分层 KV 存储

GPU 显存最快但最昂贵。

一些系统把 KV 分层：

```text
GPU HBM：热 KV
  -> CPU 内存：温 KV
  -> 本地 SSD / 远端存储：冷 KV
```

命中冷缓存时，需要搬运回 GPU。

如果传输时间超过重新 Prefill，缓存就失去意义。

因此分层策略要根据：

- 前缀长度；
- 网络带宽；
- 重算成本；
- 未来复用概率；
- SLO；

决定保留和加载。

### 二十一、多租户安全是第一等约束

共享前缀可能含有：

- 系统 Prompt；
- 私有文档；
- 用户对话；
- Adapter 产生的特定表示；
- 图片和音频特征。

缓存索引和存储都需要租户隔离。

哈希不是授权机制。

即使攻击者不知道原文，也不能让其通过时序、命中状态或可预测哈希探测其他租户内容。

可采用：

- 租户命名空间；
- 带盐哈希；
- 访问控制；
- 加密传输与静态加密；
- 配额；
- 审计；
- 敏感前缀禁用跨请求共享。

## 工程实践与边界

### 一、先规范化“影响结果的版本”

缓存键至少考虑：

```text
模型权重
+ Tokenizer
+ Adapter
+ Prompt Token IDs
+ 位置与多模态输入
+ 引擎兼容版本
+ 租户隔离域
```

Semantic Cache 再加：

```text
知识版本
+ 权限
+ 用户或会话范围
+ 时间窗口
+ 工具状态
+ 输出策略
```

### 二、缓存失效比写入更难

以下变化可能要求失效：

- 模型升级；
- Prompt 改版；
- Adapter 更新；
- 政策文档更新；
- 用户权限变化；
- 数据被删除；
- 工具状态变化；
- 安全规则变化。

优先使用版本化命名空间，让旧键自然不可命中，而不是遍历删除所有对象。

### 三、避免缓存击穿和雪崩

热门前缀失效时，大量请求可能同时重算。

可以使用：

- 单飞请求，只有一个工作者重建；
- 随机化 TTL；
- 预热；
- 限流；
- 旧值短暂保留，但只适用于允许陈旧的场景；
- 分批版本迁移。

### 四、Prefix Cache 与采样参数的关系

前缀 K/V 来自输入前向计算，Temperature 和 Top-p 主要影响后续采样。

因此在模型和前缀相同的情况下，某些采样参数变化不一定要求前缀 KV 失效。

但具体缓存键仍应遵循推理引擎文档，不能凭经验删除字段。

### 五、Semantic Cache 可只缓存“证据”，不缓存最终答案

对时效和个性化敏感任务，可以降低风险：

```text
问题
  -> 语义缓存命中相关证据/检索结果
  -> 重新做权限和时效检查
  -> 模型生成新答案
```

这比直接返回旧答案成本高，但能保留新鲜校验和当前用户语境。

### 六、缓存不能绕过授权

缓存命中后仍要检查当前请求是否有权访问结果。

“之前某人能看到”不代表“当前用户也能看到”。

授权应在返回结果前执行，而不是只在缓存写入时执行。

### 七、观察负命中和假命中

对 Semantic Cache：

- 假阳性：不该复用却复用了；
- 假阴性：可以复用但没有命中。

假阳性通常直接影响正确性和安全，成本更高。

阈值应依据业务损失函数，而不是只追求命中率。

### 八、做容量与收益实验

逐步增加缓存容量，记录：

- Token 命中率；
- Prefill 节省；
- GPU KV 压力；
- 淘汰和加载；
- 端到端延迟；
- 集群吞吐；
- 错误命中。

找到收益趋缓的容量，而不是默认“越大越好”。

## 常见误区

### 误区一：KV Cache 会缓存最终答案

KV Cache 保存的是注意力 Key/Value 中间状态，不是可直接展示的答案文本。

### 误区二：有 KV Cache 后注意力就没有成本

它避免历史 K/V 重算，但新 Query 仍需访问历史 K/V。

### 误区三：Prefix Cache 可以加速所有请求

只有存在足够长且可复用的精确前缀时才有明显收益。

### 误区四：字符串相同就可以共享 KV

模型、Tokenizer、Token ID、Adapter 和其他计算条件也要兼容。

### 误区五：Prefix Cache 会减少输出 Decode

它主要减少命中前缀的 Prefill，新答案仍需生成。

### 误区六：语义相似就意味着答案相同

身份、时间、权限和隐含条件都可能改变答案。

### 误区七：缓存键用哈希就安全了

哈希不替代租户隔离、授权、加密和审计。

### 误区八：命中率越高，缓存越成功

还要看命中 Token、节省时间、显存成本和错误命中。

### 误区九：TTL 足以解决所有失效

权限撤销和安全更新可能要求立即失效，不能等 TTL 自然过期。

### 误区十：语义缓存适合余额、价格等实时数据

这类答案强依赖当前状态，直接复用旧结果风险很高。

## 面试追问

### 追问一：KV Cache 为什么是 K 和 V，不缓存 Q？

历史位置的 K/V 会被未来每个新 Query 重复访问；当前步 Query 只用于当前注意力计算，未来会产生新的 Query，因此通常不按同样方式保留历史 Q。

### 追问二：Prefix 中间改了一个 Token 会怎样？

变化位置之前的完整相同 block 可能继续复用；变化位置及之后的 K/V 需要重新计算。

### 追问三：Prefix Cache 和 Prompt Cache 是同义词吗？

很多推理框架中二者都指复用前缀 K/V，但厂商也可能把 Prompt Caching 用于计费或服务端缓存能力。面试时应说明具体实现和复用对象。

### 追问四：如何估算 KV Cache 显存？

用 $2\times层数\times批量\times序列长度\times KV Head 数\times Head 维度\times每元素字节$ 做一阶估算，再加框架元数据、碎片和临时空间。

### 延伸问题与回答要点

| 问题 | 回答要点 |
| --- | --- |
| GQA 为什么省 KV | 多个 Query Head 共享较少 KV Head，使公式中的 $H_{kv}$ 下降 |
| 如何提高前缀命中 | 固定内容放前面，动态内容放后面，稳定模板并避免无意义随机字段 |
| 语义阈值怎么选 | 用允许复用与禁止复用的业务配对集，按风险优化 Precision—Recall |
| 缓存放 CPU 还是 GPU | 比较重算与传输时间、热度、容量和 SLO，常采用分层策略 |
| 模型升级后能否复用 | 权重变化通常使旧 KV 不兼容；语义答案也需重新验证策略与知识版本 |
| 是否缓存工具结果 | 取决于工具幂等性、权限、时效和数据版本，不能用语言相似度替代业务缓存键 |
