## 本篇目标

学完这一篇，你应该能够：

1. 准确回答 MCP 当前定义的两种标准传输：`stdio` 和 Streamable HTTP。
2. 解释 MCP 数据层使用 JSON-RPC 2.0，而传输层负责怎样搬运这些消息。
3. 说清 SSE 在 Streamable HTTP 中的角色，而不是把 SSE 误当成唯一传输。
4. 解释旧 HTTP+SSE transport 已被 Streamable HTTP 替代。
5. 说明为什么 WebSocket 不是 MCP 的标准传输，但可以作为自定义传输。
6. 根据本地、远程、多用户、会话与安全需求选择传输。
7. 排查 stdout 污染、会话丢失、代理缓冲、断线重放和 DNS rebinding 等问题。

## 60 秒回答

> MCP 2025-11-25 规范在数据层使用 UTF-8 编码的 JSON-RPC 2.0 消息，当前定义两种标准传输：`stdio` 和 Streamable HTTP。`stdio` 适合本地集成，由 Client 启动 MCP Server 子进程，通过标准输入输出传输按换行分隔的消息；stdout 只能写合法 MCP 消息，日志应写 stderr。
>
> Streamable HTTP 适合独立或远程 Server。Server 提供一个同时支持 POST 和 GET 的 MCP endpoint；Client 用 POST 发送 JSON-RPC，Server 可返回普通 JSON，也可用 `text/event-stream` 的 SSE 流发送多条消息，GET 也可建立服务端到客户端的 SSE 流。SSE 是 Streamable HTTP 内可选的服务端流通道，不等于 MCP 默认只用 SSE。
>
> 当前 Streamable HTTP 已替代 2024-11-05 版本的旧 HTTP+SSE transport。WebSocket 不是标准传输，只能作为双方明确实现的自定义传输。工程上本地、单用户工具通常优先 stdio，跨机器、多客户端和集中鉴权通常选择 Streamable HTTP；还要做 Origin 校验、认证授权、会话保护、协议版本头、超时、限流和幂等，不能因为“协议通了”就认为业务安全。

## 详细解析

### 先说人话：JSON-RPC 是“信件格式”，传输是“送信方式”

假设你要发送一封标准表格：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}
```

这封信的字段和含义由 JSON-RPC 与 MCP 数据模型规定。

但信可以通过不同方式搬运：

- 在父子进程的标准输入输出中传递；
- 通过 HTTP 请求和响应传递；
- 双方自行约定其他传输。

所以要分清两层：

```text
数据层：MCP 的 JSON-RPC 消息是什么
传输层：这些消息怎样在 Client 与 Server 之间流动
```

### 当前标准答案

MCP 2025-11-25 定义两种标准传输：

1. `stdio`；
2. Streamable HTTP。

规范也允许可插拔的自定义传输，但自定义传输要保持 MCP 消息语义和生命周期要求。

### 为什么不能回答“默认 WebSocket”

WebSocket 很适合双向长连接，所以很多人会凭经验猜 MCP 使用 WebSocket。

但当前 MCP 标准传输列表中没有 WebSocket。

准确说法是：

```text
MCP 可以由实现方定义 WebSocket transport，
但它不是规范要求所有实现都支持的标准传输。
```

如果 Client 只支持标准传输，不能假设它会连接自定义 WebSocket Server。

### JSON-RPC 消息基础

MCP 使用 JSON-RPC 2.0 编码：

- **Request**：有 `id`，期望 Response；
- **Response**：携带相同 `id`，返回 `result` 或 `error`；
- **Notification**：没有 `id`，不期望 Response。

示意：

```json
{
  "jsonrpc": "2.0",
  "id": 10,
  "method": "tools/call",
  "params": {
    "name": "get_order",
    "arguments": {
      "order_id": "A1001"
    }
  }
}
```

传输层必须保持消息完整，不能把日志、调试文字或代理错误页混进去。

### 标准传输一：stdio

`stdio` 是 standard input/output 的缩写，即标准输入和标准输出。

典型关系：

```text
MCP Client / Host
  │
  │ 启动子进程
  ▼
MCP Server Process

Client 写 Server stdin
Server 写 Server stdout
Server 日志写 stderr
```

### stdio 怎样工作

通常由 Client：

1. 确定可执行命令、参数、工作目录和环境变量；
2. 启动 MCP Server 子进程；
3. 打开该进程的 stdin、stdout 和 stderr 管道；
4. 向 stdin 写 JSON-RPC 消息；
5. 从 stdout 读取 Server 消息；
6. 从 stderr 收集日志；
7. 在会话结束时关闭输入并管理进程退出。

每条消息以换行分隔，消息本身不能包含嵌入换行。

JSON-RPC 消息使用 UTF-8 编码。

### stdout 为什么不能打印日志

错误示例：

```text
Server started successfully!
{"jsonrpc":"2.0","id":1,"result":{"tools":[]}}
```

Client 把第一行当 JSON-RPC 解析，就会失败。

正确做法：

```text
stdout -> 只写 MCP 消息
stderr -> 日志、诊断信息
```

很多“Server 一启动就断开”的根因，只是一条调试日志污染 stdout。

### stdio 的优点

- 本地接入简单；
- 不需要监听网络端口；
- 生命周期容易绑定到 Host；
- 无需单独部署远程服务；
- 延迟低；
- 适合 IDE、桌面应用和命令行工具；
- 不会直接暴露公网入口。

### stdio 的局限

- 通常一个 Client 对应一个子进程或一条会话；
- 多用户共享与横向扩展不自然；
- Server 安装和依赖要分发到本机；
- 子进程可能访问本地文件、密钥和网络；
- 进程退出就会丢失内存状态；
- 容器、桌面权限和路径差异会增加部署问题；
- 很难直接作为跨机器公共服务。

“没有网络端口”不等于“没有安全风险”。

### stdio 的常见失败路径

| 现象 | 常见原因 | 排查方式 |
| --- | --- | --- |
| 启动后立即退出 | 命令、依赖或环境变量错误 | 查看退出码和 stderr |
| JSON 解析失败 | stdout 混入日志 | 捕获原始 stdout 字节 |
| 一直没有响应 | 消息未换行或缓冲未刷新 | 检查分隔和 flush |
| 找不到文件 | 工作目录与预期不同 | 记录完整路径和 cwd |
| 本地密钥泄露 | 继承过多环境变量 | 使用显式最小环境 |
| 关闭 Host 后有孤儿进程 | 生命周期未传播 | 监控父进程与退出信号 |

### 标准传输二：Streamable HTTP

Streamable HTTP 让 MCP Server 作为独立进程运行，并处理多个 Client 连接。

Server 提供一个 MCP endpoint，例如：

```text
https://api.example.com/mcp
```

同一个 endpoint 需要处理：

- HTTP POST；
- HTTP GET。

### POST 的作用

Client 把每条 JSON-RPC 消息放进一个新的 HTTP POST 请求。

简化示意：

```http
POST /mcp HTTP/1.1
Content-Type: application/json
Accept: application/json, text/event-stream
MCP-Protocol-Version: 2025-11-25

{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}
```

对于 JSON-RPC Request，Server 可以：

- 返回 `application/json`，其中包含一个 JSON-RPC Response；
- 或返回 `text/event-stream`，在 SSE 流中发送多条 Server 消息，最终包含对应 Response。

对于 Notification 或 Response，Server 接收后通常返回表示已接受的 HTTP 状态，而不是再生成 JSON-RPC Response。

### GET 的作用

Client 可以对同一 endpoint 发 GET，以打开 SSE 流，接收 Server 主动发来的请求或通知。

简化示意：

```http
GET /mcp HTTP/1.1
Accept: text/event-stream
MCP-Protocol-Version: 2025-11-25
```

如果 Server 不提供该 SSE 流，应使用规范要求的 HTTP 行为明确拒绝，而不是返回一段普通 HTML。

### SSE 到底是什么角色

SSE，即 Server-Sent Events，是 HTTP 上的服务端事件流格式，媒体类型是：

```text
text/event-stream
```

在 Streamable HTTP 中，SSE 可以用于：

- 在一个 POST 响应里流式发送多条 Server 消息；
- 通过 GET 建立服务端到客户端的消息流；
- 携带事件 ID，支持断线后的恢复与重投递设计。

但 Client 向 Server 发送 JSON-RPC 仍通过 POST。

因此准确关系是：

```text
Streamable HTTP = POST/GET 的 MCP 传输模型
SSE = 其中可选的服务端流式编码与通道
```

### 为什么不能说“MCP 就是 SSE”

因为：

- stdio 完全不使用 SSE；
- Streamable HTTP 的简单响应可以使用普通 JSON；
- SSE 主要承载服务端向 Client 的流；
- MCP 还包括 JSON-RPC、生命周期、能力与业务对象；
- SSE 本身不知道什么是 `tools/call`。

SSE 只是传输中的一块机制。

### Streamable HTTP 的会话管理

Server 可以在初始化响应中分配：

```text
MCP-Session-Id
```

如果 Server 返回会话 ID，Client 后续 HTTP 请求必须携带它。

会话 ID 应：

- 全局唯一；
- 难以猜测；
- 安全保存；
- 不写入公开 URL；
- 与真实认证身份绑定；
- 在日志中脱敏。

会话结束时，Client 可以按规范尝试使用 DELETE 请求终止会话；Server 是否支持主动终止要按规范响应。

会话 ID 不是身份凭证，也不能替代访问令牌和服务端授权。

### 协议版本头

初始化后，Streamable HTTP Client 需要在后续请求中携带：

```text
MCP-Protocol-Version: 2025-11-25
```

Server 应验证该版本是否受支持。

协议版本和 HTTP 版本不是一回事：

```text
MCP-Protocol-Version -> MCP 规范版本
HTTP/1.1、HTTP/2、HTTP/3 -> 底层 HTTP 版本
```

### 多连接与消息路由

一个 Client 可能同时存在多个 SSE 流。

Server 必须避免：

- 把一条消息广播到所有连接；
- 把用户 A 的消息发给用户 B；
- 把一个会话的 Response 发到不相关流；
- 在断线重投递时重复执行副作用。

路由至少考虑：

- 认证主体；
- 租户；
- Session ID；
- JSON-RPC id；
- 事件流；
- 业务请求键。

### 恢复与重投递

Streamable HTTP 可以利用 SSE `id` 字段实现恢复。

Client 重连时发送：

```text
Last-Event-ID
```

Server 可从该位置继续发送后续消息。

但这不等于自动获得“恰好一次”业务执行语义。

需要区分：

```text
消息重新投递
        ≠
业务操作重新执行
```

对于 `create_refund` 等写操作，必须使用业务幂等键并在状态不明时查询结果。

### Streamable HTTP 的优点

- 适合远程部署；
- 可以集中认证、限流和审计；
- 一个 Server 可服务多个 Client；
- 易于容器化和水平扩展；
- 复用 HTTP 基础设施；
- 可以按需提供流式服务端消息；
- 可支持会话和断线恢复。

### Streamable HTTP 的局限

- 需要处理认证授权和公网攻击面；
- 代理可能缓冲 SSE；
- 网关可能有空闲超时；
- 长连接增加资源和部署排空复杂度；
- 会话状态会影响横向扩展；
- 恢复、去重和副作用幂等仍要自己设计；
- 错误可能同时存在 HTTP、JSON-RPC 和工具业务三层。

### 旧 HTTP+SSE transport 是什么

MCP 2024-11-05 协议版本曾定义独立的 HTTP+SSE transport。

当前规范明确：

```text
Streamable HTTP replaces the HTTP+SSE transport from 2024-11-05.
```

旧方案通常使用独立 SSE endpoint 和消息 POST endpoint。

新 Streamable HTTP 使用统一 MCP endpoint，支持 POST 和 GET，并允许 POST 响应本身成为 SSE 流。

所以面试时应说：

> 旧 HTTP+SSE transport 已被 Streamable HTTP 替代；SSE 没有消失，而是作为 Streamable HTTP 中可选的服务端流机制继续存在。

### 向后兼容怎样处理

规范给出了新旧实现兼容思路。

工程上可以：

- 新 Server 优先实现 Streamable HTTP；
- 迁移期按需保留旧 endpoint；
- Client 先尝试当前 Streamable HTTP 行为；
- 只有明确需要连接旧 Server 时才回退；
- 对回退进行日志记录和安全评审；
- 不把旧实现的行为误写成当前默认规范。

生产系统不应无限期静默回退，因为这可能丢失新版本安全与能力语义。

### Origin 校验与 DNS rebinding

Streamable HTTP Server 必须校验传入请求的 `Origin`。

原因是 **DNS rebinding** 攻击可能让恶意网页借浏览器访问用户本机的 MCP Server。

规范要求的关键控制包括：

- 有 `Origin` 且无效时返回 HTTP 403；
- 本地 Server 应只绑定 `127.0.0.1` 等 localhost，而不是 `0.0.0.0`；
- 所有连接实施正确认证；
- 不仅依赖“端口只有本机知道”。

### Streamable HTTP 的三层错误

调用失败时要分层判断：

| 层 | 示例 |
| --- | --- |
| HTTP | 401、403、404、429、502 |
| JSON-RPC | 方法不存在、参数无效、内部错误 |
| Tool 业务 | 订单不存在、余额不足、审批未通过 |

HTTP 200 不代表 Tool 成功。

HTTP 500 也不一定说明业务操作没有执行。

## 工程实践与边界

### 传输选型

| 场景 | 推荐起点 | 原因 |
| --- | --- | --- |
| 桌面应用调用本地脚本 | stdio | 简单，不暴露端口 |
| IDE 访问本地项目 | stdio | 生命周期跟随 Host |
| 跨机器共享能力 | Streamable HTTP | 可远程连接与集中治理 |
| 多租户企业服务 | Streamable HTTP | 便于认证、限流、审计 |
| 无法部署本地依赖 | Streamable HTTP | Server 集中部署 |
| 极特殊基础设施 | 自定义传输 | 仅在双方明确实现时采用 |

### stdio 上线检查单

- 使用固定的完整可执行路径；
- 固定依赖版本；
- 传入最小环境变量；
- 设置明确工作目录；
- stdout 只允许协议消息；
- stderr 有日志大小限制；
- 设置启动与请求超时；
- 限制文件、网络和子进程权限；
- Host 退出时清理 Server；
- 捕获退出码并提供可读诊断。

### Streamable HTTP 上线检查单

- HTTPS；
- Origin 校验；
- 认证与逐请求授权；
- 租户隔离；
- `MCP-Protocol-Version` 校验；
- Session ID 安全生成和绑定；
- 请求体与消息大小限制；
- 连接、请求、空闲和任务总超时；
- SSE 代理缓冲关闭；
- 事件恢复与去重；
- 限流和并发上限；
- 日志脱敏和审计；
- 优雅关闭与部署排空。

### 代理与网关配置

SSE 需要尽快把事件刷新给 Client。

常见问题：

- 反向代理把多个事件缓冲后一次返回；
- 压缩层等待缓冲区填满；
- 空闲超时关闭无数据流；
- 负载均衡把会话请求发到不共享状态的实例；
- CDN 缓存动态事件流。

应通过真实部署链路测试首事件延迟、持续流、断线和重连，而不是只在 localhost 演示。

### 心跳与空闲连接

可以在允许的流语义下发送保持活动的事件或注释，避免中间层误判空闲。

但心跳不能替代：

- 请求总截止时间；
- 任务超时；
- Server 健康检查；
- 用户取消；
- 资源上限。

### 背压

**背压**是消费者处理速度低于生产速度时，系统如何限制积压。

如果 Client 读不动，Server 不能无限缓存：

- 限制每连接队列；
- 限制消息和事件大小；
- 低价值进度可合并；
- 达到上限时明确关闭或降级；
- Artifact 大文件走专门存储，不塞进事件流；
- 记录慢消费者指标。

### 重试策略

| 操作 | 建议 |
| --- | --- |
| `tools/list` 等只读请求 | 指数退避、有限重试 |
| SSE 重连 | 使用 `Last-Event-ID` 和去重 |
| 无副作用读取 Tool | 确认可重试后有限重试 |
| 写 Tool | 业务幂等键 + 状态查询 |
| 初始化失败 | 根据错误分类，不无限重连 |
| 认证失败 | 刷新凭证或要求重新登录，不盲目重试 |

### 可观测性

建议关联：

```text
trace_id
transport = stdio | streamable_http
protocol_version
client_id / server_id
session_id（脱敏）
json_rpc_id
method
sse_event_id
tool_call_id
business_idempotency_key
```

分别统计：

- 初始化成功率；
- 首消息延迟；
- 请求 P50/P95/P99；
- SSE 断线与恢复率；
- stdio 异常退出率；
- 活跃会话数；
- 慢消费者数；
- HTTP、JSON-RPC 和业务错误比例。

### 自定义传输的边界

只有在标准传输确实不能满足约束时才考虑自定义传输。

需要明确：

- 消息边界；
- UTF-8 JSON-RPC 编码；
- 请求、响应和通知关联；
- 双向通信；
- 生命周期；
- 错误与关闭；
- 认证、授权与加密；
- 流控和恢复；
- Client 与 Server 的共同兼容实现。

“可以自定义”不等于其他 MCP Host 会自动支持。

## 常见误区

### 误区一：MCP 默认用 WebSocket

错误。当前标准传输是 stdio 和 Streamable HTTP。

### 误区二：Streamable HTTP 就是旧 HTTP+SSE 改名

不准确。它改为统一 endpoint 和新的 POST/GET 行为，并把 SSE 作为可选流机制；规范提供迁移兼容方案。

### 误区三：SSE 是与 HTTP 并列的独立网络协议

SSE 是 HTTP 响应中的事件流格式，媒体类型为 `text/event-stream`。

### 误区四：Streamable HTTP 的每次响应都必须是 SSE

错误。Server 可对请求返回普通 `application/json`，也可按需使用 SSE。

### 误区五：stdio 不走网络，所以可以完全信任

本地子进程可能读取文件、密钥和网络，应审查来源并最小权限运行。

### 误区六：会话 ID 就是登录凭证

不是。会话 ID 用于关联 MCP 会话，仍需独立认证和逐请求授权。

### 误区七：SSE 重连保证业务恰好执行一次

不保证。事件恢复与业务副作用幂等是两层问题。

### 误区八：HTTP 200 就代表 MCP Tool 成功

还要检查 JSON-RPC `error` 和 Tool 业务结果。

### 误区九：新 Client 应优先实现旧 HTTP+SSE

新实现应优先当前 Streamable HTTP；只有明确的旧系统兼容需求才回退。

## 面试追问

### 追问一：MCP 当前标准传输有哪些

> stdio 和 Streamable HTTP。WebSocket 不是标准传输，但双方可以实现自定义传输。

### 追问二：Streamable HTTP 中 POST 与 GET 分别做什么

POST 用来向 MCP endpoint 发送 Client 的 JSON-RPC 消息；GET 可建立 SSE 流接收 Server 主动请求和通知。POST 响应本身也可以是 SSE 流。

### 追问三：SSE 还在不在 MCP 里

在。被替代的是旧 HTTP+SSE transport，不是 SSE 技术本身。SSE 现在作为 Streamable HTTP 中可选的服务端流通道。

### 追问四：为什么 stdio 的日志要写 stderr

stdout 是协议数据通道，混入普通文本会破坏消息解析；stderr 是独立诊断通道。

### 追问五：为什么 Streamable HTTP 要校验 Origin

防止恶意网页通过 DNS rebinding 访问本机或内网 MCP Server。无效 Origin 应被拒绝，本地服务还应只绑定 localhost 并实施认证。

### 追问六：SSE 断线后怎样恢复

Server 给事件分配 ID，Client 保存最后处理位置，重连时发送 `Last-Event-ID`；双方按事件 ID 恢复和去重。写操作仍要独立使用业务幂等机制。

### 追问七：什么时候选 stdio，什么时候选 HTTP

本地、单用户、生命周期跟随 Host 时优先 stdio；跨机器、多用户、集中部署和集中治理时优先 Streamable HTTP。

### 追问八：能否用 WebSocket 实现 MCP

可以作为自定义传输，但 Client 与 Server 都要显式支持，且不能宣称这是当前规范的标准互操作方式。

### 可直接背诵的完整答案

> MCP 数据层使用 UTF-8 编码的 JSON-RPC 2.0，2025-11-25 规范定义两种标准传输：stdio 和 Streamable HTTP。stdio 由 Client 启动本地 Server 子进程，通过 stdin/stdout 交换换行分隔消息，stdout 只能写协议数据，日志写 stderr。Streamable HTTP 使用一个同时支持 POST 和 GET 的 MCP endpoint；POST 发送 JSON-RPC，Server 可以返回普通 JSON 或 SSE，GET 也可建立 SSE 流接收服务端消息。
>
> 旧 2024-11-05 的 HTTP+SSE transport 已被 Streamable HTTP 替代，但 SSE 仍是新传输中的可选流机制。WebSocket 不是标准传输，只能作为双方支持的自定义实现。本地工具通常优先 stdio，远程、多用户服务通常选择 Streamable HTTP。生产上还要处理 Origin 校验、认证授权、协议版本、会话保护、代理缓冲、背压、断线恢复和业务幂等，不能把传输重连误当成副作用可以安全重放。
