## 60 秒回答

大模型是根据输入生成输出的模型；Agent 是围绕模型构建的任务执行系统。它通常把目标、状态、工具和控制循环组合起来：模型提出下一步动作，运行时校验并执行动作，再把结果作为新状态反馈给模型，直到满足结束条件。

因此，工具调用并不自动等于 Agent。一次函数调用也可以是普通 Workflow。Agent 的关键是运行时存在有限范围的模型决策，而且系统能根据环境反馈继续、改道或停止。生产实现还必须有预算、权限、审计和人工接管，不能把“自主”理解成不受约束。

## 详细解析

可以从三个层次区分二者：

1. **能力层**：LLM 负责语言理解、生成和推断；它不会自行访问数据库或发送邮件。
2. **控制层**：Agent 运行时决定下一步是调用工具、查询记忆、请求澄清还是结束。外层程序驱动循环，模型并不会自己常驻运行。
3. **系统层**：Agent 维护任务状态、工具结果、失败信息和预算，并通过策略与守卫约束动作。

一个典型闭环是“读取目标与状态 -> 选择动作 -> 校验权限和参数 -> 执行 -> 记录结果 -> 重新决策”。模型只是闭环中的决策组件，真正的副作用由受控运行时完成。这种分工也意味着 Agent 的可靠性不能只靠换更强的模型，还取决于工具契约、状态管理和系统保护。

Agent 与 Workflow 也不是非此即彼。若路径和分支由开发者预先定义，它更接近 Workflow；若部分路径由模型根据运行结果动态选择，则具有 Agent 特征。实际系统常用 Workflow 固定高风险主流程，只在信息不完备的局部节点开放模型决策。

## 工程实践与边界

- 给每次运行设置最大步数、时间、Token 和费用预算，结束条件应由程序强制执行。
- 工具按最小权限授权；读操作与写操作分开，付款、删除、外发数据等动作需要额外审批。
- 状态按租户和用户隔离。日志中不要记录完整密钥、个人可识别信息（PII）或不必要的工具响应。
- 工具调用要有超时和可分类错误。可重试且有副作用的写操作在下游支持时使用幂等键；只读调用使用关联 ID。下游无法去重或查询状态时，超时后的写操作应转人工核对或执行补偿，不能自动重放。
- 记录动作、参数摘要、结果状态和模型/提示版本，便于复现与审计，但不依赖或公开模型的私有思维链。

## 常见误区

- **“会调工具就是 Agent”**：一次结构化工具调用没有自主循环，也可能只是 Workflow 节点。
- **“Agent 会自己执行工具”**：模型只提出调用请求，宿主程序负责鉴权、执行和回传。
- **“自主性越高越好”**：决策空间越大，评测、治理和故障定位成本通常越高。
- **“长上下文等于记忆”**：上下文是一次请求的输入；跨任务记忆还需要显式存储、检索和删除策略。

## 面试追问

> **面试官：** Agent 最小可用闭环是什么？
>
> **候选人：** 至少要有目标、状态、动作选择、受控执行、结果反馈和程序化停止条件。只有一次生成或一次工具调用还不能说明存在闭环。

> **面试官：** 为什么不让 Agent 完全自主？
>
> **候选人：** 模型决策具有不确定性，外部输入还可能包含注入内容。生产系统应限制可选工具、权限、预算和副作用，并为高风险动作保留人工审批。

> **面试官：** 如何判断 Agent 是否值得引入？
>
> **候选人：** 先看任务路径能否可靠枚举。路径稳定时优先 Workflow；只有环境反馈会改变后续步骤、规则难以穷举时，Agent 的动态决策才可能抵消新增复杂度。
