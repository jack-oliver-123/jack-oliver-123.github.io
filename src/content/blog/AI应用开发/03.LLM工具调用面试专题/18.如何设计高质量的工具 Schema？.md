## 本篇目标

学完这一篇，你应该能够：

- 理解工具 Schema 同时服务模型选择、参数校验和工程治理；
- 先划清工具边界，再设计名称、描述和参数；
- 区分可由模型提供的业务意图与必须由服务端注入的可信字段；
- 使用 JSON Schema 的类型、required、enum、范围和额外字段约束；
- 设计稳定输出、结构化错误、副作用和版本契约；
- 用选择评测、契约测试、负向测试和生产指标验证 Schema 质量。

## 60 秒回答

高质量工具 Schema 首先要有清晰边界：一个工具表达一个可审计能力，读操作与写操作分开，高风险动作不要藏进万能 action 字段。名称用稳定的动词加对象；description 说明什么时候用、什么时候不用、前置条件、返回结果和副作用，让相似工具形成互斥边界。

参数用对象结构，明确 type、required、enum、格式、单位和范围，能由服务端从会话推导的 user_id、tenant_id、角色、价格和权限不要让模型填写。工具参数始终是不可信输入，Schema 通过后还要做规范化、资源级授权、业务不变量和当前状态检查。

Schema 还要包含输出与错误契约、幂等和确认要求，并按版本治理。测试不能只验证 JSON 能解析，还要用真实任务测工具选择、参数有效率、误调用、拒调、权限拦截和端到端成功率。OpenAI、MCP 等平台支持的 JSON Schema 子集不同，跨平台时要以实际官方约束和契约测试为准。

## 详细解析

### 先说人话：工具 Schema 像办事窗口的业务说明和申请表

一个清楚的办事窗口会写：

- 这里办理什么；
- 哪些情况来这里；
- 哪些情况去另一个窗口；
- 申请表要填什么；
- 哪些信息由工作人员从身份证读取；
- 办理后会发生什么；
- 失败后怎么处理。

如果窗口只写“综合业务”，申请表只有一个“请描述需求”文本框，办错业务和越权都更容易发生。

工具 Schema 就是给模型和宿主应用看的“窗口说明 + 申请表”。它不是函数参数的简单复制。

### 贯穿案例：退款助手的工具设计

一个客服 Agent 需要：

- 查订单；
- 计算可退款金额；
- 创建退款；
- 查询退款状态；
- 取消还未执行的退款申请。

不好的设计：

~~~json
{
  "name": "order_action",
  "description": "处理订单",
  "parameters": {
    "type": "object",
    "properties": {
      "action": {"type": "string"},
      "data": {"type": "string"}
    }
  }
}
~~~

问题：

- 模型不知道有哪些 action；
- data 是自由文本；
- 读写权限混在一起；
- 审计无法快速识别真实动作；
- 不能为退款单独设置确认和幂等；
- 错误难以分类。

更好的方向是拆成 get_order、preview_refund、create_refund、get_refund_status 等清晰能力。

### Schema 的三类读者

| 读者 | 关心什么 |
| --- | --- |
| 模型 | 什么时候选这个工具、参数语义是什么 |
| 宿主应用 | 如何验证、授权、执行和处理错误 |
| 平台与运维 | 如何审计、限流、监控、升级和回滚 |

只对代码生成器友好、却让模型分不清用途的 Schema，不算高质量。

### 第一步：先设计工具边界

#### 一个工具承担一个清晰能力

好的边界具有：

- 单一业务目的；
- 明确输入输出；
- 明确读写性质；
- 明确权限；
- 明确副作用；
- 可单独限流、监控和审计。

#### 读写操作分开

~~~text
get_order        → 只读
preview_refund   → 只读计算
create_refund    → 写操作
cancel_refund    → 写操作
~~~

这样可以：

- 只给某些 Agent 读权限；
- 写操作单独确认；
- 使用不同超时与重试；
- 精确统计副作用；
- 避免一个万能工具扩大权限。

#### 工具粒度不是越细越好

过粗：

- 一个 execute_anything；
- 用 action 字段决定全部行为；
- 授权和审计困难。

过细：

- get_order_name；
- get_order_amount；
- get_order_status；
- get_order_currency；

模型需要连续调用很多次，延迟和错误面增加。

通常以“一个稳定、可授权的业务动作”为边界。

### 第二步：设计清晰名称

推荐：

~~~text
动词 + 领域对象
~~~

例如：

- get_order；
- search_policies；
- preview_refund；
- create_refund；
- send_support_email。

名称应：

- 稳定；
- 简短；
- 不含实现供应商；
- 使用团队统一领域词；
- 能区分读写；
- 避免模糊缩写。

不推荐：

- do_it；
- helper；
- api_call；
- handle_data；
- execute_action。

### 第三步：description 写选择边界

一个有用描述回答五个问题：

1. 工具做什么；
2. 什么情况下使用；
3. 什么情况下不要使用；
4. 有什么前置条件；
5. 是否有副作用。

示例：

~~~text
创建退款记录并向支付系统提交退款。
仅在已经查询订单、确认可退款余额、用户明确同意后使用。
不要用于估算退款金额；估算请使用 preview_refund。
这是有副作用的写操作，同一业务请求必须复用幂等键。
~~~

相似工具要互相说明差异：

| 工具 | 选择条件 |
| --- | --- |
| preview_refund | 只计算能否退和可退金额，不产生退款 |
| create_refund | 用户确认后真正提交退款 |
| get_refund_status | 已有退款 ID，只查询状态 |

description 不是越长越好。重复背景会挤占上下文，真正的边界反而不突出。

### 第四步：参数只让模型填写它知道的意图

#### 模型可以提供

- 用户明确说出的 order_id；
- 选择的 reason_code；
- 请求金额；
- 搜索关键词；
- 时间范围；
- 用户确认后的内容。

#### 服务端必须提供或验证

- authenticated_user_id；
- tenant_id；
- 角色和 scope；
- 数据分类；
- 当前价格；
- 可退款余额；
- 审批状态；
- 内部服务凭证；
- 权威时间；
- 幂等记录。

危险 Schema：

~~~json
{
  "properties": {
    "tenant_id": {"type": "string"},
    "is_admin": {"type": "boolean"},
    "approved": {"type": "boolean"}
  }
}
~~~

模型不能自证“我是管理员”或“已经审批”。这些值应来自受信系统。

### 第五步：使用尽量收紧的 JSON Schema

完整例子：

~~~json
{
  "type": "object",
  "properties": {
    "order_id": {
      "type": "string",
      "description": "用户订单号，必须来自当前对话或订单查询结果",
      "minLength": 1,
      "maxLength": 64
    },
    "amount_minor": {
      "type": "integer",
      "description": "退款金额，使用币种最小单位，例如人民币分",
      "minimum": 1
    },
    "currency": {
      "type": "string",
      "enum": ["CNY", "USD"]
    },
    "reason_code": {
      "type": "string",
      "enum": ["DAMAGED", "WRONG_ITEM", "NOT_AS_DESCRIBED", "OTHER"]
    },
    "user_note": {
      "type": ["string", "null"],
      "maxLength": 500
    }
  },
  "required": [
    "order_id",
    "amount_minor",
    "currency",
    "reason_code",
    "user_note"
  ],
  "additionalProperties": false
}
~~~

不同模型平台对 nullable、联合类型和关键字支持不同。示例需要根据目标官方文档调整并做契约测试。

### type：不要把所有字段都写成 string

错误：

~~~json
{
  "amount": "299元",
  "retry": "yes",
  "items": "a,b,c"
}
~~~

更明确：

~~~json
{
  "amount_minor": 29900,
  "retry": true,
  "items": ["a", "b", "c"]
}
~~~

正确类型减少后续解析歧义。

### required：明确缺一不可的字段

如果 amount 是执行退款所需，就放进 required。
可选字段要定义缺失与 null 的区别：

- 缺失：调用方没有提供该概念；
- null：明确没有值；
- 空字符串：通常不应代替 null。

跨平台 strict 模式可能要求所有 properties 都列为 required，再用 null 表示可选。以目标平台官方规则为准。

### enum：让有限业务状态可控

reason 不要是任意文本：

~~~json
{
  "reason": {
    "type": "string",
    "enum": ["DAMAGED", "WRONG_ITEM", "OTHER"]
  }
}
~~~

enum 的值应：

- 稳定；
- 机器友好；
- 描述清楚；
- 与业务系统映射；
- 变更有兼容策略。

用户原始描述可以单独保留为 note，但不能代替业务 reason_code。

### 范围、长度、格式和单位

| 约束 | 作用 |
| --- | --- |
| minimum / maximum | 限制数值结构范围 |
| minLength / maxLength | 防空值和超大输入 |
| pattern | 约束基本格式 |
| format | 表达 email、date-time 等语义 |
| minItems / maxItems | 控制批量大小 |
| uniqueItems | 表达集合去重意图 |

注意：

- format 是否被执行为校验取决于实现；
- pattern 不能替代资源存在性检查；
- minimum 不能知道真实可退余额；
- maxItems 不能代替租户配额。

### 时间、金额和标识符要消除歧义

#### 时间

说明：

- ISO 8601 或其他格式；
- 时区；
- 起止边界是否包含；
- “今天”由哪个时区解释。

#### 金额

推荐整数最小单位加币种：

~~~json
{
  "amount_minor": 29900,
  "currency": "CNY"
}
~~~

避免浮点金额和没有单位的 amount。

#### ID

区分：

- 用户可见订单号；
- 内部数据库主键；
- 工具调用 ID；
- 业务幂等键。

模型提供的 ID 只能是候选，执行端必须校验归属。

### additionalProperties

严格工具通常把 additionalProperties 设为 false，避免模型偷偷加入：

~~~json
{
  "approved": true,
  "skip_validation": true
}
~~~

但是否支持以及如何要求，要看目标模型平台的 Schema 子集。

### 不要滥用深层嵌套和复杂联合

复杂 Schema 会：

- 增加模型理解难度；
- 挤占上下文；
- 降低跨供应商兼容；
- 让错误信息难以修复；
- 扩大测试组合。

如果存在两个完全不同的业务动作，优先拆成两个工具，而不是一个深层 oneOf 万能工具。

### 第六步：设计输出契约

工具不应只返回一段自然语言：

~~~text
退款好像成功了
~~~

更稳定的结果：

~~~json
{
  "status": "succeeded",
  "refund_id": "R-9001",
  "order_id": "O-2025-42",
  "amount_minor": 29900,
  "currency": "CNY",
  "created_at": "2025-08-17T10:30:00Z"
}
~~~

输出契约应说明：

- 成功状态；
- 业务 ID；
- 关键字段；
- 是否最终完成；
- 下一步可做什么；
- 来源或更新时间；
- 哪些字段可为空。

MCP Tools 在当前规范中可以声明 outputSchema，并可返回 structuredContent。无论协议是否原生支持，执行器内部都应稳定定义输出类型。

### 第七步：错误要结构化

~~~json
{
  "status": "error",
  "error": {
    "code": "ORDER_NOT_FOUND",
    "retryable": false,
    "safe_message": "未找到当前用户可访问的订单",
    "required_action": "ask_user_to_check_order_id"
  }
}
~~~

错误至少区分：

| 类型 | 示例 | 通常动作 |
| --- | --- | --- |
| INVALID_ARGUMENT | 字段格式错 | 可有限修正 |
| UNAUTHENTICATED | 没有登录 | 引导认证 |
| PERMISSION_DENIED | 无权访问 | 不重试 |
| NOT_FOUND | 资源不存在或不可见 | 请求核对 |
| CONFLICT | 状态不允许 | 重新读取状态 |
| RATE_LIMITED | 触发限流 | 按提示退避 |
| UNAVAILABLE | 短暂不可用 | 有界重试 |
| UNKNOWN_COMMIT | 超时且提交状态未知 | 查询状态，禁止盲重试 |

错误不能泄露堆栈、密钥、内部路径和其他租户资源是否存在。

### 第八步：把副作用元数据放到执行契约

可以为工具维护机器可读元数据：

~~~yaml
tool: create_refund
risk: high
operation: write
idempotency: required
confirmation: required
default_timeout_ms: 5000
retry_policy:
  automatic: false
resource_key: order_id
audit: full
~~~

这些元数据不一定全部发送给模型，但宿主和执行平台需要。

### 第九步：版本治理

Schema 变化分三类。

#### 兼容变化

- 新增真正可选字段；
- 扩展描述但不改变语义；
- 输出新增消费端可忽略字段。

#### 可能破坏兼容

- 新增 required；
- 删除或重命名字段；
- 收紧 enum；
- 改单位；
- 改副作用；
- 同名字段改变语义；
- 错误码语义变化。

#### 发布方式

- Schema 存入版本库；
- 生成类型或契约；
- 执行器与暴露定义做一致性测试；
- 回放真实调用；
- 新旧版本并行一段时间；
- 记录模型、工具和 Schema 版本；
- 可快速撤销。

不要依赖模型记住旧工具。每个请求只应使用当前允许的工具定义。

### 第十步：如何测试 Schema

#### 1. 静态 Schema 测试

- Schema 自身是否合法；
- 是否使用目标平台支持的关键字；
- 示例能否通过；
- 负例能否被拒绝；
- 生成类型是否一致。

#### 2. 执行器契约测试

- 每个参数是否正确映射；
- 单位和时区是否一致；
- required 是否与代码一致；
- 输出是否符合契约；
- 错误码是否稳定。

#### 3. 工具选择评测

构造：

- 应调用样本；
- 不应调用样本；
- 相似工具边界样本；
- 缺少前置信息样本；
- 恶意要求跳过授权样本。

测：

- 工具选择准确率；
- 误调用率；
- 漏调用率；
- 无需工具时的拒调率；
- 相似工具混淆矩阵。

#### 4. 参数评测

- Schema 通过率；
- 字段事实准确率；
- enum 选择；
- 时间、金额、单位；
- 资源 ID 是否来自上下文；
- 是否尝试填写可信字段。

#### 5. 安全与模糊测试

- 超长字符串；
- 路径穿越；
- 内网 URL；
- 命令元字符；
- SQL 片段；
- 其他租户资源 ID；
- 提示注入；
- Unicode 混淆；
- 重复和并发写操作。

## 工程实践与边界

### 完整工具定义示例

~~~json
{
  "type": "function",
  "name": "preview_refund",
  "description": "只读计算当前用户订单可否退款及最大金额，不创建退款。用户只是询问金额或规则时使用；真正提交请使用 create_refund。",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string",
        "description": "用户提供或 get_order 返回的订单号"
      },
      "requested_amount_minor": {
        "type": ["integer", "null"],
        "description": "用户指定金额，最小币种单位；未指定时为 null",
        "minimum": 1
      }
    },
    "required": ["order_id", "requested_amount_minor"],
    "additionalProperties": false
  },
  "strict": true
}
~~~

真正执行时，服务端仍要：

~~~text
从会话得到 user_id / tenant_id
→ 查询当前用户可见订单
→ 读取权威支付金额和币种
→ 读取当前退款状态和政策
→ 计算结果
→ 返回结构化输出
~~~

### 工具数量过多怎么办

不要把数百个工具一次性塞给模型。可以：

1. 先按用户权限过滤；
2. 按任务领域路由；
3. 从工具目录检索候选；
4. 只暴露一个小而相关的集合；
5. 记录候选召回与最终选择；
6. 对目录和 Schema 版本化。

工具检索也可能漏掉正确工具，需要单独评测。

### Description 迭代方法

出现工具混淆时：

1. 看真实 Trace；
2. 标记误选原因；
3. 调整边界描述；
4. 添加正反例；
5. 固定其他变量跑回归；
6. 检查 token 和延迟；
7. 灰度发布。

不要只凭感觉把描述越写越长。

### 监控指标

| 指标 | 说明 |
| --- | --- |
| 工具选择准确率 | 该选哪个是否选对 |
| 拒调准确率 | 不需工具时是否保持不调用 |
| 参数 Schema 失败率 | 结构问题 |
| 业务校验失败率 | 资源、状态、金额问题 |
| 授权拒绝率 | 模型请求了无权动作 |
| 修正重试次数 | Schema 是否让模型容易理解 |
| 工具执行成功率 | 执行器与下游质量 |
| 单任务工具数 | 是否过度调用 |
| 写操作重复数 | 幂等是否有效 |

### 安全边界

- Schema 不是授权系统；
- description 不是策略执行器；
- enum 不是业务状态机；
- pattern 不是 SSRF 防护；
- additionalProperties=false 不是命令注入防护；
- strict 不保证事实；
- 模型传来的 user_id、tenant_id 和 approved 不可信；
- 工具结果同样可能含恶意文本。

### 跨协议复用

可以维护一份内部工具契约，再适配到不同平台或 MCP：

~~~text
内部 Tool Contract
  ├─ OpenAI Function Tool
  ├─ 其他模型平台 Tool
  ├─ MCP inputSchema / outputSchema
  └─ 执行器类型与策略
~~~

适配层要做契约测试，不能假设同一个 JSON Schema 在所有平台支持完全相同。

## 常见误区

- **“Schema 能通过校验就设计好了”**：还要让模型选对、执行安全、版本可治理。
- **“description 越长越好”**：冗余会掩盖真正的使用边界。
- **“一个万能工具最灵活”**：会混合权限、副作用和错误语义。
- **“工具越细越安全”**：过细会增加调用链、延迟和组合错误。
- **“所有字段都让模型填写”**：身份、租户、角色、价格和审批应来自受信系统。
- **“strict 后参数可信”**：只代表结构受约束，不代表值真实或已授权。
- **“format 能验证业务有效性”**：格式正确的 URL 仍可能指向内网，ID 仍可能越权。
- **“Schema 向后兼容就语义兼容”**：单位和副作用变化可能破坏业务。

## 面试追问

### 1. 高质量工具 description 应写什么？

写工具目的、使用条件、不使用条件、前置条件、返回和副作用。相似工具要明确边界，不必堆实现细节。

### 2. 为什么 read 和 write 要分开？

它们的授权、确认、重试、幂等、审计和故障处理不同。分开后可以给 Agent 最小权限，也更容易监控。

### 3. 工具粒度怎么判断？

以一个稳定、可授权、可审计的业务动作作为候选边界，再用真实任务评测调用次数、选择混淆和延迟，避免过粗或过细。

### 4. 为什么不能让模型传 tenant_id？

模型输出可被用户提示影响，不能作为身份来源。tenant_id 应从受信会话或服务身份注入，并在执行点授权。

### 5. Schema 通过后还校验什么？

参数规范化、资源存在与归属、当前状态、金额和时间、主体授权、配额、审批、幂等和安全策略。

### 6. 工具太多怎么处理？

先按权限过滤，再按任务领域或工具目录检索候选，只把小集合给模型，并评测候选工具召回和最终选择。

### 7. Schema 如何版本化？

定义显式版本，分类兼容和破坏性变更，执行器与暴露定义做契约测试，新旧版本灰度并记录调用所用版本。

### 8. 如何证明 Schema 优化有效？

固定模型和任务集，比较工具选择、拒调、参数通过、业务校验、授权拦截、端到端成功、调用次数和 token，而不是只展示几个成功 Demo。
