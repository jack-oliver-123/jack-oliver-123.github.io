## 本篇目标

学会把“大而模糊的目标”变成可以执行、验证和恢复的子任务。读完后，你应该能解释怎样拆、拆到多细、哪些步骤能并行，以及如何证明拆分真的有效。

## 60 秒回答

任务拆分的目标不是让步骤越多，而是把大目标变成边界清晰、可验证、可恢复的工作单元。每个子任务至少应说明目标、输入、输出、依赖、允许的工具、验收条件和失败策略，再把它们组织成依赖图或受控状态机。

路径稳定时用开发者定义的静态拆分；子任务数量和类型随输入变化时，可以让 Planner 动态生成任务图，但运行时必须校验节点数、依赖、权限和预算。独立且没有写冲突的任务可以并行，有强依赖或共享副作用的任务要串行或做并发控制。拆分效果要通过端到端成功率、关键路径延迟、成本、返工率和恢复能力验证。

## 详细解析

### 先说人话：不要把“大项目”当成一个动作

假设老板说：

> “下周发布一个新产品网站。”

如果只把这句话交给一个人，他会立刻遇到很多问题：

- 产品卖点是什么；
- 需要哪些页面；
- 文案谁写；
- 设计什么时候完成；
- 开发依赖什么素材；
- 怎样验收；
- 上线失败怎么办。

合理做法是把目标拆成可管理的工作：

1. 明确目标用户和成功标准；
2. 整理产品信息；
3. 设计页面结构；
4. 写文案；
5. 产出视觉设计；
6. 开发页面；
7. 做功能和兼容性测试；
8. 发布并验证。

Agent 的任务拆分也是一样。它不是为了让模型“多思考几步”，而是让系统知道每一步需要什么、完成了什么、失败后从哪里继续。

### 为什么复杂任务需要拆分

如果一次要求模型同时完成检索、分析、决策、写作和校验，会发生几个问题：

- 目标太多，模型容易漏掉约束；
- 所有信息混在同一个上下文里；
- 出错时不知道是搜索错、分析错还是写作错；
- 只想重试失败部分，却不得不全部重做；
- 不同工具权限难以精确限制；
- 无法把独立任务并行执行。

拆分后可以：

- 给每一步更小、更相关的上下文；
- 使用专用提示和工具；
- 设置独立验收条件；
- 只重试失败步骤；
- 在安全边界写检查点；
- 对独立任务做并行调度。

但是拆分也会增加模型调用、数据传递、调度和状态管理，所以不是越细越好。

### 用“完成一份市场调研报告”做完整拆分

原始目标：

> 调研国内三款客服 Agent 产品，比较功能、价格、安全能力和部署方式，给出适合 100 人公司的建议，关键结论必须引用官方来源。

#### 第一步：先提取硬约束

在规划之前，先从用户目标中提取不能被后续步骤随意修改的约束：

- 产品数量：三款；
- 比较维度：功能、价格、安全、部署；
- 目标公司规模：100 人；
- 证据要求：关键结论使用官方来源；
- 输出：选型建议。

如果 Planner 后续忘记“官方来源”，运行时仍要保留这条硬约束。

#### 第二步：拆出高层任务

可以拆成：

1. 定义统一评价表；
2. 为三款产品分别收集官方资料；
3. 校验资料时效和来源；
4. 按统一 Schema 生成对比表；
5. 结合 100 人公司的约束给出建议；
6. 检查引用和结论一致性；
7. 生成最终报告。

#### 第三步：定义依赖关系

- 任务 2 依赖任务 1，因为先有评价字段才能统一收集；
- 三款产品的资料收集彼此独立，可以并行；
- 任务 4 依赖三份资料全部或达到最低完整度；
- 任务 5 依赖对比表；
- 任务 6 依赖初稿；
- 任务 7 依赖校验通过。

这就形成了一个任务依赖图。

### 什么是有向无环图

有向无环图通常简称 DAG。初学者可以这样理解：

- **有向**：箭头表示先后依赖；
- **无环**：不能出现 A 等 B、B 又等 A 的死循环；
- **图**：一项任务可以依赖多项任务，也可以分叉并行。

一个简单结构是：

> 定义指标 -> 分别收集 A、B、C -> 汇总对比 -> 给出建议 -> 验证 -> 交付

DAG 适合依赖关系比较清楚的批处理任务。持续对话、反复审批或可能回到旧状态的系统，通常更适合受控状态机。

### 一个可执行子任务应该包含什么

只写“调研产品 A”还不够，执行器不知道怎样算完成。

一个可靠子任务可以包含：

| 字段 | 产品 A 调研示例 |
| --- | --- |
| 任务 ID | research-product-a |
| 目标 | 收集产品 A 的四类官方信息 |
| 输入 | 产品名、评价字段、允许域名 |
| 依赖 | 评价表已经定义 |
| 允许工具 | 官方网站搜索、网页读取 |
| 预期输出 | 符合统一 Schema 的产品资料 |
| 验收条件 | 四类字段齐全，关键字段有来源和日期 |
| 失败策略 | 缺少价格时标记未知，不使用非官方猜测 |
| 预算 | 最多 8 次搜索、3 分钟 |

这才是可以被调度、验证和恢复的工作单元。

### 好的子任务有哪些特征

#### 1. 完备

所有子任务合起来必须覆盖原始目标和约束。不能拆完以后漏掉用户最关心的安全能力。

#### 2. 低耦合

子任务只依赖必要输入，输出契约明确。产品 A 的调研不应该依赖产品 B 的内部状态。

#### 3. 可验证

能够用 Schema、规则、测试或人工门禁判断是否完成。“尽量写好”不可验证，“四个字段齐全并有官方引用”可以验证。

#### 4. 可恢复

每个任务有稳定 ID 和状态，失败后能从检查点继续，不必重新运行所有步骤。

#### 5. 有界

明确工具、权限、最大时间、最大调用次数和费用，不能让子任务无限扩张。

#### 6. 保留证据

输出要携带来源和版本。聚合后如果发现冲突，可以追溯是哪一步、哪条资料产生的。

### 静态拆分和动态拆分

#### 静态拆分

开发者提前写好任务和依赖。适合：

- 流程长期稳定；
- 业务规则清晰；
- 输入差异不会改变主要步骤；
- 需要高度可预测。

例如发票处理始终是：

> 识别字段 -> 校验税号 -> 查重 -> 入账 -> 归档。

#### 动态拆分

Planner 根据具体输入生成任务图。适合：

- 子任务数量无法提前知道；
- 不同输入涉及不同资源；
- 任务需要探索；
- 可以定义统一的任务契约和边界。

例如修复代码问题时，只有看过项目后才知道涉及几个文件和哪些测试。

动态拆分不等于 Planner 想建多少任务就建多少任务。运行时要校验：

- 最大节点数和深度；
- 是否存在环；
- 引用的工具是否允许；
- 依赖是否指向有效节点；
- 总预算是否超限；
- 是否包含需要审批的写操作。

### 怎样决定拆分粒度

#### 粒度过粗

例如一个任务叫“搜索、分析并写完整报告”，里面混合多个失败原因：

- 搜索失败；
- 资料过期；
- 分析遗漏；
- 输出格式错误。

失败后无法只重试相关部分。

#### 粒度过细

如果把“搜索一次网页”“读取一段文字”“写一个字段”都变成独立模型任务，会产生：

- 大量模型调用；
- 序列化和上下文传递成本；
- 更多调度失败点；
- 状态数量膨胀；
- 信息被切碎。

#### 实用判断标准

一个子任务的粒度比较合适，通常是因为它：

- 可以独立验收；
- 可以独立重试；
- 拥有完成所需的上下文；
- 不跨越不必要的权限边界；
- 失败原因容易归类；
- 调用开销仍然合理。

### 哪些任务可以并行

满足以下条件时可以考虑并行：

- 没有前置依赖；
- 不会修改同一资源；
- 不共享会被覆盖的状态；
- 工具配额允许；
- 输出可以独立校验；
- 聚合规则已经定义。

三款产品资料收集通常是只读，可以并行。

下面这些任务不能直接并行：

- 两个步骤同时修改同一订单；
- 后一步需要前一步的输出；
- 多个任务竞争同一稀缺配额；
- 一个步骤失败会使其他步骤失去意义。

涉及共享写入时，需要资源锁、乐观版本、数据库事务或 Saga 补偿，而不是让模型自己判断并发是否安全。

### 并行结果怎样聚合

聚合器不能只把三段文本拼起来。它需要：

1. 校验每个结果是否符合统一 Schema；
2. 保留结果来源和任务 ID；
3. 识别字段缺失和冲突；
4. 按预定规则处理部分失败；
5. 判断是否达到后续步骤的最低完整度。

例如三款产品中一款价格未知，可以：

- 继续生成报告，但明确标注未知；
- 触发补充检索；
- 如果价格是必须字段，则暂停并向用户说明。

这个选择应该提前定义，而不是临时由聚合器随意猜。

### 执行结果改变计划时怎么办

假设产品 B 已经停止服务，原来的对比前提发生变化。系统可以：

1. 保存这个已验证事实；
2. 暂停依赖产品 B 的未执行任务；
3. 让 Planner 只更新尚未执行的计划；
4. 保留原始约束，例如仍需比较三款产品；
5. 询问用户是更换产品还是继续保留历史对比。

已经产生副作用的步骤不能简单回滚成“未执行”。邮件已经发出、订单已经修改，只能确认状态或执行明确补偿。

### 如何判断拆分是否提升效果

不要声称“任务拆分固定提升 30%”。效果取决于任务和模型。

应该在同一评测集上比较拆分前后：

| 指标 | 要回答的问题 |
| --- | --- |
| 端到端成功率 | 最终任务是否真的完成 |
| 约束满足率 | 用户要求是否被完整保留 |
| 关键路径延迟 | 并行是否真正缩短时间 |
| Token 和工具费用 | 新增步骤带来多少成本 |
| 返工率 | 有多少步骤因为上游错误重做 |
| 恢复成功率 | 中断后能否从检查点继续 |
| 人工接管率 | 动态拆分是否经常失控 |

如果成功率没有提高，调用数却大幅增加，就应该合并过细步骤或恢复固定 Workflow。

## 工程实践与边界

- **先固定硬约束**：数据地域、权限、审批、截止时间和预算保存在外部状态中，重规划不能覆盖。
- **任务图要校验**：检查环、无效依赖、最大节点数、深度和总预算。
- **每步有稳定身份**：使用任务 ID、父步骤 ID 和关联 ID 串联追踪；可能重试的写操作额外使用幂等键。
- **并行写入做并发控制**：使用锁、乐观版本、事务或 Saga，补偿流程显式建模。
- **压缩工具结果**：限制大小并生成摘要，但关键证据保留可回溯引用。
- **失败只重试必要部分**：先判断错误类型、幂等性和外部状态，再决定重试、重规划或人工接管。
- **用相同数据做对照**：固定模型、工具和测试集，避免把模型升级的收益误认为拆分收益。

## 常见误区

- **“拆分总会提高准确率”**：错误分解会传播偏差，更多调用也意味着更多失败点。
- **“列出步骤就是任务计划”**：缺少依赖、输入输出和验收条件的清单无法可靠调度。
- **“步骤越细越容易完成”**：过细会切碎上下文，增加调度和传递成本。
- **“没有依赖就可以任意并行”**：还要检查共享资源、配额和写冲突。
- **“Planner 生成的图可以直接执行”**：运行时必须校验工具、权限、依赖、预算和高风险动作。
- **“失败步骤直接重试”**：先确认动作是否幂等，以及外部副作用是否已经发生。

## 面试追问

> **面试官：** 为什么要做任务拆分？
>
> **候选人：** 为了把大目标变成可验证、可恢复和可调度的工作单元，让每一步使用更相关的上下文和工具，也能只重试失败部分；但拆分本身会增加调用和状态成本。

> **面试官：** 如何选择拆分粒度？
>
> **候选人：** 以能否独立验收、独立重试、上下文是否足够和权限边界是否清楚为标准，再根据调用开销和真实失败数据合并或细分。

> **面试官：** 动态计划如何防止失控？
>
> **候选人：** 限制最大节点数、深度、工具集合和预算，校验任务图无环且依赖有效，把硬约束放在模型外，并让高风险步骤进入审批。

> **面试官：** 哪些任务可以并行？
>
> **候选人：** 没有前置依赖、没有共享写冲突、配额允许、输出可独立校验且聚合规则明确的任务可以并行。

> **面试官：** 并行后如何聚合？
>
> **候选人：** 先定义统一结果 Schema 和部分失败语义，聚合器只接收通过校验的结果，保留来源与任务 ID，再按规则处理缺失和冲突。

> **面试官：** 如何证明拆分有效？
>
> **候选人：** 在同一评测集上比较端到端成功率、约束满足率、关键路径延迟、成本、返工率和恢复成功率，不能只看单个子任务分数。
