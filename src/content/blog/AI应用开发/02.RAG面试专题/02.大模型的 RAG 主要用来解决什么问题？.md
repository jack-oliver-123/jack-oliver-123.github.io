## 本篇目标

读完本篇，你应该能够：

- 不再用“RAG 用来减少幻觉”概括所有价值；
- 说清知识时效、私有知识、引用追溯和数据治理四类核心问题；
- 区分“模型缺少证据”和“模型有证据但仍然答错”；
- 判断一个任务是否真的需要 RAG；
- 在面试中主动说明 RAG 的能力边界。

## 60 秒回答

RAG 主要解决的是：大模型在回答当前问题时，缺少可更新、可追溯、受权限控制的外部知识。模型训练完成后不会自动知道新制度和企业私有数据，参数也不适合像数据库那样按文档精确修改或删除。RAG 在请求时从权威知识源检索证据，因此可以更快接入新知识、按用户权限过滤，并为答案保留来源。

它还能降低因知识缺失而产生的部分事实错误，但不能消除幻觉。知识库中没有答案、检索漏召回、源文档本身错误、模型曲解证据或遭遇提示注入时，结果仍然可能出错。

所以我会把 RAG 定位为“可治理的外部知识接入机制”，而不是通用正确性保证。纯写作、翻译、规则计算或需要真正执行操作的任务，未必适合只靠 RAG。

## 详细解析

### 先说人话：给聪明人一本会更新的公司手册

假设新员工小王很聪明，表达能力也很好，但他没有读过公司的内部制度。

你问他：“上海出差每晚能报多少钱？”他可能依据常识猜一个数字，也可能坦白不知道。即使他曾经读过 2024 年的制度，也不知道 2026 年制度已经调整。

解决方法不是要求他每天重新读完世界上所有资料，而是在回答前，把当前生效、与小王权限相符的制度片段交给他。

在这个类比中：

- 聪明的新员工代表大模型；
- 公司手册代表外部知识库；
- 查找相关章节代表检索；
- 根据章节回答代表增强生成。

RAG 的价值不是让模型凭空变得更聪明，而是让它在需要时获得当前任务所需的证据。

### 问题一：模型知识有时间边界

模型的知识来自训练数据和训练过程。训练结束后，新发生的事件不会自动进入已经部署的参数。

企业数据的变化通常比模型更新更频繁：

- 商品价格和库存随时变化；
- 公司制度按月或按季度调整；
- 软件版本持续发布；
- 法规和合同存在生效与失效时间；
- 客户工单每分钟都在新增。

如果每次事实变化都重新训练模型，训练、评测、部署和回滚成本会很高，也难以保证某个事实被准确更新。

RAG 把变化频繁的事实放在外部知识层。更新流程可以是：

> 权威文档变更 -> 重新解析受影响内容 -> 重建对应 Chunk 和向量 -> 发布新索引版本。

这样，知识更新与基础模型训练被分开管理。

#### 时效不是“已经同步”这么简单

生产系统还要定义知识新鲜度。例如：

- 原系统更新后，多久进入采集队列；
- 解析和索引构建需要多久；
- 新索引什么时候对查询可见；
- 缓存多久失效；
- 同步失败怎样告警。

因此，“RAG 能使用新知识”成立的前提是数据同步链路可靠，并且应用明确了新鲜度目标。

### 问题二：模型无法天然访问企业私有知识

公司的合同、工单、技术文档、内部 Wiki 和员工制度通常不会出现在公开训练数据中。即使使用通用能力很强的模型，它也不知道这些私有事实。

RAG 可以在应用层接入这些资料，但要注意两个边界：

1. **接入不等于全部公开**：系统仍需按租户、用户、部门和文档权限过滤。
2. **向量化不等于脱敏**：Embedding 和索引仍然属于需要保护的数据资产。

例如，小李能查询公开差旅标准，但不能看到财务部门的反舞弊审核规则。正确做法是在检索过程中携带小李的身份和 ACL 条件，而不是把所有片段交给模型后再要求模型“不要泄露”。

### 问题三：模型参数不适合做精确知识管理

业务系统常要求对知识执行以下操作：

- 查明某条规则来自哪份文件；
- 删除某个客户的全部资料；
- 回滚到昨天的制度版本；
- 只让特定部门访问某个附件；
- 对过期文档停止召回；
- 审计某次回答使用了哪些依据。

模型参数不是按文档 ID 组织的数据库。很难要求模型精确地“删除文档 A 的影响，同时保留其他知识不变”。

RAG 的外部知识层可以保存文档 ID、Chunk ID、版本、有效期和 ACL，使更新、删除、审计与回滚更接近普通数据工程问题。

这并不表示 RAG 的删除天然可靠。系统还要同步删除：

- 原始文件的检索副本；
- 派生 Chunk；
- 向量和关键词索引；
- 查询或答案缓存；
- 超出保留期限的日志。

### 问题四：答案需要来源和证据

仅由模型参数生成的答案，通常难以说明某句话具体来自哪份材料。RAG 可以保留：

> 答案陈述 -> 上下文证据 -> Chunk -> 原始文档与位置

这条映射让用户可以点击引用核对，也让工程团队能够复盘。

不过，存在引用不等于引用正确。常见问题包括：

- 引用文档主题相关，但没有支持那句话；
- 答案混合了两个来源，却只标一个引用；
- 引用的是旧版本；
- 引用片段缺少适用条件；
- 模型给出了并不存在的页码或链接。

因此，引用质量要单独评测，不能只检查答案末尾是否出现来源编号。

### 问题五：降低一部分知识型幻觉

“幻觉”通常泛指模型生成了不受事实或输入支持的内容。它可能来自多种原因：

| 错误来源 | RAG 是否可能帮助 | 原因 |
| --- | --- | --- |
| 模型没见过公司制度 | 可以 | 检索可提供制度证据 |
| 事实在训练后发生变化 | 可以 | 外部知识可更新 |
| 模型算错数学题 | 通常帮助有限 | 问题主要在计算或推理 |
| 源文档金额写错 | 不能直接解决 | RAG 会把错误资料送给模型 |
| 检索没有找到答案 | 不能直接解决 | 生成阶段仍然缺证据 |
| 模型误读了否定条件 | 可能仍会失败 | 正确证据不等于正确理解 |
| 用户需要真实执行退款 | 不能只靠 RAG | 还需要受控业务工具 |

更准确的说法是：

> **RAG 可以降低由外部知识缺失引起的一部分事实错误，但不能消除所有幻觉。**

### 问题六：让知识按用户和场景定制

同一个问题，对不同用户可能有不同答案。

例如：

- 普通员工与高级管理人员的差旅标准不同；
- 中国区和欧洲区适用的制度不同；
- 客户只能查看自己购买产品的手册；
- 售后人员可以查看内部故障库，普通用户不可以。

RAG 可以在查询时结合身份和业务条件选择证据，实现受控的个性化回答。

个性化必须基于可信元数据和权限系统。不能让模型从用户输入的一句“我是管理员”自行决定权限。

### RAG 不解决什么

#### 不会修复缺失或低质量的数据源

如果知识库没有 2026 年制度，系统无法检索到它。把 Top-K 调大、换向量数据库或换更强模型，都不能替代缺失的数据。

#### 不会自动解决复杂推理

问题可能需要计算、跨多份合同推导或访问实时业务状态。RAG 只能提供资料，复杂计算可能还需要程序工具，多跳问题可能需要查询分解或 Agent。

#### 不会自动实现安全

私有知识库仍可能遭遇：

- 越权检索；
- 提示注入；
- 恶意文档投毒；
- 日志泄密；
- 缓存跨租户复用；
- 不安全的外部链接。

安全必须由数据、检索、运行时和治理机制共同实现。

#### 不会保证模型忠实使用证据

模型可能：

- 使用参数记忆覆盖当前资料；
- 忽略“不适用”或“禁止”等词；
- 在证据冲突时自行选择；
- 把多个片段错误拼接；
- 生成证据中没有的细节。

因此仍需提示约束、结构化输出、引用校验、忠实度评测和必要的人工审核。

#### 不等于业务动作

RAG 可以回答“退款政策是什么”，但不能仅靠生成文本完成真实退款。写入订单、付款、发邮件等动作需要受控工具、鉴权、幂等和审计。

### 哪些任务适合 RAG

通常适合：

- 企业知识问答；
- 产品文档助手；
- 客服政策解释；
- 合同和法规检索；
- 带来源的研究辅助；
- 基于最新资料的摘要与比较。

使用前应确认：

1. 答案需要外部知识；
2. 知识可以被收集和索引；
3. 用户问题能通过文本、表格或结构化检索找到证据；
4. 数据来源与权限可以治理；
5. 业务允许模型在证据基础上生成自然语言结果。

### 哪些任务可能不需要 RAG

| 任务 | 更直接的方案 |
| --- | --- |
| 把一段文字改成正式语气 | 直接调用模型 |
| 根据固定公式计算报销金额 | 确定性程序 |
| 查询订单当前支付状态 | 业务数据库或 API |
| 按固定流程审批请假 | Workflow |
| 让模型稳定输出特定格式 | 提示词、结构化输出或微调 |
| 对一份很短且已提供的文档做摘要 | 直接放入上下文 |

这不是说这些任务不能与 RAG 组合，而是不要把检索当成默认必需组件。

### 用四个问题判断是否该上 RAG

1. **知识在哪里？** 是模型通用知识、当前输入，还是外部数据源？
2. **知识多久变化？** 是否需要频繁更新、删除或回滚？
3. **是否需要来源和权限？** 用户能否查看证据，文档是否分级？
4. **错误来自哪里？** 是缺资料、搜不到、理解错，还是需要真正执行动作？

先回答这四个问题，再决定 RAG 是否合适。

## 工程实践与边界

### 为每个目标定义可验证指标

| 目标 | 可观察指标 |
| --- | --- |
| 知识更新更快 | 数据端到可检索端的新鲜度延迟、同步失败率 |
| 接入私有知识 | 目标问题覆盖率、无答案率 |
| 支持追溯 | 引用召回率、引用正确率、来源可打开率 |
| 权限隔离 | 越权检索测试通过率、跨租户泄露事件数 |
| 降低事实错误 | 答案正确性、忠实度、拒答质量 |
| 可维护 | 删除传播时间、索引回滚耗时、版本可复现率 |

不能只看用户点赞率。点赞会受到语气、速度和界面影响，无法直接判断证据是否正确。

### 用“无答案样本”测试边界

评测集不能只有知识库中存在答案的问题，还应包含：

- 知识库没有答案；
- 用户权限不足；
- 问题缺少地区或时间；
- 存在两个互相冲突的版本；
- 用户问题包含错误前提；
- 文档中出现恶意指令。

这些样本用来验证系统是否会澄清、拒答或安全降级。

### 保留权威源与检索副本的边界

知识库索引通常是派生数据。文档平台或业务数据库才是权威源。应监控：

- 权威源和索引的文档数量差异；
- 内容哈希差异；
- 删除传播状态；
- 同步队列积压；
- 失败重试和死信。

### 不把示例数字当生产结论

相似度阈值、Top-K、延迟和成本依赖具体模型、数据、硬件、产品版本与过滤条件。面试中可以讲评测方法，不应编造没有测量条件的“通用准确率”。

## 常见误区

- **“RAG 主要就是解决大模型幻觉”**：这是过度简化。RAG 更核心的价值是接入可更新、可追溯和可治理的外部知识。
- **“知识库有答案，系统就能答对”**：正确答案还要经过解析、切块、召回、过滤、重排和生成，任何一层都可能失败。
- **“模型不知道的都能用 RAG 解决”**：如果资料不存在、问题需要工具执行或主要是复杂推理，RAG 并不充分。
- **“私有知识库天然安全”**：向量、缓存、日志和重排请求都可能携带敏感内容，仍需完整的安全设计。
- **“有引用就代表答案正确”**：引用需要支持对应陈述，并且来自有效、允许访问的版本。
- **“数据更新后重新生成向量就够了”**：还要处理关键词索引、缓存、删除、读别名、版本和回滚。
- **“知识越多，RAG 效果越好”**：低质量、重复、过期和冲突资料会增加检索噪声。

## 面试追问

1. **RAG 为什么比把知识微调进模型更容易更新？**

   因为 RAG 主要更新外部文档和索引，可以按文档 ID 替换、删除和回滚；微调修改的是分布式模型参数，缺少数据库式的精确写入与删除保证。

2. **怎样证明 RAG 提升来自检索，而不是换了模型？**

   固定生成模型和提示词，对比无检索、旧检索和新检索；同时检查正确证据的 Recall@K 与端到端答案指标，并用消融实验隔离变量。

3. **知识库没有答案时应该怎么办？**

   根据风险选择澄清、明确说明资料不足、转人工或调用其他受控数据源。不能把模型参数中的猜测伪装成知识库结论。

4. **实时数据库查询算不算 RAG？**

   如果系统检索外部数据并把结果用于生成，可以具有检索增强特征。但结构化业务状态通常更适合直接查询数据库或 API，不需要先转成向量。

5. **RAG 能否处理用户专属数据？**

   可以，但必须把身份、租户和 ACL 作为可信过滤条件，并处理缓存隔离、日志脱敏和删除传播。不能只依赖提示词。

6. **如何区分知识问题和模型能力问题？**

   检查正确、充分的证据是否已经进入最终上下文。若没有，优先处理数据和检索；若证据完整但模型仍稳定地理解或执行错误，再评估提示、结构约束、工具或微调。
