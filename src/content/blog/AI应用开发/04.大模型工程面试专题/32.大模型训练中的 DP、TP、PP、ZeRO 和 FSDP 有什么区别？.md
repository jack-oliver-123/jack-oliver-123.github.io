假设一家工厂要生产一批很复杂的机器。

现在遇到两个问题：

1. 一台工作台放不下整台机器；
2. 即使勉强放得下，只靠一个工人也要做很久。

工厂可以采用几种不同的分工方式：

- 多个工人各自组装一整台机器，最后同步做法；
- 把一台机器拆成多个部件，让多个工人同时加工；
- 把生产线拆成多道工序，让半成品依次流动；
- 不让每个工人都保存一整套图纸和工具，而是把它们分散保管。

这几种办法，分别对应数据并行、张量并行、流水线并行和状态分片。

大模型分布式训练最容易让初学者迷糊的地方，是这些名词解决的并不是同一个问题。

有的主要缩短训练时间，有的主要让模型能够装进显存，还有的在两者之间做取舍。

## 本篇目标

读完本篇，你应该能够：

- 解释为什么大模型训练需要分布式并行；
- 理解进程、Rank、World Size、节点和设备等基础术语；
- 说清数据并行、张量并行和流水线并行分别“切什么”；
- 解释 All-Reduce、All-Gather 和 Reduce-Scatter 在做什么；
- 区分 ZeRO Stage 1、Stage 2 和 Stage 3；
- 说明 FSDP 与 ZeRO 的关系，而不是把它们当成两个完全相反的概念；
- 理解为什么生产训练经常组合多种并行方式；
- 根据显存、网络拓扑和模型规模给出初步选型；
- 回答通信瓶颈、流水线气泡、全局 Batch Size 和检查点等追问。

先记住本篇最核心的一句话：

> DP 切数据，TP 切层内计算，PP 切模型层；ZeRO 和 FSDP 则通过分片训练状态，减少每张卡上重复保存的内容。

## 60 秒回答

大模型分布式训练通常同时解决两个问题：模型或训练状态放不进单卡，以及单卡训练速度太慢。

数据并行，也就是 DP，会在每个数据并行进程上执行同一个模型，但让各进程处理不同的小批量数据，再同步梯度。它实现简单、计算扩展自然，但普通数据并行会在每张卡上复制完整参数和优化器状态，所以模型本身仍要能放进单卡。

张量并行，也就是 TP，会把同一层中的大矩阵乘法切到多张卡上。它能让单层模型跨卡放置，但每层经常需要集合通信，因此通常优先放在高速互连的同一节点内。

流水线并行，也就是 PP，会把不同层放到不同阶段，再把一个 Batch 切成多个 Micro-batch，让它们像流水线一样前后传递。它减少每张卡保存的层数，但会产生流水线气泡，并增加调度复杂度。

ZeRO 按阶段分片优化器状态、梯度和参数。FSDP 是 PyTorch 提供的全分片数据并行能力，其 FULL_SHARD 策略在目标上与 ZeRO Stage 3 相近，但具体实现、API 和生命周期不能简单画等号。

实际大规模训练常把 DP、TP、PP 组合成 3D Parallelism，再根据网络拓扑、显存、全局 Batch Size、通信量和检查点方案做压测选型。

## 详细解析

### 1. 为什么一张 GPU 往往不够

训练时显存里不只有模型参数。

通常还可能包括：

- 模型参数；
- 参数对应的梯度；
- 优化器状态，例如 Adam 的一阶矩和二阶矩；
- 混合精度训练使用的主权重副本；
- 前向传播保存的激活值；
- 临时计算缓冲区；
- 通信缓冲区；
- 框架、内核和显存分配器的额外开销。

因此，“权重文件只有 20 GB”并不表示 24 GB 显存就能训练。

除了容量，还有训练时间问题。

如果训练数据有数万亿 Token，即使模型能放进一张卡，单卡计算也可能需要不可接受的时间。

所以分布式训练有两个不同目标：

| 目标 | 要解决的问题 | 常见手段 |
| --- | --- | --- |
| 容量扩展 | 单卡放不下模型或训练状态 | TP、PP、ZeRO、FSDP、Offload |
| 计算扩展 | 单卡训练太慢 | DP、TP、PP 及其组合 |

一个方案能提升容量，不代表它也能线性提升速度。

分片越细，常常意味着通信和调度越复杂。

### 2. 先认识几个分布式术语

#### 2.1 Device 与 Node

**Device** 通常指一张 GPU 或其他加速器。

**Node** 指一台物理机或虚拟机，一台 Node 中可以有多张 GPU。

同一 Node 内的 GPU 可能通过 NVLink、NVSwitch 或 PCIe 通信。

不同 Node 之间则可能通过 InfiniBand 或以太网通信。

带宽和延迟差异会直接影响并行策略。

#### 2.2 Process 与 Rank

分布式训练通常会启动多个进程。

每个进程有一个唯一编号，叫作 **Rank**。

全部进程数量叫作 **World Size**。

例如有 2 台机器，每台 8 张 GPU，并且每张 GPU 对应一个训练进程：

```text
Node 0: Rank 0 ~ Rank 7
Node 1: Rank 8 ~ Rank 15
World Size = 16
```

#### 2.3 Process Group

并不是所有通信都需要 16 个进程一起参加。

框架可以把部分 Rank 组成通信组。

例如：

- Rank 0、1、2、3 组成一个张量并行组；
- Rank 4、5、6、7 组成另一个张量并行组；
- 每个张量并行组的对应位置再组成数据并行组。

理解 Process Group，是理解多种并行组合的关键。

### 3. 集合通信到底在同步什么

大模型并行经常出现三个名字：All-Reduce、All-Gather 和 Reduce-Scatter。

不需要先背通信库实现，可以先理解结果。

#### 3.1 All-Reduce

每个参与者先有一份数据。

系统把所有数据求和或做其他归约，再把相同结果发给每个参与者。

```text
Rank 0: [1, 2]
Rank 1: [3, 4]

求和后的 All-Reduce：
Rank 0: [4, 6]
Rank 1: [4, 6]
```

普通数据并行经常用它同步梯度。

上面的例子演示的是“求和”这一种归约。DDP 训练通常还会通过除以数据并行进程数或等价实现，让每个 Rank 最终使用全局 Batch 的平均梯度；具体缩放口径应以框架实现和 Loss Reduction 配置为准。否则把梯度求和、Loss 求平均和梯度累积重复缩放，可能改变有效学习率。

#### 3.2 All-Gather

每个参与者保存一个分片。

All-Gather 会收集全部分片，让每个参与者都得到完整数据。

```text
Rank 0: [A]
Rank 1: [B]

All-Gather 后：
Rank 0: [A, B]
Rank 1: [A, B]
```

参数分片训练中，经常在某层计算前临时收集该层参数。

#### 3.3 Reduce-Scatter

它可以粗略理解为：先对多份数据做归约，再把归约结果切开分给不同参与者。

每个 Rank 最终只保留自己负责的那一片。

ZeRO/FSDP 类方案常用它减少梯度的重复保存。

这些操作的速度会受以下条件影响：

- 参与设备数量；
- 数据量；
- 节点内和节点间带宽；
- 通信延迟；
- 通信是否能与计算重叠；
- 是否出现慢节点或负载不均。

### 4. 数据并行：切分训练数据

数据并行的英文是 Data Parallelism，简称 DP。

它的基本做法是：

1. 每个数据并行进程执行同一个模型；
2. 不同进程读取不同训练样本；
3. 各自完成前向传播和反向传播；
4. 同步梯度；
5. 使用一致的梯度更新参数。

#### 4.1 一个最小例子

假设有 4 张 GPU，每张卡一次处理 8 个样本。

如果不做梯度累积，则一次全局更新看到的样本数为：

$$
B_{global}=B_{micro}\times N_{dp}
$$

其中：

- $B_{global}$ 是全局 Batch Size；
- $B_{micro}$ 是每个数据并行进程一次处理的 Micro Batch Size；
- $N_{dp}$ 是数据并行度。

代入数字：

$$
B_{global}=8\times4=32
$$

如果每 4 次反向传播才更新一次参数，还要乘上梯度累积步数：

$$
B_{global}=B_{micro}\times N_{dp}\times N_{acc}
$$

其中 $N_{acc}$ 是梯度累积步数。

这是面试里经常出现的计算题。

#### 4.2 普通数据并行的优点

- 思路直观；
- 每个进程计算不同数据，容易提高总吞吐；
- 当通信可以被反向计算覆盖时，扩展效率可能较好；
- 对模型代码的改动通常较少。

#### 4.3 普通数据并行的局限

每张卡仍可能保存完整的：

- 模型参数；
- 梯度；
- 优化器状态。

所以如果完整模型和训练状态放不进单卡，只增加普通 DP 卡数并不能直接解决。

#### 4.4 DDP 是什么

PyTorch 的 DistributedDataParallel，简称 DDP，是常见的数据并行实现。

它通常让每个进程持有一份模型副本，并在反向传播时按 Bucket 同步梯度。

“DP”是并行思想，“DDP”是具体框架实现名称。

回答面试题时不要把概念层和产品 API 混在一起。

### 5. 张量并行：切分一层中的矩阵

张量并行的英文是 Tensor Parallelism，简称 TP。

它不是把不同层放到不同卡，而是把**同一层里的大张量计算**拆开。

Transformer 中最主要的计算常来自大矩阵乘法。

假设线性层为：

$$
Y=XW
$$

其中：

- $X$ 是输入激活；
- $W$ 是权重矩阵；
- $Y$ 是输出。

如果 $W$ 太大，可以按列或按行切分。

#### 5.1 按列切分

把权重矩阵写成：

$$
W=[W_1, W_2]
$$

则：

$$
Y=[XW_1, XW_2]
$$

两张卡可以分别计算 $XW_1$ 和 $XW_2$。

#### 5.2 按行切分

把输入和权重沿对应维度切开：

$$
X=[X_1,X_2],\quad W=
\begin{bmatrix}
W_1\\
W_2
\end{bmatrix}
$$

则：

$$
Y=X_1W_1+X_2W_2
$$

两张卡计算局部结果后，需要归约求和。

实际 Transformer 张量并行会设计相邻算子的切分方式，尽量减少中间张量的重复收集。

#### 5.3 TP 的优势

- 单层参数可以跨卡存放；
- 单层矩阵乘法可以由多卡共同完成；
- 对特别宽的模型很重要；
- 与数据并行、流水线并行可以组合。

#### 5.4 TP 的代价

同一层的前向和反向计算中可能多次发生通信。

如果把 TP 组跨到低带宽网络，通信时间可能抵消计算收益。

因此常见经验是：

> 优先把通信频繁的 TP 放在高速互连域内，再用 DP 或 PP 跨节点扩展。

这是一条工程经验，不是对所有硬件都成立的固定规则。

#### 5.5 Sequence Parallelism 又是什么

张量并行后，有些激活仍会在 TP Rank 之间重复。

Sequence Parallelism 会沿序列维度分片部分计算和激活，以进一步降低重复激活显存。

它常与 TP 配合，但不是“把所有注意力都沿序列切开”的同义词。

不同框架对 Sequence Parallel、Context Parallel 的命名和支持范围可能不同，选型时要回到所用版本的文档。

### 6. 流水线并行：切分模型层

流水线并行的英文是 Pipeline Parallelism，简称 PP。

假设一个 Transformer 有 24 层，可以分成 4 个阶段：

```text
Stage 0: Layer  1 ~  6
Stage 1: Layer  7 ~ 12
Stage 2: Layer 13 ~ 18
Stage 3: Layer 19 ~ 24
```

输入先经过 Stage 0，再把激活发给 Stage 1，依次向后传播。

反向传播则按相反方向传递梯度。

#### 6.1 为什么还要切 Micro-batch

如果整批数据一次进入流水线，Stage 0 计算完后会等待后续阶段，设备利用率很低。

因此会把一个 Batch 切成多个 Micro-batch：

```text
时间 --->

Stage 0: M1前向 M2前向 M3前向 M4前向 ...
Stage 1:        M1前向 M2前向 M3前向 ...
Stage 2:               M1前向 M2前向 ...
Stage 3:                      M1前向 ...
```

多个 Micro-batch 同时处于不同阶段，流水线才能逐渐填满。

#### 6.2 什么是流水线气泡

流水线刚启动和结束时，总有一些阶段处于等待状态。

这部分空闲时间叫作 Pipeline Bubble，也就是流水线气泡。

在非常简化的同步流水线模型中，如果有 $p$ 个阶段、$m$ 个 Micro-batch，气泡比例可近似写成：

$$
\text{bubble ratio}\approx\frac{p-1}{m+p-1}
$$

其中：

- $p$ 是流水线阶段数；
- $m$ 是 Micro-batch 数。

增加 Micro-batch 数通常能降低相对气泡，但也会影响全局 Batch、激活显存和调度。

实际 1F1B、Interleaved Pipeline 等调度的气泡和显存公式会不同，不能只套这一条近似式。

#### 6.3 1F1B 是什么

1F1B 指在流水线稳定阶段交替执行一次前向和一次反向。

与先做完全部前向再做全部反向相比，它可能减少同时保留的激活数量。

具体收益取决于流水线调度、阶段划分和框架实现。

#### 6.4 PP 的难点

- 各阶段计算量可能不均衡；
- 阶段边界要传输激活和梯度；
- Micro-batch 太少会产生较大气泡；
- 层数不一定能平均整除；
- Embedding、输出头或 MoE 层可能造成阶段热点；
- 调试和恢复比单卡训练更复杂。

所以 PP 不只是“平均分层数”。

更重要的是尽量平衡每个 Stage 的实际执行时间和显存。

### 7. ZeRO：把重复的训练状态分片

ZeRO 的全称是 Zero Redundancy Optimizer。

它观察到：普通数据并行虽然切了数据，但每个 Rank 仍保存大量相同训练状态。

ZeRO 的核心思路是：

> 既然多个数据并行 Rank 在合作训练，就不必让每个 Rank 永久保存全部重复状态。

为了便于理解，可以把训练状态分成三类：

- 优化器状态；
- 梯度；
- 模型参数。

#### 7.1 ZeRO Stage 1

Stage 1 分片优化器状态。

每个数据并行 Rank 只负责一部分参数对应的优化器状态和更新。

模型参数和梯度仍可能是完整副本。

它主要降低 Adam 一阶矩、二阶矩等优化器状态造成的重复显存。

#### 7.2 ZeRO Stage 2

Stage 2 在 Stage 1 基础上继续分片梯度。

每个 Rank 不再长期保存完整梯度副本。

参数仍通常保持复制。

#### 7.3 ZeRO Stage 3

Stage 3 进一步分片模型参数。

每个 Rank 常驻保存一部分参数。

某层即将计算时，框架收集需要的参数；计算结束后可以再次释放非本地分片。

这显著降低单卡常驻参数显存，但会增加参数收集通信和生命周期管理。

#### 7.4 三个阶段对比

| 方案 | 优化器状态 | 梯度 | 参数 | 典型取舍 |
| --- | --- | --- | --- | --- |
| 普通 DP/DDP | 每卡复制 | 每卡复制 | 每卡复制 | 简单，但重复显存多 |
| ZeRO-1 | 分片 | 每卡复制 | 每卡复制 | 优先减少优化器状态 |
| ZeRO-2 | 分片 | 分片 | 每卡复制 | 进一步减少梯度显存 |
| ZeRO-3 | 分片 | 分片 | 分片 | 容量更强，参数通信和调度更复杂 |

这张表表达的是概念模型。

不同实现可能有预取、持久化阈值、重叠通信、混合精度副本和 Offload 等细节。

#### 7.5 Offload 是什么

当 GPU 显存仍不足时，可以把部分参数或优化器状态放到 CPU 内存，甚至更慢的存储层级。

这样可以扩大可训练模型规模，但数据搬运可能成为瓶颈。

Offload 是容量和速度之间的交换，不等于“免费增加显存”。

### 8. FSDP 与 ZeRO 到底是什么关系

FSDP 是 Fully Sharded Data Parallel 的缩写。

它是 PyTorch 提供的一类全分片数据并行能力。

FSDP 会围绕模块管理参数分片、参数收集、梯度归约和状态字典等过程。

从目标上看：

- FSDP 的 FULL_SHARD 会分片参数、梯度和优化器状态，与 ZeRO Stage 3 的目标相近；
- 某些只在反向后分片梯度和优化器状态的策略，与 ZeRO Stage 2 有相似之处。

但面试中更严谨的说法是：

> ZeRO 描述了一套消除数据并行冗余的分阶段思想；FSDP 是 PyTorch 中实现全分片训练的一套 API 和运行机制。二者可以对照理解，但不能假设所有配置、通信时机和检查点语义逐项完全相同。

选用 FSDP 时还要考虑：

- 如何划分和包装模块；
- 参数何时 All-Gather；
- 是否使用 Mixed Precision；
- CPU Offload 是否适合负载；
- 原始参数视图和优化器如何配合；
- 保存完整还是分片 State Dict；
- 所用 PyTorch 版本的 API 状态。

### 9. 为什么经常组合成 3D Parallelism

单一并行方式很难同时解决所有约束。

因此大规模训练常把三种维度组合：

$$
N_{world}=N_{dp}\times N_{tp}\times N_{pp}
$$

其中：

- $N_{world}$ 是总设备数；
- $N_{dp}$ 是数据并行度；
- $N_{tp}$ 是张量并行度；
- $N_{pp}$ 是流水线并行度。

例如有 64 张 GPU：

```text
TP = 4
PP = 2
DP = 8

4 × 2 × 8 = 64
```

可以把它理解为：

1. 4 张卡一起完成每一层中的张量计算；
2. 模型层再切成 2 个流水线阶段；
3. 这样的模型副本组再复制 8 份，处理不同数据。

如果数据并行部分使用 ZeRO/FSDP，DP 组内还会进一步分片训练状态。

现代系统还可能加入：

- Expert Parallelism：把 MoE 专家分散到不同设备；
- Context Parallelism：切分长序列上下文计算；
- Sequence Parallelism：减少部分激活重复；
- Sharded Data Parallelism：分片数据并行训练状态。

这些维度的命名和可组合方式依赖框架版本，不能只看缩写做配置。

### 10. 如何根据问题选择并行方式

可以按以下顺序思考。

#### 情况一：完整训练状态能放进单卡，只是训练太慢

可以先考虑 DDP。

它通常比复杂模型并行更容易调试。

然后通过通信分析确认梯度同步是否成为瓶颈。

#### 情况二：模型参数能放进单卡，但 Adam 状态和梯度放不下

可以评估 ZeRO-1、ZeRO-2 或相应分片策略。

是否需要直接使用最强分片，要通过通信、吞吐和实现复杂度验证。

#### 情况三：单层矩阵本身太大，单卡无法容纳或计算效率很低

需要考虑 TP。

优先评估高速互连范围、矩阵尺寸和内核效率。

#### 情况四：整个模型层数很多，各层单独能放下

可以考虑 PP，把不同层放到不同 Stage。

同时评估 Micro-batch 数、阶段平衡和气泡。

#### 情况五：模型和集群都很大

通常需要组合 DP、TP、PP、参数分片，甚至专家并行或上下文并行。

此时选型不再是“哪个名词更高级”，而是一个约束优化问题。

### 11. 一张选型对照表

| 方法 | 主要切分对象 | 主要缓解什么 | 主要通信 | 常见风险 |
| --- | --- | --- | --- | --- |
| DP/DDP | 训练样本 | 训练吞吐 | 梯度 All-Reduce | 完整状态重复 |
| TP | 层内张量 | 单层容量与计算 | 层内频繁集合通信 | 对互连敏感 |
| PP | 模型层 | 总模型容量 | 阶段间激活与梯度 | 气泡、负载不均 |
| ZeRO-1 | 优化器状态 | 优化器显存 | 更新后的参数同步 | 收益依状态占比 |
| ZeRO-2 | 优化器状态和梯度 | 更多训练状态显存 | Reduce-Scatter 等 | 通信调度更复杂 |
| ZeRO-3/FSDP 全分片 | 参数、梯度、优化器状态 | 单卡常驻状态 | 参数 All-Gather、梯度归约 | 通信、预取、检查点复杂 |

这张表适合面试时快速组织答案，但真正部署还要结合模型和框架压测。

### 12. 全局 Batch Size 为什么容易算错

组合并行时，只有数据并行维度复制了不同数据。

TP 和 PP 的多张卡是在合作处理同一批数据，不应重复乘进全局 Batch。

假设：

- 总 GPU 数为 64；
- TP = 4；
- PP = 2；
- 因而 DP = 8；
- 每个流水线 Micro-batch 为 2 个样本；
- 梯度累积 16 次。

则：

$$
B_{global}=2\times8\times16=256
$$

不能写成 $2\times64\times16$，因为 TP 和 PP Rank 没有各自读取一份独立数据。

如果使用的是按 Token 打包的训练数据，还要同时记录每次更新的有效 Token 数，而不能只报样本数。

### 13. 并行之后，数值结果会不会变化

理论上，多种并行方式都希望实现与目标单卡计算等价的结果。

但工程上可能出现细微差异：

- 浮点加法顺序改变；
- 不同并行度改变随机数消费顺序；
- Dropout 或数据顺序不一致；
- 梯度累积和损失归一化口径错误；
- 混合精度溢出处理不同；
- 分片参数初始化或加载不一致。

所以切换并行策略后，应做小规模等价性测试：

- 对比前几步 Loss；
- 检查梯度范数；
- 固定种子与数据顺序；
- 验证有效 Batch 和学习率调度；
- 检查断点续训能否复现连续训练轨迹。

“能运行”不代表并行实现就是正确的。

### 14. 检查点为什么会变复杂

普通单卡训练可以直接保存完整模型和优化器状态。

分片训练中，每个 Rank 可能只拥有一部分状态。

因此检查点至少要回答：

- 保存完整权重还是分片权重；
- 谁负责聚合完整权重；
- 是否保存优化器分片；
- 是否保存学习率调度器和随机数状态；
- 恢复时能否更改 World Size；
- 如何处理写到一半的失败检查点；
- 如何验证所有分片属于同一个训练 Step。

大模型检查点很大，集中到 Rank 0 可能产生内存峰值和 I/O 热点。

生产系统通常需要分布式检查点、原子发布、校验和以及保留策略。

## 工程实践与边界

### 1. 先做单卡或小规模基线

在扩展到大量 GPU 前，先得到可解释的基线：

- 每步耗时；
- 峰值显存；
- 每秒 Token；
- 前向、反向、优化器时间；
- 数据加载时间；
- Loss 和梯度范数。

没有基线，就很难判断扩展后变慢是通信、数据、内核还是配置问题。

### 2. 按硬件拓扑放置通信组

先画出集群拓扑：

```text
机内：GPU -- NVLink/NVSwitch/PCIe -- GPU
机间：Node -- InfiniBand/Ethernet -- Node
存储：Node -- Network Storage/Object Storage
```

通信频繁的 TP 往往更依赖机内高速链路。

跨节点 DP 能否扩展，则与梯度大小、网络带宽和通信重叠有关。

不要只填一个 TP/PP/DP 数字，不看 Rank 到物理 GPU 的映射。

### 3. 逐层增加复杂度

一种更容易定位问题的顺序是：

1. 先验证单卡；
2. 再验证单节点 DDP；
3. 如果容量不足，再引入状态分片；
4. 单层仍放不下时评估 TP；
5. 层数和总容量仍受限时评估 PP；
6. 最后组合并做大规模扩展测试。

这不是唯一顺序，但它能减少一次引入过多变量的风险。

### 4. 监控计算、通信和等待

只看 GPU 利用率不够。

至少还应观察：

- 每步总时间及各阶段分解；
- All-Reduce、All-Gather、Reduce-Scatter 耗时；
- 计算与通信重叠比例；
- 每个 Rank 的最大和最小耗时；
- 流水线阶段空闲比例；
- 网络吞吐和重传；
- 数据加载等待；
- 显存峰值和 OOM；
- 每秒有效训练 Token；
- 故障恢复时间。

平均值可能掩盖一个慢 Rank 拖住整个同步训练的问题。

### 5. 学习率不能只因 GPU 变多就机械放大

增加 DP 往往会改变全局 Batch Size。

这可能需要重新评估：

- 学习率；
- Warmup；
- 梯度裁剪；
- 训练步数；
- 数据顺序；
- 收敛和泛化。

线性学习率缩放是一种经验起点，不是跨模型、跨优化器都成立的定律。

### 6. 不要把扩展效率和峰值 FLOPS 混为一谈

强扩展和弱扩展是两种不同口径。

- **强扩展**：总工作量不变，增加设备看能缩短多少时间；
- **弱扩展**：每张设备工作量近似不变，增加设备同时扩大总工作量。

报告扩展效率时要说明基线、Batch、序列长度、精度、硬件和统计区间。

否则两个“90% 扩展效率”可能不是同一个概念。

### 7. 框架名称不能替代容量计算和压测

DeepSpeed、PyTorch FSDP、Megatron Core 等框架提供不同组合能力。

最终仍需回答：

- 每张卡常驻哪些状态；
- 每步发生哪些集合通信；
- 通信组如何映射到拓扑；
- 激活和临时缓冲占多少；
- 检查点如何保存与恢复；
- 当前锁定版本支持哪些模型结构。

选框架是工程决策，不是只比较功能清单。

## 常见误区

- **GPU 数量翻倍，训练速度就会翻倍**：同步、通信、气泡、数据读取和小矩阵效率都会影响扩展
- **数据并行会把模型参数自动切开**：普通 DDP 通常仍在每个 Rank 保存完整模型副本
- **TP 就是把不同层放到不同卡**：切不同层是 PP，TP 主要切同一层内的张量计算
- **PP 只要平均分层数即可**：真正需要平衡的是执行时间、激活显存和通信，而不是层数本身
- **ZeRO-3 在所有场景都比 ZeRO-2 快**：更强的分片降低常驻显存，也可能增加参数收集和调度成本
- **FSDP 与 ZeRO 毫无关系**：二者在全分片目标上有明显联系，只是概念、API 和实现不能逐项等同
- **TP 和 PP 也要乘进全局 Batch**：全局 Batch 主要由数据并行度、Micro Batch 和梯度累积决定
- **通信带宽高就不会慢**：延迟、消息大小、拓扑、拥塞和计算重叠同样重要
- **检查点只是调用一次 save**：分片状态的一致性、聚合峰值、I/O 和跨并行度恢复都需要设计

## 面试追问

**问：模型能放进单卡，但优化器状态放不下，你会先考虑什么？**

**答：** 我会先按参数、梯度、优化器状态和激活分别估算显存，确认瓶颈。如果主要是 Adam 状态和梯度重复，可以评估 ZeRO-1/2 或相应的 FSDP 分片策略，而不一定立即引入层内 TP。然后用真实 Batch 和网络条件比较吞吐、峰值显存与检查点复杂度。

**问：为什么张量并行通常不优先跨低速网络？**

**答：** TP 把同一层的计算拆到多卡，层与层之间会反复发生集合通信。消息出现在关键计算路径上，低带宽或高延迟会让设备频繁等待。因此通常先把 TP 组放在高速互连域内，但最终仍要根据矩阵规模、框架通信方式和拓扑压测。

**问：64 张卡，TP=4、PP=2，每卡 Micro Batch=2，梯度累积 8 次，全局 Batch 是多少？**

**答：** 数据并行度是 $64/(4\times2)=8$，所以全局 Batch 为 $2\times8\times8=128$。TP 和 PP 的卡是在合作处理同一批数据，不能再次乘入全局 Batch。

**问：如何判断分布式训练扩展得不好？**

**答：** 我会先统一工作量和指标口径，再分解每步时间，比较计算、集合通信、流水线气泡、数据等待和慢 Rank。还会看每秒有效 Token、P95/P99 Step Time、网络流量和显存峰值。只有知道时间花在哪里，才能决定调整 Bucket、并行度、Micro-batch、Rank 映射还是数据管道。
