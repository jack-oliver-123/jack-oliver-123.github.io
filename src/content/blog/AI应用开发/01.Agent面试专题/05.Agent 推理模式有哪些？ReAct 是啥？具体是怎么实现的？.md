## 本篇目标

真正看懂 ReAct 的一次完整循环，分清模型、工具和宿主程序各自做什么。读完后，你应该能解释 ReAct 如何执行、如何处理失败，以及如何停止。

## 60 秒回答

Agent 常见的决策方式包括一次直接生成、分步推理、搜索多条候选路径、ReAct，以及先规划再执行。ReAct 的核心是把判断和外部行动交错进行：模型读取目标与当前观察，选择一个动作；宿主运行时校验并执行工具，把结果作为新的观察写回状态；模型再根据最新结果决定继续、改道、向用户澄清或结束。

ReAct 的循环不是模型自己常驻运行，而是应用程序反复调用模型。生产实现通常让模型输出结构化动作和简短可审计理由，不依赖或公开私有思维链。运行时还必须负责 Schema 校验、权限、错误分类、超时、有限重试、幂等、预算和程序化停止条件。

## 详细解析

### 先说人话：边走边看地图

假设你第一次去一家餐厅：

1. 先看地图，决定坐地铁；
2. 到站后发现出口关闭；
3. 根据新情况改走另一个出口；
4. 出站后查看门牌；
5. 找到餐厅，停止导航。

你没有在出发前预测所有情况，而是每走一步就观察环境，再决定下一步。ReAct 就是类似的“动作—观察—再决策”方式。

传统固定流程更像提前写好：

> 坐 2 号线 -> 从 A 口出 -> 向东走 200 米。

路径稳定时固定流程更简单；当环境反馈经常改变下一步时，ReAct 才有价值。

### ReAct 这个名字是什么意思

ReAct 来自 Reasoning 和 Acting，也就是把推断与行动结合起来。原始论文通常用三个词描述：

- **Thought**：对当前任务的内部判断；
- **Action**：准备执行的动作；
- **Observation**：动作执行后的环境结果。

工程实现不需要展示模型的完整私有 Thought。系统真正需要的是：

- 当前目标和已验证事实；
- 模型选择的结构化 Action；
- 工具返回的 Observation；
- 必要的简短决策理由；
- 运行时维护的状态和停止原因。

因此可以把生产 ReAct 记成：

> 状态 -> 动作请求 -> 受控执行 -> 观察结果 -> 状态更新 -> 下一轮或结束。

### 用“查询订单并处理退款”走一遍

用户说：“帮我退掉昨天买的耳机。”

#### 第 1 轮：发现信息不足

当前状态：

- 目标：退款；
- 已知：商品是耳机，购买时间是昨天；
- 缺失：订单号。

模型输出：

> 动作：向用户询问订单号。

用户提供订单号 12345，运行时把它写入状态。

#### 第 2 轮：查询订单

模型输出结构化动作：

> 工具：查询订单；参数：订单号 12345。

运行时执行前检查：

- 当前用户是否已登录；
- 订单号格式是否合法；
- 工具是否属于只读操作；
- 本轮是否还有调用预算。

工具观察：

> 订单存在，属于当前用户，金额 299 元，状态为已收货。

#### 第 3 轮：查询退款政策

模型发现还不能直接退款，于是调用政策检索工具。

观察结果：

> 收货 7 天内可申请退款；已拆封电子产品需要质量问题证明。

模型据此询问用户是否已经拆封，以及具体故障。

#### 第 4 轮：申请审批

用户提供故障描述后，模型提出创建退款申请。运行时发现这是写操作，先展示：

- 订单号；
- 退款金额；
- 退款原因；
- 预计影响。

根据政策进入人工审批。

#### 第 5 轮：执行并验证

审批通过后，运行时使用幂等键调用退款工具。工具返回退款单号，系统再查询一次状态确认成功，随后回复用户并结束。

这就是完整闭环。不是模型“想了一次”就完成，而是每个新结果都会影响下一步。

### 循环究竟是谁驱动的

很多初学者会认为模型在后台一直思考，其实不是。

每一次模型调用都类似：

> 应用程序发送上下文 -> 模型返回文本或工具调用 -> 本次调用结束。

随后宿主程序：

1. 解析模型结果；
2. 校验动作；
3. 执行工具；
4. 保存观察；
5. 再次调用模型。

所以“模型提出调用工具”和“工具已经执行”是两件事。即使模型输出了正确的退款参数，如果运行时权限校验失败，也不能执行。

### 一次工程化 ReAct 循环包含什么

#### 第一步：组装当前上下文

只提供本轮需要的信息：

- 用户目标；
- 当前步骤；
- 已验证事实；
- 未完成事项；
- 必要的工具说明；
- 剩余步数、时间和费用预算；
- 不可违反的政策。

不要每轮都塞入全部历史，否则上下文会越来越长，旧错误也可能不断传播。

#### 第二步：调用模型

模型应从有限动作中选择：

- 回复用户；
- 请求补充信息；
- 调用某个工具；
- 请求人工审批；
- 返回完成或无法完成。

动作最好使用结构化输出，避免从一段自然语言里用正则猜工具名和参数。

#### 第三步：校验动作

运行时检查：

- 工具是否真实存在；
- 参数是否符合 Schema；
- 当前用户是否有权限；
- 资源是否属于当前租户；
- 是否触发高风险审批；
- 是否重复了刚刚的动作；
- 是否已经超过预算。

#### 第四步：执行工具

工具执行要有超时、取消、限流和关联 ID。写操作还要考虑幂等和状态查询。

#### 第五步：记录观察

观察不仅包括成功结果，还包括可操作的错误。例如：

> 错误类型：参数错误；字段：订单号；原因：长度不合法。

这比只返回“调用失败”更有利于下一轮正确处理。

#### 第六步：判断继续还是停止

程序检查：

- 任务是否完成；
- 是否达到最大步数或截止时间；
- 是否连续多轮没有进展；
- 是否出现不可恢复错误；
- 是否必须转人工；
- 是否被用户取消。

只有没有触发停止条件时，才进入下一轮。

### 工具错误不能都用“重试”解决

| 错误类型 | 例子 | 正确处理 |
| --- | --- | --- |
| 参数错误 | 订单号格式不对 | 修正参数或询问用户 |
| 权限错误 | 用户无权读取订单 | 拒绝或转授权流程，不能重试绕过 |
| 可重试故障 | 服务临时 503 | 有限次数退避重试 |
| 限流 | 调用过于频繁 | 按提示等待或降级 |
| 永久业务错误 | 订单已取消 | 改变计划，不再重复调用 |
| 写入结果不确定 | 退款请求超时 | 用幂等键或先查询外部状态 |

“返回 500 就重试”也不完整。要知道工具是否幂等、错误是否临时，以及第一次写操作是否可能已经成功。

### ReAct 为什么有效

主要优势有三个：

1. **使用真实反馈纠偏**：搜索不到就换关键词，测试失败就修改方案。
2. **不要求提前知道完整路径**：适合诊断、搜索和开放任务。
3. **每一步都能观察**：比一次性生成长计划更容易定位失败节点。

### ReAct 为什么会失控

典型问题包括：

- 反复调用同一个工具；
- 只处理局部问题，忘记最终目标；
- 工具结果越来越多，上下文膨胀；
- 把网页中的恶意文字当成指令；
- 写操作超时后重复执行；
- 模型一直认为“再试一次可能成功”。

解决这些问题不能只靠提示模型“更谨慎”，还需要程序守卫：

- 最大轮数；
- 重复动作检测；
- 无进展检测；
- 工具 allowlist；
- Token 和费用预算；
- 高风险审批；
- 强制转人工条件。

### ReAct 适合和不适合什么任务

适合：

- 搜索资料并根据结果继续追查；
- 故障诊断；
- 客服信息收集；
- 代码修改后根据测试结果迭代；
- 工具路径无法提前穷举的任务。

不太适合：

- 固定的三步数据处理；
- 必须严格重复同一路径的合规流程；
- 延迟极低的一次性分类；
- 每一步都有重大不可逆副作用的任务。

如果任务有很长的依赖链，可以先用 Planner 生成高层计划，再让 ReAct 只负责执行当前步骤。

## 工程实践与边界

- **动作结构化**：让模型返回工具名、参数和动作类型，不从自由文本中猜调用。
- **观察可操作**：工具返回明确错误码、是否可重试和必要的上下文，不把内部堆栈直接塞给模型。
- **写操作幂等**：付款、退款、发信和创建资源携带稳定幂等键，超时后先查询外部状态。
- **程序强制停止**：最大步数、时间、费用、重复动作和无进展轮次都由运行时检查。
- **上下文做压缩**：保留目标、硬约束和关键事实，旧工具原文外置并按需检索。
- **工具结果按不可信数据处理**：标记来源，过滤无关内容，防止间接提示注入。
- **保留可回放轨迹**：记录状态摘要、调用 ID、脱敏参数、结果、延迟、模型和提示版本。

## 常见误区

- **“模型自己在循环”**：每次模型调用都会结束，循环由宿主应用再次发起。
- **“ReAct 就是展示完整思维链”**：生产系统只需要动作、观察、计划和可审计的简短理由。
- **“一次工具调用就是 ReAct”**：至少要有观察结果回到系统，并影响后续决策。
- **“报错就重试”**：权限错误和业务错误不会因为重试而消失，写操作还可能重复产生副作用。
- **“轮数越多越智能”**：无进展轮次只会增加延迟、成本和失控风险。
- **“工具返回的内容都可信”**：网页和文档可能包含错误或恶意指令，不能升级为系统政策。

## 面试追问

> **面试官：** ReAct 的核心循环是什么？
>
> **候选人：** 模型根据当前状态选择动作，运行时校验并执行工具，把结果作为观察写回状态，再让模型继续决策，直到完成或触发程序化停止条件。

> **面试官：** ReAct 如何停止？
>
> **候选人：** 模型可以返回完成状态，但运行时必须同时强制最大步数、时间、费用、重复动作、无进展和政策终止条件，不能只相信模型自报完成。

> **面试官：** 工具返回 500 怎么办？
>
> **候选人：** 先根据工具契约判断是否属于临时且可安全重试的故障，再做有限次数的退避重试。写操作还要依赖幂等键或状态查询，超过阈值就降级或转人工。

> **面试官：** 如何避免 ReAct 重复调用同一个工具？
>
> **候选人：** 保存规范化后的动作指纹和结果，检测连续重复与状态是否有进展；重复达到阈值后停止、换策略或转人工。

> **面试官：** 如何调试一条错误轨迹？
>
> **候选人：** 找到第一个偏离预期的步骤，查看当时的状态摘要、工具定义、动作参数、结果、错误类型、模型和提示版本，再用固定输入和模拟工具回放。
