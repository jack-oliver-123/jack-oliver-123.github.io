## 本篇目标

第 10 篇解释了 Multi-Agent 的概念，这一篇重点解决“实际项目怎么选、怎么画、怎么落地”。读完后，你应该能够：

- 画出一套 Single-Agent 的最小生产架构，并解释每个模块的职责。
- 画出中心编排式 Multi-Agent 架构，说明 Orchestrator 与 Worker 的边界。
- 用并行性、上下文、权限、可靠性和成本做选型，而不是凭“任务复杂”判断。
- 知道如何先做单 Agent 基线，再根据真实失败轨迹逐步拆分。
- 回答编排器故障、Agent 通信成本、跨 Agent 写操作等进阶追问。

## 60 秒回答

我的默认方案是先做 Single-Agent 基线：一个 Agent 在受控循环中读取任务状态、选择工具、接收结果并继续决策，外层运行时负责权限、超时、预算、检查点和停止条件。它链路短、状态集中、容易评测和排查，复杂步骤也可以用确定性 Workflow 包住，不必因为步骤多就拆 Agent。

只有出现可验证的边界时，我才考虑 Multi-Agent，例如多个独立耗时任务需要并行，不同业务域需要隔离上下文或真实权限，或者单一上下文长期被无关指令和资料干扰。生产中通常优先中心编排：Orchestrator 持有任务图和共享状态，Worker 只接收有界任务，返回结构化结果和证据。

选型时我会固定模型、工具和测试集，对比任务成功率、关键路径延迟、Token 与费用、越权事件、故障恢复和维护成本。多 Agent 如果没有明显净收益，就保留单 Agent；若要拆，也按失败数据逐个拆，不一次性设计大量角色。

## 详细解析

### 先说人话：一个人办事，还是一个团队办事

Single-Agent 的思路是：“让一个负责人掌握完整任务，在需要时调用不同工具。”

Multi-Agent 的思路是：“把任务拆给多个有明确边界的负责人，再设计一个协作机制。”

二者没有固定高下。一个人做事沟通少、责任清楚，但可能信息太多、能力边界太宽；团队可以专业分工和并行，但会产生交接、冲突和协调成本。

面试中最成熟的回答通常不是“复杂就用多 Agent”，而是：

> 先建立最简单、可评测的基线，再用真实瓶颈证明拆分价值。

### 贯穿案例：设计一个售后客服系统

系统需要完成这些任务：

- 回答商品和售后政策问题；
- 查询订单和物流；
- 判断是否符合退款条件；
- 对符合条件的订单发起退款；
- 高金额或争议订单转人工。

我们先分别设计 Single-Agent 和 Multi-Agent 方案，再比较。

### 一、Single-Agent 设计方案

#### 1. 最小架构

```text
用户请求
   ↓
输入校验与身份认证
   ↓
Agent 运行时
   ├─ 模型：理解任务、提出下一步动作
   ├─ 状态：目标、消息、计划、已完成步骤
   ├─ 工具注册表：查订单、查政策、申请退款
   └─ 控制循环：决策 → 校验 → 执行 → 观察
   ↓
输出校验与用户响应

外围：权限策略、检查点、日志追踪、预算、人工审批
```

这里最重要的是：模型不是整个系统。模型只提出“下一步想做什么”，运行时才决定是否允许并真正执行。

例如模型输出：

```json
{
  "action": "request_refund",
  "arguments": {"order_id": "A123", "amount": 100}
}
```

运行时仍要检查：

- 当前身份是否有权处理这个订单；
- 订单是否满足退款政策；
- 金额是否与权威系统一致；
- 是否需要人工审批；
- 本次请求是否已经执行过。

#### 2. Single-Agent 的状态机

即使只有一个 Agent，也不应让它无限自由循环。可以用外层状态机限定主要阶段：

```text
收集信息
  ↓
查询并判断
  ↓
需要补充？ ──是→ 询问用户
  ↓否
需要审批？ ──是→ 等待人工
  ↓否或已通过
执行动作
  ↓
验证结果并答复
```

每个阶段内部，Agent 可以动态决定使用哪个只读工具；高风险写操作仍由程序节点和审批控制。这样既保留适度灵活性，又把风险限制在明确边界内。

#### 3. Single-Agent 的优点

- 一个状态源，信息不需要在多个角色间复制；
- 调用链短，延迟和 Token 成本通常更低；
- 一次失败轨迹容易完整回放；
- 工具权限和停止条件更容易集中治理；
- 对小团队来说，开发和运维负担较低。

#### 4. Single-Agent 的局限

- 所有业务说明、工具和历史都挤进同一上下文时，模型容易受无关信息干扰；
- 工具很多时，选错工具的概率可能增加；
- 独立的耗时子任务如果串行执行，会拉长延迟；
- 一个角色同时拥有大量读写能力，会扩大风险面；
- 不同领域的提示和输出要求可能互相冲突。

注意，这些只是“可能的瓶颈”，不是看到其中一条就立即拆成多 Agent。先尝试检索、工具分组、上下文筛选和确定性 Workflow，往往更便宜。

### 二、Multi-Agent 设计方案

#### 1. 初学者优先选择中心编排

```text
                         ┌→ 知识 Worker：查政策 ─┐
用户 → 接入层 → Orchestrator ├→ 订单 Worker：查订单 ─┤→ 验收/聚合 → 用户
                         ├→ 退款 Worker：请求退款 ─┤
                         └→ 人工审批节点 ─────────┘

共享基础设施：任务存储、消息队列、权限策略、追踪、预算
```

职责分配可以是：

| 组件 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| Orchestrator | 拆分任务、调度、维护依赖、验收和汇总 | 不直接拥有所有高风险工具 |
| 知识 Worker | 检索政策并返回条款与来源 | 不判断订单真实状态 |
| 订单 Worker | 只读查询订单和物流 | 不执行退款 |
| 退款 Worker | 在授权范围内发起退款请求 | 不自行修改审批规则 |
| 人工审批 | 高金额、争议或低置信任务决策 | 不承担普通自动查询 |

Orchestrator 应持有全局任务图和共享状态的唯一所有权。Worker 处理有界子任务，返回结构化产物和证据，不直接随意改写全局状态。

#### 2. 任务图怎样表示

以“订单 A123 退款”为例：

```text
T1 查询订单 ─┐
              ├→ T3 判断退款资格 → T4 审批 → T5 执行退款 → T6 验证并通知
T2 查询政策 ─┘
```

T1 和 T2 互不依赖，可以并行；T3 必须等两者完成；T5 必须等审批结果。这是一个有向无环图，也常写作 DAG。

每个任务节点至少应有：

```json
{
  "task_id": "T3",
  "owner": "refund_policy_worker",
  "dependencies": ["T1", "T2"],
  "inputs": ["order_snapshot", "policy_evidence"],
  "expected_output": "EligibilityDecision",
  "acceptance_criteria": ["有政策版本", "有订单证据", "结论可解释"],
  "timeout_seconds": 30,
  "status": "pending"
}
```

#### 3. Handoff 方案什么时候更合适

如果任务不是预先可拆的图，而是会话在不同领域间流转，可以用 Handoff。例如普通客服识别到企业客户后，把控制权转给企业客服。

Handoff 设计至少要有：

- 明确的路由条件；
- 传递的最小上下文；
- 接收确认；
- 超时与失败回退；
- 最大转交次数；
- 最终责任主体。

如果业务能提前画出清楚的任务图，中心编排通常比连续 Handoff 更容易追踪。

### 三、Single-Agent 与 Multi-Agent 的核心对比

| 维度 | Single-Agent | Multi-Agent |
| --- | --- | --- |
| 状态管理 | 集中，容易理解 | 有私有与共享状态，需定义所有权 |
| 通信成本 | 低 | 需要序列化、传递、等待和重试 |
| 上下文 | 可能变大、相互干扰 | 可按职责隔离，但交接可能丢信息 |
| 并行能力 | 可由 Workflow 并行工具调用 | 适合独立 Worker 并行，但协调更复杂 |
| 权限面 | 一个 Agent 可能看到较多工具 | 可分域，但必须有真实运行时隔离 |
| 调试 | 链路短，容易回放 | 需要跨 Agent trace 和因果关联 |
| 可靠性 | 故障点较少 | 多出编排、消息、部分失败等故障面 |
| 成本 | 通常较低 | 模型调用和上下文复制通常更多 |
| 适用起点 | 默认起点 | 有明确拆分收益后采用 |

### 四、怎样做选型

#### 第一步：先问能不能用 Workflow

如果步骤和分支可以预先写清，例如“查订单 → 按规则判断 → 退款 → 通知”，固定 Workflow 通常比 Agent 更稳定。只在某些难以枚举的节点使用模型判断即可。

#### 第二步：做一个 Single-Agent 基线

先让一个 Agent 在有限工具和状态机中完成任务，收集以下数据：

- 哪类任务失败最多；
- 上下文里哪些信息互相干扰；
- 哪些步骤等待时间最长；
- 哪些工具权限不应同时出现；
- 失败后最难定位的是哪段链路。

没有基线，就无法证明 Multi-Agent 是否改进。

#### 第三步：用四个问题判断是否拆分

1. **可并行吗**：是否有彼此独立、耗时明显的子任务？
2. **可分域吗**：不同任务需要的上下文是否长期稳定地不同？
3. **可隔离吗**：身份、凭据、数据和工具权限能否在运行时真实分开？
4. **收益可测吗**：成功率、延迟、安全或维护性是否有明确指标？

如果只是“Prompt 太长”，应先做上下文检索和压缩；如果只是“工具太多”，可以先动态筛选工具；如果只是“步骤多”，可以先用 Workflow。

#### 第四步：一次只拆一个真实边界

例如失败数据表明政策文档经常干扰订单判断，就先拆出“政策检索 Worker”，而不是一口气设计十个角色。拆分后用相同评测集复测，确认收益再继续。

### 五、一个实用决策流程

```text
任务步骤是否基本固定？
  ├─ 是 → 优先 Workflow，局部使用模型
  └─ 否 → 建立 Single-Agent 基线
             ↓
         是否出现稳定且可测的瓶颈？
             ├─ 否 → 保持 Single-Agent
             └─ 是
                 ↓
      是否存在并行、上下文分域或真实权限隔离收益？
             ├─ 否 → 优化上下文、工具和状态
             └─ 是 → 小范围拆分 Multi-Agent 并复测
```

这是一条经验规则，主要用于避免“为了架构而架构”。

## 工程实践与边界

### Single-Agent 落地清单

- 工具使用窄 schema，读写能力分开注册；
- 状态放在模型外，可持久化和恢复；
- 每轮有步骤、Token、时间和费用预算；
- 高风险动作由策略校验和人工审批；
- 工具调用带关联 ID，写操作支持幂等；
- 有程序化成功、失败和停止条件；
- 能记录并回放完整轨迹。

### Multi-Agent 额外清单

- 每个 Worker 有职责、输入、输出和验收契约；
- 明确全局状态的唯一所有者；
- 消息带任务 ID、版本、发送方、接收方和关联 ID；
- 限制消息轮数、并行扇出和转交次数；
- 处理部分成功、结果冲突、乱序和重复消息；
- 各 Agent 使用最小必要数据与工具；
- 跨 Agent 写操作有统一提交或补偿机制。

### 五个典型失败场景

#### 1. 单 Agent 上下文越来越乱

政策、订单、营销和投诉历史全部放在一起，模型经常选错规则。

先做：按任务检索必要上下文、动态筛选工具、把硬约束结构化。只有不同领域仍稳定互相干扰时，再拆 Worker。

#### 2. 多 Agent 通信抵消并行收益

三个 Worker 各节省 2 秒，但 Orchestrator 多轮确认和摘要花了 8 秒。

处理：测关键路径而非只看 Worker 时间；减少交互轮数；让独立任务一次性返回结构化结果。

#### 3. Orchestrator 成为单点

编排服务崩溃后，已完成的 Worker 结果丢失。

处理：持久化任务图、派发记录和结果；服务可以多实例，但逻辑上保持单一状态所有权；消费者需要幂等。

#### 4. 跨 Agent 写操作不一致

库存 Worker 已扣库存，支付 Worker 失败，系统停在一半。

处理：优先使用数据库事务；跨服务时使用 Saga 等明确的补偿流程。补偿动作由业务代码定义，不让模型临时发明。

#### 5. 拆分后权限仍然共享

名义上有“只读 Agent”和“退款 Agent”，实际上两者使用同一把管理员密钥。

处理：使用不同服务身份和最小权限凭据，在工具服务端校验；Prompt 角色不是授权系统。

### 怎样公平比较两种架构

为了避免误判，应固定：

- 同一任务和难度分布；
- 同一模型版本与推理设置；
- 同一工具与外部数据快照；
- 同一成功判定；
- 相近的重试与预算限制。

然后比较：

- 任务成功率与关键步骤正确率；
- P50、P95 端到端延迟；
- 总 Token、模型调用次数与费用；
- 人工接管率；
- 权限违规与数据泄露；
- 故障恢复成功率；
- 平均故障定位时间和维护成本。

不能一边升级模型，一边拆 Multi-Agent，再把全部提升归功于架构。

## 常见误区

- **“任务步骤多就该拆 Multi-Agent”**：步骤数量和职责边界不是一回事，固定多步骤适合 Workflow。
- **“单 Agent 就是一段无限循环代码”**：成熟方案仍有状态机、工具校验、预算、检查点和停止条件。
- **“多 Agent 总能缩短上下文”**：Agent 之间仍要传递状态和证据，摘要还可能产生信息损失。
- **“中心编排没有自主性”**：Worker 内部仍可根据观察动态行动，中心节点只管理任务和边界。
- **“拆出 Agent 就完成权限隔离”**：必须同时拆身份、凭据、数据范围和服务端授权。
- **“Multi-Agent 失败就继续加一个审核 Agent”**：新增角色也会犯错并增加成本，应先分析失败原因和外部验证手段。

## 面试追问

> **面试官：** 你为什么默认先做 Single-Agent？
>
> **候选人：** 它状态集中、链路短，评测和调试成本低，能建立清晰基线。只有真实失败数据证明拆分有并行、上下文或权限收益时，我才增加协作复杂度。

> **面试官：** 单 Agent 上下文过长就要拆吗？
>
> **候选人：** 不一定。先做上下文筛选、外部状态、记忆压缩和按需工具检索；只有不同职责的上下文能稳定分域，拆 Agent 才可能有净收益。

> **面试官：** Orchestrator 失败怎么办？
>
> **候选人：** 持久化任务图、派发和结果，支持从检查点恢复；服务层可多实例高可用，但逻辑上保持清楚的状态所有权，避免重复提交副作用。

> **面试官：** 如何评估 Agent 间通信成本？
>
> **候选人：** 记录每次交接的 Token、序列化与排队延迟、等待时间、重试、冲突和因上下文丢失产生的返工，再和单 Agent 基线比较。

> **面试官：** Worker 可以直接执行退款吗？
>
> **候选人：** 可以请求，但是否直接执行取决于风险设计。高风险场景通常让 Worker 返回结构化意图，由统一策略、审批和提交节点校验后执行，并保证幂等。

> **面试官：** 为什么优先中心编排而不是完全去中心化？
>
> **候选人：** 中心编排的任务图、状态所有权和完成判断更清晰，便于审计与恢复。去中心化还要解决重复领取、消息乱序和全局完成检测，只有特定分布式需求才值得。
