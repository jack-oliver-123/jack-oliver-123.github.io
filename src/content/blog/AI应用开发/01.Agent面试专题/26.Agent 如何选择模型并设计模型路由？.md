## 本篇目标

理解“选择一个模型”和“运行时给不同请求选择模型”的区别；掌握能力硬约束、质量、成本、延迟、风险与治理等选型维度；能够设计可测试、可降级、不会越权的规则路由或级联路由。

## 60 秒回答

Agent 选模型不能只看公开榜单或单价，而要从真实任务出发。第一步先列出硬约束，例如是否需要图像、长上下文、结构化输出、工具调用、特定部署区域和数据合规；第二步用最强的可用模型建立质量基线和评测集；第三步再尝试把分类、抽取、简单问答等稳定步骤替换成更快、更便宜的模型，直到找到满足质量门槛的最低成本方案。

模型路由是运行时根据请求或步骤选择模型。常见方式有：按任务类型固定模型、基于规则路由、用分类器预测难度、先小模型后大模型的级联升级，以及少量高价值任务使用多模型并行。生产中通常先用可解释的硬规则过滤能力、权限和合规，再在符合条件的候选模型中优化质量、延迟与成本。

路由器本身也会出错，所以必须有接受门槛、回退模型、超时与熔断、最大升级次数和统一 trace。不要只相信模型自报的“置信度”，而要结合可执行校验、Schema、引用充分性、业务规则和真实评测。高风险动作即使路由到最强模型，也仍然需要确定性权限检查和人工审批。

一句话记忆：

> **先用硬约束排除不能用的模型，再用评测选择够用的模型；简单任务走低成本路线，困难或验证失败时升级，但安全权限不能由路由结果决定。**

## 详细解析

### 先说人话：模型路由像医院分诊

医院不会让所有患者都直接找最资深专家，也不会为了省钱让所有患者只看最便宜的门诊。分诊会先判断：

- 是否需要特定科室；
- 是否存在紧急风险；
- 普通门诊能否处理；
- 检查结果异常时是否需要升级；
- 专家不可用时怎样转诊。

Agent 的模型路由也类似：

- 简单意图分类可以由轻量模型处理；
- 复杂规划、代码修复或多工具决策可能需要更强模型；
- 图片质检必须选择支持视觉输入的模型；
- 结果未通过确定性校验时可以升级；
- 涉及付款和删除时还要进入审批，不是换成强模型就可以自动执行。

这个比喻还能提醒我们：路由不是“猜哪位医生最聪明”，而是建立一套有能力约束、质量门槛和升级规则的系统。

### 先分清模型选型和模型路由

| 概念 | 发生时间 | 例子 |
| --- | --- | --- |
| 模型选型 | 设计或发布阶段 | 为规划节点选择模型 A |
| 静态路由 | 根据确定配置分配 | 摘要节点固定使用模型 B |
| 动态路由 | 每次请求运行时判断 | 简单问题走 B，复杂问题走 A |
| 级联（Cascade） | 先调用便宜模型，未达门槛再升级 | 先抽取，Schema 失败则换强模型 |
| 故障回退（Fallback） | 主模型不可用或超时时切换 | 限流后转到兼容备用模型 |
| 集成（Ensemble） | 同一任务使用多个模型再聚合 | 多个模型独立审查高价值方案 |

“路由”和“回退”也不要混淆。路由是根据任务特征主动选模型；回退是主路线出现故障后的恢复策略。级联则是先尝试低成本路线，用结果门槛决定是否升级。

### 为什么 Agent 很适合分步骤选模型

一个 Agent 任务并不是一次模型调用。它可能包含：

```text
识别用户意图
  -> 检索资料
  -> 制定计划
  -> 选择工具和参数
  -> 整理工具结果
  -> 验证证据
  -> 生成最终回答
```

这些步骤的难度和能力需求不同：

| 步骤 | 常见特点 | 可能的模型策略 |
| --- | --- | --- |
| 意图分类 | 标签固定、输出短 | 小模型或传统分类器 |
| 字段抽取 | Schema 明确、可校验 | 支持结构化输出的低成本模型 |
| 查询改写 | 相对局部 | 中小模型，失败时升级 |
| 长程规划 | 步骤依赖多、错误传播大 | 较强推理模型 |
| 工具选择 | 需要理解描述、风险和状态 | 工具调用稳定的模型 |
| 代码生成与修复 | 需要领域能力和执行反馈 | 经代码任务评测验证的模型 |
| 图片理解 | 有模态硬要求 | 支持目标图像输入的模型 |
| 结果验证 | 取决于是否可用程序校验 | 优先规则 / 测试，必要时再用独立模型 |
| 最终表达 | 语气、语言和格式要求 | 能达到产品体验的模型 |

所以“一个 Agent 应该用哪个模型”往往不是一个问题，而是一组问题：每个关键步骤最低需要什么能力，哪个步骤最影响最终成功，哪些步骤可以用代码代替模型。

### 选模型先看硬约束，再看软指标

硬约束是不满足就不能进入候选集的条件：

| 硬约束 | 要检查什么 |
| --- | --- |
| 输入模态 | 是否支持文本、图像、音频或文件 |
| 工具能力 | 是否支持所需 Tool Calling、并行调用或严格结构 |
| 上下文与输出 | 能否容纳必要上下文并生成足够长度的结果 |
| 语言与领域 | 是否能稳定处理目标语言、代码或专业术语 |
| 部署与合规 | 数据区域、保留策略、私有部署或供应商要求 |
| 安全能力 | 是否满足目标领域的安全和内容政策 |
| 可用性 | 目标账号、区域和吞吐量是否可用 |

通过硬约束后，再比较软指标：

| 软指标 | 不能只看什么 | 应该怎样看 |
| --- | --- | --- |
| 质量 | 公开总榜平均分 | 自己任务与关键风险切片的通过率 |
| 延迟 | 单次平均响应时间 | 首 Token、完整响应和 P95 / P99 |
| 成本 | 每百万 Token 标价 | 每个成功任务的模型、重试和工具总成本 |
| 稳定性 | 一次漂亮演示 | 多次运行方差、格式失败和长尾错误 |
| 可运维性 | API 能调用 | 限流、版本、监控、支持与退役策略 |

模型价格便宜不代表任务成本低。如果它经常重试、输出冗长或引起更多工具调用，每个成功任务的总成本可能更高。

### 为什么先用强模型建立基线

直接从最便宜模型开始，遇到低质量时很难判断是模型能力不够，还是提示、工具、检索和编排有问题。

更稳妥的流程是：

1. 用能力较强且满足硬约束的模型跑真实样本；
2. 建立当前架构可以达到的质量上限和失败分类；
3. 固定提示、工具和数据，逐个比较候选模型；
4. 对简单步骤尝试更便宜、更快的模型；
5. 只有关键指标达到门槛才替换；
6. 对困难或高风险切片保留强模型或人工流程。

这不是要求生产统一使用最强模型，而是先建立可信基准，再有证据地优化。

### 路由决策可以写成一个约束优化问题

生产路由不是单纯追求最低价格，可以把目标理解为：

```text
在安全、合规、能力和最低质量全部满足的前提下，
选择能够降低成本与延迟的路线。
```

有时会用一个概念化效用函数表达：

```text
效用 = 质量收益 - λ × 成本 - μ × 延迟
```

其中 `λ` 和 `μ` 表示业务对成本和延迟的重视程度。但权限、数据区域和高风险审批不应该只是可被分数抵消的“扣分项”，而应该作为硬约束。

### 常见路由方式一：按组件静态分配

最简单的方式是为每个 Agent 节点固定模型：

```text
分类节点 -> 轻量模型
规划节点 -> 强模型
摘要节点 -> 中等模型
```

优点是可解释、延迟稳定、容易测试。缺点是同一个节点中的简单和困难请求都走同一路线。

对于任务边界明确、流量稳定的系统，静态分配常常已经足够，不需要先训练复杂路由器。

### 常见路由方式二：规则路由

规则路由根据可观测特征判断：

```text
如果包含图片 -> 视觉候选池
如果是固定标签分类且文本较短 -> 轻量模型
如果需要五个以上工具或复杂规划 -> 强模型
如果属于高风险领域 -> 受控模型路线 + 人工审批
```

可用特征包括：

- 任务类型；
- 输入模态和长度；
- 语言与领域；
- 是否需要工具；
- 预计推理复杂度；
- 用户套餐与延迟目标；
- 数据治理要求；
- 业务风险等级。

规则路由适合早期生产，因为它容易解释、审计和回放。规则多到互相冲突时，再考虑学习型路由。

### 常见路由方式三：分类器或路由模型

可以用传统分类器、小模型或专门训练的 Router，预测：

- 请求属于哪类任务；
- 弱模型是否可能处理成功；
- 哪个模型的质量与成本组合更合适。

RouteLLM 研究了利用偏好数据学习在较强和较弱模型间动态路由。学习型路由的价值是能从大量样本中发现人工规则难以表达的边界，但它也带来新问题：

- 需要高质量路由标签或偏好数据；
- 输入分布变化后可能失效；
- 更换候选模型后需要重新验证；
- 路由错误会系统性地把难题送给弱模型；
- Router 自身也增加延迟、费用和运维负担。

因此，Router 也要版本化、评测和监控，不能把它当作无需验证的可靠开关。

### 常见路由方式四：级联升级

级联的思路是先尝试低成本模型，通过则接受，不通过则升级：

```text
轻量模型生成
  ↓
确定性检查或质量门槛
  ├─ 通过 -> 返回
  └─ 不通过 -> 强模型重做或修复
```

FrugalGPT 等研究讨论了组合多个模型和级联以平衡成本与表现。

级联最关键的不是第一层模型，而是“接受门槛”。适合确定性校验的例子有：

- JSON Schema 是否通过；
- 代码测试是否通过；
- SQL 是否只读且符合语法；
- 答案是否包含所需字段；
- 引用是否来自允许来源；
- 数值是否在业务范围内。

如果只能让模型自己说“我很有信心”，门槛就不可靠。模型自报置信度可能没有校准，而且错误答案也可能语气非常肯定。

### 常见路由方式五：多模型并行与聚合

对于少量高价值、可并行的任务，可以让多个模型独立给出结果，再由规则、投票或另一个验证器聚合。例如：

- 多个模型独立审查重大迁移方案；
- 一个模型生成，一个不同模型检查遗漏；
- 多个候选答案通过可执行测试选择。

它可能提高覆盖率，但会显著增加成本和延迟，而且多个模型可能共享同类盲点。普通客服问题没有必要默认使用集成。

### 一个客服 Agent 的路由例子

假设系统处理以下四类请求：

| 请求 | 路由 |
| --- | --- |
| “退款政策链接发我” | 基础 RAG + 轻量回答模型 |
| “从这张损坏照片判断问题” | 视觉模型路线 |
| “比较新旧政策并解释我的特殊情况” | 强模型 + 多步检索 |
| “给订单退款 5000 元” | 强模型可以辅助判断，但必须走权限校验和人工审批 |

一个较稳妥的路由流程是：

```text
读取任务与可信业务上下文
  ↓
按模态、数据区域和工具能力做硬过滤
  ↓
按任务类型、复杂度和风险选择默认路线
  ↓
执行并用 Schema、证据或业务规则验证
  ↓
通过 -> 返回
失败 -> 在次数上限内升级或回退
  ↓
高风险动作始终进入独立审批
```

注意，路由器可以选择“由谁思考”，不能选择“是否绕过权限”。

### 回退模型不是随便换一个 API

主模型超时或限流时，备用模型必须满足当前任务的硬约束。切换前要检查：

- 是否支持相同输入模态；
- 是否兼容当前 Tool Schema 与结果格式；
- 上下文长度是否足够；
- 数据是否允许发送到该供应商和区域；
- 安全策略和内容限制是否一致；
- 提示是否需要针对模型调整；
- 切换后是否可能重复执行写操作。

不能为了可用性，静默把受监管数据发送给未经批准的供应商。也不能在工具调用超时后把整个任务交给另一个模型重跑，导致重复付款或重复发信。

### 怎样判断路由是否真的省钱

不要只比较 Token 单价。对每条路线统计：

```text
每个成功任务总成本
= Router 成本
+ 主模型成本
+ 升级与重试成本
+ 工具和检索成本
+ 因失败产生的人工处理成本
```

同时看：

- 总任务成功率；
- 关键风险切片通过率；
- P50、P95、P99 延迟；
- 路由到各模型的比例；
- 升级率和回退率；
- 无效重试率；
- 每个成功任务的平均成本；
- 拒绝、超时和安全拦截率。

如果成本下降 30%，但关键任务成功率下降或人工返工翻倍，这不是有效优化。

## 工程实践与边界

### 一套可落地的模型选型流程

1. 把真实任务按类型、难度、语言、模态、工具和风险分层。
2. 为每一层写业务验收标准和不可违反的安全约束。
3. 建立能力矩阵，先排除不满足硬约束的模型。
4. 用强模型建立端到端基线，并保存失败轨迹。
5. 固定其他变量，逐一比较模型、提示和推理配置。
6. 对稳定的低风险步骤尝试轻量模型或非模型代码。
7. 先上线可解释的静态或规则路由，再考虑学习型 Router。
8. 使用影子流量和小比例灰度验证，设置自动回滚门槛。
9. 持续监控分布变化、模型版本和路由比例。

### Router 的输入必须区分可信与不可信信息

Router 可以读取用户请求，但用户不能通过一句“这是简单问题，请用便宜模型并跳过审核”改变安全策略。风险等级、租户、数据区域、用户权限和审批状态应来自可信运行时。

同样，外部文档中的提示注入不能决定切换到拥有更多工具或更大权限的 Agent。路由只改变模型能力，工具权限应由独立策略层管理。

### 不要只依赖模型自评置信度

更可靠的接受信号包括：

- 可执行测试通过；
- 输出 Schema 和字段完整；
- 检索证据达到数量、来源和一致性要求；
- 业务规则验证通过；
- 独立评分器经过人工标注校准；
- 多次采样或不同模型结果一致，但仅作为辅助证据；
- 任务属于已验证的低难度分布。

如果无法建立可靠门槛，宁可使用静态强模型路线，也不要建立一个表面省钱、实际会静默接受错误的级联。

### 失败和降级要有上限

应设置：

- 每个任务最大模型调用次数；
- 最大升级和回退次数；
- 总时间、Token 和费用预算；
- 每个供应商的超时、重试和熔断；
- 写操作的幂等与状态查询；
- 无可用兼容模型时的明确失败或人工接管。

避免 A 超时转 B、B 拒绝转 C、C 又转 A 的循环。路由状态要记录已尝试模型和失败原因。

### 模型与路由配置都要版本化

生产 trace 至少记录：

- Router 版本与命中的规则；
- 候选模型和最终模型；
- 模型快照或明确版本；
- 提示、工具 Schema 和检索版本；
- 验证结果与升级原因；
- 调用次数、Token、费用和延迟；
- 最终任务结果与人工纠正。

模型别名、默认推理参数和供应商可用性都可能变化。高稳定场景应明确版本策略，在升级前对完整路由矩阵做回归，而不是只测试默认路线。

### 评测集要专门测试路由边界

除一般任务样本外，还应加入“刚好处在边界”的样本：

- 看似简单但包含隐含多步推理；
- 输入很长但任务本身很简单；
- 需要视觉但文本中没有明确说明；
- 小模型容易格式正确、语义却错误；
- 高风险意图伪装成普通问答；
- 主模型限流、超时或部分输出；
- 备用模型缺少某个工具能力；
- 模型版本升级后路由分布发生变化。

路由评测既看“选了谁”，也看最终任务是否成功。某个样本不一定只有唯一正确模型，只要路线满足质量、安全、延迟与成本目标即可。

### 什么时候不值得做动态路由

以下情况优先保持单模型或静态配置：

- 流量很小，节省的成本不足以覆盖路由复杂度；
- 任务高度一致，固定模型已经稳定；
- 没有可靠评测集和接受门槛；
- 所有任务都是高风险且都需同一强路线；
- Router 增加的延迟抵消了收益；
- 团队暂时无法监控多模型版本、限流和回退。

模型路由本身是一套生产系统，需要维护数据、规则、评测和故障策略。没有证据的复杂路由不比简单方案更成熟。

## 常见误区

- **“榜单第一就是最适合 Agent 的模型”**：公开榜单不一定覆盖你的工具、语言、数据和风险分布。
- **“最强模型应该处理所有步骤”**：分类和抽取等可验证步骤可能使用轻量模型或普通代码完成。
- **“只用最便宜模型就最省钱”**：重试、长输出、错误工具调用和人工返工会抬高每个成功任务的总成本。
- **“上下文窗口越大越好”**：更大窗口不保证模型能找到关键信息，也可能增加费用和噪声。
- **“模型说有信心就可以接受”**：自报置信度未必校准，应优先使用外部验证和真实评测。
- **“Router 只是一条 if，不用评测”**：错误规则会系统性地把任务送错路线，也要做版本、回归和监控。
- **“换备用模型只是改一个名称”**：模态、Schema、提示、上下文、合规和安全能力可能不同。
- **“高风险任务交给强模型就安全了”**：模型能力不能替代权限、审批、幂等和审计。
- **“动态路由越复杂越先进”**：稳定、可解释的静态路由常常更适合早期生产。
- **“路由只看成本和质量”**：延迟、可用性、数据区域、供应商风险和版本生命周期同样重要。

## 面试追问

> **面试官：** Agent 选模型时最重要的原则是什么？
>
> **候选人：** 先从真实任务和硬约束出发，不从厂商榜单出发。先用强模型建立质量基线，再通过分层评测找出哪些步骤可以由更快、更便宜的模型处理，最终看每个成功任务的质量、延迟、成本和安全，而不是只看单次 Token 价格。

> **面试官：** 什么是模型路由？
>
> **候选人：** 模型路由是运行时根据任务类型、模态、复杂度、风险、延迟和治理要求，从候选模型中选择本次请求或当前步骤使用哪个模型。它可以是静态配置、规则、分类器、学习型 Router 或级联。

> **面试官：** 规则路由和学习型路由怎么选？
>
> **候选人：** 早期优先规则路由，因为可解释、可审计、数据要求低。规则无法覆盖复杂边界且积累了足够的标注或偏好数据后，再考虑学习型 Router，但仍要保留能力和合规硬规则。

> **面试官：** 什么是模型级联？
>
> **候选人：** 先让低成本模型处理，再用外部门槛判断是否接受；失败时升级到强模型。关键是门槛必须可靠，例如 Schema、单元测试、引用或业务规则，而不是只相信模型自报信心。

> **面试官：** 为什么不能只用模型的置信度做升级判断？
>
> **候选人：** 语言模型的自报置信度可能没有校准，错误答案也可能很肯定。我会优先使用可执行校验、证据充分性、业务规则和经人工校准的评分器，并在无法可靠验证的任务上直接使用更稳妥路线。

> **面试官：** 规划和执行是否应该用同一个模型？
>
> **候选人：** 不一定。复杂规划可能需要强模型，固定字段抽取和结果格式化可以用轻量模型。但拆分会增加接口和错误传播，所以要用端到端评测证明收益，不能为了多模型而多模型。

> **面试官：** 怎样评估路由系统？
>
> **候选人：** 在相同分层数据集上比较单强模型基线与路由方案，检查任务成功、安全切片、P95 / P99 延迟、每个成功任务成本、升级率、回退率和路由分布。还要做影子流量、灰度和版本升级回归。

> **面试官：** 主模型不可用时，可以自动切任意备用模型吗？
>
> **候选人：** 不可以。备用模型必须满足模态、上下文、工具、数据区域和合规等硬约束，并通过兼容性测试。涉及写操作时还要确认先前调用是否已生效，避免重跑造成重复副作用。

> **面试官：** Router 自己出错怎么办？
>
> **候选人：** Router 要有独立版本、离线评测、监控和保守默认路线。记录命中规则和候选模型；异常时回到满足硬约束的默认模型，并限制升级次数。关键高风险任务不依赖 Router 决定权限和审批。

> **面试官：** 什么时候不应该引入动态模型路由？
>
> **候选人：** 流量小、任务单一、没有可靠评测和接受门槛，或者路由延迟与维护成本大于节省时，不应该引入。先用单模型或按节点静态配置更容易获得稳定系统。
