想象一道数学题：“三个人以不同速度合作，多久能完成任务？”

如果要求模型看到题目后立刻说出一个答案，它可能凭模式猜测。

如果允许它：

1. 先列出已知条件；
2. 尝试两种解法；
3. 用代入法检查结果；
4. 发现矛盾后回退；
5. 最后再输出答案；

正确率往往会提高。

这里增加的不是模型参数，而是**处理这一次问题时投入的计算**。

这就是 Test-Time Compute 的核心直觉。

## 本篇目标

学完本篇，你应该能够：

1. 解释什么是推理模型，以及它和普通对话模型的区别；
2. 区分训练时计算和测试时计算；
3. 说清楚为什么多采样、搜索和验证可能提升正确率；
4. 理解 Best-of-N、自一致性、顺序式推理和树搜索；
5. 解释结果验证器与过程验证器的区别；
6. 计算简单的成功概率、Token 成本和延迟开销；
7. 理解“计算越多越好”为什么不成立；
8. 设计按问题难度动态分配推理预算的系统；
9. 在面试中说明质量、成本和用户体验之间的工程取舍。

先记住一句话：

> Test-Time Compute 是在模型参数已经固定后，为当前输入投入更多生成、搜索或验证计算；它能扩大寻找正确答案的机会，但收益依赖候选多样性、验证能力和任务本身。

## 60 秒回答

推理模型通常指经过专门训练，能够在回答前进行更长的中间推理、检查或搜索，并能按问题难度消耗不同计算预算的模型。

Test-Time Compute，也叫 Inference-Time Compute 或 Test-Time Scaling，是模型参数固定后，在处理单个请求时增加计算。例如生成更多思考 Token、采样多个候选、用自一致性投票、让验证器打分、做树搜索，或者在执行反馈后修正。

它能提升效果，是因为一次贪心解码只探索一条路径；如果模型对正确路径分配了非零概率，多次且有差异的尝试就更可能覆盖正确答案。再由可靠的规则、执行器或验证器选出正确候选，最终成功率可能提高。

但增加计算不是自动变聪明。候选高度重复、验证器不可靠、问题缺少信息、模型根本不会解，或者长推理引入更多错误时，收益会很小甚至下降。它还会增加 Token、GPU、端到端延迟和尾延迟。

工程上我会把推理预算当作可控资源：简单问题早停，困难或高价值问题增加预算；离线画出质量—计算曲线，线上限制最大 Token、并发和超时，并同时监控准确率、平均计算量、P95/P99 延迟和单次成功任务成本。

## 详细解析

### 一、先区分“训练时”和“回答时”

一个模型生命周期里，计算大致发生在两个阶段。

#### 1. 训练时计算

训练时，模型通过大量数据更新参数。

可以粗略理解为：

```text
训练数据
  -> 前向计算
  -> 计算损失
  -> 反向传播
  -> 更新参数
```

训练结束后，参数通常固定。

更大的训练数据、更多参数和更多训练步数，都属于训练时扩展。

#### 2. 测试时计算

这里的“测试”不是只指学校考试，也不是只指评测集。

它泛指模型已经训练好后，处理新输入的推理阶段。

```text
用户问题
  -> 固定参数的模型
  -> 分配生成/搜索/验证预算
  -> 最终答案
```

增加测试时计算不会直接修改权重。

它改变的是这一次请求探索多少条路径、生成多少中间步骤、做多少检查。

### 二、什么是推理模型

“推理模型”不是一个严格到只有唯一实现的架构名。

工程上通常指具备以下特征的一类模型：

- 针对多步推理和问题求解做过后训练；
- 回答前可能产生较长的内部或可见推理过程；
- 能使用更多推理 Token 换取更高任务质量；
- 可能学习检查、回退、分解和验证；
- 在数学、代码、规划等可验证任务上尤其明显。

普通指令模型也能通过 Prompt 做 Chain-of-Thought 或多次采样。

因此，“推理模型”和“使用 Test-Time Compute”有交集，但不是同义词。

```text
普通模型 + 多次采样/外部搜索
    也可以使用 Test-Time Compute

推理模型 + 单次受限生成
    也可能只获得较小预算
```

### 三、为什么一次生成可能错过正确答案

自回归模型每一步都预测下一个 Token 的概率分布。

设当前上下文为 $x$，已经生成的前缀为 $y_{<t}$，第 $t$ 个 Token 的概率是：

$$
p(y_t\mid x,y_{<t})
$$

其中：

- $x$：用户问题和系统上下文；
- $y_t$：第 $t$ 个待生成 Token；
- $y_{<t}$：第 $t$ 步之前已经生成的 Token；
- $p$：模型给出的条件概率。

完整输出序列的概率可以写成：

$$
p(y\mid x)=\prod_{t=1}^{T}p(y_t\mid x,y_{<t})
$$

其中 $T$ 是输出长度。

贪心解码每一步只选当前概率最高的 Token。

一旦早期选择错了，后续生成会建立在错误前缀上。

即使正确解法在模型分布中存在，它也不一定是贪心路径。

测试时搜索的目标，就是不只看一条路径。

### 四、最简单的方法：让模型多试几次

假设一次独立尝试得到正确答案的概率是 $p$。

如果独立采样 $N$ 次，至少一次正确的概率是：

$$
P(\text{至少一次正确})=1-(1-p)^N
$$

其中：

- $p$：单次正确概率；
- $N$：尝试次数；
- $(1-p)^N$：所有尝试都错误的概率。

假设 $p=0.4$，采样 4 次：

$$
1-(1-0.4)^4=1-0.6^4\approx87.04\%
$$

看起来从 40% 提升到了 87%。

但这里有两个很强的假设：

1. 各次尝试近似独立；
2. 系统能知道哪个候选是正确的。

现实中，模型常常重复同一种错误，候选并不独立。

更关键的是，如果没有答案或验证器，系统只是拿到了四个互相冲突的结果。

所以，“至少生成过正确答案”不等于“最终选中了正确答案”。

### 五、Best-of-N：生成 N 个，再选一个

Best-of-N 的流程是：

```text
同一个问题
  -> 采样候选 1
  -> 采样候选 2
  -> ...
  -> 采样候选 N
  -> 评分器或验证器
  -> 选最高分候选
```

可写成：

$$
y^*=\arg\max_{y_i\in\{y_1,...,y_N\}}r(x,y_i)
$$

其中：

- $y_i$：第 $i$ 个候选；
- $r(x,y_i)$：评分器对候选的分数；
- $y^*$：最终选中的候选；
- $\arg\max$：找到分数最大的那个候选。

评分器可能是：

- 数学答案检查器；
- 单元测试；
- 编译器；
- 规则引擎；
- 奖励模型；
- 另一个 LLM Judge；
- 人工审核。

外部可执行验证通常比“让模型凭感觉给自己打分”更可靠。

### 六、自一致性：用答案投票

Self-Consistency 常用于存在多条推理路径、但最终答案较短的任务。

流程是：

1. 对同一问题采样多个推理过程；
2. 提取每条路径的最终答案；
3. 选择出现次数最多的答案。

假设 7 次采样得到：

```text
42, 42, 39, 42, 40, 42, 39
```

多数答案是 42，因此最终输出 42。

自一致性利用“不同正确路径可能汇聚到同一答案”的现象。

它适合：

- 数学题；
- 选择题；
- 有规范化短答案的推理任务。

它不适合直接处理：

- 开放写作；
- 多个答案都合理的任务；
- 所有采样都共享同一系统性偏差；
- 难以稳定提取最终答案的输出。

多数票也不等于事实真值。

如果模型 7 次都犯同一种错，投票会更自信地选错。

### 七、顺序式扩展：让一条路径思考更久

除了并行生成多个完整候选，还可以增加单条轨迹的长度。

例如让模型：

```text
理解问题
  -> 分解子问题
  -> 解答子问题
  -> 检查约束
  -> 发现错误
  -> 修正
  -> 输出结论
```

这种方式常被称为顺序式 Test-Time Compute。

它的优势是后面的步骤能利用前面的中间状态。

它的风险是错误也会沿轨迹累积。

如果早期假设错了，而模型没有真正的纠错能力，更长输出只会把错误解释得更完整。

### 八、并行扩展与顺序扩展的区别

| 方式 | 如何增加计算 | 优点 | 局限 |
| --- | --- | --- | --- |
| 并行采样 | 多生成几条候选 | 易并行、增加多样性 | 需要选择器，显存和总 Token 上升 |
| 顺序推理 | 单条轨迹生成更久 | 可逐步分解、检查和回退 | 延迟串行累积，可能越想越错 |
| 混合方式 | 多条中等长度轨迹 | 兼顾多样性和深度 | 调度、评分和成本更复杂 |

没有一种策略对所有题都最优。

一些题需要深挖一条路径；另一些题更适合多试几种方法。

### 九、搜索：不是只生成完整答案

树搜索会把中间推理状态看成节点。

```text
             初始问题
            /    |    \
         思路 A 思路 B 思路 C
          / \       |      / \
        A1  A2     B1    C1  C2
```

系统可以反复执行：

1. 扩展：从某个状态生成下一步候选；
2. 评估：估计每个候选状态的前景；
3. 剪枝：放弃低分路径；
4. 继续：把预算投入更有希望的分支；
5. 终止：找到答案或耗尽预算。

Tree of Thoughts 是这类思想的代表之一。

搜索的难点不只是“生成分支”，而是：

- 状态怎么表示；
- 每次扩展几个分支；
- 如何评分中间状态；
- 什么时候回退；
- 什么时候停止；
- 如何避免搜索成本爆炸。

分支因子为 $b$、深度为 $d$ 时，完整树节点数量大致按 $b^d$ 增长。

因此实际系统必须剪枝，而不是穷举。

### 十、验证器为什么关键

候选生成器负责“提出可能答案”。

验证器负责“判断哪个更可信”。

没有验证器，多采样的上限很难转化为最终正确率。

#### 1. 结果验证器

Outcome Reward Model，简称 ORM，只评价最终结果。

例如：

- 数学最终答案是否正确；
- 代码是否通过测试；
- SQL 查询结果是否符合预期。

它的标注和运行通常较简单，但不能指出中间哪一步错了。

#### 2. 过程验证器

Process Reward Model，简称 PRM，对中间步骤评分。

例如一条 5 步推理：

```text
步骤 1：正确
步骤 2：正确
步骤 3：符号变换错误
步骤 4：建立在错误上
步骤 5：答案错误
```

PRM 可以在步骤 3 就降低路径分数，帮助搜索提前剪枝。

过程监督更细，但也更昂贵：

- 需要定义什么是一“步”；
- 标注者要检查中间逻辑；
- 不同正确推理可能表达不同；
- 评分器也可能被表面风格欺骗。

### 十一、为什么增加计算能够提升效果

可以从四个角度理解。

#### 1. 探索更多候选

单次解码只看到概率分布中的一条路径。

多采样和搜索增加找到正确路径的机会。

#### 2. 把复杂任务分解

一步直接映射“问题 -> 答案”很难。

分成“理解 -> 计划 -> 局部求解 -> 检查”后，每一步可能更简单。

#### 3. 利用可验证反馈

代码测试、计算器和规则引擎能提供模型参数之外的可靠信号。

模型可以根据失败反馈修正。

#### 4. 按难度分配资源

不是所有问题都同样难。

简单请求只用少量 Token，复杂请求才投入更大预算，可以提高整体计算效率。

### 十二、为什么增加计算也可能不提升

#### 1. 模型没有正确解法

如果正确答案在模型分布中的概率接近零，多采样只是重复错误。

#### 2. 候选缺少多样性

采样温度太低或模型偏差太强时，多条轨迹高度相似。

#### 3. 验证器选错

验证器可能偏爱更长、更自信或更像标准模板的错误答案。

#### 4. 问题本身信息不足

用户没有给订单号，再多推理也无法确定是哪笔订单。

正确行为应该是澄清或调用工具，而不是继续“想”。

#### 5. 推理越长，错误机会越多

每一步都可能引入错误。

如果步骤正确率粗略为 $q$，并且必须连续 $L$ 步都正确，理想化成功概率为：

$$
q^L
$$

当 $q=0.95$、$L=20$ 时：

$$
0.95^{20}\approx35.8\%
$$

这个独立假设过于简化，但它说明“更长”本身不是正确性保证。

#### 6. 任务不需要推理

把一句话改成敬语，通常不需要生成几千个推理 Token。

过度计算只会增加延迟和成本。

### 十三、如何定义推理预算

推理预算可以用多种量表示：

- 最大生成 Token；
- 候选数量 $N$；
- 搜索节点数；
- 最大回退次数；
- 验证器调用次数；
- 工具调用次数；
- 最大墙钟时间；
- 最大费用。

一个请求的粗略计算成本可以写成：

$$
C=N\cdot(T_{in}+T_{reason}+T_{out})\cdot c_{token}+C_{verify}
$$

其中：

- $N$：并行候选数；
- $T_{in}$：每个候选的输入 Token 数；
- $T_{reason}$：中间推理 Token 数；
- $T_{out}$：最终输出 Token 数；
- $c_{token}$：单位 Token 的成本；
- $C_{verify}$：验证、执行和调度成本。

托管 API 可能只暴露部分 Token 统计。

自部署时还要考虑批处理效率、KV Cache、GPU 利用率和等待队列。

### 十四、质量—计算曲线与边际收益

评测时不要只测“低预算”和“高预算”两个点。

应画出多档预算：

| 推理预算 | 任务成功率 | P95 延迟 | 单次成本 |
| ---: | ---: | ---: | ---: |
| 低 | 72% | 1.2 秒 | 0.02 元 |
| 中 | 83% | 3.4 秒 | 0.06 元 |
| 高 | 86% | 9.8 秒 | 0.18 元 |

从低到中提高 11 个百分点；从中到高只提高 3 个百分点。

这叫边际收益递减。

业务可能选择中档，而不是追求最高分数。

### 十五、动态预算：简单题早停，难题多算

固定给每个请求同样预算通常不经济。

动态系统可以这样工作：

```text
请求
  -> 难度/风险估计
  -> 小预算尝试
  -> 置信或验证是否通过？
       | 是 -> 返回
       | 否 -> 增加候选/搜索/验证
  -> 达到上限后返回、拒绝或转人工
```

难度信号可能包括：

- 题目长度和结构；
- 是否需要多步约束；
- 小模型的置信度；
- 多个候选是否一致；
- 外部验证是否通过；
- 任务风险和价值；
- 是否接近超时或预算上限。

注意，模型自报“我很有信心”不一定校准良好。

最好结合可验证信号和历史数据。

### 十六、哪些任务最适合 Test-Time Compute

通常更适合：

- 数学和逻辑；
- 代码生成与修复；
- 有执行器的规划；
- 可用规则检查的结构化任务；
- 有明确答案或验证器的搜索问题；
- 高价值、低频、可接受延迟的分析任务。

通常收益较弱或需谨慎：

- 极低延迟实时交互；
- 简单分类和改写；
- 缺少信息的问题；
- 主观开放写作；
- 验证成本高于生成成本的任务；
- 高并发且预算严格的服务。

## 工程实践与边界

### 一、先做预算扫描，而不是拍脑袋设置 Token

对每个关键任务切片测多档：

- 最大推理 Token；
- 候选数量；
- 验证策略；
- 超时时间。

同时记录质量、延迟和成本，寻找拐点。

### 二、并行候选降低墙钟时间，但不降低总计算

把 8 个候选同时生成，可能比串行生成更快返回。

但它仍消耗接近 8 份生成计算，并增加瞬时显存和并发压力。

不要把“并行更快”误写成“计算更少”。

### 三、建立可取消和早停机制

如果一个候选已经通过确定性验证，可以取消其他尚未完成的候选。

如果请求断开，也应尽快终止无用生成。

否则后台继续推理会浪费 GPU 和费用。

### 四、验证器必须单独评测

验证器要测：

- 对正确候选的召回率；
- 对错误候选的拒绝率；
- 对长度和措辞的偏好；
- 是否容易被奖励黑客利用；
- 在不同难度和领域上的校准；
- 推理开销和可用性。

生成器升级后，验证器分布也可能变化。

### 五、外部执行要放在沙箱中

代码执行、浏览器、数据库和工具反馈能提高验证能力，也引入副作用和安全风险。

应设置：

- 网络和文件权限；
- CPU、内存和时间限制；
- 只读或模拟环境；
- 幂等与审批；
- 输出大小限制；
- 审计日志。

### 六、不要向用户伪造“完整思维过程”

产品应输出必要的结论、依据和可验证步骤。

内部推理 Token 是否可见，取决于模型接口和产品设计。

不能因为看不到内部推理，就声称模型没有进行额外计算；也不能把生成的一段解释自动当成真实内部计算轨迹。

### 七、监控平均值和尾部值

推理预算可能由模型动态决定，因此请求间差异很大。

线上至少监控：

- 平均和 P95/P99 推理 Token；
- TTFT、TPOT 和端到端延迟；
- 超时与取消率；
- 每任务成本；
- 验证通过率；
- 预算升级率；
- 最终任务成功率。

### 八、测试时扩展不能替代数据和工具

对于最新库存、用户余额和公司政策，模型需要可靠的数据源。

让模型“想得更久”不会自动获得缺失事实。

这类任务应优先使用检索或工具。

## 常见误区

### 误区一：推理模型就是参数更大的模型

推理行为主要与训练目标、后训练和推理策略有关，不能只看参数量。

### 误区二：Test-Time Compute 会更新模型权重

它通常发生在参数固定的推理阶段。

### 误区三：输出越长，推理越强

长度只是计算代理之一。

重复、跑题和错误链也会产生很长输出。

### 误区四：多采样就会提高最终正确率

只有生成到更好候选并可靠选中它，最终结果才会提升。

### 误区五：多数投票就是正确答案

共享偏差会让所有候选一起犯错。

### 误区六：验证器和标准答案一样可靠

学习型验证器也会误判、偏好特定风格，并可能被攻击。

### 误区七：推理预算越大越好

收益会递减，甚至可能因错误累积而下降。

### 误区八：并行 8 个候选只算一次成本

墙钟时间可能接近一次，但总 Token 和算力接近多次。

### 误区九：所有请求都应该用推理模型

简单、高并发、低延迟任务常常更适合小模型或确定性代码。

### 误区十：思考可以替代权限和事实检查

模型不能凭推理创造可信数据库状态，也不能自行决定授权。

## 面试追问

### 追问一：Test-Time Compute 和 Chain-of-Thought 有什么区别？

Chain-of-Thought 是一种生成中间推理步骤的方法。

Test-Time Compute 范围更广，还包括多采样、搜索、验证、工具反馈和动态预算。

### 追问二：Best-of-N 的 N 越大越好吗？

不一定。

候选会重复，边际收益递减；验证器错误、成本和延迟会随 N 增长。

### 追问三：为什么推理模型的延迟波动可能更大？

不同问题可能触发不同推理长度和搜索预算。

因此输出 Token 和端到端延迟分布更宽，必须关注尾延迟。

### 追问四：如何判断问题难不难？

可以用任务分类器、历史数据、候选一致性、验证失败、小模型不确定性和规则特征组合估计。

难度估计器也要独立评测。

### 追问五：结果验证和过程验证怎么选？

有可靠终态检查器时，结果验证更便宜直接。

搜索空间大、早期错误可识别时，过程验证有助于剪枝，但标注和维护更难。

### 追问六：为什么代码任务特别适合增加推理计算？

代码有编译器、类型检查和测试等低歧义反馈。

系统可以生成多个候选并执行验证，再根据失败信息修复。

### 追问七：如何控制成本失控？

设置每请求 Token、候选数、工具次数、费用和墙钟时间上限；支持早停和取消；按风险与价值分配预算；持续监控预算分布。

### 追问八：模型自我反思可靠吗？

它可能发现部分错误，但也可能坚持错误或把正确答案改坏。

应通过任务级实验验证，并优先结合外部可执行检查。

### 追问九：测试时扩展和训练 Scaling Law 有什么关系？

两者都是用更多计算换能力，但投入阶段不同。

训练扩展把计算固化进参数；测试时扩展按请求投入，能动态分配，但每次使用都重复付费。

### 追问十：怎么向业务决定采用哪档预算？

给出质量—延迟—成本曲线和关键任务门槛，让业务选择拐点，而不是只展示最高分。
