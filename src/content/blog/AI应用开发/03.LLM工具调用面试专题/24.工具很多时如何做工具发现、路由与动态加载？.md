假设公司给一个客服 Agent 接入了 300 个工具：查订单、改地址、算运费、退款、补开发票、冻结账户、查询各地区活动……

最直接的做法，是把 300 份工具说明全部塞给模型，让模型自己选。

这通常会同时带来四个问题：提示词变长、相似工具变多、选错概率上升，以及不该暴露的高风险工具被模型看见。

真正的工程方案不是单纯“换一个更聪明的模型”，而是在模型选择工具之前，先建立一条受控的工具发现流水线。

## 本篇目标

学完本篇，你应该能够回答以下问题：

1. 为什么工具数量增加后，不能继续把所有 Schema 一次性发给模型？
2. 工具注册、工具发现、工具路由和动态加载分别是什么？
3. 如何用权限过滤、候选召回、重排和按需加载缩小工具范围？
4. Function Calling、MCP 工具发现和厂商提供的 Tool Search 有什么关系？
5. 如何判断路由系统是“没有找到工具”，还是“找到后模型选错了”？
6. 面试时如何给出既有原理、又有工程细节的答案？

先记住本篇最核心的一句话：

> 工具很多时，不要让模型在整个工具仓库里盲选；应先由系统根据租户、权限、场景和语义召回少量候选，再把候选的完整 Schema 按需交给模型。

## 60 秒回答

当工具只有几个时，可以静态地把全部工具定义发给模型；当工具增加到几十、几百甚至更多时，我会设计“注册中心 + 分层路由 + 动态加载”。

首先，工具注册中心保存工具名称、用途、输入输出 Schema、版本、所属业务域、权限、风险等级和可用状态。收到请求后，系统先做确定性的租户与权限过滤，绝不能先让模型看到无权使用的工具。然后根据用户意图、会话状态和业务域做候选召回，可以结合规则、关键词、向量检索和轻量分类器。对召回结果再重排并设置最低置信度，最终只加载少量候选工具的完整 Schema 给模型。

模型仍然负责在候选中决定是否调用、调用哪个以及生成参数；宿主应用负责参数校验、授权、执行和回传。对于 MCP，可以通过 `tools/list` 发现 Server 暴露的工具，并在工具列表变化时刷新注册信息；某些模型 API 还提供厂商特定的延迟加载或 Tool Search，但这不是跨厂商标准。

评估时不能只看最终回答，要分别统计候选召回率、工具选择准确率、参数正确率、拒绝调用准确率、执行成功率、端到端任务成功率、延迟和成本。这样才能知道问题出在路由、模型还是工具本身。

## 详细解析

### 一、先建立直觉：工具目录像大型商场

一个只有 5 件商品的小卖部，可以把全部商品摆在柜台上。

一个有 10 万件商品的电商平台，不会让用户逐个翻看，而是提供：

1. 类目：先进入“家电”或“图书”；
2. 搜索：根据关键词找到候选商品；
3. 筛选：按地区、库存、权限和价格过滤；
4. 排序：把最可能需要的商品放在前面；
5. 详情页：只有打开商品后，才加载完整参数。

大型工具系统也是同一个思路：

```text
全部工具
   |
   | 1. 租户、角色、环境、风险策略过滤
   v
用户有资格看到的工具
   |
   | 2. 规则、关键词、向量或分类器召回
   v
候选工具集合
   |
   | 3. 重排、去重、阈值与冲突处理
   v
少量高相关工具
   |
   | 4. 加载完整 Schema
   v
模型决定是否调用并生成参数
   |
   | 5. 服务端再次校验、授权和执行
   v
工具结果
```

这里有一个很容易被忽略的边界：

> “让模型看见工具”本身就是一种能力暴露，不能等到执行阶段才第一次做权限控制。

即使最终执行接口会拒绝越权，工具名称和描述也可能泄露内部能力、资源名称或业务流程。

### 二、四个容易混淆的术语

#### 1. 工具注册（Tool Registration）

工具注册是把一个工具的元数据放进可管理的目录。

一条注册记录通常不只包含 Function Calling 所需的 `name`、`description` 和 `parameters`，还会包含：

| 字段 | 作用 | 示例 |
| --- | --- | --- |
| `tool_id` | 稳定的内部标识 | `billing.refund_order` |
| `version` | Schema 和行为版本 | `2.1.0` |
| `domain` | 业务域 | `billing` |
| `summary` | 用于初步检索的简短描述 | “按订单号发起退款” |
| `input_schema` | 完整输入结构 | JSON Schema |
| `output_schema` | 输出结构或契约 | 退款单对象 |
| `required_scopes` | 调用所需权限 | `refund:write` |
| `risk_level` | 风险等级 | `high` |
| `side_effect` | 是否产生副作用 | `true` |
| `tenant_policy` | 哪些租户可用 | 企业版租户 |
| `endpoint` | 实际执行适配器 | 内部服务地址或函数名 |
| `status` | 当前是否可用 | `active`、`deprecated` |

注册中心是宿主应用的工程组件，不是语言模型自动拥有的能力，也不是 MCP 强制规定必须部署的产品。

#### 2. 工具发现（Tool Discovery）

工具发现解决“系统里有哪些工具可用”。

发现可能来自：

- 静态配置文件；
- 数据库或服务注册中心；
- 插件目录；
- MCP Server 的 `tools/list`；
- 根据当前项目或租户动态生成的工具；
- 外部工具市场，但接入前必须经过信任审核。

发现得到的是工具清单，不等于模型可以使用，更不代表用户已经获得调用权限。

#### 3. 工具路由（Tool Routing）

工具路由解决“这次请求可能需要哪些工具”。

它的输出通常不是最终调用，而是一组候选：

```json
[
  { "tool": "orders.get_order", "score": 0.94 },
  { "tool": "shipping.get_tracking", "score": 0.86 },
  { "tool": "billing.get_invoice", "score": 0.41 }
]
```

模型随后在候选中继续判断，也可能决定一个都不调用。

#### 4. 动态加载（Dynamic Tool Loading）

动态加载解决“什么时候把完整工具定义放入模型上下文”。

初始阶段可以只展示业务域或工具摘要，确定候选后再加载完整 Schema。这样能减少上下文占用，也能降低大量相似参数对模型的干扰。

动态加载不是随意把任何搜索结果变成可执行工具。新加载的工具仍需经过来源校验、签名或白名单、权限过滤和 Schema 验证。

### 三、为什么工具越多不一定越强

#### 1. 工具定义会占用上下文

名称、描述、参数和枚举都需要进入模型可见的上下文。工具越多，输入 Token、延迟和成本通常越高。

更重要的是，上下文被工具定义占满后，用户需求、业务规则和历史对话可用的空间会变少。

#### 2. 相似工具会造成选择混淆

假设同时存在：

- `get_order`
- `search_orders`
- `get_order_detail`
- `query_order_status`
- `get_shipping_order`

如果描述不清，模型可能根据名称猜测，而不是根据输入前提和返回语义选择。

工具数量增长时，错误往往不是“模型缺少调用能力”，而是“从几个相似选项中选错一个”。

#### 3. 工具描述可能泄露能力

一个名为 `disable_user_without_review` 的内部工具，即使最终调用被拒绝，也不应该向普通用户会话暴露。

因此权限过滤必须发生在候选召回之前或召回过程中，而不只是执行之前。

#### 4. 全量工具会放大供应链风险

如果系统自动接入第三方工具目录，恶意工具可以通过名称或描述诱导模型调用自己，或者伪装成可信工具。

工具描述、MCP Server 返回的元数据和外部注册信息都应被当作不可信输入。

### 四、一条生产级工具路由流水线

可以把路由拆成六层，每层解决不同问题。

#### 第 1 层：确定性上下文过滤

先使用代码可以明确判断的信息：

- 当前租户；
- 用户角色与权限范围；
- 开发、测试还是生产环境；
- 地区与数据驻留要求；
- 功能开关；
- 工具健康状态；
- 是否允许写操作；
- 用户是否已完成必要确认。

这一层不应交给模型猜。

```typescript
const visibleTools = registry.filter((tool) =>
  tool.status === "active" &&
  tool.environments.includes(context.environment) &&
  context.scopes.includesAll(tool.requiredScopes) &&
  tool.tenantPolicy.allows(context.tenantId)
);
```

#### 第 2 层：业务域路由

先判断请求属于订单、物流、账单还是账户安全。

可以使用：

- 明确的页面入口；
- 规则与关键词；
- 业务分类模型；
- 会话状态；
- 上一个已确认的对象类型。

如果用户是在“订单详情页”点击“查看物流”，页面上下文已经提供了强信号，没有必要让大模型重新猜业务域。

#### 第 3 层：候选召回

在选定业务域内找出可能相关的工具。

常见方案如下：

| 方案 | 优点 | 局限 |
| --- | --- | --- |
| 规则匹配 | 可解释、稳定、便宜 | 难覆盖自然语言变化 |
| 关键词检索 | 对工具名和专有名词有效 | 不擅长同义表达 |
| 向量检索 | 能处理语义相似 | 可能召回“听起来像但不能用”的工具 |
| 小模型分类 | 速度快，可输出固定类目 | 类目变化后需要维护 |
| 大模型路由 | 理解复杂意图 | 成本、延迟和可控性较差 |
| 混合召回 | 兼顾精确词和语义 | 系统更复杂，需要融合排序 |

工程上常用混合方案：关键词保证订单号、产品名等精确概念，向量检索覆盖自然语言表达，规则注入必须出现或必须排除的工具。

#### 第 4 层：重排与冲突消解

初步召回可能得到 20 个候选，重排后只保留最相关的几个。

重排特征可以包括：

- 用户问题与工具用途的语义相关度；
- 必需参数是否能从当前上下文获得；
- 工具是否具有副作用；
- 工具历史成功率和延迟；
- 当前页面或工作流状态；
- 工具版本与租户兼容性；
- 是否存在更安全的只读替代工具。

注意：不能因为某个工具历史成功率高，就让它压过业务语义完全不同的工具。可用性只能作为相关性之后的次级特征。

#### 第 5 层：加载完整 Schema

只把最终候选的完整定义发送给模型。

这时模型要回答三个问题：

1. 是否真的需要调用工具？
2. 需要调用哪个或哪些工具？
3. 参数应该是什么？

候选集合应允许“一个都不调用”。否则路由器召回了候选，就可能迫使模型产生不必要的操作。

#### 第 6 层：执行前再次控制

路由阶段的权限过滤只能减少暴露面，不能替代执行时授权。

执行器仍需检查：

- 参数是否符合 Schema；
- 业务对象是否属于当前用户；
- 操作是否超过额度；
- 资源状态是否允许操作；
- 高风险操作是否得到用户确认；
- 幂等键是否存在；
- 工具版本是否仍与路由时一致。

这叫“纵深防御”：前面筛掉大多数风险，执行点保留最终防线。

### 五、三种常见架构

#### 架构 A：静态小工具集

适合工具少、业务固定、权限简单的场景。

```text
请求 -> 全部 8 个工具 Schema -> 模型选择 -> 执行
```

优点是简单；缺点是工具增长后难以扩展。

不要为了“架构先进”而过早引入向量库和路由服务。只有十个清晰工具时，静态方案往往更可靠。

#### 架构 B：应用侧工具注册中心与路由器

```text
请求
  -> 权限过滤
  -> 业务域分类
  -> 工具检索与重排
  -> 加载候选 Schema
  -> 模型 Function Calling
  -> 受控执行器
```

它与模型供应商无关，适合多模型、多个业务域和严格权限系统。

#### 架构 C：协议发现与模型侧按需加载组合

宿主应用先从 MCP Server 发现工具，再建立本地受控索引；模型 API 如果支持延迟加载，可在会话中继续按需加载候选。

```text
MCP Servers --tools/list--> Host 工具索引
                              |
用户请求 -> 权限/检索/路由 ------+
                              |
                              v
                     模型可见的少量工具
```

这里 MCP 负责“Host 如何从 Server 发现和调用能力”，Function Calling 负责“模型如何表达调用意图”。二者仍然是不同层级。

### 六、MCP 中的工具发现

按照 MCP 规范，支持工具的 Server 可以声明 `tools` 能力，Client 使用 `tools/list` 获取工具列表。

列表可能分页，Client 需要正确处理游标，不能只取第一页。

如果 Server 声明工具列表可变化，它可以发送 `notifications/tools/list_changed`。Client 收到通知后应重新拉取，并完成以下动作：

1. 对新工具重新做信任与权限检查；
2. 更新检索索引；
3. 让已删除或禁用工具失效；
4. 处理正在执行的旧版本调用；
5. 记录工具清单变化审计日志。

`tools/list` 返回某个工具，不等于用户已经被授权调用它，也不等于工具描述可信。

### 七、厂商 Tool Search 与通用路由的关系

一些模型 API 已经提供延迟加载工具的能力。以 OpenAI 当前文档为例，应用可以声明可延迟加载的函数、命名空间或 MCP Server，再让模型通过 Tool Search 加载需要的部分；也可以由客户端执行搜索并回传工具集合。

这类能力很有价值，但要注意三点：

1. 字段名和可用模型属于厂商接口，不是通用 Function Calling 标准；
2. 权限过滤和工具信任仍应由应用控制；
3. 如果需要跨模型运行，应在内部定义统一的工具注册与路由抽象。

不要在教程或面试中把 `defer_loading`、`tool_search` 等具体字段说成所有模型都支持。

### 八、工具注册信息该怎样写

#### 1. 名称要表达动作和对象

推荐：

- `orders.get_order`
- `orders.search_orders`
- `billing.create_refund`

不推荐：

- `do_it`
- `handle`
- `query2`

#### 2. 描述要说明“什么时候用”和“什么时候不用”

差的描述：

> 查询订单。

更好的描述：

> 已知唯一订单号时查询单个订单详情。若只有手机号、时间范围或商品信息，请使用 `orders.search_orders`；本工具不修改订单。

#### 3. 把已知值留在代码中

如果租户 ID、当前用户 ID 和订单 ID 已由页面可靠提供，不要要求模型再次生成。

```typescript
async function getCurrentOrder() {
  return orderService.get({
    tenantId: session.tenantId,
    userId: session.userId,
    orderId: pageContext.orderId,
  });
}
```

这样既减少参数错误，也避免模型伪造身份边界字段。

#### 4. 风险和副作用要进入元数据

“查退款进度”和“发起退款”语义相近，但风险完全不同。

路由器应优先暴露只读工具；写工具只有在用户意图明确、权限满足且确认条件成立时才进入候选。

### 九、如何设置候选数量

不存在适用于所有系统的固定数字。

候选太少会漏掉正确工具，候选太多又会增加混淆。因此需要通过评估寻找平衡点。

可以画一条曲线：

```text
候选数 K       3      5      10      20
召回率       88%    95%     98%     99%
最终选对率   86%    92%     90%     84%
平均延迟     低     中低     中      高
```

这个例子说明：召回率随着 K 增加，但模型最终选择准确率可能在候选过多后下降。

正确做法是同时观察：

- `Recall@K`：正确工具是否进入候选；
- `Selection Accuracy@K`：候选中有正确工具时，模型是否选对；
- 无工具请求的误调用率；
- 延迟、Token 和成本；
- 高风险工具的误暴露率。

### 十、一个端到端伪代码

```typescript
async function prepareTools(request: UserRequest, ctx: RequestContext) {
  // 1. 先做确定性安全过滤。
  const allowed = await registry.listAllowed({
    tenantId: ctx.tenantId,
    scopes: ctx.scopes,
    environment: ctx.environment,
  });

  // 2. 召回只使用安全过滤后的工具摘要。
  const recalled = await hybridRetriever.search({
    query: request.text,
    conversationState: ctx.state,
    tools: allowed.map(toSearchDocument),
    limit: 20,
  });

  // 3. 重排并应用风险策略。
  const ranked = await reranker.rank(request, recalled);
  const candidates = policyEngine.selectCandidates({
    ranked,
    maxTools: 6,
    allowSideEffects: ctx.userConfirmedAction,
  });

  // 4. 只加载候选的完整、已验证 Schema。
  return Promise.all(
    candidates.map(async ({ toolId, version }) => {
      const definition = await registry.getDefinition(toolId, version);
      validateTrustedDefinition(definition);
      return toProviderToolSchema(definition);
    }),
  );
}
```

模型返回调用后，还要进入另一个执行函数：

```typescript
async function executeToolCall(call: ModelToolCall, ctx: RequestContext) {
  const tool = await registry.getPinned(call.toolId, call.version);
  validateSchema(tool.inputSchema, call.arguments);
  await authorizeAtExecutionTime(ctx.actor, tool, call.arguments);
  await requireConfirmationIfNeeded(ctx, tool, call.arguments);

  return executor.run(tool, call.arguments, {
    timeoutMs: tool.timeoutMs,
    idempotencyKey: call.callId,
  });
}
```

这两个函数不能合并成“模型选到了就直接执行”。

### 十一、常见失败模式与定位方法

#### 失败 1：正确工具根本没进入候选

现象：模型在候选中做出了看似合理的选择，但没有正确工具。

原因可能是：

- 工具描述缺少用户常用表达；
- 权限规则误过滤；
- 业务域分类错误；
- 向量索引未更新；
- 召回数量过小。

这属于路由召回问题，不能简单归咎于模型。

#### 失败 2：正确工具在候选中，但模型选错

原因可能是：

- 两个工具职责重叠；
- 描述没有写“不适用场景”；
- 参数设计让一个工具看起来更容易调用；
- 候选太多；
- 提示中没有给出选择规则。

优先修工具契约，再考虑更换模型。

#### 失败 3：选对工具但参数错误

这属于参数生成或上下文提供问题。需要看字段级准确率，而不是工具选择准确率。

#### 失败 4：注册中心仍指向旧版本

模型按旧 Schema 生成参数，执行器却按新版本校验，导致大量失败。

解决方法包括版本固定、兼容期、双读、契约测试和可回滚发布。

#### 失败 5：无工具问题也被强制调用

如果路由器总会返回候选，模型可能产生“既然有工具就用一个”的偏差。

训练集和评估集都要包含“不应调用任何工具”的负样本，并允许模型直接回答或请求澄清。

## 工程实践与边界

### 1. 先从可解释的基线开始

推荐演进顺序：

```text
静态工具集
  -> 按业务域分组
  -> 权限过滤
  -> 关键词/规则召回
  -> 混合检索与重排
  -> 动态加载和多级路由
```

不要在没有评估集时直接上线复杂的“路由 Agent”。否则出现问题时很难知道哪一层出错。

### 2. 路由输入不能包含不必要的敏感数据

检索工具通常只需要用户意图和最少的场景信息。身份证号、访问令牌、完整订单内容不应为了“提高召回率”而写入向量库。

### 3. 工具定义需要版本化

至少记录：

- Schema 版本；
- 执行实现版本；
- 生效时间；
- 兼容范围；
- 废弃状态；
- 发布人与审批记录。

一次调用的追踪信息应能还原“模型当时看见的是哪一版定义”。

### 4. 缓存要带权限与版本维度

错误做法：只按用户问题缓存候选工具。

不同租户、角色、环境和功能开关看到的工具不同。缓存键至少需要包含这些安全边界以及工具索引版本。

### 5. 不要让可用性评分绕过语义和权限

工具健康度可以用于降级，但不能让“目前最快的工具”替代语义正确的工具，更不能让备用工具获得原本没有的权限。

### 6. 动态加载也要限制总预算

需要限制：

- 单轮最多搜索次数；
- 单次最多加载工具数；
- 总 Schema Token；
- 路由与模型总延迟；
- 递归加载深度；
- 高风险工具数量。

否则模型可能反复搜索工具，形成高延迟或高成本循环。

### 7. 设计清晰的降级路径

路由服务不可用时，可以：

1. 对低风险场景回退到业务域内的静态小工具集；
2. 对写操作停止自动执行并请求人工处理；
3. 提示用户补充信息；
4. 不要直接回退到“暴露全部工具”。

### 8. 记录分层追踪数据

一次请求至少记录以下信息，并对敏感参数脱敏：

```text
request_id
tenant / actor / policy_version
registry_version
eligible_tool_count
recalled_tools + recall_scores
reranked_tools + rank_scores
loaded_tool_ids + schema_versions
model_selected_tools
validation_result
authorization_result
execution_result
final_task_result
latency_and_token_usage
```

有了分层追踪，才能进行下一篇所讲的评估与监控。

## 常见误区

### 误区一：工具越多，模型能力越强

工具只有在被正确发现、正确选择并安全执行时才产生价值。无差别暴露只会增加上下文、混淆和风险。

### 误区二：用向量检索就能解决所有工具路由

向量相似只说明文本语义接近，不说明用户有权限、参数可获得、工具风险可接受或业务状态允许执行。

### 误区三：权限可以等到执行时再检查

执行时必须再检查，但发现阶段也应过滤，避免泄露工具能力和增加恶意诱导面。

### 误区四：MCP 会自动帮模型选对工具

MCP 标准化发现和调用消息，并不保证路由质量。Host 仍要决定暴露哪些 Server、哪些工具以及如何交给模型。

### 误区五：Tool Search 是所有模型的统一标准

不同供应商的延迟加载字段和行为不同。系统需要把厂商接口放在适配层，不能写进核心业务契约。

### 误区六：候选召回率越高越好

把所有工具都召回时，召回率当然很高，但模型可能更难选择。要联合观察召回率、选择准确率、延迟和风险。

### 误区七：路由失败就换更大的模型

如果正确工具从未进入候选，再强的模型也无法选择。必须先定位失败层级。

### 误区八：工具列表变化后立即覆盖旧版本即可

进行中的调用可能仍依赖旧 Schema。生产系统需要版本固定、兼容窗口和回滚方案。

## 面试追问

### 追问一：工具达到多少个时必须做动态路由？

没有统一阈值。要看 Schema 长度、工具相似度、模型能力、延迟预算和评估结果。十个高度相似的工具可能已经需要路由，几十个职责清晰的小工具也可能仍可静态加载。应通过候选数量实验决定，而不是背固定数字。

### 追问二：为什么不能让一个大模型先选工具，再让另一个模型执行？

可以用模型做路由，但“执行”仍是应用代码完成。模型路由之前仍需确定性权限过滤；路由结果也要有置信度、无工具分支和审计，不能把模型输出直接当授权结果。

### 追问三：工具检索应该索引哪些内容？

通常索引名称、用途、适用与不适用场景、业务域、输入前提和常见用户表达。完整 Schema 可以单独保存并按需加载。权限字段用于过滤，不应简单拼进可被所有租户检索的公共文本。

### 追问四：如何处理两个职责重叠的工具？

优先从接口设计上消除重叠：合并工具、明确边界或增加确定性路由规则。如果必须共存，在描述中写清适用条件、输入前提和失败语义，并专门加入对比评估样本。

### 追问五：MCP Server 新增工具后，Host 怎么知道？

Client 可以重新调用 `tools/list`。若 Server 与 Client 协商了工具列表变化能力，Server 可以发送工具列表变化通知；Host 收到后重新拉取、验证、授权过滤并更新索引，而不是直接信任增量内容。

### 追问六：路由器应该返回一个工具还是多个工具？

通常返回少量候选，让主模型结合完整上下文决定。只返回一个工具能降低选择难度，但路由错误会直接失去恢复空间。具体取决于召回质量与延迟预算，应通过 `Recall@K` 和最终选择准确率选择 K。

### 追问七：如何防止恶意工具通过描述诱导模型？

只接入受信来源；对注册与更新做审批、签名或白名单；把工具描述当不可信数据；限制描述长度和格式；做权限过滤；动态工具加载后仍走执行授权；对新工具进行隔离测试与审计。

### 追问八：路由服务挂了，为什么不能把全部工具发给模型兜底？

因为这会突破原有权限、成本和安全边界。更安全的降级是使用经过审核的静态小工具集，或者停止高风险自动执行并转人工。

### 追问九：工具版本变化时如何保证会话一致性？

调用记录应固定工具 ID 和版本。会话内已加载的定义要么保持到本轮结束，要么通过显式迁移重新加载；执行器发现版本不兼容时应返回可识别错误，让系统重新规划，而不是静默改写参数。

### 追问十：面试中怎样用一句话区分发现、路由和调用？

可以回答：发现是知道“有哪些工具”，路由是判断“这次可能需要哪些工具”，调用是模型表达“现在要用哪个工具和什么参数”，真正执行仍由宿主应用负责。
