假设你在培训一名客服。

你不会只告诉他“回答得像客服一点”。

你可能先给他看两份回答：

- 回答 A：准确、礼貌，还给出了可执行步骤；
- 回答 B：语气生硬，而且编造了退款规则。

然后告诉他：“A 比 B 好。”

问题是，怎样把这种偏好变成模型参数的更新？

PPO 和 DPO 都在解决这个问题，但路线不同：

- PPO 先训练一个“评分员”，再让模型反复作答、打分和改进；
- DPO 直接拿“选中回答与拒绝回答”训练，不显式训练评分员。

它们不是“一个高级、一个低级”，而是在线强化学习与离线偏好优化的两种工程路线。

## 本篇目标

读完本篇，你应该能够：

- 解释 SFT、偏好数据、奖励模型、策略模型和参考模型；
- 说清 PPO 的采样、打分、优势估计和裁剪更新流程；
- 逐个符号解释 PPO 的核心目标；
- 说清 DPO 为什么可以不显式训练奖励模型；
- 逐个符号解释 DPO 损失；
- 比较两者的数据、计算、稳定性、探索能力和风险；
- 根据业务条件选择 PPO、DPO 或先 SFT 后评估；
- 回答“DPO 是否属于强化学习”等常见追问。

先记住一句话：

> PPO 用当前策略在线生成样本，再借助奖励模型更新策略；DPO 用离线偏好对直接拉开 chosen 与 rejected 的相对概率。

## 60 秒回答

PPO 和 DPO 都可用于让大模型更符合人类偏好。

PPO 是一种在线、on-policy 的策略优化方法。在典型 RLHF 中，当前策略模型生成回答，奖励模型给回答打分，价值模型估计优势，再通过裁剪目标更新策略；通常还对参考模型加入 KL 约束，防止策略偏离过快。它能够探索当前模型的新输出，但训练链路包含 rollout、奖励、价值估计和策略更新，资源成本高，也可能出现奖励利用与不稳定。

DPO 使用固定的偏好对，每条数据包含 prompt、chosen 和 rejected。它比较策略模型与参考模型对两份回答的序列对数概率差，直接优化偏好分类目标，不需要单独训练奖励模型、价值模型或运行在线 rollout。它实现更简单，通常更容易稳定训练，但能力受离线数据覆盖限制，无法主动探索数据中没有出现的行为。

工程选型时，我会看四件事：是否有高质量成对偏好数据、奖励能否在线计算、任务是否需要探索、团队是否具备稳定的 RL 训练和评测基础设施。无论选哪种，都要防止长度偏好、标注噪声、聊天模板错误和训练指标与真实质量脱节。

## 详细解析

### 1. 先把背景补齐：为什么 SFT 之后还要做偏好对齐

SFT 是监督微调。

它让模型模仿数据中的标准答案：

~~~text
输入 x：用户问题
标签 y：人工示范回答
目标：提高模型生成 y 的概率
~~~

经典语言模型损失可写成：

$$
\mathcal{L}_{\text{SFT}}
=
-\sum_{t=1}^{T}\log \pi_\theta(y_t\mid x,y_{<t})
$$

逐个解释：

- $x$ 是输入提示；
- $y_t$ 是标准回答的第 $t$ 个 Token；
- $y_{<t}$ 是它前面的 Token；
- $\pi_\theta$ 是参数为 $\theta$ 的模型；
- $T$ 是回答长度；
- 负号表示训练时最小化损失，相当于提高标准答案概率。

但很多开放问题没有唯一标准答案。

例如“给我解释量化”可以有十种都正确的写法。

人往往更容易回答：

> A 和 B 哪个更好？

而不是从零写出全世界最好的回答。

于是产生了偏好数据：

~~~json
{
  "prompt": "用户的问题",
  "chosen": "标注者更喜欢的回答",
  "rejected": "标注者不喜欢的回答"
}
~~~

`chosen` 不一定完美。

它只表示在这一对比较中胜出。

`rejected` 也不一定完全错误。

偏好优化学习的是给定比较中的相对次序，而不是经过独立验证的事实结论。

### 2. 五个必须分清的模型角色

| 角色 | 通俗理解 | 训练时是否更新 |
| --- | --- | --- |
| 策略模型 policy | 正在学习如何回答的选手 | 是 |
| 参考模型 reference | 训练开始时的基准选手 | 通常冻结 |
| 奖励模型 reward model | 给整段回答打分的评委 | PPO 阶段通常冻结 |
| 价值模型 value model | 预测未来还能拿多少奖励 | PPO 中需要训练 |
| SFT 模型 | 偏好训练的常见起点 | 可复制成策略与参考模型 |

很多面试回答混淆奖励模型和价值模型。

奖励模型回答：

> 这份完整回答最终得多少分？

价值模型回答：

> 生成到当前 Token 时，预计后面能得到多少回报？

它们用途不同。

这里列的是五种**逻辑角色**，不表示工程上必须常驻五个彼此独立的完整模型。例如价值函数可以是策略模型共享主干上的 Value Head，参考模型也可以通过参数共享、分片或卸载降低显存；无论如何，计算策略概率、参考概率、奖励和价值这四类信号仍然存在。

### 3. PPO 的完整流程

PPO 的原论文不是专门为大语言模型提出的。

在 LLM RLHF 中，通常把生成过程视为序列决策：

- 状态：prompt 加上已经生成的 Token；
- 动作：选择下一个 Token；
- 策略：模型给出的下一个 Token 概率分布；
- 轨迹：一整段生成序列；
- 奖励：奖励模型或可验证环境给出的分数。

一轮典型训练如下：

~~~text
1. 从提示数据集中取一批 prompt
2. 当前策略在线生成回答，也叫 rollout
3. 奖励模型或环境为回答打分
4. 加入与参考模型的 KL 惩罚
5. 价值模型估计每一步的价值
6. 计算 advantage
7. 用 PPO 裁剪目标更新策略
8. 用回报目标更新价值模型
9. 重复采样和训练
~~~

这里的“在线”不是指连接互联网。

它是强化学习术语：训练样本由训练过程中的当前策略产生。

PPO 属于 on-policy 方法，但一次 rollout 通常会被冻结并重复训练若干个 mini-batch epoch；它不是每做一次梯度更新就重新采样。概率比值用于限制新策略相对“产生这批数据的旧策略”变化过大，这仍不同于长期反复使用固定离线偏好集。

### 4. 什么是 reward

最简单的序列级奖励可以写成：

$$
R(x,y)=r_\phi(x,y)
$$

其中：

- $x$ 是 prompt；
- $y$ 是完整回答；
- $r_\phi$ 是参数为 $\phi$ 的奖励模型；
- $R$ 是最终分数。

真实系统可能组合多项奖励：

$$
R
=
w_1R_{\text{helpful}}
+w_2R_{\text{correct}}
+w_3R_{\text{safety}}
-w_4R_{\text{format-error}}
$$

$w_i$ 是各项权重。

危险在于模型会优化“被测量的分数”，不一定优化真正目标。

如果评分器偏爱长答案，策略可能学会无意义地变长。

这叫 reward hacking 或 reward exploitation。

### 5. 为什么要有 KL 约束

如果只追求奖励，策略可能迅速偏离原来的语言能力。

LLM RLHF 常使用如下形式的正则化奖励：

$$
R_{\text{total}}
=
R_{\text{task}}
-\beta D_{\text{KL}}(\pi_\theta\|\pi_{\text{ref}})
$$

解释符号：

- $R_{\text{task}}$ 是任务或奖励模型分数；
- $\pi_\theta$ 是当前策略；
- $\pi_{\text{ref}}$ 是冻结参考策略；
- $D_{\text{KL}}$ 衡量两个概率分布差异；
- $\beta$ 控制“追求奖励”和“不要偏离太远”的权衡。

$\beta$ 太小：

- 策略可能偏离过快；
- 奖励利用更明显；
- 语言质量可能退化。

$\beta$ 太大：

- 策略几乎不敢改变；
- 奖励提升缓慢；
- 训练可能形同没做。

KL 约束是软约束，不是安全证明。

### 6. 什么是 advantage

PPO 不直接把“最终奖励 0.8”平均分给所有 Token。

它需要估计某一步动作比当前基线好多少。

最直观的定义是：

$$
A_t=Q(s_t,a_t)-V(s_t)
$$

其中：

- $s_t$ 是第 $t$ 步状态；
- $a_t$ 是第 $t$ 步生成的 Token；
- $Q(s_t,a_t)$ 是采取该动作后的预期回报；
- $V(s_t)$ 是不指定动作时的预期回报；
- $A_t$ 是优势。

$A_t>0$ 表示这个动作比基线好。

$A_t<0$ 表示它比基线差。

工程上常用 GAE 降低方差，但会引入估计偏差。

### 7. PPO 的概率比值

PPO 核心先计算：

$$
r_t(\theta)
=
\frac{\pi_\theta(a_t\mid s_t)}
{\pi_{\theta_{\text{old}}}(a_t\mid s_t)}
$$

其中：

- $\pi_{\theta_{\text{old}}}$ 是采样时的旧策略；
- $\pi_\theta$ 是正在更新的新策略；
- 分子分母都是对同一个动作的概率；
- $r_t=1$ 表示概率没有变化；
- $r_t>1$ 表示新策略更偏好该动作；
- $r_t<1$ 表示新策略降低了该动作概率。

为什么不能无限增大优势动作的概率？

因为一次更新太大，会让采样数据迅速过时，训练不稳定。

### 8. PPO 裁剪目标逐项解释

常见策略目标是：

$$
L^{\text{CLIP}}(\theta)
=
\mathbb{E}_t\left[
\min\left(
r_t(\theta)A_t,
\operatorname{clip}(r_t(\theta),1-\epsilon,1+\epsilon)A_t
\right)
\right]
$$

逐个解释：

- $\mathbb{E}_t$ 表示对采样步骤取平均；
- $r_tA_t$ 是普通的重要性采样目标；
- $\epsilon$ 是允许变化的裁剪范围；
- `clip` 把比值限制在 $[1-\epsilon,1+\epsilon]$；
- `min` 选择更保守的一项。

这里写的是需要**最大化**的策略目标。代码如果统一使用梯度下降最小化 loss，通常会取它的负值，并再与价值损失、熵奖励等项按实现组合。

假设：

~~~text
A_t = 2
epsilon = 0.2
r_t = 1.5
~~~

未裁剪项为：

$$
1.5\times2=3
$$

裁剪后的比值最多是 $1.2$：

$$
1.2\times2=2.4
$$

目标取较小的 2.4。

这降低了继续大幅推高该动作概率的收益。

注意：裁剪只是降低大更新的激励。

它不保证 KL 始终很小，也不能排除训练失稳。

### 9. PPO 为什么成本高

典型 PPO 训练同时涉及：

- 策略模型前向与反向；
- 参考模型前向；
- 奖励模型前向；
- 价值模型前向与反向；
- 在线 rollout；
- 轨迹和 KV Cache 存储；
- 多模型并行与调度。

而且生成是逐 Token 的，rollout 往往成为主要耗时。

这就是为什么“PPO 公式只有一行”不等于“PPO 工程简单”。

### 10. DPO 的数据和流程

DPO 训练不需要在每一步在线生成新回答。

训练数据固定为：

$$
(x,y_w,y_l)
$$

其中：

- $x$ 是 prompt；
- $y_w$ 是 winner，也就是 chosen；
- $y_l$ 是 loser，也就是 rejected。

流程是：

~~~text
1. 准备偏好对
2. 用策略模型计算 chosen/rejected 的序列对数概率
3. 用参考模型计算相同概率
4. 比较策略相对参考模型增加了多少偏好差距
5. 反向传播，只更新策略模型
~~~

它仍然需要模型前向与反向。

“不做强化学习 rollout”不等于“训练没有成本”。

### 11. 序列对数概率是什么

回答 $y=(y_1,\ldots,y_T)$ 的条件概率是：

$$
\pi_\theta(y\mid x)
=
\prod_{t=1}^{T}
\pi_\theta(y_t\mid x,y_{<t})
$$

计算机更常使用对数概率：

$$
\log\pi_\theta(y\mid x)
=
\sum_{t=1}^{T}
\log\pi_\theta(y_t\mid x,y_{<t})
$$

乘法变加法，更稳定也更方便计算。

这也说明：

- Tokenizer 会影响概率；
- 聊天模板会影响条件上下文；
- 截断会改变计算范围；
- 是否包含结束符会影响序列分数；
- 长度差异可能带来偏置。

### 12. DPO 损失逐符号解释

标准 DPO 常写成：

$$
\mathcal{L}_{\text{DPO}}
=
-\mathbb{E}_{(x,y_w,y_l)}
\left[
\log\sigma\left(
\beta
\left(
\log\frac{\pi_\theta(y_w\mid x)}{\pi_{\text{ref}}(y_w\mid x)}
-
\log\frac{\pi_\theta(y_l\mid x)}{\pi_{\text{ref}}(y_l\mid x)}
\right)
\right)
\right]
$$

逐个解释：

- $\pi_\theta$ 是正在训练的策略模型；
- $\pi_{\text{ref}}$ 是冻结参考模型；
- $y_w$ 是 chosen；
- $y_l$ 是 rejected；
- $\beta$ 是从 KL 正则策略目标推导出的尺度或温度参数：在原始推导里它也是 KL 惩罚系数；放进 DPO sigmoid 后又会缩放偏好 margin。数值增大不宜简单解读为“更新更激进”，实际偏离、梯度和质量要结合实现与验证集观察；
- $\sigma$ 是 sigmoid，把实数变成 0 到 1；
- 外层负对数把问题变成可最小化损失。

先看第一项：

$$
\log\frac{\pi_\theta(y_w\mid x)}{\pi_{\text{ref}}(y_w\mid x)}
$$

它问的是：

> 当前策略相对参考模型，把 chosen 的概率提高了多少？

第二项对 rejected 提出相同问题。

两者相减后，DPO 希望：

> chosen 的相对提升大于 rejected 的相对提升。

### 13. 一个不带复杂数学的 DPO 例子

假设参考模型的对数概率为：

~~~text
chosen： -10
rejected：-11
参考偏好差：1
~~~

训练中当前策略变成：

~~~text
chosen： -8
rejected：-12
当前偏好差：4
~~~

当前策略比参考模型多拉开了 3 的差距。

DPO 会奖励这种变化。

如果模型同时提高 chosen 和 rejected，而且提高幅度相同，相对差距没有改善，收益就有限。

### 14. DPO 为什么可以不显式训练奖励模型

DPO 论文从一个带 KL 正则的奖励最大化目标出发。

在特定假设下，最优策略与奖励之间存在可代换关系。

把这个关系代入 Bradley-Terry 风格的偏好概率模型后，可以直接用策略与参考策略的概率比表达偏好。

因此 DPO 把：

~~~text
先拟合奖励模型 → 再优化策略
~~~

改写为：

~~~text
直接从偏好对优化策略
~~~

但这不表示奖励概念消失了。

论文标题所说的“模型是隐式奖励模型”，正是在强调这种关系。

### 15. DPO 与 PPO 对照表

| 维度 | PPO | DPO |
| --- | --- | --- |
| 数据来源 | 当前策略在线 rollout | 固定 chosen/rejected 偏好对 |
| 是否显式奖励模型 | 典型 RLHF 需要 | 不需要 |
| 是否价值模型 | 通常需要 | 不需要 |
| 是否参考模型 | 通常需要 | 标准 DPO 需要 |
| 是否探索新输出 | 可以 | 受离线数据覆盖限制 |
| 训练链路 | 采样、打分、估值、更新 | 对数概率比较、更新 |
| 主要成本 | rollout 与多模型协同 | 多次前向和反向 |
| 稳定性 | 超参数和系统都较敏感 | 通常更容易管理，但仍会过拟合 |
| 典型风险 | 奖励利用、价值误差、策略崩塌 | 标签噪声、长度捷径、分布外行为 |

### 16. DPO 是否属于强化学习

这是一个容易争论的术语问题。

更严谨的回答是：

- DPO 来源于 KL 正则的策略优化推导；
- 它优化策略并使用偏好信号；
- 标准训练过程不运行在线环境交互；
- 不显式学习奖励模型和价值函数；
- 实现上更像离线的监督式二分类损失。

面试中不要只回答“是”或“不是”。

把“理论来源”和“训练实现”分别说清楚更重要。

### 17. 为什么离线偏好覆盖很重要

假设偏好数据只有两种回答：

- 冗长但正确；
- 简短但错误。

模型可能学到“越长越好”这个捷径。

它没有见过：

- 简短且正确；
- 冗长且错误。

所以无法可靠区分“长度”和“正确性”。

离线学习只能从数据支持范围内学习。

### 18. 为什么 chosen/rejected 的生成方式也重要

偏好对可能来自：

- 两个不同模型；
- 同一模型不同采样；
- 人工答案与模型答案；
- 当前策略与旧策略；
- 规则筛选或可验证奖励。

如果 rejected 太差，训练很容易，却学不到细粒度判断。

如果两者差异只在格式，模型可能只学格式。

如果 chosen 总是更长，模型可能学长度偏好。

好的数据应覆盖真正希望模型学会的决策边界。

## 工程实践与边界

### 1. 一个稳妥的训练起点

常见顺序是：

~~~text
预训练模型
  ↓
领域继续预训练（可选）
  ↓
高质量 SFT
  ↓
偏好优化
  ↓
离线评测 + 人工盲评 + 灰度验证
~~~

偏好优化不能代替基础知识和指令能力。

如果 SFT 基线很差，先修数据往往比立刻上 PPO 更有效。

### 2. DPO 数据检查清单

训练前至少检查：

- prompt 是否重复或泄漏评测集；
- chosen 与 rejected 是否真正对应同一 prompt；
- 聊天角色和模板是否完整；
- 是否出现完全相同的答案对；
- 是否有明显长度、语言、格式捷径；
- 标注者一致率是否可接受；
- 安全样本是否覆盖真实风险；
- 截断是否删掉决定偏好的关键部分；
- 数据是否包含许可证或隐私风险。

### 3. 不要忘记 response mask

训练通常只应对回答部分计算偏好概率。

prompt Token 是条件，不是模型要比较的两个答案。

如果错误地把 prompt 也纳入损失：

- chosen 和 rejected 的公共前缀会浪费计算；
- 模板差异可能污染分数；
- 实现与论文目标不再一致。

### 4. DPO 需要监控什么

建议至少记录：

- train loss 与 validation loss；
- chosen reward；
- rejected reward；
- reward margin；
- preference accuracy；
- chosen/rejected 序列长度；
- 相对参考模型的 KL 代理指标；
- 输出熵和重复率；
- 关键任务离线评测。

这里的 DPO `reward` 通常是由对数概率比构造的隐式量，不是业务真实奖励。

### 5. PPO 需要监控什么

除了任务奖励，还要监控：

- 每 Token KL 与序列 KL；
- 策略熵；
- clip fraction；
- ratio 分布；
- advantage 均值和方差；
- value loss；
- explained variance；
- 回答长度；
- 奖励各分量；
- rollout 吞吐与失败率。

只看平均奖励上涨非常危险。

### 6. 参考模型、Tokenizer 和模板必须锁定

下面任一项不一致，概率比较都可能失真：

- Tokenizer 版本；
- special tokens；
- chat template；
- BOS/EOS 处理；
- 截断位置；
- 最大长度；
- LoRA 是否合并；
- 模型精度与推理模式。

参考模型不是“随便找一个同规模模型”。

标准做法通常是偏好训练起点的冻结副本。

### 7. 长度偏差怎么处理

序列对数概率是 Token 对数概率之和。

长度会自然影响总和。

可采取的办法包括：

- 构造长度更平衡的数据；
- 分桶比较；
- 独立报告不同长度区间；
- 检查长度与胜率相关性；
- 根据算法实现评估长度归一化或其他损失变体。

不能未经验证就随意改公式。

不同归一化会改变优化目标。

### 8. 什么时候更倾向 DPO

以下条件较适合 DPO：

- 已有高质量、覆盖充分的偏好对；
- 希望快速建立稳定基线；
- 计算资源无法支撑大规模在线 rollout；
- 奖励难以实时计算；
- 任务无需强探索；
- 团队更熟悉监督训练栈。

### 9. 什么时候更倾向 PPO

以下条件让 PPO 类方法更有价值：

- 环境能在线返回可靠、可验证奖励；
- 需要探索离线数据未覆盖的行为；
- 可以持续生成并筛选新轨迹；
- 团队能运营 rollout、奖励和价值训练；
- 能对 reward hacking 做独立评测；
- 延迟和计算预算允许多轮实验。

代码、数学、游戏和可执行任务常更容易获得可验证奖励。

但这不等于 PPO 自动胜出。

### 10. 最小 DPO 训练伪代码

以下仅表达算法结构：

~~~python
for batch in preference_loader:
    prompt = batch["prompt"]
    chosen = batch["chosen"]
    rejected = batch["rejected"]

    policy_chosen = sequence_logprob(policy, prompt, chosen)
    policy_rejected = sequence_logprob(policy, prompt, rejected)

    with no_grad():
        ref_chosen = sequence_logprob(reference, prompt, chosen)
        ref_rejected = sequence_logprob(reference, prompt, rejected)

    relative_margin = (
        policy_chosen - ref_chosen
        - policy_rejected + ref_rejected
    )

    loss = -log_sigmoid(beta * relative_margin).mean()
    loss.backward()
    optimizer.step()
~~~

真实实现还要处理：

- padding mask；
- 分布式训练；
- 梯度累积；
- 混合精度；
- reference-free 变体；
- adapter 切换；
- 数值稳定性。

生产项目优先使用成熟库并核对其版本语义。

### 11. 评测必须离开训练目标

至少建立四层评测：

| 层次 | 示例 |
| --- | --- |
| 能力 | 知识、推理、代码、领域任务 |
| 偏好 | 人工盲评、成对胜率 |
| 安全 | 越权、隐私、危险请求、拒答边界 |
| 工程 | 延迟、吞吐、长度、成本、稳定性 |

奖励模型高分不能证明事实正确。

DPO preference accuracy 高也不能证明线上用户满意。

### 12. 算法边界

PPO 和 DPO 都无法自动解决：

- 偏好标准本身错误；
- 标注群体不具代表性；
- 事实知识缺失；
- 数据污染；
- 训练集覆盖不足；
- 上下文外分布变化；
- 产品权限与安全设计缺陷。

算法只能优化你提供的信号。

## 常见误区

- **认为 DPO 完全不需要参考模型**：标准 DPO 目标依赖参考策略；某些变体不同，不能混为一谈。
- **认为 PPO 的裁剪保证训练稳定**：裁剪降低大更新激励，但不能消除奖励错误、价值估计误差和系统故障。
- **认为在线 RL 是联网训练**：这里的在线指当前策略持续生成训练轨迹。
- **把奖励模型和价值模型当成同一个模型**：前者给结果打分，后者估计状态价值。
- **认为 chosen 必定正确**：它只是在给定比较中被偏好。
- **认为 DPO 就是普通 SFT**：两者都可用反向传播实现，但 DPO 比较 chosen、rejected 和参考模型的相对概率。
- **认为 DPO 无条件优于 PPO**：它更简单，但不能替代在线探索。
- **只看训练 loss**：训练目标可能被长度、风格或模板捷径优化。
- **直接把 DPO 的 $\beta$ 与 PPO 的 KL 系数等同**：两者所处目标和实现不同，数值不能直接换算。
- **忽略序列概率实现细节**：mask、EOS、截断和聊天模板会直接改变损失。
- **认为偏好优化能补齐事实知识**：它主要改变行为倾向，不保证注入可靠新知识。
- **看到奖励上涨就上线**：必须用独立任务集、人工盲评和安全集复核。

## 面试追问

### 问：一句话说 DPO 和 PPO 的核心差别

**答：** PPO 用当前策略在线采样，并借助奖励模型和价值模型做策略更新；DPO 用固定偏好对直接优化策略相对参考模型的 chosen/rejected 概率差。

### 问：DPO 为什么不需要显式奖励模型

**答：** 在特定偏好模型和 KL 正则策略优化假设下，可以把最优策略与奖励的关系代入偏好概率，从而用策略相对参考策略的概率比直接表示隐式奖励。

### 问：DPO 为什么仍然需要参考模型

**答：** 标准目标比较当前策略相对参考策略对 chosen 和 rejected 的概率变化。参考模型提供锚点，限制模型不能只通过无意义地改变两个答案的概率数值来获得目标收益。

### 问：PPO 中旧策略和参考策略一样吗

**答：** 不一定。旧策略是产生当前 rollout 的策略快照，用来计算 PPO 概率比；参考策略通常是训练起点的冻结模型，用于 KL 约束。两者职责不同。

### 延伸问题

- 为什么 PPO 需要价值模型，而奖励模型不能直接替代？
- DPO 最大的数据风险是什么？
- 什么条件下值得承担 PPO 的工程复杂度？
- 标准 DPO 为什么更接近离线或 off-policy 学习？
- 偏好准确率很高时，线上效果为什么仍可能不好？
- 没有成对偏好数据时，应怎样获得可靠比较信号？
- 面试中怎样从数据、探索、成本和评测给出选型结论？
