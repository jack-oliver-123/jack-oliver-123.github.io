## 适配逻辑

1. **题目筛选**：优先选择与岗位考察矩阵匹配的题目方向
2. **难度校准**：同一技术点，不同岗位考察深度不同
   - 例：SQL → 测试岗考复杂查询+数据校验，开发岗考索引优化+事务，运营岗考基础查询+数据提取
3. **追问方向**：追问链根据岗位调整
   - 例：排查题 → 测试岗追问"如何复现和定位"，开发岗追问"如何修复和防御"，SRE追问"如何监控和预防"
4. **评分维度**：每道题的"考察能力"标注贴合岗位 JD 常见要求

## 未列出岗位处理

当 position 不在上表中时：
1. 从岗位名称提取关键词，匹配最接近的已有岗位作为基线（关键词重叠数最多者优先）
2. 结合 domain 和 topic 动态生成考察侧重（给出推理过程）
3. 在输出开头标注 `[岗位适配:动态推断]`，提醒用户验证侧重是否准确
4. **匹配失败兜底**：若岗位名称与所有已有岗位均无关键词重叠（如"声优""插画师"），则按通用后端开发出题，标注 `[岗位适配:通用兜底]`，并建议用户指定 domain 或提供更多岗位描述

**推理示例**：
- "测试开发" → 基线取"软件测试"，但编码能力权重提升到与开发岗同级
- "数据工程师" → 基线取"后端开发"，侧重数据处理/ETL/数仓方向
- "技术管理" → 基线取对应技术方向 + "团队影响力"部分全量加载
