## 本篇目标

学完这一篇，你应该能够：

- 说清 MCP Client、MCP Server 与 Authorization Server 的 OAuth 角色；
- 解释 MCP 授权是可选的传输层能力，不是 initialize 中的普通 capability；
- 按顺序描述 Protected Resource Metadata、授权服务器元数据和客户端注册；
- 解释授权码、PKCE、state、resource、scope 和 access token 的作用；
- 区分“拿到 token”“协议支持某能力”和“有权调用具体工具”；
- 识别 token passthrough、错误受众、开放重定向、SSRF 和 confused deputy 风险。

## 60 秒回答

按 MCP 2025-11-25 规范，授权对实现是可选的；实现 HTTP 传输授权时应遵循该规范，stdio 不应套用这套浏览器 OAuth 流程，而通常从环境获取凭证。受保护的 MCP Server 是 OAuth resource server，MCP Client 是 OAuth client，Authorization Server 负责用户交互与签发 token。

流程上，Client 访问受保护 MCP 端点，遇到 401 后先通过 RFC 9728 Protected Resource Metadata 找到一个或多个授权服务器，再通过 RFC 8414 或 OpenID Connect Discovery 获取授权端点和能力；然后使用预注册、Client ID Metadata Document、Dynamic Client Registration 或人工配置获得 client_id。授权码流程必须实现 PKCE，校验服务端支持，并在技术可行时使用 S256，同时使用 state 和精确 redirect URI。

Client 在授权和换 token 时使用 RFC 8707 resource 参数，把 token 绑定到目标 MCP Server；调用时只通过 Authorization Bearer header 发送。Server 要验证签发方、受众、有效期和 scope，只接受发给自己的 token，不能把收到的 token 原样传给下游 API。OAuth 只建立访问授权基础，Server 仍要对每个工具、资源和参数做业务授权、审批和审计。

## 详细解析

### 先说人话：门票、场馆和具体房间权限是三层

想进入一个办公园区：

1. 你先在售票或身份中心证明身份；
2. 获得一张只面向这个园区的通行证；
3. 到园区门口验证通行证；
4. 进入后，财务室、机房和会议室仍分别检查权限。

OAuth access token 像园区通行证。它不等于“所有房间随便进”，更不等于“可以把通行证交给隔壁园区使用”。

### 贯穿案例：远程企业文件 MCP Server

组件：

- MCP Client：桌面 AI 助手；
- MCP Server：https://files.example.com/mcp；
- Authorization Server：https://auth.example.com；
- 用户：Alice；
- Tools：search_files、read_file、share_file。

Alice 希望搜索自己的企业文件。Client 不能预先假设授权服务器地址，也不能拿一个别的服务 token 来碰运气。它要：

1. 发现文件 Server 对应哪个 Authorization Server；
2. 注册或表明自己的 OAuth client 身份；
3. 让 Alice 在浏览器完成授权；
4. 获取面向 files.example.com/mcp 的 token；
5. 调用 search_files；
6. Server 根据 Alice、scope、租户和具体文件 ACL 再判断。

即使 scope 包含 files:read，Alice 也不能读取 Bob 的私人文件。

### 最低前置概念：三个角色

| MCP 角色 | OAuth 角色 | 责任 |
| --- | --- | --- |
| MCP Client | OAuth Client | 代表资源所有者请求访问 |
| MCP Server | Resource Server | 接收并验证 access token，保护资源 |
| Authorization Server | Authorization Server | 与用户交互，签发 token |

Authorization Server 可以与 MCP Server 同域部署，也可以是独立身份平台。

### 授权是不是 MCP 必选能力

不是。MCP 2025-11-25 明确：

- Authorization 对 MCP 实现是可选的；
- 使用 HTTP 传输并支持授权时，应遵循授权规范；
- stdio 传输不应照搬该规范，通常从环境获取凭证；
- 其他传输应采用其协议成熟的安全做法。

授权发生在传输访问层，不是在 initialize 的 capabilities 中写一个 oauth 字段完成协商。
Client 即使协议上支持 Tools，也可能尚未获得 HTTP 访问 token；有 token 也可能没有某个工具的业务权限。

### 授权完整流程总览

~~~text
1. Client 请求受保护 MCP 端点
2. Server 返回 401 和可选 WWW-Authenticate 提示
3. Client 获取 Protected Resource Metadata
4. Metadata 给出 authorization_servers
5. Client 获取 Authorization Server Metadata
6. Client 确定 client_id 注册方式
7. Client 生成 state、code_verifier、code_challenge
8. 浏览器完成授权码流程
9. Client 用 authorization code + code_verifier 换 token
10. Client 携带 resource，取得面向 MCP Server 的 token
11. Client 用 Authorization: Bearer 调用 MCP Server
12. Server 验证 token，并执行工具/资源级授权
~~~

### 第一步：访问受保护资源

Client 可能先发送未带 token 的请求：

~~~http
POST /mcp HTTP/1.1
Host: files.example.com
Content-Type: application/json
~~~

Server 返回：

~~~http
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer
  resource_metadata="https://files.example.com/.well-known/oauth-protected-resource/mcp"
  scope="files:read"
~~~

401 告诉 Client 需要认证，并可通过 resource_metadata 指向 Protected Resource Metadata。

### 第二步：发现 Protected Resource Metadata

MCP Server 必须实现 RFC 9728 Protected Resource Metadata。Client 必须支持两种发现：

1. 解析 401 的 WWW-Authenticate 中 resource_metadata；
2. 若没有，按规范尝试 well-known URI。

如果 MCP 端点是：

~~~text
https://files.example.com/team/mcp
~~~

规范定义了对应子路径和根路径的 well-known 发现顺序。生产实现应使用标准 URL 构造库，不靠字符串拼接。

### Protected Resource Metadata 包含什么

至少要关注：

~~~json
{
  "resource": "https://files.example.com/mcp",
  "authorization_servers": [
    "https://auth.example.com"
  ]
}
~~~

MCP 规范要求 authorization_servers 至少包含一个授权服务器；可能有多个，由 Client 按 RFC 9728 指引和本地策略选择。

Client 要验证：

- HTTPS；
- metadata 的 resource 与目标一致；
- URL 来源可信；
- 不因恶意 metadata 把凭证发给错误站点；
- 获取过程防 SSRF 和重定向攻击。

### 第三步：发现 Authorization Server Metadata

Client 已知道 issuer 或授权服务器 URL 后，要读取：

- RFC 8414 OAuth Authorization Server Metadata；
- 或 OpenID Connect Discovery。

得到：

- authorization_endpoint；
- token_endpoint；
- 支持的 grant；
- PKCE 方法；
- scope；
- 注册能力；
- issuer；
- 其他 OAuth 安全元数据。

对于包含路径的 issuer，RFC 8414 和 OIDC 的 well-known URL 位置不同。MCP 规范给出了尝试优先级，Client 应按规范实现，不应猜一个 URL。

### 为什么要分两次发现

Protected Resource Metadata 回答：

> 这个资源由哪些授权服务器保护？

Authorization Server Metadata 回答：

> 这个授权服务器的 authorize、token、注册端点和能力是什么？

资源服务器与授权服务器不是同一个角色，不能假设同域。

### 第四步：Client 如何获得 client_id

MCP 2025-11-25 支持三类主要机制，并给出推荐优先级。

#### 1. 预注册

Client 和 Authorization Server 已有合作，提前分配 client_id、redirect URI 等。适合受控企业环境。

#### 2. Client ID Metadata Document

Client 用一个 HTTPS URL 作为 client_id，该 URL 返回客户端元数据，例如名称、redirect URI 和认证方式。适合 Client 和 Server 事先没有关系的场景。

要注意：

- URL 必须满足规范要求；
- metadata 中 client_id 与文档 URL 精确一致；
- Authorization Server 获取远程文档时要防 SSRF；
- redirect URI 必须严格校验；
- 页面和域名可能用于钓鱼，应清楚展示。

#### 3. Dynamic Client Registration

Authorization Server 可按 RFC 7591 提供 registration_endpoint，Client 动态注册。当前 MCP 把它作为兼容或特定场景选择，不应假设每个 Server 都支持。

#### 4. 人工配置

没有自动机制时，可以请求用户或管理员配置已知客户端信息。

推荐选择顺序：

~~~text
已有预注册
→ Client ID Metadata Document
→ Dynamic Client Registration
→ 人工配置
~~~

具体还要遵守 Authorization Server 元数据和企业策略。

### 第五步：生成 PKCE 与 state

#### PKCE 是什么

Client 生成高熵随机 code_verifier，再计算 code_challenge：

~~~text
code_verifier：只保存在发起授权的 Client 实例
code_challenge：随授权请求发送
~~~

换 token 时提交 code_verifier。Authorization Server 验证它与原 challenge 匹配。截获 authorization code 的攻击者没有 verifier，就难以兑换 token。

#### MCP 2025-11-25 的要求

- MCP Client 必须实现 PKCE；
- 授权前必须从元数据确认 PKCE 支持；
- 缺少支持信息时应拒绝继续；
- 技术可行时必须使用 S256；
- Authorization Server 为兼容 MCP 需要提供相应元数据。

#### state

state 将回调绑定到发起请求，防 CSRF 和响应混淆。Client 应：

- 使用不可预测值；
- 与浏览器授权会话绑定；
- 回调时精确比较；
- 一次使用；
- 过期即失效。

PKCE 与 state 解决的风险不同，两者不能互相替代。

### 第六步：发起授权请求

概念示例：

~~~http
GET /authorize?
  response_type=code&
  client_id=https%3A%2F%2Fclient.example.com%2Fmetadata.json&
  redirect_uri=http%3A%2F%2F127.0.0.1%3A43123%2Fcallback&
  code_challenge=...&
  code_challenge_method=S256&
  state=...&
  resource=https%3A%2F%2Ffiles.example.com%2Fmcp&
  scope=files%3Aread
~~~

关键点：

- redirect URI 与注册值精确匹配；
- 非本地回调使用 HTTPS；
- Client 请求最小 scope；
- resource 明确目标 MCP Server；
- 用户看清 Client、scope 和资源。

### 第七步：回调与换 token

浏览器回调：

~~~http
GET /callback?code=...&state=...
~~~

Client：

1. 检查 state；
2. 检查回调与当前授权会话；
3. 使用一次性 code；
4. 向 token_endpoint 发送 code_verifier；
5. 再携带 resource；
6. 验证 token 响应；
7. 安全存储 token。

authorization code 不是 access token，不应直接拿去调用 MCP Server。

### resource 参数为什么重要

RFC 8707 的 resource 告诉 Authorization Server：

> 我申请的 token 要给哪个资源服务器使用。

MCP Client 必须实现并使用 resource 参数。这样可以降低同一个 token 被拿到其他服务使用的风险。

Server 也要验证 token 的受众或其他目标资源绑定，只接受面向自己的 token。

### 第八步：使用 Bearer token

~~~http
POST /mcp HTTP/1.1
Host: files.example.com
Authorization: Bearer eyJ...
MCP-Protocol-Version: 2025-11-25
~~~

access token：

- 放 Authorization header；
- 不放 query string；
- 不写日志；
- 不放模型上下文；
- 不发给错误 host；
- 不复用到另一用户或 Server。

### Server 要验证什么

根据 token 类型和部署，至少包括：

- 签名或 introspection 结果；
- issuer；
- audience / resource；
- 有效期；
- not-before；
- scope；
- token 状态和撤销；
- 主体；
- 客户端；
- 租户和本地策略。

验证通过后仍要做：

~~~text
subject
→ tool
→ action
→ resource
→ arguments
→ current state
→ policy decision
~~~

### scope 与工具级授权

scope 可以表达大范围许可：

~~~text
files:read
files:write
~~~

但不能自动回答：

- Alice 能否读取 file-123；
- 是否属于同一租户；
- share_file 的目标域是否允许；
- 文件是否机密；
- 是否需要审批；
- 批量大小是否超限。

工具执行点必须继续做资源级和业务级授权。

### 能力协商与授权要分开

| 问题 | 由什么回答 |
| --- | --- |
| Client 是否会处理 Sampling | initialize capabilities |
| Server 是否提供 Tools | initialize capabilities |
| Client 是否有 HTTP token | OAuth 授权 |
| 当前主体能否调用 share_file | Server 业务授权 |
| 参数是否允许外发到该域 | 数据与工具策略 |

协议能力表示“会不会这种消息”，授权表示“这次主体能不能做”。

### Token Passthrough 为什么禁止

危险流程：

~~~text
Client 给 MCP Server 的 token
→ MCP Server 原样转发给第三方 API
~~~

问题：

- token 受众不是第三方；
- 第三方可能误信；
- MCP Server 成为 confused deputy；
- scope 和审计边界混乱；
- token 泄露面扩大。

MCP Server 调用上游 API 时，应作为独立 OAuth client 获取面向上游的单独 token，或使用经过设计的 token exchange。收到的 MCP token 不得原样透传。

### stdio 为什么不同

stdio Server 是 Client 启动的本地进程，通信通常不经过远程 HTTP：

~~~text
Client 进程
↔ stdin / stdout
本地 Server 子进程
~~~

MCP 规范建议 stdio 不使用本章 HTTP OAuth 流程，而从环境获取凭证。Host 要控制：

- 启动命令；
- 环境变量；
- 文件和网络权限；
- 凭证最小化；
- 子进程隔离；
- 用户同意。

“不走 OAuth”不代表本地 Server 天然可信。

### 常见攻击与控制

#### 错误受众

Server 接受发给其他服务的 token。
控制：验证 audience/resource，只接受发给自己的 token。

#### Token Passthrough

Server 把 Client token 传给下游。
控制：下游使用独立 token，禁止透传。

#### Confused Deputy

攻击者诱导高权限 MCP Server 代表自己访问第三方。
控制：每个 Client 和用户明确同意，严格 redirect、state 和 token 绑定。

#### 授权码截获

攻击者拿到 code。
控制：PKCE、S256、一次性 code、短有效期和精确回调。

#### 开放重定向

恶意 redirect URI 把 code 发给攻击者。
控制：精确匹配预注册 URI，不使用宽松前缀。

#### Metadata SSRF

Authorization Server 获取 Client ID Metadata Document 时访问内网。
控制：HTTPS、网络出口限制、IP 检查、大小和重定向限制。

#### 钓鱼 Client

恶意 Client 冒充可信名称。
控制：展示 client_id 域、redirect host、权限和风险；企业可使用 allowlist。

## 工程实践与边界

### Client 端伪代码

~~~python
resource_meta = discover_protected_resource_metadata(mcp_url)
validate_resource_metadata(resource_meta, expected_resource=mcp_url)

issuer = choose_authorization_server(resource_meta.authorization_servers)
auth_meta = discover_authorization_server_metadata(issuer)
verify_pkce_support(auth_meta)

client_registration = resolve_client_id(
    pre_registered=True,
    client_id_metadata_document=True,
    dynamic_registration=True,
)

state = random_secure_value()
verifier = random_secure_value()
challenge = s256(verifier)

open_user_authorization(
    authorization_endpoint=auth_meta.authorization_endpoint,
    state=state,
    code_challenge=challenge,
    resource=mcp_url,
    scope=minimal_scope,
)

code = receive_callback_and_verify_state(state)
token = exchange_code(
    code=code,
    code_verifier=verifier,
    resource=mcp_url,
)
~~~

### Token 存储

- 操作系统安全存储或专用密钥系统；
- 按用户、MCP Server、resource 和 Client 隔离；
- 静态加密；
- 最小生命周期；
- 过期、刷新、撤销；
- 用户切换清理；
- 不进入崩溃日志和剪贴板；
- 不与模型消息持久化。

### 401、403 与增量授权

一般区分：

- 401：缺 token、token 无效或过期；
- 403 / insufficient_scope：身份有效但权限不足；
- 业务 PERMISSION_DENIED：scope 可能足够，但具体资源不允许。

需要更大 scope 时：

- 清楚告诉用户原因；
- 显示新增权限；
- 用户明确同意；
- 不在后台静默扩大；
- 拒绝后提供替代路径。

具体 HTTP 错误和 challenge 遵循 OAuth Bearer Token 与 MCP 规范。

### 多租户

Server 不能只信 token 里一个 tenant 字段就访问任意资源。要：

- 验证 issuer 和签名；
- 把主体映射到本地租户；
- 查询时强制 tenant 过滤；
- 工具、缓存和审计带租户；
- token 刷新不跨账户；
- 退出后清理会话凭证。

### 可观测性

记录但不记录 token：

- metadata 发现成功率；
- Authorization Server 选择；
- PKCE 与回调错误；
- state 不匹配；
- token 交换错误类别；
- 401、403、insufficient_scope；
- audience 拒绝；
- refresh 和撤销；
- 每工具授权拒绝；
- 用户同意与取消；
- Client、Server 和策略版本。

### 故障分支

| 故障 | 处理 |
| --- | --- |
| 找不到 Protected Resource Metadata | 停止自动授权，提示配置或不兼容 |
| metadata 指向不可信 HTTP | 拒绝 |
| 没有 PKCE 支持元数据 | 按规范拒绝继续 |
| state 不匹配 | 丢弃回调并告警 |
| code 已使用 | 重新发起授权 |
| token audience 错 | Server 拒绝 |
| scope 不足 | 用户知情的增量授权 |
| refresh 失败 | 清理旧 token，重新授权 |
| 用户取消 | 不重试弹窗轰炸，提供手动恢复 |

### OAuth 的边界

- 不验证工具参数；
- 不保证资源属于用户；
- 不代替 MCP capability negotiation；
- 不自动提供最小权限；
- 不保护工具结果中的提示注入；
- 不解决写操作幂等；
- 不代替用户确认；
- 不让本地 Server 变可信。

## 常见误区

- **“所有 MCP 都必须 OAuth”**：授权是可选的，规范流程面向 HTTP 授权实现。
- **“stdio 也在浏览器走同一套 OAuth”**：当前规范建议 stdio 从环境获取凭证。
- **“Client 猜 Authorization Server URL”**：先用 RFC 9728 资源元数据发现。
- **“拿到 access token 就能调用全部工具”**：仍要 scope、主体、资源和参数级授权。
- **“capabilities 声明等于授权”**：它只表示协议功能支持。
- **“PKCE 可以替代 state”**：二者缓解不同攻击。
- **“一个 token 可以转发给下游 API”**：MCP 明确禁止 token passthrough。
- **“Bearer token 放 URL 方便调试”**：URL 会进入历史、日志和代理，应只用 Authorization header。

## 面试追问

### 1. 为什么先发现 Protected Resource Metadata？

Client 需要确认目标 MCP resource 对应哪些 Authorization Server，并避免把凭证申请或发送到错误服务。它与 Authorization Server Metadata 分工不同。

### 2. MCP Server 和 Authorization Server 必须部署在一起吗？

不必须。资源元数据会声明对应授权服务器，它们可以同域，也可以由独立身份平台提供。

### 3. 为什么必须 PKCE？

PKCE 把 authorization code 与发起授权的 Client 实例绑定，降低 code 截获和注入后被其他客户端兑换的风险。

### 4. resource 参数解决什么？

它明确 token 的目标资源，让 Authorization Server 签发面向特定 MCP Server 的 token，Server 再验证自己是预期受众。

### 5. scope 足够后为什么还要工具级授权？

scope 通常是粗粒度许可，无法表达具体文件归属、租户、外发目标、当前状态和审批。执行点仍要做资源级业务授权。

### 6. Token Passthrough 为什么危险？

Client token 的受众是 MCP Server，不是下游 API。原样转发会破坏受众和审计边界，并引发 confused deputy 风险。

### 7. Client ID Metadata Document 与 Dynamic Registration 有什么区别？

前者用 HTTPS URL 作为 client_id 并托管客户端元数据；后者调用 RFC 7591 注册端点获得 client_id。MCP 当前规范推荐已有预注册优先，其次 CIMD，再按支持情况用 DCR。

### 8. OAuth 与 initialize 的顺序是什么？

对受保护 HTTP 端点，Client 先完成取得传输访问所需 token，之后才能正常建立 MCP 消息交互并完成 initialize。授权不是 initialize capabilities 中的普通能力。
