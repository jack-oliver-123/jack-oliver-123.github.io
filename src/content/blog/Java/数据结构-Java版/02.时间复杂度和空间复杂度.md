## 引子：如何衡量一个算法的好坏

看一个经典的例子——计算斐波那契数列第 N 项：

```java
public static long fib(int N) {
    if (N < 3) {
        return 1;
    }
    return fib(N - 1) + fib(N - 2);
}
```

这段代码能正确运行，但当 N 稍微大一点（比如 40、50），程序就会卡住很久。原因是它的时间复杂度是 O(2^N)——每增加 1，计算量翻倍。

那如何**客观地**评价一个算法的效率？靠运行时间吗？不行，同一段代码在不同机器上跑出的时间天差地别。我们需要一种**与机器无关**的度量方式——这就是复杂度分析。

## 算法效率

衡量算法效率有两个维度：

- **时间效率（时间复杂度）**：算法执行需要多少基本操作。
- **空间效率（空间复杂度）**：算法运行需要多少额外内存。

两者统称为**算法复杂度**。在实际开发中，我们更关注时间复杂度，因为现代计算机内存相对充裕，而用户对响应速度的感知更敏感。

## 时间复杂度

### 概念

时间复杂度不是计算程序跑了多少秒，而是计算算法中**基本操作的执行次数**。

以下面这个函数为例：

```java
void func1(int N) {
    int count = 0;
    for (int i = 0; i < N; i++) {         // 执行 N 次
        for (int j = 0; j < N; j++) {     // 每次执行 N 次
            count++;                       // 共 N² 次
        }
    }

    for (int k = 0; k < 2 * N; k++) {     // 执行 2N 次
        count++;
    }

    int M = 10;
    while ((M--) > 0) {                   // 执行 10 次
        count++;
    }
    System.out.println(count);
}
```

精确计算执行次数：`F(N) = N² + 2N + 10`

| N | F(N) |
|---|------|
| 10 | 130 |
| 100 | 10,210 |
| 1000 | 1,002,010 |

当 N 越来越大时，`2N + 10` 相对于 `N²` 的影响越来越小，可以忽略不计。这就引出了大 O 表示法。

### 大 O 表示法（Big O Notation）

大 O 表示法描述的是算法运行时间随输入规模增长的**渐近上界**——即最坏情况下的增长趋势。

数学定义：如果存在常数 c > 0 和 N₀，使得当 N > N₀ 时恒有 f(N) ≤ c · g(N)，则称 f(N) = O(g(N))。

简单说就是：**忽略常数和低阶项，只关注最高阶的增长趋势。**

**推导规则：**

1. 用常数 1 取代所有加法常数项。
2. 只保留最高阶项。
3. 去掉最高阶项的系数。

以 `func1` 为例：

```
F(N) = N² + 2N + 10
     → 保留最高阶：N²
     → 去掉系数：N²
     → O(N²)
```

| N | 精确值 F(N) | 大 O 估算 N² |
|---|------------|-------------|
| 10 | 130 | 100 |
| 100 | 10,210 | 10,000 |
| 1000 | 1,002,010 | 1,000,000 |

N 越大，大 O 表示法越精确地反映真实趋势。

### 常见时间复杂度量级

从快到慢排列：

```
O(1) < O(log N) < O(N) < O(N log N) < O(N²) < O(2^N) < O(N!)
```

不同量级在 N 增长时的差异：

| N | O(1) | O(log N) | O(N) | O(N log N) | O(N²) | O(2^N) |
|---|------|----------|------|-----------|-------|--------|
| 10 | 1 | 3 | 10 | 33 | 100 | 1,024 |
| 100 | 1 | 7 | 100 | 664 | 10,000 | 1.26×10³⁰ |
| 1000 | 1 | 10 | 1000 | 9,966 | 1,000,000 | — |

当 N = 100 时，O(2^N) 的计算量已经是天文数字。这就像每天折纸翻倍——第一天 1 层，第二天 2 层，到第 100 天纸张厚度已经超过可观测宇宙的直径。

### 最好、最坏与平均情况

有些算法的执行次数不仅取决于输入规模，还取决于输入数据本身：

- **最好情况（Best Case）**：输入最有利时的复杂度。
- **最坏情况（Worst Case）**：输入最不利时的复杂度。
- **平均情况（Average Case）**：所有可能输入的期望复杂度。

比如在一个无序数组中查找某个值（类似 `ArrayList.contains()` 的内部逻辑）：

```java
int find(int[] array, int value) {
    for (int i = 0; i < array.length; i++) {
        if (array[i] == value) {
            return i;
        }
    }
    return -1;
}
```

- 最好情况：第一个元素就是目标，O(1)。
- 最坏情况：目标不存在或在最后，O(N)。
- 平均情况：O(N/2) → O(N)。

**在实际分析中，我们一般关注最坏情况**，因为它给出了算法性能的底线保证。

### 面试快速判断口诀

大部分复杂度可以通过模式识别快速判断：

| 代码模式 | 复杂度 | 生活类比 |
|---------|--------|---------|
| 没有循环，直接计算 | O(1) | 查字典直接翻到页码 |
| 单层循环遍历 | O(N) | 逐页翻书找某一段 |
| 嵌套两层循环 | O(N²) | 班级里每对同学握手 |
| 每次砍掉一半 | O(log N) | 猜数字游戏（1-100 猜中间） |
| 分治后合并（排序） | O(N log N) | 归并排序：拆成小堆再合并 |
| 递归调用两次自身 | O(2^N) | 家族树：每代人数翻倍 |

## 时间复杂度实例分析

### 实例 1：O(N)

```java
// 类似场景：统计日志文件行数
void func2(int N) {
    int count = 0;
    for (int k = 0; k < 2 * N; k++) {
        count++;
    }

    int M = 10;
    while ((M--) > 0) {
        count++;
    }
    System.out.println(count);
}
```

执行次数：`2N + 10`，去掉系数和常数 → **O(N)**

### 实例 2：O(N + M)

```java
// 类似场景：合并两个数据源
void func3(int N, int M) {
    int count = 0;
    for (int k = 0; k < M; k++) {
        count++;
    }
    for (int k = 0; k < N; k++) {
        count++;
    }
    System.out.println(count);
}
```

执行次数：`M + N`。由于无法确定 M 和 N 的大小关系，两者都不能省略 → **O(N + M)**

> 如果题目明确说明 M 远大于 N，则可以简化为 O(M)；反之亦然。

### 实例 3：O(1)

```java
// 类似场景：根据配置执行固定次数的初始化操作
void func4(int N) {
    int count = 0;
    for (int k = 0; k < 100; k++) {
        count++;
    }
    System.out.println(count);
}
```

执行次数恒为 100，与 N 无关。常数次操作 → **O(1)**

> O(1) 不代表只执行 1 次，而是表示执行次数为常数，与输入规模无关。HashMap 的 get 操作就是 O(1)——无论 map 里有 100 个还是 100 万个键值对，查找速度基本不变。

### 实例 4：冒泡排序 O(N²)

```java
void bubbleSort(int[] array) {
    for (int end = array.length; end > 0; end--) {
        boolean sorted = true;
        for (int i = 1; i < end; i++) {
            if (array[i - 1] > array[i]) {
                int tmp = array[i - 1];
                array[i - 1] = array[i];
                array[i] = tmp;
                sorted = false;
            }
        }
        if (sorted) {
            break;
        }
    }
}
```

分析内层循环执行次数：
- 第 1 轮比较 N-1 次
- 第 2 轮比较 N-2 次
- ……
- 第 N-1 轮比较 1 次

总次数 = `(N-1) + (N-2) + ... + 1 = N(N-1)/2`

去掉系数和低阶项 → **O(N²)**

最好情况（数组已有序）：外层跑一轮就 break → O(N)

### 实例 5：二分查找 O(log N)

```java
int binarySearch(int[] array, int value) {
    int begin = 0;
    int end = array.length - 1;
    while (begin <= end) {
        int mid = begin + ((end - begin) / 2); // 避免整数溢出
        if (array[mid] < value)
            begin = mid + 1;
        else if (array[mid] > value)
            end = mid - 1;
        else
            return mid;
    }
    return -1;
}
```

每次循环将搜索范围缩小一半，就像猜数字游戏——每次都猜中间值，范围立刻减半：

```
N → N/2 → N/4 → N/8 → ... → 1
```

设循环 x 次后终止：`N / 2^x = 1` → `x = log₂N`

时间复杂度 → **O(log N)**

> 复杂度中的 log 底数通常是 2，书写时一般省略底数直接写 log N。

**直观感受**：在 10 亿（约 2³⁰）个有序元素中查找，二分查找最多只需要 30 次比较。

### 实例 6：归并排序 O(N log N)

```java
void mergeSort(int[] array, int left, int right) {
    if (left >= right) return;
    int mid = left + (right - left) / 2;
    mergeSort(array, left, mid);       // 递归左半部分
    mergeSort(array, mid + 1, right);  // 递归右半部分
    merge(array, left, mid, right);    // 合并，O(N) 操作
}
```

分析思路：
- 每一层递归都要对所有元素做一次合并操作 → 每层 O(N)
- 每次将问题规模对半分 → 共 log₂N 层
- 总复杂度 = O(N) × O(log N) → **O(N log N)**

> Java 的 `Arrays.sort()` 对对象数组使用 TimSort（基于归并），对基本类型数组使用双轴快排，平均复杂度都是 O(N log N)。这就是为什么上一篇中说"排序 → O(N log N)"。

### 实例 7：阶乘递归 O(N)

```java
long factorial(int N) {
    return N < 2 ? 1 : factorial(N - 1) * N;
}
```

递归调用 N 次，每次做 O(1) 的乘法操作 → **O(N)**

### 实例 8：斐波那契递归 O(2^N)

```java
int fibonacci(int N) {
    return N < 2 ? N : fibonacci(N - 1) + fibonacci(N - 2);
}
```

画出递归调用树：

```
              fib(5)
            /        \
        fib(4)       fib(3)
       /     \       /     \
   fib(3)  fib(2) fib(2) fib(1)
   /    \
fib(2) fib(1)
```

每一层的调用次数大约是上一层的 2 倍，共约 N 层，总调用次数约为 `2^N` → **O(2^N)**

### 各实例总结

| 实例 | 代码模式 | 时间复杂度 |
|------|---------|-----------|
| func2 | 单循环 | O(N) |
| func3 | 两个独立循环 | O(N + M) |
| func4 | 固定次数循环 | O(1) |
| 冒泡排序 | 嵌套循环 | O(N²) |
| 二分查找 | 每次减半 | O(log N) |
| 归并排序 | 分治 + 合并 | O(N log N) |
| 阶乘递归 | 线性递归 | O(N) |
| 斐波那契递归 | 双路递归 | O(2^N) |

## 优化思路：从 O(2^N) 到 O(N)

回到开头的斐波那契问题。分析完复杂度后我们知道问题在哪了——大量重复计算。`fib(3)` 被算了很多次。

解决方案：**用空间换时间**——把计算过的结果存起来（记忆化 / 迭代）：

```java
// 迭代版：时间 O(N)，空间 O(1)
long fib(int N) {
    if (N < 3) return 1;
    long prev = 1, curr = 1;
    for (int i = 3; i <= N; i++) {
        long next = prev + curr;
        prev = curr;
        curr = next;
    }
    return curr;
}
```

| 实现方式 | 时间复杂度 | 空间复杂度 |
|---------|-----------|-----------|
| 递归（无优化） | O(2^N) | O(N)（栈深度） |
| 记忆化递归 | O(N) | O(N)（缓存数组） |
| 迭代（滚动变量） | O(N) | O(1) |

这就是复杂度分析的实际价值——找到瓶颈，指导优化方向。

## 空间复杂度

### 概念

空间复杂度衡量的是算法在运行过程中**额外**占用的内存空间（不包含输入数据本身占的空间），同样使用大 O 表示法。

分析方法：看算法中额外申请了多少变量/数组/对象，它们的大小与输入规模 N 的关系是什么。

### 实例 1：冒泡排序 O(1)

```java
void bubbleSort(int[] array) {
    for (int end = array.length; end > 0; end--) {
        boolean sorted = true;
        for (int i = 1; i < end; i++) {
            if (array[i - 1] > array[i]) {
                int tmp = array[i - 1];
                array[i - 1] = array[i];
                array[i] = tmp;
                sorted = false;
            }
        }
        if (sorted) {
            break;
        }
    }
}
```

额外空间：只用了 `end`、`sorted`、`i`、`tmp` 几个变量，都是常数个 → **O(1)**

这属于**原地排序**算法——不需要额外的数组来辅助。

### 实例 2：斐波那契迭代版 O(N)

```java
long[] fibonacci(int n) {
    long[] fibArray = new long[n + 1];
    fibArray[0] = 0;
    fibArray[1] = 1;
    for (int i = 2; i <= n; i++) {
        fibArray[i] = fibArray[i - 1] + fibArray[i - 2];
    }
    return fibArray;
}
```

额外空间：申请了一个大小为 `n + 1` 的数组 → **O(N)**

### 实例 3：阶乘递归 O(N)

```java
long factorial(int N) {
    return N < 2 ? 1 : factorial(N - 1) * N;
}
```

每次递归调用都会在调用栈上分配一个栈帧，共递归 N 层 → **O(N)**

> 常见误区：递归斐波那契的空间复杂度是 O(N) 而不是 O(2^N)。因为递归调用树虽然有 2^N 个节点，但同一时刻栈上最多只有 N 层调用——左子树回溯后栈帧释放，才会进入右子树。

### 实例 4：归并排序 O(N)

归并排序需要一个与原数组等长的临时数组来做合并操作，空间复杂度 → **O(N)**

### 空间复杂度小结

| 实例 | 额外空间 | 空间复杂度 |
|------|---------|-----------|
| 冒泡排序 | 几个临时变量 | O(1) |
| 斐波那契迭代 | 长度为 N 的数组 | O(N) |
| 阶乘递归 | N 层递归栈帧 | O(N) |
| 归并排序 | 长度为 N 的辅助数组 | O(N) |

## 均摊复杂度简介

有些操作**单次最坏**是 O(N)，但连续执行 N 次的**总代价**是 O(N)，均摊下来每次只有 O(1)。

典型例子：`ArrayList.add()` 的尾部插入。大多数时候直接写入 O(1)；偶尔数组满了需要扩容——拷贝整个数组 O(N)。但扩容不是每次都发生（容量翻倍策略），均摊下来每次 add 仍然是 O(1)。

> 下一篇讲 ArrayList 源码时会详细分析这个过程。

## 常见复杂度速查表

| 复杂度 | 名称 | 典型场景 |
|--------|------|---------|
| O(1) | 常数 | 数组下标访问、HashMap.get/put |
| O(log N) | 对数 | 二分查找、TreeMap 操作 |
| O(N) | 线性 | 遍历数组、ArrayList.contains |
| O(N log N) | 线性对数 | Arrays.sort、归并排序 |
| O(N²) | 平方 | 冒泡排序、选择排序 |
| O(2^N) | 指数 | 递归斐波那契、穷举子集 |
| O(N!) | 阶乘 | 全排列 |

## 常见误区

1. **"循环里调用方法，复杂度不变"** ——错！如果循环内调用了 O(N) 的方法（如 `list.contains()`），总复杂度是 O(N²)。
2. **"递归空间是调用总次数"** ——错！空间复杂度看的是同一时刻栈帧的最大深度，不是调用总次数。
3. **"O(N) 一定比 O(N²) 快"** ——当 N 很小时不一定。大 O 描述的是增长趋势，常数因子可能让 O(N²) 在小 N 时更快。

## 小结

- 复杂度分析是一种**与机器无关**的算法效率度量方式，关注增长趋势而非绝对数值。
- 大 O 表示法取最高阶项、去系数，描述最坏情况下的渐近上界。
- 时间复杂度看基本操作执行次数，空间复杂度看额外申请的内存大小。
- 快速判断口诀：单循环 O(N)、嵌套 O(N²)、减半 O(log N)、分治合并 O(N log N)、双路递归 O(2^N)。
