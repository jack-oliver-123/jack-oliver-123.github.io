# AI 面试系列 2026 全面升级设计

## 目标

将 `src/content/blog/AI应用开发/` 下现有 79 篇文章升级为一套适用于 2026 年面试与工程实践的原创内容，并新增 28 道高价值题目。升级后的系列必须技术准确、结构统一、可维护、可验证，不再依赖“小林面试笔记”的文字、结构或图片。

## 当前基线

- 现有文章：79 篇，共约 11,881 行
- 带“小林面试笔记”来源标记：76 篇
- 第三方 `cdn.xiaolincoding.com` 图片：3 张
- 无语言标签代码块：100 个
- 工作区已有用户进行中的文章重命名和内容修改，实施时必须保留
- 文件名决定公开 URL 与系列顺序，现有文件不得重命名

## 非目标

- 不修改博客视觉设计、路由或内容集合 schema
- 不把面试系列改成纯学术教材
- 不为了追热点加入缺少稳定资料或工程价值的问题
- 不在构建阶段访问外网验证链接
- 不提交或覆盖用户当前未提交的其他改动

## 编辑策略

采用“系列级升级”，而不是局部修补或完全推倒重写。

### 现有文章

1. 保留现有文件名、题目编号和 frontmatter 字段。
2. 从题目本身和权威资料重新组织正文，不逐句改写原有来源文本。
3. 修正上一轮审阅中发现的技术错误、版本过时、代码错误和过度绝对化表述。
4. 删除套路化、贬低候选人的对话；模拟面试最多保留 2 至 4 轮专业追问。
5. 保留有价值的原理讲解，但补齐适用条件、反例、失败场景、安全边界和工程取舍。
6. 只有在文章完成独立重写后，才删除末尾“小林面试笔记”来源标记。
7. 删除或替换第三方图片；站点发布内容不得继续引用 `xiaolincoding.com` 图片。

### 新文章

新文章沿用现有数字前缀排序，追加到各专题末尾。每道题必须满足以下条件：

- 2026 年仍有明确工程价值或高概率面试价值
- 能找到官方规范、官方文档或原始论文支撑
- 与现有题目边界清楚，不能只换标题重复内容
- 包含可以直接用于面试的答案和生产实践中的限制

## 统一文章结构

普通技术题统一使用以下结构，专题介绍页可以按目录用途精简：

1. frontmatter
2. 2 至 4 轮专业面试追问
3. `## 60 秒回答`
4. `## 详细解析`
5. `## 工程实践与边界`
6. `## 常见误区`
7. `## 面试追问`
8. `## 参考资料`

具体规则：

- “60 秒回答”控制在 150 至 300 个中文字符，先给结论再给理由
- 每个主要章节首段先说明该章节解决的问题
- 缩写首次出现时给出中文或英文全称
- 不使用“永远”“必然”“完全”“事实标准”等无法证明的绝对表述
- 百分比、延迟、吞吐和成本倍数必须标注模型、硬件、数据集、版本或来源；无法补足时改为定性表述
- 代码示例必须标注语言，并展示完成该知识点所需的最小路径
- 涉及外部输入或副作用时，示例必须说明校验、授权、超时、重试、幂等或注入风险
- 不要求模型公开隐藏思维链；涉及推理过程时区分可验证中间产物与私有推理
- 语气专业、直接、有压力感，但不嘲讽候选人

## 事实与引用规则

资料优先级如下：

1. 正式协议规范和标准
2. 官方产品或项目文档
3. 原始论文和作者维护的项目仓库
4. 可复现实验或公开评测
5. 可信二手资料，仅用于背景，不作为关键技术结论的唯一依据

每篇技术文章的“参考资料”列出 2 至 6 个真正支撑正文的链接。正文不堆砌引用，但时效性强、容易争议或包含数字的结论应就近注明来源。引用内容必须转述，不能大段复制。

对时效性强的协议、模型或框架使用“截至 2026 年 7 月”的语境。避免把某个当前 API、模型 ID 或实验特性写成永久规则。实验功能必须明确标注实验状态。

## 必须纠正的高风险结论

本次升级不能只调整措辞，至少要逐项消除以下已确认问题：

- Agent：Single-Agent 可以并行调用工具；Workflow 的确定性取决于节点和外部系统；上下文超限不会自动以统一方式丢弃最旧消息；角色切换和 Reflection 不自动带来客观评审
- RAG：RAG 不能从根源消除幻觉；向量检索可以通过查询分解和迭代检索完成多跳任务；增大 `top-k` 可以提高召回但会增加噪声与成本；Embedding 维度越高不等于效果必然越好
- 工具调用：MCP 不依赖 Function Calling；MCP Tools 不能按“是否有副作用”定义；推理模型的工具调用不要求暂停同一次生成并长期占用 KV Cache；A2A 文章必须按当前 Message、Task、Agent Card 和多协议绑定修订
- 大模型工程：LoRA 矩阵维度和初始化必须可计算且符合原论文；训练数据量、FLOPs 和显存数字必须重新核算；量化推理不包含优化器状态；vLLM 已支持 Automatic Prefix Caching；MoE 权重容量必须与设备显存总量一致
- 全系列：删除没有条件的固定性能倍数、命中阈值和“最佳参数”；无法复现实验的数据不能包装成普遍规律

## 图像与可视化

使用项目级 `baoyu-infographic` 配置为文章生成原创信息图：

- 布局：按内容自动选择
- 风格：按内容自动选择
- 比例：横版 16:9
- 语言：简体中文
- 图片后端：自动选择，当前环境优先使用 Codex 原生 `imagegen`

删除现有 3 个第三方 CDN 图片。102 道面试题每篇至少生成并上传 1 张信息图，根总览和 4 个专题介绍也各生成 1 张，因此公开图片链接不得少于 107 个。复杂文章在单张图无法清楚表达多个独立关系时可以增加第 2 张或第 3 张。

主信息图放在“60 秒回答”之后、“详细解析”之前；补充信息图放在它实际解释的章节附近。不得为了满足数量而重复正文大段文字，图应承担比较、流程、结构、层级、指标或决策信息。

布局与风格按信息结构映射：

- 流程与生命周期：`linear-progression` 或 `circular-flow`，优先 `technical-schematic`
- 二元或多方案比较：`binary-comparison` 或 `comparison-matrix`
- 架构和组件：`structural-breakdown` 或 `hierarchical-layers`
- 指标与评测：`dashboard` 或 `dense-modules`
- 分类与知识地图：`tree-branching`、`hub-spoke` 或 `bento-grid`
- 专题导航：`winding-roadmap` 或 `bento-grid`

正式生成前，按 `baoyu-infographic` 的确认规则向用户提交一次全系列布局、风格、比例、语言和后端映射确认。确认可以覆盖同一批次，不逐图重复询问；若单篇需要偏离已确认映射，应单独说明。

每张图保留完整可复现记录：

```text
docs/infographics/ai-interview/<series>/<nn-slug>/
├── source.md
├── analysis.md
├── structured-content.md
├── prompts/infographic.md
├── infographic.png
└── upload.json
```

生成结果通过 `picgo-upload` 上传。当前环境优先调用正在运行的 PicGo 桌面端本地服务，复用用户已有图床配置；桌面端不可用时才按技能规则检查 CLI。上传成功后把 `imgUrl`、`type`、文件名、尺寸和上传时间写入 `upload.json`，文章使用该公开 URL 和描述性 alt 文本。

`infographic.png` 作为本地生成与验收文件保留，但通过 `.gitignore` 排除，不提交 107 张以上位图。`source.md`、`analysis.md`、`structured-content.md`、最终 prompt 和 `upload.json` 纳入版本控制。所有上传图片均视为有意公开，任何 prompt 或图片不得包含凭据、内部地址或未公开数据。

上传只发生在图片通过文字与事实检查之后。网络或 5xx 错误最多重试一次；登录、配额、文件类型或配置错误不盲目重试。重新生成图片后必须重新上传、更新 `upload.json` 和文章 URL，不能继续引用旧图。

图片文字必须清晰、准确且与文章一致。发现错字、乱码或错误数据时必须修改 prompt 后重新生成，不得通过 SVG、Canvas、Pillow、ImageMagick 或其他程序覆盖位图文字。只允许不改变文字与主体构图的裁剪、缩放、压缩和格式转换。

## 自动内容检查

新增无第三方运行时依赖的 `scripts/lint-content.mjs`，并在 `package.json` 增加：

```json
{
  "scripts": {
    "lint:content": "node scripts/lint-content.mjs",
    "test:content": "node --test scripts/__tests__/lint-content.test.mjs scripts/__tests__/upload-infographics.test.mjs",
    "upload:infographics": "node scripts/upload-infographics.mjs"
  }
}
```

检查范围固定为 `src/content/blog/AI应用开发/**/*.{md,mdx}`。

硬错误：

- 缺少必需 frontmatter 字段
- 未标注语言的代码块
- 未闭合的代码块
- `原文来源`、`xiaolinnote.com` 或 `xiaolincoding.com` 残留
- `xiaolinnote.com`、`xiaolincoding.com` 或没有对应 `upload.json` 的远程图片
- 文章缺少至少一个由 PicGo 上传的 AI 面试信息图链接
- 信息图分析记录、结构化内容、最终 prompt 或上传记录缺失
- 同一目录数字编号重复或断裂
- 指向不存在文件的本地图片或 Markdown 链接
- 非介绍类技术文章缺少“参考资料”章节

警告：

- `100%`、固定性能倍数、固定阈值等需要人工复核的强断言
- 过时模型 ID、协议版本或带年份但未注明语境的内容
- 过长的模拟面试对话
- 缺少 alt 文本的图片

脚本输出使用 `file:line - message` 格式。硬错误返回非零退出码，警告不阻断构建。

`scripts/upload-infographics.mjs` 读取信息图 manifest，按专题和批大小调用 `picgo-upload` 自带的 GUI 上传脚本，将返回结果按输入顺序写入各自的 `upload.json` 并更新远程 URL。它必须提供 `--dry-run`，不得输出或持久化 PicGo secret；部分失败时保留成功记录并明确列出失败文件。

## 风险控制

- 现有文章存在用户未提交修改：按当前工作树内容继续编辑，只修改任务范围内的段落，不恢复旧版本。
- 全量重写范围大：按专题分批完成，每批先通过内容检查再进入全站构建。
- 官方文档可能继续变化：文章标注时间语境，参考资料使用稳定规范页或项目仓库。
- 107 张以上信息图生成与上传量大：按专题和文章编号分批处理，每批先检查文字和事实，再上传和接入正文；禁止在未验收上一批时继续大规模生成。
- 图床 URL 属于外部依赖：保留 prompt、分析和上传记录；最终验收时检查公开可访问性，但不在站点构建过程中访问网络。
- 图片可能影响版面：统一以 16:9 为主，保持稳定宽高，构建后使用浏览器抽查桌面和移动端。
- 内容原创性不足：以权威资料重新搭建论证结构，不把删除来源标记当成原创完成标准。
