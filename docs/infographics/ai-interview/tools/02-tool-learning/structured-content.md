# 2. LLM 是如何学会调用外部工具的？

## Overview

介绍 LLM 学会调用外部工具的训练机制，梳理工具描述、参数生成、监督数据和反馈优化在工具使用中的作用

## Learning Objectives

The viewer will understand:

1. 复述“LLM 是如何学会调用外部工具的？”的关键流程与控制点
2. 说明关键工程边界、失败模式与验证方法

---

## Source Content (Verbatim)

👔面试官：模型参数量变大后，会自然学会工具调用吗？

🙋我：它可能学到 API 和代码知识，但稳定输出指定调用格式通常还需要工具使用数据和接口约束。

👔面试官：模型具体学哪些能力？

🙋我：至少包括判断是否调用、选择工具、填参数、读取结果，以及在失败后调整计划。

## 60 秒回答

LLM 学会工具调用，不是获得了执行权限，而是学会在给定工具描述和对话状态时预测合适的调用表示。预训练提供语言、代码和 API 语义基础；监督微调把“请求、工具清单、调用、结果、回答”串成目标序列；偏好优化或基于执行结果的反馈继续改善选择和参数质量。

推理时，宿主把可用工具及其 Schema 放进上下文，模型按训练过的格式产生调用。宿主执行并返回观察结果，模型再决定回答还是继续调用。模型能力、供应商 API 是否暴露工具通道、宿主是否实现执行循环，是三个独立条件。

## 详细解析

### 模型需要学会四类决策

- **调用判断**：能直接回答时不乱调，需要实时或私有数据时才调用
- **工具选择**：在语义相近的工具中选对名称
- **参数绑定**：从对话抽取值，并满足类型、枚举与必填约束
- **结果利用**：识别成功、空结果和错误，再回答、澄清或改用其他工具

只训练“每个问题都调用一个工具”会形成过度调用。数据必须包含无需调用、信息不足、工具不存在、拒绝执行和调用失败等负例。

### 常见学习路径

Toolformer 展示了模型可通过自监督方式筛选有助于语言建模的 API 调用。Gorilla 和 ToolLLM 等工作则研究了工具文档检索、调用数据构造和泛化。商业模型的具体训练配方通常不公开，因此面试中应把这些方法称为可验证的研究路径，而不是断言某个闭源模型采用了某一套流程。

“会写 JSON”也不等于“会用工具”。模型要同时理解工具语义、参数约束和多轮协议角色。

## 工程实践与边界

应用侧要给模型提供清晰、互斥的工具描述，并只暴露当前身份可用的工具。工具数量过多时，可先检索候选工具，再把小集合交给模型，减少上下文开销和误选。

评估不能只看最终回答。至少分别统计：

- 是否应调用的判断准确率
- 工具名称准确率
- 参数结构与参数值准确率
- 执行成功率和任务成功率
- 越权、注入和高风险误调用率

线上失败样本可以进入人工标注和回归集，但要清除隐私数据，避免把错误执行结果直接当成正确标签。

## 常见误区

- **涌现能力会自动产生稳定协议格式**：语义知识不能保证符合产品接口
- **只依赖监督微调**：数据覆盖、解码约束和宿主反馈同样影响结果
- **工具越多能力越强**：大而相似的工具集合会增加选择歧义
- **最终回答正确就说明调用正确**：模型可能绕过工具猜中答案，评估会被高估

## 面试追问

**追问：为什么需要“不调用工具”的训练样本？**

它建立调用门控。缺少负例时，模型容易把所有请求都映射成工具调用。

**追问：工具执行失败要不要用于训练？**

要覆盖，但标签应教模型识别可重试错误、需要澄清的错误和不可恢复错误，不能统一写成“再试一次”。
