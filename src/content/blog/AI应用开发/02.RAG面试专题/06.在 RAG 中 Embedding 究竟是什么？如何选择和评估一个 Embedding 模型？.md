## 本篇目标

读完本篇，你应该能够：

- 用地图坐标类比解释 Embedding；
- 说清查询向量、文档向量与向量相似度怎样参与召回；
- 区分余弦相似度、点积和欧氏距离，并说明归一化条件；
- 从语言、领域、长度、部署、成本和许可等维度筛选模型；
- 使用 Recall@K、MRR、nDCG 和端到端指标完成公平评测；
- 解释更换 Embedding 模型为什么通常需要重建索引。

## 60 秒回答

Embedding 是把文本映射到固定维度向量的过程。检索模型通过训练，让语义相关的查询和文档在向量空间中具有更高相似度。RAG 离线为文档 Chunk 生成向量并建立索引，在线为用户 Query 生成向量，再按余弦、点积或欧氏距离寻找近邻候选。

选型时我先看语言与领域覆盖、查询和文档是否需要不同前缀、最大输入长度、向量维度、许可、隐私和部署约束；再用自有查询与相关文档标注，在相同 Chunk、距离函数和检索参数下比较 Recall@K、MRR、nDCG，并测编码吞吐、P95 延迟、索引体积和成本。公开 MTEB 榜单只适合初筛，不能代替业务数据。

相似度分数没有通用阈值。结果取决于模型版本、文本分布、是否归一化和数据库返回的是相似度还是距离。更换模型后，新旧向量空间通常不可比较，因此需要双索引迁移、离线验收和灰度切换。

## 详细解析

### 先说人话：把文字放到一张“意思地图”上

普通地图用经纬度表示地点：

- 上海和苏州在地图上较近；
- 上海和巴黎较远。

Embedding 可以粗略理解成一张高维“意思地图”。模型把文本转换成一串数字坐标，让相关内容更接近。

例如：

- 查询：“上海出差酒店能报多少？”
- 文档 A：“上海地区普通员工住宿费报销上限”
- 文档 B：“办公电脑采购流程”

训练良好的检索 Embedding 应让查询更靠近文档 A，而不是文档 B。

实际向量不是二维地图，而是几百或几千维数字。每一维通常没有可以直接命名的业务含义，多个维度共同表达文本特征。

### Embedding 不是什么

#### 不是原文的可逆压缩

通常不能只根据一个向量精确还原原文。向量也不能代替用于生成和引用的正文。

#### 不是加密

Embedding 不能被当成安全保护。向量可能泄露语义或被用于推断，仍需访问控制、加密存储和合规管理。

#### 不是一个通用语义真值

不同模型训练目标不同。同一段文本经过两个模型，会落在两个不同向量空间。分数不能跨模型直接比较。

#### 不是知识库本身

知识库还需要原文、元数据、版本、权限、索引和更新流程。Embedding 只是其中一种检索表示。

### RAG 中 Embedding 怎样工作

#### 离线：编码文档

1. 把文档切成 Chunk；
2. 按模型要求添加文档前缀或任务说明；
3. 使用指定 tokenizer 截断或分批；
4. 生成文档向量；
5. 根据模型要求做归一化；
6. 将向量、Chunk ID 和元数据写入索引。

文档向量可以提前计算，所以在线查询不需要重新编码全部文档。

#### 在线：编码查询

1. 接收用户 Query；
2. 必要时结合对话历史进行改写；
3. 按模型要求添加查询前缀；
4. 使用同一模型系列生成查询向量；
5. 采用与索引一致的归一化和距离函数；
6. 在向量索引中搜索近邻；
7. 返回候选 Chunk。

这类架构常被称为双塔或双编码器：查询和文档分别编码，然后快速比较向量。

### 查询和文档为什么可能使用不同输入格式

查询通常较短，表达“我想找什么”；文档较长，表达“我包含什么”。

有些模型使用同一个编码器，但要求加入不同前缀，例如：

- 查询前加“query:”；
- 文档前加“passage:”；
- 或使用任务说明告诉模型这是检索任务。

这些前缀是训练协议的一部分，不是可随意忽略的装饰。漏掉前缀可能造成质量下降。

选型和上线时应记录：

- 模型名称与修订版本；
- tokenizer 版本；
- 查询模板；
- 文档模板；
- 是否归一化；
- 向量维度；
- 距离函数。

### 相似度到底怎样计算

#### 余弦相似度

余弦相似度关注两个向量方向的夹角，较少受向量长度影响。

数学范围通常是负一到一，但真实模型输出可能只覆盖其中一部分。某个模型中的 0.8，不能当成所有模型通用的相关阈值。

#### 点积

点积同时受到方向和向量长度影响。许多检索模型直接使用点积作为训练与排名目标。

如果两个向量都经过 L2 归一化，点积与余弦排序等价；未归一化时，两者不是同一回事。

#### 欧氏距离

欧氏距离衡量空间中的直线距离，越小表示越近。

一些数据库返回距离，一些返回相似度。前者通常越小越相似，后者通常越大越相似。必须查看当前产品和字段配置，不能只看列名叫 score 就猜方向。

#### 为什么配置要完全对齐

假设模型文档说明要求归一化后使用点积，但索引中的文档向量没有归一化，查询向量却归一化了。系统仍会返回结果，但排序语义已经偏离模型训练方式。

因此，Embedding 模型、预处理、归一化和索引距离函数要作为一个整体版本管理。

### Embedding 模型是怎样学会“相关”的

许多现代检索模型使用对比学习。

训练数据中包含：

- 查询；
- 相关文档，也叫正样本；
- 不相关或容易混淆的文档，也叫负样本。

训练目标让查询靠近正样本，远离负样本。

“上海住宿上限”与“上海住宿标准及审批条件”是正对；“上海办公用品采购”主题有部分重合但不是答案，可作为较难负样本。

困难负样本很重要，因为真实检索中的错误通常不是随机文本，而是“看起来很像但没有回答问题”的内容。

不过，错误标注的负样本会伤害模型。如果一段文档其实也能支持答案，却被当成负样本，训练会错误地把相关内容推远。

### 模型选型第一步：先列硬约束

硬约束不满足，公开分数再高也不能上线。

#### 语言覆盖

系统是否包含：

- 简体中文；
- 繁体中文；
- 中英混合；
- 多语言查询与跨语言文档；
- 特殊缩写和内部代号。

应按语言分别评测，不能只看总体平均。

#### 领域覆盖

通用问答、法律、医疗、代码、电商和企业制度的词汇与相关性标准不同。目标领域数据比公开通用榜单更有说服力。

#### 最大输入长度

模型是否能完整编码当前 Chunk？超过限制后是报错还是截断？截断方向是什么？这些行为要通过模型文档和测试确认。

#### 部署与隐私

- 是否允许把文本发送给外部 API；
- 是否要求本地或特定地域部署；
- 模型权重和许可证是否允许商用；
- 是否支持所需并发和可用区；
- 供应商如何处理输入数据。

#### 模型与索引兼容

向量维度、数据类型、归一化和距离函数必须被目标检索系统支持。

### 模型选型第二步：比较软指标

#### 检索质量

在自有标注集上比较 Recall@K、MRR、nDCG 等指标。

#### 编码性能

分别测：

- 文档离线吞吐；
- 单条与批量查询延迟；
- P50、P95 和 P99；
- CPU、GPU 或 API 配额；
- 批处理效率。

#### 存储与检索成本

向量维度越高，原始向量存储和距离计算通常越重，但质量不随维度单调提升。有些模型支持 Matryoshka 表示，可在经过相应训练的前提下截断维度并评测质量与成本取舍。

#### 稳定性和版本管理

托管 API 是否可能升级默认模型？能否固定修订版本？自部署模型是否具备长期维护能力？

#### 总拥有成本

包括：

- 离线全量编码；
- 增量编码；
- 在线 Query 编码；
- 索引存储和副本；
- 计算资源；
- 迁移重建；
- 运维与监控。

### 公共榜单怎样正确使用

MTEB 是覆盖多种 Embedding 任务和数据集的基准。它适合：

- 了解模型大致能力；
- 按语言和任务初步筛选候选；
- 排除明显不满足需求的模型。

它不能替代业务评测，因为你的系统可能具有：

- 内部缩写；
- 极短查询；
- 长制度文档；
- 高比例编号；
- 多语言混合；
- 特殊“相关”标准；
- 完全不同的负样本分布。

合理流程是：

> 公共榜单初筛 3–5 个候选 -> 自有数据离线评测 -> 目标硬件性能测试 -> 端到端灰度。

### 怎样建立自有检索评测集

每条样本至少包含：

| 字段 | 含义 |
| --- | --- |
| query | 真实或代表性的用户问题 |
| relevant_chunk_ids | 可以支持答案的 Chunk |
| minimum_evidence_set | 完整回答所需的最小证据集合 |
| hard_negatives | 主题相似但不支持答案的片段 |
| slice | 语言、部门、问题类型等分组 |
| answerable | 知识库是否存在答案 |

数据来源可以是：

- 真实搜索日志脱敏采样；
- 客服常见问题；
- 文档作者提供的问题；
- 线上失败案例；
- 人工构造的边界和无答案样本。

训练集、调参集和最终测试集应隔离，避免反复调参把测试集“调熟”。

### 常见检索指标

#### Recall@K

前 K 个结果中是否包含相关证据。

适合回答：“正确文档有没有进入候选集？”

如果问题需要多个证据，还应测充分证据集召回，而不是命中任意一个片段就算成功。

#### MRR

关注第一个相关结果出现得多靠前。相关结果排在第一位得分更高。

适合只有一个主要答案或关注首个命中的任务。

#### nDCG

考虑多个结果的相关程度和排序位置，允许“高度相关、部分相关、不相关”等分级标注。

适合一条查询存在多个不同价值证据的场景。

#### Precision@K

前 K 个结果中有多少相关。它反映候选噪声，但在第一阶段召回中通常需要与 Recall 一起观察。

### 把模型质量和 ANN 索引质量分开

ANN 是近似最近邻搜索，用速度和资源换取接近精确搜索的结果。

如果线上召回差，可能有两个原因：

1. Embedding 本身把相关文档排得不靠前；
2. Embedding 排序正确，但 ANN 索引近似搜索漏掉了它。

排查方法：

1. 在小规模语料上使用精确向量搜索作为参考；
2. 比较不同 Embedding 的精确检索结果；
3. 固定 Embedding 后，再比较 ANN 与精确结果的 Recall@K；
4. 调整 HNSW、IVF 等索引参数，画召回与延迟曲线。

这样不会把索引参数问题误判为模型问题。

### 端到端评测仍然不可缺少

Embedding Recall 提升，不代表最终答案同步提升。

可能出现：

- 召回了更多相关文档，也带来更多冲突；
- Top-K 变长，正确证据被放到上下文中间；
- 重排器没有利用新增候选；
- 模型无法综合多个证据；
- 延迟和成本超出业务目标。

所以最终还要测答案正确性、忠实度、引用、拒答、延迟和成本。

### 相似度阈值怎样设置

不存在跨模型通用的“0.8 代表相关”规则。

阈值会受到：

- 模型与版本；
- 归一化方式；
- 距离函数；
- Query 和文档长度；
- 领域与语言；
- 数据库分数转换；
- 问题类型；
- 文档规模。

可以在标注集上画正样本和负样本分数分布，根据误召回与漏召回成本选取阈值。不同路由或问题类型也可以采用不同策略。

很多系统不会只靠一个阈值拒答，而会同时使用：

- Top-1 与 Top-2 分差；
- 重排分数；
- 是否存在充分证据；
- 模型的引用覆盖；
- 业务规则。

### 更换 Embedding 模型为什么要重建索引

不同模型的坐标系不同。新模型生成的查询向量，与旧模型生成的文档向量通常没有可解释的距离关系。

安全迁移流程可以是：

1. 固定旧索引继续服务；
2. 使用新模型重算文档向量；
3. 写入独立新索引；
4. 跑离线质量与性能评测；
5. 影子查询比较新旧结果；
6. 灰度读取新索引；
7. 观察线上质量、延迟与错误；
8. 切换读别名；
9. 保留旧索引回滚窗口。

不要在同一个向量字段中逐条混入两个模型的向量。

### 员工制度案例的选型实验

候选模型 A、B、C 都支持中文。可以这样比较：

1. 固定同一批制度、Chunk 和元数据；
2. 为每个模型构建独立索引；
3. 使用同一组真实问题；
4. 按模型文档分别使用正确前缀、归一化和距离函数；
5. 先跑精确检索，再跑相同目标延迟下的 ANN；
6. 按地区、职级、编号查询、口语查询和无答案问题分组；
7. 比较检索、端到端质量和资源。

某模型总体 Recall 更高，但如果它在“制度编号精确查询”分组表现差，生产系统可能仍需混合关键词检索。

## 工程实践与边界

### 建立 Embedding 模型卡

至少记录：

- 模型名和修订版本；
- 许可证与来源；
- 支持语言和任务；
- 最大输入长度；
- 向量维度；
- 查询与文档模板；
- 归一化要求；
- 推荐距离函数；
- 部署硬件或 API 版本；
- 离线和线上评测结果。

### 对长文本做截断监控

记录：

- 超过最大输入长度的 Chunk 比例；
- 被截断的 Token 数；
- 截断发生在哪一侧；
- 不同文档类型的异常比例。

如果大量 Chunk 被截断，应先调整解析或切块，不要假设模型仍看到了完整内容。

### 按业务切片报告

至少按以下维度拆分：

- 语言；
- 查询长短；
- 关键词型与语义型；
- 高频与长尾；
- 文档类型；
- 有答案与无答案；
- 新旧制度；
- 权限过滤强度。

总体均值可能掩盖某个关键用户群的回归。

### 缓存要绑定版本

查询向量缓存键应包含规范化 Query、模型版本和输入模板版本。模型更新后不能继续复用旧向量缓存。

### 设置回退方案

Embedding API 失败时，可以：

- 使用短时安全缓存；
- 回退到关键词检索；
- 返回明确的服务降级提示；
- 对高风险查询转人工。

不应静默使用与文档索引不兼容的备用 Embedding 模型。

## 常见误区

- **“向量维度越高，检索越准”**：质量由训练数据、目标和任务匹配决定，维度还会增加存储与计算。
- **“余弦分数 0.8 是通用阈值”**：分数分布依赖模型、数据、归一化和数据库实现。
- **“公开榜单第一就适合生产”**：榜单任务与业务数据、延迟、许可和部署约束可能不同。
- **“所有 Embedding 都直接用余弦”**：应按模型训练协议和索引配置选择点积、余弦或距离。
- **“查询和文档用同一个模型就不会出错”**：还要遵循查询前缀、文档前缀、tokenizer 和归一化要求。
- **“换模型只需修改模型名称”**：新旧向量空间通常不兼容，需要重算文档向量并重建索引。
- **“ANN 召回差就是 Embedding 差”**：先与精确检索比较，隔离模型和索引损失。
- **“Embedding 不包含原文，所以不敏感”**：向量仍可能携带语义信息，必须纳入数据安全治理。

## 面试追问

1. **为什么归一化后点积和余弦排序相同？**

   L2 归一化把每个向量长度变为 1，点积中的长度因素被去除，此时点积等于余弦值。未归一化时不能这样推断。

2. **如何评估多语言 Embedding？**

   分别构造同语言检索和跨语言检索样本，按语言报告指标，再观察低资源语言和中英混合查询，不能只看总体平均。

3. **怎样选择向量维度？**

   先按模型支持方式生成或截断维度，在同一数据上比较质量、索引体积、构建时间和查询延迟。普通模型不能随意截断后假设语义不受影响。

4. **精确搜索很好，ANN 很差怎么办？**

   说明表示模型可能没问题，重点调整索引类型、构建参数和搜索参数，并检查过滤策略和资源限制。

5. **为什么要加入困难负样本？**

   随机负样本太容易。困难负样本与查询主题相近但不支持答案，可以训练或评测模型是否理解细微相关性；同时要过滤假负例。

6. **Embedding 模型离线 Recall 提升后如何上线？**

   构建独立新索引，做端到端回归和影子流量，对比检索、答案、延迟、成本与安全切片，再灰度切换并保留旧索引回滚。
