你家厨房里有一把只给自己使用的菜刀。
为了使用它，没有必要建立一套“厨具发现服务”、网络连接和版本协商。
拿起来用就够了。
但如果你经营一座大型共享厨房，几十个团队需要使用不同厨具，厨具由不同供应商维护，还要统一登记、授权和更新，那么标准化目录与接入方式就有价值。
选择直接 Function Calling 还是 MCP，也不能只看“项目大不大”。
真正应该看的是：
- 能力归谁维护；
- 会被多少 Host 复用；
- 是否需要运行时发现；
- 是否有独立部署边界；
- 引入协议后的收益能否覆盖成本。

## 本篇目标
读完本篇，你应该能够：
- 说明这不是简单二选一；
- 判断何时直接 Function Calling 更合适；
- 判断何时值得把能力封装为 MCP Server；
- 从所有权、复用、发现、传输、安全、延迟和成本做权衡；
- 分析已有 REST API 是否需要 MCP Adapter；
- 避免按项目规模或工具数量机械选型；
- 设计 Function Calling 与 MCP 的组合架构；
- 给出渐进迁移方案；
- 回答本地工具、远程服务、单体应用和平台化等追问。

## 60 秒回答
Function Calling 和 MCP 不是同一层的替代方案，所以选型不是简单二选一。
如果工具只服务一个应用，由同一团队维护，接口稳定，需要紧贴业务事务或进程内低延迟调用，Host 直接维护工具注册表并执行函数通常最简单。
如果能力有独立所有者和发布周期，需要被多个 AI Host 复用，需要标准化的 Tools、Resources、Prompts、运行时发现、生命周期、版本协商或本地与远程传输，那么封装为 MCP Server 更有价值。
生产系统也常组合使用：模型通过 Function Calling 表达调用意图，Host 通过 MCP 调用外部 Server。
选型重点不是项目大小或工具数量，而是集成边界与长期维护收益。
无论选择哪种方式，参数校验、用户授权、确认、超时、幂等、补偿、审计和提示注入防御都不会自动消失。

## 详细解析

### 1. 先明确两个“使用”的含义
“使用 Function Calling”通常表示：
> Host 把工具定义提供给模型，让模型返回结构化调用意图。
“使用 MCP”通常表示：
> Host 通过 MCP Client 连接一个或多个 MCP Server，发现和调用外部能力。
同一个系统完全可以同时做两件事。
所以问题不应是：
~~~text
Function Calling 还是 MCP？
~~~
更准确的问题是：
~~~text
模型怎样表达调用？
能力怎样接入 Host？
~~~

### 2. 三种常见架构
#### 架构一：直接 Function Calling
~~~text
模型
  ↓ 调用意图
Host
  ↓ 本地注册表
内部函数 / 业务服务
~~~
适合专用、稳定、同团队维护的能力。
#### 架构二：Function Calling + MCP
~~~text
模型
  ↓ 调用意图
Host
  ↓ MCP Client
MCP Server
  ↓
业务系统
~~~
适合模型决策与标准化外部接入结合。
#### 架构三：确定性 Workflow + MCP
~~~text
固定工作流
  ↓ MCP Client
MCP Server
~~~
适合步骤固定、不需要模型选择的自动化任务。

### 3. 适合直接 Function Calling 的场景
#### 场景一：应用内部专用工具
例如客服应用内部有：
~~~text
get_current_conversation()
set_draft_reply()
mark_ticket_resolved()
~~~
这些能力与当前内存状态和 UI 紧密绑定。
没有其他 Host 需要使用。
直接注册为模型工具更简单。
#### 场景二：同团队、同发布周期
工具和 Host 由同一代码库、同一团队发布。
接口变化可以同步修改。
引入独立 Server 和协议适配可能没有明显收益。
#### 场景三：低延迟进程内调用
例如读取当前编辑器选区。
数据就在 Host 内存中。
把它变成远程服务会增加序列化和网络成本。
即使使用 MCP，本地 stdio 也仍有进程通信成本。
是否值得要实测。
#### 场景四：紧贴业务事务
例如一个单体应用已经有成熟的事务服务层。
Host 可以直接调用领域服务。
为了形式统一把每个方法拆成 MCP Server，可能破坏原有事务边界。
#### 场景五：短期原型
原型只有两个只读工具，目标是验证用户价值。
先直接调用能减少接入成本。
但原型进入生产前仍要补安全和治理。

### 4. 直接 Function Calling 的优势
- 链路短；
- 调试简单；
- 进程内延迟低；
- 与应用状态容易结合；
- 不需要单独管理 Server；
- 事务和权限可复用现有代码；
- 初始开发成本低。

### 5. 直接 Function Calling 的代价
当多个 Host 出现后，可能产生：
- 每个 Host 维护一套工具定义；
- 多种模型格式重复适配；
- 工具版本不同步；
- 生命周期各自实现；
- 能力发现不统一；
- 认证与审计重复建设；
- 业务团队难以独立发布。
这些问题可能推动能力协议化。

### 6. 适合 MCP 的场景
#### 场景一：能力需要跨 Host 复用
同一个代码仓库能力要被：
- IDE；
- 桌面助手；
- 代码评审 Agent；
- 自动化平台
共同使用。
一个标准 MCP Server 比四套专用适配更容易治理。
#### 场景二：能力有独立所有者
例如财务团队维护报销能力。
AI 平台团队不应该复制财务规则。
财务团队可维护 MCP Server 或 Adapter，并独立发布。
#### 场景三：需要运行时发现
Server 的工具、资源或提示会变化。
Host 需要在运行时列举和感知变化。
MCP 提供相应协议语义。
#### 场景四：不仅需要 Tool
一个能力域同时提供：
- Tool：创建工单；
- Resource：工单字段说明；
- Prompt：工单总结模板。
MCP 的多种原语比单纯函数列表更完整。
#### 场景五：需要本地进程标准接入
桌面应用需要连接本地：
- 文件系统；
- Git；
- 开发工具；
- 命令行程序。
stdio MCP Server 可以形成统一接入边界。
#### 场景六：需要远程标准接入
企业服务需要跨网络为多个 Host 提供能力。
Streamable HTTP、生命周期和授权规范能减少专用协议设计。
#### 场景七：希望利用现有 MCP 生态
如果目标 Host 和 Server 已支持同一版本与能力，采用 MCP 可以复用 SDK、Inspector 和现有集成。
但仍要做供应链和安全审查。

### 7. MCP 的优势
- 标准化角色和消息；
- 支持能力列举；
- 支持版本与能力协商；
- Tools、Resources、Prompts 语义统一；
- 本地和远程都有标准传输；
- 多个 Host 可复用；
- Server 可独立演化；
- 有官方 SDK 和调试工具。

### 8. MCP 的代价
- 多一层协议适配；
- 连接和生命周期管理；
- Schema 映射；
- 版本兼容；
- 本地子进程治理；
- 远程认证与授权；
- 网络故障；
- 多一层可观测性；
- Server 供应链风险；
- 团队学习成本。
标准化不是免费。
只有复用和治理收益足够大时，成本才值得。

### 9. 选型决策表

| 问题 | 更偏直接 Function Calling | 更偏 MCP |
| --- | --- | --- |
| 工具归谁维护 | Host 同团队 | 独立团队或第三方 |
| 使用者数量 | 单一 Host | 多个 Host |
| 发布周期 | 与 Host 同步 | 独立发布 |
| 能力发现 | 静态注册即可 | 需要运行时列举与变化通知 |
| 能力类型 | 主要是内部函数 | Tool、Resource、Prompt 组合 |
| 部署位置 | 进程内 | 本地子进程或远程服务 |
| 延迟要求 | 极低、内存态 | 可接受协议边界 |
| 生态复用 | 不需要 | 希望兼容现有 Client/Server |
| 版本协商 | 应用内部处理 | 需要协议化协商 |
| 初始成本 | 希望最低 | 可以承担平台化投入 |
这些维度都只是决策信号，需要结合实际约束判断。
选型是综合权衡。

### 10. 一个实用决策流程
按顺序问：
~~~text
1. 能力是否只给一个 Host 使用？
   是 → 继续评估直接调用
   否 → MCP 收益增加

2. 能力是否与 Host 同团队、同发布周期？
   是 → 直接调用更简单
   否 → 独立 Server 边界更清晰

3. 是否需要 Tools 之外的 Resources/Prompts？
   是 → MCP 收益增加

4. 是否需要运行时发现、版本与能力协商？
   是 → MCP 收益增加

5. 是否有极低延迟或强事务耦合？
   是 → 谨慎引入进程外边界

6. 复用收益能否覆盖协议、运维和安全成本？
   否 → 暂不引入
   是 → 设计 MCP 接入
~~~

### 11. 不能按项目大小判断
大项目也可能适合直接调用。
例如一个大型单体内有稳定领域服务，工具只服务这一个产品。
小项目也可能适合 MCP。
例如一个小型客户端需要接入已有标准 MCP Server。
所以：
~~~text
项目规模 ≠ 集成边界
~~~

### 12. 不能只按工具数量判断
五百个内部工具不一定都要变成 MCP。
它们可能只是同一应用的内部能力。
五个工具也可能值得 MCP。
如果它们由独立团队维护并被多个 Host 复用。
工具数量主要影响：
- 目录治理；
- 路由；
- 上下文预算；
- 评测。
不是协议选型的唯一条件。

### 13. 已有 REST API 要不要包 MCP
不需要全部重写。
常见做法是增加 Adapter：
~~~text
MCP Client
    ↓
MCP Adapter Server
    ↓
已有 REST API
~~~
值得包装的条件：
- 多个 AI Host 需要接入；
- 希望统一 Tool/Resource/Prompt 语义；
- 业务团队愿意维护稳定契约；
- 认证授权能正确透传；
- 长期复用收益明确。
不值得的条件：
- 只有一个内部调用者；
- 接口变化非常频繁；
- 只是为了展示“用了 MCP”；
- Adapter 会绕过现有安全网关；
- 没有维护责任人。

### 14. 本地工具怎样选
如果工具逻辑就在 Host 内：
- 当前选区；
- 当前会话；
- UI 状态；
- 内存草稿；
直接工具通常更自然。
如果是独立本地程序：
- Git；
- 文件服务；
- 数据分析 CLI；
- 可复用开发工具；
stdio MCP 可能更合适。
本地不等于无延迟，也不等于安全。

### 15. 远程工具怎样选
已有远程 API 且只有一个 Host 使用：
~~~text
Host → REST API
~~~
可能已经足够。
多个 Host 需要统一 AI 语义：
~~~text
多个 Host → MCP Server → REST API
~~~
MCP Adapter 才可能产生复用价值。

### 16. 组合架构中的责任
~~~text
模型
  负责：选择候选工具、生成参数

Host
  负责：筛选、校验、授权、确认、映射、审计

MCP Client
  负责：协议会话与消息

MCP Server
  负责：暴露并执行能力、服务端授权

业务系统
  负责：领域规则、事务、幂等和数据一致性
~~~

### 17. 渐进迁移方案
不要一次性重写全部工具。
可以分四步：
#### 第一步：建立内部工具接口
把模型格式与业务执行解耦。
~~~typescript
interface ToolExecutor {
  execute(name: string, args: unknown, context: ExecutionContext): Promise<Result>;
}
~~~
#### 第二步：选择稳定可复用能力
优先：
- 只读；
- 契约稳定；
- 多 Host 需求明确；
- 所有者清晰；
- 风险可控。
#### 第三步：增加 MCP Adapter
复用原业务服务，不复制业务规则。
#### 第四步：灰度双通道
在测试环境对比：
- 结果一致性；
- 延迟；
- 错误语义；
- 权限；
- Trace；
- 运维成本。
稳定后再逐步切换。

### 18. 什么时候不该上 MCP
- 只是为了追新技术；
- 没有跨 Host 复用需求；
- 工具紧贴当前进程内状态；
- 业务接口还在每天大改；
- 团队无法维护 Server 生命周期；
- 无法建立安全授权；
- 延迟预算极紧且收益有限；
- 现有 API 已经很好满足唯一调用方。

### 19. 什么时候不该只用直接 Function Calling
- 每个 Host 都复制一套工具适配；
- 工具定义严重漂移；
- 多团队发布互相阻塞；
- 能力需要 Tool、Resource、Prompt；
- 本地与远程接入各自为政；
- 版本和发现已经成为主要维护成本；
- 业务团队需要独立治理外部能力。

### 20. 成本模型
可以把总成本粗略理解为：
~~~text
总成本
= 初始开发
+ 协议适配
+ 运维
+ 安全治理
+ 版本升级
+ 故障排查
- 复用节省
- 重复适配节省
~~~
MCP 的收益往往在多 Host、多团队和长期演化中体现。
单一小链路的短期成本可能反而更高。

## 工程实践与边界

### 1. 先做能力清单
为每项能力记录：
- 所有者；
- 使用 Host；
- 风险等级；
- 是否只读；
- 调用频率；
- 延迟要求；
- 发布周期；
- 输入输出；
- 认证方式；
- 是否需要 Resource/Prompt；
- 是否值得复用。
有了清单，选型才有事实基础。

### 2. 不要复制业务规则到 Adapter
MCP Adapter 应调用现有领域服务。
如果把报销政策复制一份到 Adapter，会产生双重事实来源。
正确结构：
~~~text
MCP Adapter
  ↓
统一领域服务
  ↓
数据库
~~~

### 3. 保留 Host 侧安全策略
即使 Server 做授权，Host 仍要：
- 隐藏无权工具；
- 控制数据流；
- 请求用户确认；
- 限制高风险工具；
- 记录模型决策；
- 防止跨 Server 泄露。

### 4. Server 侧也不能信任 Host
Server 必须：
- 验证身份；
- 检查资源级权限；
- 校验参数；
- 实施租户隔离；
- 限流；
- 审计；
- 不相信 Host 传来的“已授权”布尔值。

### 5. 延迟必须实测
对比：
- 进程内直接函数；
- 本地 stdio；
- 同机 HTTP；
- 跨网络 Streamable HTTP；
- 真实业务依赖。
MCP 本身不是固定延迟数字。
瓶颈可能在模型或业务 API。

### 6. 设计降级路径
MCP Server 不可用时，可选策略：
- 只读缓存；
- 备用 Server；
- 直接说明不可用；
- 转人工；
- 使用原有内部接口；
- 延迟任务。
高风险写操作不应静默降级到不受控路径。

### 7. 契约版本化
工具变化要考虑：
- 名称；
- Schema；
- 默认值；
- 返回格式；
- 错误码；
- 权限；
- 幂等语义。
兼容更新可原地演化。
破坏性变化应有新版本、迁移期和回归测试。

### 8. 同一套评测对比两种链路
固定：
- 用户任务；
- 模型；
- 工具语义；
- 数据；
- 并发；
- 网络环境。
比较：
- 任务成功率；
- P50/P95 延迟；
- 错误率；
- 恢复能力；
- 运维复杂度；
- 接入新 Host 的成本。

### 9. 防止工具列表过大
无论直接注册还是 MCP，都不要把全部工具一次交给模型。
使用：
- 权限过滤；
- 领域分组；
- 语义检索；
- 规则路由；
- 动态加载；
- 候选重排。

### 10. 不因协议隐藏业务风险
工具名称叫“update_profile”也可能覆盖敏感字段。
风险评估应看真实副作用，而不是协议名称。

## 常见误区

### 误区一：小项目用 Function Calling，大项目用 MCP
不准确。
关键是边界、复用和所有权。

### 误区二：工具超过某个数量就必须 MCP
错误。
数量只会增加目录治理需求。

### 误区三：用了 MCP 就不用 Function Calling
错误。
模型仍可用 Function Calling 选 MCP Tool。

### 误区四：已有 REST API 必须全部改造
错误。
只有复用收益明确时才需要 Adapter。

### 误区五：认为 MCP 通常比直接调用慢
不能一概而论。
本地与远程、进程内与网络链路差异很大。

### 误区六：MCP 自动统一所有模型 API
错误。
Host 仍要做供应商工具格式适配。

### 误区七：MCP Server 做授权后 Host 不用管
错误。
两侧责任不同，都需要控制。

### 误区八：认为标准化会直接降低成本
短期可能增加成本。
收益取决于复用规模和生命周期。

## 面试追问

### 追问一：只有两个工具，要不要 MCP
不能只看数量。
如果两个工具被多个 Host 复用且独立维护，可能值得。
如果只是应用内部函数，通常不值得。

### 追问二：已有 API Gateway，MCP 会替代它吗
通常不会。
API Gateway 继续负责流量、认证、限流等基础设施。
MCP Server 可以在其上提供 AI 能力语义。

### 追问三：怎样避免过度设计
先用内部执行接口解耦模型与业务。
只有跨 Host 复用和独立治理需求明确时，再增加 MCP Adapter。

### 追问四：本地文件为什么适合 stdio MCP
独立文件 Server 可被多个 Host 复用，且无需开放网络端口。
但仍需限制目录和进程权限。

### 追问五：MCP 引入后事务怎么办
事务仍由业务服务负责。
MCP 不应把一个原子领域操作拆成大量低层调用交给模型组合。

### 追问六：如何渐进迁移
保留统一内部 ToolExecutor，先包装稳定只读能力，双通道验证结果、延迟和权限，再逐步迁移。

### 追问七：最终选型文档应包含什么
需求、候选方案、能力所有者、复用范围、成本、延迟、安全、故障模式、评测结果、迁移和回滚。

### 可背诵完整版
> Function Calling 和 MCP 不是同层替代。工具只服务一个 Host、同团队维护、紧贴进程内状态或业务事务时，直接 Function Calling 加内部执行器更简单。能力有独立所有者、要被多个 Host 复用、需要 Tools/Resources/Prompts、运行时发现、生命周期或标准本地与远程接入时，MCP 更有价值。生产系统常让模型通过 Function Calling 表达意图，再由 Host 通过 MCP 调外部能力。选型不能只看项目大小或工具数量，而要比较复用收益、延迟、协议适配、运维、安全和版本成本。无论哪种方案，校验、授权、确认、幂等、补偿和审计都必须保留。
