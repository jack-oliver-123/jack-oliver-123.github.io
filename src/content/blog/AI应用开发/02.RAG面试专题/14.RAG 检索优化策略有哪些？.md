## 本篇目标

学完这一篇，你应该能够：

1. 不再把“检索优化”理解成只调 Top-K 或盲目更换 Embedding 模型。
2. 按数据、表示、索引、查询、召回、排序和上下文七层定位问题。
3. 根据“正确证据掉在哪一层”选择对应策略，而不是一次改很多变量。
4. 建立离线评测、消融实验、影子流量、灰度发布和回滚闭环。
5. 同时优化相关性、延迟、成本、安全与时效，理解它们之间的取舍。

## 60 秒回答

> 我做 RAG 检索优化不会先罗列技巧，而是先建立带相关证据标注的评测集，沿链路判断正确证据掉在哪一层：知识库没有或解析错，是数据问题；文档存在但候选里没有，是切块、Embedding、索引、过滤或 Query 问题；候选里有但排序靠后，是融合或 Rerank 问题；排得靠前却没进 Prompt，是去重和上下文预算问题。
>
> 对应策略包括：数据层做解析校验、去重、版本和元数据治理；切块层按标题、段落和语义边界切分并保留父子关系；表示层比较领域 Embedding、标题增强和多向量表示；检索层结合 BM25 与向量、多 Query 和结构化过滤；排序层用 RRF、Reranker 和权威性/时效特征；上下文层去重、合并邻接片段并控制证据多样性。
>
> 每次尽量只改变一个主要变量，用 Recall@K 看召回覆盖，用 MRR/nDCG 看排序，用上下文命中率看是否进入 Prompt，再结合答案正确性、Faithfulness、P95、成本和无答案误召判断收益。离线通过后再影子、灰度和回滚，避免只让一个指标变漂亮。

## 详细解析

### 先说人话：先找“掉件环节”，再谈优化

用户问：“X2 在华东区超过 30 天还能换货吗？”

正确政策明明存在，系统却答错。可能是：

1. PDF 解析时表格错位，政策根本没有正确入库；
2. 切块把“适用条件”和“例外条款”分开了；
3. Query Rewrite 丢掉“超过 30 天”；
4. 向量路没有召回，但 BM25 本可找到；
5. 正确文档在候选中，却被融合算法排到后面；
6. Reranker 选中了过期版本；
7. 正确证据排得很靠前，但上下文预算被重复片段占满；
8. 证据进入 Prompt，模型仍然没有忠实使用。

这些问题的解决方式完全不同。先定位，才叫工程优化。

### 建立一张“检索漏斗”

```text
知识库中存在正确证据吗？
        |
        v
正确证据被解析、切块并成功索引吗？
        |
        v
它进入各路 Top-K 候选了吗？
        |
        v
融合/Rerank 后仍在前列吗？
        |
        v
它进入最终上下文了吗？
        |
        v
模型依据它给出正确答案了吗？
```

给每一层设置可观测指标，才能知道改动影响了哪里。

### 第 0 层：先把评测集建好

没有评测集，优化只是“看几个 Demo 感觉不错”。每条样本至少包含：

```json
{
  "query": "X2 在华东区超过 30 天还能换货吗？",
  "query_type": "policy_with_constraints",
  "relevant_document_ids": ["policy_x2_east_2026_06"],
  "required_evidence": [
    "超过30天的例外条件",
    "华东区生效版本"
  ],
  "answerable": true,
  "tenant": "demo_tenant"
}
```

评测集要覆盖：

- 常见和长尾问题；
- 精确编号与自然语言表达；
- 单跳与多跳；
- 时间、地区、否定和比较；
- 多版本与冲突来源；
- 权限边界；
- 知识库没有答案的问题；
- OCR、表格、代码和长文档等不同内容类型。

### 第 1 层：数据与解析优化

#### 常见故障

- PDF 多栏内容顺序错乱；
- 表格行列关系丢失；
- 页眉页脚反复进入正文；
- OCR 把 `E-1047` 识别成 `E-I047`；
- 网页菜单、广告和隐藏文本混入；
- 同一政策的草稿、正式版和缓存副本并存；
- 文档删除了，旧 chunk 仍可检索。

#### 优化策略

1. 按文档类型选择解析器，不用一个通用方案处理所有格式；
2. 保存原文件、解析结果和解析器版本，支持复现；
3. 对标题层级、表格、列表、代码块保留结构；
4. 使用内容哈希、来源 ID 和版本做去重；
5. 建立抽样质检和解析异常指标；
6. 为文档记录来源、权威性、生效时间、过期时间和 ACL；
7. 删除与权限撤销要能传播到索引、缓存和引用服务。

很多检索问题在这一层解决，比更换大模型更直接。

### 第 2 层：Chunking 优化

**Chunk（文本块）**是进入检索和上下文的基本单元。太小会丢上下文，太大会混入多个主题。

#### 常见策略

| 策略 | 优点 | 风险 |
| --- | --- | --- |
| 固定字符/token | 简单、吞吐稳定 | 可能切断语义和表格 |
| 按段落/标题 | 保留文档结构 | 段落长短差异大 |
| 语义切分 | 主题边界更自然 | 算法和阈值增加成本 |
| 滑动窗口重叠 | 缓解边界信息丢失 | 重复候选和索引膨胀 |
| 父子块 | 小块检索、大块阅读 | 版本和映射管理更复杂 |
| 问题/命题级切分 | 精准匹配事实 | 生成预处理可能引入错误 |

#### 选择原则

- FAQ：一问一答通常是自然单元；
- 政策：标题、条款、例外条件应尽量一起保留；
- API 文档：接口、参数、返回值和错误码需要结构关联；
- 表格：保留表头与行语义，不只截取单个单元格；
- 长报告：可用层级摘要和细节块组合。

不要只比较“chunk 长度”，还要比较边界完整率、重复率、索引体积和最终答案。

### 第 3 层：表示与 Embedding 优化

#### 可能的问题

- 模型对中文或领域术语能力不足；
- Query 和文档使用了不匹配的编码方式；
- 文档只有正文，没有标题和章节上下文；
- 文本超过模型输入上限被截断；
- 型号、数字和错误码在稠密表示中权重不足；
- 更换 Embedding 后旧向量没有全部重建。

#### 可验证的策略

1. 在目标语料上比较多个 Embedding，而不是只看公开榜单；
2. 遵循模型要求的 Query/Document 前缀或指令格式；
3. 把标题、章节路径和必要元数据加入表示，但正文与元数据分区；
4. 对精确标识保留 BM25 或字段检索；
5. 对长文档探索多向量表示或 Late Interaction；
6. 评估向量维度、量化和存储成本；
7. 模型切换使用新索引版本、双读和灰度。

更强的通用榜单模型，不代表在你的政策、代码或医学语料上更好。

### 第 4 层：索引与 ANN 优化

ANN 参数控制速度和召回的取舍：

- HNSW 的搜索宽度提高，召回可能上升，延迟也会上升；
- IVF 搜索更多桶，可能找到更多近邻，也增加计算；
- 量化减少内存和计算，但会带来表示误差；
- 分片提升容量，却可能增加跨分片合并和网络成本。

优化步骤：

1. 在一部分数据上建立精确检索参考；
2. 固定 Query、硬件和并发；
3. 扫描 ANN 参数，画出 Recall—Latency 曲线；
4. 加入真实 ACL 和元数据过滤；
5. 观察 P95/P99、内存、IOPS 和过滤后结果不足率；
6. 根据质量底线选择参数，不只选择最快点。

### 第 5 层：Query 优化

可能采用：

- 规则归一型号、日期和错误码；
- 结合授权会话做指代消解；
- 保留原 Query，同时生成适配 BM25 和 Dense 的表示；
- 对复合问题做受控分解；
- 只在验证有效的类型启用多 Query、HyDE 或 Step-back；
- 信息不足时向用户澄清。

防止 Query Rewrite 丢失实体、时间、数字、否定和地区约束。详细策略见上一题，本层在优化时重点看“改写是否提高有效召回且没有放大误召”。

### 第 6 层：召回与过滤优化

#### 混合检索

BM25 和 Dense 错误模式不同，组合后可提升覆盖。先用 RRF 建立基线，再按 Query 类型评估权重和路由。

#### 元数据过滤

过滤可以提高精度和安全性，但错误过滤会把正确证据直接排除。重点检查：

- 字段值是否规范一致；
- 生效时间逻辑是否正确；
- ACL 是预过滤还是后过滤；
- 过滤后 ANN 是否还能返回足够候选；
- 缺失元数据如何处理；
- 多租户缓存是否包含权限版本。

#### 候选深度

第一阶段 Top-K 太小，正确证据进不了 Rerank；太大则增加重排成本和噪声。应画出 K 增加时 Recall、延迟和答案质量的变化，而不是套固定值。

### 第 7 层：融合与 Rerank 优化

如果正确证据在候选中但名次低，检查：

- RRF 参数和每路候选深度；
- 某一路是否以大量噪声占据名额；
- Reranker 是否匹配语言和领域；
- 输入是否被截断；
- Reranker 是否只看正文而丢了标题；
- 过期版本是否因文本更相似而排在新版本前；
- 权威性、时效和来源是否需要作为排序特征。

Reranker 的评测应在“实际召回候选分布”上进行。用过于容易的随机负例，会高估效果。

### 第 8 层：上下文选择优化

相关证据已经排在前面，仍可能没进入 Prompt：

- 多个重复 chunk 占满 token；
- 同一文档垄断前 N；
- 只选了规则，漏了例外条款；
- 切块上下文不足；
- 重要证据处在长上下文中间，模型利用率下降；
- 引用元数据占用过多预算。

可采用：

- 精确去重和语义近重复去重；
- 合并相邻片段并恢复标题路径；
- 每个来源/文档设置上限；
- 按“覆盖问题中的必要事实”选择证据集合；
- 先压缩候选，但保留事实、数字、否定和引用映射；
- 根据模型上下文预算动态选择，而非固定拼 Top-K。

### 一张“症状—根因—行动”表

| 症状 | 优先检查 | 可能行动 |
| --- | --- | --- |
| 正确文档完全不存在 | 数据源、解析、同步 | 补数据、修解析、重建版本 |
| BM25 命中、Dense 不命中 | Embedding、标题、领域漂移 | 保留混合检索、换/微调表示 |
| Dense 命中、BM25 不命中 | 同义表达、分词 | 同义词、字段分析器、混合检索 |
| 无过滤命中、加过滤丢失 | 元数据、ACL、预/后过滤 | 修字段、索引与过滤策略 |
| 候选有、Rerank 后丢失 | 精排模型、输入截断、负例 | 换模型、加标题、重做训练/评测 |
| 排名前列却没进 Prompt | 去重、配额、token 预算 | 上下文选择与父块恢复 |
| 证据正确但答案错 | Prompt、模型、冲突处理 | 生成约束、引用校验、拒答 |
| 离线好、线上差 | 分布漂移、延迟、缓存、权限 | 线上分层监控和回放 |

## 工程实践与边界

### 指标分层

#### 召回指标

- Recall@K：相关证据是否进入前 K；
- Hit Rate@K：是否至少命中一个相关证据；
- 证据集合覆盖率：回答需要多条证据时，是否全部覆盖。

#### 排序指标

- MRR：第一个相关结果的位置；
- nDCG：考虑多级相关性和整个排序；
- Rerank 前后名次变化。

#### 上下文指标

- Context Recall：必要证据是否进入 Prompt；
- Context Precision：Prompt 中有多少内容真正有用；
- 重复率、来源多样性和冲突覆盖率。

#### 端到端与系统指标

- 答案正确性、Faithfulness、引用准确率；
- 有答案误拒答、无答案误回答；
- P50/P95/P99、吞吐、错误率；
- token、模型调用和单位请求成本；
- 权限违规、过期内容命中率。

### 一次只改一个主要变量

如果同时换 Chunking、Embedding、Query Rewrite 和 Reranker，即使结果变好，也无法知道哪个组件贡献最大、哪个组件带来风险。

建议：

1. 固定数据快照和评测集；
2. 记录基线；
3. 改一个主要变量；
4. 报告总体和分类型结果；
5. 运行多次并报告方差；
6. 保留失败样本；
7. 再进入下一项实验。

### 离线结果怎样上线

```text
离线回归 -> 影子检索 -> 小流量灰度 -> 扩大流量 -> 全量
                    |             |
                    +----异常回滚-+
```

- 影子阶段：新链路处理真实 Query，但不影响用户答案；
- 灰度阶段：按租户、查询类型或流量比例逐步切换；
- 监控质量代理、P95、错误率和成本；
- 索引、模型、Prompt 和配置都要有版本；
- 保留旧索引直到新版本通过观察窗口。

### 优化边界

- 提高 Recall 可能增加噪声和重排成本；
- 增加 Reranker 可能提高排序，也增加尾延迟；
- 更大 chunk 提供上下文，也可能稀释主题；
- 更多重叠缓解切断，也会增加重复和存储；
- 严格过滤提高安全与时效，但错误元数据会降低召回；
- 压缩上下文节省 token，也可能丢失否定、数字和例外。

优化目标应是满足业务质量、安全和 SLO 的整体方案，而不是单个离线分数。

## 常见误区

### 误区一：检索优化就是调 Top-K

Top-K 只是候选深度。问题可能出在数据、解析、切块、Query、索引、过滤、融合或 Rerank。

### 误区二：换一个榜单更强的 Embedding 就能提升

公开榜单与目标领域、语言、文档长度和 Query 分布可能不同。还要重建索引并评估成本。

### 误区三：Recall@K 越高，答案质量越高

高 Recall 只说明候选覆盖，噪声和上下文选择仍会影响模型。需要 MRR/nDCG、Context Precision 和端到端指标。

### 误区四：Reranker 分数可以当作答案置信度

Reranker 衡量 Query 与候选相关性，不直接证明文档真实、最新或足以支持答案。

### 误区五：线上点击率可以替代相关性标注

点击受展示位置、标题吸引力和用户习惯影响。可作为信号，但需要去偏和人工校准。

### 误区六：把全部相关文档塞进长上下文最稳

长上下文有成本、延迟和注意力分配问题，相关文档之间还可能冲突。应选择最小充分证据集合。

## 面试追问

### 1. 正确证据没进候选，第一步看什么？

先确认知识库中是否存在且已正确解析、切块和索引；再分别检查 ACL/时间过滤、Query 表示和各召回路，不能直接调 Reranker。

### 2. 正确证据在 Top-100 但不在 Top-10，怎么优化？

检查融合权重、Reranker、标题与正文表示、过期版本和困难负例；用 MRR/nDCG 观察排序，而不是继续扩大最终上下文。

### 3. Chunk 大小如何选择？

按文档结构和回答所需证据跨度设计候选方案，在固定评测集上比较 Recall、Context Precision、重复率、token 和答案质量。不存在跨领域通用值。

### 4. 怎样判断 Reranker 值不值得？

比较加入前后的 nDCG、上下文命中和最终答案质量，再看新增 P95、吞吐和成本；按 Query 类型路由可能比全量启用更经济。

### 5. 无答案问题为什么很重要？

只测有答案问题会鼓励系统“总能找点东西”。无答案集用于校准拒答、候选阈值和误召，降低看似相关但错误回答。

### 6. 如何处理过期文档排得更高？

用生效时间和版本先过滤；同一政策只保留有效版本；排序可加入来源权威性和时效特征；冲突时显式展示并按业务规则裁决。

### 7. 为什么离线提升，线上没有提升？

可能是评测集分布偏差、线上 Query 改写不同、权限过滤、缓存、延迟降级、真实文档更新或用户反馈指标有偏。需要基于 Trace 回放线上请求。

### 8. 如何避免评测数据泄漏？

按时间或文档版本切分；避免同一文档近重复块跨训练和测试；模型调参集与最终测试集分开；记录人工样本来源和变更。
