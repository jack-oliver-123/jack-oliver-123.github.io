## 本篇目标

读完这篇，你应该能够：

- 用自己的话解释 Multi-Agent，而不是只说“多个 Agent 一起工作”。
- 分清角色提示、独立 Agent 和真正的安全边界。
- 看懂中心编排、Handoff、并行 Worker 和共享任务空间四种协作方式。
- 判断哪些任务值得使用 Multi-Agent，哪些任务只会被它变得更复杂。
- 讲清任务、消息、状态、结果和失败在多个 Agent 之间如何传递。

## 60 秒回答

Multi-Agent 是多个具有明确职责边界的 Agent，通过约定的协作方式共同完成一个目标。不同 Agent 可以拥有不同的指令、上下文、状态和工具，也可以使用相同或不同的模型。重点不在数量，而在职责是否真的可以拆分，以及拆分后是否带来并行、专业上下文或权限隔离的净收益。

常见架构有四类：中心编排由 Orchestrator 分解任务、调度 Worker 并聚合结果；Handoff 由当前 Agent 把任务转交给更合适的 Agent；并行 Worker 同时处理彼此独立的子任务；共享任务空间让多个 Agent 通过队列或黑板发布和领取任务。

工程上必须定义任务契约、消息格式、状态所有权、完成条件、超时重试和冲突处理。仅换一个角色名或 system prompt 不构成安全隔离，真正的权限边界要靠独立身份、最小权限凭据和服务端授权。若一个 Agent 配合普通 Workflow 就能稳定完成任务，多 Agent 通常只会增加 Token、延迟、通信和故障成本。

## 详细解析

### 先说人话：从“一个全能员工”变成“一个小团队”

Single-Agent 像让一个人从头到尾完成工作：理解需求、查资料、做决定、写答案。

Multi-Agent 像一个小团队：有人负责接待，有人查订单，有人审核政策，还有人汇总结果。团队可能更专业，也可能并行工作；但它同时需要任务分配、交接、沟通和负责人。分工不清时，三个人反而可能重复做同一件事，甚至互相把任务推来推去。

因此，判断 Multi-Agent 的第一原则是：

> 不是“Agent 越多越强”，而是“拆分带来的收益是否大于协作成本”。

### 什么才算一个独立 Agent

工程上没有所有人都接受的唯一判定标准。通常可以从四个边界理解一个 Agent：

| 边界 | 含义 | 例子 |
| --- | --- | --- |
| 职责与指令 | 它负责什么、不负责什么 | 退款 Agent 只处理退款，不做营销推荐 |
| 可见上下文 | 它能看到哪些信息 | 酒店 Agent 只看住宿需求，不看支付卡号 |
| 工具与权限 | 它可以请求哪些动作 | 查询 Agent 只能读，支付 Agent 才能发起扣款 |
| 运行状态 | 它独立跟踪什么任务 | 每个 Worker 有自己的子任务状态和预算 |

只把同一个模型连续调用三次，并分别说“你是研究员”“你是写作者”“你是审核员”，可以看成一种多角色工作流，但是否值得称为 Multi-Agent 并不重要。面试时更应说明：这些角色有没有独立职责、状态和协作契约。

另外，多个 Agent 不等于多个不同的大模型。三个 Agent 可以使用同一个模型，只是指令、上下文和工具不同；一个 Agent 也可以在不同步骤调用不同模型。

### 贯穿案例：电商客服团队

假设目标是“解决用户关于订单 A123 的投诉”。系统里有：

- **接待 Agent**：理解诉求、收集订单号；
- **订单 Agent**：只读查询订单与物流；
- **退款 Agent**：检查退款规则并请求退款工具；
- **人工服务**：处理高金额、争议和异常情况。

一次正常协作可能是：

```text
用户投诉
  ↓
接待 Agent：确认是物流延误，并取得订单号
  ↓ Handoff
订单 Agent：查询物流，发现包裹已丢失
  ↓ 返回结构化结果与证据
退款 Agent：判断符合退款政策
  ↓ 请求审批与执行退款
接待 Agent：向用户说明结果
```

这个例子有三个关键点：

1. 订单 Agent 不必知道所有营销规则。
2. 接待 Agent 不能因为自己“认为应该退款”就获得退款权限。
3. Agent 之间传递的是任务、必要状态和证据，不是毫无筛选地复制完整聊天记录。

### Multi-Agent 的四种常见协作方式

#### 1. 中心编排：有一个项目经理

中心节点常叫 Orchestrator、Supervisor 或 Coordinator。它负责：

- 理解总目标；
- 把目标拆成子任务；
- 选择合适的 Worker；
- 处理依赖和并行关系；
- 检查每个结果；
- 聚合为最终产物；
- 判断整个任务何时结束。

```text
                 ┌→ 订单 Worker ─┐
用户 → Orchestrator              ├→ 聚合与验收 → 用户
                 ├→ 政策 Worker ─┤
                 └→ 风险 Worker ─┘
```

优点是状态所有权和审计路径清楚，适合大多数生产系统。缺点是 Orchestrator 可能成为瓶颈或单点，需要持久化状态和故障恢复。

#### 2. Handoff：把接力棒交给下一位

Handoff 是当前 Agent 判断另一个 Agent 更适合继续处理，于是把控制权和必要上下文转交出去。

例如接待 Agent 识别到“我要申请企业合同退款”，就转交企业退款 Agent。接收方此后直接面对任务，直到完成或再次转交。

Handoff 必须定义：

- 什么条件可以转交；
- 传递哪些字段；
- 接收方如何确认；
- 接收失败怎么办；
- 最多允许转交几次；
- 是否可以退回，以及退回给谁。

如果这些条件只写成一句模糊提示，两个 Agent 可能互相判断“这不是我的职责”，形成来回转交的死循环。

#### 3. 并行 Worker：几个人同时做独立工作

当子任务彼此独立时，可以并行执行。例如制定旅行方案时，同时查询航班、酒店和天气，最后再汇总。

并行只有在关键路径上真正节省等待时间时才有价值。如果酒店查询必须等航班日期确定，就不能假装并行。并行还会带来限流、资源峰值、结果乱序和部分失败问题。

#### 4. 共享任务空间：大家在同一块任务板上协作

多个 Agent 可以从任务队列或“黑板”读取待办、发布发现和领取新任务。这适合动态出现、彼此较松耦合的工作。

但系统必须处理：

- 两个 Agent 同时领取同一任务；
- 一个 Agent 崩溃后任务无人接管；
- 旧结果覆盖新结果；
- 大家都认为自己完成了，却没有人判断全局是否完成。

对初学者来说，中心编排通常更容易实现和调试。去中心化并不天然更智能，只是把协调问题分散了。

### Agent 之间到底传什么

最常见的错误是直接复制完整上下文。更稳妥的消息应像一个明确的工作单：

```json
{
  "task_id": "task_1001",
  "message_id": "msg_2001",
  "sender": "reception_agent",
  "recipient": "order_agent",
  "schema_version": "1.0",
  "goal": "查询订单 A123 当前物流状态",
  "inputs": {"order_id": "A123"},
  "constraints": {"read_only": true},
  "expected_output": "OrderStatusResult",
  "deadline": "2026-07-15T15:00:00+08:00",
  "correlation_id": "trace_3001"
}
```

接收方返回的也应是可校验的产物：

```json
{
  "task_id": "task_1001",
  "status": "completed",
  "result": {"shipping_status": "lost"},
  "evidence": ["logistics_event_789"],
  "produced_at": "2026-07-15T14:01:00+08:00"
}
```

自然语言适合解释，结构化字段适合校验和调度。二者可以并存。

### 谁拥有状态，谁判断完成

一个 Multi-Agent 系统至少有三种状态：

- 每个 Worker 的私有执行状态；
- 多个 Agent 共享的任务状态；
- 外部真实世界状态，例如订单是否真的退款。

中心编排模式下，最好由 Orchestrator 维护任务图和共享状态的唯一事实来源。Worker 返回结果，不直接随意修改其他 Worker 的状态。

完成也不能靠某个模型说“我觉得好了”。每个子任务要有验收条件，总任务也要有程序化条件，例如：所有必要节点成功、退款状态从权威系统确认、用户通知已投递，或者任务已转人工。

### Multi-Agent 什么时候有价值

#### 1. 真正可并行

多个耗时子任务互不依赖，并行后能缩短关键路径。例如同时检索多个独立资料源。

#### 2. 上下文可以稳定分域

不同职责需要完全不同的知识和指令，拆开后能减少无关信息干扰。例如法律条款审核与前端代码检查。

#### 3. 权限与数据边界可以真实隔离

高风险工具只交给少数 Agent，并且运行时确实使用不同服务身份、凭据和授权策略。注意：只在提示词里说“你没有权限”不算安全边界。

#### 4. 需要不同责任主体或人工参与

例如模型负责收集材料，人工审批员负责最终签字；或内部 Agent 与外部合作方 Agent 通过明确协议交互。

### 什么时候不应该用

- 任务只是步骤多，但步骤强依赖、必须串行；
- 所有角色需要几乎相同的上下文和工具；
- 一个 Workflow 加一个 Agent 已经稳定解决；
- 没有评测基线，只是觉得“多 Agent 更高级”；
- 团队尚未解决单 Agent 的状态、权限和可观测问题。

此时拆分只会让同一份信息被反复总结和传递，增加错误机会。

## 工程实践与边界

### 最小可落地设计

1. 先定义总任务的成功和失败条件。
2. 为每个 Agent 写清职责、输入、输出、可见数据和请求的工具。
3. 选择一个明确的状态所有者，初期优先中心编排。
4. 用 schema 校验 Agent 间消息，不只传自然语言。
5. 为超时、重试、取消、部分成功和转人工定义语义。
6. 限制总 Agent 数、消息轮数、并行扇出、Token 和费用。
7. 保留完整 trace，让一次任务可以按消息和工具调用回放。

### 五个典型失败场景

#### 场景一：相互 Handoff 死循环

A 说“应该找 B”，B 又说“应该找 A”。

处理：服务端限制转交次数；记录已访问角色；明确回退或人工接管节点；检测重复的 Agent 路径。

#### 场景二：并行重复写入

两个 Worker 都判断应该退款，并同时调用退款接口。

处理：Worker 只提出结构化请求，由统一提交节点执行；使用幂等键、锁或事务确保副作用只提交一次。

#### 场景三：结果互相冲突

政策 Agent 说可以退，风险 Agent 说必须人工审核。

处理：聚合器按权威来源和明确规则处理，无法决断时升级人工；不能简单“多数投票”，因为三个错误意见不会变成事实。

#### 场景四：上下文越传越失真

每次 Handoff 都重新摘要，最终丢失“退款不超过 100 元”的关键约束。

处理：硬约束使用结构化字段传递，原始证据保留引用；摘要只负责说明背景，不能成为唯一真相。

#### 场景五：Orchestrator 崩溃

Worker 已完成部分任务，但协调节点重启后不知进度，重复派发。

处理：持久化任务图、消息 ID 和 Worker 结果；消费者幂等；恢复时从检查点与外部状态核对。

### 协议能解决什么，不能解决什么

Agent2Agent（A2A）等协议可以规范 Agent 能力发现、消息、任务和产物的交换，适合跨系统互操作。但协议不会自动替你决定：

- 退款业务怎样拆分；
- 哪个来源更权威；
- 谁有权限看客户数据；
- 冲突时应当接受哪个结果；
- 整体任务何时算成功。

协议解决“怎么说话”，应用仍要解决“说什么、谁能说、说完怎样负责”。

### 如何评测 Multi-Agent

必须与 Single-Agent 或 Workflow 基线使用同一模型、工具和任务集比较：

- 端到端任务成功率；
- 关键路径延迟，而不是所有调用时间相加；
- 总模型调用数、Token 和费用；
- 转交错误率、重复任务率和冲突率；
- 工具越权或敏感数据暴露事件；
- 故障后能否恢复；
- 人工接管率和运维复杂度。

只有收益能够抵消新增成本，拆分才是合理架构，而不是演示效果。

## 常见误区

- **“多个模型就是 Multi-Agent”**：模型数量不是关键，职责、状态、工具和协作边界才是。
- **“复杂任务都要上 Multi-Agent”**：复杂任务也可以由 Workflow 或一个 Agent 完成。
- **“角色名不同就实现了安全隔离”**：安全隔离依靠独立身份、凭据、存储和服务端授权，不靠提示词。
- **“多 Agent 总会更快”**：只有关键路径上的独立任务可并行，且通信开销可控时才可能更快。
- **“多个 Agent 投票就会更正确”**：它们可能共享相同模型、数据和偏差，投票不能替代证据。
- **“每个 Agent 都应该知道全部信息”**：最小必要上下文更利于隐私、成本和稳定性。
- **“有了 A2A 协议就有了协作逻辑”**：协议提供互操作格式，业务语义、授权和责任仍需应用设计。

## 面试追问

> **面试官：** Multi-Agent 和调用多次大模型有什么区别？
>
> **候选人：** 多次调用只是实现手段。Multi-Agent 更强调独立职责、上下文或工具边界，以及任务分配、消息和完成协议；没有这些边界，通常只是普通多步骤工作流。

> **面试官：** 多 Agent 必须使用不同模型吗？
>
> **候选人：** 不必须。它们可以使用同一模型但拥有不同职责、上下文和工具；也可以按任务成本和能力选择不同模型。

> **面试官：** Agent 之间应该传什么？
>
> **候选人：** 传任务目标、结构化输入、约束、必要状态、验收标准、已验证结果和证据引用。不要复制完整对话、密钥或无关个人信息。

> **面试官：** 谁判断整体任务完成？
>
> **候选人：** 中心编排模式由 Orchestrator 根据任务图和程序化验收条件判断；去中心化模式也必须有明确完成协议，不能让各 Agent 自行猜测。

> **面试官：** 角色拆分能防止工具越权吗？
>
> **候选人：** 角色拆分本身不能。只有运行时使用可信身份、最小权限凭据，并由服务端校验每次工具请求时，才形成真实权限边界。

> **面试官：** Multi-Agent 的最小收益证据是什么？
>
> **候选人：** 在同一任务集上，相比单 Agent 基线，成功率、关键路径延迟、上下文稳定性或权限隔离有可测提升，而且足以抵消通信、Token 和运维成本。
