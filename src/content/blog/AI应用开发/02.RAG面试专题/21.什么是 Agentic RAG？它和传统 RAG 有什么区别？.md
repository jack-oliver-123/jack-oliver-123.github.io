## 本篇目标

学完这一篇，你应该能够：

- 用通俗语言区分传统 RAG、复杂工作流和 Agentic RAG；
- 解释 Agentic RAG 中的规划器、工具、状态、验证器和停止条件；
- 画出一次“判断—行动—观察—再判断”的执行循环；
- 判断哪些问题值得使用 Agentic RAG，哪些问题固定流程更合适；
- 说清它带来的质量收益、延迟成本、循环、权限和提示注入风险。

## 60 秒回答

传统 RAG 通常是工程师预先定义好的管线，例如 Query 改写、检索、重排、拼上下文和生成，每个请求大体走同样的步骤。Agentic RAG 则把一部分控制流交给模型：模型根据问题和中间证据决定是否检索、选择哪个数据源、如何拆分子问题、是否更换查询、是否验证以及何时停止。

它适合多跳、开放式、跨数据源或需要根据结果继续调查的问题。典型组件包括规划器、受权限约束的工具网关、结构化证据状态、检索与重排工具、验证器，以及最大步数、时间和 token 预算。模型只能在授权的工具和数据范围内决策，ACL、参数校验、审批和预算必须由确定性代码强制执行。

生产选型不能默认“越 Agent 越高级”。简单 FAQ 和稳定制度查询通常用固定 RAG 更便宜、可预测、好测试；复杂查询可以先走快速路径，证据不足时再升级到 Agentic 路径。评测除了最终答案，还要看工具选择、步骤成功率、无效循环、证据新增、延迟、安全和单位成功成本。

## 详细解析

### 先说人话：传统 RAG 像固定套餐，Agentic RAG 像会调查的办事员

传统 RAG 像在餐厅点固定套餐：

~~~text
下单 → 做前菜 → 做主菜 → 上菜
~~~

步骤由餐厅提前设计好，每位顾客大体走同一流程。

Agentic RAG 更像让一名办事员调查问题：

~~~text
先理解任务
→ 决定查哪个系统
→ 看结果
→ 发现信息不足
→ 再查另一份资料
→ 比较冲突
→ 证据足够后回答
~~~

关键不在“检索了很多次”，而在于下一步动作由当前问题和观察动态决定。

### 贯穿案例：回答一个多条件报销问题

用户问：

> 我是上海研发中心的普通员工，下周去深圳参加客户会议 12 天，中间有一个周末。酒店每晚能报多少，周末住宿也能报吗？

要完整回答，可能需要：

1. 查城市住宿标准；
2. 查员工职级规则；
3. 查连续出差和周末规则；
4. 确认“客户会议”是否属于业务出差；
5. 处理新旧制度版本；
6. 汇总并给出来源。

固定 RAG 可以一次检索多个关键词，但不一定知道缺了哪份证据。Agentic RAG 可以先拆分子问题，每得到一份证据就更新状态，再判断还缺什么。

### 先把 Agent、Workflow 和 Tool 说清楚

| 概念 | 通俗解释 | 例子 |
| --- | --- | --- |
| Tool | 系统可调用的一个受控能力 | 搜制度、查员工属性、查日历 |
| Workflow | 工程师写好的固定或条件流程 | 每次都执行改写、检索、重排 |
| Agent | 模型根据目标和观察动态选择下一步 | 先查身份，再决定查哪套制度 |
| Agentic RAG | Agent 的可选动作主要围绕获取、验证和组织外部证据 | 多步检索后形成有引用答案 |

“用了大模型”不等于 Agent；“调用了两次检索”也不等于 Agentic RAG。
判断重点是：谁拥有下一步控制权？如果步骤和分支全部由代码预先写死，仍更接近工作流。

### 传统 RAG 与 Agentic RAG 的核心区别

| 维度 | 传统 RAG | Agentic RAG |
| --- | --- | --- |
| 控制流 | 工程师预先定义 | 模型在受限范围内动态决定 |
| 检索次数 | 通常固定 | 可根据证据继续或停止 |
| 数据源 | 通常预先确定 | 可动态选择授权工具 |
| 子问题 | 常一次性改写 | 可拆分、排序和逐步处理 |
| 状态 | 主要是查询和上下文 | 保存计划、证据、缺口和冲突 |
| 可预测性 | 较高 | 较低，需要硬边界 |
| 延迟与成本 | 容易估算 | 随步骤数变化 |
| 调试 | 链路相对固定 | 还要评决策和状态转移 |
| 适合问题 | FAQ、单跳、结构稳定 | 多跳、开放调查、跨系统 |

### Agentic RAG 的基本循环

~~~text
目标
  ↓
观察当前问题和证据状态
  ↓
判断下一步：检索、改写、换数据源、验证或停止
  ↓
通过工具网关执行
  ↓
把新证据写入状态
  ↓
检查证据是否充分、是否冲突、预算是否耗尽
  ├─ 不充分：进入下一轮
  └─ 充分或到达上限：生成、引用或转人工
~~~

这与 ReAct 中“推理与行动交替”的思想有关，但生产系统不需要暴露模型的私有思维链。应该记录可观察的决策摘要、工具参数、结果、状态变化和停止原因。

### 五个核心组件

#### 1. 规划器（Planner）

规划器负责：

- 识别任务目标；
- 拆分子问题；
- 判断依赖顺序；
- 给每个子问题选择候选数据源；
- 根据新证据修改计划。

例如：

~~~text
子问题 A：用户属于哪种员工类型？
子问题 B：深圳住宿标准是多少？
子问题 C：周末住宿是否属于可报销区间？
子问题 D：规则在出差日期是否有效？
~~~

计划不是越长越好。过度拆分会增加调用、错误和延迟。

#### 2. 工具网关（Tool Gateway）

工具可以包括：

- 向量或关键词检索；
- SQL 查询；
- 图查询；
- 企业搜索；
- 文档读取；
- 业务 API；
- 可信网页搜索。

模型只产生“请求调用”的结构化参数，真正执行由应用负责。网关要检查：

- 当前用户是否有权调用；
- 参数是否符合 Schema；
- 数据源是否在 allowlist；
- 超时、返回大小和速率；
- 是否包含敏感写操作；
- 结果中的内容是否需要隔离。

模型不能自己扩大工具权限。

#### 3. 证据状态（Evidence State）

不要只把所有内容堆进对话历史。更清晰的状态可以是：

| 字段 | 示例 |
| --- | --- |
| 已解决子问题 | 员工类型、城市标准 |
| 未解决子问题 | 周末规则 |
| 证据 | 文档 ID、页码、片段、版本 |
| 冲突 | 2023 版和 2025 版金额不同 |
| 来源质量 | 正式制度高于会议纪要 |
| 查询历史 | 已搜索过哪些规范化查询 |
| 预算 | 剩余步骤、token 和时间 |

结构化状态便于去重、验证、调试和恢复。

#### 4. 验证器（Verifier）

验证器检查：

- 每个子问题是否有证据；
- 关键结论是否被来源支持；
- 是否只使用当前有效版本；
- 不同来源是否冲突；
- 引用是否可访问且定位正确；
- 是否应该拒答或转人工。

验证器可以由规则、检索分数、模型判断和业务数据对账共同组成。高风险条件优先使用确定性规则。

#### 5. 停止器（Stop Controller）

“模型觉得够了”不能成为唯一停止条件。还要有硬上限：

- 最大步骤数；
- 最大工具调用数；
- 最大总 token；
- 最大墙钟时间；
- 单次和总费用预算；
- 连续若干轮没有新增证据时停止；
- 重复查询或重复结果时停止；
- 权限、工具或安全策略阻断时停止。

停止结果可以是回答、明确说明证据不足、请求用户补充信息或转人工。

### 常见的 Agentic RAG 模式

#### 模式一：动态路由

模型先判断问题属于哪类：

~~~text
制度问题 → 企业知识库
订单问题 → 业务数据库
公开时效问题 → 受控网页搜索
跨部门关系问题 → 图检索
~~~

它比固定搜索所有数据源节省成本，但路由错误会直接漏证据，所以要评路由准确率，并为高风险问题提供回退。

#### 模式二：查询规划与多跳检索

把复杂问题拆成有依赖的子问题。后一步查询可能使用前一步结果，例如先确定员工等级，再检索对应政策。

#### 模式三：迭代检索

第一次结果太宽泛，系统提取关键实体或条件，生成更精确查询再次检索。
重点是跟踪新增证据，避免只是换同义词反复搜索。

#### 模式四：纠错式检索

系统判断已检索文档相关性不足时，改写查询、扩大或更换数据源，再对证据做过滤。CRAG 研究了检索质量评估和纠错思路。

#### 模式五：检索与自我检查

模型在生成过程中判断哪些陈述需要检索、证据是否支持输出。Self-RAG 研究了带反思 token 的训练方法。
它可以为 Agentic 设计提供启发，但 Self-RAG 是具体研究方法，不能与所有 Agentic RAG 实现直接画等号。

#### 模式六：多代理检索

不同 Agent 分别负责计划、搜索、验证或特定数据源。它只在任务可并行、角色边界清楚且收益经评测验证时值得考虑。多 Agent 会增加通信、状态一致性和故障面。

### 同一个问题，两种系统怎么执行

#### 固定 RAG

~~~text
改写完整问题
→ 混合检索 top 30
→ ACL 过滤
→ 重排 top 6
→ 生成答案
~~~

优点是步骤少、延迟稳定、易回放。
如果检索和重排已经能覆盖所需三份制度，这可能就是更合适的方案。

#### Agentic RAG

~~~text
读取问题
→ 查用户身份
→ 查深圳住宿标准
→ 观察到还缺周末规则
→ 查连续出差与周末条款
→ 检查规则有效期
→ 发现证据齐全
→ 生成并逐条引用
~~~

它对复杂问题更灵活，但每一步都可能选错工具、使用错误参数、超时或被恶意内容影响。

### 什么时候值得使用

更适合：

- 问题需要多跳证据；
- 不同问题应访问不同数据源；
- 后续查询依赖前一步结果；
- 需要比较冲突来源；
- 需要根据证据不足动态补查；
- 调查型任务允许较高延迟；
- 有完整 Trace、评测和安全网关。

不一定适合：

- 简单 FAQ；
- 单一数据源且一次检索效果稳定；
- 延迟和成本预算很紧；
- 流程受法规严格约束，需要确定性步骤；
- 数据权限和工具边界尚未建立；
- 团队连基础 RAG 的失败原因都无法观测。

### 一个实用的分层路由方案

~~~text
请求进入
  ↓
规则与轻量分类判断复杂度
  ├─ 简单：固定 RAG 快速路径
  ├─ 中等：固定多阶段检索与验证
  └─ 复杂：Agentic RAG，带严格预算
~~~

这样避免每个“报销标准是多少”都经历多轮规划。

## 工程实践与边界

### 权限由代码强制，不由 Agent 决定

Agent 可以选择“搜索制度库”，但不能自己决定用户是否有权查看财务文档。每次工具调用都要由服务端：

1. 从已认证会话获取用户和租户；
2. 计算当前权限；
3. 注入不可由模型覆盖的 ACL 条件；
4. 对返回结果再次校验；
5. 记录审计信息。

无权内容不能先进入模型再过滤答案。

### 防止循环和无效搜索

可以组合：

- 规范化查询去重；
- 文档 ID 和证据哈希去重；
- 记录每轮新增证据数；
- 连续无新增时停止；
- 限制同一工具连续调用；
- 设置步骤、时间、token 和成本上限；
- 检测状态是否反复出现；
- 必要时转固定回退流程。

### 处理工具失败

| 故障 | 推荐行为 |
| --- | --- |
| 检索超时 | 有界重试、换健康副本或安全降级 |
| 返回空结果 | 调整一次查询或换授权数据源 |
| 参数错误 | 根据 Schema 修正，限制重试次数 |
| 来源冲突 | 保留冲突，按权威性和时间处理 |
| 权限拒绝 | 不绕过，说明无法访问或请求授权 |
| 预算耗尽 | 用已有证据给条件性回答或转人工 |
| 外部内容含指令 | 当数据处理，不提升其权限 |

### 评测不能只看最终答案

| 维度 | 示例指标 |
| --- | --- |
| 最终质量 | 正确性、忠实度、引用覆盖、任务成功 |
| 规划质量 | 子问题覆盖、依赖顺序、冗余步骤 |
| 工具质量 | 工具选择正确率、参数有效率 |
| 检索质量 | 各步证据召回、边际新增证据 |
| 停止质量 | 过早停止、无效循环、预算耗尽率 |
| 效率 | 步数、工具调用、P95 延迟、token、成本 |
| 安全 | 越权、注入成功、敏感工具误用 |

要与强固定流程做消融对比。如果 Agentic 路径只增加了调用，没有显著提升复杂问题成功率，就没有证明其生产价值。

### 可观测性至少保存什么

- 请求、用户权限域和风险等级；
- 每次状态转移；
- 工具名称、结构化参数摘要和结果哈希；
- 证据来源、版本、页码和 ACL；
- 每一步延迟、token 和费用；
- 停止原因；
- 最终答案和引用；
- 模型、Prompt、工具 Schema 与索引版本。

敏感数据应脱敏，工具密钥不能进入上下文或日志。

### Agentic RAG 的安全边界

- 检索内容和网页都是不可信数据；
- 文档中的指令不能修改系统策略；
- 工具采用最小权限和 allowlist；
- 有副作用的操作与“查资料”分离；
- 付款、发信、删除等动作需要额外审批；
- Agent 不能因为检索到一段文字就获得新权限；
- 超时和降级不能绕过 ACL；
- 高风险回答仍需要引用、规则或人工审核。

### 成本预算

可以为每类请求设置：

~~~text
最大规划调用：2
最大检索调用：6
最大验证调用：2
最大总步骤：10
最大墙钟时间：20 秒
最大 token / 费用：按业务预算
~~~

这些数字只是结构示例，实际值要通过压测和业务 SLO 确定。

### 方法边界

- “Agentic RAG”没有唯一、固定的行业实现标准；
- 模型的自我评价可能过度自信，不能替代外部验证；
- 动态控制流增加了测试组合数量；
- 多步调用让延迟和成本方差更大；
- Agentic RAG 仍不能消除幻觉；
- 基础数据质量差时，增加代理步骤通常只是更复杂地使用错误数据。

## 常见误区

- **“Agentic RAG 就是多检索几次”**：关键是根据观察动态选择下一步和停止。
- **“用了 ReAct 提示词就是生产 Agent”**：还需要状态、工具网关、权限、预算、评测和故障处理。
- **“模型可以自己判断权限”**：ACL 必须由服务端和数据层强制执行。
- **“模型会在证据足够时自然停止”**：可能过早停止或循环，需要硬上限。
- **“复杂问题都应该用 Agent”**：固定多阶段流程通常更可预测、更便宜。
- **“反思一次就能验证答案”**：自我评价也会出错，需要规则、证据和独立检查。
- **“工具越多能力越强”**：工具描述相似会增加误选，权限面和测试成本也会扩大。
- **“多 Agent 总比单 Agent 好”**：它还会增加通信、协调、状态和费用。

## 面试追问

### 1. Agentic RAG 与普通多路召回有什么区别？

多路召回通常是工程师预先规定同时查关键词、向量等通道；Agentic RAG 会根据问题与中间证据动态决定查哪些源、是否继续和如何改写。前者是固定编排，后者包含受约束的动态控制。

### 2. 怎样判断证据已经足够？

检查所有必要子问题是否有来源支持、关键结论是否有引用、是否存在未处理冲突，再结合检索分数和模型判断。无论评分如何，还要受最大步骤、时间和预算约束。

### 3. 如何防止重复检索？

保存规范化查询、工具参数、返回文档 ID 和证据哈希；相同状态不重复执行；跟踪每轮边际新增证据，连续没有新增时停止或换策略。

### 4. 为什么简单 FAQ 不适合默认走 Agent？

固定 RAG 已能稳定回答时，动态规划只会增加延迟、费用和故障路径。可以先走快速路径，检测证据不足或问题复杂时再升级。

### 5. Agentic RAG 怎么评测？

同时评最终任务、规划、工具、每步检索、停止、效率和安全。尤其要与固定流程对比复杂问题成功率、P95 延迟、循环率和单位成功成本。

### 6. 如何回放一次 Agentic RAG？

保存模型与 Prompt 版本、状态转移、工具 Schema、结构化参数、结果哈希、证据版本和停止原因。外部系统易变化时，用固定快照或模拟器复现。

### 7. 工具返回恶意指令怎么办？

把返回值标记为不可信数据，与系统指令分隔；模型不能因此扩大权限；后续工具调用仍由网关按当前用户、allowlist 和参数规则重新授权。

### 8. Self-RAG、CRAG 和 Agentic RAG 是什么关系？

Self-RAG 和 CRAG 是具体研究方法，分别强调检索/生成中的自我反思以及检索质量判断与纠错。它们的思想可用于 Agentic RAG，但 Agentic RAG 是更宽泛的系统设计概念，不能把三者当同义词。
