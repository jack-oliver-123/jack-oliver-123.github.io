## 本篇目标

读完本篇，你应该能够：

- 识别句子截断、限定条件丢失和跨章节依赖等不同语义断裂；
- 理解为什么单纯增加 Chunk 长度或重叠不能解决所有问题；
- 从解析、切分、索引、取回和生成五个阶段设计补救方案；
- 解释父子检索、句子窗口和查询分解的适用范围；
- 建立专门验证边界问题的评测集。

## 60 秒回答

规避语义被切断不能只靠一个固定 overlap，要同时处理切分和取回。切分前先恢复 PDF、标题、段落、列表与表格结构；切分时优先沿自然边界，块过长再递归切分，并为 Chunk 补充标题路径和必要的局部上下文。

检索时可以用小块做精确匹配，命中后取回父章节、句子窗口或相邻片段，再经过权限和版本过滤、去重、重排与 Token 预算控制。对于定义和例外相距很远、需要多份文档组合的问题，还要使用查询分解、多跳检索或结构化关系，单纯扩大 Chunk 无法稳定解决。

我会专门构造跨句、跨段、跨章节和表格边界样本，比较正确证据召回、最小充分证据完整率、重复率、上下文 Token、答案忠实度和延迟，依据结果选择策略。

## 详细解析

### 先说人话：把“规则”和“但是”分开会发生什么

假设制度原文是：

> 普通员工上海住宿标准为每晚某金额。
>
> 但是，每年展会期间需要提前审批，未审批时仍按普通标准报销。

如果系统恰好在“但是”前面切开：

- Chunk A 只有基础金额；
- Chunk B 只有展会例外和审批条件。

用户问“展会期间能不能按上浮标准报销”，检索器可能只召回 Chunk A。大模型看到的证据并不完整，即使完全忠实地回答，也可能给出错误结论。

这就是语义断裂：组成完整含义的信息被切到了不同片段，而检索或上下文组装没有把它们重新组合回来。

### 先分清五类语义断裂

#### 第一类：硬截断

句子、数字、URL、代码或表格行在中间被切开。

例子：

> “普通员工每晚住宿标准不得超过”
>
> “600 元。”

这种问题通常可以通过句子边界、表格结构和递归切分缓解。

#### 第二类：局部依赖断裂

句子完整，但主语、定义、否定或限定条件在前后句。

例子：

- “该标准”指代上一段的上海住宿标准；
- “以下情况除外”出现在下一段；
- 表格上方的单位写着“元/人/天”；
- 章节标题包含“仅适用于销售部门”。

只保存命中句会丢失含义。

#### 第三类：父级语境丢失

正文片段本身合理，但缺少标题路径。

例如正文只有“住宿标准为某金额”，却不知道它属于：

- 国内还是海外；
- 普通员工还是经理；
- 2024 年旧版还是 2026 年新版。

这类问题适合把标题、章节路径和关键元数据加入检索表示。

#### 第四类：远距离依赖

定义在第 2 章，例外在第 8 章，或者一个问题需要组合两份文档。

增加几十个 Token 重叠无法覆盖跨章节距离，需要多跳检索、查询分解或关系索引。

#### 第五类：解析造成的伪断裂

原始文档没有断裂，但解析器把顺序弄错：

- 双栏 PDF 左右交叉；
- 页眉插入正文；
- 表格按错误顺序展开；
- OCR 把数字和单位识别错；
- 列表层级消失。

这类错误必须先修复解析。后续语义切块无法从错误文本恢复原版面含义。

### 策略一：切分前先恢复结构

处理顺序应该是：

> 文档解析 -> 结构恢复 -> 清洗 -> Chunking

应尽量识别：

- 标题层级；
- 段落和句子；
- 列表项及其父项；
- 表格标题、表头和脚注；
- 图片说明；
- 页码、坐标和阅读顺序。

对于低质量 OCR，可以记录置信度，把异常文件送入人工复核，而不是悄悄进入知识库。

### 策略二：优先沿自然边界切分

常见优先级可以是：

> 章节 -> 小节 -> 段落 -> 句子 -> Token

先按较大结构分组，超过目标上限时再递归向下切。这样比直接按字符位置截断更容易保留语义。

自然边界也不是完美答案。一个长段落仍可能包含多个主题，一个短条款也可能引用远处定义，所以还需要其他策略。

### 策略三：加入受控重叠

重叠窗口让相邻 Chunk 共享少量句子或 Token。

它适合解决：

- 句子刚好跨过边界；
- 定义和解释相邻；
- 局部上下文需要前后几句。

它不适合解决：

- 定义和例外相隔数页；
- 问题需要组合多个章节；
- 表格结构已经解析错误；
- 权限不同的内容被错误合并。

重叠的副作用包括：

- 存储和 Embedding 成本增加；
- 同一证据重复召回；
- Top-K 被近似重复片段占满；
- 生成上下文浪费；
- 引用去重更复杂。

因此要在结果合并阶段按 document_id 和原始偏移去重。

### 策略四：为小块补充标题与上下文

原始 Chunk：

> “每晚标准为某金额，旺季需提前审批。”

补充标题后的检索文本：

> “2026 年差旅制度 > 国内差旅 > 上海 > 普通员工住宿：每晚标准为某金额，旺季需提前审批。”

标题可以让查询和 Chunk 更容易匹配，也减少父级语境丢失。

可以使用：

- 原始标题路径；
- 文档名和版本；
- 人工规则生成的上下文；
- 模型生成的简短情境说明。

如果使用模型生成上下文，要保留原文并评测生成错误。模型摘要只能作为检索表示，不能成为未经核验的权威证据。

### 策略五：小块检索，父块生成

父子检索把粒度拆成两层：

| 层级 | 作用 |
| --- | --- |
| 子 Chunk | 内容集中，用于 Embedding 和命中 |
| 父 Chunk | 包含标题、上下文和例外，用于生成 |

流程是：

1. 用户查询匹配到子 Chunk；
2. 系统根据 parent_id 找到父章节；
3. 对父章节重新执行权限、版本和有效期检查；
4. 按 Token 预算取回完整父块或必要部分；
5. 与其他证据去重后送给模型。

父块不是越大越好。整章可能包含多个无关主题，仍需裁剪。

### 策略六：句子窗口与邻居扩展

如果命中的是一个句子，系统可以取回：

- 前一句；
- 命中句；
- 后一句；
- 所属标题。

也可以根据原始偏移扩展前后相邻 Chunk。

扩展前后要检查：

- 是否属于同一文档版本；
- 是否有相同 ACL；
- 是否跨入另一个章节；
- 是否已经失效；
- 是否和其他结果重复；
- 是否超过上下文预算。

邻居扩展是一次新的数据访问，不能跳过权限校验。

### 策略七：让索引同时保存多种表示

同一内容可以保存：

- 子 Chunk 向量；
- 标题加正文向量；
- 父章节摘要向量；
- 关键词索引；
- 从原文生成的可能问题。

查询可以从多个表示命中，然后统一映射回原文。

这种方法适合查询表达与原文表达差异很大的场景，但会增加索引和调试复杂度。需要记录“命中的是哪种表示，最终返回的是哪段原文”。

### 策略八：对远距离问题做查询分解

用户问：

> “上海旺季出差，普通员工住宿上浮后是否仍低于经理的普通标准？”

这可能需要：

1. 查询普通员工基础标准；
2. 查询旺季上浮规则；
3. 查询经理基础标准；
4. 由确定性程序或模型比较。

一个大 Chunk 很难覆盖所有事实。系统可以把问题分解为多个子查询，分别检索后合并证据。

分解也可能出错：

- 漏掉子问题；
- 改写了原意；
- 合并了不同版本；
- 将不同适用对象的数据放在一起。

所以应保留原问题、子查询和证据关系，并对最终组合做一致性校验。

### 策略九：对关系明确的数据使用结构化检索

如果制度已经结构化成：

> 城市 -> 职级 -> 季节 -> 标准 -> 审批条件

直接用数据库查询可能比从自然语言 Chunk 中猜关系更可靠。

图数据库也适合表达实体与多跳关系，但只有当关系检索收益能覆盖建图、消歧和维护成本时才值得使用。

RAG 不要求所有知识都变成向量。关键词、SQL、图和向量可以组合。

### 策略十：上下文组装时恢复完整证据

召回后还要：

- 合并相邻且连续的片段；
- 去掉重叠重复；
- 保留标题和来源；
- 把定义与例外放在一起；
- 检测版本冲突；
- 按证据价值裁剪；
- 标明不同证据之间的关系。

如果只按相似度分数把 Top-K 依次拼接，可能把同一句话放入五次，却漏掉关键例外。

### 怎样建立边界评测集

普通随机查询很难暴露语义断裂。应专门收集：

| 样本类型 | 示例 |
| --- | --- |
| 跨句 | 金额和单位分属两句 |
| 跨段 | 规则在上一段，例外在下一段 |
| 跨标题 | 正文依赖父级适用范围 |
| 跨页 | 表格延续到下一页 |
| 跨章节 | 定义和例外相距较远 |
| 跨文档 | 总则与部门补充规定组合 |
| 表格边界 | 表头和数据行分离 |
| 否定与条件 | “不适用”“仅限”“除外”等信息 |

每个问题标注“最小充分证据集”：能够完整支持答案所需的最少片段集合。

然后比较：

- 至少一个相关片段是否召回；
- 最小充分证据是否全部召回；
- 扩展后是否带入过多无关内容；
- 重复率；
- 上下文 Token；
- 答案是否忠实使用限定条件；
- 延迟和成本。

### 一个排查流程

遇到疑似语义断裂时，依次检查：

1. 原始文档是否正确；
2. 解析后的阅读顺序是否正确；
3. Chunk 边界和标题是否合理；
4. 正确子块是否进入候选；
5. 父块或邻居是否成功取回；
6. 权限和版本过滤是否误删证据；
7. 去重与 Token 裁剪是否删掉条件；
8. 最终上下文是否包含充分证据；
9. 模型是否正确理解证据。

这样才能区分是切块问题还是后续组装问题。

## 工程实践与边界

### 保存位置关系

每个 Chunk 建议保存：

- document_id；
- section_path；
- parent_id；
- 前后邻居 ID；
- 字符或 Token 偏移；
- 页码或版面坐标；
- 文档版本。

没有位置关系，命中后就很难稳定扩展上下文。

### 扩展后重新过滤

父块、邻居和跨文档证据都要重新检查 ACL、租户、有效期和数据分类。不能假设子块有权限，所有关联内容就都有权限。

### 控制上下文预算

扩展后的文本需要重新计算 Token。可以按以下优先级裁剪：

1. 直接回答问题的证据；
2. 必需的定义与限定条件；
3. 冲突版本说明；
4. 辅助背景；
5. 重复或低相关内容。

### 监控重复与膨胀

建议观察：

- 每份文档平均 Chunk 数；
- 重叠带来的内容重复率；
- Top-K 中同文档近邻占比；
- 扩展前后 Token 增幅；
- 扩展前后答案质量差异；
- 父块读取失败率。

### 上线时做消融实验

分别关闭标题补全、重叠、父子扩展和重排，观察哪一项真正改善边界样本。否则多个策略叠加后，无法知道收益来自哪里。

## 常见误区

- **“把 overlap 调大就能解决语义断裂”**：重叠主要覆盖局部边界，不能解决跨章节和跨文档依赖。
- **“Chunk 越大，语义越完整”**：大块会混入更多主题，降低检索区分度并占用上下文。
- **“只要按句子切就不会断”**：句子可能依赖标题、主语、定义和例外。
- **“语义切块可以理解所有文档”**：它依赖表示模型和阈值，也无法修复错误的 PDF 阅读顺序。
- **“召回命中句就够了”**：完整证据可能包含单位、条件、否定和适用范围。
- **“父子检索就是返回整篇文档”**：父级仍需按问题和预算裁剪。
- **“邻居扩展不需要再鉴权”**：扩展内容可能有不同权限、版本或数据分类。
- **“平均 Recall 很高就没有边界问题”**：少数高风险边界样本可能被平均值掩盖。

## 面试追问

1. **怎样判断问题真的是 Chunking，而不是 Embedding？**

   先看原文和 Chunk。若完整答案被拆散且任何单块都不充分，是切块或取回问题；若存在完整相关 Chunk 但排名很低，再检查 Embedding、查询表达和索引。

2. **父子切块怎样更新？**

   父文档版本变化时，重建受影响的父子关系和向量，写入新版本后原子切换；不能让新版子块指向旧版父块。

3. **怎样选择句子窗口大小？**

   在跨边界评测集上比较不同窗口，观察充分证据完整率、无关 Token 和延迟。不同文档类型可以使用不同配置。

4. **标题补全会不会污染 Embedding？**

   标题有助于补语境，但重复的宽泛标题可能降低区分度。需要对原文向量与标题加正文向量做对照，并保留原文用于生成。

5. **多跳检索最大的风险是什么？**

   子查询遗漏或错误会传播，多个来源还可能属于不同时间和权限范围。应保存检索轨迹并验证实体、版本和条件一致性。

6. **为什么要标注最小充分证据集？**

   只标一个“相关文档”无法判断条件是否完整。最小充分证据可以评测系统是否同时找回结论、适用范围和例外。
