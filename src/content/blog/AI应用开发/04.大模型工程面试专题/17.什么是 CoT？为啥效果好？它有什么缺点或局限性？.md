问一个人：

> 一本书 24 元，买 3 本再优惠 12 元，需要多少钱？

他如果直接报“60 元”，你不知道他是算出来的还是猜的。

如果他先写：

~~~text
3 本原价：24 × 3 = 72
减去优惠：72 - 12 = 60
答案：60 元
~~~

复杂任务被拆成了更小步骤。

Chain-of-Thought，简称 CoT，通常指模型在给出最终答案前，生成一段中间推理文本。

它在许多算术、常识和符号推理任务上能改善表现，但“写出来的理由”不应被直接当成模型真实、完整和可信的内部因果过程。

## 本篇目标

读完本篇，你应该能够：

- 解释 CoT、rationale、scratchpad 和 final answer；
- 区分 few-shot CoT 与 zero-shot CoT；
- 说明 CoT 可能有效的几个机制；
- 解释自洽采样为什么不是简单重复询问；
- 区分结果监督与过程监督；
- 说明 CoT 的非忠实性、错误传播、成本和安全风险；
- 区分普通聊天模型与推理模型的提示方式；
- 设计“简洁答案、可验证证据、内部推理”之间的产品边界；
- 回答“CoT 是否等于真实思考”等面试追问。

核心结论：

> CoT 是一种可见的中间推理文本或推理策略，不是模型内部计算的完整读出；它可能提高复杂任务正确率，也可能给错误答案编出流畅理由。

## 60 秒回答

CoT 是让模型在最终答案前生成中间推理步骤的方法。Few-shot CoT 在示例中同时给问题、推理和答案；Zero-shot CoT 用类似“逐步分析”的指令诱导模型展开推理。它可能有效，是因为复杂映射被分解成多个较短的条件生成步骤，中间 Token 充当额外计算状态，也让模型有机会复用训练中见过的推理模式。

进一步可以对同一问题采样多条推理路径，再对最终答案投票，这叫 self-consistency。它利用不同路径的共识提高稳健性，但成本随样本数增长，路径之间也未必独立。

CoT 的局限包括：中间步骤会出错并传播；文字解释可能不忠实于真正影响答案的特征；长推理增加 Token、延迟和攻击面；公开详细推理还可能泄露提示、敏感信息或安全策略。工程上不能把 CoT 当事实证明，应优先使用可验证工具、计算器、代码执行、引用和测试。对于专门的推理模型，要遵循对应官方提示建议，有些模型更适合只给清晰目标和约束，而不是强行规定“逐步思考”。

## 详细解析

### 1. CoT 的基本定义

一条普通问答数据：

~~~text
问题 x → 答案 y
~~~

加入 CoT 后：

~~~text
问题 x → 中间推理 r → 最终答案 y
~~~

可写成：

$$
p(r,y\mid x)
=
\prod_{t=1}^{|r,y|}
p(z_t\mid x,z_{<t})
$$

其中：

- $x$ 是问题；
- $r$ 是 reasoning/rationale；
- $y$ 是 final answer；
- $z$ 表示把推理与答案连接后的 Token 序列；
- 模型仍然只是逐 Token 预测。

CoT 没有增加新的网络层。

它改变了生成轨迹和可用的中间上下文。

### 2. 几个容易混淆的词

| 术语 | 常见含义 |
| --- | --- |
| Chain-of-Thought | 逐步中间推理文本 |
| Rationale | 支持答案的解释或理由 |
| Scratchpad | 用于中间计算的工作区文本或状态 |
| Final answer | 面向用户或评测器的最终结论 |
| Reasoning trace | 一段推理轨迹，可能可见也可能隐藏 |
| Process supervision | 对中间步骤提供监督或评价 |
| Outcome supervision | 只评价最终结果 |

这些词在不同论文和产品中语义可能略有差别。

讨论时应说明上下文。

### 3. Few-shot CoT

经典 CoT prompting 在 few-shot 示例中展示推理：

~~~text
示例：
问题：商店有 10 个苹果，卖出 3 个，还剩几个？
推理：初始 10 个，卖出 3 个，10-3=7。
答案：7

现在：
问题：仓库有 25 箱，运走 8 箱，还剩几箱？
推理：
~~~

模型不仅模仿答案格式，也模仿“先分解再作答”的模式。

示例质量非常重要。

错误步骤会被模型模仿。

### 4. Zero-shot CoT

Zero-shot CoT 不给完整示例，而加入诱导推理的短语。

经典研究使用过类似：

> Let's think step by step.

中文可以表达为“请逐步分析”。

它在部分任务和模型上能激活分步推理模式。

但不能把一句短语当成跨模型固定技巧。

现代推理模型可能已经在训练中学会分配推理计算，额外指令反而冗余或限制策略。

### 5. 为什么 CoT 可能有效：任务分解

直接从问题映射到答案：

~~~text
复杂输入 ─────────→ 最终答案
~~~

加入中间步骤：

~~~text
复杂输入 → 子问题 1 → 子问题 2 → 检查 → 最终答案
~~~

每一步条件更具体。

模型可以把复杂组合问题拆成训练中更常见的小模式。

### 6. 为什么 CoT 可能有效：额外计算 Token

Transformer 一次前向对当前上下文做固定深度计算。

自回归生成更多中间 Token，相当于把中间结果写回上下文，再进行新的前向步骤。

可以把它理解为：

- Token 不只是语言输出；
- 也能充当可读写的中间状态；
- 更长轨迹提供更多串行计算机会。

这也是 test-time compute 的一种直觉。

更多 Token 只提供机会，不保证计算方向正确。

### 7. 为什么 CoT 可能有效：模式提示

预训练数据中包含：

- 教科书推导；
- 代码执行步骤；
- 数学证明；
- 解释性问答；
- 论坛解题过程。

CoT 示例可能帮助模型识别“当前任务应采用哪类推理模式”。

这种能力仍受训练分布和模型规模影响。

### 8. 一个分解例子

问题：

> 一个项目原计划 40 天，已完成 25%，剩余工作按每天 1.5% 的总项目量推进，还需多少天？

可拆成：

~~~text
已完成：25%
剩余：100% - 25% = 75%
每天推进：1.5%
所需天数：75 / 1.5 = 50
~~~

题目中的“原计划 40 天”在已经给定新速度为“每天完成总项目量的 1.5%”后，不参与剩余天数计算，是一个干扰条件。把相关条件与无关条件分开，本身也是任务分解的一部分。

这里中间表示让除法目标更清楚。

但如果第一步误把剩余算成 85%，后续形式再整齐也会错。

### 9. CoT 不是搜索本身

单条 CoT 通常仍是一条自回归生成路径。

它不自动：

- 回退错误步骤；
- 枚举全部可能方案；
- 验证事实；
- 执行代码；
- 查询外部数据。

要探索多条路径，需要采样、搜索、规划或外部验证机制。

### 10. Self-Consistency

Self-Consistency 的典型流程：

~~~text
同一问题
  ├─ 采样推理路径 1 → 答案 A
  ├─ 采样推理路径 2 → 答案 B
  ├─ 采样推理路径 3 → 答案 A
  ├─ 采样推理路径 4 → 答案 A
  └─ 采样推理路径 5 → 答案 C

多数答案：A
~~~

它不是选择最高概率的单条推理，而是从多样路径的答案分布中聚合。

### 11. 多数投票公式

生成 $N$ 条轨迹，最终答案为 $a_1,\ldots,a_N$。

多数投票可写成：

$$
\hat{a}
=
\arg\max_a
\sum_{i=1}^{N}\mathbf{1}[a_i=a]
$$

其中：

- $\mathbf{1}[a_i=a]$ 在相等时为 1，否则为 0；
- 对每个候选答案计数；
- $\hat a$ 是票数最多的答案。

开放文本需要先做规范化或语义聚类，不能直接比较字符串。

### 12. Self-Consistency 为什么可能有效

如果：

- 正确答案可由多条路径到达；
- 错误是相对分散的；
- 采样具有足够多样性；

多数正确路径可能压过偶然错误。

但如果模型有系统性偏差，所有路径都可能得到同一个错误答案。

相关性很高的样本也不能视为独立投票者。

### 13. 自洽采样的成本

若每条推理平均 $T$ 个 Token，采样 $N$ 条：

$$
\text{输出计算量}\propto N\times T
$$

还可能增加：

- 峰值并发；
- 汇总步骤；
- 解析失败；
- 用户等待时间；
- 成本方差。

因此可先在难题或高价值请求上触发，而不是所有请求都固定采样十次。

### 14. CoT 的“忠实性”问题

Faithfulness 问的是：

> 模型写出的理由，是否真实反映了导致答案的内部因果因素？

研究发现，可见 CoT 可能：

- 忽略实际影响答案的偏置提示；
- 事后为答案编出合理解释；
- 在答案不变时给出不同理由；
- 在理由看似正确时得到错误答案；
- 在理由含错误时偶然得到正确答案。

所以可读不等于忠实。

### 15. 一个非忠实解释例子

假设多选题 Prompt 暗中标记“多数示例答案是 C”。

模型受这个偏置影响选择 C，却输出一段看似基于知识的解释。

解释中没有提到偏置。

人会误以为模型完全按显式理由决策。

这说明 CoT 不能直接作为审计证明。

### 16. 正确答案也可能配错误 CoT

例子：

~~~text
问题：7 × 8 等于多少？
推理：7 × 8 = 54，再加 2，所以答案 56。
答案：56
~~~

最终答案碰巧正确，但步骤错误。

若评测只看 final answer，会漏掉过程缺陷。

在不可验证的新问题上，这种缺陷可能造成错误。

### 17. 错误传播

CoT 是串行轨迹。

早期错误会成为后续上下文：

~~~text
错误假设
  ↓
基于错误假设的正确运算
  ↓
结构严谨的错误结论
~~~

文本越流畅，越容易让用户高估可靠性。

### 18. 结果监督与过程监督

#### Outcome Supervision

只根据最终答案给分。

优点：

- 标签更容易获得；
- 可用单元测试、数学答案等验证；
- 不需要逐步标注。

缺点：

- 无法区分正确推理与偶然猜中；
- 奖励信号稀疏；
- 难定位哪一步出错。

#### Process Supervision

对中间步骤提供反馈。

优点：

- 信号更细；
- 可发现第一处错误；
- 可能帮助学习更可靠过程。

缺点：

- 标注成本高；
- “标准步骤”未必唯一；
- 标注规则可能限制有效的新路径；
- 过程评分器也会犯错。

### 19. 可见 CoT 与隐藏推理

产品可有三层内容：

1. 模型内部或隐藏的推理 Token；
2. 面向系统的结构化证据和工具轨迹；
3. 面向用户的简洁说明。

它们不必相同。

用户通常需要：

- 结论；
- 关键依据；
- 可验证来源；
- 重要计算；
- 不确定性。

用户不一定需要完整自由文本推理轨迹。

### 20. 为什么不应盲目展示完整 CoT

潜在风险：

- 暴露系统提示或策略；
- 复述敏感上下文；
- 给攻击者提供绕过线索；
- 让错误解释显得更可信；
- 增加输出成本；
- 产生无关冗长内容；
- 泄漏受版权或隐私保护的数据。

更稳妥的是返回简洁、可验证的理由摘要。

### 21. 推理模型与普通模型

“推理模型”通常经过专门训练，在回答前分配更多推理计算。

对这类模型：

- 清晰描述目标与约束通常很重要；
- 不必总要求输出逐字 CoT；
- 过度规定每一步可能限制其策略；
- 供应商可能隐藏原始推理并返回摘要；
- 具体提示方法应看该模型当前官方指南。

不能把 2022 年的一个 CoT 短语机械复制到所有新模型。

### 22. CoT 与 ReAct

ReAct 把推理与动作交错：

~~~text
判断需要什么信息
  ↓
调用搜索或工具
  ↓
读取观察结果
  ↓
继续判断
  ↓
最终回答
~~~

它把纯文本推理连接到外部事实。

但工具执行必须由宿主授权和校验。

模型写“调用数据库”不表示数据库已被访问。

### 23. CoT 与代码执行

对精确计算，模型可以：

1. 生成计算计划；
2. 调用计算器或 Python 沙箱；
3. 获取真实结果；
4. 用简洁解释呈现。

这通常比让语言模型手算长小数更可靠。

代码执行仍需资源限制、输入校验和沙箱。

### 24. CoT 与检索

CoT 可以帮助模型列出需要哪些证据。

但不能凭推理生成不存在的事实。

理想流程：

~~~text
分解问题
  ↓
检索每个子问题的证据
  ↓
校验证据来源
  ↓
综合答案
~~~

推理链和证据链要区分。

## 工程实践与边界

### 1. 先判断任务是否需要 CoT

通常不需要长 CoT：

- 简单分类；
- 文本改写；
- 单字段抽取；
- 固定模板填充；
- 用户只需快速答复。

可能受益：

- 多步数学；
- 复杂规则判断；
- 规划；
- 多跳问答；
- 代码调试；
- 需要分解的分析任务。

### 2. 建立三种基线

可以比较：

~~~text
A. 直接回答
B. 单条分步推理
C. 多路径采样 + 验证/投票
~~~

评测：

- 最终正确率；
- 过程错误率；
- 输出 Token；
- 延迟；
- 成本；
- 安全泄露；
- 用户可读性。

### 3. 最终答案要可解析

即便允许中间推理，也可要求明确最终字段：

~~~json
{
  "answer": 60,
  "unit": "CNY",
  "confidence": "medium",
  "evidence_ids": ["calc_1"]
}
~~~

不要用正则从一大段自由文本最后一句猜答案。

更好的是结构化输出或专门答案通道。

### 4. 验证优先级

不同任务使用不同验证器：

| 任务 | 可验证手段 |
| --- | --- |
| 数学 | 计算器、符号系统 |
| 代码 | 编译、单元测试、静态检查 |
| SQL | 只读沙箱、Schema 校验 |
| 事实问答 | 权威来源、引用支持 |
| 规则判断 | 规则引擎、决策表 |
| JSON | Schema 校验 |

自然语言 CoT 只能作为辅助信号。

### 5. Adaptive Compute

不是每题都需要同样多计算。

可以按：

- 问题复杂度；
- 模型不确定性；
- 业务风险；
- 首次验证是否通过；
- 用户延迟预算；

决定是否增加推理 Token、采样更多路径或升级模型。

### 6. Self-Consistency 的实现检查

- 样本使用适度多样的采样；
- 最终答案规范化；
- 数值包含单位；
- 无法比较时使用验证器；
- 设置最大样本数和截止时间；
- 允许提前形成稳定多数；
- 记录路径相关性和分歧率；
- 不把多数票当事实证明。

### 7. 过程评测

可记录：

- 第一个错误步骤；
- 无依据步骤比例；
- 计算器结果一致率；
- 引用支持率；
- 自相矛盾率；
- 最终答案与过程一致率；
- 过程长度。

过程评分最好与最终结果评分分开。

### 8. 安全与隐私

如果推理上下文包含：

- 系统策略；
- 其他用户数据；
- 密钥；
- 内部文档；
- 工具原始结果；

不应通过“展示思考过程”直接返回。

日志也要脱敏、限权和设置保留期限。

### 9. 面向用户的解释设计

推荐输出：

~~~text
结论：60 元
依据：3 × 24 = 72；72 - 12 = 60
~~~

而不是输出数百 Token 的自我对话。

解释应围绕可验证依据，不需要模仿人的意识流。

### 10. 模型升级必须重测

升级可能改变：

- 是否需要显式 CoT 提示；
- 推理长度；
- 隐藏推理 API；
- 对相同提示的遵循方式；
- 成本计费；
- 延迟；
- 自洽采样收益。

提示策略要绑定模型版本。

### 11. CoT 的边界

CoT 无法单独保证：

- 中间事实真实；
- 推理规则正确；
- 解释忠实；
- 最终答案正确；
- 外部操作已执行；
- 没有敏感信息泄露。

工程结论仍需验证。

## 常见误区

- **认为 CoT 是模型内部神经活动的完整转录**：它是模型生成的文本轨迹，不应直接当成忠实读出。
- **认为写得越长推理越强**：更多 Token 也可能重复、跑题和传播错误。
- **认为“逐步思考”对所有模型都有效**：模型训练方式不同，应遵循对应指南并实测。
- **看到正确 final answer 就认为过程正确**：可能猜中或用错误步骤碰巧得到正确值。
- **看到过程合理就认为答案可靠**：模型能为错误答案生成流畅解释。
- **把 Self-Consistency 当独立专家投票**：样本来自同一模型，错误高度相关。
- **所有请求都生成十条路径**：成本和延迟可能不符合业务价值。
- **把 CoT 当外部证据**：事实需要来源、工具或可执行验证。
- **公开完整推理能提高透明度**：也可能泄露敏感信息并制造虚假可信感。
- **推理模型必须被要求展示每一步**：不少推理模型更适合直接给目标和约束。
- **过程监督没有偏差**：步骤标注和过程评分器也会带来偏好。
- **CoT 可以替代测试**：代码、数学和结构仍应使用确定性验证器。

## 面试追问

### 问：CoT 为什么可能提升推理能力

**答：** 它把复杂映射拆成多个条件生成步骤，中间 Token 可保存子结果并提供额外串行计算机会，同时提示模型复用训练数据中的推理模式。

### 问：CoT 是模型真实思考过程吗

**答：** 不应这样断言。它是模型生成的中间文本，可能有用但未必忠实反映真正影响答案的内部因素，也可能是事后合理化。

### 问：Self-Consistency 怎样工作

**答：** 对同一问题采样多条具有差异的推理路径，抽取并规范化最终答案，再用多数投票或验证器聚合。它以更多推理成本换取稳健性。

### 问：生产系统应不应该展示完整 CoT

**答：** 通常更适合展示简洁、可验证的结论依据，而不是原始自由文本轨迹。后者可能不忠实、冗长，并泄露提示、安全策略或敏感上下文。
