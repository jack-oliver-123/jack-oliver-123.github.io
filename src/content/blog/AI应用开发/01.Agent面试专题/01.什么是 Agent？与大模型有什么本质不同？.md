## 本篇目标

先分清大模型、Agent 和 Workflow。读完后，你应该能够判断一个系统是否真的具有 Agent 特征。

## 60 秒回答

大模型和 Agent 不是同一层级的概念。大模型是根据输入生成输出的模型，可以负责理解、生成和推断；Agent 是围绕模型搭建的任务执行系统，它还包含目标、状态、工具、控制循环和安全规则。

Agent 的典型过程是：模型根据当前状态提出下一步动作，外层运行时检查权限和参数后执行，再把工具结果反馈给模型，让它继续、改道或结束。因此，一次大模型调用或一次工具调用不一定是 Agent。真正的判断标准是：系统是否会根据环境反馈，在受限范围内动态决定下一步，并且有程序化停止条件。

## 详细解析

### 先说人话：大模型像大脑，Agent 像办事系统

你可以把大模型理解成一位知识丰富的顾问。你问它一个问题，它给出一个回答，但这次调用结束后，它不会自己继续工作。

Agent 更像一名在公司制度内办事的员工。它不仅可以思考，还能读取任务进度、选择工具、查看结果，再决定下一步。当然，这名“员工”不能随便操作生产系统，它的权限和行动必须由外层程序控制。

所以最重要的一句话是：

> **大模型主要负责生成和判断，Agent 负责围绕目标持续完成任务。**

### 用客服退款理解三者区别

假设用户说：“这件商品有质量问题，请帮我退款。”

**只有大模型调用：**

大模型读取这句话，生成一段退款说明或客服回复。它没有查询真实订单，也没有执行退款。

**使用固定 Workflow：**

开发者提前写好流程：查询订单 -> 检查退款期限 -> 符合条件就退款 -> 发送通知。无论由谁执行，主要路径已经由代码规定。

**使用 Agent：**

系统先读取用户目标和当前状态。如果缺少订单号，就向用户询问；获得订单号后查询订单；查询失败时判断是重试、换工具还是转人工；发现退款金额超过自动处理上限时发起人工审批。后续步骤会随着真实结果变化。

这三种方式没有谁天然更高级。简单任务用一次模型调用就够了，稳定流程优先使用 Workflow，只有后续路径难以提前枚举时才需要 Agent 的动态决策。

### Agent 的最小闭环

一个最小可用 Agent 至少包含六部分：

| 部分 | 作用 | 退款例子 |
| --- | --- | --- |
| 目标 | 明确最终要完成什么 | 解决用户退款问题 |
| 状态 | 保存当前知道的信息和任务进度 | 已获得订单号，尚未确认退款资格 |
| 动作选择 | 决定下一步做什么 | 查询订单或询问补充信息 |
| 受控执行 | 校验后真正调用工具 | 运行时调用订单查询接口 |
| 结果反馈 | 把执行结果写回任务状态 | 订单已超过自动退款金额 |
| 停止条件 | 决定成功、失败或转人工 | 审批完成、超时或达到最大步数 |

完整闭环可以记成：

> 读取目标和状态 -> 选择动作 -> 校验权限和参数 -> 执行 -> 记录结果 -> 重新决策或结束。

这里容易忽略的一点是：**模型并不会自己常驻运行。** 每一次模型调用都会结束，是外层程序把新状态再次交给模型，从而形成循环。

### 模型提出动作，运行时负责执行

假设模型提出“为订单 123 退款 100 元”。这只是一个动作请求，不能直接等于真实退款。外层运行时还要检查：

- 当前用户是否拥有这个订单；
- 订单是否符合退款政策；
- 金额是否超过自动处理上限；
- 是否需要人工审批；
- 同一退款是否已经执行过。

检查通过后，运行时才会调用退款接口。这个边界非常重要，因为模型输出具有不确定性，外部网页、邮件或文档还可能包含恶意指令。

### 从三个层次看本质区别

#### 第一层：能力

大模型擅长理解语言、生成内容、抽取信息和进行判断。它可以根据工具说明生成调用请求，但模型本身不会拿着生产密钥访问数据库，也不会亲自完成付款。

这就像一个人可以提出“应该查询订单”，但真正访问公司数据库还需要账号、权限和系统接口。

#### 第二层：控制

一次大模型调用结束后，不会自己在后台继续运行。Agent 的外层程序会保存任务状态，再次调用模型，并决定：

- 是否接受模型提出的动作；
- 是否需要调用工具；
- 是否要求用户补充信息；
- 是否暂停等待审批；
- 是否已经达到停止条件。

因此，Agent 的“持续工作”来自宿主程序维护的循环，而不是模型成为了一个永不停止的进程。

#### 第三层：系统

生产 Agent 还必须处理模型之外的问题：

- 用户身份和租户隔离；
- 工具权限；
- 状态持久化；
- 超时、重试和幂等；
- Token 与费用预算；
- 审计和人工接管；
- 评测与故障回放。

这也是为什么只换一个更强的大模型，不能自动解决 Agent 的全部可靠性问题。

### Agent 与 Workflow 是一条连续光谱

它们不是非此即彼：

- 路径和分支主要由开发者预先定义，更接近 Workflow；
- 部分路径由模型根据运行结果动态选择，开始具有 Agent 特征；
- 生产系统通常混合两者：高风险主流程用 Workflow 固定，信息不完整的局部步骤允许 Agent 判断。

判断时不要只看系统是否使用了大模型，也不要只看它是否调用工具，要看**下一步的控制权主要属于谁**。

### 判断一个系统是否具有 Agent 特征

面试中遇到具体案例，可以按下面六个问题判断：

1. 系统是否围绕一个目标持续执行，而不是只生成一次结果？
2. 是否显式保存当前任务状态？
3. 模型是否会根据状态选择下一步动作？
4. 工具结果是否会反馈回来并改变后续决策？
5. 是否允许继续、改道、澄清、暂停或结束？
6. 是否存在由程序强制执行的预算和停止条件？

满足的项目越多，Agent 特征越明显。只有“调用了 LLM”或“调用了一个函数”，都不足以单独判断。

还要注意，Agent 的自主性不是开关，而是程度：

| 自主程度 | 示例 |
| --- | --- |
| 很低 | 模型只在三个固定分支中分类 |
| 中等 | 模型在一个节点内选择有限工具并循环 |
| 较高 | 模型能规划多步任务并根据结果重规划 |

生产系统通常不会追求最高自主性，而是选择能够解决问题的最小决策空间。

## 工程实践与边界

- **限制预算**：为每次运行设置最大步数、时间、Token、费用和工具调用次数，由程序强制结束。
- **限制权限**：工具按最小权限授权，查询与写入分开；付款、删除和向外发送数据等动作需要额外审批。
- **隔离数据**：状态按租户和用户隔离，日志中不保存完整密钥、不必要的个人可识别信息（PII）和大段工具原文。
- **正确处理失败**：所有工具调用都使用关联 ID 便于追踪；可能重试且有副作用的写操作，在下游支持时还要使用幂等键。超时不能证明写入失败，无法确认状态时应转人工核对或执行补偿，不能盲目重放。
- **保留可审计记录**：记录动作、参数摘要、结果状态、模型和提示版本，便于复现问题，但不依赖或公开模型的私有思维链。

## 常见误区

- **“会调工具就是 Agent”**：一次工具调用也可能只是 Workflow 中的固定节点，关键是有没有根据反馈继续决策的闭环。
- **“Agent 会自己执行工具”**：模型只提出动作请求，宿主程序负责鉴权、执行和回传。
- **“自主性越高越好”**：决策空间越大，测试、治理和故障定位成本通常越高。
- **“用了大模型就是 Agent”**：固定流程可以把大模型当作分类、抽取或生成节点，但控制权仍属于 Workflow。
- **“长上下文等于记忆”**：上下文是一次请求中模型看到的内容；跨任务记忆还需要存储、检索、权限和删除策略。

## 面试追问

> **面试官：** Agent 最小可用闭环是什么？
>
> **候选人：** 至少要有目标、状态、动作选择、受控执行、结果反馈和程序化停止条件。只有一次生成或一次工具调用，还不能说明系统形成了 Agent 闭环。

> **面试官：** 工具调用为什么不一定是 Agent？
>
> **候选人：** 如果工具和执行顺序都是开发者预先规定的，模型只负责生成一次参数，它仍然可以是 Workflow。只有模型会根据工具反馈继续选择、改道或停止，才体现出 Agent 特征。

> **面试官：** 为什么不让 Agent 完全自主？
>
> **候选人：** 模型决策具有不确定性，外部输入也可能包含提示注入。生产系统必须限制工具、权限、预算和副作用，并为付款、删除、外发数据等高风险动作保留人工审批。

> **面试官：** 如何判断一个任务是否值得引入 Agent？
>
> **候选人：** 先看任务路径能否可靠枚举。路径稳定时优先 Workflow；只有环境反馈会改变后续步骤、规则难以穷举时，Agent 的动态决策才可能抵消新增的成本和风险。
