## 前言

Git 命令很多，初学时不需要一次记住全部命令。更实际的做法是先掌握日常高频命令：会查看状态、提交修改、切换分支、同步远程仓库、撤销错误操作，就已经能完成大多数开发工作。

这篇文章按真实使用顺序整理 Git 常用命令，适合作为日常速查笔记。

## 常用命令速查

| 场景 | 命令 | 作用 |
| --- | --- | --- |
| 初始化仓库 | `git init` | 把当前目录变成 Git 仓库 |
| 克隆仓库 | `git clone <地址>` | 从远程复制一份仓库到本地 |
| 查看状态 | `git status` | 查看哪些文件被修改、暂存或未跟踪 |
| 查看差异 | `git diff` | 查看工作区中的修改内容 |
| 添加暂存 | `git add <文件>` | 把修改加入暂存区 |
| 提交修改 | `git commit -m "说明"` | 把暂存区内容提交为一个版本 |
| 查看历史 | `git log` | 查看提交记录 |
| 创建分支 | `git switch -c <分支名>` | 创建并切换到新分支 |
| 切换分支 | `git switch <分支名>` | 切换到已有分支 |
| 拉取远程 | `git pull` | 获取远程更新并合并到当前分支 |
| 推送远程 | `git push` | 把本地提交推送到远程仓库 |
| 撤销文件修改 | `git restore <文件>` | 丢弃工作区中的文件修改 |
| 临时保存现场 | `git stash` | 暂时保存未提交的修改 |

## 仓库初始化与克隆

### 初始化本地仓库

进入项目目录后执行：

```bash
git init
```

执行后，当前目录会多出一个隐藏的 `.git` 文件夹。这个文件夹保存 Git 仓库的历史记录、分支、配置等信息，不要手动删除或随意修改。

### 克隆远程仓库

如果项目已经在 GitHub、GitLab 或 Gitee 上，可以使用：

```bash
git clone https://github.com/用户名/仓库名.git
```

克隆完成后会自动生成一个项目目录，并且默认配置好远程仓库地址。

如果想指定本地目录名，可以写成：

```bash
git clone https://github.com/用户名/仓库名.git my-project
```

## 查看仓库状态

### 查看当前状态

```bash
git status
```

这是最常用的 Git 命令之一。它会告诉你：

- 哪些文件被修改了；
- 哪些文件已经加入暂存区；
- 哪些文件还没有被 Git 跟踪；
- 当前所在分支是否领先或落后远程分支。

日常使用 Git 时，建议在提交前先执行一次 `git status`。

### 简洁模式

```bash
git status -s
```

常见输出含义：

```txt
 M README.md
A  src/main.js
?? notes.md
```

说明：

- `M`：文件被修改；
- `A`：新增文件已加入暂存区；
- `??`：新文件还没有被 Git 跟踪。

## 查看文件差异

### 查看工作区修改

```bash
git diff
```

这个命令会显示还没有加入暂存区的修改。

### 查看暂存区修改

```bash
git diff --staged
```

也可以写成：

```bash
git diff --cached
```

这个命令会显示已经 `git add`、但还没有 `git commit` 的修改。

### 查看某个文件的差异

```bash
git diff README.md
```

如果只想检查某一个文件，可以在命令后面加文件路径。

## 暂存修改

Git 提交前通常需要先把修改加入暂存区。

### 添加指定文件

```bash
git add README.md
```

### 添加多个文件

```bash
git add README.md package.json
```

### 添加当前目录下所有修改

```bash
git add .
```

`git add .` 很方便，但使用前最好先执行 `git status`，确认不会把临时文件、日志文件或不该提交的内容加进去。

### 取消暂存

如果文件已经加入暂存区，但暂时不想提交，可以执行：

```bash
git restore --staged README.md
```

这只会把文件从暂存区移出，不会删除你的实际修改。

## 提交修改

### 普通提交

```bash
git commit -m "完善 Git 常用命令文章"
```

提交说明应该简洁描述这次修改的目的。不要只写 `update`、`fix` 这类过于模糊的说明。

### 跳过暂存并提交已跟踪文件

```bash
git commit -am "更新文档"
```

这个命令相当于对已经被 Git 跟踪的修改文件执行 `git add`，然后提交。

注意：它不会提交新建文件。新文件仍然需要先执行 `git add <文件>`。

### 修改上一次提交说明

```bash
git commit --amend -m "新的提交说明"
```

如果刚提交完就发现提交说明写错了，可以用这个命令修改。

如果这次提交已经推送到远程仓库，并且别人可能已经基于它继续开发，就不要随意修改提交历史。

## 查看提交历史

### 查看完整日志

```bash
git log
```

会显示提交哈希、作者、时间和提交说明。

### 单行显示日志

```bash
git log --oneline
```

输出类似：

```txt
f3a1c2d 完善 Git 常用命令文章
9b8e7a6 初始化博客结构
```

这适合快速查看提交历史。

### 查看最近几次提交

```bash
git log --oneline -5
```

### 查看某个文件的提交历史

```bash
git log -- README.md
```

### 查看每次提交的改动

```bash
git log -p
```

`-p` 会展示每次提交的具体差异，适合排查某段代码是在哪次提交中修改的。

## 分支管理

分支可以让你在不影响主分支的情况下开发新功能、修复问题或尝试方案。

### 查看分支

```bash
git branch
```

当前所在分支前面会有 `*` 标记。

### 创建并切换分支

```bash
git switch -c feature-search
```

这会创建 `feature-search` 分支，并立即切换过去。

### 切换已有分支

```bash
git switch main
```

旧版本 Git 中常见写法是：

```bash
git checkout main
```

现在更推荐使用 `git switch` 切换分支，因为语义更清晰。

### 删除本地分支

```bash
git branch -d feature-search
```

如果分支还没有合并，Git 会阻止删除。确认不需要这个分支时，可以强制删除：

```bash
git branch -D feature-search
```

## 合并与变基

### 合并分支

假设你在 `main` 分支，想把 `feature-search` 合并进来：

```bash
git switch main
git merge feature-search
```

如果两个分支修改了同一处内容，可能会出现冲突，需要手动解决后再提交。

### 变基分支

```bash
git rebase main
```

`rebase` 会把当前分支的提交“移动”到目标分支最新提交之后，让提交历史更线性。

使用建议：

- 自己本地的功能分支可以使用 `rebase` 整理历史；
- 已经推送并被别人使用的公共分支，不要随意 `rebase`；
- 不确定时，优先使用 `merge`。

## 远程仓库

### 查看远程仓库

```bash
git remote -v
```

输出示例：

```txt
origin  https://github.com/用户名/仓库名.git (fetch)
origin  https://github.com/用户名/仓库名.git (push)
```

`origin` 是默认远程仓库名称。

### 添加远程仓库

```bash
git remote add origin https://github.com/用户名/仓库名.git
```

### 修改远程仓库地址

```bash
git remote set-url origin https://github.com/用户名/新仓库名.git
```

### 拉取远程更新

```bash
git pull
```

`git pull` 相当于先获取远程更新，再合并到当前分支。

### 只获取远程更新

```bash
git fetch
```

`git fetch` 只把远程更新下载到本地，不会自动修改当前工作分支。想先查看远程变化，再决定是否合并时，可以使用它。

### 推送当前分支

```bash
git push
```

第一次推送新分支时，通常需要指定远程分支：

```bash
git push -u origin feature-search
```

`-u` 会建立本地分支和远程分支的追踪关系。之后在这个分支上就可以直接使用 `git push` 和 `git pull`。

## 撤销修改

撤销命令要谨慎使用。执行前建议先用 `git status` 和 `git diff` 看清楚会影响哪些内容。

### 丢弃工作区中某个文件的修改

```bash
git restore README.md
```

这会把文件恢复到最近一次提交或暂存区中的状态。未保存的修改会丢失。

### 丢弃所有工作区修改

```bash
git restore .
```

这个命令会丢弃当前目录下所有未暂存修改，使用前一定要确认。

### 回退到某个提交

```bash
git reset --soft <提交哈希>
```

`--soft` 会回退提交历史，但保留修改在暂存区。

```bash
git reset --mixed <提交哈希>
```

`--mixed` 是默认模式，会回退提交历史，并把修改保留在工作区。

```bash
git reset --hard <提交哈希>
```

`--hard` 会回退提交历史，同时丢弃工作区和暂存区的修改。这个命令破坏性较强，不要随便使用。

### 用新的提交撤销旧提交

```bash
git revert <提交哈希>
```

`git revert` 不会删除历史，而是生成一个新的提交来抵消指定提交的修改。

如果代码已经推送到远程仓库，并且多人协作，通常优先使用 `git revert`，不要直接改写公共历史。

## 临时保存修改

有时你正在写代码，突然需要切换分支处理其他问题，但当前修改还不适合提交。这时可以使用 `stash`。

### 保存当前修改

```bash
git stash
```

也可以添加说明：

```bash
git stash push -m "临时保存搜索页样式"
```

### 查看保存列表

```bash
git stash list
```

### 恢复最近一次保存

```bash
git stash pop
```

`pop` 会恢复修改，并从 stash 列表中删除这条记录。

### 只恢复但保留记录

```bash
git stash apply
```

如果你不确定恢复后是否会有冲突，可以先用 `apply`。

### 删除某条 stash

```bash
git stash drop stash@{0}
```

## 标签管理

标签通常用来标记版本发布点，例如 `v1.0.0`。

### 查看标签

```bash
git tag
```

### 创建轻量标签

```bash
git tag v1.0.0
```

### 创建带说明的标签

```bash
git tag -a v1.0.0 -m "发布 1.0.0 版本"
```

更推荐使用带说明的标签，因为它能记录标签说明、作者和时间。

### 推送标签到远程

```bash
git push origin v1.0.0
```

### 推送所有标签

```bash
git push origin --tags
```

### 删除本地标签

```bash
git tag -d v1.0.0
```

### 删除远程标签

```bash
git push origin --delete v1.0.0
```

## 常见工作流程

### 第一次提交项目

```bash
git init
git add .
git commit -m "初始化项目"
git remote add origin https://github.com/用户名/仓库名.git
git push -u origin main
```

如果你的默认分支不是 `main`，需要根据实际情况调整分支名。

### 日常开发提交流程

```bash
git status
git diff
git add .
git diff --staged
git commit -m "说明这次修改"
git push
```

推荐在 `commit` 前查看一次 `git diff --staged`，确认提交内容和预期一致。

### 新功能分支开发流程

```bash
git switch main
git pull
git switch -c feature-login

# 修改代码后
git add .
git commit -m "添加登录功能"

git switch main
git pull
git merge feature-login
git push
```

如果团队使用 Pull Request 或 Merge Request，通常是在功能分支推送后，通过平台页面发起合并请求。

## 常见问题

### 提交时提示没有配置用户名和邮箱

先配置全局用户信息：

```bash
git config --global user.name "你的名字"
git config --global user.email "你的邮箱"
```

查看配置：

```bash
git config --global --list
```

### 不小心把文件加入暂存区

```bash
git restore --staged 文件名
```

如果只是不想提交它，而不是删除修改，用这个命令即可。

### 不小心改错文件

先查看差异：

```bash
git diff 文件名
```

确认可以丢弃后再执行：

```bash
git restore 文件名
```

### 拉取代码时出现冲突

常见处理步骤：

1. 打开冲突文件；
2. 找到 `<<<<<<<`、`=======`、`>>>>>>>` 标记；
3. 手动保留正确内容；
4. 删除冲突标记；
5. 重新暂存并提交。

```bash
git add 冲突文件
git commit -m "解决合并冲突"
```

### 想看某个命令的帮助

```bash
git help status
```

或者：

```bash
git status --help
```

如果只想快速看参数说明，也可以使用：

```bash
git status -h
```

## 总结

Git 的学习重点不是背命令，而是理解工作区、暂存区、本地仓库和远程仓库之间的关系。

日常开发中，最常用的流程可以记成：

```txt
查看状态 → 查看差异 → 加入暂存区 → 提交 → 推送
```

对应命令就是：

```bash
git status
git diff
git add .
git commit -m "提交说明"
git push
```

先把这条主线用熟，再逐步掌握分支、撤销、stash、tag 等命令，Git 就会从“容易出错的黑盒工具”变成日常开发中非常可靠的版本管理工具。
