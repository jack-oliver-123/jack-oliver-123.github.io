“把模型部署起来”可能指四种完全不同的事情：

- 在笔记本上离线聊天；
- 在一张 GPU 上给内部十个人使用；
- 在 GPU 集群上服务上千并发请求；
- 运行一个包含检索、结构化生成和多轮工具调用的程序。

这四种场景不应只凭“哪个框架跑分最高”选择同一个引擎。

vLLM、TGI、llama.cpp、SGLang 都能提供模型推理能力，但定位、硬件、调度、生态和当前维护状态不同。

真正的选型问题是：

> 在我的模型、硬件、流量和功能约束下，哪个方案以可接受的质量和运维复杂度达到目标 SLO？

## 本篇目标

读完本篇，你应该能够：

- 解释模型权重、推理引擎、API Server 和业务网关的区别；
- 说清 Continuous Batching、PagedAttention、Prefix Cache 的价值；
- 描述 vLLM、TGI、llama.cpp、SGLang 的主要定位；
- 正确说明 TGI 当前处于维护模式的官方状态；
- 根据 GPU 集群、本地 CPU/Apple Silicon 和复杂生成程序做初选；
- 判断“支持模型”和“高效支持模型”的差别；
- 设计公平的质量、延迟、吞吐、显存和稳定性压测；
- 给出分阶段上线、回滚与可观测性方案；
- 避免用过时榜单直接决定生产架构。

核心结论：

> 先确定硬件、模型和业务负载，再用官方支持矩阵筛选，最后在真实请求分布上压测；框架名称本身不是性能结论。

## 60 秒回答

我会先区分场景。vLLM 是面向高吞吐 LLM 推理和服务的活跃项目，核心能力包括 PagedAttention、连续批处理、Prefix Cache、分布式并行以及兼容常见 API，适合 GPU 服务基线。SGLang 同样面向高性能服务，强调 RadixAttention、复杂 LLM 程序的前缀复用、结构化输出、推理与大规模并行，适合共享前缀多、Agent/推理工作负载或需要深度调优的场景。两者实际优劣必须按目标模型和版本压测。

llama.cpp 是 C/C++ 本地推理生态，支持 GGUF、多种低比特量化、CPU、Apple Silicon、GPU 和 CPU+GPU 混合卸载，适合桌面、边缘、离线和资源有限环境，也提供 `llama-server`。它也能做服务，但大规模 GPU 集群调度通常不是最典型定位。

TGI 是 Hugging Face 的生产推理服务器，具备连续批处理、张量并行、流式和可观测性等能力；但其官方仓库当前明确标注 maintenance mode，并建议新方向使用 vLLM、SGLang 或本地引擎。已有稳定 TGI 系统不必因此立即迁移，但新项目应把长期维护状态纳入选型。

最后我会用相同模型、精度、模板、并发和输入输出长度，对候选方案比较正确性、TTFT、TPOT、吞吐、峰值显存、Prefix 命中、错误率和运维复杂度，再决定，而不是引用别人的单一 tokens/s。

## 详细解析

### 1. 部署栈不是一个程序

典型链路：

~~~text
客户端
  ↓
鉴权 / 限流 / 路由网关
  ↓
模型 API Server
  ↓
调度器 / Batching
  ↓
推理 Runtime / Kernel
  ↓
模型权重 / KV Cache
  ↓
GPU、CPU、NPU
~~~

vLLM 等项目通常覆盖 API Server、调度和推理运行时的一部分或大部分。

业务权限、计费、租户隔离和产品逻辑仍需上层系统负责。

### 2. 先认识部署术语

| 术语 | 通俗解释 |
| --- | --- |
| Model weights | 训练得到的参数文件 |
| Runtime | 真正执行算子的运行时 |
| Inference engine | 管理模型执行、KV 和批处理 |
| API server | 把推理能力暴露为 HTTP/gRPC 等接口 |
| Scheduler | 决定哪些请求在何时进入计算 |
| Gateway | 鉴权、限流、路由、配额和审计 |
| Kernel | 在硬件上执行矩阵乘、Attention 等底层程序 |

一个框架“支持 OpenAI-compatible API”不表示它覆盖完整生产治理。

### 3. 自回归服务为什么难

请求长度不同：

~~~text
请求 A：输入 100，输出 20
请求 B：输入 8000，输出 200
请求 C：输入 500，输出 2000
~~~

而 Decode 每一步依赖上一步。

服务需要同时处理：

- 动态到达；
- 不同 Prefill 长度；
- 不同 Decode 长度；
- KV Cache 增长；
- 中途取消；
- 流式返回；
- 显存不足；
- 多模型和 LoRA。

### 4. Static Batching 的问题

传统静态 batch 把一批请求一起开始、一起结束。

如果某条输出很长：

~~~text
A 已结束 ─────等待─────┐
B 已结束 ─────等待─────┼→ C 结束后整个 batch 完成
C 持续生成──────────────┘
~~~

GPU 槽位被已完成请求占用。

### 5. Continuous Batching

连续批处理允许每个 Decode 步重新组织活跃请求：

~~~text
时间 1：[A, B, C]
时间 2：A 完成，加入 D → [B, C, D]
时间 3：B 完成，加入 E → [C, D, E]
~~~

目标是提高 GPU 利用率和吞吐。

代价包括：

- 调度更复杂；
- batch 变化带来 kernel 管理；
- 大 Prefill 可能阻塞 Decode；
- 公平性和尾延迟需要治理。

### 6. PagedAttention

KV Cache 长度动态变化，连续预留会产生碎片和浪费。

PagedAttention 把 KV 分成固定块，通过逻辑到物理块映射管理。

优点包括：

- 减少内存碎片；
- 更灵活地增长序列；
- 支持块共享；
- 提高可容纳并发。

它是 vLLM 早期论文和系统设计的核心贡献之一。

如今多个引擎也实现了分页式 KV 管理。

### 7. Prefix Cache

若请求共享长前缀：

~~~text
同一系统提示 + 同一文档 + 问题 A
同一系统提示 + 同一文档 + 问题 B
~~~

Prefix Cache 可复用公共前缀的 KV 块，减少重复 Prefill。

选型时应关注：

- 命中粒度；
- 哈希和隔离；
- 淘汰；
- 多副本路由；
- LoRA 与多模态兼容；
- 命中指标。

### 8. Chunked Prefill

一个很长的 Prefill 可能让正在 Decode 的请求等待。

Chunked Prefill 把长输入分块调度，与 Decode 更灵活地交错。

可能改善公平性和吞吐，但参数会影响：

- TTFT；
- TPOT；
- 调度开销；
- GPU 利用率。

需要按负载压测。

### 9. vLLM 的定位

vLLM 官方将自己定位为高效、易用的 LLM 推理与服务库。

当前项目能力覆盖：

- PagedAttention；
- 连续批处理；
- Chunked Prefill；
- Prefix Caching；
- 多种量化；
- 张量、流水线、数据、专家等并行；
- 流式输出；
- 结构化输出；
- 工具与推理解析；
- 常见兼容 API；
- 多种 GPU、CPU 和硬件插件。

支持范围持续变化，应以部署版本文档为准。

### 10. vLLM 的典型优势

- 社区活跃；
- 模型覆盖广；
- GPU 高吞吐服务常用；
- OpenAI-compatible 接入方便；
- Paged KV 与连续调度成熟；
- 分布式与量化选项丰富；
- 监控和生态工具较多。

### 11. vLLM 的常见挑战

- 新模型首日支持可能需要特定版本；
- “支持架构”不代表所有特性都支持；
- 高级并行和量化组合兼容性复杂；
- 默认配置未必适合目标流量；
- GPU/CUDA/驱动依赖需要管理；
- 兼容 API 不能覆盖供应商所有语义；
- 频繁升级要做回归。

### 12. SGLang 的定位

SGLang 同时包含面向模型服务的 Runtime 与面向复杂语言模型程序的能力。

当前官方项目强调：

- RadixAttention；
- Prefix Caching；
- 连续批处理；
- Paged Attention；
- Chunked Prefill；
- Prefill/Decode 分离；
- 结构化输出；
- 推测解码；
- 多种并行和硬件；
- 多 LoRA；
- 推理与 RL rollout 集成。

具体模块和 API 随版本演进。

### 13. RadixAttention 的直觉

复杂程序常产生树状共享前缀：

~~~text
公共系统提示
  ├─ 文档 A
  │   ├─ 问题 1
  │   └─ 问题 2
  └─ 文档 B
      ├─ 问题 3
      └─ 问题 4
~~~

RadixAttention 使用 radix tree 管理前缀 KV，匹配、复用和淘汰共享前缀。

它对：

- 多轮对话；
- few-shot；
- 多分支采样；
- 树搜索；
- 共享文档问答；

可能有价值。

收益取决于真实前缀重复率。

### 14. SGLang 的典型优势

- 强调复杂生成程序的前缀复用；
- 高性能服务特性丰富；
- 结构化生成与推理工作负载投入较多；
- 大规模 MoE 和 Prefill/Decode 分离支持活跃；
- 可作为 RL rollout 后端；
- 社区和硬件支持持续扩展。

### 15. SGLang 的常见挑战

- 特性演进快，升级与配置需要持续跟踪；
- 不同模型、硬件和后端成熟度不同；
- 高级能力增加运维和调优复杂度；
- 不能只引用项目方峰值跑分；
- 与现有平台、监控和鉴权集成仍需工程工作。

### 16. llama.cpp 的定位

llama.cpp 是 C/C++ 大模型推理项目，强调低依赖和广泛本地硬件。

当前官方项目包括：

- CPU 推理；
- Apple Silicon / Metal；
- CUDA、HIP、Vulkan、SYCL 等后端；
- CPU+GPU 混合卸载；
- 多种低比特量化；
- GGUF 模型格式；
- `llama-cli`；
- `llama-server`；
- 常见 API 与本地 Web 使用方式；
- 文本和部分多模态模型支持。

### 17. GGUF 是什么

GGUF 是 ggml 生态使用的模型文件格式。

通常包含：

- 权重张量；
- 模型元数据；
- Tokenizer 信息；
- 量化类型。

它便于分发多种量化版本。

文件名中的 `Q4`、`Q5` 等只表示某种量化族，还要看具体后缀和当前实现。

### 18. llama.cpp 的典型优势

- 本地和离线部署方便；
- CPU 与消费级硬件友好；
- Apple Silicon 生态成熟；
- 低比特量化选择丰富；
- 单一二进制和嵌入式集成较轻；
- 可做 CPU+GPU 混合卸载；
- 适合桌面、边缘和隐私场景。

### 19. llama.cpp 的常见挑战

- 大规模多节点 GPU 服务不是最典型路径；
- 不同后端性能差异大；
- 量化格式多，质量和速度需实测；
- 模型支持可能依赖社区转换；
- 服务治理仍需上层补齐；
- 高并发调度目标与集群框架不同。

### 20. TGI 的定位与当前状态

Text Generation Inference 是 Hugging Face 的 LLM 推理服务器。

它提供过并继续保有的能力包括：

- 连续批处理；
- 张量并行；
- 流式 SSE；
- 常见模型支持；
- 多种量化；
- Prometheus 与 OpenTelemetry；
- 消息 API；
- 结构化生成等。

但截至 2026-07-15 核验时，TGI 官方仓库顶部明确标注：

> 项目进入 maintenance mode，主要接受小型修复、文档和轻量维护，并建议后续优先使用 vLLM、SGLang 或本地兼容引擎。

这是选型的重要长期信号。

### 21. 维护模式意味着什么

它不意味着：

- 现有服务立刻停止；
- 所有功能不能用；
- 必须当天迁移；
- 历史架构突然错误。

它意味着新项目要考虑：

- 新模型支持速度；
- 新硬件与 kernel；
- 安全修复承诺；
- 社区活跃度；
- 长期升级路线；
- 团队迁移成本。

现有稳定 TGI 系统应基于风险和收益规划迁移，不应只因标签仓促重写。

### 22. 四者对比

| 维度 | vLLM | SGLang | llama.cpp | TGI |
| --- | --- | --- | --- | --- |
| 典型定位 | 高吞吐模型服务 | 高性能服务与复杂生成程序 | 本地、边缘、广硬件推理 | Hugging Face 生产服务 |
| GPU 集群 | 强 | 强，含活跃大规模特性 | 非最典型定位 | 有分布式能力 |
| CPU/Apple 本地 | 有部分支持，需核对 | 依后端与版本 | 强项 | 非主要定位 |
| 低比特生态 | 多种格式 | 多种格式 | GGUF 与本地量化丰富 | 多种格式 |
| 前缀复用 | Prefix Cache | RadixAttention/Prefix Cache | 有上下文缓存能力，语义依版本 | 有相关优化，需查版本 |
| 结构化生成 | 支持 | 强调并持续优化 | 支持程度依版本 | 支持 |
| 当前维护方向 | 活跃 | 活跃 | 活跃 | 官方维护模式 |

表格用于初筛，不能代替版本化测试。

### 23. “OpenAI-compatible” 的边界

兼容接口通常方便迁移基础请求。

但可能存在差异：

- 支持的字段；
- 错误码；
- Streaming event；
- Tool Calling 模板；
- Structured Outputs；
- logprobs；
- usage 统计；
- reasoning 字段；
- seed 和停止语义。

客户端接通不代表行为等价。

### 24. 模型支持的三个层次

#### 能加载

模型不会报架构错误。

#### 能正确运行

Tokenizer、模板、位置编码、输出头和多模态处理正确。

#### 能高效运行

走 fused kernel、分页 KV、量化、并行和图优化路径。

选型要达到第三层，而不是只验证第一层。

### 25. 硬件支持也有层次

“支持 AMD”可能表示：

- 可以编译；
- 一部分模型可运行；
- 核心 kernel 已优化；
- 分布式可用；
- 特定量化可用。

同理适用于 NVIDIA、Intel、TPU、NPU 和 CPU。

必须查版本、型号和已知限制。

### 26. 为什么第三方跑分不能直接用

跑分可能使用不同：

- 模型；
- 精度；
- GPU；
- batch；
- 输入长度；
- 输出长度；
- 并发；
- Prefix 命中；
- 调度参数；
- 功能开关；
- 测量口径。

tokens/s 高可能以更高排队延迟为代价。

### 27. 延迟指标

#### TTFT

Time To First Token，从请求到首个输出 Token。

受排队、Prefill、缓存和网络影响。

#### TPOT

Time Per Output Token，输出阶段平均每 Token 时间。

#### ITL

Inter-Token Latency，相邻输出 Token 间隔分布。

#### E2E Latency

请求到完整回答结束的总时间。

平均值之外还应看 P95、P99。

### 28. 吞吐指标

可报告：

- requests/s；
- input tokens/s；
- output tokens/s；
- total tokens/s；
- 每 GPU 吞吐；
- 满足 SLO 下的最大并发。

不说明是 input 还是 output tokens/s，数字很难比较。

## 工程实践与边界

### 1. 按场景做第一轮筛选

#### 本地离线与 Apple Silicon

优先评估 llama.cpp。

理由：

- 本地生态；
- GGUF；
- Metal 和 CPU；
- 低依赖；
- 混合卸载。

#### 常规 GPU API 服务

优先把 vLLM 与 SGLang 放入候选。

根据目标模型和团队栈比较。

#### 共享前缀、复杂推理程序

重点测试 SGLang 的前缀复用和程序执行能力，同时保留 vLLM 基线。

#### 已有 TGI 生产系统

先盘点稳定性、模型路线和维护风险，再计划迁移验证。

### 2. 需求清单

- 模型架构和 commit；
- 精度与量化；
- GPU/CPU 型号；
- 单机或多机；
- 输入/输出长度分布；
- 峰值并发；
- TTFT、TPOT、P99 SLO；
- Tool Calling；
- Structured Outputs；
- 多 LoRA；
- Prefix Cache；
- 多模态；
- 许可证与离线要求。

没有这张清单，框架讨论容易变成偏好争论。

### 3. 建立正确性门槛

性能测试前先验证：

- Tokenizer 一致；
- Chat Template 正确；
- Stop Token 正确；
- Tool Call 可解析；
- JSON Schema 通过；
- 长上下文没有静默截断；
- 量化质量可接受；
- 多模态预处理一致。

错误输出再快也没有业务价值。

### 4. 公平压测配置

固定：

- 模型与权重；
- 精度或相近质量量化；
- 硬件；
- Prompt；
- 解码参数；
- 最大长度；
- API 客户端；
- 并发曲线；
- 预热策略；
- 测试时长。

记录每个引擎的调优参数，不隐藏特例。

### 5. 请求分布而不是单一长度

构造：

| 桶 | 输入 | 输出 | 占比示例 |
| --- | ---: | ---: | ---: |
| 短对话 | 256 | 128 | 40% |
| 普通知识问答 | 2048 | 256 | 35% |
| 长文档 | 16000 | 512 | 20% |
| 长生成 | 1024 | 4096 | 5% |

占比应来自真实日志，而不是照抄表中数字。

### 6. 并发压测

逐步增加：

~~~text
1 → 2 → 4 → 8 → 16 → 32 → 64 → ...
~~~

观察：

- 吞吐何时饱和；
- P99 何时越过 SLO；
- KV 水位；
- 排队时间；
- OOM；
- 请求拒绝与取消；
- GPU 利用率。

最大吞吐点未必是生产运行点。

### 7. Prefix Cache 压测

至少两组：

- 冷缓存，前缀不重复；
- 热缓存，按线上比例重复。

报告：

- 命中 Token；
- TTFT 改善；
- 显存占用；
- 淘汰；
- 多副本路由影响。

不要把全热缓存跑分当普通流量性能。

### 8. 稳定性测试

- 长时间 soak test；
- 模型热重载；
- worker 崩溃；
- GPU OOM；
- 客户端中途取消；
- 超长输入；
- 高低优先级混合；
- 下游网络变慢；
- 多租户突发流量。

生产系统的价值不只在峰值 tokens/s。

### 9. 可观测性

至少需要：

- 请求量和错误码；
- 队列长度；
- TTFT/TPOT/E2E 分位数；
- Prefill/Decode tokens/s；
- KV 使用率；
- Prefix 命中；
- GPU 利用率和显存；
- 批大小；
- 抢占/重计算；
- 模型和模板版本。

### 10. 安全与治理

不要直接把裸推理端口暴露给外部用户。

上层通常还需：

- TLS；
- 身份认证；
- 租户配额；
- 速率限制；
- 最大 Token 限制；
- 内容和工具策略；
- 审计；
- 隐私和数据保留；
- 请求取消；
- 滥用防护。

### 11. 容器与版本

锁定：

- 引擎镜像 digest；
- 模型 commit；
- 驱动；
- CUDA/ROCm；
- kernel 依赖；
- Tokenizer；
- Chat Template；
- 量化配置。

仅写 `latest` 会让回归难以定位。

### 12. 升级策略

~~~text
离线正确性
  ↓
离线性能
  ↓
影子流量
  ↓
少量灰度
  ↓
逐步扩量
  ↓
保留旧版本回滚
~~~

快速演进的推理框架尤其需要此流程。

### 13. TGI 迁移思路

已有 TGI 系统可：

1. 列出当前依赖的 API 和特性；
2. 选择 vLLM/SGLang/llama.cpp 候选；
3. 做协议兼容测试；
4. 比较同模型质量；
5. 压测性能与资源；
6. 影子复制流量；
7. 分租户迁移；
8. 保留回滚。

维护模式是规划输入，不是紧急停机命令。

### 14. 什么时候自建不划算

若团队：

- 请求量很小；
- 没有 GPU 运维能力；
- 模型更新频繁；
- 需要多区域弹性；
- 更重视上市速度；

托管 API 或云端推理服务可能总成本更低。

自建还要计算人力、闲置 GPU、值班和升级成本。

### 15. 选型结论模板

~~~text
我们先按模型和硬件支持排除不兼容方案。
然后用真实长度与并发对 vLLM、SGLang 等候选做正确性和性能测试。
本地离线场景单独评估 llama.cpp。
已有 TGI 因官方维护模式制定迁移路线。
最终按满足 SLO 时的单位请求成本、稳定性和团队运维复杂度选择。
~~~

## 常见误区

- **认为部署只是把权重加载进 GPU**：还涉及调度、API、网关、监控、安全和升级。
- **认为 tokens/s 最高就应直接选用**：还要满足正确性、P99 延迟、功能和运维约束。
- **把别人的跑分直接套用**：硬件、模型、长度、并发和缓存条件可能不同。
- **认为 OpenAI-compatible 就行为完全一致**：字段、Streaming、工具和错误语义可能不同。
- **认为模型能加载就已高效支持**：可能没有 fused kernel 或正确量化路径。
- **认为 llama.cpp 只能 CPU 单用户**：它支持多种 GPU 后端和 server，但典型定位仍与集群服务不同。
- **认为 vLLM 适合所有硬件和模型组合**：应查具体版本支持并压测。
- **认为 SGLang 只是一门 Prompt DSL**：当前项目包含高性能 Runtime 和丰富服务能力。
- **认为 TGI 维护模式等于立即不可用**：现有系统仍可运行，但长期演进风险需要管理。
- **只测平均延迟**：排队和长请求会影响 P95/P99。
- **用全热 Prefix Cache 数据宣传常规性能**：应同时报告冷、热和真实命中分布。
- **裸露推理服务到公网**：推理引擎不能替代业务鉴权与滥用治理。

## 面试追问

### 问：vLLM 和 SGLang 怎么选

**答：** 两者都可作为活跃的高性能 GPU 服务候选。先看目标模型、硬件和功能支持，再用真实流量压测；共享前缀和复杂推理程序可重点测试 SGLang，常规模型 API 可把 vLLM 作为强基线。

### 问：什么时候选 llama.cpp

**答：** 桌面、边缘、离线、Apple Silicon、CPU 或消费级硬件场景通常优先评估它，尤其需要 GGUF 低比特量化和 CPU+GPU 混合卸载时。

### 问：TGI 现在还能用于新项目吗

**答：** 技术上仍可使用，但官方仓库已声明维护模式。新项目应把新模型、硬件、安全修复和长期维护风险纳入决策，并优先比较官方建议的活跃替代方案。

### 问：怎样公平比较推理框架

**答：** 固定模型、精度、硬件、模板、解码参数和请求分布，先过正确性门槛，再比较 TTFT、TPOT、P99、吞吐、显存、缓存收益、错误率和长时间稳定性。

### 延伸问题

- Continuous Batching 为什么能提高吞吐？
- Chunked Prefill 如何影响 TTFT 与 TPOT？
- PagedAttention 与 Prefix Cache 有何区别？
- 多 LoRA 服务会影响哪些缓存和调度？
- Prefill/Decode 分离适合什么规模？
- 如何为 MoE 选择 Expert Parallel 配置？
- 什么情况下托管 API 的总成本低于自建？
