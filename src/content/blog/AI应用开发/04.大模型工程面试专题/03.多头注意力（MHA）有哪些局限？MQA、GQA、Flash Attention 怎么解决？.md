想象一家图书馆有 32 位研究员。每位研究员都有自己的一套索引卡和原文复印件，这对应 MHA：每个 Query 头都有独立的 Key 和 Value。

后来发现，真正拖慢夜间值班的不是“研究员不会提问”，而是要反复搬运 32 套资料。于是可以让所有研究员共享一套资料，即 MQA；也可以每 4 位共享一套，即 GQA。

还有另一种改进：资料仍然不变，但管理员重新安排搬运路线，让资料尽量留在快速的小仓库里，少往返慢速大仓库。这更像 FlashAttention。

MQA/GQA 与 FlashAttention 经常一起出现，但解决的不是同一层问题。

## 本篇目标

读完本篇，你应该能够：

- 回顾 MHA 中 Query 头与 KV 头的关系；
- 推导 KV Cache 大小及其主要影响因素；
- 解释为什么 Decode 常受显存带宽限制；
- 区分 MHA、MQA 和 GQA 的结构差异；
- 解释 FlashAttention 的分块、在线 Softmax 与 IO-aware 思路；
- 明确 FlashAttention 通常不改变精确注意力的数学结果；
- 根据质量、延迟、吞吐和模型兼容性做选型；
- 回答“是否能叠加”“为什么不总用 MQA”等追问。

先记住：

> MQA/GQA 通过减少 KV 头数量，主要降低 KV Cache 容量和 Decode 时的 KV 读取量；FlashAttention 通过重新组织精确注意力计算，主要减少显存层级间的数据搬运和中间张量存储。两者可以组合使用。

## 60 秒回答

MHA 为每个注意力头分别生成 Q、K、V，表达能力强，但自回归推理需要为历史 Token 保存每层所有 KV 头。长上下文、大批量服务时，KV Cache 会占用大量显存，Decode 每一步还要读取历史 K/V，容易受显存带宽限制。

MQA 保留多个 Query 头，但所有 Query 头共享一组 K/V，能显著减少 KV Cache 和读取带宽，不过共享过强可能影响部分任务质量。GQA 把 Query 头分组，每组共享一组 K/V，在 MHA 的质量与 MQA 的效率之间折中。MHA、MQA、GQA 改的是模型参数结构和 KV 头数量，通常要与 checkpoint 配置匹配。

FlashAttention 则是精确注意力的 IO-aware 实现。它按块读取 Q/K/V，在片上存储器中计算，并用在线 Softmax 保持数值等价，避免把完整 $L\times L$ 注意力矩阵反复写入和读出高带宽显存。它通常降低内存占用并提升速度，但并没有把标准注意力的理论计算量自动从二次变成线性。

工程上可同时使用 GQA 和 FlashAttention；最终收益取决于硬件、序列长度、批量、精度、内核支持和服务阶段，应以真实负载基准测试为准。

## 详细解析

### 1. 先回顾多头注意力

设隐藏状态：

$$
X\in\mathbb{R}^{B\times L\times d_{model}}
$$

其中：

- $B$：批大小；
- $L$：序列长度；
- $d_{model}$：模型隐藏维度。

如果有 $h_q$ 个 Query 头，每个头维度为 $d_h$，通常有：

$$
d_{model}=h_qd_h
$$

第 $i$ 个头计算：

$$
Q_i=XW_i^Q,\qquad K_i=XW_i^K,\qquad V_i=XW_i^V
$$

$$
H_i=\operatorname{softmax}\left(\frac{Q_iK_i^\top}{\sqrt{d_h}}+M\right)V_i
$$

最后拼接所有头：

$$
Y=\operatorname{Concat}(H_1,\ldots,H_{h_q})W_O
$$

在标准 MHA 中，Query 头数与 KV 头数相同：

$$
h_{kv}=h_q
$$

### 2. 训练注意力与生成注意力的负载不同

#### Prefill 阶段

Prompt 已全部给定，可以并行处理多个 Token：

```text
输入 2,000 个 Prompt Token
→ 一次处理整段
→ 计算并保存每层 K/V
```

Prefill 通常有较大的矩阵乘法，更容易利用 GPU 计算能力。

#### Decode 阶段

每次只生成一个或少量新 Token：

```text
新 Query
→ 读取此前全部 K/V
→ 算注意力
→ 生成一个 Token
→ 把新 K/V 追加到缓存
```

Decode 的矩阵较小，却要反复读取长历史 KV，因此常更受显存带宽和调度影响。这里描述的是常见现象，具体瓶颈要用 profiler 判断。

### 3. KV Cache 是什么

在第 $t$ 步生成时，过去 Token 的 K/V 不会因新 Token 到来而改变。若每一步都重新计算历史 K/V，会浪费计算。

KV Cache 保存每层历史 Token 的 Key 和 Value：

```text
第 1 步：缓存 K1, V1
第 2 步：复用 K1, V1，只新增 K2, V2
第 3 步：复用 K1..K2, V1..V2，只新增 K3, V3
```

它用显存换计算，是自回归推理的关键优化。

### 4. KV Cache 大小怎样估算

忽略分页、对齐、元数据和并行切分，单请求 KV Cache 可近似为：

$$
M_{KV}
\approx
2\times N\times L\times h_{kv}\times d_h\times s
$$

其中：

- $2$：同时保存 Key 和 Value；
- $N$：Transformer 层数；
- $L$：缓存 Token 数；
- $h_{kv}$：KV 头数量；
- $d_h$：每个头的维度；
- $s$：每个元素的字节数，例如 FP16/BF16 通常为 2 字节。

若批量有 $B$ 条等长请求：

$$
M_{KV,batch}
\approx
2BNLh_{kv}d_hs
$$

示例：

- 32 层；
- 上下文 4,096 Token；
- 32 个 KV 头；
- 头维度 128；
- BF16，每元素 2 字节。

则单请求约为：

$$
2\times32\times4096\times32\times128\times2
=2,147,483,648\ \text{bytes}
$$

约 2 GiB。真实引擎还会受缓存块、张量并行、量化和未用容量影响。

如果 KV 头从 32 减到 8，理论主体容量约降到四分之一；减到 1，则约降到三十二分之一。

### 5. MHA 的优势与局限

#### 优势

- 每个 Query 头有独立 K/V 投影；
- 表示自由度高；
- 训练生态成熟；
- 大量已有 checkpoint 使用这种结构。

#### 局限

- KV Cache 随 KV 头数线性增长；
- Decode 每步要读取更多历史 K/V；
- 长上下文和高并发下容易挤压可批处理请求数；
- 服务成本可能受显存容量与带宽限制。

MHA 的问题不是“计算结果错误”，而是规模化推理效率。

### 6. MQA 怎样共享 K/V

Multi-Query Attention 保留多个 Query 头：

$$
h_q>1
$$

但只使用一组 K/V：

$$
h_{kv}=1
$$

可以表示为：

$$
Q_i=XW_i^Q,\qquad K=XW^K,\qquad V=XW^V
$$

每个 Query 头都读取同一 K/V：

$$
H_i=\operatorname{Attention}(Q_i,K,V)
$$

这样大幅减少缓存和读取量。

#### 为什么还保留多个 Query 头

不同 Query 头仍可以从不同子空间提出不同“问题”。共享的是被查询的 K/V 表示，不是把所有注意力头完全合成一个头。

#### 代价

所有 Query 头只能读取同一组 K/V，表示瓶颈更强。论文和不同模型实践显示，它可以保持较好质量，但不能保证对所有规模、任务和训练配方都无损。

### 7. GQA 怎样折中

Grouped-Query Attention 把 Query 头分组。设：

$$
1<h_{kv}<h_q
$$

每组 Query 头共享一组 K/V。

例如：

```text
32 个 Query 头
8 个 KV 头
每 4 个 Query 头共享 1 个 KV 头
```

关系可以写为：

$$
g=\frac{h_q}{h_{kv}}
$$

其中 $g$ 是每组 Query 头数，通常要求头数可以整除。

极端情况：

```text
h_kv = h_q  → MHA
h_kv = 1    → MQA
1 < h_kv < h_q → GQA
```

GQA 让架构设计者在缓存效率和表示能力之间选择中间点。

### 8. MHA、MQA、GQA 对比

| 方式 | Query 头 | KV 头 | KV Cache | 典型取舍 |
| --- | ---: | ---: | ---: | --- |
| MHA | $h_q$ | $h_q$ | 最大 | 表示自由度高，缓存成本高 |
| MQA | $h_q$ | 1 | 最小 | 解码高效，共享最强 |
| GQA | $h_q$ | $1<h_{kv}<h_q$ | 中间 | 质量与效率折中 |

不能仅看名称判断性能。隐藏维度、层数、上下文、缓存精度、内核和硬件都会影响结果。

### 9. 能否把 MHA checkpoint 直接改成 GQA

不能只改配置中的头数。K/V 投影权重形状和语义发生了变化。

GQA 论文研究了把已有 MHA checkpoint 的 KV 头分组并做继续训练的 uptraining 方法。工程上需要：

- 明确权重聚合或初始化方式；
- 继续训练让模型适应共享；
- 重新做质量、安全和性能评测；
- 确保推理引擎支持新结构。

这不是无训练、零风险的开关。

### 10. FlashAttention 要解决什么

普通注意力的直观实现会产生大中间矩阵：

```text
QKᵀ → 写回显存
读取分数 → Softmax → 再写回显存
读取概率与 V → 输出
```

GPU 有不同层级存储：

- HBM/显存：容量大，但相对慢；
- SRAM/片上存储：容量小，但更快。

很多注意力实现的时间不只是花在乘加运算上，还花在不同存储层级之间搬数据。FlashAttention 从 IO-aware 角度重排计算。

### 11. 分块计算为什么有用

FlashAttention 把 Q、K、V 切成块：

```text
取一小块 Q
  ↓
依次读取 K/V 小块
  ↓
在片上计算局部 QKᵀ、Softmax 统计量和输出
  ↓
合并局部结果
```

它避免把完整注意力分数和概率矩阵长期存到 HBM。这样可以把更多运算留在快速片上存储，并减少 HBM 读写。

### 12. 分块后 Softmax 怎样保持正确

Softmax 看起来必须一次知道整行所有分数：

$$
\operatorname{softmax}(x_i)=\frac{e^{x_i}}{\sum_j e^{x_j}}
$$

为数值稳定，通常减去行最大值 $m$：

$$
\operatorname{softmax}(x_i)
=
\frac{e^{x_i-m}}{\sum_j e^{x_j-m}}
$$

FlashAttention 使用在线 Softmax，在处理每个块时维护：

- 当前最大值 $m$；
- 当前指数和 $l$；
- 当前加权输出累积。

新块到来时更新最大值，并按新旧最大值差重新缩放之前的累积量。概念更新为：

$$
m_{new}=\max(m_{old},m_{block})
$$

$$
l_{new}
=
e^{m_{old}-m_{new}}l_{old}
+
e^{m_{block}-m_{new}}l_{block}
$$

其中：

- $m_{old}$：已处理块的最大分数；
- $m_{block}$：新块最大分数；
- $l_{old}$：已处理块的稳定指数和；
- $l_{block}$：新块的稳定指数和。

因此不需要保存整行分数，也能得到与标准注意力相同的数学结果，允许正常浮点舍入差异。

### 13. FlashAttention 没有做什么

它通常没有：

- 把注意力变成稀疏近似；
- 删除历史 Token；
- 减少模型 KV 头数；
- 自动扩大模型训练过的上下文能力；
- 把理论乘加量从 $O(L^2)$ 变成 $O(L)$；
- 对任何硬件和形状都保证同样倍数加速。

它的关键收益来自更少的中间存储和 HBM IO。

### 14. FlashAttention、FlashAttention-2、FlashAttention-3

后续版本继续优化工作划分、并行度和针对新硬件特性的利用。面试中不必死背每个内核细节，但要理解：

- 它们是一系列高效精确注意力算法和实现；
- 支持的 GPU、数据类型、头维度、Mask 和反向传播特性会随版本变化；
- 选择时应以当前框架和官方兼容性文档为准。

不要把“FlashAttention 3”理解为第三种模型注意力语义。

### 15. 两类优化为什么可以叠加

GQA 改的是张量结构：

```text
Query 头多，KV 头少
```

FlashAttention 改的是计算调度：

```text
同一注意力结果，用更少 HBM 往返完成
```

因此常见组合是：

```text
GQA 模型
  +
FlashAttention 兼容内核
  +
分页 KV Cache/连续批处理服务引擎
```

每层优化各自负责不同瓶颈。

### 16. 与 PagedAttention 的区别

PagedAttention 主要解决服务中 KV Cache 的内存管理问题：将缓存分成块，减少内存碎片并支持请求动态增长和共享。

可以这样区分：

| 技术 | 主要层次 | 主要目标 |
| --- | --- | --- |
| MQA/GQA | 模型架构 | 减少 KV 头与缓存主体容量 |
| FlashAttention | 算子/内核 | 减少注意力计算的 IO 和中间存储 |
| PagedAttention | 服务内存管理 | 更灵活地分配和复用 KV Cache 块 |

它们也可以共同存在。

## 工程实践与边界

### 1. 先测 Prefill 与 Decode，不只看总延迟

建议分别记录：

- TTFT：首 Token 延迟，常受排队和 Prefill 影响；
- TPOT：每个输出 Token 的时间，常反映 Decode 效率；
- Prompt 吞吐；
- Generation 吞吐；
- P50/P95/P99 延迟；
- 每请求 KV Cache 占用；
- 并发达到显存上限时的吞吐。

一个优化可能改善长 Prompt，却不明显改善短 Prompt；也可能提高吞吐但使尾延迟变差。

### 2. 用目标负载做基准测试

基准至少覆盖：

- 真实 Prompt 长度分布；
- 真实输出长度分布；
- 不同并发和批量；
- 实际 GPU 型号；
- 实际精度和量化方案；
- 相同模型与输出约束；
- 冷启动与稳定运行阶段。

只用单条短请求测得的速度不能代表生产服务。

### 3. 检查模型配置与引擎支持

关键字段包括：

- `num_attention_heads`；
- `num_key_value_heads`；
- `head_dim`；
- 位置编码类型；
- 缓存数据类型；
- 滑动窗口设置；
- 张量并行切分方式。

错误解释 KV 头数可能导致权重加载失败或隐蔽的输出错误。

### 4. KV Cache 量化是另一条优化线

把缓存从 FP16/BF16 改为 FP8 或更低精度可以继续节省容量和带宽，但可能引入量化误差，并要求内核支持。它与 GQA 不是同一件事，需要独立评测质量和速度。

### 5. 短序列未必明显受益

内核启动、数据布局转换和编译成本在短序列下可能占比更高。FlashAttention 的收益通常随序列、批量和硬件变化，不能引用论文某个峰值作为本系统承诺。

### 6. 数值等价不等于逐位相同

不同分块和归约顺序会改变浮点加法顺序，因此可能存在末位差异。验证时应使用适当容差，并评估生成分布和任务质量，而不是要求所有 logits 位级相同。

### 7. 质量评测不能省略

架构转换、缓存量化或更换内核后都应检查：

- 困惑度或验证损失；
- 任务准确率；
- 长上下文检索；
- 代码和数学任务；
- 安全与拒答；
- 不同长度下的回归。

## 常见误区

### 误区一：MQA 只有一个注意力头

错误。它可以有多个 Query 头，只是共享一组 K/V。

### 误区二：GQA 和 MQA 是 FlashAttention 的不同版本

错误。前两者是 KV 头共享结构，FlashAttention 是注意力计算实现。

### 误区三：FlashAttention 把复杂度变成线性

错误。对标准稠密精确注意力，它主要减少 IO 和中间内存，核心乘加量仍随 $L^2$ 增长。

### 误区四：使用 FlashAttention 就不需要 KV Cache

错误。自回归 Decode 仍需复用历史 K/V；两者解决不同问题。

### 误区五：把 MHA 配置改成 GQA 就能直接加载

错误。K/V 权重形状和训练分布改变，需要明确转换、继续训练和回归评测。

### 误区六：KV Cache 大小只由上下文长度决定

错误。还与层数、KV 头数、头维度、精度、批量和并行切分有关。

### 误区七：MQA 必定不损失质量

错误。它是更强共享约束，效果取决于模型、训练配方和任务。GQA 正是常见中间折中。

### 误区八：论文加速比可以直接当生产加速比

错误。硬件、内核版本、输入长度、批量、精度和基线实现都会改变结果。

## 面试追问

### 问：MHA、GQA、MQA 的关系怎么一句话说明

答：它们可以用 KV 头数统一表示：$h_{kv}=h_q$ 是 MHA，$h_{kv}=1$ 是 MQA，介于两者之间是 GQA。

### 问：为什么减少 KV 头能提高 Decode 吞吐

答：每个生成步骤都要读取历史 K/V。KV 头更少意味着缓存容量和读取字节数更少，能缓解显存容量与带宽压力，并容纳更大批量。

### 问：GQA 为什么可能比 MQA 质量更好

答：GQA 保留多组 K/V，给不同 Query 组更丰富的表示，比所有 Query 共享唯一 K/V 的约束弱。但是否更好必须由训练和评测验证。

### 问：FlashAttention 为什么能省显存

答：它按块计算并在线维护 Softmax 统计量，不需要把完整 $L\times L$ 分数和概率矩阵保存到 HBM，从而减少中间张量和数据搬运。

### 延伸追问速答

| 追问 | 回答要点 |
| --- | --- |
| FlashAttention 是近似算法吗 | 经典方案计算精确标准注意力，允许正常浮点误差，不靠稀疏或低秩近似换速度 |
| 为什么 Decode 容易 memory-bound | 新 Query 少、算术密度较低，却要读取全部历史 K/V；是否确实受带宽限制仍应用 profiler 确认 |
| 怎样算每 Token 的 KV Cache | 主体约为 $2Nh_{kv}d_hs$ 字节，再乘缓存 Token 数和批量；真实值还受分页、对齐和并行影响 |
| 已用 GQA 还需要 FlashAttention 吗 | 两者可叠加：GQA 减少 KV 数据量，FlashAttention 优化内核 IO；实际收益取决于负载和硬件 |
