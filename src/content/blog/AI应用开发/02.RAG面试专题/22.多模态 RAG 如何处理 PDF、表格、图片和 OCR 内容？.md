## 本篇目标

学完这一篇，你应该能够：

- 解释为什么处理 PDF 不能只做“复制全部文字”；
- 区分文本 PDF、扫描 PDF 和混合 PDF；
- 理解 OCR、版面分析、阅读顺序、Bounding Box 和多模态检索；
- 为段落、表格、图片、图表和公式选择不同的表示与检索方式；
- 设计保留页码、坐标、来源、置信度和权限的完整证据链；
- 说清多模态 RAG 的常见失败、评测指标、安全边界与成本取舍。

## 60 秒回答

多模态 RAG 的难点不是把 PDF 粗暴转成字符串，而是保留内容结构和跨模态关系。入库时我会先判断页面有没有可靠文本层，再做版面分析，识别标题、段落、多栏阅读顺序、表格、图片、图表、公式和坐标；对缺失或不可靠文本层的页面、图片或局部区域再使用 OCR。每个元素保留 document_id、页码、Bounding Box、父级章节、解析置信度、版本和 ACL。

不同内容采用不同表示：段落用于文本和向量检索；表格保留行列、表头、单位和合并单元格，并同时生成可检索文本；图片和图表保留原始区域、标题、图例、邻近文字，可增加视觉向量或视觉模型描述。在线用文本、视觉、结构化字段和元数据多路召回，命中后把原始页面裁剪或结构化表格交给模型，并把回答引用到页码和坐标。

评测要分开测阅读顺序、OCR 字词错误、表格结构、视觉检索、证据定位和端到端问答。OCR 置信度高不等于数字和单位可靠；精确筛选与算术优先交给程序。图片和 OCR 内容同样是不可信输入，ACL 和提示注入防护不能因为换了模态而放松。

## 详细解析

### 先说人话：PDF 更像一张排版好的纸，不像一段天然有顺序的文字

人看到一页文件，会自然理解：

- 哪个是标题；
- 左栏读完再读右栏；
- 表头属于下面的单元格；
- 图片说明属于哪张图；
- 脚注不是正文；
- 红色箭头指向哪个数字。

计算机拿到 PDF 时，可能只看到一堆散落在坐标上的字符和图形。
如果直接按文件内部顺序提取文字，双栏内容可能交叉，表格可能被打平成无法理解的数字串，图片中的信息则可能全部丢失。

### 贯穿案例：从差旅制度 PDF 回答表格与流程图问题

一份 30 页的差旅制度包含：

- 第 3 页：适用员工范围；
- 第 8 页：各城市住宿标准表；
- 第 9 页：表格跨页续表；
- 第 14 页：报销审批流程图；
- 第 20 页：一张低清扫描的发票样例；
- 页脚：版本号和生效日期。

用户可能问：

1. “深圳普通员工每晚能报多少？”——需要理解表格；
2. “超过标准后谁审批？”——需要理解流程图；
3. “这张发票哪个字段是发票号码？”——需要图像和 OCR；
4. “第 8 页的金额适用于高管吗？”——需要结合表格与适用范围。

如果系统只有扁平文本，这些问题很容易答错。

### 先认识七个基础术语

| 术语 | 通俗解释 |
| --- | --- |
| OCR | Optical Character Recognition，把图片中的文字识别出来 |
| 版面分析（Layout Analysis） | 识别标题、段落、表格、图片等区域 |
| 阅读顺序（Reading Order） | 判断内容应该按什么先后顺序阅读 |
| Bounding Box | 元素在页面上的矩形坐标 |
| 模态（Modality） | 文本、图片、音频等不同信息形式 |
| VLM | Vision-Language Model，同时理解视觉和语言的模型 |
| Grounding | 让结论能够定位到具体原始证据 |

多模态 RAG 的“多模态”不是只指图片。版面位置、表格结构、视觉关系和文字都可以成为证据。

### 第一步：先判断 PDF 属于哪一类

| PDF 类型 | 特征 | 处理重点 |
| --- | --- | --- |
| 文本型 PDF | 可以选中文字 | 提取文本层，但仍要分析版面和顺序 |
| 扫描型 PDF | 每页基本是一张图片 | 图像预处理、OCR 和版面分析 |
| 混合型 PDF | 一部分有文本，一部分是扫描或图片 | 逐页或逐区域判断，避免重复识别 |

常见错误是对所有 PDF 都做 OCR：

- 文本型 PDF 原本字符准确，OCR 反而可能引入错误；
- OCR 文字与原文本层同时保留，会产生重复；
- 计算和入库时间增加；
- 字体、公式和小数点可能识别错误。

更合理的是先检测文本层质量，再决定页面或区域是否需要 OCR。

### 第二步：做版面分析，而不是只提取字符

版面解析器通常输出元素树：

~~~text
文档
├─ 第 8 页
│  ├─ 标题：住宿费标准
│  ├─ 段落：适用说明
│  ├─ 表格
│  │  ├─ 表头
│  │  ├─ 行
│  │  └─ 单元格
│  └─ 页脚：版本 2025.1
└─ 第 9 页
   ├─ 续表标记
   └─ 表格续页
~~~

每个元素应尽量保留：

- 元素类型；
- 文本或结构化内容；
- 页码；
- Bounding Box；
- 父级标题和章节；
- 阅读顺序；
- OCR 或解析置信度；
- 与图片、标题、脚注的关系；
- 源文件、版本和 ACL。

这些信息既用于检索，也用于最终引用和人工核验。

### 第三步：处理阅读顺序

PDF 内部字符顺序不一定等于人类阅读顺序。常见复杂情况：

- 双栏或三栏；
- 侧边栏和批注；
- 标题跨多栏；
- 浮动图片；
- 页眉页脚重复；
- 竖排或旋转文字；
- 跨页段落。

如果阅读顺序错了，可能得到：

~~~text
深圳 500 北京 450 普通员工 高管 800 600
~~~

模型很难判断哪个数字对应哪个城市和职级。
因此要把“阅读顺序正确率”作为解析层独立评测，而不是等最终问答出错才发现。

### 第四步：OCR 不只是识别率

扫描件通常先经过：

1. 旋转与方向检测；
2. 去噪、裁边和透视校正；
3. 文本区域检测；
4. 字符识别；
5. 版面和阅读顺序恢复；
6. 语言、数字和单位后处理。

#### 为什么少量 OCR 错误也可能很严重

普通句子错一个字，语义可能还能理解；金额、日期和编号错一个字符，就可能改变结论：

- 500 被识别成 900；
- 2025 被识别成 202S；
- 0.5% 丢掉小数点；
- “不得”被识别成“得”；
- 发票号码中的 0 和 O 混淆。

所以除了整体字符错误率，还要对金额、日期、否定词、编号和单位设置专项规则或人工复核。

#### 置信度怎么用

OCR 置信度可以用于：

- 低置信区域重新识别；
- 使用另一引擎交叉验证；
- 标记答案不确定；
- 高风险字段转人工；
- 在排序时降低不可靠证据权重。

它只是模型信号，不应被当作事实正确的证明。

### 第五步：表格要保存结构

#### 为什么表格不能只转成一段文字

原表：

| 城市 | 普通员工 | 高管 | 单位 |
| --- | ---: | ---: | --- |
| 深圳 | 500 | 800 | 元/晚 |
| 成都 | 400 | 600 | 元/晚 |

粗暴提取后可能变成：

~~~text
城市 普通员工 高管 单位 深圳 500 800 元/晚 成都 400 600 元/晚
~~~

当表格很大或跨页时，行列对应关系会更难恢复。

#### 推荐保存两种表示

1. **结构化表示**：行、列、表头、单元格、合并关系、单位；
2. **可检索文本表示**：把每一行展开成明确语句。

例如：

~~~text
表：住宿费标准
城市=深圳；员工类型=普通员工；金额=500；单位=元/晚
来源：第 8 页，2025 版差旅制度
~~~

结构化表示用于准确筛选和计算，文本表示用于关键词和语义召回。

#### 跨页表格

跨页表格要判断：

- 下一页是否是同一张表；
- 表头是否重复；
- 列宽和列语义是否一致；
- 单位是否在上一页；
- 行是否在分页处被拆开。

合并后仍要保存每一行的原始页码，不能让引用只指向合成结果。

#### 计算交给程序

如果用户问“出差 12 晚最多报销多少”，先从结构化表格取出 500，再由程序执行 500 × 12。
让模型对截图心算会增加数值错误和不可复现性。

### 第六步：图片、图表和流程图怎么表示

不同图像承载的信息不同：

| 类型 | 主要信息 | 推荐表示 |
| --- | --- | --- |
| 普通照片 | 物体和场景 | 原图、视觉向量、简短描述 |
| 图表 | 坐标轴、图例、趋势、数值 | 原图、标题、图例、邻近文字、可选结构化数据 |
| 流程图 | 节点、箭头和顺序 | 原图、节点/边结构、文本描述 |
| 截图 | 界面文字和位置 | OCR、区域坐标、原图 |
| 发票/表单 | 字段与值 | Key-Value 结构、OCR、原始区域 |

可以生成图片描述帮助文本检索，但描述不是原图的等价替代：

- 描述模型会遗漏细节；
- 细小数值和颜色编码可能丢失；
- 描述本身也可能幻觉；
- 用户需要看到可核验的原始证据。

因此应保留原始区域及其来源。

### 第七步：如何建立多模态索引

常见有三条检索通道。

#### 文本通道

索引：

- 原生文本层；
- OCR 文本；
- 图片标题和邻近段落；
- 表格展开文本；
- 经过审核的图像描述。

适合用户以文字提问的大多数场景。

#### 视觉通道

对页面或区域生成视觉向量，用于：

- 图表外观；
- 版面相似性；
- 难以可靠 OCR 的页面；
- 用户上传图片查相似文档。

ColPali 等研究探索从文档页面图像生成多向量并做 late interaction，减少对传统扁平文本表示的依赖。但索引大小、显存、语言、领域和延迟需要在自己的数据上评估。

#### 结构化通道

把表格、表单和业务字段写入关系数据库、搜索引擎字段或分析引擎。
适合精确条件、范围过滤、聚合和计算。

#### 多路融合

~~~text
用户问题
→ 判断是否涉及文本、表格或图像
→ 文本检索 + 视觉检索 + 结构化查询
→ ACL 过滤
→ 融合与重排
→ 返回文本块、表格区域和页面裁剪
~~~

不能简单比较不同通道的原始分数，因为分数分布不同。可以用 RRF、归一化或学习排序融合，并通过评测集调优。

### 第八步：上下文如何交给生成模型

命中证据后，不要只发送一段失去来源的文字。可以附带：

- 文档标题和版本；
- 页码；
- 章节；
- 元素类型；
- Bounding Box；
- OCR 置信度；
- 结构化表格行；
- 原始页面或区域裁剪；
- 与相邻元素的关系。

对于支持视觉输入的模型，可同时提供页面裁剪和结构化文本。这样模型既能看原始布局，也能利用可搜索、可计算的结构。

### 第九步：引用要能回到原始页面

理想引用不是“来自差旅制度.pdf”，而是：

~~~text
2025 版差旅制度，第 8 页，
表格“住宿费标准”，深圳行、普通员工列，
页面坐标 x1,y1,x2,y2
~~~

前端可高亮对应区域，让用户直接核验。
如果用户没有原文权限，引用预览也不能绕过 ACL。

### 第十步：多模态 RAG 怎么评测

#### 解析层

| 内容 | 指标或检查 |
| --- | --- |
| OCR | 字符错误率、词错误率、关键字段准确率 |
| 版面 | 元素检测、类型、Bounding Box 匹配 |
| 阅读顺序 | 段落顺序准确率、人工抽检 |
| 表格 | 行列、单元格、表头和结构识别 |
| 图片 | 图片与标题、邻近文字关联准确率 |
| 来源 | 页码、坐标、版本和 ACL 完整率 |

#### 检索层

- 文本问题能否找到表格或图片；
- 图片问题能否找到正确页面；
- 多模态候选的 Recall@K 和 nDCG；
- 正确区域是否进入最终上下文；
- 不同页面类型和扫描质量切片的表现。

#### 生成层

- 答案事实正确性；
- 对视觉与结构化证据的忠实度；
- 数字、单位、方向和颜色理解；
- 引用能否定位到支持结论的区域；
- 低置信证据时是否表达不确定或拒答。

#### 端到端

用真实文档覆盖：

- 双栏；
- 旋转页；
- 低清扫描；
- 手写批注；
- 合并单元格；
- 跨页表格；
- 图表和流程图；
- 文本中嵌入恶意指令。

## 工程实践与边界

### 为每一阶段保存版本

- 原始文件哈希；
- 页面渲染器版本；
- OCR 引擎和语言包版本；
- 版面模型版本；
- 表格解析器版本；
- 图片描述或 VLM 版本；
- 文本和视觉 Embedding 版本；
- 索引与发布版本。

解析器升级后，不要原地覆盖全部结果。先重建影子索引，用解析回归集和问答集比较。

### 常见失败与降级

| 失败 | 风险 | 可选处理 |
| --- | --- | --- |
| OCR 服务失败 | 扫描页无文本 | 保留旧版本，重试或转人工 |
| 阅读顺序不确定 | 上下文混乱 | 使用版面块独立检索，附原页 |
| 表格结构低置信 | 数字对应错误 | 返回原表裁剪，不做自动计算 |
| 视觉模型超时 | 图片未理解 | 降级到 OCR、标题和邻近文字 |
| 跨页表合并失败 | 条件遗漏 | 按页展示并提示用户核验 |
| 原文权限变化 | 预览泄露 | 立即失效裁剪和缓存 |

### 安全边界

- OCR 文本、图片中的文字和二维码内容都是不可信输入；
- 文档里写“忽略系统指令”不能改变系统权限；
- 文件上传要做类型、大小、恶意文件和来源校验；
- OCR、解析和 VLM 服务只接触任务所需数据；
- 临时页面图和裁剪遵守加密、访问和保留要求；
- ACL 在召回、预览、缓存和模型输入前强制执行；
- 图片描述不能成为高风险事实的唯一依据。

### 成本与性能

多模态摄取比纯文本多出页面渲染、OCR、版面、视觉模型和图像存储成本。可以按文档路由：

~~~text
可靠文本页 → 文本提取 + 版面分析
扫描页 → OCR + 版面分析
含关键表格 → 表格结构解析
含业务关键图片 → 视觉索引或 VLM
普通装饰图 → 不进入高成本视觉链路
~~~

在线也可以先走文本通道，只有问题涉及“图中、截图、趋势、流程”或文本证据不足时，才启用视觉检索。

### 技术边界

- OCR 和视觉模型都有错误，不能把输出视为权威原文；
- VLM 上下文支持页面，不代表它能稳定读准所有细小数字；
- 图片描述会压缩信息；
- 视觉多向量索引通常占用更多存储和计算；
- 表格结构恢复对复杂跨页、嵌套表仍有挑战；
- 高风险数字和业务动作需要结构化校验或人工复核。

## 常见误区

- **“PDF 能复制文字，就不需要版面解析”**：阅读顺序、表格结构和图片关系仍可能丢失。
- **“所有 PDF 先 OCR 最省事”**：会引入重复、误识别和额外成本。
- **“OCR 置信度高就代表内容正确”**：少量数字、单位和否定词错误也可能改变结论。
- **“把表格转成 Markdown 就结束了”**：还要保留单元格、表头、单位、跨页关系和来源坐标。
- **“生成图片描述后可以丢掉原图”**：描述会遗漏细节，也可能产生错误。
- **“多模态模型可以代替结构化计算”**：精确筛选、统计和算术优先使用程序。
- **“引用文档名就足够”**：用户需要定位页码和具体区域。
- **“图片不会提示注入”**：恶意指令可以藏在截图、OCR 和二维码中。

## 面试追问

### 1. 文本型 PDF 还需要 OCR 吗？

通常先使用可靠文本层并做版面分析。只有文本层缺失、乱码或局部是图片时，才对对应页或区域 OCR，避免重复文本和识别误差。

### 2. 跨页表格怎么处理？

利用重复表头、列位置、标题、续表标记和上下文判断是否属于同一表；合并结构后仍保留每行原始页码和坐标，低置信时不自动计算。

### 3. 表格应该怎么切块？

小表可以整体保留；大表按逻辑行组切分，同时为每个块重复表名、表头、单位和适用条件。结构化存储与展开文本通常并存。

### 4. 图片描述和视觉 Embedding 有什么区别？

图片描述把视觉信息压缩成可检索文字，便于文本搜索但可能遗漏；视觉 Embedding 直接编码视觉特征，适合版面和图像相似检索。二者可以互补，原始区域仍应保留。

### 5. 如何评估多模态解析？

分别测 OCR 字词、关键字段、阅读顺序、版面区域、表格结构和图片关联，再测检索与最终问答。只看端到端答案无法定位是解析还是生成出错。

### 6. 为什么保留 Bounding Box？

它让系统把答案引用到页面具体区域，支持高亮核验，也能关联图片、标题、脚注和表格单元格。

### 7. 视觉检索什么时候值得使用？

当关键信息依赖版面、图形、图表或低质量 OCR，且文本检索在目标评测集上确实漏召回时。需要比较质量收益与索引大小、延迟和计算成本。

### 8. 怎样防止多模态链路泄露数据？

ACL 覆盖原文件、解析结果、页面图片、裁剪、缓存和模型输入；第三方 OCR/VLM 只接收授权数据；临时文件按保留策略删除；引用和预览再次鉴权。
