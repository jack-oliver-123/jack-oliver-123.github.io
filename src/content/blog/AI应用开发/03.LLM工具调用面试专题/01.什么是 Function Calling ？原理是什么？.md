想象你给一位前台助理发消息：
> “帮我查一下上海明天会不会下雨。”
助理本人并不知道实时天气。
但他知道公司里有一个“天气查询系统”，也知道应该填写城市和日期。
于是他写下一张办事单：
~~~json
{
  "name": "get_weather",
  "arguments": {
    "city": "上海",
    "date": "2026-07-16"
  }
}
~~~
这张办事单很像 Function Calling 的输出。
关键点是：**写办事单的人没有亲自查询天气。**
真正访问天气服务的是拿到办事单的应用程序。
Function Calling 最容易被误解的地方，就在这里。

## 本篇目标
读完本篇，你应该能够：
- 用一句话解释 Function Calling；
- 说清模型、宿主应用和真实工具的职责；
- 描述一次完整工具调用的闭环；
- 理解工具名称、描述、参数 Schema 和调用 ID 的作用；
- 区分“结构正确”“业务正确”和“执行成功”；
- 解释为什么模型不能直接获得数据库或网络权限；
- 写出一个最小工具调用执行循环；
- 回答超时、重试、幂等、授权和提示注入等追问。
先记住本篇核心结论：
> Function Calling 是模型表达工具调用意图的一种结构化机制，不是模型亲自执行外部函数。

## 60 秒回答
Function Calling，也常被称为 Tool Calling，是模型与宿主应用之间约定的一种结构化输出机制。
宿主应用先把可用工具的名称、用途和参数 Schema 提供给模型。
模型根据用户问题决定直接回答、拒绝回答，还是返回一个或多个工具调用意图。
调用意图通常包含工具名、结构化参数和调用标识。
模型生成这些内容，并不意味着它已经访问了网络、数据库或操作系统。
宿主应用收到调用后，还要完成参数解析、Schema 校验、业务规则校验、身份授权、用户确认、真实执行、超时控制和审计。
执行完成后，宿主把工具结果与原调用关联，再交回模型。
模型才能基于真实结果生成最终答案，或者继续发起下一次调用。
因此它解决的是“模型怎样告诉应用自己想用哪个工具”，不自动解决工具发现、部署、安全执行、幂等和业务事务。

## 详细解析

### 1. 先分清三个参与者
一次工具调用至少有三个参与者。

| 参与者 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| 大模型 | 理解问题，选择工具，生成参数，阅读结果 | 通常不直接访问外部系统 |
| 宿主应用 | 管理会话、校验、授权、执行、回传、停止循环 | 不能盲目信任模型输出 |
| 真实工具 | 查询或修改外部世界 | 不负责理解完整自然语言对话 |
“宿主应用”可以是聊天后端、Agent Runtime、桌面客户端或工作流服务。
“真实工具”可以是本地函数、数据库查询、REST API、消息队列任务或 MCP Tool。
模型看到的是工具的**说明书**。
宿主应用持有的是工具的**执行权**。
这两件事必须分开。

### 2. 为什么需要结构化调用
如果没有 Function Calling，我们也可以让模型输出一句话：
> “请调用天气接口，城市是上海，日期是明天。”
但应用很难稳定解析这句话。
模型可能换一种说法，漏掉字段，或者在参数旁边输出解释。
结构化调用把自由文本收敛为更容易验证的对象：
~~~json
{
  "name": "get_weather",
  "arguments": {
    "city": "上海",
    "date": "2026-07-16"
  }
}
~~~
它主要带来三项收益：
1. 工具名称可以精确匹配注册表；
2. 参数可以按 Schema 自动校验；
3. 调用和结果可以通过 ID 关联。
但“更容易验证”不等于“可以直接信任”。
模型输出仍然属于外部输入。

### 3. 工具定义包含什么
不同模型供应商的 API 字段并不完全一致。
不要把某一家接口的字段名当成跨厂商标准。
从概念上，一项工具通常至少包含：

| 部分 | 作用 | 示例 |
| --- | --- | --- |
| name | 稳定标识工具 | get_weather |
| description | 告诉模型何时使用 | 查询指定城市某天的天气 |
| input schema | 约束参数结构 | city、date 必填 |
| output contract | 约定返回结构 | temperature、condition |
| policy metadata | 供宿主执行策略使用 | 只读、超时 2 秒 |
其中输入 Schema 常借用 JSON Schema 表达。
例如：
~~~json
{
  "type": "object",
  "properties": {
    "city": {
      "type": "string",
      "description": "城市名称，例如上海"
    },
    "date": {
      "type": "string",
      "format": "date",
      "description": "ISO 8601 日期，例如 2026-07-16"
    }
  },
  "required": ["city", "date"],
  "additionalProperties": false
}
~~~
Schema 告诉模型和校验器“参数长什么样”。
描述告诉模型“什么时候应该使用”。
两者都重要。
只有字段而没有清晰描述，模型可能选错工具。
只有描述而没有结构约束，参数容易漂移。

### 4. 一次完整调用的七步闭环
完整流程可以拆成七步。
~~~text
1. 宿主注册工具
        ↓
2. 用户提出请求
        ↓
3. 宿主把消息和候选工具交给模型
        ↓
4. 模型返回文本或工具调用意图
        ↓
5. 宿主校验、授权并执行
        ↓
6. 宿主把工具结果关联到调用
        ↓
7. 模型生成最终回答或继续调用
~~~
下面逐步解释。
#### 第一步：注册工具
宿主维护工具注册表。
注册表至少记录工具名、描述、参数 Schema 和执行函数。
高风险系统还会记录：
- 工具所有者；
- 风险等级；
- 所需权限；
- 默认超时；
- 是否允许重试；
- 是否需要用户确认；
- 审计字段；
- 版本号。
#### 第二步：用户提出请求
用户说：
> “查一下上海明天会不会下雨。”
“明天”是相对时间。
宿主最好把当前日期、时区等必要上下文提供给模型。
否则模型可能生成错误日期。
#### 第三步：发送候选工具
宿主不一定要把全部工具都发给模型。
工具很多时，可以先按任务、用户权限和场景筛选。
候选集合越相关，模型越容易选对，也能减少上下文成本。
#### 第四步：模型生成调用意图
模型可能有四种合理行为：
1. 直接回答；
2. 提出澄清问题；
3. 拒绝不允许的请求；
4. 生成一个或多个工具调用。
“没有调用工具”不一定是失败。
如果用户只是问“天气预报是什么意思”，直接回答反而正确。
如果缺少城市，先追问也比猜测更好。
#### 第五步：宿主执行
这是最重要的责任边界。
宿主至少要做：
1. 解析参数；
2. 验证工具是否在允许名单中；
3. 执行 JSON Schema 校验；
4. 执行业务规则校验；
5. 检查当前用户权限；
6. 必要时请求用户确认；
7. 设置超时、取消和重试策略；
8. 调用真实工具；
9. 记录审计日志。
模型不能替代这些步骤。
#### 第六步：回传工具结果
宿主把工具结果与原始调用 ID 关联。
调用 ID 的作用是避免并行调用时对错结果。
例如同时查询上海和北京天气时，两个结果必须回到各自的调用。
工具结果也应该经过裁剪和脱敏。
不要把访问令牌、内部堆栈、数据库连接串或整份超大响应直接放回上下文。
#### 第七步：继续生成
模型读到真实结果后，可以：
- 生成最终自然语言答案；
- 再调用另一个工具；
- 发现结果不足并追问用户；
- 说明工具失败，给出可恢复建议。
因此 Function Calling 往往不是一次请求，而是一个受控循环。

### 5. 最小伪代码
下面是跨供应商的概念伪代码。
字段名不对应任何一家 API 的完整实现。
~~~typescript
const tools = registry.listAllowedTools(user);
let messages = [userMessage];

for (let step = 0; step < MAX_STEPS; step++) {
  const modelOutput = await model.generate({
    messages,
    tools: tools.map(toModelDefinition),
  });

  if (modelOutput.type === "final_text") {
    return modelOutput.text;
  }

  for (const proposedCall of modelOutput.toolCalls) {
    const tool = registry.get(proposedCall.name);

    assertAllowed(user, tool);
    const args = validateSchema(tool.inputSchema, proposedCall.arguments);
    validateBusinessRules(user, tool, args);

    if (tool.requiresConfirmation) {
      await askUserToConfirm(tool, args);
    }

    const result = await executeWithPolicy(tool, args, {
      timeoutMs: tool.timeoutMs,
      idempotencyKey: proposedCall.id,
    });

    messages.push(
      toToolResultMessage(proposedCall.id, sanitize(result))
    );
  }
}

throw new Error("工具调用步数超过上限");
~~~
这段代码最值得注意的不是语法。
而是模型输出与执行函数之间有明确的验证层。

### 6. Schema 正确不等于业务正确
假设订票工具要求：
~~~json
{
  "flightId": "MU0001",
  "price": 1800
}
~~~
JSON 可以解析。
字段类型也正确。
但用户的差旅上限是 1500 元。
这就是“结构正确，但业务不合法”。
还可能出现：
- 用户没有替他人订票的权限；
- 航班已售罄；
- 报价已经过期；
- 员工信息不完整；
- 同一申请已创建；
- 当前操作需要主管审批。
JSON Schema 无法表达全部动态业务状态。
因此，服务端业务校验仍是必要步骤。

### 7. 工具调用不是普通文本生成
普通文本回答错误，可能只影响阅读体验。
工具调用错误，可能改变外部世界。

| 输出类型 | 可能后果 |
| --- | --- |
| 错误解释 | 用户得到错误信息 |
| 错误查询参数 | 返回无关数据 |
| 错误发信参数 | 邮件发给错误的人 |
| 错误预订参数 | 产生费用 |
| 错误删除参数 | 数据丢失 |
因此风险越高，自动化程度越不能只由模型置信度决定。
高风险动作通常需要确定性校验、用户确认或人工审批。

### 8. Function Calling 与 JSON Mode 的区别
JSON Mode 通常只要求模型输出合法 JSON。
它不一定表达“要调用外部工具”。
例如从简历中提取姓名、技能和工作年限，可以只需要结构化输出。
Function Calling 除了结构化参数，还包含工具选择和调用语义。
但具体产品如何定义两者，仍需查看供应商文档。
不要用一个厂商的实现推断所有平台。

### 9. 单次调用与多步调用
单次调用：
~~~text
用户问天气
→ get_weather
→ 返回天气
→ 最终回答
~~~
多步调用：
~~~text
用户申请差旅
→ get_employee_profile
→ get_travel_policy
→ search_flights
→ 用户确认
→ create_travel_request
→ book_flight
→ 最终回答
~~~
多步调用需要额外控制：
- 最大步骤数；
- 总超时；
- 总费用；
- 重复调用检测；
- 中间状态；
- 失败恢复；
- 用户中断；
- 结果压缩。
Function Calling 只提供调用表达。
这些编排能力仍由 Host 或 Agent Runtime 实现。

## 工程实践与边界

### 1. 把模型输出当作不可信输入
无论模型多强，都要假设它可能：
- 生成不存在的工具名；
- 漏掉必填参数；
- 生成额外字段；
- 把字符串写成数字；
- 使用过期资源 ID；
- 复述用户给出的恶意参数；
- 重复调用同一写操作。
因此执行前要经过统一入口。
不要在不同工具中各写一套零散校验。

### 2. 区分只读工具和写工具
只读工具通常可以：
- 设置较短超时；
- 在明确条件下有限重试；
- 使用缓存；
- 降级到备用数据源。
写工具通常需要：
- 幂等键；
- 用户确认；
- 资源版本校验；
- 审批；
- 更严格审计；
- 不确定结果后的状态查询；
- 补偿流程。
“调用失败”不等于“操作没有发生”。
网络在服务端成功后断开，客户端可能只看到超时。
此时盲目重试可能重复扣款或重复下单。

### 3. 设计明确错误类型
不要只向模型返回：
~~~json
{"error": "failed"}
~~~
可以设计有限、稳定的错误码：
~~~json
{
  "ok": false,
  "error": {
    "code": "MISSING_REQUIRED_FIELD",
    "message": "缺少乘客证件信息",
    "retryable": false,
    "userActionRequired": true
  }
}
~~~
模型据此才能决定：
- 修改参数重试；
- 向用户追问；
- 停止调用；
- 转人工处理。
错误消息不要泄露内部堆栈和密钥。

### 4. 控制工具结果
工具结果可能来自网页、邮件、文档或第三方 API。
这些内容可能包含：
- 提示注入；
- 恶意链接；
- 个人隐私；
- 访问令牌；
- 超长文本；
- 不可信 HTML。
Host 应做：
- 来源标记；
- 长度限制；
- 字段白名单；
- HTML 清洗；
- 密钥脱敏；
- 权限过滤；
- 将数据与指令分离。
工具结果是数据，不应自动获得系统指令的优先级。

### 5. 建立停止条件
至少设置：
- 最大模型轮数；
- 最大工具调用次数；
- 单工具超时；
- 全链路超时；
- 最大并行度；
- 总成本预算；
- 重复调用阈值。
否则模型可能在两个失败工具之间循环。

### 6. 做完整可观测性
一次调用链至少关联：
- trace ID；
- 会话 ID；
- 模型响应 ID；
- tool call ID；
- 工具名与版本；
- 参数摘要；
- 用户与权限决策；
- 开始时间和结束时间；
- 状态码；
- 重试次数；
- 幂等键；
- 最终结果。
敏感字段只能记录脱敏值或哈希。

### 7. 供应商字段不是通用协议
不同模型 API 对以下能力的表达可能不同：
- 工具定义字段；
- 强制或禁止工具选择；
- 并行调用；
- 严格结构化输出；
- 工具结果消息；
- 调用 ID；
- 流式事件。
工程上应该在自己的领域模型中抽象。
然后为不同供应商编写适配层。
但不要为了“统一”抹掉供应商真实能力差异。

## 常见误区

### 误区一：Function Calling 就是模型执行函数
错误。
模型通常只输出调用意图。
执行由宿主代码完成。

### 误区二：返回合法 JSON 就可以执行
错误。
还要校验工具白名单、字段语义、用户权限、资源状态和副作用风险。

### 误区三：工具描述只给人看
错误。
模型会依据名称、描述和参数说明选择工具。
含糊描述会直接降低选择准确率。

### 误区四：Schema 越复杂越好
错误。
过深嵌套、模糊联合类型和大量可选字段会增加模型填参难度。
优先设计小而清晰的接口。

### 误区五：模型没调工具就是失败
错误。
直接回答、拒绝或追问可能是正确决策。
评测必须包含“不该调用”的样本。

### 误区六：调用失败后让模型无限重试
错误。
重试要由错误类型、工具幂等性和预算共同决定。

### 误区七：工具返回值天然可信
错误。
外部内容可能错误、恶意或越权。

### 误区八：Function Calling 自动等于 Agent
错误。
单次结构化调用只是能力。
Agent 还需要状态、循环、规划、停止条件和治理。

## 面试追问

### 追问一：为什么要把工具结果再交给模型
因为真实执行发生在模型外部。
模型只有收到结果，才能基于最新事实继续回答。
并行调用时还要用调用 ID 正确关联。

### 追问二：严格 Schema 后还要服务端校验吗
必须要。
Schema 主要约束结构。
金额上限、库存状态、用户归属和审批规则属于动态业务语义。

### 追问三：什么时候应该先追问用户
当缺失信息会显著改变结果或产生副作用时。
例如订票缺少日期、乘客或预算，不能靠猜测补齐。

### 追问四：怎样避免重复下单
为业务动作设计稳定幂等键。
服务端以幂等键去重。
超时后先查询订单状态，不直接再次创建。

### 追问五：模型生成了不存在的工具怎么办
执行层拒绝。
返回稳定的“未知工具”错误。
同时记录评测样本，检查工具命名、描述、候选集合和模型表现。

### 追问六：工具调用可以并行吗
互相独立的只读调用通常可以并行。
存在数据依赖或副作用的调用不能盲目并行。
例如先拿到差旅政策，才能判断航班是否合规。

### 追问七：Function Calling 和 MCP 是什么关系
Function Calling 解决模型怎样表达调用意图。
MCP 解决 Host 怎样连接和调用外部能力提供方。
Host 可以把 MCP Tools 映射给模型，再用 Function Calling 驱动。

### 可背诵完整版
> Function Calling 是模型和宿主应用之间的结构化调用机制。应用把工具名称、描述和参数 Schema 提供给模型，模型决定直接回答还是生成工具名与参数。模型通常不执行真实函数，也不会自动获得网络或数据库权限。宿主收到调用后，要做白名单、Schema、业务规则和权限校验，必要时请求用户确认，再以超时、幂等和审计策略执行工具。结果通过调用 ID 回传给模型，模型再生成最终答案或继续调用。生产上还要防止重复写入、无限循环、敏感信息泄露和工具结果中的提示注入。
