假设你正在读一本侦探小说。

每看一个新字，如果都要从第一页重新读到当前位置，速度会越来越慢。

更合理的方式是把已经读过的关键信息记在便签上，新一页只查便签和新内容。

自回归大模型的 KV Cache 就像这组便签。

它保存历史 Token 在每层注意力中的 Key 和 Value，生成下一个 Token 时不必重新计算全部历史表示。

如果很多用户都使用相同的长系统提示，还可以把这段前缀已经算好的 KV 块跨请求复用。

这就是 Prefix Cache 或某些 API 所称的 Prompt Caching。

两者相关，但不是同一个范围的缓存。

## 本篇目标

读完本篇，你应该能够：

- 从注意力公式解释 K、V 为什么可以缓存；
- 区分 Prefill 与 Decode；
- 说清请求内 KV Cache 的读写过程；
- 推导 KV Cache 的近似显存公式；
- 解释 MHA、MQA、GQA 为什么占用不同；
- 区分 KV Cache、Prefix Cache、供应商 Prompt Caching 和语义缓存；
- 说明跨请求缓存的精确命中条件；
- 分析分页、淘汰、并发、租户隔离和版本失效；
- 判断缓存到底改善 TTFT、TPOT 还是成本。

先记住：

> 请求内 KV Cache 避免每个 Decode 步骤重算历史；跨请求 Prefix/Prompt Cache 避免相同前缀重复 Prefill。

## 60 秒回答

在自回归 Transformer 中，每层注意力会为 Token 计算 Query、Key 和 Value。生成新 Token 时，历史 Token 的 Key 和 Value 在模型参数不变的前提下不会变化，所以可以缓存。下一步只计算新 Token 的 Q、K、V，用新 Query 与历史及当前 Key 做注意力，再读取对应 Value。这样避免每一步对完整前缀重复投影，代价是 KV Cache 随层数、序列长度、KV 头数、头维度和数据精度线性占用显存。

Prompt Caching 需要先区分语义。自托管引擎的 Prefix Cache 通常把已计算前缀的 KV 块按 Token 与执行上下文做哈希，跨请求精确复用；云 API 的 Prompt Caching 是供应商产品能力，命中长度、有效期、计费和隔离规则必须看具体文档。它们主要减少重复前缀的 Prefill，因此改善首 Token 延迟或输入计算成本，不会免除未命中后缀和新输出 Token 的 Decode。

缓存键和隔离边界至少要考虑模型版本、Tokenizer、聊天模板、位置、LoRA、租户和权限。语义相似但 Token 不同通常不能直接复用底层 KV；KV 隐状态也可能携带敏感信息，不能因为它不可读就忽略隔离和清理。

## 详细解析

### 1. 先回顾自注意力

一层注意力的核心公式是：

$$
\operatorname{Attention}(Q,K,V)
=
\operatorname{softmax}
\left(
\frac{QK^\top}{\sqrt{d_k}}
+M
\right)V
$$

逐个解释：

- $Q$ 是 Query，表示当前 Token 想查什么；
- $K$ 是 Key，表示每个 Token 可按什么特征被匹配；
- $V$ 是 Value，表示命中后实际取回什么信息；
- $d_k$ 是单个注意力头的 Key 维度；
- $M$ 是掩码；
- 因果掩码禁止当前位置看到未来 Token；
- softmax 把匹配分数变成注意力权重。

通俗类比：

- Query 是检索问题；
- Key 是索引；
- Value 是索引指向的内容。

### 2. Q、K、V 从哪里来

输入隐藏状态 $X$ 通过线性投影得到：

$$
Q=XW_Q
$$

$$
K=XW_K
$$

$$
V=XW_V
$$

其中：

- $X$ 是当前层输入；
- $W_Q,W_K,W_V$ 是训练得到的参数矩阵；
- Q、K、V 不是数据库中的键值；
- 它们是模型内部张量。

每一层都有自己的投影和缓存。

不能只缓存最后一层。

### 3. 没有 KV Cache 会怎样

假设 prompt 有 1000 个 Token，准备生成第一个新 Token。

模型对整个前缀做一次前向。

生成第二个 Token 时，上下文变为 1001 个 Token。

如果不缓存，就要再次计算前面 1000 个 Token 的 K、V。

随后是 1002、1003……

大量历史投影被重复执行。

### 4. 为什么历史 K、V 可以复用

在标准因果 Transformer 推理中：

- 模型参数固定；
- 历史 Token 固定；
- 历史位置固定；
- 历史 Token 不能看到未来；
- 因此历史 Token 的层内 K、V 不因新 Token 到来而改变。

新 Token 的 Query 仍需要读取所有历史 Key。

所以缓存减少了重复计算，却没有让上下文长度从注意力读取中消失。

### 5. Prefill 与 Decode

推理常分为两个阶段。

#### Prefill

一次处理输入 prompt：

~~~text
输入：N 个 prompt Token
输出：首个生成位置的 logits + N 个 Token 的各层 KV
~~~

Prefill 通常：

- 并行处理多个输入 Token；
- 计算密集；
- 长 prompt 会明显影响首 Token 延迟；
- 产生初始 KV Cache。

#### Decode

之后逐步生成：

~~~text
每一步输入：通常 1 个新 Token
读取：全部历史 KV
追加：新 Token 的 K、V
输出：下一个 Token logits
~~~

Decode 通常：

- 串行依赖上一 Token；
- 更受显存带宽和 KV 读取影响；
- 决定每 Token 延迟；
- 缓存随输出继续增长。

### 6. 请求内 KV Cache 流程

以 prompt “北京天气”为例：

~~~text
Prefill：
北京天气 → 计算各层 K/V → 写入缓存

Decode 第 1 步：
新 Token A → 计算 Q_A/K_A/V_A
Q_A 读取 [K_北京 ... K_A]
写入 K_A/V_A

Decode 第 2 步：
新 Token B → 计算 Q_B/K_B/V_B
Q_B 读取 [K_北京 ... K_A, K_B]
写入 K_B/V_B
~~~

历史 Query 通常不必缓存。

因为下一步只需要新 Token 的 Query 去查询全部 Key。

### 7. KV Cache 容量公式

对于一个序列，近似容量为：

$$
M_{\text{KV}}
\approx
2\times L\times S\times H_{kv}\times D\times B
$$

逐个解释：

- $2$：Key 与 Value 两份张量；
- $L$：Transformer 层数；
- $S$：已缓存 Token 数；
- $H_{kv}$：KV 头数；
- $D$：每个头的维度；
- $B$：每个元素的字节数。

如果还有 batch 或并发序列数 $N$：

$$
M_{\text{total}}
\approx
N\times2LSH_{kv}DB
$$

这是理论近似。

真实引擎还有块对齐、元数据、并行分片和碎片开销。

### 8. 手算一个容量例子

假设：

~~~text
层数 L       = 32
序列长度 S   = 4096
KV 头数 Hkv  = 8
头维度 D     = 128
数据类型     = FP16，即 B=2 字节
~~~

代入：

$$
2\times32\times4096\times8\times128\times2
$$

得到：

$$
536{,}870{,}912\ \text{bytes}
$$

约为 512 MiB。

这只是一个序列的近似 KV 容量。

并发 20 条相似长度请求，理论上就可能接近 10 GiB。

### 9. 为什么用 KV 头数而不是 Query 头数

多头注意力有三种常见形态：

| 结构 | Query 头 | KV 头 | KV Cache 特点 |
| --- | ---: | ---: | --- |
| MHA | 多个 | 与 Query 头相同 | 最大 |
| MQA | 多个 | 通常 1 个共享 | 显著减小 |
| GQA | 多个 | 若干组共享 | 介于两者之间 |

容量保存的是 K 和 V。

所以公式使用 $H_{kv}$，不是 $H_q$。

这也是 MQA/GQA 能提高解码吞吐的重要原因之一。

### 10. KV Cache 没有消除全部注意力成本

有缓存后，新 Query 仍要与历史 Key 计算匹配并读取历史 Value。

随着上下文增长：

- 每步读取的 KV 增加；
- 显存带宽压力增加；
- 单 Token Decode 可能变慢；
- 缓存容量线性增长。

所以“用了 KV Cache，生成复杂度就是常数”不准确。

它消除的是历史 K/V 投影的重复计算。

### 11. Dynamic、Static 和滑动窗口 Cache

推理库可能提供多种缓存实现。

#### Dynamic Cache

- 随 Token 增长；
- 易于使用；
- 动态形状可能不利于某些编译优化。

#### Static Cache

- 预先分配最大容量；
- 形状固定，可能利于编译；
- 未使用空间也被预留；
- 最大长度预测错误会浪费或溢出。

#### Sliding Window Cache

- 只保留模型允许窗口内的 KV；
- 容量达到窗口后不再线性增长；
- 前提是模型注意力结构支持；
- 不能擅自丢弃需要全局注意力的历史。

### 12. 什么是 PagedAttention

传统连续内存分配会遇到：

- 请求长度不可预测；
- 预留过多产生内部碎片；
- 不同请求结束时间不同；
- 扩容或复制困难。

PagedAttention 把 KV Cache 分成固定大小的逻辑块和物理块，借鉴虚拟内存分页思想。

~~~text
序列逻辑块 0 ─→ 物理块 7
序列逻辑块 1 ─→ 物理块 2
序列逻辑块 2 ─→ 物理块 9
~~~

逻辑上连续，不要求物理显存连续。

主要收益是更灵活地管理显存、减少碎片并支持块共享。

它不会让 KV 本身变成零成本。

### 13. 什么是 Prefix Cache

考虑许多请求共享系统提示：

~~~text
[同一段 8000 Token 系统说明] + 用户问题 A
[同一段 8000 Token 系统说明] + 用户问题 B
[同一段 8000 Token 系统说明] + 用户问题 C
~~~

如果每次都 Prefill 公共前缀，会重复计算。

Prefix Cache 保存公共前缀对应的 KV 块。

后续请求命中时，从最长可复用前缀之后继续计算。

### 14. Prefix Cache 与请求内 KV Cache 的区别

| 维度 | 请求内 KV Cache | 跨请求 Prefix Cache |
| --- | --- | --- |
| 复用范围 | 同一次生成 | 多个请求 |
| 主要阶段 | Decode | Prefill |
| 命中依据 | 已生成历史 | 兼容的 Token 前缀 |
| 生命周期 | 通常随请求结束释放 | 可跨请求保留并淘汰 |
| 主要收益 | 避免每步重算历史 | 避免重复前缀 Prefill |

Prefix Cache 建立在 KV Cache 之上，但多了跨请求索引、共享和淘汰。

### 15. vLLM 的块哈希直觉

以块为单位，可构造类似键：

~~~text
hash(
  parent_block_hash,
  current_block_token_ids,
  model_context,
  tenant_salt
)
~~~

父块哈希把当前块与此前完整前缀关联起来。

因此相同 Token 块出现在不同前缀中，不会被错误当成同一状态。

真实实现字段以当前官方文档为准。

### 16. 为什么通常要求 Token 前缀一致

KV 是模型对精确 Token 序列的计算结果。

下面两句话语义相近：

~~~text
请简洁回答。
回答要简洁。
~~~

但 Token 序列不同，位置也可能不同。

底层 KV 通常不能直接互换。

因此：

$$
\text{语义相似}\not\Rightarrow\text{KV 可复用}
$$

### 17. 部分前缀命中

请求不必整段完全相同。

例如：

~~~text
公共系统提示 + 公共文档 + 问题 A
公共系统提示 + 公共文档 + 问题 B
~~~

缓存可命中公共系统提示和公共文档对应的完整块。

问题部分重新计算。

块大小会影响命中粒度：

- 块太大，尾部少量差异可能浪费复用；
- 块太小，索引和元数据开销增加。

### 18. Prompt Caching 不是单一标准

“Prompt Caching”可能指：

1. 推理引擎自动复用 Token 前缀 KV；
2. API 供应商自动识别公共前缀；
3. API 用户显式标记缓存断点；
4. 应用缓存模型最终回答；
5. 应用按语义相似度复用回答。

这些机制的命中、有效期、计费和风险完全不同。

讨论前必须先定义语义。

### 19. 四类缓存对照

| 机制 | 缓存内容 | 命中依据 | 节省什么 |
| --- | --- | --- | --- |
| 请求内 KV Cache | 当前序列各层 K/V | 同一生成历史 | Decode 重复投影 |
| Prefix Cache | 前缀 KV 块 | 精确兼容 Token 前缀 | 重复 Prefill |
| Provider Prompt Cache | 供应商定义 | 产品规则 | 延迟或输入计费，依文档 |
| Semantic Cache | 最终回答或中间结果 | 相似度 + 策略 | 整次模型调用 |

语义缓存风险更高。

相似问题不一定有相同答案，尤其涉及用户、时间、权限和实时数据。

### 20. Prompt Caching 主要改善什么

如果命中长公共前缀，通常改善：

- Prefill 计算量；
- Time To First Token，TTFT；
- 某些供应商的缓存输入计费；
- GPU Prefill 压力。

通常不会直接减少：

- 未命中后缀计算；
- 新输出 Token 数；
- 每个输出 Token 的 Decode；
- 模型权重占用。

### 21. 一个延迟拆分

端到端延迟可粗略表示为：

$$
T_{\text{total}}
=
T_{\text{queue}}
+T_{\text{prefill}}
+N_{\text{out}}\times T_{\text{decode-step}}
+T_{\text{network}}
$$

Prefix Cache 主要降低 $T_{\text{prefill}}$。

如果 prompt 很短、输出很长，整体收益可能有限。

如果公共 prompt 很长、输出很短，收益可能明显。

### 22. 哪些上下文会让缓存失效

常见因素：

- Token ID 不同；
- 模型权重或版本不同；
- Tokenizer 不同；
- chat template 不同；
- LoRA adapter 不同；
- 位置编码状态不同；
- 多模态输入哈希不同；
- 精度或量化路径不兼容；
- 租户或权限边界不同；
- 引擎重启或缓存淘汰。

缓存键不能只包含 prompt 原始字符串。

### 23. 位置为什么重要

同一个 Token 出现在不同位置，位置编码不同。

它的 K/V 也可能不同。

所以不能把任意中间片段仅按文本哈希后搬到另一个位置。

Prefix Cache 从序列起点沿共同前缀复用，更容易保持位置一致。

### 24. 淘汰策略

缓存空间有限，常见策略包括：

- LRU：淘汰最近最少使用；
- TTL：超过时间失效；
- 按块引用计数；
- 按大小或成本加权；
- 租户配额；
- 高水位主动回收。

目标不是单纯追求命中率。

还要防止热点租户挤占全部显存。

## 工程实践与边界

### 1. 容量规划不能只看最大窗口

不能简单计算：

~~~text
模型最大上下文 × 最大并发
~~~

真实规划应使用线上分布：

- 输入长度分位数；
- 输出长度分位数；
- 峰值并发；
- KV 数据类型；
- GQA/MQA 配置；
- 张量并行分片；
- 块大小和碎片；
- prefix 命中率；
- 抢占与调度策略。

### 2. 一个容量监控面板

建议监控：

- GPU KV Cache 使用率；
- 可用 KV 块数；
- 活跃序列数；
- 被抢占或重计算次数；
- 请求排队时间；
- Prefix Cache 命中 Token 数；
- 命中率与节省 Prefill Token；
- 缓存淘汰率；
- OOM 和拒绝请求数。

只有命中率没有节省 Token 数，可能误导。

命中大量短前缀价值未必高。

### 3. 设计容易命中的 prompt

一般把稳定内容放前面，动态内容放后面：

~~~text
稳定系统说明
稳定工具定义
稳定少样本示例
稳定长文档
动态用户信息
动态问题
~~~

但安全上下文和权限不能为了命中而错误共享。

缓存优化必须服从语义与隔离。

### 4. 不要在公共前缀中放随机值

下列字段放在最前面会破坏精确前缀：

- 每次变化的时间戳；
- 随机请求 ID；
- 用户昵称；
- 动态排序工具列表；
- 无关追踪信息。

如果业务允许，可把动态信息移到公共前缀之后。

### 5. 聊天模板要版本化

以下细节都会改变 Token：

- 角色前缀；
- 换行；
- BOS/EOS；
- 工具消息序列化；
- JSON 字段顺序；
- 空格与 Unicode 规范化。

模板升级后应主动使不兼容缓存失效。

### 6. 多租户隔离

至少考虑：

- 缓存命名空间包含租户；
- 敏感租户禁止跨请求共享；
- 权限不同的上下文不能共享；
- 对哈希碰撞采取可靠处理；
- 记录缓存复用审计；
- 请求删除时考虑缓存生命周期；
- 对共享显存做侧信道风险评估。

KV 是张量，不代表没有敏感信息。

### 7. LoRA 与 Prefix Cache

同一基础模型加载不同 LoRA 时，层输出可能不同。

因此缓存键或隔离必须包含 adapter 身份与版本。

否则可能把 A 客户定制模型的 KV 复用于 B 客户。

这既会产生错误，也可能造成数据泄露。

### 8. Cache Salting

一些引擎支持把额外 salt 加入块哈希。

作用是：

- 阻止不同信任边界意外共享；
- 降低通过已知前缀探测其他租户缓存命中的风险。

salt 不是加密，也不能代替访问控制。

### 9. 量化 KV Cache

将 KV 从 FP16/BF16 压缩到更低精度可能显著减小容量。

代价包括：

- 量化和反量化开销；
- 精度损失；
- 内核与硬件支持差异；
- 长上下文误差累积；
- 与模型结构的兼容性。

必须在目标长度和任务上测质量与吞吐。

### 10. Prefix Cache 的收益评估

建议做对照实验：

~~~text
缓存关闭：记录 TTFT、吞吐、GPU 利用率
缓存开启：使用相同请求分布重测
~~~

并按公共前缀长度分桶：

- 0–512 Token；
- 513–2048 Token；
- 2049–8192 Token；
- 更长前缀。

这样才能知道收益来自哪里。

### 11. 缓存预热

热点系统提示可在流量到来前预热。

但需考虑：

- 预热会占显存；
- 热点可能预测错误；
- 模型发布后旧缓存失效；
- 多副本需要分别预热；
- 自动扩缩容的新实例仍是冷缓存。

预热是否划算取决于命中收益和驻留成本。

### 12. 供应商 Prompt Caching 检查表

使用云 API 前确认：

- 是否自动启用或需显式标记；
- 最小可缓存 Token 数；
- 精确前缀要求；
- TTL 与延长规则；
- 写入和读取计费；
- 缓存是否跨区域；
- 数据保留和隐私条款；
- 使用量字段如何报告命中；
- 模型版本变化如何失效。

这些不是行业统一常量。

### 13. 语义缓存的额外防线

如果应用按向量相似度复用最终答案，缓存键还应包含：

- 用户或权限范围；
- 数据版本；
- 时间有效性；
- 地区与语言；
- 工具参数；
- 模型和 prompt 版本；
- 业务实体；
- 相似度阈值。

涉及账户余额、医疗、法律和实时价格时，应非常谨慎或禁用。

### 14. 失效比写入更难

以下变化应触发失效或新命名空间：

- 模型发布；
- LoRA 发布；
- 系统 prompt 更新；
- 工具 Schema 更新；
- 权限策略更新；
- 知识文档更新；
- Tokenizer 或模板更新。

如果无法证明兼容，就按不兼容处理。

### 15. 缓存边界

缓存无法解决：

- 模型权重加载过慢；
- 新输出 Decode 很长；
- prompt 本身不共享；
- 模型事实错误；
- 工具延迟；
- 请求排队；
- 跨区域网络延迟。

优化前先用性能剖析找到真正瓶颈。

## 常见误区

- **认为 KV Cache 缓存 Token 文本**：它保存各层 Key/Value 张量，不只是字符串。
- **认为 Q、K、V 都必须缓存**：标准自回归 Decode 主要缓存历史 K、V，新一步只需要新 Query。
- **用 Query 头数计算 GQA 容量**：容量取决于 KV 头数。
- **认为 KV Cache 让每步复杂度完全不随长度增长**：新 Query 仍需读取并关注历史 KV。
- **把 Prefix Cache 当语义缓存**：底层复用通常要求精确兼容的 Token 前缀。
- **认为 Prefix Cache 加速所有 Decode Token**：它主要跳过重复前缀 Prefill。
- **把供应商 Prompt Caching 当统一标准**：命中、TTL、计费和隔离均依产品。
- **只看请求命中率**：还要看命中 Token 数和实际节省的 Prefill 时间。
- **忽略聊天模板**：一个换行或特殊 Token 都可能破坏命中。
- **不同 LoRA 共用缓存**：模型计算不同，结果不兼容。
- **认为哈希天然解决隔离**：仍需租户命名空间、权限和侧信道防护。
- **容量耗尽后才处理**：应在高水位前调度、抢占或拒绝，避免 OOM。

## 面试追问

### 问：为什么缓存 K 和 V，不缓存历史 Q

**答：** 生成下一 Token 时，只需要当前新 Token 的 Query 去匹配全部历史 Key，再聚合历史 Value。历史 Query 已经完成自己的注意力计算，下一步通常不会再使用。

### 问：KV Cache 的显存为什么随长度线性增长

**答：** 每新增一个 Token，每层都要追加该 Token 的 Key 与 Value。层数、KV 头数、头维度和元素字节固定时，容量与 Token 数成正比。

### 问：MHA、MQA、GQA 谁的 KV Cache 更小

**答：** 在 Query 头数相同的条件下，MQA 通常只有一组 KV 头，最小；GQA 由若干 Query 头共享一组 KV，居中；MHA 的 KV 头通常与 Query 头相同，最大。

### 问：Prefix Cache 命中后还要算什么

**答：** 仍要计算未命中的输入后缀，并逐 Token 执行所有新输出的 Decode。命中只复用兼容前缀已有的 KV 块。
