## 本篇目标

学完这一篇，你应该能够：

1. 用一句话说明 MCP 与 Agent Skill 不在同一层。
2. 解释 MCP 为什么是运行时协议，Agent Skill 为什么更像任务知识包。
3. 对比二者的参与者、载体、发现、加载、执行和生命周期。
4. 说明一个 Skill 怎样指导 Agent 使用一个或多个 MCP Server。
5. 判断某项能力应该做成 Skill、MCP Server、普通代码，还是组合实现。
6. 识别权限放大、提示注入、脚本供应链和副作用重放等风险。

## 60 秒回答

> MCP 和 Agent Skill 不是竞争方案。MCP 是 Host 内的 Client 与 MCP Server 之间的运行时协议，标准化初始化、能力协商、Tools、Resources、Prompts 等能力以及消息和传输；它主要回答“外部能力怎样被发现、连接和调用”。
>
> Agent Skills 规范则把完成某类任务所需的指令与配套资源打包成目录，至少包含 `SKILL.md`，还可以带脚本、参考资料和模板；它主要回答“拿到这些能力后，Agent 应该按什么方法完成任务”。Skill 不定义 MCP 的连接、JSON-RPC 或远程鉴权，MCP 也不会自动提供完整业务流程。
>
> 二者可以组合。例如退款 Skill 规定先核验身份、查询订单、判断政策、请求确认，再执行退款；订单查询和退款操作由 MCP Tools 提供。工程上仍由 Host 做 Skill 发现、上下文加载、工具裁剪、参数校验、授权、审批、幂等和审计。是否采用其中一个取决于问题：只传递操作知识可只用 Skill，只标准化工具接入可只用 MCP，跨工具的 Agent 任务通常适合组合。

## 详细解析

### 先说人话：一个像“操作手册”，一个像“标准接口”

继续用退款场景理解。

公司有一份退款操作手册：

```text
1. 核验用户和订单。
2. 读取对应地区的退款政策。
3. 判断是否符合资格。
4. 超过 500 元请求主管确认。
5. 创建退款。
6. 核验状态并通知用户。
```

这更像 Agent Skill。

公司还提供统一接口：

```text
get_order
get_refund_policy
create_refund
get_refund_status
```

Host 需要发现这些接口、理解参数并调用远程服务。MCP 可以标准化这部分连接。

最短记忆法是：

```text
Skill：告诉 Agent 怎样做。
MCP：让 Host 知道去哪里拿能力、怎样调用。
```

### 先明确本文中的 Agent Skill

本文的 Agent Skill 特指 Agent Skills 文件规范，而不是所有平台都可能使用的“技能”名称。

一个最小 Skill：

```text
refund-review/
└── SKILL.md
```

完整一些可以是：

```text
refund-review/
├── SKILL.md
├── scripts/
├── references/
└── assets/
```

`SKILL.md` 的 YAML frontmatter 至少包含：

- `name`；
- `description`。

正文使用 Markdown 描述任务方法、边界、验证和资源导航。

### MCP 的基本形态

MCP 的典型结构是：

```text
Agent Host
  ├── MCP Client A ── transport ── MCP Server A
  └── MCP Client B ── transport ── MCP Server B
```

Client 与 Server 使用 JSON-RPC 消息，并通过标准传输或自定义传输通信。

当前标准传输包括：

- `stdio`；
- Streamable HTTP。

Server 可以暴露：

- **Tools**：可由模型或应用调用的操作；
- **Resources**：应用可读取的上下文数据；
- **Prompts**：可发现的提示模板；
- 以及规范定义的其他能力。

### 核心对比表

| 维度 | MCP | Agent Skill |
| --- | --- | --- |
| 本质 | Client 与 Server 的运行时协议 | 指令与配套资源的目录规范 |
| 主要问题 | 怎样连接、发现和调用能力 | 怎样完成一类任务 |
| 主要参与者 | Host、Client、Server | Host、Agent、技能包 |
| 核心载体 | JSON-RPC 消息与传输 | `SKILL.md` 和目录文件 |
| 能力发现 | 初始化协商、`tools/list` 等 | `name`、`description` 等元数据 |
| 是否定义网络传输 | 是 | 否 |
| 是否定义工具调用 | 是，Tools 有协议语义 | 可指导调用，但不定义通用远程协议 |
| 是否携带脚本 | Server 内部可有实现，但协议不分发源码 | 可以在 `scripts/` 附带 |
| 是否携带知识资料 | Resources 可提供数据 | `references/` 可打包资料 |
| 生命周期 | 连接初始化、运行、关闭 | 发现、选择、加载、执行、卸载由 Host 决定 |
| 权限来源 | Host、Server、授权协议与业务系统 | Host 与执行环境，不由文本自动授予 |
| 典型更新 | Server 或协议版本更新 | 目录内容与版本更新 |

### 差异一：解决的问题不同

MCP 想减少的是 **N 对 M 集成成本**。

没有统一协议时：

```text
Host A -> 数据库插件专用接口
Host A -> 文件插件专用接口
Host B -> 数据库插件另一套接口
Host B -> 文件插件另一套接口
```

采用统一协议后，Host 与 Server 围绕相同消息和能力模型适配。

Agent Skill 想减少的是 **任务方法重复编写**。

没有 Skill 时，每次都要重新告诉 Agent：

```text
如何发布版本、如何审核合同、如何制作周报、如何排查事故……
```

打包后，同类任务可以复用说明、脚本和资料。

### 差异二：所在层次不同

可以把 Agent 应用粗略分层：

```text
任务知识层      Agent Skill
                   ↓
模型决策层      大模型 / Agent 循环
                   ↓
能力接入层      MCP Client
                   ↓
能力提供层      MCP Server / 业务系统
```

这是逻辑分层，不代表每个项目都要部署四个独立进程。

Skill 可能只是一组本地文件；MCP Server 可能是本地子进程，也可能是远程服务。

### 差异三：发现机制不同

#### MCP 的发现

MCP Client 与 Server 初始化后，根据双方声明的能力进行交互。

例如 Host 可以请求工具列表，获取：

- 工具名；
- 描述；
- 输入 Schema；
- 输出相关元数据。

Server 还可以在能力变化时发出列表变更通知，前提是协商支持。

#### Skill 的发现

Host 通常扫描配置位置或安装源，先读取 `SKILL.md` 的元数据。

根据 `name` 和 `description` 建立索引后，再按任务选择是否加载正文。

两种“发现”不能混为一谈：

```text
MCP 发现的是运行时协议能力。
Skill 发现的是可能适用于任务的知识包。
```

### 差异四：生命周期不同

MCP 生命周期由协议明确约束：

```text
建立传输
  -> initialize
  -> 协商版本与能力
  -> initialized
  -> 正常请求、响应和通知
  -> 关闭
```

Skill 没有这套网络连接生命周期。

它更可能经历：

```text
安装
  -> 验证
  -> 索引元数据
  -> 匹配任务
  -> 加载正文
  -> 按需读取资源或运行脚本
  -> 从上下文移除
```

具体步骤由 Host 实现，不应把“Skill 被激活”描述成 MCP `initialize`。

### 差异五：执行语义不同

MCP Tool 调用具有明确协议动作，例如：

```text
tools/list
tools/call
```

Server 收到调用后负责执行或转发真实操作。

Skill 正文主要是给 Agent 的指令。

即使 Skill 附带脚本，脚本也要由 Host 或运行环境显式启动。

目录本身不会自动执行。

### 差异六：知识与数据的表达方式不同

MCP Resource 可以在运行时提供数据：

```text
resource://company/refund-policy/current
```

Skill 的 `references/` 可以附带随包分发的参考资料：

```text
references/refund-policy-explanation.md
```

选择时要考虑数据新鲜度：

| 内容 | 更合适的方式 |
| --- | --- |
| 随代码版本发布、变化很少的说明 | Skill `references/` |
| 频繁更新的实时政策或数据库记录 | MCP Resource 或 Tool |
| 需要事务一致性的业务状态 | Tool/API，不要依赖静态文件 |
| 生成输出用的固定模板 | Skill `assets/` |

### 差异七：兼容性边界不同

一个 Host 支持 MCP，不代表它支持 Agent Skills 目录。

一个 Host 支持 Agent Skills，也不代表它有 MCP Client。

还要分别确认：

- 支持哪个 MCP 协议版本；
- 支持哪些 MCP 能力与传输；
- 如何发现 Skill；
- 支持哪些 frontmatter 字段；
- 能否运行 Skill 脚本；
- 文件引用规则是否一致；
- 权限模型是否兼容。

不要只看到两个“支持”标签就假设可以无缝组合。

### 二者怎样组合

#### 示例：退款审核 Skill

`SKILL.md` 可以规定：

```text
1. 从用户消息提取订单号，但不要猜测缺失值。
2. 通过订单工具读取订单。
3. 根据地区读取当前退款政策。
4. 运行确定性脚本计算最大可退金额。
5. 如果金额超过用户权限，生成审批请求。
6. 用户明确确认后才能创建退款。
7. 查询最终状态并生成通知。
```

MCP Server 可以分别提供：

```text
Order Server
  - get_order
  - get_payment

Policy Server
  - read_policy

Refund Server
  - create_refund
  - get_refund_status
```

完整执行链：

```text
用户请求
  -> Host 选择退款 Skill
  -> 加载 SKILL.md
  -> 模型按流程提出工具调用
  -> Host 校验权限和参数
  -> MCP Client 调用相应 Server
  -> Host 回传结果
  -> 模型继续流程
  -> 高风险步骤请求确认
  -> 执行并审计
```

### 谁负责什么

组合系统最怕责任模糊。

| 责任 | 推荐承担者 |
| --- | --- |
| Skill 是否适用于当前任务 | Host 路由器或模型，配合规则 |
| Skill 是否可信 | 安装与供应链治理 |
| 哪些 MCP Tools 对当前用户可见 | Host 权限过滤 |
| MCP 连接与协议消息 | MCP Client |
| Tool 输入结构校验 | Host 与 Server 双重校验 |
| 业务权限与事务规则 | Server / 下游业务系统 |
| 是否请求用户确认 | Host 策略，不仅依赖模型 |
| 工具调用是否幂等 | Host 与业务服务共同保证 |
| 日志、追踪和脱敏 | Host、Server 和网关 |

Skill 可以提醒“必须确认”，但强制确认应由代码状态机实现。

### 什么时候只用 Skill

适合：

- 写作规范；
- 代码审查清单；
- 本地已有工具足够；
- 主要需求是领域知识和流程；
- 不需要统一接入外部服务。

例如“按公司风格撰写事故复盘”可以只用 Skill 和模板。

### 什么时候只用 MCP

适合：

- 确定性应用要统一访问数据或工具；
- 固定按钮调用固定 API；
- 业务流程已在代码或工作流引擎中；
- 不需要向 Agent 注入复杂任务方法。

例如 IDE 直接通过 MCP Resource 读取项目文件，不一定需要独立 Skill。

### 什么时候组合

适合：

- 任务需要领域步骤和多个外部工具；
- 需要复用同一操作方法；
- 工具来自不同 MCP Server；
- 既要灵活判断，又要统一接入；
- 需要按需加载长资料和脚本。

### 什么时候两者都不是首选

固定、强事务、高风险的核心流程更适合业务代码或工作流引擎强制执行。

例如：

```text
资金清算
工资发放
权限授予
数据库迁移
```

Skill 可以提供操作说明，MCP 可以提供受控接口，但不能用模型的自然语言判断替代强一致性状态机。

## 工程实践与边界

### 设计组合系统的四道门

#### 第一道：来源门

只有经过批准的 Skill 和 MCP Server 才能安装或连接。

检查：

- 发布者；
- 许可证；
- 签名或哈希；
- 代码与脚本；
- 外部依赖；
- 更新渠道。

#### 第二道：发现门

发现不等于可执行。

Host 应按当前用户、租户和任务过滤：

- 可见 Skills；
- 可见 MCP Servers；
- 可见 Tools、Resources 和 Prompts。

#### 第三道：执行门

执行前进行：

```text
Schema 校验
  -> 业务校验
  -> 身份和权限校验
  -> 风险确认
  -> 幂等检查
  -> 调用
```

#### 第四道：结果门

工具结果和参考资料都视为不可信数据：

- 限制大小；
- 校验类型；
- 脱敏；
- 标记来源；
- 不把其中指令提升为系统规则；
- 只把必要内容送回模型。

### 防止权限放大

危险设计：

```text
安装了“管理员运维 Skill”
        ↓
Host 自动开放所有运维 MCP Tools
```

正确关系是：

```text
用户身份与策略决定权限
        ↓
Host 过滤可见工具
        ↓
Skill 只能在已授权集合内指导使用
```

Skill 的文字不能创造权限。

### 处理提示注入

攻击可能来自：

- 第三方 `SKILL.md`；
- `references/` 中的文档；
- MCP Resource；
- Tool 返回的网页或邮件；
- 用户输入。

防护原则：

1. 明确指令层级；
2. 把外部结果标记为数据；
3. 不让数据修改工具权限；
4. 高风险操作由代码确认；
5. 对读取和执行做最小权限；
6. 记录来源和调用链。

### 版本和兼容性治理

一次行为至少受这些版本影响：

```text
Skill 版本
MCP 协议版本
MCP Server 版本
工具 Schema 版本
Host 版本
模型与提示版本
```

升级时要进行契约测试：

- Skill 引用的工具是否仍存在；
- 工具名称和参数是否变化；
- Resource URI 是否稳定；
- 错误码是否变化；
- 高风险步骤是否仍被确认；
- 旧会话能否安全结束或迁移。

### 可观测性

建议关联：

```text
trace_id
skill_name + skill_version
mcp_server_id + protocol_version
tool_name + tool_call_id
json_rpc_id
user_id / tenant_id（脱敏或内部标识）
business_idempotency_key
policy_decision
```

这样才能回答：

- 为什么选择这个 Skill；
- 为什么暴露这个 Tool；
- 谁批准了副作用；
- 失败发生在哪一层；
- 重试是否安全。

### 分层评测

不要只看最后一句回答。

| 层 | 评测问题 |
| --- | --- |
| Skill 发现 | 是否该触发、是否误触发 |
| 指令遵循 | 关键步骤是否遗漏或越界 |
| MCP 连接 | 初始化、发现、调用是否稳定 |
| 工具选择 | 是否选对 Server 和 Tool |
| 参数质量 | 结构和业务约束是否正确 |
| 安全 | 是否越权、绕过确认或受注入影响 |
| 业务结果 | 是否满足真实业务不变量 |
| 性能 | 延迟、成本、工具调用数是否可接受 |

### 选型决策表

| 需求 | Skill | MCP | 业务代码 |
| --- | --- | --- | --- |
| 复用任务说明 | 强 | 弱 | 中 |
| 分发脚本与模板 | 强 | 弱 | 中 |
| 标准化远程工具接入 | 弱 | 强 | 中 |
| 动态列出能力 | 中 | 强 | 中 |
| 强事务与一致性 | 弱 | 只提供调用通道 | 强 |
| 灵活处理非结构化例外 | 强 | 弱 | 弱到中 |
| 服务端权限强制 | 弱 | 需结合授权 | 强 |

## 常见误区

### 误区一：MCP 和 Skill 是两种竞争的插件标准

不准确。一个主要标准化运行时连接，一个主要标准化任务知识包，可以互补。

### 误区二：Skill 可以替代工具实现

指令不能替代真实 API、数据库事务和执行器。

### 误区三：MCP 会自动编排多步业务流程

MCP 提供能力模型和消息语义，不决定退款先查订单还是先审批。

### 误区四：在 Skill 中写了工具名，Host 就能直接调用

Host 还必须连接相应 Server、完成协商、获得权限，并兼容工具 Schema。

### 误区五：MCP Resource 比 Skill 参考资料更新鲜

不一定。新鲜度取决于数据源和更新机制；只是 Resource 更适合运行时读取动态数据。

### 误区六：Skill 的 `allowed-tools` 是最终授权

不是。运行时仍要根据用户身份、租户和业务策略执行授权。

### 误区七：本地 stdio Server 没有远程安全风险

本地子进程仍可能读取文件、密钥、环境变量或访问网络，应限制启动来源和运行权限。

### 误区八：组合越多，Agent 越强

无关 Skill 和过多工具会增加上下文、冲突、误调用和攻击面，应按需选择。

## 面试追问

### 追问一：一句话说出 MCP 与 Skill 的区别

> MCP 解决外部能力怎样通过标准协议接入和调用，Agent Skill 解决 Agent 应按什么知识与流程完成一类任务。

### 追问二：一个 Skill 能调用多个 MCP Server 吗

可以。

Skill 可以指导 Agent 依次使用订单、政策、支付等不同 Server 的工具，但实际连接、权限和路由由 Host 管理。

### 追问三：一个 MCP Server 能服务多个 Skill 吗

可以。

例如订单 Server 可被退款、物流查询、客服申诉等多个 Skill 复用。Server 应保持能力职责清晰，不把某个 Skill 的整套流程硬编码进通用工具。

### 追问四：政策应该放 Skill 参考资料还是 MCP Resource

如果政策与 Skill 版本一起发布且变化慢，可以放参考资料；如果频繁更新、按地区或用户动态变化，更适合从权威系统通过 MCP Resource 或 Tool 读取。关键业务规则最终仍应由服务端校验。

### 追问五：为什么高风险流程不能只写在 Skill 中

模型可能漏步骤、误解或受注入影响。审批、额度、幂等和事务不变量必须由确定性代码强制执行，Skill 只能提供指导和解释。

### 追问六：如何处理 Skill 引用的 Tool 被 Server 删除

安装和启动时做依赖检查；运行时处理 `tools/list` 变化；发现缺失后禁止执行相关步骤，给出明确降级或升级人工，不要让模型猜一个相似工具替代。

### 追问七：两者的安全边界分别在哪里

Skill 的重点是内容与脚本供应链、加载上下文和执行环境；MCP 的重点是连接、认证授权、Server 信任、工具参数和返回数据。组合后还要防权限放大与跨边界提示注入。

### 追问八：怎样证明组合方案比只写 Prompt 更好

用同一任务集比较触发准确率、关键步骤完成率、工具调用正确率、安全违规率、延迟和维护成本；同时验证 Skill 与 Server 可以独立升级并通过契约测试。

### 可直接背诵的完整答案

> MCP 和 Agent Skill 不在同一层。MCP 是 Host 中 Client 与 Server 的运行时协议，负责初始化、能力协商、Tools、Resources、Prompts 的发现和调用，以及标准传输；它回答“外部能力怎样接入”。Agent Skills 是以 `SKILL.md` 为入口的目录规范，可以携带指令、脚本、参考资料和模板；它回答“Agent 应怎样完成一类任务”。
>
> 二者可以组合：退款 Skill 定义核验、查询、审批和通知流程，订单与退款 MCP Server 提供实际工具。Host 仍要负责 Skill 选择、工具裁剪、参数校验、授权、确认、幂等和审计。只需要操作知识时可以只用 Skill，只需要标准能力接口时可以只用 MCP；固定强事务流程则应由业务代码强制，不能交给自然语言指令。安全上，Skill 按内容和代码供应链治理，MCP Server 按外部服务治理，任何一方都不能自动提升另一方的权限。
