## 本篇目标

学完这一篇，你应该能够：

- 按顺序描述 MCP 初始化、正常运行和关闭三个阶段；
- 解释 initialize 请求、响应和 initialized 通知各自作用；
- 说清日期协议版本如何协商，以及与 SDK 版本的区别；
- 区分 Client capabilities、Server capabilities 和用户授权；
- 正确处理请求超时、progress、取消和连接关闭；
- 设计兼容测试、会话状态、失败恢复和可观测性。

## 60 秒回答

MCP 2025-11-25 的生命周期分 Initialization、Operation 和 Shutdown。初始化必须是 Client 与 Server 的第一次协议交互：Client 发送 initialize，包含自己支持的 protocolVersion、clientInfo 和 capabilities；Server 返回选定协议版本、serverInfo 和 capabilities；Client 接受后发送 notifications/initialized，才进入正常操作。

版本协商不是交换完整版本列表。Client 发送自己支持的一个版本，通常是最新支持版；Server 支持就原样返回，否则返回自己支持的另一个版本，通常是 Server 最新支持版。Client 如果不支持返回版本，应断开。HTTP 后续请求必须携带 MCP-Protocol-Version。SDK 包版本和 MCP 日期协议版本是两个概念。

能力必须显式声明。Client 侧可声明 Roots、Sampling、Elicitation 等；Server 侧可声明 Tools、Resources、Prompts 等。双方只能使用已经协商的能力，listChanged 等子能力也要单独声明。能力只表示“会说这种协议消息”，不代表当前用户有权使用具体工具。所有请求应有可配置超时，progress 可以重置等待但仍要总上限；取消不保证副作用撤销；关闭没有专用 MCP shutdown RPC，而由 stdio 或 HTTP 传输完成。

## 详细解析

### 先说人话：先对暗号、确认语言和服务，再开始办事

两个陌生团队合作前要先确认：

- 我们使用哪一版合同；
- 双方分别能做什么；
- 联系人和软件版本；
- 什么时候算正式开始；
- 出现超时如何停止；
- 最后怎样结束连接。

如果 Client 一连上就调用 tools/list，而 Server 还不知道版本和能力，双方可能用不同字段解释同一消息。

### 贯穿案例：桌面 Client 连接文件 MCP Server

Client 支持：

- Roots；
- Sampling；
- Form 和 URL Elicitation。

Server 提供：

- Tools；
- Resources；
- Tools 列表变化通知。

初始化后，Client 才能：

- 请求 tools/list；
- 调用 read_file；
- 接收 Server 发起的 sampling/createMessage；
- 响应 roots/list。

但如果 Client 没声明 Sampling，Server 不能因为“这个 Client 看起来有模型”就发送 Sampling 请求。

### 最低前置概念：MCP 基于 JSON-RPC 消息

常见三类消息：

| 类型 | 特征 | 示例 |
| --- | --- | --- |
| Request | 有 id，需要 Response | initialize、tools/list |
| Response | 与 Request id 对应 | result 或 error |
| Notification | 没有 id，不等待 Response | notifications/initialized |

初始化顺序：

~~~text
Client                         Server
  |                               |
  |------ initialize Request ---->|
  |<----- initialize Response ----|
  |-- notifications/initialized ->|
  |                               |
  |======= Normal Operation ======|
~~~

### 生命周期三个阶段

#### Initialization

完成：

- 协议版本兼容；
- 能力声明和协商；
- 实现信息交换。

#### Operation

按已协商版本和能力收发消息。

#### Shutdown

通过底层传输关闭，没有通用 MCP shutdown 方法。

### 第一步：Client 发送 initialize

概念示例：

~~~json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-11-25",
    "capabilities": {
      "roots": {
        "listChanged": true
      },
      "sampling": {
        "tools": {}
      },
      "elicitation": {
        "form": {},
        "url": {}
      }
    },
    "clientInfo": {
      "name": "example-client",
      "version": "1.4.0"
    }
  }
}
~~~

字段作用：

| 字段 | 作用 |
| --- | --- |
| protocolVersion | Client 提议的 MCP 协议版本 |
| capabilities | Client 能接收和处理的可选能力 |
| clientInfo | Client 实现名称和软件版本 |

clientInfo.version 是产品或 SDK 实现版本，不是 protocolVersion。

### initialize 必须是第一次交互

规范要求初始化阶段是第一次 Client-Server 协议交互。

在收到 initialize 响应前：

- Client 不应发送除 Ping 外的其他请求。

在收到 initialized 通知前：

- Server 不应发送除 Ping 和 Logging 外的其他请求。

这两个例外用于连接检查和诊断，但不能借此提前调用 Tools 或 Sampling。

### 第二步：Server 返回 initialize 结果

~~~json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-11-25",
    "capabilities": {
      "logging": {},
      "resources": {
        "subscribe": true,
        "listChanged": true
      },
      "tools": {
        "listChanged": true
      }
    },
    "serverInfo": {
      "name": "example-files-server",
      "version": "2.3.0"
    },
    "instructions": "Optional usage guidance"
  }
}
~~~

Server 告诉 Client：

- 最终选择的协议版本；
- Server 能提供哪些能力；
- 实现信息；
- 可选使用说明。

instructions 只是说明，不是绕过 Host 策略和授权的高权限命令。

### 第三步：Client 发送 initialized 通知

~~~json
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}
~~~

它表示：

> 我接受初始化结果，已经准备进入正常操作。

它是 Notification：

- 没有 id；
- Server 不返回普通 Response；
- 发送后才进入 Operation。

### 协议版本如何协商

#### Client 发送什么

Client 必须发送一个自己支持的 protocolVersion，应该是它支持的最新版本。

#### Server 如何响应

- 支持请求版本：必须返回同一版本；
- 不支持：必须返回另一个自己支持的版本，应该是 Server 支持的最新版本。

#### Client 最终决定

Client 检查返回版本：

- 支持：继续，发送 initialized；
- 不支持：应断开。

### 协商不是“随便找一个差不多的版本”

错误想法：

~~~text
Client：2025-11-25
Server：2025-03-26
Client：字段看着差不多，继续吧
~~~

正确做法是 Client 明确知道自己是否实现该版本。日期版本之间可能新增字段、约束和安全语义，不能猜测兼容。

### 日期协议版本与 SDK 版本

| 版本 | 示例含义 |
| --- | --- |
| MCP protocolVersion | 消息和字段遵循哪版规范 |
| Client 软件版本 | 某个桌面应用发布版 |
| Server 软件版本 | 某个工具服务实现版 |
| SDK 包版本 | npm、PyPI 等依赖版本 |
| Tool Schema 版本 | 某个具体工具的业务契约 |

升级 SDK 不代表会话自动使用最新协议；还要看实现支持、配置和协商结果。

### HTTP 的 MCP-Protocol-Version Header

完成初始化后，HTTP Client 在后续请求中必须带：

~~~http
MCP-Protocol-Version: 2025-11-25
~~~

Server 应按这个版本解析请求。
Header 值应与已协商版本和会话一致，网关不能随意改写。

### 能力协商是什么

能力对象表达可选功能：

#### Client 侧常见能力

| Capability | Client 能做什么 |
| --- | --- |
| roots | 向 Server 提供文件系统 roots |
| sampling | 响应 Server 的模型采样请求 |
| elicitation | 响应 Server 的用户输入请求 |
| tasks | 支持相应任务增强请求 |
| experimental | 协商非标准实验能力 |

#### Server 侧常见能力

| Capability | Server 提供什么 |
| --- | --- |
| tools | 可调用工具 |
| resources | 可读取资源 |
| prompts | Prompt 模板 |
| logging | 结构化日志通知 |
| completions | 参数补全 |
| tasks | 任务增强请求 |
| experimental | 非标准实验能力 |

### 子能力也要协商

例如：

~~~json
{
  "tools": {
    "listChanged": true
  }
}
~~~

表示 Server 不仅提供 Tools，还会在列表变化时发送通知。

~~~json
{
  "sampling": {
    "tools": {}
  }
}
~~~

表示 Client 支持 Sampling 内的工具使用。只声明 sampling: {} 不等于支持 tool-enabled sampling。

### 能力声明是“会不会”，不是“准不准”

Client 声明 Roots：

- 表示能处理 roots/list；
- 不代表必须暴露所有目录；
- 用户可以只选一个工作区。

Server 声明 Tools：

- 表示能处理 tools/list 和 tools/call；
- 不代表当前用户能看到或调用所有工具；
- 列表可以按身份过滤；
- 每次调用仍要授权。

### 能力、发现和授权的区别

| 概念 | 回答的问题 |
| --- | --- |
| Capability negotiation | 双方支持哪些协议功能 |
| tools/list | 当前会话能看到哪些具体工具定义 |
| OAuth authorization | HTTP Client 是否获准访问 Server |
| Tool authorization | 当前主体能否执行该动作和资源 |
| User approval | 用户是否同意这次具体高风险操作 |

这些层次不能合并成一个布尔值。

### Operation 阶段

进入正常运行后，双方必须：

- 遵守协商版本；
- 只使用成功协商的能力；
- 关联 Request 与 Response id；
- 校验消息和 Schema；
- 设置超时；
- 处理错误和取消；
- 维持会话身份与授权；
- 对列表变化重新发现。

### 工具列表变化

如果 Server 声明 tools.listChanged，可以发送相应列表变化通知。Client 收到后重新调用 tools/list。

这不等于重新 initialize：

- 具体工具列表变了：按通知重新列举；
- 协议 capability 结构变了：通常建立新会话并重新初始化。

Client 不应继续永久使用旧工具 Schema。

### 请求超时

规范建议所有请求设置 timeout，避免挂起和资源耗尽。超时应：

1. 停止等待；
2. 发送 cancellation notification；
3. 记录状态；
4. 按工具语义决定后续。

SDK 和中间件应允许按请求配置。

### Progress 与最大上限

收到 Progress 表示工作仍在进行，实现可以重置等待计时。但规范也建议始终有最大 timeout，防止恶意或故障对端不断发进度让请求永不结束。

~~~text
每段空闲 timeout：可被 progress 重置
不可突破的最大 deadline：不能无限延长
~~~

### Cancellation 不等于副作用回滚

取消通知表达：

> 调用方不再需要这个结果。

它不能保证：

- 远端立即停止；
- 数据库没有提交；
- 邮件没有发出；
- 退款没有创建；
- 补偿已经完成。

写工具仍需要幂等键、状态查询、Saga 和审计。

### Error Handling

至少处理：

- 协议版本不匹配；
- 必需 capability 未协商；
- initialize 乱序；
- 请求超时；
- 无效消息；
- 未知 Request id；
- 连接中断；
- 会话过期；
- 授权失效；
- 工具列表或 Schema 变化。

错误要区分协议层和业务层：

~~~json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32602,
    "message": "Unsupported protocol version"
  }
}
~~~

与：

~~~json
{
  "status": "error",
  "error": {
    "code": "ORDER_NOT_REFUNDABLE",
    "retryable": false
  }
}
~~~

前者是协议请求错误，后者是工具业务错误。

### Shutdown 没有专用 MCP RPC

#### stdio

Client 通常：

1. 关闭子进程输入流；
2. 等待 Server 退出；
3. 超时后发送温和终止信号；
4. 再超时才强制终止。

具体信号机制取决于操作系统。

#### HTTP

通过关闭相关 HTTP 连接表示关闭。若使用会话或可恢复流，还要遵守传输规范管理会话状态。

不要虚构一个通用 mcp/shutdown 方法。

### HTTP 会话和断线恢复

Streamable HTTP 可能使用 MCP-Session-Id 管理会话，并通过 SSE 事件 ID 支持恢复。需要区分：

- 传输连接短暂中断；
- Server 会话仍存在；
- Server 已丢失会话；
- Client 明确开始新会话。

能恢复事件流时，不应自动重复 initialize 或重放已经产生副作用的请求。Server 明确会话不存在时，才建立新会话并重新协商。

### 初始化状态机

~~~text
DISCONNECTED
→ CONNECTING
→ INITIALIZING
→ WAITING_FOR_INITIALIZED
→ READY
→ CLOSING
→ CLOSED
~~~

异常状态：

~~~text
VERSION_UNSUPPORTED
CAPABILITY_MISSING
AUTH_REQUIRED
SESSION_LOST
PROTOCOL_ERROR
~~~

用状态机比一组零散布尔值更容易防乱序。

## 工程实践与边界

### Client 端伪代码

~~~python
connection.open()
state = "INITIALIZING"

response = request(
    method="initialize",
    params={
        "protocolVersion": latest_supported_version,
        "capabilities": client_capabilities,
        "clientInfo": client_info,
    },
    timeout=initialize_timeout,
)

if response.protocolVersion not in supported_versions:
    connection.close()
    raise UnsupportedProtocolVersion()

negotiated = validate_server_capabilities(response.capabilities)

notify("notifications/initialized")
state = "READY"
~~~

### 每次发送前检查能力

~~~python
def request_sampling(params):
    if state != "READY":
        raise LifecycleError("not initialized")
    if not negotiated.client_sampling:
        raise CapabilityError("sampling not negotiated")
    return send("sampling/createMessage", params)
~~~

Server 发送 Client 侧请求时也做同样检查。

### 连接池隔离

连接与会话至少按以下维度隔离：

- Server URL；
- 用户身份；
- 租户；
- authorization context；
- protocolVersion；
- MCP-Session-Id；
- Client 配置；
- 数据地域。

不能让 Alice 的连接池会话被 Bob 复用。

### 兼容测试

每个支持版本测试：

- initialize 顺序；
- 同版本响应；
- Server 降级版本；
- Client 不支持时断开；
- capability 缺失；
- listChanged；
- HTTP header；
- Ping；
- timeout、progress 和 cancellation；
- stdio / HTTP 关闭；
- 断线与会话恢复；
- OAuth 过期；
- 工具 Schema 变化。

### 协议升级流程

1. 明确 Client 与 Server 支持版本矩阵；
2. 契约测试新旧组合；
3. 影子连接或测试环境回放；
4. 先让一侧支持双版本；
5. 小范围协商新版本；
6. 监控初始化和消息错误；
7. 扩大流量；
8. 观察期后再停止旧版本。

不要同时升级协议、工具 Schema 和业务行为，避免回归无法归因。

### 监控指标

| 类别 | 指标 |
| --- | --- |
| 初始化 | 成功率、延迟、乱序错误 |
| 版本 | 协商版本分布、不支持率 |
| 能力 | 缺失能力错误、能力违规消息 |
| 请求 | 数量、超时、取消、未知 id |
| Progress | 请求持续时间、超过最大 deadline |
| 会话 | 创建、恢复、丢失、跨身份拒绝 |
| 关闭 | 正常关闭、强制终止、孤儿进程 |
| 授权 | 401、403、token 过期 |

### Trace 字段

- Client 与 Server implementation；
- 提议和最终 protocolVersion；
- 双方 capabilities；
- 会话 ID 的安全摘要；
- Request id 和 method；
- timeout、progress 和 cancellation；
- HTTP request ID；
- 授权主体摘要；
- 关闭原因；
- 错误码。

不要在 Trace 中记录 access token。

### 安全边界

- implementation 名称和 instructions 来自对端，不可信；
- capability 声明不能自动扩大本地权限；
- Server 声明 Tools 不代表所有工具安全；
- Client 声明 Roots 不代表暴露整个磁盘；
- 会话恢复必须绑定同一身份和授权；
- MCP-Session-Id 不是用户身份；
- 取消不能撤销已提交副作用；
- 实验 capability 要隔离并显式启用。

### 生命周期的边界

它解决协议状态一致性，不解决：

- HTTP OAuth 授权；
- 工具业务授权；
- 工具参数安全；
- 写操作幂等；
- Prompt Injection；
- 数据驻留；
- 模型质量；
- 用户审批。

## 常见误区

- **“TCP 或 HTTP 连上就初始化完成”**：还要 initialize、响应和 initialized。
- **“可以先 tools/list 再 initialize”**：初始化必须是第一次协议交互，只有规范列出的 Ping 等例外。
- **“Server 必须接受 Client 提议版本”**：不支持时返回另一个支持版本。
- **“Client 可以猜测兼容继续运行”**：不支持 Server 返回版本时应断开。
- **“SDK 版本就是 MCP 协议版本”**：两者独立。
- **“能力声明等于用户授权”**：能力只表达协议支持。
- **“Progress 可以无限续命”**：仍应执行不可突破的总 timeout。
- **“MCP 定义统一 shutdown RPC”**：当前规范通过底层传输关闭。

## 面试追问

### 1. initialize 为什么必须最先执行？

双方需要先确定协议版本、能力和实现信息，之后才能正确解释可选消息与字段，避免状态和语义不一致。

### 2. 版本协商失败怎么办？

Server 返回自己支持的版本；Client 若不支持，应关闭连接。不能靠“字段看起来一样”猜兼容。

### 3. Client 与 Server capabilities 分别有哪些？

Client 常见 Roots、Sampling、Elicitation；Server 常见 Tools、Resources、Prompts、Logging。具体还包括 Tasks、Completions 和实验能力，以当前规范为准。

### 4. capability 与 authorization 有什么区别？

Capability 表示实现支持哪类协议消息；authorization 判断当前主体能否访问 Server、工具和资源。支持 Tools 不代表当前用户有调用权限。

### 5. Server 新增工具需要重新 initialize 吗？

若已协商 tools.listChanged，可发列表变化通知，Client 重新 tools/list。若协议 capability 本身改变，通常建立新会话重新初始化。

### 6. Progress 收到后是否一直等？

可以重置当前等待计时，但仍要不可突破的最大 deadline，防止对端持续发进度造成资源耗尽。

### 7. Cancellation 能取消退款吗？

不能保证。取消只表达不再需要结果；远端可能已提交。退款类写工具还要业务幂等、状态查询和补偿。

### 8. HTTP 为什么要带 MCP-Protocol-Version？

它让后续 HTTP 请求明确使用已协商协议版本，Server 或网关据此解析和验证。
