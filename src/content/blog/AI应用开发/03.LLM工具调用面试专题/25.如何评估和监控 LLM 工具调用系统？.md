一个工具调用 Demo 跑通，并不代表系统可以上线。

用户说“把昨天那笔重复扣款退掉”，系统可能经历以下结果：

- 选对退款工具，但退错订单；
- 参数格式正确，但金额超过可退余额；
- 第一次执行成功，超时重试后又退了一次；
- 工具执行正确，但模型告诉用户“退款已经到账”，实际上只是提交审核；
- 最终回答看起来合理，但整个过程中根本没有调用真实工具。

如果只看最终文字是否“像一个好答案”，这些问题会被混在一起。

因此，工具调用评估的核心不是算一个笼统的准确率，而是把整条链路逐层拆开。

## 本篇目标

学完本篇，你应该能够：

1. 解释为什么工具调用系统不能只评估最终回答；
2. 把一次工具任务拆成触发、选择、参数、执行和任务结果等层级；
3. 设计包含正样本、负样本、歧义样本和攻击样本的离线评估集；
4. 计算工具触发精确率、召回率、选择准确率和参数正确率；
5. 区分离线评估、回归测试、线上监控、灰度发布和人工审核；
6. 用追踪数据定位问题究竟发生在模型、路由器、权限策略还是工具服务；
7. 在面试中给出完整的生产级回答。

先记住一条总原则：

> 评估工具调用，要同时回答“该不该调用、调用哪个、参数对不对、执行是否可靠、结果是否被正确使用、最终任务是否完成”。

## 60 秒回答

我会把 LLM 工具调用评估拆成四层。

第一层是模型决策：该调用时是否调用、不该调用时能否克制，以及工具选择是否正确。常用指标包括触发精确率、召回率、无工具请求误调用率和 Top-K 工具选择准确率。

第二层是参数质量：先检查 JSON 和 Schema 是否有效，再检查字段值、单位、对象归属、时间范围等业务语义是否正确。不能用字符串完全相等作为唯一标准，因为参数顺序和等价表达可能不同。

第三层是执行可靠性：统计授权拒绝、执行成功率、超时、重试、重复副作用、补偿成功率、P95/P99 延迟和工具自身错误率。

第四层是端到端任务结果：用户目标是否完成、最终回答是否忠于工具结果、有没有把“已受理”说成“已完成”，同时观察成本、安全违规和人工升级率。

离线时我会建立包含正常、歧义、并行、多轮、无需工具、权限不足和提示注入的版本化数据集，每次模型、提示词、Schema、路由器或工具版本变化都跑回归。线上则用统一 Trace 关联模型响应、工具调用 ID、Schema 版本、授权、执行和最终回答，通过灰度、告警和人工抽检发现分布漂移。只有分层指标，才能知道问题出在哪一层。

## 详细解析

### 一、先把一次工具任务拆成可观测步骤

一次完整任务通常不是“模型输出一个 JSON”这么简单。

```text
用户请求
   |
   v
是否需要工具？
   |
   v
候选工具是否包含正确工具？
   |
   v
模型是否选对工具？
   |
   v
参数是否满足结构与业务语义？
   |
   v
授权、审批与风控是否通过？
   |
   v
工具是否可靠执行？
   |
   v
模型是否正确理解工具结果？
   |
   v
用户目标是否真正完成？
```

每一个箭头都可能失败。

如果没有分层记录，最后只会看到一句“抱歉，操作失败”，无法判断应该修提示词、工具描述、权限策略、重试逻辑还是后端服务。

### 二、先定义“正确”，再写评估代码

评估的第一步不是选指标，而是定义期望行为。

以“查询订单”为例，正确可能表示：

- 应调用 `orders.get_order`；
- 订单号必须来自用户或可信页面上下文；
- 不允许调用写工具；
- 用户无权查看时应拒绝，而不是换另一个工具绕过；
- 工具返回 `not_found` 时应说明未找到，不能编造订单状态；
- 最终回答不能泄露工具返回中的内部字段。

以“退款”为例，正确还可能要求：

- 用户意图明确；
- 订单归属正确；
- 金额不超过可退余额；
- 高风险场景需要二次确认；
- 使用幂等键；
- 返回“申请已提交”时不能说成“资金已到账”。

因此，一条测试用例的标准答案不应只有 `expected_text`，还应包含过程期望。

```json
{
  "case_id": "refund_duplicate_charge_001",
  "input": "把昨天重复扣的 99 元退掉",
  "context": {
    "user_id": "user_42",
    "visible_orders": ["order_a", "order_b"]
  },
  "expected": {
    "should_call_tool": true,
    "allowed_tools": ["orders.search_orders"],
    "forbidden_tools": ["billing.create_refund"],
    "must_ask_clarification": true,
    "reason": "存在两笔候选订单，尚不能确定退款对象"
  }
}
```

这条用例的正确行为不是“直接退款”，而是先澄清。

### 三、评估集应该包含哪些类型

#### 1. 正常单工具样本

用户意图明确，只需要一个工具。

例如：

- “查一下订单 A123 的物流”；
- “北京现在的天气怎么样”；
- “读取项目 README”。

这类样本只能验证基础能力，不能代表生产难度。

#### 2. 不需要工具的负样本

例如：

- “帮我解释什么是退款”；
- “把这句话改得礼貌一点”；
- “你能做哪些事情”；
- 用户只是在表达情绪，没有要求执行动作。

如果评估集只有需要工具的样本，模型很容易学会“看到问题就调用”，却无法学会克制。

#### 3. 工具不可用或不存在的样本

用户要求的能力系统没有提供时，正确行为可能是说明限制、给出替代方案或转人工，而不是虚构工具名。

#### 4. 多个相似工具样本

专门区分：

- `get_order` 与 `search_orders`；
- `preview_refund` 与 `create_refund`；
- `send_email_draft` 与 `send_email`；
- `read_file` 与 `write_file`。

这能发现工具职责重叠和描述不清的问题。

#### 5. 参数边界样本

包括：

- 缺少必填字段；
- 日期跨时区；
- 金额单位不明确；
- 枚举值接近但不合法；
- 空字符串、超长文本和特殊字符；
- 用户说“下周一”，但当前日期位于周末；
- 页面上下文和用户文本互相冲突。

#### 6. 多工具与依赖样本

有些工具可以并行：

```text
查天气 ─┐
查日历 ─┼─> 汇总出行建议
查路况 ─┘
```

有些工具必须串行：

```text
搜索订单 -> 确认唯一订单 -> 查询可退金额 -> 用户确认 -> 发起退款
```

评估时要判断工具集合、顺序、依赖和停止条件，而不只是某一步有没有出现。

#### 7. 多轮与状态变化样本

例如第一轮查询库存，第二轮用户改了数量，第三轮库存已被其他请求占用。

这类测试需要真实或可控的状态机，不能只用静态问答对。

#### 8. 权限与安全样本

包括：

- 普通用户请求管理员工具；
- A 租户用户读取 B 租户订单；
- 工具结果中包含“忽略之前指令”的提示注入；
- 用户让模型生成任意 URL 再由工具访问；
- 文件路径包含 `../`；
- 命令参数包含 shell 操作符；
- MCP Server 描述冒充可信系统工具。

安全评估要验证系统拒绝危险行为，并且不会在最终回答中泄露敏感细节。

#### 9. 故障与恢复样本

主动注入：

- 超时；
- 限流；
- 临时 5xx；
- 永久业务错误；
- 部分并行调用失败；
- 工具返回超长结果；
- 重试后得到不同结果；
- 补偿操作失败。

没有故障注入，无法证明重试、幂等和降级真的有效。

### 四、第一层指标：该不该调用工具

把“需要工具”看作一个二分类问题。

| 真实情况 | 系统调用了工具 | 系统没有调用 |
| --- | --- | --- |
| 需要工具 | 真阳性 TP | 假阴性 FN |
| 不需要工具 | 假阳性 FP | 真阴性 TN |

#### 1. 触发精确率（Precision）

```text
Precision = TP / (TP + FP)
```

含义：系统决定调用工具时，有多少次确实应该调用。

精确率低说明系统爱乱调用，可能增加成本和副作用风险。

#### 2. 触发召回率（Recall）

```text
Recall = TP / (TP + FN)
```

含义：真正需要工具的请求中，有多少次成功触发。

召回率低说明系统经常“凭空回答”，没有使用外部真实数据。

#### 3. F1

```text
F1 = 2 * Precision * Recall / (Precision + Recall)
```

F1 用于综合观察，但不能取代业务权重。

对于转账、删除和退款，误调用的代价远高于漏调用，通常更重视精确率和人工确认；对于只读搜索，可能更重视召回率。

#### 4. 澄清准确率

不是所有不完整请求都应该“不调用”。有时正确行为是先向用户询问必要信息。

因此还要评估：

- 应澄清时是否澄清；
- 问的问题是否正好补齐缺失参数；
- 已有可靠上下文时是否重复询问；
- 澄清后是否正确恢复任务。

### 五、第二层指标：工具是否选对

#### 1. Top-1 工具选择准确率

模型最终选出的第一个工具是否在允许集合中。

一条用例可能有多个等价工具，因此标准答案最好是允许集合，而不是唯一字符串。

#### 2. Recall@K

对于前置路由器，正确工具是否出现在前 K 个候选中。

```text
Recall@K = 正确工具进入前 K 的用例数 / 需要该工具的用例总数
```

如果 `Recall@5` 很低，应该先修路由器；如果 `Recall@5` 很高但最终选择率低，问题更可能在模型、候选数量或工具描述。

#### 3. 混淆矩阵

统计模型经常把哪些工具混在一起。

```text
真实工具 \ 预测工具    get_order   search_orders   get_tracking
get_order                 920            65              15
search_orders              88           870              42
get_tracking               21            34             945
```

这比一个总体准确率更有用，因为它直接暴露职责重叠。

#### 4. 无效工具率

统计模型生成不存在、已下线或未暴露工具名的比例。

严格 Function Calling 通道通常能降低此问题，但动态工具、文本解析和版本不一致时仍可能发生。

### 六、第三层指标：参数是否正确

参数正确至少分四级。

#### 第 1 级：语法有效

参数能否被解析为 JSON 或指定格式。

#### 第 2 级：Schema 有效

类型、必填字段、枚举、嵌套结构和额外字段是否符合约束。

#### 第 3 级：语义正确

例如用户说“摄氏度”，模型是否传入 `celsius`；用户说“明天下午”，时间范围是否解析正确。

#### 第 4 级：业务有效

例如：

- 退款金额没有超过可退余额；
- 订单属于当前用户；
- 目标账户处于可操作状态；
- 文件路径位于允许根目录；
- URL 在允许域名列表内。

Schema 正确不代表业务正确。

#### 不要只做字符串完全匹配

下面两段 JSON 在语义上相同：

```json
{"city":"北京","unit":"celsius"}
```

```json
{"unit":"celsius","city":"北京"}
```

评估器应先解析、标准化，再比较。

对于日期、单位、大小写、默认值和数组顺序，需要定义领域规则。

参数指标可以包括：

- JSON 解析成功率；
- Schema 通过率；
- 必填字段召回率；
- 字段级准确率；
- 业务校验通过率；
- 危险参数拦截率；
- 缺参时正确澄清率。

### 七、第四层指标：多工具计划是否正确

多工具任务要评估的不只是集合，还包括顺序和依赖。

可以把期望过程表示成有向图：

```text
search_orders
      |
      v
select_unique_order
      |
      v
preview_refund
      |
      v
user_confirmation
      |
      v
create_refund
```

需要检查：

- 是否漏掉必要步骤；
- 是否出现多余调用；
- 是否在依赖完成前执行后续工具；
- 可并行步骤是否被不必要地串行化；
- 某一步失败后是否仍继续危险操作；
- 是否在达到目标后停止，而不是无限循环。

常见指标包括：

- 必要工具覆盖率；
- 多余调用率；
- 顺序约束满足率；
- 平均调用步数；
- 最大循环深度；
- 首次成功步数；
- 任务中止正确率。

### 八、第五层指标：工具执行是否可靠

模型做对了，不代表后端执行可靠。

#### 1. 成功率要分错误类型

不要把所有失败都记成 `tool_error`。

至少区分：

- 参数校验失败；
- 身份认证失败；
- 授权拒绝；
- 用户取消确认；
- 限流；
- 超时；
- 临时服务错误；
- 永久业务错误；
- 幂等冲突；
- 补偿失败；
- 工具版本不兼容。

#### 2. 延迟看分位数，不只看平均值

平均 500 毫秒可能掩盖少量 20 秒请求。

常见监控包括：

- P50：典型体验；
- P95：大多数慢请求上界；
- P99：长尾问题；
- 超时率；
- 排队时间；
- 模型等待工具结果的时间。

#### 3. 重试必须和幂等一起看

只统计“重试后成功率”会掩盖重复副作用。

对于写工具，需要统计：

- 幂等键覆盖率；
- 同一调用 ID 的重复执行次数；
- 重复扣款、重复发信等副作用事故数；
- 补偿触发率与补偿成功率。

#### 4. 并行调用要看部分成功

三个并行工具成功两个时，系统是降级回答、重试失败项，还是整批失败？

应记录每个子调用结果以及聚合策略，不能只记录一个总状态。

### 九、第六层指标：模型是否正确使用工具结果

工具返回真实数据后，模型仍可能理解错误。

例如工具返回：

```json
{
  "status": "submitted",
  "estimated_arrival_days": 5
}
```

错误回答：

> 退款已经到账。

正确回答：

> 退款申请已提交，预计 5 天内到账。

因此要评估：

- 最终回答是否引用了正确字段；
- 是否忠于工具状态；
- 是否混淆“请求已接受”和“操作已完成”；
- 是否遗漏关键限制；
- 是否泄露内部字段或敏感数据；
- 工具失败时是否编造成功结果；
- 多个工具结果冲突时是否指出不确定性。

确定性字段可以用代码比较；开放式表达可以用规则、人工评分和经过校准的模型评分结合。

### 十、第七层指标：用户任务是否完成

最终指标是用户目标，而不是模型是否调用了某个函数。

任务成功可能包括：

- 用户拿到了正确订单状态；
- 退款申请被正确创建且只创建一次；
- 文件被写到正确位置；
- 会议被安排在双方都有空的时间；
- 无权限请求被安全拒绝；
- 信息不足时系统成功收集补充信息。

“安全拒绝”也可以是成功结果。

端到端指标包括：

- Task Success Rate；
- 首次解决率；
- 人工升级率；
- 用户重新表述率；
- 用户取消率；
- 平均完成步数；
- 端到端延迟；
- 单个成功任务成本；
- 安全违规率。

### 十一、自动评分、人工评分与模型评分怎么选

#### 1. 能用确定性规则时，优先用规则

例如：

- 工具名是否在允许集合；
- JSON 是否符合 Schema；
- 金额是否相等；
- 调用顺序是否满足依赖；
- 是否出现重复写操作；
- 权限不足时是否执行了工具。

确定性评分可重复、便宜、易解释。

#### 2. 开放式质量使用人工评分

例如回答是否清楚、是否解释了风险、澄清问题是否自然。

人工标注需要评分标准和示例，否则不同标注者会理解不同。

#### 3. 模型评分适合辅助，不应成为唯一真相

模型评分器适合扩展开放式评估，但要：

- 用人工标注集校准；
- 隐藏被评模型身份，减少偏见；
- 固定评分规则和输出结构；
- 评估评分器自身的一致性；
- 对高风险决策保留人工复核；
- 不让评分器看到不应泄露的敏感参数。

### 十二、BFCL 等公开基准能解决什么

Berkeley Function Calling Leaderboard（BFCL）等公开评测可以测试单函数、多函数、并行、无关性和多轮工具调用能力，并使用结构化方式比较函数与参数，而不是只看输出字符串。

公开基准适合：

- 比较模型基础工具调用能力；
- 发现升级模型后的普遍变化；
- 学习评估分类与匹配方法；
- 建立内部基线。

但公开基准不能替代业务评估，因为它不知道：

- 你的权限系统；
- 你的工具职责重叠；
- 你的 Schema 和错误码；
- 你的写操作风险；
- 你的用户表达与地区差异；
- 你的真实延迟和服务故障。

正确做法是“公开基准看通用能力，私有数据集看业务可靠性”。

### 十三、离线评估流水线

一个实用的离线流水线如下：

```text
版本化数据集
   |
   v
固定模型/提示词/工具清单/策略版本
   |
   v
运行工具调用，但写操作进入模拟器
   |
   v
收集完整 Trace
   |
   v
确定性评分 + 模型评分 + 人工抽检
   |
   v
按场景、工具、语言、风险等级切片
   |
   v
与基线比较并执行发布门禁
```

#### 为什么写操作要进入模拟器

离线测试不能真的退款、发邮件或删除文件。

模拟器需要尽量保留真实语义：

- 状态转换；
- 错误码；
- 幂等行为；
- 延迟和超时；
- 权限拒绝；
- 并发冲突。

只返回成功结果的 Mock 容易制造虚假的高分。

#### 什么变化需要跑回归

- 模型或模型版本；
- System/Developer Prompt；
- 工具名称与描述；
- 输入输出 Schema；
- 路由器或检索索引；
- 权限和风险策略；
- 工具后端版本；
- 上下文压缩方式；
- 并行与重试策略。

### 十四、线上监控需要统一 Trace

线上一次请求通常跨越多个服务。

```text
HTTP 请求
  -> 路由器
  -> 模型调用 1
  -> 工具 A
  -> 工具 B
  -> 模型调用 2
  -> 最终响应
```

需要一个统一 `trace_id`，每个模型响应和工具调用再有自己的 `span_id`、`response_id` 或 `call_id`。

一条脱敏后的工具 Span 可以记录：

```json
{
  "trace_id": "trace_123",
  "span_id": "span_tool_7",
  "tool_id": "billing.create_refund",
  "tool_version": "2.1.0",
  "model_call_id": "call_abc",
  "risk_level": "high",
  "schema_valid": true,
  "authorization": "allowed",
  "confirmation": "accepted",
  "execution_status": "submitted",
  "latency_ms": 842,
  "retry_count": 0,
  "idempotency_key_present": true
}
```

日志中不应直接保存访问令牌、银行卡号、完整身份证号或未脱敏的工具结果。

OpenTelemetry 的生成式 AI 语义约定可以作为统一 Trace 字段的参考，但该规范仍可能演进，生产使用时要固定版本并维护内部稳定字段。

### 十五、线上看板应该怎样分层

#### 模型决策看板

- 工具调用率；
- 无工具回答率；
- 澄清率；
- 无效工具名率；
- 每轮工具调用数；
- 最大调用循环次数。

#### 路由看板

- 权限过滤前后工具数；
- Recall@K 的抽样估计；
- 候选数量分布；
- 路由延迟；
- 索引版本和更新失败；
- 各工具被召回与被选择的比例。

#### 工具执行看板

- 各工具成功率；
- P50/P95/P99 延迟；
- 超时与限流率；
- 重试率；
- 幂等冲突；
- 授权拒绝；
- 补偿失败。

#### 业务结果看板

- 任务成功率；
- 首次解决率；
- 人工升级率；
- 重复操作事故；
- 用户投诉；
- 单个成功任务成本。

总体平均值还要按工具、租户、语言、地区、客户端版本和风险等级切片，否则少数严重问题会被大流量正常请求稀释。

### 十六、告警不能只看服务是否存活

工具接口返回 200，不代表语义正确。

可以设置以下告警：

- 某工具选择率突然翻倍；
- 无工具请求误调用率上升；
- Schema 失败集中在新字段；
- 某租户授权拒绝异常增加；
- 写工具重试率上升；
- 同一幂等键出现多次成功副作用；
- 工具结果为失败但最终回答声称成功；
- 高风险工具确认率突然下降；
- 平均调用步数持续增加，可能出现循环；
- 新模型版本在某个语言切片明显回退。

告警要关联版本发布信息，否则排查时仍然不知道变化来源。

### 十七、灰度发布与版本对比

不要把新模型或新提示词一次性切给全部用户。

#### 1. Shadow Test

复制真实脱敏请求给新版本，但不执行真实写操作，也不把结果返回用户。

比较新旧版本的工具决策和参数。

#### 2. Canary

只让少量低风险流量使用新版本，观察错误、延迟和业务结果。

#### 3. A/B Test

比较用户任务成功率，但必须确保安全策略一致，并避免同一写操作被两个实验组重复执行。

#### 4. 发布门禁

例如：

```text
高风险工具误调用率不得上升
核心工具选择准确率下降不得超过 0.5%
P95 端到端延迟不得增加超过 10%
单任务平均成本不得增加超过 15%
安全攻击集必须 100% 通过确定性策略拦截
```

这些数字只是示例，真实阈值应由业务风险和历史基线决定。

### 十八、一个退款系统的分层评估示例

用户输入：

> 把订单 A123 退掉。

#### 第一步：是否需要工具

期望：需要，但不能立即执行写操作。

#### 第二步：工具选择

期望先调用 `billing.preview_refund`，不应直接调用 `billing.create_refund`。

#### 第三步：参数

期望订单号为 `A123`，用户 ID 从可信会话注入，不由模型填写。

#### 第四步：状态检查

如果工具返回“已发货，需要选择退货原因”，模型应询问用户，而不是继续调用。

#### 第五步：确认

模型展示金额与后果，用户明确同意后才能创建退款。

#### 第六步：执行

写调用必须带幂等键；超时后先查询已有结果，再决定是否重试。

#### 第七步：最终回答

如果状态是 `submitted`，回答“申请已提交”，不能说“退款已到账”。

#### 对应指标

| 层级 | 指标 |
| --- | --- |
| 触发 | 工具触发召回率 |
| 选择 | `preview_refund` 选择准确率 |
| 参数 | 订单号字段准确率 |
| 安全 | 未确认写操作拦截率 |
| 执行 | 幂等覆盖率、重复退款事故数 |
| 结果理解 | 状态陈述一致率 |
| 端到端 | 正确提交率、人工升级率、用户投诉率 |

这比一个“回答得分 0.87”更能指导优化。

## 工程实践与边界

### 1. 数据集必须版本化

记录：

- 用例版本；
- 标注规则版本；
- 工具与 Schema 版本；
- 权限策略版本；
- 运行时间；
- 模型和提示词版本；
- 评估器版本。

否则两次分数没有可比性。

### 2. 从真实失败中持续补样本

线上事故、用户纠正、人工升级和低置信度请求都可以成为候选样本，但进入评估集前要脱敏、去重和人工确认标签。

不能把含隐私的完整生产日志直接复制到评估平台。

### 3. 训练集与评估集要隔离

如果不断用同一批评估样本调提示词，最终会对测试集过拟合。

可以维护：

- 开发集：日常迭代可查看；
- 回归集：每次发布运行；
- 隐藏集：关键版本验收；
- 红队集：安全团队维护。

### 4. 指标要按风险加权，但保留原始分项

删除数据的误调用不能和天气查询选错工具等权平均。

可以构造风险加权得分用于发布门禁，但必须同时展示原始分项，避免综合分掩盖严重安全问题。

### 5. 先定位，再优化

```text
正确工具没进候选 -> 修路由与索引
正确工具在候选但没选 -> 修描述、候选集或模型提示
工具选对但参数错 -> 修 Schema、上下文和参数评估
参数正确但执行错 -> 修授权、工具服务或重试
执行正确但回答错 -> 修结果契约和结果解释提示
全链路正确但用户不满意 -> 修产品交互与任务定义
```

### 6. 不要记录原始思维过程

监控需要记录可观测决策、调用、结果和错误，不需要也不应依赖模型的隐藏推理文本。

可记录简短、面向审计的决策原因，例如“缺少订单号，请求澄清”，但不要把不可用的内部推理当成稳定接口。

### 7. 线上自动判分必须谨慎

真实请求通常没有现成标准答案。线上指标更多依赖业务状态、用户行为、抽样人工标注和可验证结果。

不要把“用户没有继续追问”直接当成功，也不要把“点击了确认”当作工具执行成功。

### 8. 评估框架不是业务正确性的替代品

供应商评估平台、开源框架或公开基准可以提高效率，但业务标签、权限语义、错误成本和发布门禁仍由团队定义。

### 9. 设定数据保留与访问策略

工具 Trace 可能包含客户数据和内部系统信息，需要：

- 字段级脱敏；
- 最小化采集；
- 分级访问；
- 保留期限；
- 审计日志；
- 删除机制；
- 跨地区存储合规检查。

### 10. 评估本身也要测试

验证评分器是否：

- 能识别等价参数；
- 对字段顺序不敏感；
- 不把安全拒绝判成失败；
- 不泄露隐藏标准答案；
- 在相同输入上稳定；
- 与人工判断具有足够一致性。

## 常见误区

### 误区一：最终回答看起来正确，工具调用就正确

模型可能没有调用工具而猜对，也可能工具执行错误但话术很好。必须检查过程与真实业务状态。

### 误区二：Schema 通过率就是参数准确率

Schema 只验证结构。订单归属、金额上限、日期语义和权限需要业务校验。

### 误区三：公开排行榜高，就能直接用于业务

公开基准测试通用能力，不覆盖你的工具、权限、数据分布、风险和后端可靠性。

### 误区四：评估集全部是“应该调用工具”的问题

这样无法测量误调用和拒绝能力，会鼓励模型过度调用。

### 误区五：参数必须和标准 JSON 字符串完全一致

字段顺序、默认值和等价表达可能不同。应解析并按领域规则标准化后比较。

### 误区六：平均延迟足够说明体验

平均值会掩盖长尾。工具链路必须观察 P95/P99、超时和每一步耗时。

### 误区七：重试成功率越高越好

写操作重试可能造成重复副作用。必须同时观察幂等、重复执行和补偿。

### 误区八：模型评分可以完全替代人工

模型评分需要校准，会有偏差，也不适合独自裁决高风险行为。

### 误区九：一个综合分足以做发布决策

综合分可能掩盖某个高风险工具的严重回退。必须设置硬性安全门禁并查看分项。

### 误区十：线上失败都属于模型问题

错误可能来自路由、Schema 版本、权限、后端、重试、网络或结果解释。分层 Trace 才能归因。

## 面试追问

### 追问一：工具调用最核心的三个离线指标是什么？

如果只能先选三个，我会选：是否该调用的精确率/召回率、正确工具选择率、参数业务正确率。但生产上线还必须补充执行成功率、安全违规率和端到端任务成功率。

### 追问二：为什么要评估“无需工具”的样本？

因为模型必须学会克制。没有负样本，就无法测量误调用；对写工具而言，误调用的风险可能远高于漏调用。

### 追问三：路由器和主模型如何分别评估？

路由器主要看正确工具是否进入候选，即 `Recall@K`、候选数和延迟；主模型在固定候选集上看最终工具选择、拒绝和参数。端到端再把二者组合起来测任务成功。

### 追问四：怎样比较两个工具参数是否相同？

先解析并按 Schema 规范化，再按字段类型比较。对象忽略键顺序；日期统一时区；单位转换到标准单位；数组是否忽略顺序由业务语义决定；敏感或服务端注入字段不要求模型生成。

### 追问五：工具执行失败后，模型的正确行为怎么评估？

根据错误类型定义期望：临时只读错误可有限重试；权限拒绝应停止；缺参应澄清；写操作超时应先查询幂等结果；不可恢复错误应透明说明并提供替代方案。不能统一要求“最终成功”。

### 追问六：如何评估并行工具调用？

检查每个调用是否必要、是否真的可以并行、参数是否独立、部分失败策略是否正确、结果关联是否使用正确的调用 ID，并比较端到端延迟与串行基线。

### 追问七：模型升级时如何防止工具能力回退？

固定版本化回归集，保持其他变量不变做对比；按工具、语言和风险切片；对写工具进行模拟；通过 Shadow 和 Canary 观察真实分布；设置安全硬门禁和一键回滚。

### 追问八：线上没有标准答案，怎么判断任务成功？

优先使用可验证业务状态，例如退款单是否只创建一次、会议是否真的存在、查询对象是否匹配。再结合用户行为、人工抽检和模型评分。不同信号要区分因果，不能用单一点击或无追问作为成功。

### 追问九：如何监控提示注入导致的工具滥用？

记录工具结果来源、风险标签、随后产生的调用链、授权拒绝和异常工具切换；用注入攻击集做离线回归；线上对高风险工具设置确认、允许列表和行为告警。模型“看起来没被影响”不是安全证明。

### 追问十：你会把完整工具参数写入日志吗？

不会默认全量记录。应按字段分类，敏感字段哈希、掩码或不采集；只保存评估所需的最小信息；限制访问和保留时间。调试便利不能凌驾于隐私与密钥安全。

### 追问十一：BFCL 分数很高，为什么内部准确率仍可能低？

因为内部工具名称、职责重叠、用户表达、权限和状态机与公开基准不同。BFCL 说明模型具备较好的通用函数调用能力，不保证它理解你的业务契约。

### 追问十二：如何向面试官总结整个评估体系？

可以回答：我会用离线版本化数据集做分层回归，用统一 Trace 做线上归因，用业务状态验证端到端成功，再通过 Shadow、Canary、安全门禁和人工抽检控制发布风险。
