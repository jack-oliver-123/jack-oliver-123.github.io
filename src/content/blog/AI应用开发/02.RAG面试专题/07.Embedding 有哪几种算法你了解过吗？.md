## 本篇目标

读完本篇，你应该能够：

- 解释“Embedding 算法”这个问题为什么需要先分类；
- 区分词频表示、静态词向量、上下文表示、句向量和检索向量；
- 理解 Word2Vec、GloVe、fastText、BERT、SBERT 和 DPR 的基本思路；
- 说清稠密单向量、学习稀疏表示和多向量 Late Interaction 的取舍；
- 根据 RAG 的数据、查询和基础设施选方法，而不是只报模型名称。

## 60 秒回答

Embedding 可以按表示粒度和检索方式分类。早期有 One-hot、词袋和 TF-IDF 这类词频表示；Word2Vec、GloVe 和 fastText 生成每个词固定的静态向量；ELMo、BERT 等模型会根据上下文为同一个词生成不同的 Token 表示。

用于现代 RAG 语义检索时，更常见的是 Sentence-BERT、DPR、E5、BGE 等双编码器或对比学习模型，把查询和文档分别编码成单个稠密向量，便于大规模 ANN 检索。除此之外，还有 SPLADE 这类学习稀疏表示，以及 ColBERT 这类为每个 Token 保留向量、在查询时做 Late Interaction 的多向量方法。

这些方法不是简单的新旧排名。词法检索擅长编号和专有名词，单稠密向量成本较低且适合语义召回，多向量保留更细的匹配信息但索引与计算更重。工程上应在自有数据上比较质量、延迟、存储、更新和基础设施兼容性。

## 详细解析

### 先说人话：同一句“苹果”可以有不同表示

看两个句子：

1. “我买了一个苹果。”
2. “苹果发布了新款电脑。”

“苹果”在第一句是水果，在第二句是公司。

早期静态词向量通常给“苹果”一个固定坐标，很难根据句子改变含义。上下文模型会结合周围词语，为两个“苹果”生成不同表示。

这说明 Embedding 不是单一算法。它可以表示：

- 一个词；
- 一个 Token；
- 一个句子；
- 一个文档 Chunk；
- 一张图片；
- 查询与文档之间的匹配关系。

面试官问“有哪些 Embedding 算法”时，先按发展思路和检索结构分类，比堆一串模型名更清楚。

### 第一类：离散词频表示

#### One-hot

假设词表只有四个词：

- 差旅；
- 住宿；
- 报销；
- 电脑。

每个词用只有一个位置为 1 的向量表示：

| 词 | 向量 |
| --- | --- |
| 差旅 | [1, 0, 0, 0] |
| 住宿 | [0, 1, 0, 0] |
| 报销 | [0, 0, 1, 0] |
| 电脑 | [0, 0, 0, 1] |

它能区分词，却不能表示“差旅”和“住宿”比“电脑”更相关。词表很大时，向量也非常稀疏。

#### Bag of Words

词袋模型统计一段文本中每个词出现多少次，不考虑词序。

“上海住宿报销”和“报销住宿上海”得到相近表示。它适合词项匹配，但难以理解句子结构和同义表达。

#### TF-IDF

TF-IDF 在词频基础上降低常见词权重，提高对当前文档更有区分力的词。

直觉是：

- 一个词在当前文档经常出现，可能重要；
- 一个词在所有文档都出现，区分力较低。

TF-IDF 不是神经网络 Embedding，但它是理解稀疏表示和关键词检索的重要基础。

#### BM25

BM25 是经典的概率相关性排序函数，会考虑词频饱和和文档长度。它通常不被叫作 Embedding 模型，但在 RAG 中经常与稠密向量检索组合。

编号、产品型号、人名和精确制度用语，往往由关键词检索稳定命中。

### 第二类：静态词向量

“静态”表示每个词在所有上下文中共享一个向量。

#### Word2Vec

Word2Vec 的核心直觉来自分布假设：经常出现在相似上下文中的词，含义往往相关。

它有两种经典训练方式：

- **CBOW**：根据周围词预测中间词；
- **Skip-gram**：根据中间词预测周围词。

经过训练，“酒店”“住宿”和“宾馆”可能在向量空间中靠近。

优势：

- 计算和存储相对轻；
- 便于学习词级语义；
- 对后续词向量研究影响很大。

局限：

- 多义词只有一个向量；
- 未登录词难以处理；
- 不能直接得到高质量句子检索表示。

#### GloVe

GloVe 使用整个语料的词共现统计来学习向量。它强调全局共现矩阵，而 Word2Vec 更像在局部窗口上做预测训练。

两者都属于静态词向量，主要区别在训练目标，不应简单说其中一个在所有任务都更好。

#### fastText

fastText 不只把词当作整体，还使用字符 n-gram。

例如“报销单”和“报销流程”共享“报销”相关字符片段。这样有助于：

- 处理词形变化；
- 表示罕见词；
- 缓解部分未登录词问题；
- 支持形态丰富语言。

它仍属于静态表示，无法充分处理同一个词在不同上下文中的含义。

### 第三类：上下文化 Token 表示

#### ELMo

ELMo 使用双向语言模型，根据整句话为每个词产生上下文相关表示。同一个词在不同句子中可以得到不同向量。

它开启了将深层上下文表示用于下游任务的重要路线。

#### BERT

BERT 使用 Transformer 编码器，通过掩码语言模型等预训练任务学习双向上下文。

输入：

> “苹果发布了新电脑。”

BERT 中“苹果”的 Token 表示会受到“发布”“电脑”等词影响。

不过，原始 BERT 的训练目标不是直接优化“整句余弦相似度”。简单取 CLS Token 或平均 Token，并不自动得到适合大规模语义检索的句向量。

这是一个常见面试陷阱：

> **BERT 可以产生上下文表示，但原始 BERT 不等于已经训练好的句向量检索模型。**

### 第四类：句子和文档的单向量表示

#### Sentence-BERT

Sentence-BERT 使用孪生或三塔结构，让句子可以分别编码，再通过余弦等方式比较。

为什么这是重要变化？

如果使用 Cross-encoder，把查询和每个候选文档拼在一起编码，比较全部语料成本很高。

双编码器可以：

- 离线预先编码所有文档；
- 在线只编码查询；
- 用向量索引快速寻找近邻。

这使大规模语义检索更可行。

#### DPR

DPR 是面向开放域问答的稠密段落检索方法。它使用独立的查询编码器和段落编码器，并通过问答数据中的正负段落训练。

查询“谁提出了某理论”与包含答案的百科段落应得到更高点积分数。

DPR 展示了稠密检索在开放域问答中的价值，也影响了很多 RAG 检索器设计。

#### E5、BGE 等检索模型

这类现代模型通常采用大规模弱监督数据、对比学习、困难负样本和任务指令等方式训练。

工程上不要只说“我用过 BGE”或“E5 排名高”，而要说明：

- 使用的具体版本；
- 查询和文档模板；
- 支持语言；
- 最大输入长度；
- 向量维度；
- 归一化和距离函数；
- 自有数据 Recall 和延迟；
- 是否需要本地部署。

模型家族名称不足以复现实验。

### 第五类：学习稀疏表示

传统稀疏检索的维度对应词表，只有少量位置非零。

SPLADE 使用神经网络学习词表维度上的稀疏权重，还可以扩展与原文不完全相同但相关的词项。

它保留了稀疏表示的特点：

- 可以使用倒排索引；
- 匹配词项具有部分可解释性；
- 对精确词汇较友好。

同时也引入：

- 模型推理成本；
- 索引膨胀控制；
- 稀疏正则；
- 与现有搜索引擎的工程适配。

学习稀疏表示不等于 BM25。两者都可使用词表和倒排索引，但权重来源和扩展机制不同。

### 第六类：多向量与 Late Interaction

单向量方法把整个查询或 Chunk 压缩成一个向量，效率高，但可能丢失细粒度 Token 匹配信息。

ColBERT 为查询和文档保留多个 Token 向量。查询时，让每个查询 Token 找到文档中最相似的 Token，再聚合分数，这种方式称为 Late Interaction。

直觉上：

- 双编码器：两段文本先各自压缩成一个点，再比较；
- ColBERT：保留许多词级点，最后再做细粒度匹配。

收益可能是更精细的匹配；代价是：

- 每个文档保存多个向量；
- 索引更大；
- 在线交互计算更重；
- 基础设施和调优更复杂。

是否值得使用，要看质量收益能否覆盖存储、延迟和运维成本。

### 第七类：Cross-encoder 重排

Cross-encoder 把 Query 与候选文档一起输入模型，直接输出相关性分数。

它严格来说不是把整个语料预先编码成可独立检索的 Embedding，但在 RAG 检索链路中非常重要。

优点：

- Query 与文档可以充分交互；
- 通常比单向量分数更精细。

缺点：

- 每个 Query-文档对都要在线计算；
- 不适合直接与海量文档逐一比较。

典型组合是：

> BM25 或双编码器快速召回 -> Cross-encoder 对少量候选重排。

### 一张分类总表

| 类别 | 每个文本的表示 | 典型方法 | 优点 | 主要代价 |
| --- | --- | --- | --- | --- |
| 词频稀疏 | 词表维度稀疏向量 | TF-IDF、BM25 | 精确词匹配、可解释 | 同义表达能力有限 |
| 静态词向量 | 每个词一个固定向量 | Word2Vec、GloVe、fastText | 轻量、学习词语相似 | 多义词处理弱 |
| 上下文 Token | 每个 Token 随上下文变化 | ELMo、BERT | 上下文理解强 | 不是直接检索句向量 |
| 稠密单向量 | 每个 Query 或 Chunk 一个向量 | SBERT、DPR、E5、BGE | 可预计算、ANN 高效 | 可能压缩掉细节 |
| 学习稀疏 | 每个文本一个学习得到的稀疏向量 | SPLADE | 词项可解释、可倒排 | 模型与索引成本 |
| 多向量 | 每个文本多个 Token 向量 | ColBERT | 细粒度匹配 | 存储和查询更重 |
| 联合编码 | 每个 Query-文档对一个分数 | Cross-encoder | 交互充分、排序精细 | 难以全库逐一计算 |

### 这些方法背后的训练思想

#### 分布假设

出现在相似上下文中的词，语义往往相关。Word2Vec、GloVe 等都利用了这个思想。

#### 语言建模

通过预测词或被遮盖内容学习上下文表示。ELMo 和 BERT 属于这条路线。

#### 对比学习

拉近 Query 与正样本，推远负样本。现代句向量和检索模型常使用这种目标。

#### 困难负样本

使用“看起来很相关但不能回答问题”的文档训练模型识别细微差异。

例如：

- 正样本：上海普通员工住宿上限；
- 困难负样本：北京普通员工住宿上限；
- 随机负样本：电脑采购制度。

困难负样本比随机负样本更有训练价值，但也更容易出现假负例。

#### 蒸馏

使用更强但更慢的模型为轻量模型提供软标签，让轻量检索器学习更精细的相关性。

### RAG 中怎样选

#### 查询包含精确编号

例如“制度 HR-2026-017 第四条”。关键词或混合检索通常不可缺少。

#### 用户表达口语化

例如“出差住店能报几百”。稠密检索更有机会匹配“住宿费报销上限”。

#### 需要高吞吐和较低存储

单向量双编码器通常更容易扩展。具体容量和延迟仍需压测。

#### 细粒度匹配收益很高

法律条款、长段落或精准证据检索中，可以评估 Late Interaction 或 Cross-encoder 重排。

#### 现有搜索基础设施成熟

可先从 BM25 + 稠密向量 + 融合开始，不需要为了追逐模型名称立即替换全部系统。

### 员工制度案例

用户查询：

> “HR-17 里上海旺季酒店怎么算？”

它同时包含：

- 精确编号“HR-17”；
- 城市“上海”；
- 条件“旺季”；
- 口语表达“酒店怎么算”。

不同检索方式的作用：

| 路径 | 可能命中 |
| --- | --- |
| BM25 | 精确找到 HR-17 和“上海” |
| 稠密向量 | 将“酒店怎么算”匹配到“住宿费报销标准” |
| 学习稀疏 | 结合词项匹配和语义扩展 |
| Cross-encoder | 区分上海普通标准与旺季例外 |
| ColBERT | 保留编号、城市、旺季等细粒度交互 |

生产方案常是组合，而不是只选一种表示。

### 怎样公平比较不同类别

必须控制：

- 同一文档语料；
- 同一查询和标注；
- 同一权限与过滤条件；
- 相同或可解释的候选规模；
- 相同目标硬件和并发；
- 固定模型与产品版本。

报告：

- Recall@K、MRR、nDCG；
- 不同查询切片；
- 索引大小；
- 构建和更新吞吐；
- P95 查询延迟；
- 资源与成本；
- 端到端答案指标。

不要只比较“某个公开数据集上的模型分数”，也不要让一种方法使用重排、另一种方法不使用，却把结果归因于 Embedding。

## 工程实践与边界

### 从简单基线开始

一个可解释的实验顺序：

1. BM25；
2. 稠密单向量；
3. BM25 + 稠密融合；
4. 在融合候选上加入 Cross-encoder；
5. 质量仍受细粒度匹配限制时，再评估学习稀疏或多向量。

每增加一层，都用消融实验验证收益。

### 明确表示与索引的耦合

- 稠密单向量需要向量索引；
- 稀疏表示通常使用倒排索引；
- 多向量方法需要支持多个 Token 向量及相应交互；
- Cross-encoder 需要在线推理服务。

换模型可能意味着换索引结构，不只是替换一个接口。

### 训练时防止数据泄漏

如果业务查询或测试答案出现在训练集中，离线指标会虚高。应按时间、文档或用户划分训练和测试数据，并保存数据版本。

### 困难负样本需要审核

可从当前检索器的高排名错误中挖掘困难负样本，但要排除：

- 实际也能支持答案的片段；
- 标注遗漏的相关文档；
- 不同版本中同样有效的规则。

### 不按“代际”做统一判断

静态词向量对现代端到端 RAG 通常不够，但词法稀疏检索仍然很有价值。较新的模型也可能因语言、许可、延迟或硬件不满足而不适用。

## 常见误区

- **“BERT 就是句子 Embedding”**：BERT 产生上下文 Token 表示，原始训练目标不直接优化句子检索相似度。
- **“词法检索已经过时”**：编号、专有名词和精确短语仍然是它的优势。
- **“静态词向量和上下文向量只差模型大小”**：核心差异是同一个词能否随上下文改变表示。
- **“SPLADE 就是 BM25”**：两者都可使用稀疏索引，但 SPLADE 的权重和扩展由模型学习。
- **“ColBERT 是一个更大的单向量模型”**：它保留多个 Token 向量，并在查询时做 Late Interaction。
- **“Cross-encoder 适合直接扫描全库”**：联合编码每个 Query-文档对的成本较高，通常用于候选重排。
- **“新模型在所有任务都优于旧方法”**：任务分布、基础设施和成本可能改变结论。
- **“模型名相同，结果就能复现”**：还需要版本、前缀、池化、归一化、距离函数和索引参数。

## 面试追问

1. **Word2Vec 的 CBOW 和 Skip-gram 有什么区别？**

   CBOW 用周围上下文预测中心词；Skip-gram 用中心词预测周围词。两者都学习静态词向量，具体表现依赖数据和训练配置。

2. **为什么原始 BERT 的 CLS 不适合直接做语义检索？**

   原始预训练目标没有直接优化句子向量之间的相似度排序。需要合适池化和句对或检索目标训练，例如 SBERT 路线。

3. **双编码器和 Cross-encoder 怎样组合？**

   双编码器离线编码文档并快速召回较多候选，Cross-encoder 在线联合读取 Query 与少量候选做精细重排。

4. **为什么 RAG 常用混合检索？**

   稀疏方法擅长精确词项，稠密方法擅长同义和语义表达。融合可以覆盖不同查询类型，但需处理分数不可直接比较和重复候选。

5. **什么时候考虑 ColBERT？**

   当单向量压缩丢失的细粒度匹配明显影响关键查询，并且目标数据上的质量收益能够覆盖多向量索引、延迟和运维成本时。

6. **模型微调怎样选负样本？**

   混合随机负样本和当前检索器返回的困难负样本，按实体、时间和适用范围检查假负例，并保留独立测试集。
