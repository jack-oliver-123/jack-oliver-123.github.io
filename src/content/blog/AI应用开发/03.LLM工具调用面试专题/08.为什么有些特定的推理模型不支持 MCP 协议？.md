## 本篇目标

学完这一篇，你应该能够：

1. 纠正“模型原生支持 MCP”这个不够准确的说法。
2. 解释模型、模型 API、Agent Host、MCP Client 和 MCP Server 各自负责什么。
3. 区分协议连不通、平台没适配、API 没有工具通道和模型不会正确用工具。
4. 解释为什么某些推理模型在产品里显示“不支持 MCP”，但这通常不是 MCP 对模型下了禁令。
5. 给出一套从网络、协议、适配、模型到安全策略的排查流程。
6. 回答“没有原生 Function Calling，是否还能使用 MCP”这类进阶追问。

## 60 秒回答

> 这道题的前提需要先纠正：MCP 通常不是由大模型直接实现的协议。按照 MCP 架构，Host 里运行 MCP Client，Client 与 MCP Server 建立连接、发现工具并发起调用；模型看到的通常只是 Host 转换后的工具定义和工具结果。
>
> 因此，产品所说的“某模型不支持 MCP”，一般表示这个模型、API 端点、SDK 和 Host 的组合还不能稳定完成工具调用闭环。常见原因包括：API 没有结构化工具调用通道，工具调用事件格式尚未适配，流式事件无法正确拼装，不能把工具结果送回模型，模型没有经过足够的工具使用训练，或者平台出于安全和产品策略暂时关闭了该能力。
>
> 推理时间长、输出思考内容或上下文成本高，可能增加适配难度，但都不是 MCP 协议层面的“不兼容”。排查时我会分五层：先验证 MCP Client 与 Server 是否能独立初始化和调用；再验证 Host 的 Schema 映射；然后检查模型 API 的工具事件和多轮回传；再测模型选工具、填参数、利用结果的准确率；最后检查授权、审批和执行策略。这样才能找对责任层，而不是把所有问题都归因于模型。

## 详细解析

### 先说人话：MCP 更像“插座标准”，模型更像“做决定的人”

可以把一个 Agent 应用想成一家办公室：

- **大模型**像负责判断下一步做什么的员工；
- **Agent Host** 像员工使用的办公系统；
- **MCP Client** 像办公系统里的插座适配器；
- **MCP Server** 像提供数据库、文件、搜索或业务操作的设备；
- **MCP** 规定适配器和设备如何互相介绍能力、发送请求和返回结果。

员工不需要亲自理解插座内部如何握手，也不需要直接发送 JSON-RPC 消息。

他只需要向办公系统表达：

```text
我想调用“查询订单”工具，参数是订单号 A1001。
```

Host 再把这个意图转换成 MCP 的 `tools/call` 请求。

因此，“某个模型会不会讲 MCP 协议”通常不是关键问题。真正的问题是：

```text
Host 能不能把 MCP 能力翻译成模型看得懂的工具定义，
再把模型的调用意图翻译回 MCP 请求？
```

### MCP 的真实参与者是谁

MCP 采用 Host、Client、Server 的架构：

```text
用户
  │
  ▼
Agent Host
  ├── 模型 API 适配器 ──► 大模型
  │
  ├── MCP Client A ─────► MCP Server A
  │                       ├── tools
  │                       ├── resources
  │                       └── prompts
  │
  └── MCP Client B ─────► MCP Server B
```

几个角色不能混淆：

| 角色 | 主要职责 | 是否通常由模型承担 |
| --- | --- | --- |
| 大模型 | 理解任务、选择动作、生成参数、利用结果 | 是，负责认知判断 |
| Host | 管理会话、模型调用、权限、工具循环和用户交互 | 否，属于应用程序 |
| MCP Client | 与一个 MCP Server 建立会话并发送协议消息 | 否，通常嵌在 Host 中 |
| MCP Server | 暴露 Tools、Resources、Prompts 等能力 | 否，是能力提供方 |
| 工具执行器 | 真正访问数据库、文件或业务 API | 否，位于 Server 或下游系统 |

规范还强调 Host 与不同 Server 之间保持隔离。一个 Server 不应该自动看见其他 Server 的完整上下文。

### 一次 MCP 工具调用究竟经过哪些步骤

以“查询订单”为例：

1. Host 启动或连接 MCP Server。
2. MCP Client 与 Server 执行 `initialize`。
3. 双方协商协议版本和能力。
4. Host 通过 Client 获取工具列表。
5. Host 把 MCP Tool 的名称、描述和输入 Schema 转换成模型 API 接受的工具定义。
6. 用户说：“帮我查 A1001 的物流。”
7. 模型返回结构化调用意图，例如工具名和参数。
8. Host 校验工具是否允许、参数是否合法、是否需要用户确认。
9. MCP Client 向 Server 发送 `tools/call`。
10. Server 执行查询并返回结果。
11. Host 把结果转换成模型 API 的工具结果消息。
12. 模型结合结果生成最终回答。

其中只有第 7 步和第 12 步主要依赖模型能力。

第 1～5、8～11 步主要依赖 Host、Client、Server 和平台工程。

这就是为什么不能简单说“MCP 是模型原生协议”。

### “支持”至少要拆成五层

面试时最重要的能力不是直接回答“支持”或“不支持”，而是先问：

```text
你说的是哪一层支持？
```

#### 第一层：MCP 协议层

需要确认：

- Client 与 Server 能否建立标准传输；
- 能否完成初始化和版本协商；
- 能否正确处理 JSON-RPC 请求、响应和通知；
- 能否列出并调用 Tools；
- 会话、取消、超时和错误是否按规范处理。

这一层甚至可以完全不调用大模型。

可以写一个确定性测试程序：

```text
初始化 -> tools/list -> tools/call -> 校验结果
```

如果这个测试都失败，问题肯定不在模型。

#### 第二层：Host 适配层

Host 要做双向翻译：

```text
MCP Tool Schema
        ↓
模型 API 的工具定义

模型工具调用事件
        ↓
MCP tools/call

MCP 工具结果
        ↓
模型 API 的工具结果消息
```

适配层常见问题包括：

- MCP 的 JSON Schema 特性超出模型 API 支持范围；
- 工具名长度、字符集或命名规则不同；
- SDK 丢失工具调用 ID；
- 流式参数被拆成多个事件，Host 没有正确拼接；
- 并行调用顺序与结果关联错误；
- 工具结果太长，超过上下文或接口限制；
- 取消请求后，后台工具仍继续执行。

#### 第三层：模型 API 通道

模型能力与 API 能力也不是一回事。

某个模型可能具有工具使用训练，但某个具体端点只暴露文本输入输出；也可能批处理端点支持工具，实时端点暂时不支持；还可能 SDK 尚未暴露底层事件。

一个便于记忆的判断是：

```text
模型“可能会”做什么，不等于当前 API“允许应用可靠地拿到”什么。
```

可靠工具循环通常需要这些接口语义：

- 传入结构化工具定义；
- 区分普通文本和工具调用；
- 提供稳定的调用 ID；
- 传回工具结果并关联原调用；
- 支持多轮继续生成；
- 对流式、并行调用和错误有明确事件模型。

#### 第四层：模型行为能力

即使 API 格式完全兼容，模型也未必能把工具用好。

需要分别评估：

- **工具选择**：该不该调用，调用哪个；
- **参数生成**：字段、类型和约束是否正确；
- **任务分解**：多步任务的顺序是否合理；
- **结果利用**：是否真正依据工具结果回答；
- **停止判断**：何时继续调用，何时结束；
- **抗干扰能力**：是否会被工具返回中的恶意指令带偏。

一个模型可能几乎总能生成合法 JSON，却经常选错工具。

这属于模型行为质量，不属于 MCP 兼容性。

#### 第五层：产品与安全策略

平台还可能主动限制能力：

- 该模型只用于只读问答；
- 某类端点还没有通过安全评审；
- 远程 MCP Server 只允许白名单；
- 高风险写操作必须人工确认；
- 企业租户关闭了第三方连接；
- 灰度发布只开放给一部分用户。

此时技术上“可以接”，产品上仍会显示“不可用”。

### 为什么推理模型容易让人误以为“不兼容”

“推理模型”通常指会在回答前投入更多计算、进行多步分析的模型或运行模式。

它可能增加四类适配难度，但没有一类等于 MCP 禁止它使用工具。

#### 1. 输出被分成更多事件类型

流式 API 可能分别产生：

```text
推理状态 -> 工具调用开始 -> 参数增量 -> 工具调用完成 -> 最终文本
```

如果 Host 只认识“文本增量”，就可能把工具参数当普通文本，或者漏掉调用完成事件。

#### 2. 工具调用可能出现在不同阶段

有的端点允许模型在分析中间调用工具，有的只允许在某个阶段输出调用。

Host 若写死状态机，就会出现：

- 调用事件被忽略；
- 同一个调用被提交两次；
- 工具结果回传后无法继续；
- 最终文本抢先结束会话。

#### 3. 上下文与成本压力更明显

几十个 MCP Server 暴露几百个工具时，工具描述会占用大量上下文。

推理内容、历史消息和工具结果叠加后，可能造成：

- 工具定义被截断；
- 选择准确率下降；
- 延迟和费用增加；
- 工具结果没有空间回传。

解决办法是能力检索、按需暴露、压缩结果和会话管理，而不是断言协议不兼容。

#### 4. 平台需要额外做兼容性验证

新的事件模型、参数和终止原因都要经过 SDK、网关、日志、计费和安全链路验证。

产品晚一点开放 MCP，可能只是适配尚未完成。

### Function Calling 与 MCP 的关系

**Function Calling** 在这里泛指：模型 API 允许应用传入工具定义，并让模型返回结构化调用意图。

它解决的是：

```text
模型怎样告诉 Host：“我想调用哪个工具，参数是什么？”
```

MCP 解决的是：

```text
Host 怎样发现、连接并调用外部能力提供方？
```

常见组合是：

```text
模型 Function Calling
        ↓
Host 工具循环
        ↓
MCP Client
        ↓
MCP Server
```

二者不是同一个协议，也不是互相替代。

### 没有原生 Function Calling，还能不能用 MCP

可以，但要区分两种情况。

#### 情况一：确定性程序直接调用 MCP

例如用户点击“同步日历”按钮，应用不需要模型判断：

```text
按钮事件 -> 参数校验 -> MCP tools/call -> 展示结果
```

这完全可以使用 MCP，也完全不需要 Function Calling。

#### 情况二：希望模型自主选择 MCP Tool

如果 API 只返回普通文本，可以要求模型输出约定 JSON，再由 Host 解析：

```json
{
  "tool": "get_order",
  "arguments": {
    "order_id": "A1001"
  }
}
```

但这种方案通常比原生工具通道脆弱：

- 模型可能在 JSON 前后输出解释文字；
- 字段类型可能不合法；
- 无法可靠区分回答和调用；
- 流式拼接更复杂；
- 调用 ID、多调用和结果回传需要自己设计；
- 文本中的恶意内容可能伪造调用。

低风险原型可以尝试，生产写操作不应只依赖正则表达式解析。

如果 API 支持受约束解码或严格结构化输出，可以提升语法可靠性，但仍不会自动解决工具选择、授权和副作用安全。

### 一张表定位“不支持”的真实含义

| 观察到的现象 | 更可能的责任层 | 首要验证方式 |
| --- | --- | --- |
| Client 无法完成 `initialize` | MCP 协议或传输 | 不经过模型，直接跑协议测试 |
| `tools/list` 正常，但模型看不到工具 | Host 映射或模型 API | 打印最终发送给 API 的工具定义 |
| 模型返回一段“我要调用工具”的文字 | API 通道或模型行为 | 检查原始响应事件与结构化能力 |
| 参数流式输出后 JSON 不完整 | SDK 或事件状态机 | 保存原始事件流并离线重放 |
| 模型总选错工具 | 工具描述或模型能力 | 固定数据集评测工具选择准确率 |
| 调用成功，但模型无视结果 | 回传格式或结果利用能力 | 检查结果消息与后续请求 |
| 只有某个产品入口不能用 | 产品策略或端点差异 | 对比端点、SDK 和功能开关 |
| 高风险工具被拒绝 | 授权或审批策略 | 查看策略决策与审计日志 |

### 完整排查流程

#### 第一步：绕过模型验证 MCP

至少覆盖：

```text
连接
  -> initialize
  -> initialized
  -> tools/list
  -> tools/call
  -> 关闭
```

如果失败，记录：

- 协议版本；
- 传输类型；
- JSON-RPC id；
- HTTP 状态或进程退出码；
- Server 返回的错误对象；
- 会话 ID。

#### 第二步：验证 Schema 映射

准备最小工具：

```json
{
  "name": "add",
  "description": "计算两个整数之和",
  "inputSchema": {
    "type": "object",
    "properties": {
      "a": { "type": "integer" },
      "b": { "type": "integer" }
    },
    "required": ["a", "b"],
    "additionalProperties": false
  }
}
```

检查它被发送给模型 API 后有没有字段丢失、类型变化或名称重写。

#### 第三步：验证最小工具循环

测试六组样本：

1. 明确不需要工具；
2. 明确需要一个工具；
3. 参数缺失，需要追问；
4. 工具返回业务错误；
5. 工具超时；
6. 工具结果返回后继续回答。

#### 第四步：增加复杂场景

再测：

- 多个相似工具；
- 多工具串行；
- 允许时的并行调用；
- 长参数和长结果；
- 流式中断与取消；
- 重试和幂等；
- 工具列表变化；
- 上下文接近上限。

#### 第五步：验证安全边界

至少测试：

- 未授权工具不会暴露；
- 参数经过服务端校验；
- 用户 A 的会话不能使用用户 B 的凭证；
- 工具结果中的提示注入不会提升权限；
- 高风险操作需要明确确认；
- 断线重试不会重复扣款或发消息。

## 工程实践与边界

### 推荐的适配接口

不要让业务代码直接依赖某家模型的原始事件格式。

可以在 Host 内定义统一事件：

```text
TextDelta
ToolCallStarted
ToolArgumentsDelta
ToolCallCompleted
ToolResultRequired
ResponseCompleted
ResponseFailed
```

模型适配器负责把供应商事件转成统一事件，MCP 适配器负责把统一工具调用转成 `tools/call`。

这样更换模型或 MCP SDK 时，影响不会扩散到整个应用。

### 做能力探测，不要只看型号白名单

接入一个模型端点时，动态或离线验证：

| 能力 | 测试内容 |
| --- | --- |
| 工具定义 | 是否接受所需 Schema 子集 |
| 单工具调用 | 能否稳定返回名称、参数和调用 ID |
| 多工具调用 | 是否支持，事件顺序是否稳定 |
| 工具结果回传 | 能否继续生成并引用结果 |
| 流式 | 参数增量能否完整重组 |
| 错误处理 | 拒绝、超时、取消是否有明确状态 |

产品可以基于探测结果降级，而不是简单报“模型不支持 MCP”。

### 控制工具数量与描述质量

不要一次把所有 MCP Server 的所有工具塞给模型。

更稳妥的做法是：

1. 先按用户、租户和会话权限过滤；
2. 再按任务语义检索候选工具；
3. 去掉重复或高度相似工具；
4. 使用清晰、可区分的名称和描述；
5. 只发送当前回合真正可能使用的工具。

工具描述应该说明：

- 什么时候使用；
- 什么时候不要使用；
- 参数的业务含义；
- 是否有副作用；
- 可能的错误和前置条件。

### Schema 合法不等于业务合法

模型生成：

```json
{
  "amount": 1000000,
  "currency": "CNY"
}
```

可能完全符合 JSON Schema，却超出当前用户的退款额度。

所以执行前至少要有：

```text
结构校验
  -> 业务规则校验
  -> 身份与权限校验
  -> 风险审批
  -> 幂等检查
  -> 执行
```

模型的推理能力不能替代这些控制。

### 失败时要可降级

可根据风险分级：

- 只读查询失败：允许有限重试或转人工查询；
- 参数不确定：让用户补充，不要擅自猜；
- 工具不可用：明确告知，保留任务状态；
- 写操作状态不明：先查询结果，不要直接重放；
- 模型不支持工具通道：退回确定性工作流或只读回答；
- 安全策略拒绝：不能通过换模型绕过。

### 可观测性要贯穿两条链路

一次请求至少关联：

```text
用户请求 ID
  ├── 模型响应 ID / 工具调用 ID
  └── MCP JSON-RPC id / session id / 业务幂等键
```

应记录状态、耗时和错误类型，但对访问令牌、用户数据和工具结果做脱敏。

### 选型边界

| 场景 | 建议 |
| --- | --- |
| 固定按钮触发固定动作 | 直接程序调用 MCP，不必让模型决策 |
| 低风险、工具少的对话助手 | 原生工具通道 + MCP 适配 |
| API 只有文本输出的原型 | 可用严格结构化协议验证可行性 |
| 高风险写操作 | 原生工具通道仍需审批、授权和幂等 |
| 工具数很多 | 先做权限过滤与工具检索 |
| 端点事件格式频繁变化 | 建立统一适配层和契约测试 |

## 常见误区

### 误区一：MCP 运行在模型内部

不准确。MCP Client 通常运行在 Host 中，模型一般不直接维护 MCP 连接。

### 误区二：推理模型都不支持工具

不能按“推理模型”这个大类下结论。要看具体 API 端点、事件格式、Host 适配和模型行为评测。

### 误区三：能输出 JSON 就等于支持 MCP

JSON 语法正确只解决很小一部分问题。还需要调用 ID、结果回传、多轮状态、错误处理、授权和执行控制。

### 误区四：Host 接上 MCP SDK 就完成了适配

SDK 只处理部分协议工作。工具 Schema 映射、模型事件状态机、结果裁剪、安全和业务幂等仍由应用负责。

### 误区五：上下文越长，工具调用就越准

塞入更多工具可能增加混淆和成本。相关、清晰、可区分的候选集合通常比“全部暴露”更重要。

### 误区六：模型足够聪明，可以省掉参数校验

模型输出是不可信输入。服务端必须重新执行结构、业务、权限和风险校验。

### 误区七：写操作失败后再调一次就行

网络失败不代表业务没有执行。必须使用幂等键、状态查询或补偿机制，不能盲目重放。

### 误区八：产品标注“不支持”就是协议层不兼容

它也可能是 SDK、端点、灰度、安全策略或商业配置的限制。要查看具体组合与原始事件。

## 面试追问

### 追问一：如何一句话纠正这道题的前提

可以回答：

> MCP 主要是 Host 中的 Client 与 Server 之间的协议，不是大模型直接实现的“母语”；所谓模型不支持，通常是工具 API、Host 适配或模型工具使用能力还不能完成可靠闭环。

### 追问二：模型没有 Function Calling，是否完全不能使用 MCP

不是。

确定性程序仍可直接调用 MCP；如果想让模型自主选工具，可以用严格结构化输出做适配，但可靠性、安全性和状态管理成本更高。

### 追问三：如何区分 Host 适配问题和模型能力问题

先保存并检查原始 API 事件：

- 工具事件缺失或被 SDK 丢掉，偏向接口与适配问题；
- 工具事件格式正确，但在固定评测集中经常选错工具或参数，偏向描述与模型能力问题；
- 工具结果已正确回传，但模型忽略证据，属于结果利用能力问题。

### 追问四：为什么结构化输出不能完全替代 Function Calling

结构化输出可以约束返回 JSON 的形状，但 Function Calling 通道通常还明确区分调用与文本、提供调用 ID、支持结果关联和多轮工具循环。前者主要保证格式，后者定义交互语义。

### 追问五：怎样判断一个模型端点可以上线 MCP 工具

不能只跑一个演示。

要在代表性数据集上测试不调用、正确调用、参数缺失、多工具、工具失败、长结果、流式中断和注入攻击，并观察正确率、非法参数率、误调用率、P95 延迟、成本与高风险拒绝率。

### 追问六：推理时间长会带来什么工程问题

会放大总超时、用户取消、连接保持、成本预算和状态恢复问题。应把模型超时、工具超时和任务总截止时间分开，并让取消信号能传播到尚未开始或可中断的工具。

### 追问七：多个 MCP Server 有同名工具怎么办

Host 应使用稳定命名空间或内部唯一 ID，向模型提供可区分的名称与描述，并把模型看到的名称映射回正确 Server。不能只按展示名路由。

### 追问八：最容易被忽略的安全问题是什么

工具结果和 MCP Resource 都是不可信数据，可能包含提示注入。Host 不能把其中的指令当更高优先级规则，也不能让它们改变授权范围。

### 可直接背诵的完整答案

> 首先要纠正前提：MCP 不是通常由模型直接实现的协议。Host 里的 MCP Client 负责与 Server 初始化、发现和调用工具，模型只通过 Host 提供的工具通道表达调用意图。因此“某模型不支持 MCP”一般是一个产品层简称，可能指 API 没有结构化工具事件、SDK 或 Host 尚未适配流式状态、工具结果不能回传、模型工具使用准确率不足，或者安全策略没有开放。
>
> 我会按五层排查：第一，不经过模型验证 MCP 连接、初始化和 `tools/call`；第二，检查 MCP Tool 到模型工具 Schema 的映射；第三，检查 API 的调用 ID、参数流、结果回传和多轮语义；第四，用固定任务集评测模型选工具、填参数和利用结果；第五，检查授权、确认、幂等和产品开关。推理链长可能增加事件适配、上下文、延迟和超时难度，但不是 MCP 协议层不兼容。没有 Function Calling 时，确定性工作流仍能用 MCP；让模型自主选工具则需要额外的结构化适配，而且生产写操作必须保留服务端校验和安全控制。
