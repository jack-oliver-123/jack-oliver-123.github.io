一家综合医院有很多专科医生。

病人到院后，不会让所有医生同时完整会诊。

分诊台会根据症状，把病人交给少数相关科室。

这样医院可以拥有很大的总知识容量，但每位病人只消耗一部分计算资源。

Mixture of Experts，MoE，混合专家模型的直觉与此相似：

- 模型里放入许多专家子网络；
- Router 为每个 Token 选择少数专家；
- 只激活被选中的专家计算；
- 总参数可以很大，每 Token 激活参数较少。

但专家不是预先命名的“数学专家”“法律专家”。

它们是在训练中通过路由和梯度共同形成的神经网络子结构。

## 本篇目标

读完本篇，你应该能够：

- 解释稠密 Transformer 中 FFN 的作用；
- 说清 MoE 通常替换哪一部分；
- 逐符号解释 Router、softmax、Top-k 和专家加权；
- 区分总参数、激活参数和实际显存；
- 解释负载不均、专家塌缩和 capacity；
- 说明共享专家、细粒度专家和辅助损失的作用；
- 解释专家并行与 All-to-All 通信；
- 严谨描述 DeepSeek-V3 与 Qwen3 MoE 的公开配置；
- 分析 MoE 的训练、部署和业务选型边界。

先记住：

> MoE 用条件计算扩大参数容量，但模型权重仍要被存储和调度；激活参数少不等于部署显存只需激活参数大小。

## 60 秒回答

MoE 是条件计算架构。Transformer 的注意力层通常仍保持共享，部分 FFN 层被多个专家 FFN 替代。Router 根据每个 Token 的隐藏状态给专家打分，选择 Top-k 个专家执行，再按路由权重合并输出。因此模型可以拥有很大的总参数量，但单个 Token 只经过少数专家，理论上用接近较小稠密模型的每 Token 计算获得更大参数容量。

主要代价是路由和系统复杂度：热门专家可能过载、冷门专家学不到东西；分布式推理需要把 Token 发到专家所在设备，产生 All-to-All 通信；所有专家权重通常仍需驻留在集群内，所以显存、跨卡带宽和小 batch 利用率可能成为瓶颈。

DeepSeek-V3 技术报告公开的是 671B 总参数、每 Token 约激活 37B 参数，并使用 DeepSeekMoE 的细粒度专家、共享专家和无辅助损失负载均衡策略。Qwen 不是所有型号都用 MoE；Qwen3 技术报告同时提供稠密与 MoE 变体，例如 30B-A3B、235B-A22B，A3B/A22B 表示约激活参数规模。采用 MoE 的共同动机是扩大容量并控制每 Token 计算，但最终训练和部署收益取决于路由质量、通信、内核和负载分布。

## 详细解析

### 1. 先理解稠密模型

Dense Transformer 中，每个 Token 通常经过每一层的相同参数：

~~~text
Token
  ↓
Self-Attention
  ↓
Feed-Forward Network，FFN
  ↓
下一层
~~~

所有 Token 都使用同一组 FFN 权重。

因此称为稠密激活。

### 2. FFN 是什么

一个常见 FFN 可简化为：

$$
\operatorname{FFN}(x)
=
W_2\,\phi(W_1x)
$$

其中：

- $x$ 是 Token 的隐藏状态；
- $W_1$ 把隐藏维度扩展到更大中间维度；
- $\phi$ 是激活函数；
- $W_2$ 投影回隐藏维度。

现代模型可能使用 SwiGLU 等门控 FFN，公式更复杂。

FFN 往往占据模型参数的大部分。

因此 MoE 常把 FFN 换成专家集合，而不是复制完整 Transformer。

### 3. MoE 层长什么样

~~~text
                    ┌→ Expert 1 ─┐
Token hidden state ─┼→ Expert 2 ─┼→ 加权合并
        │           ├→ Expert 3 ─┤
        └→ Router   └→ ...       ┘
~~~

Router 为每个 Token 独立选择专家。

同一句话中的不同 Token 可能进入不同专家。

### 4. Router 打分

给定 Token 隐藏状态 $x$，线性 Router 可计算：

$$
g=W_rx
$$

其中：

- $W_r$ 是路由参数；
- $g\in\mathbb{R}^{E}$；
- $E$ 是专家数量；
- $g_i$ 是第 $i$ 个专家的 logit。

再做 softmax：

$$
p_i
=
\frac{\exp(g_i)}
{\sum_{j=1}^{E}\exp(g_j)}
$$

$p_i$ 表示 Router 对专家 $i$ 的相对权重。

Softmax 是便于入门的常见形式，不是所有 MoE 的固定实现。Router 也可以使用 Sigmoid affinity、Top-k 前后不同的归一化方式或额外偏置；后文 DeepSeek-V3 就采用 Sigmoid affinity，不能把本节公式直接当成它的逐项实现。

### 5. Top-k 路由

如果有 8 个专家，`top_k=2`：

1. 计算 8 个路由分数；
2. 选分数最高的 2 个；
3. 只运行这 2 个专家；
4. 对两份输出加权合并。

概念公式：

$$
y
=
\sum_{i\in\operatorname{TopK}(p,k)}
\tilde{p}_iE_i(x)
$$

逐个解释：

- $E_i(x)$ 是第 $i$ 个专家 FFN 输出；
- $\operatorname{TopK}(p,k)$ 返回最高的 $k$ 个专家；
- $\tilde p_i$ 是选中专家间重新归一化后的权重；
- $y$ 是 MoE 层输出。

### 6. 一个最小路由例子

Router 概率：

| 专家 | 概率 |
| --- | ---: |
| E1 | 0.40 |
| E2 | 0.30 |
| E3 | 0.20 |
| E4 | 0.10 |

Top-2 选择 E1 和 E2。

二者概率和为 0.70。

重新归一化：

$$
\tilde p_1=0.40/0.70\approx0.571
$$

$$
\tilde p_2=0.30/0.70\approx0.429
$$

输出：

$$
y=0.571E_1(x)+0.429E_2(x)
$$

未选中的 E3、E4 对这个 Token 不做 FFN 计算。

### 7. 条件计算的意义

假设有 $E$ 个等大的专家，每 Token 只激活 $k$ 个。

专家部分的计算比例粗略为：

$$
\frac{k}{E}
$$

但完整模型还有：

- Attention；
- Embedding；
- Router；
- 共享专家；
- 输出层；
- 通信和调度。

所以不能用 $k/E$ 直接推导整模型 FLOPs。

### 8. 总参数与激活参数

#### 总参数 Total Parameters

模型所有专家与共享层的参数总和。

影响：

- 模型文件；
- 集群权重显存；
- 加载时间；
- checkpoint 存储。

#### 激活参数 Activated Parameters

单个 Token 在一次前向中实际经过的参数规模近似值。

影响每 Token 计算量，但不是完整的硬件占用说明。

### 9. 为什么激活 37B 不等于只需存 37B

假设一个 MoE 总参数 671B，每 Token 约激活 37B。

服务时下一个 Token 可能路由到任意专家。

因此集群通常需要让所有专家权重可访问：

- 分散驻留在多张 GPU；
- 或在较慢层级按需加载；
- 或使用复杂的专家缓存与卸载。

如果专家不在设备上，加载延迟会抵消条件计算收益。

### 10. 专家会自然变成学科专家吗

不应机械理解。

专家可能按：

- Token 形态；
- 语言；
- 句法；
- 频率；
- 抽象特征；
- 训练阶段动态；

形成分工。

这种分工由优化产生，未必有人类可解释标签。

同一个专家也可能处理多种看似无关 Token。

### 11. 最大难题：负载不均

如果 Router 总把 Token 发给少数专家：

~~~text
Expert 1：50%
Expert 2：30%
Expert 3：2%
Expert 4：1%
……
~~~

会产生：

- 热门专家过载；
- 其他 GPU 等待；
- 冷门专家训练不足；
- 吞吐下降；
- Token 被丢弃或延迟；
- 专家塌缩。

### 12. Capacity Factor

早期或部分 MoE 实现为每个专家设置容量。

若 batch 中有 $N$ 个 Token，$E$ 个专家，每 Token 选 $k$ 个专家，平均分配量为：

$$
\frac{N\times k}{E}
$$

专家容量可近似：

$$
C
=
\left\lceil
\text{capacity factor}
\times
\frac{N k}{E}
\right\rceil
$$

容量过小：

- 热门专家溢出；
- 可能丢 Token 或走备用路径；
- 质量受影响。

容量过大：

- 预留和填充浪费；
- 利用率下降。

具体模型和框架可能采用不同策略。

### 13. 负载均衡辅助损失

Switch Transformer 一类方法加入辅助损失，鼓励：

- 各专家接收 Token 比例接近；
- Router 概率质量不要集中在少数专家。

概念上可写成：

$$
\mathcal{L}
=
\mathcal{L}_{\text{LM}}
+\lambda\mathcal{L}_{\text{balance}}
$$

其中：

- $\mathcal{L}_{\text{LM}}$ 是语言模型损失；
- $\mathcal{L}_{\text{balance}}$ 是负载均衡项；
- $\lambda$ 控制均衡与主任务的权衡。

辅助损失太强可能干扰模型自然选择更合适专家。

### 14. Router Z-loss

部分架构还对 Router logits 加正则，避免数值过大。

动机包括：

- 改善训练数值稳定性；
- 防止 softmax 过度饱和；
- 控制路由分数规模。

它与负载均衡损失目标不同。

### 15. 共享专家

共享专家对所有 Token 都执行。

路由专家只对被选中 Token 执行。

直觉：

- 共享专家学习通用知识；
- 路由专家学习更有差异的表示；
- 减少多个专家重复学习公共模式。

代价是共享专家增加每 Token 固定计算。

### 16. 细粒度专家

把一个大专家拆成多个较小专家，可以：

- 增加可组合的专家数量；
- 每 Token 选择多个细粒度专家；
- 形成更灵活的组合。

同时会增加：

- Router 选择数；
- 调度复杂度；
- 小矩阵计算和通信挑战。

DeepSeekMoE 的公开设计强调细粒度专家与共享专家隔离。

### 17. 专家并行 Expert Parallelism

模型太大时，把不同专家放在不同 GPU：

~~~text
GPU 0：Expert 0–7
GPU 1：Expert 8–15
GPU 2：Expert 16–23
GPU 3：Expert 24–31
~~~

Router 选择专家后，需要把 Token 隐状态发送到对应 GPU。

专家计算完成，再把输出发回原序列所在位置。

这叫 dispatch 和 combine。

### 18. All-to-All 通信

一个 batch 的 Token 可能被路由到所有设备。

每张 GPU 既要发送自己的 Token，也要接收其他 GPU 的 Token。

因此常出现 All-to-All 通信模式。

性能受：

- 网络带宽；
- 拓扑；
- Token 数；
- 每专家负载；
- 通信与计算重叠；
- 数据类型；
- 节点内与跨节点比例；

影响。

### 19. 为什么小 batch 可能利用率差

如果每次只有少量 Token：

- 分到每个专家的 Token 更少；
- 矩阵乘规模小；
- kernel 启动占比高；
- 设备间通信仍存在；
- 热门专家波动更明显。

高吞吐批处理能聚合更多 Token，但会增加排队延迟。

这与在线服务的延迟目标形成权衡。

### 20. Training 与 Inference 的差异

训练还要保存：

- 激活；
- 梯度；
- 优化器状态；
- Router 统计；
- 专家梯度通信。

推理不做反向传播，但要：

- 保存全部可访问权重；
- 管理动态路由；
- 优化专家并行；
- 处理 KV Cache；
- 支持连续批处理。

MoE 训练可扩展性好，不代表单机部署简单。

### 21. DeepSeek-V3 的公开配置

根据 DeepSeek-V3 技术报告：

- 总参数为 671B；
- 每 Token 激活参数约 37B；
- 采用 Multi-head Latent Attention，MLA；
- 采用 DeepSeekMoE；
- 使用更细粒度的路由专家；
- 包含共享专家；
- 路由 affinity 使用 Sigmoid，而不是本篇通用示例中的 Softmax；
- 报告提出 auxiliary-loss-free 的负载均衡策略；
- 还使用多 Token 预测训练目标。

模型能力不是由 MoE 单独造成。

数据、训练规模、优化、MLA 和后训练同样重要。

### 22. DeepSeek-V3 的专家数量要怎样表述

报告中的典型 MoE 层配置包括：

- 1 个共享专家；
- 256 个路由专家；
- 每个 Token 选择 8 个路由专家。

该版本先根据加入动态负载偏置后的 affinity 选择 Top-8；被选专家的合并权重仍由原始 Sigmoid affinity 归一化得到。偏置影响“选谁”，但不直接替代原始 affinity 成为合并权重。

面试中应说明这是该技术报告公开的具体配置，而不是所有 DeepSeek 模型的固定配置。

不同版本或层类型应以对应模型配置为准。

### 23. 无辅助损失负载均衡

传统辅助损失可能影响主任务优化。

DeepSeek-V3 报告描述的策略通过为各专家引入动态偏置项影响路由选择，按近期负载调整偏置，同时路由权重仍来自原始 affinity score。

直觉：

~~~text
专家过热 → 路由偏置下降
专家过冷 → 路由偏置上升
~~~

它仍需要监控和调节，不应理解为负载问题从此消失。

### 24. Qwen 是否都用 MoE

不是。

Qwen 是模型家族，包含不同规模和架构。

以 Qwen3 技术报告为例：

- 提供 Dense 模型；
- 也提供 MoE 模型；
- 公开命名包括 Qwen3-30B-A3B；
- 公开命名包括 Qwen3-235B-A22B。

对应官方模型配置还显示，这两个 Qwen3 MoE 变体均为 128 个路由专家、每 Token 选择 8 个专家，且 `shared_expert_intermediate_size` 为空；因此不能把 DeepSeek-V3 的“1 个共享专家 + 256 个路由专家”配置套到 Qwen3。

因此严谨说法是“Qwen 家族中的部分型号采用 MoE”。

### 25. A3B、A22B 是什么

这类命名中的 `A` 通常表示 activated。

例如：

- 30B-A3B：总参数约 30B，每 Token 激活规模约 3B；
- 235B-A22B：总参数约 235B，每 Token 激活规模约 22B。

它是参数规模描述，不等于：

- 只需 3B 或 22B 的权重显存；
- 实际 FLOPs 精确等于同规模 Dense；
- 延迟会与对应稠密模型相同。

### 26. 为什么这些模型使用 MoE

共同动机通常包括：

- 增加模型总容量；
- 控制每 Token 计算；
- 让不同 Token 使用条件化子网络；
- 在固定训练 FLOPs 下扩大参数规模；
- 配合大规模分布式训练与推理。

能否兑现收益取决于系统实现。

### 27. MoE 与 Dense 的对比

| 维度 | Dense | MoE |
| --- | --- | --- |
| 每 Token 参数路径 | 大部分参数固定参与 | 只激活部分专家 |
| 总参数存储 | 与计算规模直接相关 | 可远大于激活规模 |
| 路由 | 无专家路由 | 动态 Top-k 路由 |
| 通信 | 张量/流水线等 | 额外专家 All-to-All |
| 负载问题 | 相对规则 | 需处理专家不均衡 |
| 单机部署 | 通常更直观 | 大总权重更难放下 |
| 批量效率 | 较稳定 | 依专家 Token 聚合 |

### 28. MoE 也可能有专家冗余

即使专家数量很多，也可能：

- 多个专家学习相似功能；
- 少数专家主导；
- 路由对输入扰动敏感；
- 微调后专家分布漂移。

总参数多不等于有效容量按比例增长。

需要路由分析和消融实验。

## 工程实践与边界

### 1. 部署前先读模型配置

至少确认：

- 总参数；
- 激活参数；
- 层数；
- MoE 层位置；
- 路由专家数；
- 每 Token 选择数；
- 共享专家数；
- hidden size；
- 权重精度；
- KV 结构；
- 推理引擎支持状态。

不能只按模型名称中的 `A3B` 估显存。

### 2. 权重显存估算

粗略权重存储：

$$
M_{\text{weights}}
\approx
P_{\text{total}}\times B
$$

其中：

- $P_{\text{total}}$ 是总参数；
- $B$ 是每参数字节。

还要加：

- 量化 scale；
- 未量化层；
- KV Cache；
- 激活和工作区；
- 通信 buffer；
- 图捕获；
- 碎片。

### 3. 推理引擎支持

检查：

- 是否识别该 MoE 架构；
- 是否有 fused MoE kernel；
- 支持哪种量化；
- expert parallel 是否成熟；
- 是否支持跨节点；
- Router 和 shared expert 是否实现一致；
- tool calling、长上下文等上层能力是否兼容。

“能加载”不等于“高效”。

### 4. 并行策略

MoE 可能组合：

- Tensor Parallelism；
- Pipeline Parallelism；
- Data Parallelism；
- Expert Parallelism；
- Context Parallelism。

组合空间很大。

选择应基于：

- 单机 GPU 数；
- 节点内互联；
- 跨节点网络；
- 权重大小；
- KV 压力；
- 请求分布；
- 目标延迟与吞吐。

### 5. 监控路由

建议监控：

- 每专家 Token 数；
- 负载最大/平均比；
- 路由概率熵；
- Top-k 分布；
- Token drop；
- 每专家计算时间；
- All-to-All 字节与时延；
- 每节点等待时间；
- 不同任务和语言的路由差异。

平均吞吐可能掩盖个别专家热点。

### 6. 负载指标示例

假设每专家 Token 数为 $n_i$，平均为：

$$
\bar n
=
\frac{1}{E}\sum_{i=1}^{E}n_i
$$

最大负载比：

$$
R_{\max}
=
\frac{\max_i n_i}{\bar n}
$$

$R_{\max}$ 越大，说明最热专家相对平均越拥挤。

阈值应由具体系统压测确定。

### 7. 微调 MoE 的注意点

- 只训练 LoRA 时覆盖哪些专家层；
- Router 是否冻结；
- 领域数据是否让少数专家过热；
- 不同专家的梯度是否充分；
- adapter 与 expert parallel 是否兼容；
- 微调后是否破坏负载均衡；
- 保存和合并格式是否受引擎支持。

只看训练 loss 不够。

### 8. 量化 MoE 的注意点

专家权重占总参数大头，量化可显著降低存储。

但要确认：

- fused MoE kernel 支持该格式；
- Router 和共享层精度；
- 每专家 scale 布局；
- 小矩阵解包开销；
- 专家并行通信数据类型；
- 质量在不同专家负载下是否稳定。

### 9. 性能压测矩阵

至少覆盖：

- batch 1 低延迟；
- 高并发吞吐；
- 短输入长输出；
- 长输入短输出；
- 多语言；
- 结构化输出；
- 节点内与跨节点；
- 热门领域和长尾领域。

报告 TTFT、TPOT、吞吐、显存、网络和错误率。

### 10. 模型选型不应只看激活参数

还要看：

- 业务准确率；
- 总权重与部署拓扑；
- 延迟；
- 并发吞吐；
- 量化后质量；
- 上下文能力；
- 许可证；
- 生态与版本稳定性；
- 运营复杂度。

有时较小 Dense 模型的端到端成本更低。

### 11. 什么时候更适合 MoE

可能适合：

- 有多卡或多节点资源；
- 需要较大模型容量；
- 请求量足以形成有效批处理；
- 网络和引擎对专家并行优化良好；
- 团队能监控动态路由。

不利条件：

- 单卡边缘部署；
- 极低并发；
- 跨节点带宽弱；
- 引擎尚未优化该架构；
- 运维团队希望极简部署。

### 12. 面试回答的严谨措辞

推荐说：

> 根据 DeepSeek-V3 技术报告，该版本为 671B 总参数、约 37B 激活参数。

不要说：

> DeepSeek 都是 37B，Qwen 都是 MoE。

模型配置会随版本和型号变化。

### 13. MoE 的能力边界

MoE 架构本身不能证明：

- 模型知识更准确；
- 每个专家可解释；
- 推理在所有负载下都更快；
- 显存只需激活参数；
- 训练不会不稳定；
- 某个基准分数必定更高。

最终要看完整训练与系统评测。

## 常见误区

- **认为一个专家就是一个人工学科**：专家是训练形成的子网络，分工未必可解释。
- **认为 MoE 每次只加载被选专家**：主流高吞吐服务通常需要让全部专家权重在集群中可快速访问。
- **用激活参数估算模型文件**：存储主要取决于总参数。
- **认为 MoE 把整个 Transformer 都复制成专家**：常见做法主要替换部分 FFN 层。
- **认为 Top-2 表示每个请求选两个专家**：通常是每个 Token、每个 MoE 层分别路由。
- **认为专家越多越好**：路由、利用率、通信和训练难度也会增加。
- **认为负载均衡只影响速度**：专家训练不足和 Token overflow 也可能影响质量。
- **认为 Qwen 全系列都是 MoE**：Qwen 家族同时存在 Dense 和 MoE 型号。
- **把 A3B 当总参数 3B**：它通常表示约激活规模，总参数需看完整型号。
- **认为 MoE 在所有场景都比 Dense 延迟低**：All-to-All、小 batch 和权重访问可能抵消计算优势。
- **只看单卡能否加载**：大 MoE 的目标部署通常涉及专家并行和网络拓扑。
- **把某份技术报告配置推广到所有版本**：应核对具体模型配置和发布时间。

## 面试追问

### 问：MoE 的核心原理是什么

**答：** Router 根据每个 Token 的隐藏状态选择少量专家 FFN，只执行选中专家并加权合并，使总参数容量可以很大，而每 Token 计算保持稀疏。

### 问：为什么激活参数少，显存仍可能很大

**答：** 不同 Token 会访问不同专家，服务集群需要让全部专家权重可用。激活参数描述单 Token 计算路径，不描述完整模型权重存储。

### 问：MoE 最大的系统挑战是什么

**答：** 动态路由造成专家负载不均和跨设备 All-to-All 通信。热门专家会形成尾延迟，冷门专家又可能训练不足，因此需要均衡策略、专家并行和持续监控。

### 问：DeepSeek-V3 和 Qwen 为什么用 MoE

**答：** 这些具体 MoE 型号希望扩大总参数容量，同时控制每 Token 激活计算。需要强调 Qwen 不是所有型号都采用 MoE，并且实际收益还依赖训练、路由、通信和推理内核。

### 延伸问题

- Top-1 与 Top-2 路由如何权衡质量和计算？
- 共享专家解决什么问题？
- capacity factor 过小会发生什么？
- Router 辅助损失为什么可能伤害主任务？
- Expert Parallel 与 Tensor Parallel 有何区别？
- 如何判断某个专家已经过载或塌缩？
- MoE 量化为什么依赖 fused kernel？
