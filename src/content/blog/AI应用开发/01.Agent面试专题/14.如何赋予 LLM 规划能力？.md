## 本篇目标

读完这篇，你应该能够：

- 分清“推理过程”“执行计划”“固定 Workflow”三个概念。
- 把自然语言目标转换成包含步骤、依赖、输入输出和验收条件的结构化计划。
- 解释 Plan-and-Execute、边执行边规划、候选计划搜索和外部规划器的适用场景。
- 知道什么时候应该重规划，什么时候绝不能简单重做。
- 用计划有效率、无效步骤率、重规划率和端到端成功率评测规划能力。

## 60 秒回答

赋予 LLM 规划能力，不是只在 Prompt 里加一句“一步步思考”，而是让它围绕明确目标生成可执行、可校验的计划，并由外部运行时负责调度、观察和调整。计划至少要包含步骤、依赖、输入输出、请求的工具、验收条件、预算和失败处理；模型提出能力需求，运行时根据真实身份与策略决定是否授权。

常见方法有：用清晰约束、Few-shot 和结构化输出提高计划质量；采用 Planner-Executor 或 Plan-and-Execute 先生成高层计划再逐步执行；用 ReAct 在不确定环境中边观察边决定下一步；用 ToT、GoT 等搜索多个候选路径；在动作和约束可形式化的领域，把问题交给确定性规划器或求解器。

工程上还要校验计划 schema、依赖环、工具可用性、硬约束和副作用。当前置条件失效、关键结果未达验收或用户约束变化时才重规划；已经付款、退款或发送的步骤不能当作没发生。最终看端到端任务成功率和成本，不把计划写得长或思维过程写得多当成规划能力强。

## 详细解析

### 先说人话：规划就是“先想清楚怎样把目标变成一组可完成的步骤”

用户说：“帮我安排一次三天的北京出差。”这只是目标，不是计划。

一个可执行计划至少要回答：

- 成功是什么意思？
- 有哪些硬约束？
- 要做哪些步骤？
- 哪些步骤可以并行，哪些必须等待？
- 每一步需要什么输入和工具？
- 怎样知道这一步真的完成了？
- 失败后重试、换方案还是找人？

所以可以记住：

```text
目标 + 约束 + 步骤 + 依赖 + 验收 + 执行反馈 = 可落地的规划
```

### 先分清推理、计划和 Workflow

| 概念 | 回答的问题 | 特点 |
| --- | --- | --- |
| 推理（Reasoning） | “这个结论怎么得到？” | 可能只服务于当前判断，不一定可执行 |
| 计划（Plan） | “为了完成目标，接下来做什么？” | 有步骤、依赖、状态和验收 |
| Workflow | “程序预先规定必须怎样走？” | 路径由开发者定义，通常更确定 |

Chain-of-Thought（CoT，思维链）通过生成中间推理步骤帮助解决某些多步问题，但一段推理文字不自动等于计划。

例如：

> 用户预算有限，所以应该先查航班，再选酒店。

这是一段合理分析，却还不能被调度。程序需要更结构化的信息：

```json
{
  "step_id": "S1",
  "action": "search_flights",
  "dependencies": [],
  "inputs": ["origin", "destination", "date", "budget"],
  "expected_output": "FlightCandidates",
  "acceptance_criteria": ["至少一个符合时间和预算的候选"]
}
```

生产系统通常需要模型输出简洁的计划、依据和可验证产物，而不是公开或依赖模型完整的私有思维过程。

### 贯穿案例：规划一次三天北京出差

用户要求：

> 下周去北京出差三天，从杭州出发，总预算 5000 元；周二上午 10 点前要到客户办公室；不要凌晨航班；酒店离客户公司 5 公里内。订票前必须让我确认。

#### 第一步：目标建模

先把模糊自然语言整理成明确状态：

```json
{
  "goal": "生成并在确认后预订北京三日出差方案",
  "hard_constraints": {
    "origin": "杭州",
    "destination": "北京",
    "duration_days": 3,
    "arrival_deadline": "周二 10:00 前",
    "max_total_budget_cny": 5000,
    "forbid_overnight_flight": true,
    "hotel_max_distance_km": 5,
    "booking_requires_confirmation": true
  },
  "success_criteria": [
    "时间满足会议要求",
    "总价不超过预算",
    "用户确认后才预订",
    "所有预订返回有效凭证"
  ]
}
```

目标建模非常重要。如果模型没有理解“10 点前到客户办公室”包含机场到办公室的交通时间，它即使找到 9:30 落地的航班，也可能无法按时到达。

#### 第二步：任务分解

```text
S1 补齐客户公司地址和准确日期
  ↓
S2 查询符合到达要求的去程航班 ─┐
S3 查询酒店与价格 ─────────────┼→ S5 组合候选并计算总预算
S4 查询机场到公司的交通时间 ───┘
                                      ↓
                               S6 让用户确认方案
                                      ↓
                         S7 预订航班与酒店并验证凭证
```

其中 S2、S3、S4 在输入齐全后可以并行；S5 依赖它们；S7 依赖用户确认。

#### 第三步：执行和观察

调度器只执行依赖已满足的步骤。每次工具返回后，程序更新状态：

```text
计划步骤：查询去程航班
工具观察：符合时间的航班全部超过 2500 元
验收结果：有候选，但整体预算风险高
下一步：继续查询酒店后再组合，或生成替代交通方案
```

模型不能只说“查好了”，程序应检查返回结构、候选数量、时间和价格。

#### 第四步：必要时重规划

用户在确认前说：“预算可以增加到 6000 元，但酒店必须含早餐。”这改变了硬约束，需要更新目标状态，并重新执行受影响节点：酒店搜索和方案组合。已经正确的客户地址查询不必重做。

这叫局部重规划，而不是把整个任务从头开始。

### 规划系统的三个层次

#### 1. 目标层：定义成功和边界

包括目标、硬约束、软偏好、资源、预算、截止时间和不可接受结果。

- **硬约束**：违反就不可接受，例如“订票前必须确认”。
- **软偏好**：尽量满足，但必要时可以权衡，例如“优先靠窗座位”。

硬约束应存到模型外部状态，由程序校验。重规划时模型不能偷偷删除它们。

#### 2. 计划层：生成任务图

把目标拆成带依赖的步骤。每个步骤建议包含：

```json
{
  "step_id": "S2",
  "description": "查询满足到达时间的去程航班",
  "dependencies": ["S1"],
  "inputs": ["origin", "date", "arrival_deadline"],
  "requested_tools": ["flight_search"],
  "expected_output": "FlightCandidates",
  "acceptance_criteria": [
    "到达后有足够时间前往客户办公室",
    "不是凌晨航班",
    "价格字段和来源完整"
  ],
  "failure_policy": "replan",
  "status": "pending"
}
```

`requested_tools` 只是模型请求使用的能力，不代表已经授权。服务端工具 allowlist、用户身份和审批策略才是权限边界。

#### 3. 执行层：调度、验收和恢复

执行器负责：

- 找出依赖已完成的步骤；
- 在权限和预算内调用工具；
- 校验工具结果与验收条件；
- 保存检查点；
- 标记成功、失败、未知或等待审批；
- 触发局部重试、重规划或人工接管。

这一步让“模型写的一段计划”变成真正的 Agent 系统。

### 赋予规划能力的常见方法

### 方法一：清晰指令、Few-shot 与结构化输出

`Few-shot` 是在提示中给模型少量正确示例。可以展示怎样把一个目标拆成步骤，以及什么叫合格的验收条件。

配合 JSON Schema 或其他结构化输出，可以减少漏字段和格式漂移。它是最便宜的起点，适合结构相对清楚的任务。

局限是：格式正确不代表计划可行。程序还要检查依赖、工具和约束。

### 方法二：Plan-and-Execute / Planner-Executor

将规划和执行分开：

```text
Planner 生成高层计划
  ↓
Executor 逐步执行
  ↓
结果与观察返回 Planner
  ↓
继续、局部修改或结束
```

优点是全局目标更清楚，计划可以提前校验和审批；缺点是环境变化后，早期计划可能过时。

适合：研究报告、数据分析、软件改造等可以先形成高层任务图的工作。

### 方法三：边执行边规划，例如受限 ReAct

ReAct 把推理与动作交替进行。工程上可以理解为：模型根据当前观察只决定下一步，而不是一次写完全部细节。

```text
观察当前状态 → 选择下一步动作 → 执行 → 获得新观察 → 再选择
```

优点是能适应工具结果和环境变化；缺点是容易短视、循环或偏离总目标。因此常见做法是保留一个高层计划，再让 ReAct 只负责当前步骤内的低层动作。

### 方法四：候选计划搜索，ToT 与 GoT

普通线性生成往往产生一条路径后一路写下去。Tree of Thoughts（ToT，思维树）让系统生成多个候选中间状态，评价、剪枝，并在必要时回退。Graph of Thoughts（GoT，思维图）允许中间结果形成更一般的图，还能合并多个候选结果。

这类方法适合：

- 需要探索多个方案；
- 初始选择对最终结果影响很大；
- 有较可靠的评价函数；
- 任务价值足以承担额外模型调用。

如果评价函数本身不可靠，多生成十个方案并不会自动找到正确答案，只会增加成本。

### 方法五：外部规划器或求解器

在动作、前置条件和目标可以形式化的领域，可以让 LLM 负责理解自然语言并转换问题，再由确定性工具求解。

例如排班问题：

- 员工 A 周一不可用；
- 每个班次至少两人；
- 每人每周最多五个班次；
- 技术负责人必须在场。

这些约束适合交给约束求解器、优化器或经典规划器。LLM 擅长理解用户表达，求解器擅长搜索满足硬约束的方案。生成后还应把结果转换回可读说明，并验证转换没有遗漏约束。

### 方法六：检索历史计划和工具说明

规划失败有时不是模型不会拆分，而是不知道当前有哪些工具、工具前置条件是什么。可以按任务检索：

- 相似任务的已验证计划；
- 工具能力、参数和限制；
- 业务规则与失败案例；
- 当前资源和可用性。

历史计划只能作为参考，因为旧工具、价格和政策可能已变化。

### 高层计划与低层动作怎样配合

一个实用的分层设计是：

```text
高层计划：目标、里程碑、依赖、验收、预算
        ↓
当前步骤执行器：在有限工具中选择具体动作
        ↓
工具观察与状态更新
        ↓
仍在边界内？继续当前步骤
偏离边界？返回 Planner 重规划
```

高层计划不要过早规定每一个点击和 API 参数，否则环境稍有变化就全部过期。低层执行也不能随意改目标和硬约束。

### 什么时候应该重规划

常见触发条件包括：

- 前置条件不再成立；
- 关键工具不可用或数据源变化；
- 结果没有达到验收条件；
- 预算、时间或步骤即将耗尽；
- 用户新增或修改约束；
- 外部世界状态发生变化；
- 计划出现无法满足的依赖。

普通步骤成功后，不必每次都重新生成完整计划。过度重规划会增加成本，也可能让已经正确的部分反复变化。

### 重规划最危险的边界：真实副作用

如果系统已经完成付款、退款、发送邮件或创建工单，这些动作不会因为模型重写计划就消失。

计划状态必须区分：

```text
planned     已计划但未执行
running     正在执行
succeeded   已由外部证据确认成功
failed      已确认失败
unknown     请求发出但结果不确定
compensated 已执行明确补偿
```

`unknown` 不能直接当 `failed` 重试。先查询外部状态或使用幂等键，否则可能重复扣款。

## 工程实践与边界

### 计划生成后的四层校验

#### 1. 结构校验

- 字段是否齐全；
- ID 是否唯一；
- 状态值是否有效；
- 输入输出类型是否匹配。

#### 2. 图校验

- 是否存在循环依赖；
- 是否有引用不存在的步骤；
- 是否有因依赖缺失而无法开始的节点；
- 节点数、深度和并行扇出是否超限。

#### 3. 能力与权限校验

- 请求的工具是否存在；
- 当前用户和 Agent 是否有权请求；
- 参数范围是否符合策略；
- 高风险步骤是否需要人工审批。

#### 4. 业务校验

- 硬约束是否全部覆盖；
- 验收条件是否可测；
- 总预算和截止时间是否可行；
- 是否考虑已发生的外部状态。

### 五个典型失败场景

#### 场景一：计划看起来完整，但不能执行

模型写“预订最佳航班”，却没有航班日期、工具和验收条件。

处理：使用结构化 schema；执行前检查必需输入；缺失时先生成澄清步骤。

#### 场景二：无限拆分

模型把“查航班”继续拆成“理解航班”“研究查询”“思考结果”等没有产出的步骤。

处理：要求每步有外部可观察产物；限制图深度和节点数；合并无法独立验收的步骤。

#### 场景三：计划违反权限

模型为完成目标加入“读取其他公司的订单”或“绕过审批”。

处理：模型只能请求工具，运行时根据可信身份和策略授权；任何重规划都不能修改权限规则。

#### 场景四：重规划后重复付款

付款请求超时，模型将步骤标记失败并重新执行。

处理：状态标为 `unknown`；通过幂等键或查询支付系统确认；不可确认时转人工，而不是盲目重试。

#### 场景五：多候选搜索成本失控

ToT 为每一步生成大量分支，每个分支又继续展开。

处理：限制分支、深度、模型调用和总费用；先用便宜规则剪枝；只有高价值且评价函数可靠的任务才使用搜索。

### 如何做计划审批

对低风险只读任务，可以自动执行经过校验的计划。对付款、退款、删除、外部发送等动作，审批内容至少包括：

- 要达到的目标；
- 具体对象和参数范围；
- 预计影响；
- 哪些步骤已经完成；
- 审批后计划还能修改到什么程度。

如果执行中超出已批准范围，应重新审批，不能拿旧审批覆盖新计划。

### 如何评测规划能力

- **计划有效率**：生成的计划有多少通过结构、图和约束校验；
- **可执行步骤率**：步骤是否有输入、工具、产物和验收；
- **无效步骤率**：重复、无产出或不必要步骤占比；
- **重规划率**：多少任务需要修改计划，以及为什么；
- **约束违反率**：硬约束被遗漏或破坏的比例；
- **端到端成功率**：最终目标是否真的完成；
- **恢复成功率**：工具失败或服务重启后能否继续；
- **成本与延迟**：计划带来的额外调用是否值得。

一份很长、很漂亮的计划，如果执行不了或违反约束，规划得分应该很低。

### 如何选方法

| 场景 | 优先方法 |
| --- | --- |
| 任务结构清楚、步骤较少 | 结构化计划 + 程序校验 |
| 可先形成里程碑，执行中会变化 | Planner-Executor + 局部重规划 |
| 环境高度不确定，需要频繁观察 | 高层计划 + 受限 ReAct |
| 需要探索多条路线，且可可靠评分 | ToT/GoT 或候选搜索 |
| 约束与动作可形式化 | LLM 做转换 + 外部规划器/求解器 |
| 高风险且流程固定 | 确定性 Workflow，模型只做局部辅助 |

## 常见误区

- **“在 Prompt 中说一步步思考，就有规划能力”**：这可能改善某些推理，却没有任务状态、依赖、验收和执行控制。
- **“CoT 就是计划”**：CoT 是中间推理组织方式，计划是可调度、可检查、可恢复的执行结构。
- **“计划越详细越好”**：过早锁定低层细节会让计划更快过期，高层稳定、低层按观察调整通常更实用。
- **“ToT、GoT 总比线性计划好”**：没有可靠评价函数时，多路径只会扩大成本和噪声。
- **“模型写了 requested_tools 就获得权限”**：它只是在请求能力，真正授权必须由运行时策略决定。
- **“重规划就是从头再做一遍”**：应识别受影响节点，保留已验证结果，并尊重已发生的外部副作用。
- **“能生成计划就代表完成任务”**：最终仍要以外部结果和验收条件判断成功。

## 面试追问

> **面试官：** 怎样赋予 LLM 规划能力？
>
> **候选人：** 先明确目标和硬约束，让模型输出带步骤、依赖、输入输出、工具和验收条件的结构化计划，再由外部执行器校验、调度、观察和按需重规划。

> **面试官：** CoT 和规划有什么区别？
>
> **候选人：** CoT 组织中间推理，主要回答“怎么得到结论”；规划回答“为了完成目标下一步做什么”，还需要可调度步骤、依赖、状态和验收。

> **面试官：** 高层计划和低层动作如何分工？
>
> **候选人：** 高层计划保存目标、里程碑、依赖、验收和预算；执行器根据当前观察选择有限的低层动作，偏离边界时返回 Planner。

> **面试官：** 什么时候触发重规划？
>
> **候选人：** 前置条件失效、工具不可用、结果未达验收、用户约束变化或预算即将耗尽时触发。普通成功步骤不必重复规划。

> **面试官：** 怎样发现计划生成错误？
>
> **候选人：** 依次做 schema 校验、依赖图校验、工具和权限校验、硬约束与预算校验；关键计划再通过模拟、规则或人工审批。

> **面试官：** 为什么不总用 ToT 搜索更多计划？
>
> **候选人：** 多分支会增加调用和成本，效果依赖评价函数。只有任务价值高、确实需要探索，而且候选能可靠评分时才值得。

> **面试官：** 已执行的步骤在重规划时怎么办？
>
> **候选人：** 以外部真实状态为准。已成功副作用必须保留；结果未知先查询或依赖幂等；需要撤销时走预先定义的补偿或人工流程。
