想象一家快递公司要处理客户留言。

传统做法像一条分工明确的流水线：先用一个模型判断“投诉还是咨询”，再用另一个模型抽取订单号，最后由固定模板生成回复。每个工位只做一件事，输出字段也早已规定好。

大语言模型更像一位读过大量材料的综合客服。你可以用自然语言告诉它：“判断诉求、提取订单号，并写一段礼貌回复。”它会通过同一个文本生成接口完成多项任务。不过，这位客服只是根据上下文续写最可能出现的内容，不会天然知道公司数据库中的实时事实，也不保证每句话都正确。

这就是 LLM 的能力来源与工程边界。

## 本篇目标

读完本篇，你应该能够：

- 用一句话解释什么是大语言模型；
- 理解 NLP、语言模型、预训练模型、基础模型和聊天模型之间的关系；
- 看懂自回归语言建模的概率公式与交叉熵损失；
- 解释“预测下一个 Token”为什么能产生问答、摘要和代码能力；
- 从任务接口、数据、成本、可控性和适用场景比较 LLM 与传统 NLP；
- 说明参数更多并不直接代表更强，“大”也没有统一参数门槛；
- 回答幻觉、知识时效、上下文学习和微调等常见追问；
- 在真实项目中判断应使用 LLM、小模型、规则，还是组合方案。

先记住核心结论：

> 大语言模型是在大规模数据上训练的通用语言模型。当前主流生成式 LLM 通常把文本表示成 Token，并反复预测下一个 Token；它能用统一的生成接口适配很多任务，但输出本质上仍是概率生成结果，不自动等于事实、权限或业务执行结果。

## 60 秒回答

大语言模型，英文是 Large Language Model，简称 LLM，是在大规模语料和计算资源上训练的参数化语言模型。当前常见的生成式 LLM 多采用 Transformer Decoder 架构，通过自回归目标学习：给定前面的 Token，预测下一个 Token 的概率分布。

它和传统 NLP 模型的主要差别不只是参数量，而是能力组织方式。传统 NLP 系统通常围绕分类、命名实体识别、机器翻译等单一任务设计，输入输出固定，常依赖任务标注数据；LLM 先通过大规模预训练学习通用语言规律，再通过 Prompt、上下文示例、监督微调或偏好优化适配多种任务，很多任务共享同一个生成接口。

但两者不是简单的新旧替代关系。LLM 灵活、迁移成本低，适合开放式理解和生成；规则或专用小模型通常延迟更低、成本更可控、输出更稳定，适合高吞吐和封闭标签任务。工程上要根据业务评测比较效果、延迟、成本、隐私和可控性，必要时组合使用。

另外，流畅不代表正确。预训练目标主要奖励对文本分布的拟合，不直接保证事实准确、实时知识、权限判断或安全执行，因此生产系统还需要检索、工具、校验、监控和人工兜底。

## 详细解析

### 1. 先把几个容易混淆的术语分开

#### NLP 是一个领域，不是一种具体模型

NLP 是 Natural Language Processing，即自然语言处理，研究计算机怎样处理人类语言。它包含很多任务：

- 文本分类；
- 情感分析；
- 命名实体识别；
- 信息抽取；
- 搜索与排序；
- 机器翻译；
- 摘要；
- 问答；
- 对话和文本生成。

因此，“LLM 和 NLP 有什么区别”这个问法并不严格。更准确的说法是：

> LLM 是实现 NLP 任务的一类模型，传统任务专用模型也是实现 NLP 的方式。

#### 语言模型学习“文本出现的概率”

语言模型的目标是为 Token 序列建模。例如：

```text
今天天气很 ____
```

模型可能给出：

```text
P(好) = 0.42
P(冷) = 0.21
P(热) = 0.13
P(贵) = 0.0001
```

它没有先在代码里写死“天气后面只能接形容词”，而是从训练数据中学习上下文与后续 Token 的统计关系。

#### 预训练模型是先学习通用规律，再适配任务

传统监督学习常从某个具体任务开始：

```text
输入：这个商品太差了
标签：负面
```

预训练则先使用规模更大的通用数据学习语言表示或生成能力，再用少量任务数据、Prompt 或偏好数据适配下游任务。

#### 基础模型强调可复用性

“基础模型”通常指在广泛数据上训练、能适配许多下游任务的模型。LLM 是基础模型的重要类别，但基础模型还可以处理图像、音频或多模态数据。

#### 聊天模型不等于原始基础模型

一个常见产品链路是：

```text
预训练基础模型
    ↓ 监督微调（学习指令和对话格式）
指令模型
    ↓ 偏好优化、安全训练、系统规则
聊天模型
```

聊天模型的礼貌风格、拒答行为和指令遵循能力，很多来自后训练，而不只是预训练。

### 2. 文本为什么要先变成 Token

神经网络不能直接接收“苹果很好吃”这几个抽象汉字。Tokenizer 会把文本转换为离散编号：

```text
文本：苹果很好吃
Token：['苹果', '很', '好吃']
ID：[3812, 96, 7421]
```

真实切分由词表和分词算法决定，同一句话在不同模型中可能得到不同结果。

每个 Token ID 会通过嵌入矩阵映射为向量：

$$
\mathbf{e}_t = E[x_t]
$$

逐个解释符号：

- $x_t$：序列第 $t$ 个 Token 的整数 ID；
- $E$：可训练的嵌入矩阵；
- $E[x_t]$：取出矩阵中对应 Token 的那一行；
- $\mathbf{e}_t$：该 Token 的连续向量表示。

模型内部处理的是这些向量，最后再把输出概率映射回词表中的 Token。

### 3. 自回归语言模型到底在学什么

假设 Token 序列是：

$$
x_1,x_2,\ldots,x_T
$$

根据概率的链式法则，整个序列的联合概率可以写成：

$$
P(x_1,\ldots,x_T)
=
\prod_{t=1}^{T}P(x_t\mid x_{<t})
$$

其中：

- $T$：序列长度；
- $x_t$：第 $t$ 个 Token；
- $x_{<t}$：第 $t$ 个 Token 之前的全部 Token；
- $P(x_t\mid x_{<t})$：给定前文后，下一个 Token 是 $x_t$ 的条件概率；
- $\prod$：把每个位置的条件概率相乘。

例如序列“我 喜欢 编程”：

$$
P(我,喜欢,编程)
=P(我)\cdot P(喜欢\mid 我)\cdot P(编程\mid 我,喜欢)
$$

训练时通常最小化负对数似然，也可以理解为逐 Token 交叉熵：

$$
\mathcal{L}
=
-\frac{1}{T}\sum_{t=1}^{T}\log P_\theta(x_t\mid x_{<t})
$$

其中：

- $\mathcal{L}$：训练损失，通常越低表示对训练分布预测得越好；
- $\theta$：模型的全部可训练参数；
- $P_\theta$：由参数 $\theta$ 决定的概率分布；
- $\log$：对数函数，把概率乘积转为求和并放大低概率错误；
- $\sum$：累加所有预测位置的损失；
- $1/T$：对序列长度取平均。

如果正确 Token 的概率越高，$-\log P$ 越小；如果正确 Token 的概率很低，损失就很大。反向传播会根据损失调整参数。

### 4. “只预测下一个 Token”为什么能学会很多任务

训练语料里不只有普通句子，还包含问答、教程、表格、代码、推导和对话。为了在这些文本中持续预测正确的后续内容，模型会受益于学习：

- 词法和语法规律；
- 常见实体与事实关联；
- 文档和代码结构；
- 问题与答案的对应模式；
- 示例中输入和输出的映射；
- 某些可复用的推理模式。

例如训练数据经常出现：

```text
2 + 3 = 5
7 + 4 = 11
```

那么面对：

```text
6 + 8 =
```

模型不仅可以记住固定字符串，也可能学习到可泛化模式。

但必须注意，“为了预测而形成有用内部表示”不等于模型拥有和人相同的理解机制。仅凭流畅输出，也无法证明模型建立了稳定的事实模型或因果模型。

### 5. 模型生成答案时发生了什么

推理阶段是一个循环：

```text
输入 Prompt
  ↓
计算下一个 Token 的概率
  ↓
按解码策略选择一个 Token
  ↓
把新 Token 追加到上下文
  ↓
继续预测，直到停止条件满足
```

概念伪代码：

```python
prompt_tokens = tokenizer.encode(prompt)
tokens = list(prompt_tokens)

while len(tokens) < max_length:
    logits = model(tokens)
    probabilities = softmax(logits[-1])
    next_token = sample(probabilities)
    tokens.append(next_token)

    if next_token == EOS_TOKEN:
        break

generated_tokens = tokens[len(prompt_tokens):]
answer = tokenizer.decode(generated_tokens)
```

这段代码只解码新生成部分，避免把输入 Prompt 重复放进答案。它隐藏了 KV Cache、批处理、并行计算和采样参数等工程细节，但展示了自回归生成的核心循环。

### 6. 传统 NLP 系统一般怎样工作

以客服工单为例，传统方案可能是：

```text
文本清洗
  ↓
意图分类模型
  ↓
实体抽取模型
  ↓
规则引擎
  ↓
回复模板
```

每个组件都有固定契约：

```json
{
  "intent": "refund",
  "order_id": "A1024",
  "confidence": 0.97
}
```

优点是容易验证、延迟低、行为边界清楚。缺点是新增任务时经常要重新收集标注、训练模型并改流水线。

### 7. LLM 系统一般怎样工作

LLM 方案可能把任务描述放进 Prompt：

```text
请分析工单：
1. 判断用户意图；
2. 提取订单号；
3. 写一段不超过 80 字的回复。
```

模型可以直接生成多个结果，也可以通过结构化输出返回字段。新增相近任务时，可能先修改指令和示例，而不必立即训练新模型。

不过，真实系统仍常组合多个组件：

```text
规则过滤 → LLM 理解 → 数据库工具 → 业务校验 → LLM 表达
```

LLM 不是整个系统，只是其中一个概率组件。

### 8. LLM 与传统任务模型的系统比较

| 维度 | 传统任务专用模型 | 生成式 LLM |
| --- | --- | --- |
| 训练目标 | 常直接优化分类、标注等任务目标 | 常先优化通用语言建模目标 |
| 任务接口 | 每个任务有固定输入输出 | 多任务共享文本或消息生成接口 |
| 适配方式 | 特征、标注数据、单独训练 | Prompt、上下文示例、检索、工具、微调 |
| 输出空间 | 通常封闭，如固定类别 | 开放式 Token 序列 |
| 数据需求 | 依赖任务标注数据 | 预训练依赖大规模语料，适配可少样本起步 |
| 确定性 | 较容易约束和校准 | 生成路径复杂，需额外约束与验证 |
| 推理成本 | 小模型通常更低 | 参数量、上下文和生成长度会增加成本 |
| 可解释接口 | 输出字段往往明确 | 自然语言灵活，但错误边界更开放 |
| 新任务速度 | 往往要改数据与训练流程 | 可先通过 Prompt 快速验证 |
| 典型优势 | 高吞吐、封闭任务、稳定决策 | 开放理解、生成、跨任务迁移 |

这里说的是常见趋势，不是适用于所有系统的硬性划分。现代 NLP 小模型也可能经过预训练，LLM 也可以被约束成分类器。

### 9. “大”到底是什么意思

业界没有统一标准规定超过多少参数才叫 LLM。原因包括：

- 参数量只是能力影响因素之一；
- 数据数量和质量很重要；
- 训练计算量和训练配方很重要；
- 稠密模型与 MoE 的“总参数”和“激活参数”不同；
- 小模型经过蒸馏或高质量训练也可能超过更大的旧模型；
- 同样参数量在不同架构、词表和上下文长度下成本不同。

因此面试中不要回答“超过 10 亿参数就是大模型”之类的固定门槛。更稳妥的说法是：它是一个相对概念，通常强调规模化预训练与通用迁移能力。

### 10. 参数是什么，参数多为何可能有帮助

神经网络参数可以理解为训练中被调整的数值。最简单的线性层是：

$$
\mathbf{y}=W\mathbf{x}+\mathbf{b}
$$

其中：

- $\mathbf{x}$：输入向量；
- $W$：权重矩阵；
- $\mathbf{b}$：偏置向量；
- $\mathbf{y}$：输出向量。

Transformer 中有大量嵌入、注意力和前馈网络权重。更多参数可以提供更大的函数表示空间，但是否转化为能力还取决于训练数据、优化、计算分配和评测任务。

### 11. 什么是上下文学习

上下文学习是模型仅根据当前 Prompt 中的说明或示例调整行为，而不修改权重。

```text
示例 1：好评 → POSITIVE
示例 2：太糟糕了 → NEGATIVE
待判断：物流很快 →
```

模型可能续写 `POSITIVE`。这叫 few-shot prompting，而不是一次新的梯度训练。

区别如下：

| 方式 | 是否修改权重 | 信息持续多久 |
| --- | ---: | --- |
| Prompt 指令 | 否 | 当前请求或会话 |
| 上下文示例 | 否 | 当前上下文 |
| RAG 检索 | 否 | 当前请求引用到的材料 |
| 微调 | 是 | 写入新模型参数或适配器 |

### 12. 为什么 LLM 会“幻觉”

自回归模型优化的是下一个 Token 的似然，不是一个内置数据库事务。出现错误内容的原因可能包括：

- 训练数据中缺少、冲突或过时的信息；
- Prompt 信息不足或问题本身含糊；
- 模型从相似模式错误泛化；
- 采样引入不同生成路径；
- 上下文太长，关键信息利用不足；
- 后训练鼓励“有帮助地回答”，却没有足够的“不知道”样本；
- 外部检索结果不相关或被错误引用。

“模型就是随机鹦鹉，所以必定产生幻觉”过于简单；“扩大模型就能彻底消除幻觉”同样没有依据。工程上应按任务风险组合检索、引用、工具、约束、验证和人工审核。

### 13. 最小可运行的任务选型思路

假设要识别 20 个固定工单类别：

1. 先定义准确率、召回率、P95 延迟、单次成本和拒识要求；
2. 建立代表真实分布的测试集；
3. 测规则基线和小分类模型；
4. 测零样本或少样本 LLM；
5. 测“规则/小模型处理简单样本，LLM 处理长尾”的路由方案；
6. 用同一测试集比较，而不是凭演示效果选型。

最终方案可能并不是全量使用 LLM。

## 工程实践与边界

### 1. 把模型选型变成指标问题

至少同时评估：

- 任务质量：准确率、召回率、人工评分或任务成功率；
- 延迟：首 Token 延迟和完整响应延迟；
- 吞吐：单位时间处理请求数或 Token 数；
- 成本：推理、检索、人工审核和维护成本；
- 可控性：格式遵循、拒答、稳定性和版本回归；
- 隐私：数据能否发送到外部服务，日志如何脱敏；
- 可运维性：监控、回滚、限流和降级能力。

### 2. 不要把模型知识当实时数据库

模型参数中的知识具有时间边界，也无法保证精确记忆。涉及实时或权威信息时，应访问受控数据源：

```text
用户问题
  ↓
检索或工具查询
  ↓
获得当前数据
  ↓
LLM 组织答案
  ↓
引用和校验
```

### 3. 开放生成必须配确定性验证

如果业务需要 JSON、金额、订单号或动作参数：

- 使用明确 Schema；
- 服务端再次解析和校验；
- 检查权限和业务状态；
- 高风险动作要求确认；
- 不把“模型说执行成功”视为真实执行结果。

### 4. 评测集必须来自真实业务

公开榜单可以帮助了解模型能力，但不能替代业务评测。测试集应覆盖：

- 高频正常样本；
- 长尾表达；
- 输入缺失和冲突；
- 超长文本；
- 不应回答的请求；
- 对抗和提示注入；
- 不同语言、地区和用户群体；
- 新旧模型版本回归。

### 5. 为概率组件准备降级路径

常见降级方案包括：

- 模型超时后返回固定模板；
- 低置信或高风险样本转人工；
- 外部模型不可用时切换本地小模型；
- 检索失败时明确说明信息不足；
- 输出校验失败时有限重试，超过上限停止。

### 6. LLM 未必适合的场景

下列情况应认真比较规则或专用模型：

- 标签集合稳定且样本充足；
- 每秒请求量很高，延迟预算很小；
- 输出必须完全确定；
- 错一次就会产生重大副作用；
- 输入极短，任务规律简单；
- 数据不能离开受控环境；
- 现有规则已经高质量覆盖需求。

### 7. 常见组合架构

一个更现实的系统是：

```text
输入安全检查
  ↓
规则或小模型路由
  ├─ 简单固定任务 → 专用模型
  └─ 开放复杂任务 → LLM
                         ↓
                    检索或工具
                         ↓
                    结构化校验
                         ↓
                    人工或自动执行
```

组合方案能让不同组件承担各自擅长的部分。

## 常见误区

### 误区一：NLP 已经被 LLM 取代

错误。NLP 是领域，LLM 是技术路线之一。搜索排序、拼写纠错、高吞吐分类和规则合规检查仍可能使用专用方案。

### 误区二：参数超过某个数字就可认定为 LLM

错误。不存在公认固定门槛，参数量也不能单独代表能力。

### 误区三：模型能流畅回答，所以它理解并记住了事实

错误。语言连贯、任务能力和事实可靠性是不同维度，必须分别评测。

### 误区四：下一 Token 预测只是背诵

不完整。模型确实可能记忆训练片段，但也能从数据中学习可泛化模式。具体能力中记忆与泛化各占多少，需要通过专门实验判断。

### 误区五：LLM 天然能访问互联网和数据库

错误。基础模型只根据输入和参数计算输出。联网、检索和数据库访问由宿主系统提供。

### 误区六：温度设为 0 就可完全确定

错误。贪心解码可减少采样随机性，但底层实现、并行数值误差、模型版本和服务路由仍可能造成差异。

### 误区七：公开榜单第一就是业务最佳模型

错误。榜单数据、提示模板、语言和指标可能与真实业务不同，还需考虑成本、延迟、安全和部署约束。

### 误区八：LLM 必定比小模型效果好

错误。对于定义清晰、数据充足的封闭任务，专用小模型可能更准、更快、更便宜。

## 面试追问

### 问：BERT 算不算大语言模型

答：要先说明语境。BERT 是大规模预训练语言表示模型，采用双向 Transformer Encoder 和掩码语言建模，适合理解类任务；当前产品语境中的“LLM”常特指自回归生成式基础模型。不要只争论名称，要说清架构、训练目标和使用接口。

### 问：语言模型和聊天模型有什么区别

答：语言模型学习 Token 序列分布；聊天模型通常是在预训练语言模型基础上，通过指令数据、偏好优化、安全训练和对话模板形成的产品形态。聊天能力不是仅靠把基础模型套一个聊天页面得到的。

### 问：为什么生成式模型可以做分类

答：可以把标签写成目标文本，让模型计算或生成标签 Token；也可以读取某些候选序列的条件概率。但生产中仍要约束可选标签、解析输出并评估校准性。

### 问：什么是 perplexity，它能代表模型最终能力吗

困惑度常定义为平均负对数似然的指数：

$$
\operatorname{PPL}=\exp(\mathcal{L})
$$

同一 Tokenizer 和评测分布下，较低困惑度通常表示更好的下一 Token 预测。但不同词表间不宜直接比较，而且低困惑度不自动代表更好的指令遵循、事实性或安全性。

### 延伸追问速答

| 追问 | 回答要点 |
| --- | --- |
| Prompt 和微调有什么区别 | Prompt 提供临时上下文且不改权重；微调更新全部参数或适配器，适合稳定行为，但需要数据、训练、评测和版本管理 |
| 为什么预训练后还要 SFT | 预训练主要学习续写广泛文本；SFT 用“指令—理想回答”示范任务格式、对话风格和基本指令遵循 |
| 模型继续变大，能力会持续提升吗 | Scaling Law 是特定条件下的经验趋势；数据质量、计算分配、评测饱和、后训练和推理预算都会影响结果 |
| 如何决定是否使用 LLM | 用同一真实测试集比较规则、小模型、LLM 和混合架构的质量、延迟、成本、隐私与风险，再做灰度回归 |
