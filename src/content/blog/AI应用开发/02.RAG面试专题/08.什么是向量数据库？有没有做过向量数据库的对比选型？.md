## 本篇目标

读完本篇，你应该能够：

- 解释向量数据库保存什么、查询什么；
- 区分向量搜索库、数据库扩展、搜索引擎和专用向量数据库；
- 理解精确搜索与 ANN 的取舍；
- 用人话说明 HNSW、IVF 和量化；
- 说清元数据过滤为什么会影响召回和延迟；
- 设计包含质量、延迟、更新、故障恢复和成本的对比选型；
- 在没有真实生产数据时诚实回答面试题。

## 60 秒回答

向量数据库或具备向量检索能力的数据库，用来保存向量、主键、原文引用和元数据，并根据余弦、点积或欧氏距离搜索近邻。生产能力不只包括 ANN，还包括元数据过滤、增删改、持久化、副本、备份恢复、多租户、权限、监控和混合检索。

选型时我不会直接给产品排名，而是先明确现有技术栈、向量数量与维度、增长和更新频率、过滤选择性、查询并发、延迟 SLO、可用性、数据地域和运维模式。然后用同一批真实向量、查询、过滤条件和硬件，在固定版本下比较 Recall@K 与 P95 延迟曲线、吞吐、索引构建、增量更新、删除可见时间、资源成本和故障恢复。

产品方向上，已有 PostgreSQL 且规模与负载合适时可评估 pgvector；已有全文搜索和混合检索体系时可评估 Elasticsearch 或 OpenSearch；需要独立向量服务时可比较 Qdrant、Milvus、Weaviate 等；若想减少运维可评估托管服务。最终结论只对测过的数据和条件负责。

## 详细解析

### 先说人话：普通数据库按条件找，向量检索按“像不像”找

普通数据库擅长回答：

> 找出 city = 上海并且 level = 普通员工的记录。

这是精确条件查询。

向量检索擅长回答：

> 找出语义上最像“上海出差住酒店能报多少”的制度片段。

用户没有使用制度原文中的“住宿费报销上限”，但 Embedding 可能把两种表达映射到相近位置。

生产 RAG 往往同时需要两者：

1. 使用向量找语义相近内容；
2. 使用租户、部门、时间、权限等元数据过滤；
3. 取回原文给模型；
4. 必要时再与关键词检索融合。

### 一条向量记录包含什么

最小记录可能有：

| 字段 | 示例 | 作用 |
| --- | --- | --- |
| id | chunk-2026-4-2 | 主键 |
| vector | 一串浮点数 | 近邻比较 |
| content | 上海住宿标准正文 | 生成和重排 |
| document_id | policy-2026 | 回到父文档 |
| tenant_id | company-cn | 租户隔离 |
| department | all | 部门过滤 |
| valid_from | 2026-01-01 | 时效过滤 |
| acl | employee | 权限控制 |
| embedding_version | embedding-v3 | 复现向量空间 |
| source_uri | 文档地址 | 引用与回源 |

数据库可能把正文存在外部对象存储，只保存引用；也可能直接存一份可检索文本副本。具体设计取决于延迟、容量和一致性要求。

### 向量数据库和普通数据库有什么区别

关键能力是对高维向量做相似性搜索。

普通 B-tree 索引适合排序和精确范围查询，但高维空间中不能直接照搬一维索引。向量系统使用专门的近邻搜索算法和索引结构，在质量、延迟、内存和构建时间之间取舍。

不过，现代数据库边界已经很模糊：

- PostgreSQL 可以通过 pgvector 增加向量字段和索引；
- Elasticsearch、OpenSearch 等搜索引擎支持向量与全文检索；
- Qdrant、Milvus、Weaviate 等提供专门的向量服务；
- FAISS 是近邻搜索库，不是完整数据库；
- 云平台还提供托管向量检索服务。

所以更准确的问题是：

> 当前系统需要哪组向量检索与数据治理能力，而不是名字里有没有“向量数据库”。

### 四种常见形态

#### 向量搜索库

以 FAISS 为例，它提供高性能向量索引与搜索算法。

适合：

- 单机实验；
- 离线检索；
- 团队愿意自己补充持久化和服务层；
- 作为自定义系统组件。

需要自己考虑：

- 网络服务；
- 副本；
- 数据更新；
- 权限；
- 备份恢复；
- 多租户；
- 监控。

#### 关系数据库扩展

以 pgvector 为例，在 PostgreSQL 中保存向量，并使用 SQL、事务、元数据和向量查询。

适合考虑：

- 团队已有 PostgreSQL 运维经验；
- 向量需要与业务行事务性关联；
- 希望复用 SQL、备份和权限体系；
- 规模与目标负载经过压测可满足。

限制与成本取决于数据规模、索引、过滤、并发、更新和硬件，不能仅凭“PostgreSQL 很熟”就下结论。

#### 搜索引擎的向量能力

Elasticsearch 或 OpenSearch 可同时处理倒排全文检索、过滤和向量搜索。

适合考虑：

- 已有搜索平台；
- 关键词和向量混合检索是核心；
- 需要丰富文本分析与查询能力；
- 运维团队熟悉搜索集群。

同时要评估向量索引资源、更新、过滤行为和版本特性。

#### 专用或托管向量数据库

专用系统通常围绕向量索引、分片、过滤和集合管理设计；托管服务进一步减少部分基础设施运维。

适合考虑：

- 向量规模或服务独立性要求较高；
- 需要专门向量索引与过滤能力；
- 希望独立扩容检索层；
- 团队接受新增服务或托管成本。

仍需验证备份、恢复、租户隔离、地域、配额、迁移和供应商锁定。

### 精确最近邻和近似最近邻

#### 精确搜索

精确搜索会计算查询向量与所有候选向量的距离，然后取最近结果。

优点：

- 在给定距离函数下得到精确排名；
- 适合小数据集；
- 可作为 ANN 评测基准。

缺点：

- 数据规模和查询量增大后，计算成本高。

#### ANN

ANN 是 Approximate Nearest Neighbor，近似最近邻。

它不逐个比较全部向量，而是通过索引快速寻找很可能接近的候选，以少量召回损失换取更低延迟或资源。

ANN 不表示结果随便近似。工程目标是：

> 在可接受 Recall 下满足延迟、吞吐和资源目标。

### HNSW：像在多层导航图中找邻居

HNSW 可以粗略理解成一张多层小世界图：

- 上层节点少，适合快速跳到目标附近；
- 下层节点多，适合精细搜索；
- 每个向量与一部分邻居连接。

查询时先在高层快速移动，再逐层下降到更精细区域。

常见取舍：

- 更密的图可能提高召回，也增加内存和构建成本；
- 更大的查询搜索范围可能提高召回，也增加延迟；
- 增量写入和删除行为依产品实现不同；
- 元数据过滤会改变实际候选范围。

参数名称和具体行为需要看目标产品版本，不能把一个产品的参数结论直接套给另一个产品。

### IVF：先分区，再在少量分区内搜索

IVF 是 Inverted File Index，可以粗略理解为：

1. 先把向量聚成许多区域；
2. 查询到来后找到最接近的若干区域；
3. 只在这些区域中搜索。

搜索更多区域通常提高召回并增加延迟；搜索过少可能漏掉边界附近结果。

IVF 索引通常需要训练聚类中心。数据分布变化后，旧聚类是否仍然合适需要评估。

### PQ 与量化：用更少空间近似表示向量

Product Quantization 等方法将向量拆分并使用紧凑编码，减少存储和距离计算。

收益：

- 降低内存或磁盘占用；
- 在某些配置下提高吞吐。

代价：

- 引入近似误差；
- 增加训练和参数选择；
- 可能影响特定查询切片。

量化后的质量应与未量化精确或 ANN 基线比较，不能只看节省比例。

### 距离函数必须与模型对齐

常见方式：

- 余弦相似度；
- 点积；
- 欧氏距离。

应确认：

- Embedding 模型按什么目标训练；
- 文档和查询是否归一化；
- 数据库字段配置是什么；
- 返回值是距离还是相似度；
- 分数方向是大更相关还是小更相关。

同一模型换了距离函数，排名可能变化。

### 元数据过滤为什么很关键

员工制度查询不能只按向量相似度。还要过滤：

- tenant_id = 当前租户；
- 用户拥有对应 ACL；
- valid_from 不晚于当前时间；
- valid_to 为空或仍有效；
- 文档状态 = 已发布；
- 地区和部门符合问题。

这既是质量问题，也是安全问题。

### 过滤为什么会拖慢或降低召回

考虑 100 万个向量，其中用户只有权访问 1000 个。

不同系统可能采用：

#### 预过滤

先根据元数据缩小到允许集合，再做向量搜索。

优点是权限语义直观；但允许集合很小时，索引能否有效利用要看实现。

#### 后过滤

先找向量近邻，再删除不符合条件的结果。

如果前 100 个近邻中大部分无权限，过滤后可能剩不够 K 个。系统需要扩大候选或重复搜索。

#### 过滤感知搜索

索引遍历过程中结合过滤条件。具体效果依产品和数据分布。

所以压测必须包含真实过滤：

- 无过滤；
- 低选择性过滤；
- 高选择性过滤；
- 多条件组合；
- 热门和长尾租户。

只测无过滤查询会高估生产性能。

### 多租户怎样设计

常见方式：

#### 共享集合，加 tenant_id 过滤

优点是集合数量较少、资源复用高。

风险是：

- 每次查询都要正确携带租户过滤；
- 高选择性过滤影响性能；
- 需要防止缓存和日志串租户。

#### 每个租户独立集合或索引

优点是隔离边界直观。

代价是：

- 大量小租户造成集合管理负担；
- 资源碎片；
- 发布、备份和监控对象增多。

#### 分层方案

大租户独立，小租户共享。需要清晰路由、迁移和容量治理。

选型应根据租户数量、规模分布、合规和运维能力决定。

### 写入、更新和删除也要压测

RAG 知识库不是只读系统。

#### 批量导入

需要测：

- 向量生成吞吐；
- 写入吞吐；
- 索引构建耗时；
- 构建期间资源峰值；
- 服务查询是否受影响。

#### 增量更新

文档变更可能对应多个 Chunk。要检查：

- 新向量多久可查询；
- 更新是原地还是删除后插入；
- 并发读会看到旧版还是新版；
- 是否会同时召回两个版本。

#### 删除

删除语义很重要：

- API 返回成功后，何时不再可搜索；
- 索引中的标记删除何时物理回收；
- 副本和备份如何处理；
- 缓存何时失效；
- 是否支持按 document_id 级联删除 Chunk。

#### 索引重建

Embedding 模型或索引参数变化时，需要独立构建新索引。要测重建期间的容量和切换时间。

### 一致性与可见性

数据库的“写成功”不等于所有查询节点立刻看到新内容。

要明确：

- 写入确认语义；
- 副本同步方式；
- 读写一致性；
- 索引刷新间隔；
- 缓存传播；
- 故障切换后的数据状态。

业务可以据此定义知识新鲜度 SLO。

### 可用性与灾难恢复

选型不能只测正常查询。

至少验证：

- 单节点或单副本故障；
- 网络抖动；
- 磁盘接近上限；
- 索引构建失败；
- 备份能否恢复；
- 恢复后向量数量和抽样结果是否一致；
- 恢复时间目标 RTO；
- 可接受数据丢失目标 RPO。

备份是否包含原文、元数据和索引，要看具体产品。有时可以从权威源重建索引，但重建时间也要纳入恢复方案。

### 向量容量怎样粗估

原始 float32 向量大小可以从下面公式起步：

> 向量数量 × 向量维度 × 4 字节

但这只包含原始数值，不包含：

- HNSW 图或 IVF/PQ 结构；
- 文本和元数据；
- 主键；
- 副本；
- 缓存；
- 日志；
- 数据库内部开销。

不同索引和产品开销差异很大，最终容量需要用真实数据测量。

### 一套完整选型流程

#### 第一步：明确业务目标

列出：

- 查询功能；
- 延迟 SLO；
- 目标可用性；
- 数据新鲜度；
- 安全和合规；
- 预算；
- 上线时间。

#### 第二步：描述数据

记录：

- 当前和预计向量数量；
- 维度与数据类型；
- 文本和元数据大小；
- 每日新增、更新和删除；
- 租户数量与大小分布；
- 过滤字段基数；
- 备份保留。

#### 第三步：描述查询

记录：

- QPS 与并发；
- Top-K；
- 距离函数；
- 过滤组合与选择性；
- 混合检索需求；
- 热点分布；
- 批量查询；
- 高峰模式。

#### 第四步：筛选候选

按硬约束排除：

- 不满足数据地域；
- 缺少需要的过滤或距离函数；
- 许可证不符合；
- 团队无法运维；
- 与现有平台集成成本不可接受。

#### 第五步：构建可复现压测

固定：

- 数据快照；
- Query 与相关文档标注；
- 过滤条件；
- Embedding 模型和版本；
- 数据库版本；
- 索引类型与参数；
- 硬件、节点和副本；
- 并发与预热方式。

#### 第六步：先测精确基准

在可承受规模上使用精确搜索，得到距离函数下的参考 Top-K。ANN Recall 用它作为 ground truth。

#### 第七步：画质量与延迟曲线

不要只报一个 QPS。对每个候选调整搜索参数，报告：

- ANN Recall@K；
- P50、P95、P99 延迟；
- 吞吐；
- CPU、内存、磁盘和网络；
- 索引大小。

在相近 Recall 下比较延迟，或在相近延迟下比较 Recall，结论才公平。

#### 第八步：加入真实过滤和更新

测试：

- 不同过滤选择性；
- 并发写入；
- 删除；
- 索引刷新；
- 冷启动；
- 节点故障；
- 备份恢复。

#### 第九步：计算成本和迁移风险

包括：

- 基础设施；
- 托管费用；
- 网络流量；
- 运维人力；
- 全量重建；
- 双写迁移；
- 供应商锁定；
- 退出方案。

#### 第十步：做技术决策记录

记录候选、数据、测试条件、结果、未解决风险和最终原因。以后规模变化时可以重新评估，而不是把结论当成长期不变。

### 产品类别怎样初筛

下面是思考方向，不是产品排名：

| 现状或需求 | 可优先评估的方向 | 仍需验证 |
| --- | --- | --- |
| 已有 PostgreSQL，向量与业务数据关联紧密 | pgvector | 目标规模下的索引、过滤、并发和维护 |
| 已有全文搜索，混合检索是核心 | Elasticsearch 或 OpenSearch 向量能力 | 资源、版本、更新和混合查询质量 |
| 需要独立扩展向量服务 | Qdrant、Milvus、Weaviate 等 | 分片、过滤、备份、运维和客户端 |
| 团队希望减少基础设施运维 | 托管向量服务 | 价格、地域、配额、数据政策和迁移 |
| 单机实验或自定义离线系统 | FAISS 等库 | 持久化、服务化、权限和高可用需自建 |

### 怎样回答“你有没有做过选型”

如果真实做过，可以按这个结构：

1. 业务场景和约束；
2. 数据规模、维度、更新与过滤；
3. 候选为什么入围；
4. 怎样构建 ground truth；
5. 质量、延迟、资源和故障测试；
6. 最终选择和主要取舍；
7. 上线后的瓶颈与改进。

如果没有做过生产选型，可以诚实回答：

> “我没有在生产规模完成过多产品对比，但我会先明确数据、查询、过滤、更新、SLO 和运维约束，用精确搜索建立参考，再在相同硬件与版本下比较 ANN Recall-延迟曲线、资源、恢复和成本，最后灰度验证。”

这种回答展示了方法，不会虚构经验。

## 常见误区

- **“向量数据库就是 HNSW”**：HNSW 是索引算法之一，数据库还负责数据、过滤、更新、持久化和运维。
- **“有向量字段的产品都一样”**：索引、过滤、一致性、备份、分片和客户端行为差异很大。
- **“QPS 越高就越适合”**：需要在相近 Recall、数据、硬件、过滤、并发和延迟分位数下比较。
- **“ANN 返回的就是语义真相”**：结果同时受 Embedding、距离函数、索引近似和过滤影响。
- **“元数据过滤只是附加功能”**：它直接影响权限、正确性、召回和延迟。
- **“托管服务不需要运维”**：仍要管理容量、配额、数据、成本、监控、备份和迁移。
- **“有向量库后不需要原数据库”**：权威业务事实、事务和复杂关系通常仍由原系统管理。
- **“删除 API 成功后数据已经从所有地方消失”**：还要确认索引、缓存、副本、备份和日志的处理语义。
- **“产品选型结论可以脱离版本长期复用”**：产品能力和实现会变化，结论应绑定测试版本与条件。

## 面试追问

1. **HNSW 和 IVF 怎样选？**

   在目标产品、数据和硬件上比较 Recall-延迟曲线、内存、构建、更新和过滤。不能只凭算法名称下结论。

2. **为什么高选择性过滤可能让召回不足？**

   如果先做 ANN 再过滤，大量近邻会被删除；若预过滤集合很小，索引路径也可能变化。应使用真实过滤分布压测并调整候选范围或数据布局。

3. **怎样计算 ANN Recall@K？**

   对同一查询使用精确搜索得到参考 Top-K，再计算 ANN 结果覆盖了多少参考结果。要固定距离函数和过滤条件。

4. **为什么不能只比较 P50？**

   P50 只代表中位请求，高并发、缓存未命中、过滤和故障常体现在 P95 或 P99。业务体验通常受尾延迟影响。

5. **向量库怎样做零停机迁移？**

   回填、双写、影子读、灰度切换和回滚窗口组合；同时校验数据、结果差异和删除事件，不直接停旧库再搬迁。

6. **什么情况下先考虑 pgvector？**

   已有 PostgreSQL、团队熟悉其运维、向量与事务数据紧密相关，且真实规模和负载压测满足质量与 SLO 时。它不是按数据量名称直接决定的。

7. **为什么备份恢复也属于选型指标？**

   检索服务故障会让 RAG 无法提供证据。能否在 RTO/RPO 内恢复数据和索引，直接决定可用性与业务风险。

8. **向量数据库能替代重排模型吗？**

   数据库近邻分数主要来自 Embedding 和距离函数；Cross-encoder 等重排器能让 Query 与候选充分交互。两者作用不同，是否需要重排应按质量和延迟评测。
