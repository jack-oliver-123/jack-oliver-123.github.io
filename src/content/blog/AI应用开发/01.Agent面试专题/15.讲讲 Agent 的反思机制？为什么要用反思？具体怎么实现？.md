## 本篇目标

学完后，你应该能用自己的话解释：反思和普通重试有什么不同、为什么外部反馈比模型自评可靠、怎样实现一个可停止的反思循环，以及哪些任务不适合使用反思。

## 60 秒回答

反思机制是让 Agent 在得到一次执行结果后，根据测试、环境反馈或明确的评价标准，找出具体问题，再进行有限次修正。它的核心不是让模型重复说“我再想想”，而是形成“生成结果、验证结果、得到失败证据、针对性修改、重新验证”的闭环。

它适合结果能够验证、修改成本可接受、一次生成容易遗漏的任务，例如代码测试失败、JSON 不符合 Schema、报告缺少引用。实现时我会把生成器、验证器和优化器分开：验证器输出具体失败项，优化器只修改相关部分，每轮都重新验证，并在成功、没有进展、达到轮次、时间或费用上限时由程序强制停止。

反思不能把模型自评当成真值。可靠性顺序通常是可执行测试和确定性规则优先，其次是权威数据或人工反馈，最后才是模型评分。涉及权限、安全政策和高风险审批时，不能让 Agent 通过“反思”自行改变规则。

## 详细解析

### 先说人话：反思就是“根据批改意见改作业”

你写完一篇作文后，有三种不同做法：

1. 原封不动再抄一遍，这是**普通重试**。
2. 自己看一眼，觉得“好像还能更好”，然后随意改写，这是不稳定的**自我重采样**。
3. 老师指出“第二段没有证据，结尾偏题”，你只修这两个问题，再交给老师检查，这才接近 **Reflection（反思）**。

所以，反思真正有用的部分不是“再生成一次”，而是下一次生成拿到了**可操作的失败反馈**。

### 贯穿案例：让代码 Agent 修复一个函数

假设用户让 Agent 实现一个计算优惠价格的函数。

第一次执行后，Agent 给出了代码。测试系统运行了 20 个测试，其中 18 个通过，2 个失败：

```text
失败 1：折扣为 0 时返回了 null，期望返回原价
失败 2：折扣超过 100% 时没有抛出参数错误
```

一个合理的反思循环是：

```text
生成代码
   ↓
运行测试
   ↓
把失败用例、期望值和实际值交给优化器
   ↓
只修改折扣边界处理
   ↓
重新运行相关测试和必要的回归测试
   ↓
全部通过则结束；仍失败则继续或停止
```

注意：真正证明代码修复成功的是**测试结果**，不是模型说“我已经修好了”。

### 先认识五个术语

| 术语 | 通俗解释 | 在案例中的角色 |
| --- | --- | --- |
| 候选结果 | Agent 当前交出的答案 | 第一版代码 |
| 验证器（Validator） | 按标准检查结果的程序或角色 | 测试系统 |
| 反馈（Feedback） | 指出哪里错、证据是什么 | 两个失败用例 |
| 优化器（Refiner） | 根据反馈修改结果 | 修复边界处理的模型调用 |
| 停止条件 | 决定什么时候不再循环 | 测试通过或最多修改 3 轮 |

论文中还常见两个名字：

- **Reflexion**：通过语言形式的反馈总结失败经验，并把经验用于后续尝试。
- **Self-Refine**：让模型对自己的输出给出反馈，再迭代改进输出。

它们的具体方法不同，但初学阶段先记住共同骨架即可：

```text
生成（Generate）→ 评价（Evaluate）→ 反馈（Feedback）→ 修正（Refine）
```

### 为什么反思可能有效

大模型一次生成时，可能已经具备解决问题所需的知识，却因为以下原因答错：

- 漏掉了一个约束；
- 工具返回了新的信息；
- 长任务中前后不一致；
- 第一版只是草稿，没有经过验证；
- 代码、格式或引用存在可以明确指出的错误。

反思把“大问题”变成“依据失败证据修一个小问题”。修改目标越具体，模型越容易稳定改进。

但反思并不会凭空创造正确答案。数据源本来就是错的、工具没有权限、接口已经宕机、业务规则互相冲突，这些都不是多想几轮能够解决的。

### 反馈从哪里来

不同反馈源的可靠程度不同：

| 反馈来源 | 例子 | 优点 | 局限 |
| --- | --- | --- | --- |
| 可执行测试 | 单元测试、SQL 结果对账 | 客观、可重复 | 只能覆盖写出的测试 |
| 确定性规则 | JSON Schema、字段完整性、权限规则 | 快、便宜、稳定 | 难判断开放式质量 |
| 权威数据比对 | 官方价格、数据库状态 | 能校验事实 | 数据也要考虑时效和权限 |
| 人工反馈 | 专家审阅、用户确认 | 能处理复杂语义 | 慢、贵，也可能出错 |
| 模型评分 | 按评分量表打分 | 扩展快 | 有偏差，可能被答案迎合 |

选型原则可以背成一句话：

> **能用确定性证据验证，就不要只靠模型感觉；必须用模型评分时，要用清晰量表并经过人工校准。**

### 反馈应该写成什么样

无效反馈：

```text
答案不够好，请再优化。
```

有效反馈：

```json
{
  "issue_id": "PRICE-002",
  "severity": "high",
  "criterion": "折扣必须位于 0 到 100 之间",
  "evidence": "输入 discount=120 时函数返回 -20，而不是抛出错误",
  "suggested_scope": "只修改参数校验，不改变价格计算公式"
}
```

有效反馈至少回答四件事：哪条标准没满足、错误证据是什么、影响多严重、允许修改哪里。

### 步骤级反思与结果级反思

| 类型 | 什么时候检查 | 适合的问题 |
| --- | --- | --- |
| 步骤级反思 | 每个关键动作之后 | 工具调用失败、计划走偏、缺少参数 |
| 结果级反思 | 完整产物生成之后 | 报告结构、全局一致性、引用覆盖 |

步骤级反思能早点止损，但每一步都检查会增加延迟和费用；结果级反思更省调用，但可能到最后才发现前面走错。生产系统通常只在关键节点做步骤检查，完成后再做一次整体验收。

### 怎样判断是否值得使用反思

可以问四个问题：

1. **结果能验证吗？** 没有标准，改得更长不代表更正确。
2. **错误能局部修复吗？** 如果每次都必须全部重做，成本会很高。
3. **多一轮的收益大于成本吗？** 要考虑延迟、Token、工具费用和副作用。
4. **任务允许试错吗？** 付款、删除等动作不能靠多次尝试来“反思”。

## 工程实践与边界

### 一个可落地的状态结构

反思过程不能只存在聊天记录里，最好保存结构化状态：

```json
{
  "candidate_version": 2,
  "issues": ["PRICE-002"],
  "evidence_refs": ["test-run-185"],
  "attempt": 2,
  "max_attempts": 3,
  "score_history": [0.80, 0.90],
  "status": "refining"
}
```

这样程序才能判断是否有进展，也能在失败后排查是哪一轮开始退化。

### 必须由程序控制的停止条件

至少应包含：

- 验证全部通过；
- 达到最大反思轮次；
- 达到总时间、Token 或费用预算；
- 连续两轮没有改善；
- 出现不可恢复错误；
- 需要更高权限或人工判断；
- 新版本比上一版本更差，回退到最佳版本。

不能只问模型“你觉得还要继续吗”，因为模型既可能过早结束，也可能一直修改。

### 常见失败场景及处理

| 失败场景 | 会发生什么 | 工程处理 |
| --- | --- | --- |
| 验证器给错反馈 | Agent 把正确内容改坏 | 多种证据交叉验证，保留最佳版本 |
| 只优化公开评分 | 迎合评分器，隐藏场景退化 | 隐藏测试、多个指标、人工抽样 |
| 无限循环 | 成本和延迟失控 | 轮次、时间、费用三重上限 |
| 修改范围过大 | 修好一个问题又引入新问题 | 限定修改范围并跑回归测试 |
| 把错误经验写入长期记忆 | 后续任务持续受污染 | 只有经验证、可复用的经验才能入库 |
| 外部系统失败 | 一直改 Prompt 也没用 | 熔断、降级、修复工具或转人工 |

### 不可越过的边界

- 安全政策、权限规则和审批结果由外层系统控制，模型不能自行改写。
- 有副作用的动作先判断能否安全重试，不能为了反思重复付款、发信或删除。
- 反思日志应记录结果和证据，不需要保存或展示模型的私有思维链。
- 反思记忆要按任务、租户、版本和有效期隔离，避免错误经验跨场景传播。

## 常见误区

- **“再问模型一次就是反思”**：没有失败证据和明确修改目标，只是重新抽样。
- **“同一个模型既回答又打分会更客观”**：生成器和评分器可能共享偏差，模型分数不能替代外部证据。
- **“轮次越多，答案越好”**：后续版本可能退化、迎合评分器或引入新错误。
- **“反思能解决所有错误”**：权限、数据、工具和业务逻辑问题需要系统层处理。
- **“把整段思考保存下来就能复用”**：应保存经过验证的简洁经验，而不是未经证实的推理文本。

## 面试追问

> **面试官：Reflection 和普通重试有什么区别？**
>
> **候选人：** 普通重试可能只是重复同一个请求；Reflection 会把具体失败证据、评价标准和修改范围带入下一轮，并在修改后重新验证。

> **面试官：为什么纯模型自评不够可靠？**
>
> **候选人：** 模型可能看不出自己的错误，也可能和生成答案时共享同一种偏差。因此应优先用测试、规则和权威数据；模型评分只作为补充，并与人工结果校准。

> **面试官：如何防止 Agent 无限反思？**
>
> **候选人：** 由程序设置成功、最大轮次、时间、费用和无进展停止条件，同时保存每轮得分和最佳版本，必要时回退或转人工。

> **面试官：如何防止 Agent 迎合评估器？**
>
> **候选人：** 使用隐藏测试、多个独立指标和人工抽样，不把单一模型评分暴露为唯一优化目标，并检查关键业务切片是否退化。

> **面试官：什么时候不该使用反思？**
>
> **候选人：** 结果不可验证、延迟要求很严、修改成本高，或失败来自权限和外部系统时不适合。高风险副作用也不能靠重复执行来反思，应直接审批、降级或转人工。
