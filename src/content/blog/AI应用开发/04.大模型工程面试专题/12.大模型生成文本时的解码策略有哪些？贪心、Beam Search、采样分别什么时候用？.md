同一个大模型收到“写一句雨天文案”，可能想到很多下一个词：

- “雨”；
- “窗”；
- “城市”；
- “今天”。

模型只负责给这些候选分配概率。

究竟选概率最高的那个，保留多条候选路径，还是按概率随机抽一个，是**解码策略**决定的。

因此：

> 模型参数决定“它认为哪些 Token 有可能”，解码策略决定“这次到底沿哪条路生成”。

贪心、Beam Search 和采样不是三种模型，而是同一个自回归模型的不同输出决策方式。

## 本篇目标

读完本篇，你应该能够：

- 解释 logits、softmax、Token 概率和自回归生成；
- 说清训练与解码不是同一个阶段；
- 手算一个最小贪心和采样例子；
- 解释 Beam Search 的候选扩展、累计分数与长度惩罚；
- 比较贪心、Beam Search、随机采样和对比搜索；
- 说明为什么 Beam Search 不一定适合开放式对话；
- 根据事实问答、翻译、创作、代码和结构化输出选择策略；
- 设计可复现、可评测的生产参数方案。

先记住：

> 没有脱离任务目标的“最佳解码策略”；稳定、质量、多样性、延迟和格式约束必须一起权衡。

## 60 秒回答

大语言模型按自回归方式逐 Token 生成。每一步先输出词表上的 logits，再经过 softmax 得到条件概率，解码策略从这个分布中选出下一个 Token。

贪心解码每步都选当前概率最高的 Token，速度快、结果较稳定，但局部最优不保证整段最优，也容易重复或形成模板化输出。Beam Search 每步保留累计分数最高的 $B$ 条序列，适合翻译、摘要等输出空间较受约束且有明确序列目标的任务，但计算与显存随 beam 数增加，对开放式对话常产生相似、保守或过短的候选。随机采样按概率抽取 Token，配合 Temperature、Top-k、Top-p 控制候选分布，更适合对话和创作，但结果不完全可复现，参数过松会增加跑题与幻觉。

生产中我会先确定目标：事实与格式任务优先低随机性或约束解码；开放创作使用受控采样；需要多个候选时使用独立采样或特定搜索。然后固定模型、模板、随机种子和完整生成参数，在真实任务集上比较正确率、格式成功率、多样性、延迟和成本，而不是照抄一组所谓最佳参数。

## 详细解析

### 1. 解码发生在哪里

训练阶段，模型学习预测下一个 Token。

推理阶段，模型已经训练完毕，开始真正生成：

~~~text
输入 prompt
  ↓
模型计算下一个 Token 分数
  ↓
解码策略选择一个 Token
  ↓
把新 Token 接回上下文
  ↓
重复，直到结束条件
~~~

“解码”不是把乱码翻译成人话。

这里指从模型概率分布中选择输出序列的过程。

### 2. 三个基础术语

#### Logit

模型最后一层会为词表中每个 Token 输出一个实数分数。

它叫 logit。

logit 可以是正数或负数，不是概率。

#### Softmax

softmax 把 logits 转成总和为 1 的概率：

$$
p_i
=
\frac{e^{z_i}}
{\sum_{j=1}^{V}e^{z_j}}
$$

其中：

- $V$ 是词表大小；
- $z_i$ 是第 $i$ 个 Token 的 logit；
- $e^{z_i}$ 把分数映射成正数；
- 分母对全词表求和；
- $p_i$ 是第 $i$ 个 Token 的概率。

#### 条件概率

第 $t$ 步的概率依赖 prompt 与此前输出：

$$
p(y_t\mid x,y_{<t})
$$

- $x$ 是输入；
- $y_{<t}$ 是已经生成的 Token；
- $y_t$ 是下一个 Token。

下一步上下文变了，概率也会重新计算。

### 3. 一个最小概率例子

假设某一步只有四个候选：

| Token | 概率 |
| --- | ---: |
| 北京 | 0.50 |
| 上海 | 0.25 |
| 广州 | 0.15 |
| 深圳 | 0.10 |

按该分布执行贪心时会选“北京”。

随机采样则像按比例转转盘：

- 约一半机会选北京；
- 四分之一机会选上海；
- 小概率也可能选深圳。

这不是模型突然“不确定”。

模型的不确定性已经体现在分布中，采样只是使用了整个分布。

### 4. 序列概率为什么是乘积

一段回答 $y=(y_1,\ldots,y_T)$ 的概率是：

$$
p(y\mid x)
=
\prod_{t=1}^{T}p(y_t\mid x,y_{<t})
$$

实际计算使用对数：

$$
\log p(y\mid x)
=
\sum_{t=1}^{T}\log p(y_t\mid x,y_{<t})
$$

原因有两个：

- 小概率连乘容易数值下溢；
- 对数把乘法变成加法。

搜索策略比较完整候选时，通常使用累计对数概率。

### 5. 贪心解码

贪心解码每一步选择概率最高的 Token：

$$
y_t
=
\arg\max_{v\in\mathcal{V}}
p(v\mid x,y_{<t})
$$

解释：

- $\mathcal{V}$ 是词表；
- $v$ 是候选 Token；
- `argmax` 返回概率最大者；
- 每步只保留一条路径。

伪代码：

~~~python
tokens = tokenize(prompt)

while not should_stop(tokens):
    logits = model(tokens)
    next_token = argmax(logits[-1])
    tokens.append(next_token)

return decode(tokens)
~~~

#### 贪心的优点

- 实现简单；
- 不需要随机数；
- 每步只保留一条序列；
- 通常延迟和显存开销较低；
- 在同一实现和确定性算子下较容易复现。

#### 贪心的缺点

- 当前一步最优不保证整段最优；
- 一旦选错不能回退；
- 可能陷入重复；
- 开放式任务容易模板化；
- 不能自然提供多样候选。

### 6. 为什么局部最优不等于全局最优

假设第一步有两条路：

~~~text
A：概率 0.60
B：概率 0.40
~~~

第二步：

~~~text
A → C：概率 0.20
B → D：概率 0.90
~~~

两条完整序列概率：

$$
p(A,C)=0.60\times0.20=0.12
$$

$$
p(B,D)=0.40\times0.90=0.36
$$

贪心第一步选 A，最终得到 0.12。

如果保留 B，则能得到更高的 0.36。

这就是搜索多条路径的动机。

### 7. Beam Search 的核心思想

Beam Search 每一步保留分数最好的 $B$ 条部分序列。

$B$ 叫 beam width 或 `num_beams`。

以 $B=2$ 为例：

~~~text
第一步：保留 A、B
          ↓
第二步：扩展 A1、A2、B1、B2……
          ↓
按累计分数只保留最好的两条
          ↓
重复直到结束
~~~

它是启发式搜索。

只保留有限条路径，所以不保证找到整个空间中分数最高的序列。

### 8. Beam Search 最小例子

第一步：

| 路径 | 概率 | 对数概率近似 |
| --- | ---: | ---: |
| A | 0.60 | -0.51 |
| B | 0.40 | -0.92 |

保留 A 和 B。

第二步扩展：

| 路径 | 条件概率 | 完整概率 |
| --- | ---: | ---: |
| A-C | 0.20 | 0.12 |
| A-E | 0.10 | 0.06 |
| B-D | 0.90 | 0.36 |
| B-F | 0.05 | 0.02 |

保留 B-D 与 A-C。

相比贪心，它没有在第一步过早丢弃 B。

### 9. Beam Search 伪代码

~~~python
beams = [(tokenize(prompt), 0.0)]

while not all_finished(beams):
    candidates = []

    for tokens, score in beams:
        if is_finished(tokens):
            candidates.append((tokens, score))
            continue

        log_probs = log_softmax(model(tokens)[-1])

        for token in top_tokens(log_probs, beam_width):
            candidates.append((
                tokens + [token],
                score + log_probs[token]
            ))

    beams = select_best(candidates, beam_width)

return select_best_finished(beams)
~~~

真实实现会批量计算 beam，并管理 EOS、缓存重排和提前停止。

### 10. 为什么需要长度惩罚

每个 Token 的概率不超过 1。

对数概率通常不大于 0。

序列越长，累计对数概率通常越负。

搜索可能偏好过短输出。

常见长度归一化形式为：

$$
s(y)
=
\frac{\log p(y\mid x)}{|y|^\alpha}
$$

其中：

- $|y|$ 是输出长度；
- $\alpha$ 是长度惩罚强度；
- $s(y)$ 是排序分数。

不同框架的 `length_penalty` 定义可能不同。

不能只看参数名就假设公式一致。

### 11. Beam Search 的成本

beam 数增大通常意味着：

- 同时维护更多序列；
- KV Cache 占用增加；
- 每步候选选择开销增加；
- 延迟上升；
- 候选可能高度相似。

$B=8$ 不表示质量始终比 $B=4$ 好。

搜索偏差和模型本身的概率偏差仍然存在。

### 12. 为什么开放式对话不总适合 Beam Search

开放式问题可能有很多合理答案。

最大化模型概率容易偏向：

- 常见表达；
- 安全而泛化的句子；
- 高频模板；
- 相互相似的候选。

经典研究还观察到，纯似然最大化搜索可能产生退化和重复。

因此现代聊天模型常默认受控采样，而不是大 beam 搜索。

这不是不带条件的规则。

最终仍应以具体模型和任务评测为准。

### 13. 随机采样

最基础的采样直接按分布抽取：

$$
y_t\sim p(\cdot\mid x,y_{<t})
$$

波浪号表示“从这个分布中采样”。

高概率 Token 更容易被选中，但低概率 Token 也有机会。

优点：

- 输出更有多样性；
- 能探索多个合理表达；
- 可生成多个独立候选；
- 适合对话和创作。

缺点：

- 结果有随机性；
- 参数过松会选到低质量 Token；
- 多样性可能伴随事实风险；
- 线上回归更难完全复现。

### 14. 受控采样

实际系统很少毫无限制地从整个词表采样。

常见控制包括：

- Temperature：调整分布平滑程度；
- Top-k：只保留概率最高的 $k$ 个候选；
- Top-p：保留累计概率达到阈值的最小集合；
- min-p：按最高概率的相对阈值过滤；
- repetition penalty：抑制已出现 Token；
- no-repeat n-gram：禁止重复片段；
- bad words：禁止特定序列；
- stop strings：遇到停止序列后终止。

Temperature、Top-k、Top-p 的公式和选型会在下一篇单独展开。

### 15. 独立采样与 `num_return_sequences`

需要多个候选时，可以多次采样：

~~~text
同一个 prompt
  ├─ 样本 1
  ├─ 样本 2
  ├─ 样本 3
  └─ 样本 4
~~~

然后用：

- 可执行测试；
- 规则；
- 奖励模型；
- 人工；
- 另一个评审模型；

进行 rerank。

这类“生成多个再选择”常用于代码和推理。

它的成本与候选数近似增长。

### 16. 对比搜索是什么

Contrastive Search 试图同时兼顾：

- 模型概率高；
- 新 Token 的表示不要与已有上下文过度相似。

概念目标可以理解为：

$$
\text{score}
=
(1-\alpha)\times\text{model confidence}
-\alpha\times\text{degeneration penalty}
$$

$\alpha$ 控制两者权衡。

它希望减少重复退化，同时维持连贯性。

是否优于采样高度依赖模型与任务。

### 17. Typical Decoding 是什么

Typical Decoding 不只看概率排名。

它倾向保留“信息量接近当前分布熵”的 Token。

直觉上，它试图避开：

- 过于确定、单调的 Token；
- 过于意外、离谱的 Token。

不同推理引擎支持程度不同。

生产中要先确认 API 是否真正实现该策略。

### 18. 约束解码与普通采样不同

如果输出必须满足 JSON Schema，仅降低温度不够。

约束解码会在每一步屏蔽无法构成合法结果的 Token。

~~~text
模型 logits
  ↓
语法或 Schema 状态机计算允许 Token
  ↓
屏蔽不允许的 Token
  ↓
在剩余候选中贪心或采样
~~~

它保证的是语法或结构约束。

它不保证字段事实正确，也不保证业务规则正确。

### 19. 停止条件也是解码的一部分

常见停止条件：

- 生成 EOS；
- 达到 `max_new_tokens`；
- 匹配 stop string；
- 达到时间预算；
- 外部取消；
- 约束状态接受完整输出。

`max_length` 与 `max_new_tokens` 经常被混淆：

- 前者在一些框架中包含输入长度；
- 后者只限制新输出 Token 数。

以所用框架当前文档为准。

### 20. 解码不是事实校验器

低温或贪心只能降低输出随机性。

如果模型稳定地相信错误答案，贪心会稳定输出错误。

Beam Search 也可能更坚定地找到一个高概率错误。

事实性还需要：

- 可靠上下文；
- 检索或工具；
- 训练与对齐；
- 引用验证；
- 任务评测。

## 工程实践与边界

### 1. 先按任务选择起点

| 任务 | 可考虑的起点 | 主要原因 |
| --- | --- | --- |
| 分类标签生成 | 贪心或约束解码 | 追求稳定与合法集合 |
| JSON / SQL 结构 | 约束解码 + 低随机性 | 格式必须满足语法 |
| 机器翻译 | Beam Search 或任务验证后的策略 | 输出空间相对受约束 |
| 摘要 | Beam 或低温采样，需防事实错误 | 同时看覆盖与事实性 |
| 普通助手对话 | 受控采样 | 避免过度模板化 |
| 创意写作 | 较宽松的受控采样 | 需要多样性 |
| 代码生成 | 低温单样本或多样采样后执行筛选 | 可用测试验证 |
| 数学推理 | 多样采样 + 验证或投票 | 单一路径可能偶然出错 |

表格是起点，不是固定答案。

### 2. 最小参数配置示意

以下是概念配置，不代表跨模型最佳值：

~~~yaml
strategy: sampling
temperature: 0.7
top_p: 0.9
top_k: null
max_new_tokens: 512
stop:
  - "</final>"
seed: 42
~~~

必须一起记录：

- 模型与版本；
- Tokenizer；
- chat template；
- 系统提示；
- 解码参数；
- 推理引擎版本；
- 随机种子；
- 硬件和并行配置。

### 3. seed 不保证跨环境逐 Token 相同

设置随机种子能改善复现性，但不一定保证：

- 不同 GPU 相同；
- 不同 CUDA 或内核相同；
- 不同批次调度相同；
- 不同推理引擎相同；
- 模型升级后相同。

浮点非确定性和采样实现都可能改变结果。

生产回归应比较指标与行为，不只比较整段字符串。

### 4. 评测维度

比较策略时至少看：

- 任务正确率；
- 事实性；
- 格式成功率；
- 拒答与安全表现；
- 重复率；
- 输出长度；
- 候选间多样性；
- 首 Token 延迟；
- 每 Token 延迟；
- 总成本。

创作任务只看准确率不合理。

结构化任务只看“读起来不错”也不合理。

### 5. 参数实验方法

建议流程：

1. 固定模型、prompt 和评测集；
2. 选一个简单基线；
3. 每次只改变少量参数；
4. 随机策略对每题生成多个样本；
5. 报告均值、方差和失败分布；
6. 用真实线上长度分布压测；
7. 保存完整配置与原始输出。

不要只挑最好看的三条示例。

### 6. Beam Search 的工程检查

使用 beam 前确认：

- `num_beams` 的显存增量；
- 是否开启 early stopping；
- 长度惩罚定义；
- 是否返回多个候选；
- beam 是否允许采样；
- KV Cache 如何复制或重排；
- 批处理吞吐是否下降；
- 输出是否真的优于简单基线。

### 7. 采样的工程检查

使用采样前确认：

- 参数应用顺序；
- 是否同时启用多种截断；
- Temperature 为 0 时 API 的实际语义；
- seed 是否受支持；
- stop 字符串是否可能被分词拆开；
- 生成后是否做事实或格式验证；
- 是否限制最大输出与成本。

### 8. 防重复不能只靠一个参数

重复可能来自：

- prompt 中的重复模板；
- 模型训练偏差；
- 过于尖锐的分布；
- 生成过长；
- 错误停止条件；
- 采样或搜索配置。

repetition penalty 太强也会破坏正常重复，例如姓名、代码变量和表格字段。

### 9. 流式输出不改变核心策略

Streaming 只是逐步把已选 Token 发给用户。

它不会把贪心变成采样，也不会提高模型正确率。

但它会影响：

- 用户感知延迟；
- 中途取消；
- 内容审核；
- 结构化输出的消费方式。

### 10. 服务端默认值不是业务结论

不同供应商、模型和版本的默认值可能不同。

同一个参数名也可能有边界差异。

上线时显式传入关键参数，并把默认值版本化。

## 常见误区

- **认为模型直接输出文字**：模型先输出词表 logits，解码器再选 Token。
- **认为贪心能找到整段最高概率序列**：它只保证每一步局部最大。
- **认为 Beam Search 无条件优于贪心**：开放生成中可能更保守、重复，而且成本更高。
- **认为 beam 越大越好**：搜索收益会饱和，甚至受长度与模型偏差影响而下降。
- **认为采样等于胡乱生成**：受控采样仍主要从高概率区域选择。
- **认为低温能消除幻觉**：它只能减少随机性，不能纠正稳定错误。
- **认为 seed 保证所有环境完全一致**：内核、批处理和实现差异都可能破坏逐 Token 复现。
- **把 `max_length` 和 `max_new_tokens` 混用**：是否包含输入长度取决于字段语义。
- **把约束解码当业务校验**：合法 JSON 仍可能包含错误金额。
- **同时调整所有参数再下结论**：无法判断质量变化由哪个参数造成。
- **忽略 EOS 和 stop**：停止条件错误会截断答案或产生无意义长输出。
- **把解码参数跨模型照搬**：不同模型的概率校准与训练方式不同。

## 面试追问

### 问：贪心解码为什么不是全局最优

**答：** 它每一步只保留当前概率最高的 Token，过早丢弃其他前缀。某个次优前缀后续可能具有更高条件概率，使完整序列概率更高。

### 问：Beam Search 的 beam width 是什么

**答：** 它是每一步保留的候选部分序列数量。更大 beam 探索更多路径，也增加计算、KV Cache 和候选管理成本，并不保证质量单调提高。

### 问：为什么 Beam Search 会偏短

**答：** 完整序列对数概率是每步对数概率之和，通常越长越负，因此直接比较累计分数会偏向较短序列，常需长度归一化或惩罚。

### 问：什么时候用采样而不是 Beam Search

**答：** 当任务有多个合理答案、重视自然度与多样性，例如开放式对话和创作时，通常使用受控采样。最终仍应通过任务评测决定。

### 延伸问题

- 随机采样要固定哪些条件，为什么 seed 仍不能保证跨环境复现？
- 生成 JSON 时为什么应优先考虑约束解码？
- 需要多个候选时，beam 与独立采样如何选择？
- 贪心没有随机抽样，为什么不同引擎仍可能输出不同结果？
- 解码策略能否改变模型已有知识？
- 线上怎样结合质量、格式、延迟和成本选最终策略？
