# 泛型进阶

## 回顾泛型基础
- 类型参数化
- 编译期检查
- 类型擦除 → Object
- 泛型上界 extends
- 泛型方法

## 通配符 ?
- 解决的问题
  - 接收任意泛型类型
  - 不允许修改数据
- 与泛型 T 的区别
  - T 是类型参数（定义时用）
  - ? 是通配符（使用时用）

## 上界通配符
- 语法：? extends 类
- 含义：接收该类及其子类
- 特性：只能读，不能写
- 原因：不确定具体是哪个子类
- 场景：从容器中取数据

## 下界通配符
- 语法：? super 类
- 含义：接收该类及其父类
- 特性：只能写，不能读
- 原因：能确定写入的下限
- 场景：往容器中存数据

## PECS 原则
- Producer Extends
  - 容器是数据生产者 → 用 extends
- Consumer Super
  - 容器是数据消费者 → 用 super
- JDK 中的应用
  - Collections.copy
  - Comparable<? super T>

## 泛型基础回顾

快速回顾第 3 篇的核心内容：

```java
// 泛型类：T 是类型参数
class MyArray<T> {
    public T[] array = (T[]) new Object[10];
    public T getPos(int pos) { return array[pos]; }
    public void setVal(int pos, T val) { array[pos] = val; }
}

// 使用：指定类型实参
MyArray<Integer> list = new MyArray<>();

// 泛型上界：限制 T 的范围
class MyArray<T extends Number> { ... }

// 泛型方法：方法自己声明类型参数
public static <E> void swap(E[] arr, int i, int j) { ... }

// 类型擦除：编译后 T → Object（或上界类型）
```

这些都是在**定义**泛型类/方法时使用的。但在**使用**泛型时，会遇到一个新问题——通配符。

## 通配符 `?` 解决什么问题

### 问题引出

假设有一个打印方法，希望能打印任意类型的 `MyArray`：

```java
// 只能打印 MyArray<Integer>
public static void printArray(MyArray<Integer> arr) {
    for (int i = 0; i < arr.size(); i++) {
        System.out.println(arr.getPos(i));
    }
}

// 如果想打印 MyArray<String> 呢？再写一个？
public static void printArray(MyArray<String> arr) { ... }
```

你可能会想：用 `MyArray<Object>` 不就能接收所有类型了吗？

```java
// ❌ 编译错误！
public static void printArray(MyArray<Object> arr) { ... }

MyArray<Integer> intArr = new MyArray<>();
printArray(intArr); // ❌ 类型不匹配！MyArray<Integer> 不是 MyArray<Object> 的子类
```

**关键点**：`MyArray<Integer>` **不是** `MyArray<Object>` 的子类型！泛型没有协变性——这和数组不一样（`Integer[]` 是 `Object[]` 的子类型）。

### 解决方案：通配符 `?`

```java
// ✅ 使用通配符：接收任意类型的 MyArray
public static void printArray(MyArray<?> arr) {
    for (int i = 0; i < arr.size(); i++) {
        System.out.println(arr.getPos(i));
    }
}

MyArray<Integer> intArr = new MyArray<>();
MyArray<String> strArr = new MyArray<>();
printArray(intArr); // ✅
printArray(strArr); // ✅
```

`?` 表示"未知类型"——我不关心具体是什么类型，但我能接收任何类型。

### 通配符的限制

使用 `?` 后，**不能往容器中写入数据**（null 除外）：

```java
public static void addElement(MyArray<?> arr) {
    arr.setVal(0, 10);    // ❌ 编译错误！
    arr.setVal(0, "abc"); // ❌ 编译错误！
    arr.setVal(0, null);  // ✅ null 可以（任何引用类型都能接收 null）
}
```

**为什么不能写？** 因为 `?` 代表未知类型——编译器不知道容器里实际存的是 `Integer` 还是 `String`，如果允许写入 `Integer`，万一容器实际是 `MyArray<String>` 就出问题了。

但**可以读取**（以 `Object` 类型）：

```java
public static void readElement(MyArray<?> arr) {
    Object obj = arr.getPos(0); // ✅ 可以读，但类型是 Object
}
```

### 通配符 `?` vs 类型参数 `T`

| | `T`（类型参数） | `?`（通配符） |
|--|---------------|-------------|
| 使用位置 | 定义泛型类/方法时 | 使用泛型类型时 |
| 含义 | "我定义一个类型占位符" | "我不关心具体是什么类型" |
| 能否用于声明变量 | 能（`T val`） | 不能 |
| 能否限定 | `T extends X` | `? extends X` / `? super X` |

## 上界通配符 `? extends`

### 语法与含义

`? extends 类` 表示：接收**该类及其所有子类**。

```java
// 接收 Number 及其子类（Integer、Double、Long...）
public static void printNumbers(List<? extends Number> list) {
    for (Number num : list) {
        System.out.println(num);
    }
}

List<Integer> intList = Arrays.asList(1, 2, 3);
List<Double> doubleList = Arrays.asList(1.1, 2.2, 3.3);

printNumbers(intList);    // ✅ Integer extends Number
printNumbers(doubleList); // ✅ Double extends Number
```

### 只能读，不能写

```java
public static void process(List<? extends Number> list) {
    // ✅ 可以读——取出来的一定是 Number 或其子类
    Number num = list.get(0);
    double d = list.get(0).doubleValue();

    // ❌ 不能写——不确定具体是 List<Integer> 还是 List<Double>
    list.add(1);     // 编译错误！
    list.add(1.0);   // 编译错误！
    list.add(null);  // ✅ null 可以
}
```

**为什么不能写？** 假设 `list` 实际上是 `List<Integer>`，如果允许你 `add(1.0)`，就往 Integer 列表里放了一个 Double——类型安全被破坏了。编译器无法确定具体是哪个子类，所以干脆禁止所有写入。

**为什么能读？** 无论是 `List<Integer>` 还是 `List<Double>`，取出来的元素一定**是** `Number`（因为 Integer 和 Double 都是 Number 的子类），所以可以安全地赋值给 `Number` 变量。

### 使用场景

**当你只需要从容器中取数据（生产数据）时**，用 `? extends`。

## 下界通配符 `? super`

### 语法与含义

`? super 类` 表示：接收**该类及其所有父类**。

```java
// 接收 Integer 及其父类（Number、Object）
public static void addIntegers(List<? super Integer> list) {
    list.add(1);
    list.add(2);
    list.add(3);
}

List<Integer> intList = new ArrayList<>();
List<Number> numList = new ArrayList<>();
List<Object> objList = new ArrayList<>();

addIntegers(intList);  // ✅ Integer super Integer
addIntegers(numList);  // ✅ Number super Integer
addIntegers(objList);  // ✅ Object super Integer
```

### 只能写，不能读（读出来是 Object）

```java
public static void process(List<? super Integer> list) {
    // ✅ 可以写——Integer 一定能安全放入 List<Integer>/List<Number>/List<Object>
    list.add(1);
    list.add(2);

    // ❌ 不能"有意义地"读——不确定取出来是 Integer 还是 Number 还是 Object
    // Integer num = list.get(0); // 编译错误！
    Object obj = list.get(0);     // ✅ 只能当 Object 用
}
```

**为什么能写？** 无论容器实际是 `List<Integer>`、`List<Number>` 还是 `List<Object>`，放入一个 `Integer` 都是安全的——因为 Integer 是它们所有泛型类型的子类。

**为什么不能有意义地读？** 如果容器是 `List<Object>`，取出来的可能是任何东西；如果是 `List<Number>`，取出来的也不一定是 Integer。所以编译器只能让你以 `Object` 类型读取。

### 使用场景

**当你只需要往容器中存数据（消费数据）时**，用 `? super`。

## PECS 原则

### 定义

PECS = **P**roducer **E**xtends, **C**onsumer **S**uper

- 如果容器是**数据的生产者**（你从它里面取数据）→ 用 `? extends`
- 如果容器是**数据的消费者**（你往它里面存数据）→ 用 `? super`
- 如果既要取又要存 → 不用通配符，用确切类型

### JDK 中的实际应用

#### Collections.copy 的签名

```java
public static <T> void copy(List<? super T> dest, List<? extends T> src) {
    // src 是数据来源（Producer）→ extends
    // dest 是数据目的地（Consumer）→ super
    for (int i = 0; i < src.size(); i++) {
        dest.set(i, src.get(i));
    }
}
```

- `src` 只需要读取 → `? extends T`（生产者）
- `dest` 只需要写入 → `? super T`（消费者）

#### Comparable 为什么常写成 `Comparable<? super T>`

在集合框架中，TreeSet 的构造和 Collections.sort 的签名经常是：

```java
public static <T extends Comparable<? super T>> void sort(List<T> list)
```

为什么不是简单的 `T extends Comparable<T>`？考虑这个场景：

```java
class Animal implements Comparable<Animal> {
    int age;
    @Override
    public int compareTo(Animal other) {
        return Integer.compare(this.age, other.age);
    }
}

class Dog extends Animal { }
```

`Dog` 继承了 `Animal` 的 `compareTo`——它实现的是 `Comparable<Animal>`，不是 `Comparable<Dog>`。

- `T extends Comparable<T>` → Dog 不满足（Dog 没有实现 `Comparable<Dog>`）
- `T extends Comparable<? super T>` → Dog 满足（Dog 实现了 `Comparable<Animal>`，Animal 是 Dog 的 super）

### PECS 速查表

| 场景 | 用什么 | 记忆 |
|------|--------|------|
| 只从容器取数据 | `? extends T` | Producer Extends（生产者用 extends） |
| 只往容器存数据 | `? super T` | Consumer Super（消费者用 super） |
| 既取又存 | `T`（不用通配符） | — |
| 不关心类型 | `?` | 无界通配符 |

## 通配符总结对比

| 通配符 | 含义 | 能读吗 | 能写吗 | 场景 |
|--------|------|--------|--------|------|
| `?` | 任意类型 | ✅（Object） | ❌（只能 null） | 只遍历/打印，不修改 |
| `? extends T` | T 及其子类 | ✅（T 类型） | ❌（只能 null） | 从容器取数据 |
| `? super T` | T 及其父类 | ⚠️（只能 Object） | ✅（T 及子类） | 往容器存数据 |

## 常见面试题

### 面试题 1：`List<?>` 和 `List<Object>` 的区别

```java
List<?> list1 = new ArrayList<Integer>(); // ✅ 可以
List<Object> list2 = new ArrayList<Integer>(); // ❌ 编译错误！

// List<?> 可以引用任何类型的 List
// List<Object> 只能引用 List<Object>
```

### 面试题 2：为什么 `List<Integer>` 不是 `List<Number>` 的子类型

如果允许协变：

```java
List<Integer> intList = new ArrayList<>();
List<Number> numList = intList; // 假设允许
numList.add(3.14); // 往里放了一个 Double！
Integer n = intList.get(0); // 💥 取出 Double 赋给 Integer → ClassCastException
```

泛型不允许协变正是为了防止这种类型安全漏洞。通配符是提供有限协变的安全方式——通过禁止写入（extends）或限制读取（super）来保证安全。

### 面试题 3：类型擦除后通配符如何工作

通配符是**编译期**的约束——编译器在编译时检查你对 `? extends` 容器的写入操作并报错。类型擦除后字节码中没有通配符信息，安全性已经在编译期保证了。

## 小结

| 概念 | 核心要点 |
|------|---------|
| `?` 通配符 | 表示未知类型，用于接收任意泛型 |
| `? extends T` | 上界通配符，只能读不能写 |
| `? super T` | 下界通配符，只能写不能有意义地读 |
| PECS | Producer Extends, Consumer Super |
| 泛型不协变 | `List<Integer>` 不是 `List<Number>` 的子类型 |

**关键认知**：

- 通配符是使用侧的概念（使用别人写的泛型类时），泛型参数 T 是定义侧的概念（自己写泛型类时）
- `? extends` 的"只读"和 `? super` 的"只写"不是设计缺陷，而是类型安全的保证
- PECS 原则是阅读 JDK 源码的钥匙——看到 `? extends` 就知道这个参数是"数据来源"，看到 `? super` 就知道是"数据目的地"
- 如果你写 API 时不确定用哪个，先写具体类型 `T`，只在确实需要灵活性时才引入通配符
