## 本篇目标

学完这一篇，你应该能够：

1. 解释复杂 RAG 不是“组件越多越高级”，而是为具体失败模式增加决策或结构。
2. 区分 Self-RAG、CRAG、FLARE、RAPTOR、GraphRAG、Adaptive RAG 和 Agentic RAG。
3. 说清每种范式解决什么问题、增加什么成本、在哪些场景不值得使用。
4. 避免把论文原方法和工程上的简化 Prompt 实现混为一谈。
5. 通过基线、分类型评测和消融实验选择最小可行复杂度。

## 60 秒回答

> 我会把复杂 RAG 按增加的能力来理解。Self-RAG 在生成过程中学习何时检索并用反思 token 评价证据和回答；CRAG 先评估检索结果，低质量时触发知识纠正或补充搜索；FLARE 在生成过程中根据后续 token 的不确定性主动检索；RAPTOR 把文档递归聚类和摘要成树，支持不同抽象层级检索；GraphRAG 从实体关系和社区摘要支持跨文档、全局性问题；Adaptive RAG 按问题复杂度选择不检索、单步或多步策略；Agentic RAG 则由 Agent 根据中间结果规划数据源和下一步动作。
>
> 这些不是从低到高的固定升级路线。它们分别解决检索时机、证据质量、长文档层次、关系聚合和多步决策等问题，也增加模型调用、索引构建、延迟、成本和失败面。我会先用 Naive/Advanced RAG 建立可观测基线，按失败类型选择一个机制做消融，并同时比较答案质量、检索覆盖、停止正确率、P95、成本和安全风险。

## 详细解析

### 先说人话：普通 RAG 是固定套餐，复杂 RAG 是按情况加步骤

基础 RAG 的流程通常固定：

```text
Query -> 检索一次 -> 拼上下文 -> 生成一次
```

它像每个顾客都收到同一份套餐，无论问题简单还是复杂。

复杂 RAG 会增加某种“判断”：

- 这个问题需要检索吗？
- 第一次找到的证据可靠吗？
- 需要继续检索吗？
- 应该查细节块、摘要树还是关系图？
- 证据足够回答了吗？

关键不是名字多，而是判断解决了什么失败。

### 先建立三个层级概念

#### Naive RAG：基础固定管线

```text
切块 -> Embedding -> 向量检索 -> Top-K -> 生成
```

优点：简单、稳定、容易评测。适合范围明确的 FAQ 和文档问答。

#### Advanced RAG：在固定管线上增强

常见增强：

- 更好的解析和 Chunking；
- Query Rewrite；
- BM25 + Dense 多路召回；
- RRF、Rerank；
- 上下文压缩、引用和拒答；
- 评测、缓存和权限治理。

很多生产问题到这一层已经可以解决。

#### Modular RAG：把组件变成可选择模块

系统根据 Query 和中间结果选择：

- 是否检索；
- 用哪个数据源；
- 检索几次；
- 是否改写、纠错或反思；
- 何时停止。

Self-RAG、CRAG、Adaptive RAG 和 Agentic RAG 可以视为不同的模块化决策方案，但各论文的具体训练和实现并不相同。

### 范式一：Self-RAG

#### 它想解决什么

固定 RAG 每次都检索，可能出现：

- 简单常识也付出检索成本；
- 检索到的内容不相关，模型仍硬用；
- 生成与证据不一致，却没有自检信号。

#### 核心思路

Self-RAG 原论文通过训练模型生成特殊的**反思 token**，学习：

- 是否需要检索；
- 检索片段是否相关；
- 回答是否被证据支持；
- 回答整体是否有用。

简化理解：

```text
判断是否检索 -> 读取证据 -> 生成片段 -> 评价相关性与支持度 -> 继续或调整
```

#### 重要边界

“生成后再让同一个模型说一句请反思”不等于论文中的 Self-RAG。工程上可以借鉴自评思想，但应准确描述为“基于 Prompt 的自检流程”。

#### 适用与代价

适合需要动态检索和证据自检的开放问题。代价包括专门训练/模型支持、更多推理步骤，以及自评信号也可能带有模型偏差。

### 范式二：CRAG

#### 它想解决什么

基础 RAG 认为“检索到了就可以用”，但候选可能分为：

- 基本正确；
- 模糊，既有相关也有噪声；
- 明显错误或知识库没有答案。

#### 核心思路

**CRAG（Corrective Retrieval-Augmented Generation，纠错式 RAG）**使用检索评估器判断候选质量，再选择动作：

```text
正确/高置信 -> 精炼内部证据 -> 生成
模糊       -> 内部证据 + 补充搜索 -> 生成
错误/低置信 -> 丢弃错误候选，改用补充搜索或拒答
```

原论文还包含对检索文档的知识精炼，并利用外部 Web 搜索补充。

#### 工程落地

企业场景未必允许搜公网，可以把补充动作替换为：

- 查另一个权威知识库；
- 改写 Query 再检索；
- 调用内部 API；
- 澄清或转人工。

需要评测检索评估器的误判：把好证据判坏会造成误拒答，把坏证据判好会继续传播错误。

### 范式三：FLARE

#### 它想解决什么

一次性检索在长答案生成过程中可能不够。模型开始写后才发现后面的事实没有证据。

#### 核心思路

**FLARE（Forward-Looking Active Retrieval-Augmented Generation）**先预测接下来要生成的内容，遇到低置信 token 时，使用预测内容构造检索 Query，补充证据后继续生成。

```text
生成一段 -> 预测后续内容 -> 低置信？
                        | 否 -> 继续
                        | 是 -> 检索 -> 加证据 -> 继续
```

#### 适用与代价

适合长篇、多个事实逐步出现的回答。代价是生成与检索频繁交替，延迟、调用次数和状态管理更复杂；模型置信度也需要校准。

### 范式四：RAPTOR

#### 它想解决什么

长文档中，用户可能问：

- 一个段落中的细节；
- 一个章节的主题；
- 整份报告的总体结论。

只用固定大小 chunk 难以同时支持不同抽象层级。

#### 核心思路

**RAPTOR**对文本块递归聚类并生成摘要，形成树状结构：

```text
                 全局摘要
               /          \
          主题摘要 A      主题摘要 B
          /      \        /      \
       细节块   细节块  细节块   细节块
```

查询可以命中细节节点，也可以命中较高层摘要。

#### 风险

- 摘要由模型生成，可能压缩掉重要例外或引入错误；
- 构建和增量更新成本高于平铺 chunk；
- 摘要必须能追溯到底层原文；
- 文档变化后需要确定哪些祖先摘要重算。

### 范式五：GraphRAG

#### 它想解决什么

向量检索擅长找相似片段，却不天然擅长回答：

- 哪些实体互相关联？
- 多份文档共同描述了哪些主题？
- 某个变化可能影响哪些上游和下游？
- 整个语料库的主要社区和全局趋势是什么？

#### 核心思路

GraphRAG 通常包含：

```text
文档 -> 实体/关系抽取 -> 图构建 -> 社区发现 -> 社区摘要
                                        |
Query -> 本地实体检索或全局社区检索 -----|
```

微软 GraphRAG 论文重点讨论从局部到全局的、面向 Query 的摘要。

#### 风险

- 实体抽取、关系方向和实体消歧会出错；
- 错误边可能沿多跳路径传播；
- 社区摘要可能丢细节；
- 图更新、删除、来源追踪和成本治理较复杂。

图检索的详细选型见下一题。

### 范式六：Adaptive RAG

#### 它想解决什么

不同问题复杂度不同：

```text
“2+2 等于几？”                 -> 可能无需检索
“X2 保修多久？”                -> 单步检索
“负责 X2 的团队最新版本修了什么？” -> 多步检索
```

所有请求都走最复杂链路会浪费成本，所有请求都走单次检索又覆盖不了多跳问题。

#### 核心思路

Adaptive RAG 先用分类器判断 Query 复杂度，再路由到：

- 不检索；
- 单步检索；
- 多步迭代检索。

#### 风险

分类器是新的错误点。复杂问题被错分为简单问题时，系统可能在证据不足时作答。要评测路由准确率、错路由代价和安全回退。

### 范式七：Agentic RAG

#### 它想解决什么

当问题涉及多个异构数据源、步骤依赖和中间决策时，固定管线会很僵硬。

#### 核心思路

Agent 根据当前状态决定下一步：

```text
理解目标 -> 选择工具/数据源 -> 检索 -> 观察结果
         -> 是否足够？否 -> 改写/换源/继续
                     是 -> 组织答案并引用
```

它可以使用文档检索、SQL、图数据库、Web 或内部 API。

#### 风险

- 循环和无效检索；
- 工具选错或参数错误；
- 延迟和成本不可预测；
- 外部内容提示注入；
- 工具权限过大；
- 难以复现和评测。

因此需要最大步骤、总 token、工具白名单、超时、停止条件、Trace 和人工审批边界。

### 七种范式对比

| 范式 | 主要解决的问题 | 新增机制 | 主要代价 |
| --- | --- | --- | --- |
| Self-RAG | 是否检索、证据/回答自评 | 反思 token 与动态检索 | 训练与推理复杂度 |
| CRAG | 初次检索质量不可靠 | 检索评估与纠错动作 | 评估器误判、额外搜索 |
| FLARE | 长生成过程中证据逐步不足 | 生成中主动检索 | 多轮调用与状态管理 |
| RAPTOR | 长文档多层级问题 | 聚类摘要树 | 构建、更新与摘要误差 |
| GraphRAG | 跨文档关系和全局主题 | 实体关系图与社区摘要 | 抽取、消歧和图维护 |
| Adaptive RAG | 问题复杂度差异 | 分类与路由 | 错路由风险 |
| Agentic RAG | 多源、多步、开放任务 | 自主规划和工具循环 | 不确定成本与安全面 |

### 贯穿案例：售后知识助手应该选哪一种

#### 场景 A：固定政策 FAQ

问题稳定、一个文档即可回答。Advanced RAG 加混合检索、Rerank、引用和拒答通常更经济。

#### 场景 B：长设备手册的细节和章节总结

可比较父子检索与 RAPTOR。若只用父子 chunk 已满足指标，不需要额外摘要树。

#### 场景 C：供应商故障影响哪些产品和客户

关系路径是答案核心，可以评估图检索或 GraphRAG。

#### 场景 D：初次检索经常被噪声文档污染

先改数据和 Rerank；若仍需要动态评估与补搜，再实验 CRAG 类流程。

#### 场景 E：用户要求跨订单、政策、工单做综合判断

固定工作流或 Agentic RAG 都可候选。步骤可预测时优先固定工作流；路径高度依赖中间结果时再引入 Agent。

## 工程实践与边界

### 选型顺序

```text
1. 建立 Naive/Advanced RAG 基线
2. 对失败样本分类
3. 找到占比和业务损失都高的失败类型
4. 选择直接对应的一个复杂机制
5. 做消融与成本评测
6. 影子、灰度、监控和回滚
```

不要先决定“上 GraphRAG”，再寻找适合它的问题。

### 复杂范式的统一预算

无论采用哪种机制，都建议设置：

- 最大检索轮数；
- 最大模型调用数和 token；
- 单步与端到端超时；
- 最大并发和外部搜索数；
- 允许的数据源和工具；
- 证据不足时的停止与拒答；
- 每一步的 Query、证据、决策和停止原因。

### 评测指标

#### 质量

- 答案正确性与 Faithfulness；
- 引用准确率与证据完整性；
- 初次检索、补充检索和最终证据 Recall；
- 复杂问题的步骤完成率；
- 无答案误回答和误拒答。

#### 决策

- 是否检索判断准确率；
- 复杂度路由准确率；
- 检索评估器准确率；
- 有效补搜率；
- 正确停止率与无效循环率。

#### 系统

- P50/P95/P99；
- 每请求平均/最大模型调用数；
- token、搜索和图构建成本；
- 超时、降级和工具错误率；
- 安全违规和跨权限访问。

### 论文实现与工程实现要分开说

面试时使用这样的表达更严谨：

> Self-RAG 原论文包含专门训练的反思 token。我们项目没有复现完整训练，而是借鉴其思想，使用检索评估器和受控二次检索。两者机制和效果不能直接等同。

同样，GraphRAG、CRAG 和 RAPTOR 的简化实现也应说明删减了哪些部分。

### 安全边界

复杂 RAG 会扩大攻击面：

- Web 和知识库内容都可能包含提示注入；
- Agent 不应因为文档指令而扩大权限；
- 外部搜索结果要记录来源并隔离指令；
- 图和摘要的派生事实应保留原始来源；
- 高风险动作需要确定性校验或人工批准；
- 低置信不是调用更多工具的无限理由，应有停止上限。

## 常见误区

### 误区一：复杂 RAG 普遍优于基础 RAG

简单 FAQ 中，复杂机制可能只增加延迟、成本和失败点。是否更好要看目标数据和指标。

### 误区二：Self-RAG 就是多问模型一句“请反思”

原论文包含训练得到的反思 token 和特定推理方法。Prompt 自检只是工程近似。

### 误区三：CRAG 就是检索不到时再检索一次

它的关键是先评价检索质量，并根据正确、模糊或错误采取不同纠正动作。

### 误区四：RAPTOR 就是给每篇文档写摘要

RAPTOR 递归聚类并摘要，形成多层树结构；单层摘要不等同于完整方法。

### 误区五：GraphRAG 就是图数据库加向量搜索

GraphRAG 还涉及实体关系抽取、消歧、社区发现、摘要和查询策略。图数据库只是基础设施之一。

### 误区六：Agentic RAG 会自己把答案做对

Agent 只是获得了动态决策能力，也会选错工具、循环、越权和误判证据，需要严格治理。

## 面试追问

### 1. 如何选择复杂 RAG 范式？

先对基础管线失败做分类，再选择能直接处理主要失败的机制。例如长文档抽象层级问题对应 RAPTOR，关系聚合对应图方案，初次证据质量不稳对应 CRAG 类流程。

### 2. Self-RAG 和 CRAG 有什么核心区别？

Self-RAG 通过模型训练和反思信号控制检索与生成评价；CRAG 重点评估外部检索结果并触发纠正。二者都关注证据质量，但机制不同。

### 3. FLARE 与普通多轮检索有什么区别？

FLARE 在生成过程中依据对后续 token 的置信信号主动触发检索，检索时机与正在生成的内容绑定。

### 4. 什么情况下 RAPTOR 不值得？

文档短、问题主要是局部事实、父子检索已满足质量，或内容更新频繁而摘要树重建成本过高时。

### 5. Adaptive RAG 的路由器如何评估？

标注问题复杂度和期望路径，测分类准确率、不同错分类型的业务损失、端到端答案、成本和回退成功率。

### 6. 复杂 RAG 怎样防止无限循环？

设置最大步骤、总预算、重复 Query 检测、边际新证据阈值、端到端超时和明确停止状态；每一步记录原因。

### 7. 如何证明复杂机制有增益？

和固定基线做消融，按目标失败类型报告质量、延迟、成本和安全指标；不仅展示成功案例，也报告新引入的失败。

### 8. 为什么先做 Advanced RAG，再考虑 Agentic RAG？

许多问题来自解析、切块、召回和重排，固定管线更易控制和评测。只有任务路径确实依赖中间结果时，动态 Agent 才有明确价值。
