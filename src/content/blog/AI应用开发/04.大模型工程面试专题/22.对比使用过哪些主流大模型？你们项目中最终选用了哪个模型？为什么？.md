假设公司要做一个售后客服助手。

产品经理问：“到底选哪个大模型？”

如果只回答“我选参数最多的”或“我选排行榜第一名”，这个答案通常经不起追问。

客服系统真正关心的是：

- 能不能正确理解公司的退款规则；
- 会不会泄露客户信息；
- 高峰期能不能及时响应；
- 一次工单到底花多少钱；
- 模型升级后行为会不会突然变化；
- 某个供应商故障时能不能切换。

所以，模型选型不是一场“背模型名字”的比赛，而是一个有约束的工程决策。

## 本篇目标

学完本篇，你应该能够：

1. 解释为什么公开排行榜不能直接替代项目选型；
2. 区分硬约束、质量指标、系统指标和商业风险；
3. 设计一套来自真实业务的候选模型评测集；
4. 正确计算延迟、吞吐和“单次成功任务成本”；
5. 用 Pareto 前沿解释为什么不存在所有维度都最优的模型；
6. 比较托管 API、开放权重模型和混合部署；
7. 说明什么时候应该采用多模型路由；
8. 在面试中给出具体、可信、又不过时的项目答案。

先记住一句话：

> 选型不是找“世界上最强的模型”，而是在业务硬约束内，找质量、延迟、成本、风险和运维复杂度最合适的系统方案。

## 60 秒回答

我不会先根据榜单指定模型，而会先把业务目标和硬约束写清楚。

第一步是准入筛选，例如数据能否出域、许可证是否允许商用、部署地区、上下文长度、目标语言、工具调用、多模态和硬件条件。不满足硬约束的模型，即使公开分数很高也不会进入下一轮。

第二步是用真实业务样本做离线评测。我会覆盖正常样本、长尾样本、拒答样本、风险样本和历史故障样本，并同时比较任务成功率、事实性、安全性和人工接管率，而不是只看一个平均分。

第三步是在真实输入长度、输出长度和并发下压测 TTFT、TPOT、P95/P99 延迟、吞吐、错误率和限流表现。

第四步计算单次成功任务总成本，把输入输出 Token、重试、检索、工具、GPU、运维和人工接管都算进去。

最终我会选满足所有硬约束、并位于质量—延迟—成本 Pareto 前沿上的方案。项目不一定只用一个模型：复杂或高风险请求走能力更强的模型，分类、抽取和批处理走小模型，但路由本身也必须评测、监控和可回滚。

面试中我会给出模型的具体版本、测试日期、候选范围、关键数据和最终取舍，不会声称“某模型在所有时期都最好”。

## 详细解析

### 一、先纠正问题：“主流”不等于“适合”

“主流模型”至少可能指五种不同的东西：

1. 用户量大；
2. 公开基准成绩高；
3. 开发者生态成熟；
4. 某个垂直领域表现好；
5. 在你的预算和硬件上容易部署。

这五个定义并不等价。

一个模型可能推理能力强，但价格或响应时间不适合实时客服。

另一个模型可能公开知识题成绩一般，却对公司内部分类任务足够准确，而且成本只有前者的一小部分。

因此，回答“用过哪些模型”时，重点不是列品牌，而是说明你比较过哪些**模型类别和能力边界**。

可以按下面的方式归类：

| 类别 | 典型用途 | 需要重点验证的内容 |
| --- | --- | --- |
| 通用对话模型 | 问答、总结、写作 | 指令遵循、事实性、语言风格 |
| 推理模型 | 数学、规划、复杂分析 | 正确率、推理预算、长尾延迟 |
| 小型高效模型 | 分类、抽取、路由 | 稳定性、边界样本、吞吐 |
| 代码模型 | 生成、补全、修复 | 可执行率、仓库级理解、安全 |
| 多模态模型 | 图片、音频、视频理解 | 跨模态定位、幻觉、输入成本 |
| 开放权重模型 | 私有部署、定制训练 | 许可证、硬件、量化、运维 |
| 嵌入或重排模型 | 检索和排序 | Recall@K、NDCG、领域迁移 |

这个分类比背诵一串会迅速过时的产品名更稳定。

### 二、模型选型的完整漏斗

一个可靠的选型过程可以画成漏斗：

```text
业务目标与风险等级
        |
        v
硬约束准入筛选
        |
        v
真实业务离线评测
        |
        v
真实负载性能压测
        |
        v
成本与运维风险评估
        |
        v
灰度验证与最终决策
```

每层都会淘汰不合适的候选。

先筛硬约束可以避免浪费评测时间。

例如公司要求数据只能留在指定区域，那么不满足数据驻留要求的 API 根本不应进入质量对比。

### 三、第一层：先把业务任务说清楚

“做一个聊天机器人”不是可评测的任务定义。

更好的任务定义是：

- 输入是什么；
- 输出是什么；
- 谁使用；
- 错误会造成什么后果；
- 延迟要求是多少；
- 一天有多少请求；
- 什么时候必须转人工。

以客服为例，可以拆为：

1. 识别工单意图；
2. 检索对应政策；
3. 生成基于证据的回答；
4. 判断是否需要工具；
5. 提取工具参数；
6. 对高风险动作请求确认；
7. 生成给客户的最终说明。

不同步骤不一定需要同一个模型。

意图分类可能用小模型就足够；复杂申诉分析可能需要更强模型；退款执行还必须经过确定性的权限系统。

### 四、第二层：硬约束先于分数

硬约束是“不满足就不能用”的条件。

#### 1. 数据与合规

需要明确：

- 输入是否包含个人信息、医疗数据或商业秘密；
- 数据会发送到哪个地区；
- 服务方是否保存输入输出；
- 数据是否用于训练；
- 日志保存多久；
- 是否支持删除、审计和访问控制；
- 子处理方和事故通知条款是什么。

模型能力再强，也不能替代法律、合同和安全审查。

#### 2. 许可证

“开放权重”不等于“没有限制”。

要核对：

- 是否允许商业使用；
- 是否限制特定行业或用途；
- 是否要求署名或公开衍生物；
- 是否限制用输出训练其他模型；
- 模型、代码和数据是否采用不同许可证。

许可证应由组织的法务或合规人员确认，不能靠 README 中的一句摘要做最终判断。

#### 3. 部署与硬件

开放权重模型还要回答：

- 单机能否容纳权重和 KV Cache；
- 目标 GPU 是否支持所需精度；
- 是否需要张量并行或流水线并行；
- 推理框架是否支持模型结构；
- 量化后质量是否仍达标；
- 故障切换和扩容需要多久。

#### 4. 必需能力

例如：

- 是否支持中文及业务术语；
- 是否支持所需上下文；
- 是否支持结构化输出；
- 是否支持工具调用；
- 是否能理解图片、表格或音频；
- 是否允许固定模型版本；
- 是否满足流式输出要求。

注意，“API 有这个字段”和“在你的任务中可靠可用”是两回事。

结构化输出、工具调用和长上下文都要实测。

### 五、第三层：建立真实业务评测集

公开基准可以帮助发现候选，但不能代替内部评测。

原因包括：

- 公开题目与业务分布不同；
- 训练数据可能包含公开测试题；
- 基准通常不包含你的系统提示词、检索和工具；
- 平均分会掩盖关键客户或高风险切片；
- 模型在选择题上的能力不等于端到端任务成功。

#### 1. 数据从哪里来

内部评测样本可以来自：

- 经过脱敏的历史请求；
- 产品需求和业务规则；
- 客服或领域专家编写的边界案例；
- 线上失败记录；
- 对抗测试和安全团队构造的攻击样本；
- 合成后再由专家审核的数据。

#### 2. 至少包含哪些切片

| 切片 | 目的 |
| --- | --- |
| 常见正常请求 | 验证主体体验 |
| 低频长尾请求 | 防止只优化高频模板 |
| 信息不足请求 | 检查是否会合理澄清 |
| 无法完成请求 | 检查是否会编造能力 |
| 应拒绝请求 | 检查安全边界 |
| 不应拒绝请求 | 检查过度拒答 |
| 长输入与噪声 | 检查有效上下文利用 |
| 多语言与方言 | 检查人群公平性 |
| 历史事故样本 | 防止回归 |

#### 3. 不要只设置一个“总分”

一个客服答案可能：

- 语言很流畅；
- 引用政策错误；
- 最终结论错误；
- 还泄露了内部字段。

把这些压成一个分数会失去诊断能力。

更好的方法是分层打分：

```text
任务是否完成
证据是否支持结论
关键事实是否正确
是否遵守业务规则
是否发生安全违规
是否需要人工接管
回答是否清楚、简洁
```

高风险错误可以设置为“一票否决”，而不是被文风分数抵消。

### 六、评分方法怎么选

#### 1. 确定性任务

分类、字段抽取、SQL 执行和代码测试等任务，可以使用：

- Accuracy；
- Precision、Recall、F1；
- Exact Match；
- Schema 通过率；
- 执行成功率；
- pass@k；
- 端到端任务成功率。

#### 2. 开放生成任务

总结、解释和对话通常没有唯一标准答案。

可以组合：

- 领域专家盲评；
- 成对偏好比较；
- 基于细化 Rubric 的 LLM-as-a-Judge；
- 事实声明与证据逐条核对；
- 用户任务是否完成的行为指标。

Judge 也会有位置偏好、长度偏好、风格偏好和自偏好。

因此应随机左右顺序、隐藏模型身份，并用人工标注样本校准 Judge。

#### 3. 报告不确定性

假设模型 A 在 500 条样本上成功 410 条，成功率是：

$$
\hat{p}=\frac{410}{500}=82\%
$$

其中：

- $\hat{p}$ 表示样本估计成功率；
- $410$ 是成功样本数；
- $500$ 是总样本数。

但 82% 不是总体分布上的真值。

样本有限时应报告置信区间，模型差距很小时还应做配对分析或 bootstrap。

如果 A 是 82%，B 是 81%，不能仅凭这 1 个百分点就断言 A 更强。

### 七、第四层：在真实负载下测系统性能

离线生成一条短答案很快，不代表生产环境也快。

至少记录：

| 指标 | 含义 | 用户感受 |
| --- | --- | --- |
| TTFT | 从请求到首个输出 Token 的时间 | 等多久开始看到回答 |
| TPOT | 稳态下每生成一个 Token 的平均时间 | 回答流动是否顺畅 |
| E2E latency | 从请求到完整回答的时间 | 整个任务多久结束 |
| Throughput | 单位时间处理的请求或 Token 数 | 系统承载能力 |
| P95/P99 | 95%/99% 请求不超过的延迟 | 尾部用户是否受苦 |
| Error rate | 超时、限流和服务错误比例 | 服务可靠性 |

压测必须固定并报告：

- 输入长度分布；
- 输出长度分布；
- 并发或到达率；
- 是否命中缓存；
- 流式还是非流式；
- 模型精度与量化方式；
- 硬件型号和并行策略；
- 测试日期和软件版本。

否则两个延迟数字不可比较。

例如，一个测试只输入 50 Token，另一个输入 20,000 Token，TTFT 差异可能主要来自 Prefill 工作量，而不是模型“更聪明或更慢”。

### 八、第五层：不要只比 Token 单价

真正有意义的是“完成一次成功任务的总成本”。

可以写成：

$$
C_{success}=\frac{C_{model}+C_{retrieval}+C_{tool}+C_{infra}+C_{human}}{N_{success}}
$$

其中：

- $C_{success}$：每次成功任务成本；
- $C_{model}$：模型输入、输出和推理费用；
- $C_{retrieval}$：嵌入、检索与重排费用；
- $C_{tool}$：外部工具和第三方 API 费用；
- $C_{infra}$：GPU、网关、存储、网络和运维费用；
- $C_{human}$：人工审核与接管费用；
- $N_{success}$：真正成功完成的任务数。

假设模型 A 每次调用 0.02 元，成功率 70%；模型 B 每次调用 0.03 元，成功率 95%。

只看调用价格，A 更便宜。

忽略其他成本时：

$$
C_A=\frac{0.02}{0.70}\approx0.0286
$$

$$
C_B=\frac{0.03}{0.95}\approx0.0316
$$

两者已经很接近。

如果 A 还会产生更多重试和人工接管，它的真实总成本可能更高。

### 九、用 Pareto 前沿做取舍

现实中很少有一个模型同时满足：

- 质量最高；
- 延迟最低；
- 成本最低；
- 风险最低；
- 运维最简单。

如果方案 A 在质量、延迟和成本上都不差于 B，并且至少一个维度更好，那么称 A **支配** B。

没有被其他方案支配的候选构成 Pareto 前沿。

例如：

| 方案 | 任务成功率 | P95 延迟 | 单次成功成本 |
| --- | ---: | ---: | ---: |
| A | 94% | 4.0 秒 | 0.12 元 |
| B | 91% | 1.6 秒 | 0.05 元 |
| C | 88% | 2.2 秒 | 0.08 元 |

C 同时比 B 质量低、延迟高、成本高，因此可以淘汰。

A 和 B 各有优势，都可能在 Pareto 前沿上。

最终选谁取决于业务的最低质量门槛和延迟预算。

### 十、托管 API 和开放权重模型怎么选

#### 托管 API 的典型优势

- 上手快；
- 不需要自行管理 GPU；
- 弹性扩缩容通常更简单；
- 厂商可能提供完善的多模态和工具能力；
- 模型更新和底层优化由服务方负责。

#### 托管 API 的典型边界

- 数据条款和地区受服务方限制；
- 受速率限制、配额和供应商故障影响；
- 版本迁移节奏可能不完全可控；
- 高稳定流量下成本未必最低；
- 厂商特有能力会增加迁移成本。

#### 开放权重模型的典型优势

- 可以在指定环境部署；
- 能控制推理栈、版本和优化方式；
- 可做领域微调、量化和特定硬件适配；
- 对高稳定负载可能更容易控制边际成本。

#### 开放权重模型的典型边界

- 需要 GPU、容量规划和故障处理；
- 需要维护推理框架、驱动和安全补丁；
- 自己承担伸缩、监控和 SLA；
- 许可证与训练数据风险仍需审查；
- “下载成功”距离“生产可用”还有很远。

选择不是价值判断，而是总拥有成本和控制权的权衡。

### 十一、为什么有时应该用多模型路由

单模型方案最简单，但可能浪费成本或无法兼顾延迟与质量。

多模型路由可以根据请求特征选择模型：

```text
简单分类、格式转换 -> 小模型
普通知识问答       -> 中等模型
复杂推理、高风险任务 -> 强模型 + 审核
```

路由信号可以包括：

- 任务类型；
- 输入长度；
- 风险等级；
- 是否包含图片或音频；
- 小模型置信度；
- 用户服务等级；
- 当前负载和预算。

但路由不是免费优化。

它会带来：

- 路由误判；
- 多套 Prompt 和 Schema；
- 缓存命中率下降；
- 调试路径增多；
- 不同模型输出风格不一致；
- 供应商适配和回归成本。

所以应比较：

$$
G_{route}=C_{single}-C_{routed}-C_{routing\_overhead}
$$

其中：

- $G_{route}$：路由带来的净收益；
- $C_{single}$：全量使用单一强模型的成本；
- $C_{routed}$：路由后的模型成本；
- $C_{routing\_overhead}$：路由器、误判、维护和额外延迟成本。

只有净收益稳定为正，而且质量门禁仍满足时，路由才值得上线。

### 十二、一份可复现的选型决策表

不要只在会议上口头比较。

可以保存如下记录：

| 项目 | 记录内容 |
| --- | --- |
| 测试日期 | 例如 2026-07-15 |
| 模型版本 | 固定版本号，不只写 `latest` |
| API 或权重版本 | 包含量化和推理框架 |
| Prompt 版本 | 系统提示词、模板和工具定义 |
| 数据集版本 | 样本 ID、切片和隐藏集 |
| 采样配置 | Temperature、Top-p、最大输出 |
| 质量结果 | 总体和关键切片，含置信区间 |
| 安全结果 | 越狱、隐私、误拒和过度拒答 |
| 性能结果 | TTFT、TPOT、P95/P99、吞吐 |
| 成本结果 | 每次调用及每次成功任务成本 |
| 已知边界 | 失败模式和不适用场景 |
| 决策 | 为什么选、为什么没选其他候选 |

这张表既服务于面试，也服务于未来审计和迁移。

### 十三、面试中如何讲一个真实项目

可以使用“场景—候选—实验—取舍—结果”结构。

示例：

> 我们做的是中文售后知识助手，硬约束是客户数据不用于训练、回答必须引用内部证据、P95 首 Token 小于两秒。我们先按数据条款、中文、结构化输出和版本固定能力筛了三个候选，再用 1,200 条脱敏工单评测，按普通咨询、退款、物流、信息不足和越权请求分层。最终没有选择公开基准最高的候选，因为它在我们的长政策文档上引用错误更多、P95 也超预算。上线方案在关键工单成功率达到门槛的前提下，单次成功成本更低。分类和改写走小模型，退款解释走强模型；灰度期间保留旧模型回滚。

这个回答中的模型名称可以替换，但决策逻辑长期有效。

不要编造自己没有做过的数字。

如果不能披露公司数据，可以说“经过脱敏的相对变化”或只解释方法。

## 工程实践与边界

### 一、统一接口，但不要抹平供应商差异

适配层可以统一：

- 消息结构；
- 超时和重试；
- 使用量记录；
- Trace ID；
- 常见错误分类；
- 基础的工具调用表示。

但不应该静默忽略不同供应商的能力差异。

如果某模型不支持严格 Schema，适配层不能假装支持然后悄悄降级为 Prompt 约束。

能力差异要显式暴露并纳入测试。

### 二、固定版本，谨慎使用 `latest`

生产环境应记录：

- 模型 ID；
- 快照或权重哈希；
- Tokenizer 版本；
- Prompt 版本；
- 推理参数；
- 路由策略；
- 工具 Schema 版本。

模型别名指向的后端可能变化。

升级前应跑完整回归和影子流量，确认质量、延迟和安全没有不可接受的退化。

### 三、灰度发布与回滚

典型流程是：

```text
离线门禁
  -> 影子请求
  -> 内部用户
  -> 1% 稳定分流
  -> 逐步扩大
  -> 全量或回滚
```

稳定分流可以根据用户 ID 哈希，避免同一用户在多个模型间反复跳动。

提前定义回滚阈值，例如：

- 关键任务成功率下降超过阈值；
- P95 延迟超过 SLO；
- 安全事件上升；
- 工具失败或人工接管显著增加；
- 供应商错误率持续异常。

### 四、不要把模型评测和系统评测混为一谈

模型输出错误，可能确实是模型能力问题。

但端到端失败也可能来自：

- 检索没找到正确文档；
- Prompt 截断了关键规则；
- 工具 Schema 描述不清；
- 权限服务拒绝；
- 网络超时；
- 输出解析器错误；
- 缓存返回了过期答案。

因此 Trace 要能把一次请求串起来，按组件定位失败。

### 五、定期重评，但不要无理由追新

以下事件应触发重评：

- 模型或 API 版本变化；
- Prompt、检索、工具或路由变化；
- 业务规则变化；
- 用户分布明显变化；
- 成本或配额变化；
- 发现新的安全攻击；
- 生产指标持续漂移。

“出了新模型”可以触发候选探索，但不等于必须立即迁移。

### 六、选型结论有适用范围

一次实验只对当时的以下组合负责：

```text
模型版本
+ Prompt
+ 数据分布
+ 推理参数
+ 工具和检索
+ 硬件或服务配置
+ 评测时间
```

任何一项变化，都可能改变结论。

这也是为什么本文不提供固定排行榜。

## 常见误区

### 误区一：参数越多，业务效果就越好吗

参数规模只是影响能力的一个因素。

训练数据、后训练、架构、推理预算和任务匹配同样重要。

业务还受延迟、成本和安全约束。

### 误区二：公开榜单第一就是项目第一

公开榜单只能提供候选线索。

数据污染、评分方法、Prompt 和任务分布都会影响排名。

### 误区三：只比较每百万 Token 单价

应比较每次成功任务总成本。

输出更长、失败更多、重试更多的便宜模型，最后可能更贵。

### 误区四：上下文窗口越大越好

标称窗口表示可以接受的长度，不代表能可靠利用每个位置的信息。

长上下文还会增加 Prefill、KV Cache 和成本。

### 误区五：开放权重等于免费

权重下载费用可能为零，但 GPU、网络、工程、监控和故障值班都不是零。

### 误区六：托管 API 无法保护数据

数据保护取决于具体产品、合同、地区、保留策略和架构。

不能用一句笼统结论替代审查。

### 误区七：抽象一层就完全没有供应商锁定

不同模型的工具、结构化输出、多模态、缓存和错误语义不同。

适配层能降低迁移成本，但不能消除能力差异。

### 误区八：多模型路由只会省钱

路由会误判，也会增加测试、监控、缓存和运维复杂度。

### 误区九：平均分提高就可以上线

平均分可能掩盖退款、隐私或特定语言切片的严重退化。

关键风险切片应单独设门槛。

### 误区十：模型选定后就不需要再评测

模型、业务和流量都会变化。

选型是持续的生命周期，而不是一次采购会议。

## 面试追问

### 追问一：你最终到底选了哪个模型？

先给具体版本和测试日期，再解释决定它的两个或三个关键约束。

不要只回答品牌。

示例结构：

> 在某年某月的固定版本对比中，我们选择了候选 B。它不是所有公开基准最高，但满足数据要求，在核心业务集上达到质量门槛，P95 延迟和单次成功成本更合适。候选 A 作为复杂任务回退，候选 C 因许可证或尾延迟被淘汰。

### 追问二：两个模型成功率只差 1%，怎么判断？

检查样本量、置信区间和配对结果。

还要看 1% 是否集中在高价值或高风险切片，而不是只看总体平均。

### 追问三：如何避免测试集被调参调坏？

分成开发集、回归集和隐藏发布集。

开发集允许迭代；隐藏集只用于少量正式门禁，并限制查看明细。

### 追问四：如何比较输出长度不同的模型？

固定最大输出和停止条件，同时报告任务成功率、实际输出 Token、延迟和总成本。

不要只固定请求数。

### 追问五：为什么要测 P95/P99，而不是只测平均延迟？

平均值会掩盖排队、限流和超长请求造成的尾部问题。

用户体验和 SLO 往往由尾延迟决定。

### 追问六：什么时候选择开放权重模型？

当数据和部署控制、深度定制、稳定高负载成本或特定硬件优化具有明确价值，并且团队有能力承担推理运维时。

### 追问七：什么时候不该做多模型路由？

流量小、任务单一、模型成本占比低，或团队尚未建立稳定评测和可观测性时，单模型通常更可靠。

### 追问八：如何应对供应商故障？

维护经过回归的备用模型，设置超时和熔断，做容量与配额预留，并定期进行故障切换演练。

备用模型不是在事故当天临时换一个名字。

### 追问九：如何判断一次迁移成功？

不仅要看离线分数，还要看线上任务成功、人工接管、投诉、安全、延迟和成本，并观察足够长的业务周期。

### 追问十：不能披露公司模型和数据时怎么回答？

可以说明模型类别、候选数量、硬约束、评测设计、相对变化和取舍，不应编造具体名称或未经许可的数据。
