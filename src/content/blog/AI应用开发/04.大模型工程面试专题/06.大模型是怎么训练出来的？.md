训练大模型不像给学生看一本教材，更像建立一所学校：

- 先决定招生目标和考试标准；
- 收集、清洗并编排教材；
- 设计文字编码规则；
- 在大量计算设备上进行基础学习；
- 再用示范和反馈教它怎样回答；
- 最后经过多轮考试、安全检查和小范围试运行。

任何一环出问题，都可能得到“训练损失正常、产品却不好用”的模型。

## 本篇目标

读完本篇，你应该能够：

- 从需求定义开始讲完整的大模型训练生命周期；
- 区分预训练、继续预训练、SFT 和偏好优化；
- 推导下一 Token 预测损失、Token 数和训练计算的基本关系；
- 解释数据清洗、去重、混合和污染检查为什么重要；
- 理解数据并行、张量并行、流水线并行、ZeRO/FSDP 的职责；
- 说明 checkpoint、优化器状态和混合精度的工程意义；
- 设计离线评测、安全评测、灰度发布和持续监控；
- 回答“为什么训练贵”“loss 降低为何不代表产品变好”等追问。

先记住整体路线：

```text
需求与评测 → 数据治理 → Tokenizer → 预训练
→ 继续预训练（可选）→ SFT → 偏好/强化学习（可选）
→ 能力与安全评测 → 压缩部署 → 灰度发布 → 监控迭代
```

## 60 秒回答

大模型训练不是只有一次反向传播，而是一条从数据到发布的工程链路。

首先根据目标语言、领域、上下文和部署预算定义模型架构与评测集。然后收集经过授权的数据，做格式解析、语言和质量过滤、隐私处理、去重、污染检查及数据混合；再训练并冻结 Tokenizer，把文本打包成固定长度 Token 序列。

预训练阶段通常采用自回归下一 Token 预测，用交叉熵损失通过反向传播更新参数。由于模型、激活和优化器状态放不进单卡，会组合数据并行、张量并行、流水线并行，以及 ZeRO 或 FSDP 分片，并使用混合精度、梯度累积、学习率调度和 checkpoint 容错。

预训练得到的是基础模型。为了让它按指令工作，通常还要进行监督微调；为了优化人类偏好、安全或可验证任务，可继续做 DPO、PPO、GRPO、拒绝采样等后训练，但它们不是每个模型都必须全部使用。

最后要同时评估验证损失、下游能力、事实性、安全、偏差、记忆泄漏、长上下文、延迟和成本，通过灰度发布监控回归。训练 loss 降低只说明更好地拟合特定数据分布，不等于业务质量、安全和真实性都提高。

## 详细解析

### 1. 第一步不是找 GPU，而是定义目标

训练前要回答：

- 模型面向通用对话、代码、金融还是医疗；
- 主要语言和地区是什么；
- 需要多少上下文；
- 目标参数和推理硬件预算；
- 是否开源权重；
- 哪些能力不允许退化；
- 哪些安全风险必须控制；
- 用什么测试集判断成功。

如果没有预先冻结的评测标准，团队容易在训练后挑选有利样本证明“模型变好了”。

### 2. 数据来源与治理

预训练数据可能来自：

- 获得许可的网页和出版物；
- 代码仓库；
- 学术资料；
- 企业自有文档；
- 人工创建数据；
- 合成数据；
- 对话和任务记录。

“网上可以访问”不自动等于可以不受限制地训练。数据管线需要考虑版权、许可、隐私、地域法律、用户协议和删除请求。

建议记录数据谱系：

```text
source_id
license
collection_time
parser_version
filter_version
dedup_cluster
privacy_status
split
```

### 3. 文档解析与清洗

原始数据常包含：

- HTML 导航和广告；
- 重复页眉页脚；
- OCR 错误；
- 乱码；
- 自动生成垃圾页；
- 恶意脚本；
- 密钥和个人信息；
- 模板化重复内容。

典型流水线：

```text
原始对象存储
  ↓ 解析正文和元数据
  ↓ 语言/格式识别
  ↓ 质量与安全过滤
  ↓ 去重和隐私处理
  ↓ 数据混合与切分
  ↓ Tokenize、打包、分片
```

清洗规则也会塑造模型。例如过度删除口语可能降低聊天能力；按字符比例过滤可能误伤代码、数学和少数语言。

### 4. 为什么去重重要

如果同一文档出现很多次，它会获得更大训练权重，可能造成：

- 过拟合与逐字记忆；
- 训练计算浪费；
- 测试集污染；
- 数据源比例失衡；
- 隐私文本被重复强化。

#### 精确去重

对规范化文本计算哈希：

$$
h=H(normalize(x))
$$

相同哈希视为重复。适合完全相同文档。

#### 近似去重

网页可能只改标题或少量句子。可用 n-gram、MinHash、局部敏感哈希或向量相似度发现近重复。

近似阈值太松会漏重复，太严会删掉合法模板或同主题文章，需要抽样审计。

### 5. 数据污染是什么

如果基准测试题及答案出现在训练数据中，评测分数可能反映记忆而不是泛化。

污染检查可包括：

- 精确字符串匹配；
- n-gram 重叠；
- 题目与答案分别匹配；
- 语义近似搜索；
- 按发布日期隔离；
- 私有、延后公开的测试集。

没有一种检测能证明“完全无污染”，所以报告中应说明方法和局限。

### 6. 数据混合决定模型“在哪些课上花时间”

假设有网页、代码、数学三类数据，各自采样概率：

$$
p_{web}+p_{code}+p_{math}=1
$$

训练时先按 $p_i$ 选择数据源，再采样样本。提高代码比例可能改善代码任务，也可能减少自然语言训练预算。

原始数据量不应直接决定采样比例。高质量小数据可上采样，但反复次数过多会过拟合。常用小规模消融实验选择混合比例。

### 7. Tokenizer 在训练链路中的位置

Tokenizer 通常在正式预训练前训练并冻结：

```text
代表性语料 → 训练词表/规则 → 固定 Token ID
→ 全量数据编码 → 模型训练
```

一旦模型嵌入矩阵与 Token ID 对齐，不能任意更换 Tokenizer。多语言、代码、数学和特殊 Token 都应在词表设计时考虑。

### 8. 文本怎样打包成训练样本

假设最大序列长度是 4,096。很多短文档如果各自补零会浪费计算，因此可进行 packing：

```text
[文档 A][EOS][文档 B][EOS][文档 C][EOS]...
```

直到接近固定长度。需要明确：

- 是否允许跨文档注意力；
- Loss 是否计算特殊 Token；
- 文档边界 Mask；
- 长文档怎样切块；
- 每个数据源的打包策略。

### 9. 预训练目标

对 Token 序列 $x_1,\ldots,x_T$，自回归模型分解：

$$
P_\theta(x_1,\ldots,x_T)
=
\prod_{t=1}^{T}P_\theta(x_t\mid x_{<t})
$$

训练损失：

$$
\mathcal{L}
=
-\frac{1}{N_{valid}}
\sum_{t=1}^{T}m_t\log P_\theta(x_t\mid x_{<t})
$$

其中：

- $\theta$：模型参数；
- $m_t\in\{0,1\}$：该位置是否计入损失；
- $N_{valid}=\sum_tm_t$：有效目标 Token 数；
- $P_\theta$：模型给正确 Token 的概率；
- $\mathcal{L}$：平均负对数似然。

Padding、Prompt 或文档边界是否计入损失，取决于训练阶段和配方。

### 10. 一次训练更新发生什么

```text
读取 Token Batch
  ↓ 前向传播
得到 logits 和 loss
  ↓ 反向传播
计算每个参数梯度
  ↓ 梯度同步/分片通信
  ↓ 优化器更新参数
  ↓ 学习率调度
  ↓ 记录指标并定期保存 checkpoint
```

最简伪代码：

```python
for batch in dataloader:
    optimizer.zero_grad()

    logits = model(batch.input_ids)
    loss = cross_entropy(logits, batch.labels)
    loss.backward()

    clip_grad_norm_(model.parameters(), max_norm=1.0)
    optimizer.step()
    scheduler.step()
```

大规模训练会把这几行拆到许多设备，并加入梯度累积、混合精度、检查点和容错。

### 11. Batch、Micro-batch 与梯度累积

单卡放不下全局 Batch 时，把一次更新拆成多个 Micro-batch：

$$
B_{global}
=
B_{micro}\times G\times D
$$

其中：

- $B_{micro}$：每设备一次前向的样本数；
- $G$：梯度累积步数；
- $D$：数据并行副本数；
- $B_{global}$：一次优化器更新看到的总样本数。

若样本长度不同，更准确的训练规模常按有效 Token 计数，而不是只看样本条数。

### 12. 优化器状态为什么很占显存

以混合精度 Adam 为例，训练中可能保存：

- 低精度模型参数；
- FP32 主参数副本；
- 梯度；
- 一阶动量；
- 二阶动量；
- 激活值。

所以“70B 参数 × 2 字节 = 140GB”远不是完整训练显存。具体字节数取决于优化器、精度、是否保留主权重、分片和激活重计算。

### 13. 混合精度训练

使用 BF16/FP16 等低精度进行大量矩阵运算，可以减少显存并利用 Tensor Core。

但要注意：

- FP16 数值范围较小，常需要 Loss Scaling；
- BF16 指数范围接近 FP32，通常更易训练，但尾数精度较低；
- 某些归约、优化器状态或关键算子仍保留 FP32；
- 低精度可能出现溢出、下溢和非确定性差异。

### 14. 数据并行

每个设备持有模型副本，读取不同数据：

```text
GPU 0：Batch A → 梯度 A
GPU 1：Batch B → 梯度 B
GPU 2：Batch C → 梯度 C
          ↓ All-Reduce
       聚合梯度后共同更新
```

优点是概念简单；局限是每张卡都要放完整模型、梯度和优化器状态。

### 15. ZeRO 与 FSDP

它们通过在数据并行设备之间分片训练状态，避免每卡保存完整副本。

ZeRO 常按阶段理解：

- Stage 1：分片优化器状态；
- Stage 2：再分片梯度；
- Stage 3：再分片参数。

FSDP 也会分片参数、梯度和优化器状态，并在计算某一层时按需 All-Gather 参数，计算后重新分片。

代价是通信和实现复杂度。分片越彻底，并不代表速度越快，要平衡网络带宽、计算规模和重叠策略。

### 16. 张量并行

把单个大矩阵乘法切到多卡。例如线性层：

$$
Y=XW
$$

可以按列拆分：

$$
W=[W_1,W_2]
$$

各设备计算 $XW_i$，再按需要拼接或通信。

张量并行让单层参数跨设备，但层内经常通信，适合高速互联节点。

### 17. 流水线并行

把不同层分到不同设备：

```text
GPU 0：层 1–8
GPU 1：层 9–16
GPU 2：层 17–24
GPU 3：层 25–32
```

再把 Batch 切成 Micro-batch 形成流水。局限包括流水线气泡、阶段负载不均和调度复杂。

### 18. 并行策略常组合使用

大规模训练可能使用：

```text
数据并行 × 张量并行 × 流水线并行 × 序列/上下文并行
```

总设备数近似为各维并行度乘积。选择取决于：

- 模型能否放入单节点；
- 节点内与节点间带宽；
- 序列长度；
- Batch 大小；
- 故障率；
- 框架支持。

### 19. 激活检查点

反向传播需要前向激活。全部保存会占大量显存。Activation Checkpointing 只保存部分节点，反向时重新计算中间激活。

```text
省显存 ↔ 增加重复计算
```

不要与“训练状态 checkpoint 文件”混淆：

- Activation Checkpointing：单步内部以计算换显存；
- Training Checkpoint：把模型和优化器状态写到持久存储以便恢复。

### 20. 学习率调度与 Warmup

训练初期参数和优化器统计不稳定，直接使用峰值学习率可能导致发散。常见策略：

```text
Warmup：学习率逐步升高
  ↓
主训练：余弦或其他方式衰减
```

没有适合所有规模的固定学习率。应依据 Batch、优化器、参数化、模型规模和小规模实验选择。

### 21. Checkpoint 应保存什么

为了精确恢复训练，通常需要：

- 模型参数；
- 优化器状态；
- 学习率调度器状态；
- 当前 step 和已见 Token 数；
- 随机数状态；
- 数据迭代位置；
- 混合精度 scaler；
- 并行和模型配置；
- 代码、数据和 Tokenizer 版本。

只保存模型权重可以用于推理，但未必能无缝续训。

### 22. 继续预训练是什么

继续预训练，也称领域自适应预训练等，是在基础模型上继续使用语言建模目标学习新领域或语言分布。

```text
通用基础模型
  ↓ 金融文档下一 Token 训练
金融领域基础模型
```

它与 SFT 不同：

- 继续预训练的数据多为原始领域文本；
- SFT 数据是输入指令与目标回答；
- 前者补领域分布，后者教任务行为。

继续训练过猛可能遗忘通用能力，常混入部分通用数据并做回归测试。

### 23. SFT 在做什么

监督微调数据：

```text
输入：解释什么是光合作用
目标：一段高质量回答
```

模型仍可使用交叉熵，只是通常只对 Assistant 目标 Token 计算损失：

$$
\mathcal{L}_{SFT}
=
-\sum_{t\in assistant}\log P_\theta(y_t\mid x,y_{<t})
$$

其中：

- $x$：指令和上下文；
- $y_t$：第 $t$ 个目标回答 Token；
- `assistant`：需要计入损失的回答位置集合。

SFT 让基础模型学习对话格式、指令遵循、任务风格和工具协议。

### 24. 偏好与强化学习阶段

SFT 后可使用：

- 奖励模型 + PPO 的经典 RLHF；
- DPO 等离线偏好优化；
- GRPO 等基于成组采样和奖励的方法；
- 拒绝采样后再 SFT；
- AI 反馈或可验证规则奖励。

它们利用的数据和优化目标不同，不是固定全套流程。奖励定义错误可能造成 Reward Hacking、长度偏好或能力回退。

### 25. 训练中的验证损失

训练集 loss 下降可能只是记忆。验证集来自未参与更新的分布：

```text
训练 loss 下降，验证 loss 同步下降 → 通常仍在泛化改善
训练 loss 下降，验证 loss 上升 → 可能过拟合或分布问题
```

但验证 loss 也不是全部。它可能不敏感于指令遵循、安全、事实引用和复杂任务成功率。

### 26. 能力评测与安全评测

评测层次：

| 层次 | 示例 |
| --- | --- |
| 语言建模 | Loss、Perplexity |
| 知识与推理 | 问答、数学、代码 |
| 行为 | 指令遵循、格式、拒答 |
| 事实性 | 引用、可验证事实、幻觉率 |
| 安全 | 越狱、有害内容、隐私泄漏 |
| 公平性 | 不同语言和群体表现 |
| 系统 | 延迟、吞吐、显存、成本 |

自动评测应抽样人工复核；模型裁判需要检查位置、长度、自我偏好和提示注入风险。

### 27. 模型发布前的系统环节

训练完成后通常还要：

- 合并或转换 checkpoint；
- 导出推理格式；
- 量化或蒸馏；
- 构建模型卡与使用限制；
- 安全红队；
- 压力和故障测试；
- 小流量灰度；
- 准备回滚和版本路由。

“训练结束”不等于“可直接上线”。

### 28. 训练计算怎样粗略理解

对于稠密自回归 Transformer，常见粗略估算把训练 FLOPs 写成与参数量 $N$ 和训练 Token 数 $D$ 成正比：

$$
C\approx kND
$$

其中：

- $C$：训练计算量；
- $N$：模型参数量；
- $D$：训练 Token 数；
- $k$：与前向、反向和架构实现有关的常数，常见粗估会取约 6，但不是普适精确值。

MoE、长上下文、Embedding、重计算和硬件利用率都会让真实成本偏离粗估。

### 29. 为什么 Token 吞吐比样本吞吐更有意义

不同样本长度不同：

```text
Batch A：100 个样本，每个 100 Token
Batch B：100 个样本，每个 4,000 Token
```

两者样本数相同，计算量完全不同。训练进度常记录：

- 已见总 Token；
- 有效训练 Token；
- 每秒 Token；
- 每 GPU 每秒 Token；
- 模型 FLOPs 利用率。

## 工程实践与边界

### 1. 建立可复现的数据谱系

每个 checkpoint 应能追溯：

- 数据快照和许可；
- 过滤与去重代码版本；
- 混合权重；
- Tokenizer；
- 模型配置；
- 训练代码和依赖；
- 随机种子；
- 评测结果。

否则发现隐私或版权问题时难以定位和删除。

### 2. 小规模实验先验证配方

正式大训练前运行缩小版实验，验证：

- loss 是否按预期下降；
- 学习率和 Batch 是否稳定；
- 数据是否乱码或泄漏；
- checkpoint 能否保存和恢复；
- 并行通信是否正确；
- 评测流水线是否可用。

小模型规律不能完全保证大模型结果，但能提前发现大量工程错误。

### 3. 对训练故障做自动恢复

监控：

- 节点失联；
- GPU ECC 或 Xid 错误；
- NaN/Inf；
- 数据读取卡死；
- 通信超时；
- 磁盘和对象存储故障；
- checkpoint 损坏。

定期做恢复演练，不要等昂贵长跑中断后才验证 checkpoint。

### 4. 防止静默数据错误

比程序崩溃更危险的是程序继续运行但读到错误数据。可检查：

- Token ID 范围；
- 各数据源采样比例；
- 空文档和重复率；
- 语言分布；
- 每批 loss 异常；
- 解码抽样；
- 数据校验和。

### 5. 把隐私删除能力设计进管线

记录文档来源、去重簇和训练快照，有助于：

- 响应删除请求；
- 阻止未来训练继续使用；
- 判断哪些 checkpoint 受影响；
- 做敏感信息记忆测试。

已经写入权重的信息删除并不简单，前置治理很重要。

### 6. 后训练不能只优化单一分数

建立多目标门禁：

- 主要任务提升；
- 通用能力不超过允许回退；
- 安全指标不过线不发布；
- 输出长度和拒答率不过度漂移；
- 推理成本满足预算。

### 7. 发布必须可回滚

灰度时记录模型版本、Prompt 版本、Tokenizer、采样参数和工具版本。出现质量或安全回归时，应能快速切回上一稳定版本，而不是重新训练后再处理。

### 8. 不要把一次训练当永久完成

数据分布、用户需求、攻击方式和硬件都在变化。持续迭代需要：

```text
线上反馈 → 清洗与标注 → 离线评测 → 训练候选
→ 回归测试 → 灰度 → 监控 → 下一轮
```

线上数据进入训练前必须重新做隐私、授权和质量治理。

## 常见误区

### 误区一：大模型训练就是抓网页然后跑 GPU

错误。目标设计、许可、清洗、去重、数据混合、分布式训练、后训练、评测和发布都是必要工程环节。

### 误区二：预训练完就是聊天模型

错误。基础模型主要学语言分布，聊天中的指令遵循、角色格式和偏好通常还需后训练。

### 误区三：模型参数能放进显存就能训练

错误。还要容纳梯度、优化器状态、激活和通信缓冲区。

### 误区四：训练 loss 越低，模型所有能力越好

错误。Loss 只反映特定数据和目标，事实、安全、指令与业务指标需要单独评测。

### 误区五：数据越多越好

错误。低质量、重复、污染或非法数据可能浪费计算并伤害模型。规模与质量、覆盖和混合共同决定效果。

### 误区六：分片级别越高，训练必定越快

错误。更彻底分片节省显存但增加通信，速度取决于模型、Batch 和互联。

### 误区七：从 checkpoint 恢复只需模型权重

错误。精确续训还需要优化器、调度器、随机状态、数据位置等信息。

### 误区八：公开基准高就可以直接发布

错误。还需要业务、安全、隐私、系统成本、灰度和回滚验证。

## 面试追问

### 问：为什么大模型预训练这么贵

答：参数量和训练 Token 数都很大，每个 Token 需要多层前向和反向；还要支付设备通信、低利用率、故障恢复、数据处理和大量实验成本。粗略计算与 $N\times D$ 成正比，但真实成本受架构和系统效率影响。

### 问：数据并行和张量并行有什么区别

答：数据并行让不同设备持有模型副本并处理不同 Batch，再同步梯度；张量并行把同一层大矩阵切到多设备，解决单层放不下的问题，但需要频繁层内通信。

### 问：继续预训练和 SFT 有什么区别

答：继续预训练通常对领域原始文本继续做语言建模，补充分布和知识；SFT 使用指令与目标回答，教模型执行任务和遵循格式。两者数据与目标不同，可以按需组合。

### 问：怎样判断训练是否正常

答：同时看训练/验证 loss、梯度和激活、吞吐与硬件错误、数据分布抽样、定期下游评测和 checkpoint 恢复；仅看一条平滑 loss 曲线不够。

### 延伸追问速答

| 追问 | 回答要点 |
| --- | --- |
| ZeRO-3 为什么省显存 | 参数、梯度和优化器状态都跨数据并行设备分片 |
| 激活重计算代价是什么 | 减少保存激活，但反向传播要重复部分前向计算 |
| 为什么做 Warmup | 训练初期逐步提高学习率，降低不稳定和发散风险 |
| 为什么训练前先建测试集 | 防止训练后挑指标，也减少评测数据被混入训练的风险 |
