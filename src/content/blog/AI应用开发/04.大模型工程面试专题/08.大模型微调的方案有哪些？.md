公司新来一位通用能力很强的员工。让他适应财务部门，有几种完全不同的办法：

- 给他阅读大量财务材料，补领域语言和知识；
- 给他看“问题—标准答案”，教具体工作方式；
- 给两份答复，让他学习哪份更符合偏好；
- 重新训练他的大部分能力，或只给他一本小型岗位手册。

这些都可能被口头叫作“微调”，但它们回答的是两个不同问题：

1. 用什么训练目标和数据适配；
2. 更新模型的哪些参数。

## 本篇目标

读完本篇，你应该能够：

- 区分训练阶段、训练目标与参数更新方式；
- 解释继续预训练、SFT 和偏好优化分别解决什么；
- 比较全量微调、Partial Fine-tuning 与 PEFT；
- 理解 Adapter、Prefix Tuning、Prompt Tuning、LoRA、QLoRA、BitFit；
- 用显存、质量、数据、部署和多租户维度选型；
- 判断什么时候应该先用 Prompt、RAG 或工具；
- 设计数据切分、训练、评测和发布闭环；
- 回答灾难性遗忘、过拟合和适配器合并等追问。

先建立二维分类：

```text
训练目标：继续预训练 / SFT / 偏好优化 / 强化学习
参数方式：全量更新 / 部分层更新 / Adapter / Prefix / LoRA / QLoRA
```

例如“用 LoRA 做 SFT”是完整描述；只说“LoRA 和 SFT 哪个好”是在比较不同层次。

## 60 秒回答

大模型微调应从两个维度回答。

第一个维度是训练目标。继续预训练用领域原始文本继续做语言建模，适合补充领域分布、语言或代码风格；SFT 用“指令—理想回答”学习任务、格式和对话行为；DPO、PPO、GRPO 等偏好或强化学习方法进一步利用比较、奖励或可验证反馈优化回答偏好与策略。

第二个维度是更新哪些参数。全量微调更新全部权重，表达能力强但显存、存储和灾难性遗忘风险较高。PEFT 只训练少量参数：Adapter 在层中插入小网络；Prefix/Prompt Tuning 学习连续提示向量；LoRA 用低秩矩阵表示权重增量；QLoRA 把冻结基座量化后训练 LoRA，以进一步降低显存。BitFit 只更新 Bias，是更轻量的基线。

选型时先问问题能否由 Prompt、RAG、工具或结构化约束解决。需要实时知识优先 RAG/工具；需要稳定风格、格式和任务行为可做 SFT；需要领域语言分布可考虑继续预训练；需要偏好排序再考虑 DPO/RL。参数方式则根据质量差距、GPU、并发适配器数量和部署引擎决定，并通过真实测试集验证，不能默认 LoRA 始终等价于全量微调。

## 详细解析

### 1. “微调”为什么容易说乱

下面几句话混合了不同分类轴：

```text
我们用 SFT 微调
我们用 LoRA 微调
我们用 DPO 微调
我们用 QLoRA 微调
```

- SFT：描述监督训练目标；
- DPO：描述偏好优化目标；
- LoRA：描述参数增量的低秩形式；
- QLoRA：描述量化冻结基座上训练 LoRA 的资源方案。

一个项目可以：

```text
LoRA + SFT
LoRA + DPO
QLoRA + SFT
全量参数 + SFT
```

### 2. 先判断是否真的需要训练

先把“是否训练”与“训练什么”分开决策：

```text
第一层：能否不改权重
Prompt / 少样本 / 结构化约束 / RAG / 工具

第二层：若需训练，选择数据与目标
继续预训练 / SFT / 偏好优化 / 强化学习

第三层：再选择参数更新方式
PEFT / 部分参数 / 全量参数
```

这不是必须逐层走完的固定阶梯。继续预训练、SFT 和偏好优化描述训练数据与目标；PEFT 和全量微调描述更新多少参数，两类选择可以组合。工程上应先验证成本较低、可逆性较强的无训练方案，再为确实需要训练的问题选择目标和参数方式。

#### 适合 Prompt 的问题

- 模型已有能力，只是任务说明不清；
- 示例可以放入上下文；
- 规则经常变化；
- 尚处于需求验证阶段。

#### 适合 RAG/工具的问题

- 知识实时变化；
- 需要引用来源；
- 数据必须按用户权限查询；
- 需要精确执行数据库或业务动作。

微调不适合作为实时数据库更新机制。

### 3. 继续预训练

继续预训练使用原始领域文本，目标通常保持下一 Token 预测：

$$
\mathcal{L}_{CPT}
=
-\sum_t\log P_\theta(x_t\mid x_{<t})
$$

其中：

- $x_t$：领域文本中的下一个 Token；
- $x_{<t}$：前文；
- $\theta$：待更新参数。

适合：

- 新语言或低资源语言；
- 专业术语与写作分布；
- 代码库或领域文档风格；
- 基座预训练覆盖不足的领域。

风险：

- 通用能力遗忘；
- 领域数据重复和污染；
- 大量计算；
- 只学会“像领域文本”，未必学会按指令回答。

常混入通用数据并做领域/通用双重回归。

### 4. 监督微调 SFT

数据格式：

```json
{
  "messages": [
    {"role": "user", "content": "解释资产负债表"},
    {"role": "assistant", "content": "资产负债表反映……"}
  ]
}
```

通常只对目标回答 Token 计算损失：

$$
\mathcal{L}_{SFT}
=
-\sum_{t\in y}\log P_\theta(y_t\mid x,y_{<t})
$$

其中：

- $x$：指令与上下文；
- $y$：理想回答；
- $y_t$：回答第 $t$ 个 Token。

SFT 适合：

- 指令遵循；
- 固定输出格式；
- 领域问答风格；
- 工具调用格式；
- 拒答和边界示范；
- 多轮对话行为。

高质量、多样和覆盖边界的数据通常比简单堆数量更重要。

### 5. 偏好优化

偏好数据常为：

```text
Prompt x
更好回答 y_w
较差回答 y_l
```

DPO 等方法直接从偏好对优化策略；经典 RLHF 可能先训练奖励模型，再用 PPO 在线采样优化；GRPO 等方法用成组样本和奖励估计相对优势。

适合：

- 多个答案都可接受，但质量有高低；
- 需要调整帮助性、安全性或风格；
- 有可靠人类或规则反馈；
- 可验证任务有清晰奖励。

偏好优化不能修复错误或偏置的评分标准。奖励可能被钻漏洞。

### 6. 全量微调

更新模型全部参数：

$$
\theta' = \theta - \eta\nabla_\theta\mathcal{L}
$$

其中：

- $\theta$：原模型全部参数；
- $\eta$：学习率；
- $\nabla_\theta\mathcal{L}$：对所有参数的梯度；
- $\theta'$：更新后参数。

优势：

- 可调整所有层；
- 对分布变化很大或数据充足任务，能力上限可能更高；
- 部署时不必额外挂载适配器。

代价：

- 梯度、优化器状态与 checkpoint 很大；
- 每个任务保存完整模型；
- 更容易破坏原能力；
- 分布式训练复杂；
- 学习率和数据质量更敏感。

### 7. 部分层微调

冻结大部分模型，只训练：

- 最后若干 Transformer 层；
- 输出头；
- Embedding；
- LayerNorm；
- Bias。

优点是简单，不引入新模块。缺点是选择哪些层需要实验，训练参数仍可能很大，而且更新连续整层未必是参数效率更高的方式。

### 8. 什么是 PEFT

PEFT 是 Parameter-Efficient Fine-Tuning，即参数高效微调。核心目标：冻结大部分基座，只训练少量新增或选中的参数。

注意两个显存概念：

- 可训练参数显存：梯度与优化器状态只为小部分参数保存；
- 基座权重显存：即使冻结，前向仍需加载基座权重。

因此“只训练 1% 参数”不表示训练显存只剩 1%。激活和基座权重仍占空间。

### 9. Adapter

Adapter 在 Transformer 层中插入小型瓶颈网络：

$$
\operatorname{Adapter}(h)
=
h+W_{up}\,\sigma(W_{down}h)
$$

其中：

- $h\in\mathbb{R}^d$：原隐藏状态；
- $W_{down}\in\mathbb{R}^{r\times d}$：降维；
- $r\ll d$：瓶颈维度；
- $\sigma$：非线性激活；
- $W_{up}\in\mathbb{R}^{d\times r}$：升维；
- 残差 $h+$：保留原表示。

优势：模块化、每任务保存小权重。边界：新增推理层可能增加延迟，是否能融合取决于实现。

### 10. Prefix Tuning

Prefix Tuning 为每层注意力学习一组连续的前缀 K/V 向量。真实输入 Token 不变，但注意力像是在每层前面看到可训练“虚拟上下文”。

```text
[可训练 Prefix K/V] + [真实 Token K/V]
```

它未必对应可读文字。优势是参数少；代价是增加注意力需要处理的前缀长度和 K/V 存储。某些框架会把这部分计入可用长度或缓存预算，某些只在内部管理，因此应按具体实现核对，而不是直接等同于用户可输入文本窗口。

### 11. Prompt Tuning

Prompt Tuning 通常只在输入 Embedding 前添加可训练软提示：

$$
H_0=[P;E(x)]
$$

其中：

- $P\in\mathbb{R}^{m\times d}$：$m$ 个可训练 Prompt 向量；
- $E(x)$：真实输入 Embedding；
- $[;]$：序列维拼接。

它比 Prefix Tuning 更接近输入层，参数更少，但效果受模型规模、任务和初始化影响。

### 12. LoRA

对冻结权重 $W_0\in\mathbb{R}^{d_{out}\times d_{in}}$，不直接训练完整 $\Delta W$，而写成：

$$
W'=W_0+\Delta W
$$

$$
\Delta W=BA
$$

其中：

- $A\in\mathbb{R}^{r\times d_{in}}$；
- $B\in\mathbb{R}^{d_{out}\times r}$；
- $r\ll\min(d_{in},d_{out})$；
- $W_0$ 冻结，只更新 $A,B$。

前向：

$$
y=W_0x+\frac{\alpha}{r}BAx
$$

LoRA 可挂在注意力和 FFN 投影上。适配器可保存、切换，也可合并进基座，后文会单独详讲。

### 13. QLoRA

QLoRA 通常把冻结基座以 4-bit 等低精度表示用于训练前向，并在其上训练 BF16/FP16 LoRA 参数。

关键点：

- 基座权重被量化以降低存储；
- 基座不通过普通全量梯度更新；
- LoRA 参数保持较高精度训练；
- 计算时可能按块反量化到计算精度；
- 论文提出 NF4、Double Quantization 和 Paged Optimizers 等技术。

QLoRA 让有限 GPU 训练更大基座成为可能，但速度未必总比非量化 LoRA 快，质量也要按任务验证。

### 14. BitFit

BitFit 只更新模型 Bias 参数，其余权重冻结。参数极少，适合作为轻量基线。

现代某些 LLM 线性层本来不使用 Bias，此时可训练参数更有限，方法适用性取决于架构。

### 15. 各参数方式比较

| 方法 | 更新范围 | 训练资源 | 独立任务存储 | 推理影响 | 典型边界 |
| --- | --- | --- | --- | --- | --- |
| 全量微调 | 全部参数 | 最高 | 完整模型 | 无新增模块 | 成本高、遗忘风险 |
| 部分层 | 若干原参数 | 中等 | 参数差分或完整模型 | 通常无新增模块 | 层选择依赖实验 |
| Adapter | 新增瓶颈层 | 低 | 小 | 可能增加算子 | 需引擎支持 |
| Prefix | 每层连续前缀 | 很低 | 很小 | 增加前缀 K/V | 占上下文/缓存 |
| Prompt Tuning | 输入软提示 | 极低 | 很小 | 占输入长度 | 能力上限依任务 |
| LoRA | 低秩权重增量 | 低 | 小 | 可动态挂载或合并 | Rank 和目标层需调参 |
| QLoRA | 量化基座 + LoRA | 更省权重显存 | 小 | 训练/部署路径复杂 | 量化误差与内核支持 |

“训练资源”是相对趋势，不是统一倍数。

### 16. 灾难性遗忘

模型在新领域训练后，原有通用能力下降，称为灾难性遗忘。常见原因：

- 新数据分布窄；
- 学习率过大；
- 训练轮数过多；
- 全量参数大幅更新；
- 数据中有错误模式；
- 只优化单一任务。

缓解方式：

- 混入代表性通用数据；
- 降低学习率和训练轮数；
- 使用 PEFT 限制更新；
- 加 KL 或权重正则；
- 同时监控新旧任务；
- 保留回滚 checkpoint。

PEFT 可以降低部分风险，但不保证没有行为回归。

### 17. 过拟合在微调中怎样表现

```text
训练 loss 持续下降
验证 loss 开始上升
固定模板表现很好
换一种表达就失败
回答越来越像训练样本
```

小数据微调尤其容易让模型记住答案格式。需要：

- 按用户、文档或时间切分数据；
- 去重近似问题；
- 保留真实分布的盲测集；
- Early Stopping；
- 控制 Epoch 和学习率；
- 检查训练样本复述。

### 18. 数据质量比格式数量更重要

SFT 样本应检查：

- 指令是否有足够信息；
- 答案是否事实正确；
- 风格是否符合目标；
- 是否包含错误拒答；
- 是否泄漏系统或隐私信息；
- 多轮角色是否正确；
- 工具结果是否与调用对应；
- 是否覆盖“不知道”和澄清场景。

自动合成数据应保留生成模型、Prompt、过滤器和人工抽查记录。

### 19. 训练集、验证集、测试集怎样切

不要简单随机打散高度相似样本，否则同一模板或文档片段会跨集合泄漏。

可按：

- 用户；
- 原始文档；
- 问题簇；
- 时间；
- 业务场景；
- 数据来源。

测试集应在训练决策中尽量少被反复查看，另保留持续回归集。

### 20. 一个微调选型决策树

```text
问题主要是缺实时/私有事实？
  ├─ 是 → 先 RAG/工具
  └─ 否
      ↓
模型已有能力，只是指令不清？
  ├─ 是 → Prompt/示例/约束
  └─ 否
      ↓
缺领域语言分布？
  ├─ 是 → 评估继续预训练
  └─ 否
      ↓
缺稳定任务行为与格式？
  ├─ 是 → SFT
  └─ 否
      ↓
需要比较式偏好或可验证奖励？
  └─ 是 → DPO/RL/拒绝采样

参数更新方式：
GPU/存储有限、多任务多租户 → 优先 PEFT 基线
PEFT 质量达不到且资源足够 → 再评估全量微调
```

### 21. 最小 LoRA SFT 配置要说明什么

即使不写框架 API，也应记录：

```yaml
base_model: model-version-hash
method: lora
target_modules: [q_proj, v_proj]
rank: 16
alpha: 32
dropout: 0.05
trainable_dtype: bf16
max_sequence_length: 4096
learning_rate: 0.0002
epochs: 2
chat_template: template-version
```

还要记录数据版本、随机种子、梯度累积、优化器和评测结果。

### 22. 多适配器服务

PEFT 适合一个基座服务多个任务：

```text
共享基座
  ├─ 客服 LoRA
  ├─ 法务 LoRA
  └─ 代码 LoRA
```

收益：减少存储、快速切换。挑战：

- 动态批处理中的适配器不同；
- 热加载和缓存；
- 版本与租户隔离；
- 合并冲突；
- 引擎是否支持高效多 LoRA。

### 23. 评测应覆盖四层

#### 模型层

- 验证 loss；
- 困惑度；
- 基准任务。

#### 行为层

- 格式遵循；
- 指令成功率；
- 拒答与澄清；
- 长度和风格。

#### 安全层

- 越狱；
- 隐私；
- 有害内容；
- 提示注入；
- 工具越权。

#### 系统层

- TTFT/TPOT；
- 吞吐；
- 显存；
- 适配器切换；
- 量化兼容；
- 单次成本。

## 工程实践与边界

### 1. 先做无训练基线

没有 Prompt/RAG 基线，就无法判断微调投入带来多少增益。先固定同一测试集测：

- 原始模型；
- 优化 Prompt；
- Few-shot；
- RAG/工具；
- LoRA；
- 全量微调候选。

### 2. 微调不能替代权限系统

训练模型“不要读取别人的数据”不是强制授权。数据访问必须由应用和数据库权限控制，模型只接收当前用户获准的数据。

### 3. 微调不能可靠写入频繁变化事实

价格、库存、政策和订单状态会变化。将其写进权重成本高、难删除、无法保证召回。优先使用检索或工具，并让模型引用当前结果。

### 4. 使用小学习率和完整回归

微调是在已有能力上做调整。学习率过大可能迅速破坏表示。每个 checkpoint 都应同时测目标任务和原有通用/安全任务。

### 5. 适配器也属于生产制品

记录：

- 基座精确版本；
- 适配器权重和配置；
- 目标模块；
- Tokenizer/模板；
- 训练数据版本；
- 兼容量化；
- 评测报告；
- 许可证。

适配器不能随意挂到同名但不同权重的基座。

### 6. 合并前后都要测试

LoRA 可合并到基座以减少动态算子，但合并涉及精度、量化顺序和权重转换。对比：

- 未合并输出；
- 合并后输出；
- 重新量化后输出；
- 性能与显存。

### 7. 防止训练数据中的提示注入

抓取的对话、网页和工具结果可能包含“忽略之前指令”等内容。训练数据应区分普通数据和受信任控制指令，避免模型学到错误角色边界。

### 8. 人工反馈要测一致性

偏好标注应记录指南、标注者分歧和裁决规则。低一致性可能说明任务含糊，而不是简单增加标注数量就能解决。

## 常见误区

### 误区一：SFT 和 LoRA 是互斥方案

错误。SFT 是训练目标，LoRA 是参数更新方式，可以用 LoRA 做 SFT。

### 误区二：微调可以实时更新知识库

错误。频繁变化且需引用的事实更适合 RAG 或工具。

### 误区三：只训练 1% 参数就只需 1% 显存

错误。冻结基座和激活仍要放入计算图；节省主要来自梯度与优化器状态，QLoRA 才进一步压缩基座存储。

### 误区四：LoRA 必定和全量微调等效

错误。低秩约束可能足够也可能不足，效果受 Rank、目标层、数据和任务影响。

### 误区五：全量微调必定质量最高

错误。小数据下可能更易过拟合和遗忘，且优化更难。必须实验比较。

### 误区六：继续预训练会自动教会指令遵循

错误。它通常学习领域文本分布，SFT 才直接提供指令—回答示范。

### 误区七：适配器文件可以挂到任意同架构模型

错误。它绑定具体基座权重、模块名称、维度和 Tokenizer/模板。

### 误区八：训练集表现好就可以上线

错误。还要独立测试、通用能力回归、安全评测和灰度监控。

## 面试追问

### 问：怎样判断用全量微调还是 LoRA

答：先在同一真实测试集上做 LoRA 基线。如果目标行为可通过低秩更新学到、资源或多租户要求高，优先 LoRA；若质量仍有明确差距、数据充分且能承担分布式训练与完整模型存储，再评估全量微调。

### 问：继续预训练、SFT、DPO 的顺序是什么

答：常见顺序是先补领域分布，再用 SFT 教指令，最后用偏好数据优化；但不是固定流水线。若基座领域能力够，可跳过继续预训练；若没有可靠偏好数据，也不必做 DPO。

### 问：为什么微调会灾难性遗忘

答：新数据窄或更新过强时，参数向新分布移动，原任务需要的表示被破坏。可通过数据混合、较小学习率、PEFT、正则和全量回归缓解。

### 问：Adapter 与 LoRA 的结构区别是什么

答：Adapter 在网络中插入带非线性的瓶颈模块；LoRA 不增加串行主干层，而为既有线性权重添加低秩增量。前者可能增加推理算子，后者可合并进权重。

### 延伸追问速答

| 追问 | 回答要点 |
| --- | --- |
| Prompt Tuning 与 Prefix Tuning | 前者主要学习输入软提示，后者通常为每层注意力提供连续 Prefix K/V |
| QLoRA 是否等于 4-bit 全量微调 | 不是，典型 QLoRA 冻结量化基座，只训练 LoRA 参数 |
| 微调后为何更啰嗦 | 训练答案长度分布、偏好标注的长度偏置或解码配置可能改变 |
| 如何避免测试泄漏 | 按文档/用户/时间切分、近似去重、保留私有盲测集 |
