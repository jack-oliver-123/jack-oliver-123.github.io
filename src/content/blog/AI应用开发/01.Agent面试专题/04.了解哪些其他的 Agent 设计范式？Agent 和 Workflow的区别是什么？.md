## 本篇目标

建立一张 Agent 设计模式地图。你不需要死记英文名，而要理解每种模式解决什么问题、增加什么成本，以及什么时候不该使用。

## 60 秒回答

我会先把常见设计分成 Workflow 模式和 Agent 控制策略。常见 Workflow 有提示链、路由、并行、编排者—执行者和评估者—优化者；常见 Agent 控制策略有 ReAct、Plan-and-Execute 和 Reflection。Workflow 的主要路径由开发者定义，Agent 则允许模型根据环境反馈动态选择部分路径。

选型原则是从最小复杂度和最小自主性开始：一步能完成就用单次调用；固定阶段用提示链；类别清晰用路由；独立子任务用并行；子任务数量无法提前确定时用编排者—执行者；环境反馈不断改变下一步时再用 ReAct 或计划型 Agent；只有结果可验证且重复修改有价值时才使用 Reflection。最终要用成功率、延迟、成本和安全评测证明复杂结构值得。

## 详细解析

### 为什么会有这么多“范式”

“范式”可以简单理解成一种组织工作的方法。

假设你要让 AI 完成“调研三款手机并写一份购买建议”。系统可以：

- 一次性让模型直接回答；
- 先列评价标准，再逐款分析，最后写结论；
- 把三款手机交给三个任务并行处理；
- 让一个编排者决定要查哪些资料；
- 写完后再让评估器检查引用和预算；
- 让 Agent 边搜索边决定下一步。

它们都可能完成任务，但控制方式、成本和风险不同。设计模式不是为了给系统起一个高级名字，而是为了让我们讨论：

- 谁决定下一步；
- 任务是否能提前拆清；
- 中间结果怎样传递；
- 失败时从哪里恢复；
- 什么时候应该停止。

### 第一类：提示链

提示链是把一个任务拆成固定的连续阶段，前一步输出进入后一步。

例如：

> 提取用户需求 -> 生成文章大纲 -> 检查大纲 -> 根据大纲写正文。

适合场景：

- 阶段顺序固定；
- 每一步都能清楚定义输入和输出；
- 把大任务拆小后，模型更容易做好。

优势是容易理解、测试和定位错误。缺点是每一步都会增加延迟，前一步的错误还可能传给后一步。

使用提示链时，最好在关键节点加程序校验。例如大纲必须包含“预算、性能、售后”三个字段，缺失就不进入写作步骤。

### 第二类：路由

路由是先判断输入属于哪一类，再发送到不同的提示、模型、工具或流程。

客服例子：

| 用户问题 | 路由目标 |
| --- | --- |
| “怎么修改密码？” | 账号帮助流程 |
| “商品坏了，想退款” | 售后流程 |
| “接口返回 500” | 技术支持流程 |

分类器可以是规则、传统模型，也可以是 LLM。即使由 LLM 分类，只要允许的类别和后续流程由开发者预设，整体仍然是 Workflow。

适合路由的前提是类别能够稳定区分。如果大量问题同时跨越多个类别，强行路由会丢失信息，这时可以允许多标签或先询问用户。

### 第三类：并行

并行是让多个互不依赖的子任务同时执行，再汇总结果。

例如比较三款手机时：

- 任务 A 查手机一；
- 任务 B 查手机二；
- 任务 C 查手机三；
- 聚合器统一按相同指标比较。

并行的主要价值是缩短关键路径，也可以收集多个独立视角。但“多调用几次模型然后投票”并不代表结果更准确。

并行前要确认：

- 子任务之间没有先后依赖；
- 不会同时修改同一个资源；
- 输出使用统一 Schema；
- 部分任务失败时有明确策略；
- 汇总器能保留来源并处理冲突。

三个任务同时写同一份订单状态，就可能发生覆盖和重复副作用，不能简单并行。

### 第四类：编排者—执行者

编排者先根据当前输入动态拆分子任务，再把每个子任务交给有界的执行者，最后汇总。

它与固定并行的区别是：**子任务数量和内容无法提前完全确定。**

例如代码修改任务：

> “为项目增加邮箱登录。”

编排者需要先查看代码库，才能知道要修改几个文件、是否要更新数据库、需要哪些测试。不同项目产生的子任务不同。

这种模式适合复杂搜索、代码修改和开放调研，但编排者不能只输出一段自然语言清单。可靠的任务应包含：

- 子任务 ID；
- 输入和预期输出；
- 依赖关系；
- 可用工具；
- 完成标准；
- 失败策略。

### 第五类：评估者—优化者

一个组件生成结果，另一个组件按照明确标准评价，生成器再根据反馈修改。

例如写一篇带引用的调研报告：

1. 生成器完成初稿；
2. 评估器检查是否覆盖三个产品、引用是否存在、价格是否来自权威来源；
3. 评估器输出具体缺失项；
4. 生成器只修改相关部分；
5. 重新验证，直到通过或达到上限。

这种模式适合“结果能够判断好坏，并且反馈后有机会改进”的任务。代码测试、Schema 校验、引用核验通常比“另一个模型觉得不错”更可靠。

如果评价标准模糊，两个模型可能共同犯错；如果没有最大轮次，系统还可能不断润色却没有真实收益。

### Agent 控制策略一：ReAct

ReAct 可以先理解为“每获得一个新观察，就决定下一步动作”：

> 读取问题 -> 选择搜索 -> 得到结果 -> 判断信息不足 -> 换关键词搜索 -> 得到证据 -> 生成答案。

它适合路径较短、环境反馈频繁、无法提前知道要调用几次工具的任务。风险是只顾局部动作，出现重复搜索、目标漂移或上下文不断膨胀。

下一篇会专门解释 ReAct 的运行循环。

### Agent 控制策略二：Plan-and-Execute

Plan-and-Execute 先生成高层计划，再逐步执行；当工具不可用、前置条件变化或结果不符合预期时，再受控重规划。

例如组织一次会议：

> 确认参会人 -> 收集可用时间 -> 选择时间 -> 预订会议室 -> 发送邀请。

这类任务有明显依赖，先保留整体计划可以防止 Agent 只盯着眼前一步。但计划也可能一开始就错，所以每一步需要验收条件和重规划触发器。

### Agent 控制策略三：Reflection

Reflection 是根据失败证据或评价反馈，进行有限次修正。

它不是简单地“再想一遍”。有效反思必须知道：

- 哪条标准没有满足；
- 失败证据是什么；
- 允许修改哪些部分；
- 修改后怎样重新验证；
- 什么时候停止。

例如代码测试失败时，把具体测试错误交给修复步骤；如果只是让同一个模型重复评价“答案好不好”，价值通常有限。

### 一张表比较常见模式

| 模式 | 谁规定主要路径 | 最适合解决什么 | 主要代价或风险 |
| --- | --- | --- | --- |
| 单次调用 | 开发者 | 简单理解、抽取或生成 | 能力边界有限 |
| 提示链 | 开发者 | 固定阶段的复杂任务 | 延迟、错误传播 |
| 路由 | 开发者定义类别 | 不同类型走不同专用流程 | 分类错误、边界模糊 |
| 并行 | 开发者定义子任务 | 独立任务加速或多视角 | 汇总冲突、调用成本 |
| 编排者—执行者 | 模型拆分，系统约束 | 子任务无法提前确定 | 状态和调度复杂 |
| 评估者—优化者 | 固定循环加评价信号 | 可验证结果的迭代改进 | 评分偏差、无效循环 |
| ReAct | 模型逐轮选择 | 反馈频繁、路径未知 | 漂移、重复和成本 |
| Plan-and-Execute | 模型规划，系统调度 | 长依赖任务 | 错误计划、计划过期 |
| Reflection | 反馈驱动修正 | 有可靠验证的任务 | 迎合评分器、退化 |

### 选型可以按五个问题进行

1. **任务能否一次完成？** 能就从单次调用开始。
2. **步骤是否稳定？** 稳定就使用 Workflow。
3. **子任务是否独立且已知？** 已知用固定并行，未知再考虑编排者。
4. **环境结果是否会改变下一步？** 会才需要 Agent 循环。
5. **结果是否可以可靠验证？** 可以才考虑 Reflection。

还要看四项工程约束：

- 延迟要求；
- Token 和工具费用；
- 副作用风险；
- 测试和运维能力。

### 模式可以组合，但不要一开始全用

一个合理组合可能是：

1. 路由判断客服问题类型；
2. 售后节点由 ReAct Agent 处理信息不完整的问题；
3. 固定 Workflow 执行审批和退款；
4. 最后用规则检查回复中是否包含退款状态。

组合模式的原则不是越多越强，而是每增加一层都要有明确收益。否则错误发生时，很难判断是路由错、计划错、工具错还是评估器错。

## 工程实践与边界

- **先建立最简单基线**：先测单次调用或固定 Workflow，再用失败轨迹证明某个节点需要动态决策。
- **每层定义契约**：明确输入、输出、错误出口、最大轮次、预算和人工接管条件。
- **并行避免写冲突**：共享资源使用锁、版本号、事务或明确的合并规则。
- **评价依赖真实证据**：优先可执行测试、Schema、业务规则和权威数据，再考虑模型评分。
- **计划不能扩大权限**：Planner 可以请求工具，但实际可执行集合由可信运行时计算。
- **记录可观察轨迹**：保留计划、动作、工具结果和简短理由，不记录或展示模型私有思维链。
- **持续做简化**：当线上路径长期稳定时，把重复决策固化为 Workflow，减少成本和故障面。

## 常见误区

- **“范式名称越新越先进”**：名称只描述控制结构，最终效果取决于任务、工具、数据和评测。
- **“Workflow 完全不能动态”**：它可以有条件、循环和 LLM 路由，关键是边界由开发者定义。
- **“编排者—执行者等同于 Multi-Agent”**：执行者可以只是普通模型调用或函数。
- **“Reflection 总能提高质量”**：没有可靠评价信号时，它可能只增加延迟和成本。
- **“多个模型投票就能得到事实”**：相关模型可能共享同一偏差，外部证据更重要。
- **“把所有模式叠加起来最强”**：复杂度会扩大状态空间，使测试和故障归因更困难。

## 面试追问

> **面试官：** 什么时候用编排者—执行者而不是固定并行？
>
> **候选人：** 当子任务数量或类型无法提前确定，但可以定义执行者能力、任务契约和汇合协议时使用动态编排；子任务已知且独立时，固定并行更简单。

> **面试官：** 评估者—优化者什么时候有效？
>
> **候选人：** 结果有明确标准、反馈能指出具体问题、修改确实能提高通过率时有效，例如代码测试或引用校验；只有主观模型评分时要谨慎。

> **面试官：** Reflection 的退出条件是什么？
>
> **候选人：** 外部测试通过、评价达到门槛、连续两轮没有改进，或者达到轮次、时间和费用预算，任一条件都应该能终止。

> **面试官：** 如何判断 Agent 应该退化成 Workflow？
>
> **候选人：** 如果线上轨迹长期集中在少数稳定路径，而且动态决策在同一评测集上没有改善成功率，就应固化这些路径并减少模型决策点。

> **面试官：** 如何避免过度设计？
>
> **候选人：** 从单次调用或固定 Workflow 建立基线，每增加一种模式都要求在任务成功率、延迟或安全上产生可测的净收益。
