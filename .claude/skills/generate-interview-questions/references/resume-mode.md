# 简历驱动模式详细规则

## 简历读取

| 格式 | 处理方式 |
|------|----------|
| `.md` / `.txt` | 直接 Read |
| `.pdf` | 通过 Read 工具读取（仅文本型 PDF，扫描件需用户粘贴文本） |
| `.docx` | 提示用户导出为 PDF 或粘贴文本（需 pandoc 支持，不保证可用） |
| 直接粘贴文本 | 最可靠方式，推荐 |

## 简历信息提取

### 基本信息
- 技术栈清单（语言/框架/中间件/工具）
- 项目经历（每个项目的职责、技术难点、业务规模）
- 工作年限 → 自动推断 level（用户可覆盖）
- 主要领域 → 自动推断 domain
- 目标岗位 → 若 `$ARGUMENTS.position` 未指定，尝试从简历目标/求职意向推断
- 区分正式工作经历 vs 实习经历

### 职业轨迹信号
- 跳槽频率：<1年/家标注[高频跳槽风险]，面试大概率被问
- 技术栈转型：标注[转型候选人]，新领域基础需重点验证
- 空窗期：>3个月标注[空窗期]，建议准备 behavioral 解释
- 教育背景：非CS科班标注[非科班]，基础算法/数据结构题权重+1

### 包装度评估（内部参考，不直接输出给用户）
- "精通"出现 ≥3 次 → 技术验证题要深一层
- 多个项目描述高度同构 → 可能模板化包装，追问具体细节
- 职责描述含大量被动语态（"参与了""配合了"）→ 追问个人具体贡献

## 三模块题目生成

### 模块一：专业技能面试题

来源：简历技术栈 × 岗位考察矩阵

**分类判定优先级**（当一个技术点可归入多个方向时）：
1. 岗位考察矩阵中标注为该岗位核心的技术 → 优先出题，A级
2. 技术栈列表中出现、项目中有实际使用场景 → 标准出题，按频率定级
3. 该 level 应掌握但简历完全未提及的核心技术 → 薄弱探测，B级

**岗位化筛选**：
根据 `positions.md` 岗位考察矩阵中对应岗位的"专业技能侧重"列确定筛选方向。未列出岗位按 positions.md 的动态推断规则处理。

### 模块二：实习/工作经历行为面试题

来源：简历中的每段工作/实习经历

**生成规则**：
- 每段经历生成 1 组行为面试题（包含六维度追问，见 behavioral-interview.md "实习经历专属深挖"）
- 标注 [实习] 或 [正式工作]，追问深度不同
- 长期实习（≥6个月）按准正式工作标准出题
- 关键信号（跳槽/空窗/转型）额外生成对应 behavioral 题目

**与 A/B/C 的映射**：行为面试题统一按 A 级格式输出（完整追问链），但使用专属 BQ 格式模板。

### 模块三：项目综合面试题

来源：简历中有具体描述（≥2句话）的项目

**分类判定**：
1. 简历有具体项目描述（≥2句话讲了做什么/怎么做）→ 纳入模块三
2. 仅一词带过（如"用了Redis"无后文）→ 不纳入模块三，归入模块一薄弱探测

**生成规则**：
- 每个项目生成 1 组综合面试题（覆盖七维度追问，见 output-templates.md 项目综合题格式）
- 标注 [简历项目#N]，N 对应简历中项目顺序
- 实习项目的追问深度降一级（不追问架构决策，侧重"你学到了什么"）

**与 A/B/C 的映射**：项目综合题统一按 A 级格式输出，使用专属 PJ 格式模板。

### 三模块间互斥性

每道题只归入一个模块：
- 技术点出现在项目描述中 → 归入模块三（项目中追问），不在模块一重复
- 但如果该技术点需要独立验证基础理解（如"Java多线程"在项目中用到但基础概念需单独问）→ 模块一出基础题，模块三出应用题，不重复追问方向

## Resume + Topic 组合规则

| 场景 | 处理 |
|------|------|
| topic 在简历技术栈内 | 深挖该方向，模块一中该方向题目权重加大 |
| topic 不在简历技术栈内 | 作为扩展考察，薄弱探测题权重加大，标注[扩展考察] |
| topic 范围 > 简历经验 | 超出部分题目级别降一档 |

## 跨领域简历处理

当简历涉及多个领域时：

1. **主领域判定**（按优先级）：
   - 用户通过 domain 参数显式指定 → 该领域为主
   - 最近工作经历的领域 → 主领域
   - 项目数量最多的领域 → 主领域

2. **副领域处理**：
   - 加载主领域 domain 配置作为评估基线
   - 副领域项目仍纳入模块三（项目综合题），但追问深度降一级
   - 副领域技术栈归入模块一（技术验证题），验证基础理解即可
   - 在题目中标注 [跨域:AI] 或 [跨域:前端]

3. **输出组织**：主领域占 60-70%，副领域占 30-40%

## 简历分析摘要

输出为 `00-resume-analysis.md`，包含：
- 识别到的技术栈和经验等级
- 推断的目标岗位（若未指定 position）
- 三模块题目数量预览（模块一 X 题 / 模块二 X 组 / 模块三 X 组）
- 面试官最可能追问的 3-5 个方向
- 简历中的潜在风险点（如"写了精通但项目中看不到深度使用"）
- 职业轨迹信号（跳槽/空窗/转型，如有）
- 建议准备的 behavioral 问题（如有空窗/跳槽信号）

## 隐私保护

- 输出文件中公司名用 [前公司A]、[前公司B] 代替（除非用户明确要求保留）
- 不在输出文件中重复手机号、邮箱、地址等联系信息
- 项目名可保留（通常不构成 PII）
- 如检测到身份证号、详细住址等高敏感信息，在输出开头提醒用户注意文件存储和分享范围

## 调研策略（简历模式专用）

- 聚焦简历中 Top 3-5 核心技术的高频面试追问方向
- 岗位相关：额外搜索 1-2 次该岗位特色面试题（如测试岗搜"软件测试面试高频题"）
- 不搜索具体公司/项目的内部信息（搜不到）
- 调研总搜索不超过 8 次
- 标注哪些追问方向来自调研，哪些来自 domain 配置
