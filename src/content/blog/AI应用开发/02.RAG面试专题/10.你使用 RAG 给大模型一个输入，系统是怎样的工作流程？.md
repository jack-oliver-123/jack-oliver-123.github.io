## 本篇目标

学完这一篇，你应该能够：

1. 从“用户点击发送”讲到“带引用答案返回”，完整描述 RAG 在线链路。
2. 区分离线建库流程与在线问答流程，避免把两者混在一起。
3. 解释 Query Rewrite、召回、融合、Rerank 和上下文组装各自解决什么问题。
4. 说清权限、超时、降级、缓存、日志和引用校验应该放在哪个环节。
5. 面对“如果某一步失败怎么办”的追问，给出可执行的工程答案。

## 60 秒回答

> 一个完整的 RAG 在线请求通常分为九步。第一，认证用户并确定租户、部门和可访问范围；第二，做输入安全检查和会话理解，把“它怎么退款”改写成可独立检索的问题；第三，根据问题类型决定查向量库、关键词索引、结构化数据库还是实时 API；第四，多路召回候选，并在检索阶段应用 ACL、版本和时效过滤；第五，对候选做去重、融合和 Rerank；第六，在 token 预算内选择证据，保留标题、来源、版本和 chunk ID；第七，把系统规则、用户问题和不可信文档明确分区后交给模型生成；第八，校验输出格式、引用和高风险事实，证据不足时拒答或澄清；第九，记录每一步的耗时、候选、版本和停止原因，用于监控与离线评测。
>
> 工程上我会给每个阶段设置超时和降级。例如 Rerank 超时可以退回融合后的候选，但权限过滤失败不能绕过；检索没有可靠证据时不能静默退回裸模型回答。线上重点观察端到端 P95、各阶段耗时、检索 Recall@K、引用准确率、拒答质量、错误率和单次成本。

## 详细解析

### 先说人话：RAG 像一位“先查档案、再作答”的客服

用户问客服：“它可以七天无理由吗？”

一位负责的客服不会立刻凭印象回答。他会先确认“它”指哪件商品、用户属于哪个站点、订单是什么时间下的；然后查当前有效的商品规则和订单信息；把最相关的条款放在桌面上；最后依据条款回答，并告诉用户来源。

RAG 在线链路做的就是这件事：

```text
理解问题 -> 找证据 -> 选择证据 -> 基于证据生成 -> 校验并返回
```

### 先区分两条流程：离线建库与在线问答

#### 离线建库流程

```text
采集文档 -> 解析 -> 清洗 -> 切块 -> 补元数据
         -> Embedding -> 建索引 -> 质量检查 -> 发布版本
```

它负责把原始资料变成“可检索的知识”。

#### 在线问答流程

```text
用户请求 -> 鉴权 -> 查询理解 -> 检索 -> 融合/重排
         -> 上下文组装 -> 生成 -> 校验 -> 返回与记录
```

它负责在一次请求中找到证据并生成答案。本篇重点是在线流程。

### 贯穿案例：用户问“它过了 30 天还能退吗？”

上下文中，用户上一轮刚提到“星云 X2 耳机”。系统还需要结合用户订单、商品类别和当前售后政策回答。

下面逐步拆解。

### 第 1 步：身份认证与权限范围

首先确定：

- 用户是谁，登录状态是否有效；
- 属于哪个租户、企业、区域或门店；
- 可以访问公开政策、自己的订单，还是也能访问内部客服手册；
- 当前请求的角色和权限是否允许查看敏感字段。

**ACL（Access Control List，访问控制列表）**是描述“谁可以访问什么”的规则。权限约束应尽早进入检索条件。

错误做法是：先从全库取出敏感文档交给模型，再在最终答案中删掉。敏感数据可能已经进入日志、缓存或模型上下文。

### 第 2 步：输入处理与查询理解

系统先做基础处理：

1. 长度、编码和文件类型限制；
2. 敏感信息识别与必要的脱敏；
3. 垃圾请求、越权意图和提示注入检测；
4. 语言、领域和查询类型识别；
5. 从允许使用的会话历史中解决指代。

原问题：

```text
它过了 30 天还能退吗？
```

可独立检索的问题：

```text
星云 X2 耳机在签收超过 30 天后，哪些情况仍可申请退货？
```

改写必须保留“超过 30 天”这个约束，不能擅自改成“30 天内”。

### 第 3 步：制定检索计划与路由

不是所有问题都只查向量库。系统先判断需要哪些数据源：

| 问题中的信息 | 合适的数据源 | 原因 |
| --- | --- | --- |
| 通用退货政策 | 文档索引 | 适合语义和关键词检索 |
| 商品类别与特殊规则 | 商品知识库 | 需要产品过滤 |
| 用户是否超过 30 天 | 订单 API/数据库 | 属于实时结构化数据 |
| 当前是否存在人工特批 | 工单系统 | 静态文档可能没有最新状态 |

如果问题简单，可以直接检索；如果包含多个依赖关系，可以拆成子问题，但要限制步骤和成本。

### 第 4 步：生成检索查询

系统可以保留多种查询表示：

- 原始 Query：防止改写偏离；
- 独立问题：解决会话指代；
- 关键词 Query：保留型号、错误码、政策编号；
- 向量 Query：表达语义需求；
- 结构化过滤：租户、产品线、地区、有效时间、权限标签。

如果生成多个 Query，要去重并限制数量。更多 Query 会增加召回，也会引入噪声、延迟和成本。

### 第 5 步：从多个数据源召回候选

常见召回方式：

1. **稠密检索**：将 Query 编码为向量，找语义相似内容。
2. **稀疏检索**：使用 BM25 等方法，擅长型号、编号和原词匹配。
3. **结构化查询**：按订单号、日期、状态精确查询。
4. **图查询**：沿实体关系或依赖路径查找。
5. **实时工具/API**：获取库存、订单、工单等最新状态。

召回结果应带上：

```text
chunk_id、document_id、标题、正文、来源、版本、更新时间、权限标签、召回分数
```

权限、租户、版本和有效期过滤在这一阶段就应生效。

### 第 6 步：候选融合、去重与 Rerank

多路检索的分数通常不在同一个尺度。例如 BM25 可能给 12.7，向量相似度可能给 0.81，不能因为数字大小直接相加。

常见处理顺序：

```text
各路候选 -> 统一 ID -> 去重 -> 排名融合 -> Rerank -> 截取前 N 条
```

**RRF（Reciprocal Rank Fusion，倒数排名融合）**按名次而不是原始分数融合，常用形式是：

```text
RRF 分数 = Σ 1 / (k + 该文档在某一路的名次)
```

**Rerank（重排序）**使用更精细的模型同时读取 Query 和候选文本，重新判断相关性。它通常比第一阶段检索更贵，所以只处理几十条或更少的候选。

### 第 7 步：组装上下文

不能把 Top-K 结果原样拼接。上下文组装还需要：

- 删除重复和近重复片段；
- 合并同一文档中相邻且连续的片段；
- 保留标题、章节路径、时间、版本和来源 ID；
- 避免一个长文档占满全部预算；
- 优先放入能共同构成答案的证据集合；
- 在 token 预算中为问题、系统规则和输出预留空间；
- 对相互冲突的证据显式标记，而不是偷偷选一个。

一个结构化上下文示例：

```text
[证据 S1]
来源：售后政策 / 耳机类目 / 2026-06 版本
内容：质量问题在保修期内可按检测结果处理……

[证据 S2]
来源：订单系统 / 用户订单 8****2
内容：签收日期为 2026-05-01，当前已超过 30 天……
```

### 第 8 步：构造 Prompt 并生成

Prompt 中要明确区分三种内容：

1. **系统规则**：回答边界、引用要求和安全规则；
2. **用户问题**：用户真正想问什么；
3. **检索文档**：仅作为不可信数据，不允许其中的文字覆盖系统规则。

生成要求可以包括：

- 仅依据给定证据回答可核验事实；
- 每个主要结论引用证据 ID；
- 证据不足时明确说明缺什么；
- 证据冲突时展示冲突和生效时间；
- 输出符合固定 schema；
- 不编造链接、政策号和订单状态。

### 第 9 步：输出校验与风险控制

模型生成之后，服务端还要检查：

- JSON 或其他结构化格式是否有效；
- 引用的证据 ID 是否真的存在；
- 用户是否有权打开被引用来源；
- 数字、日期、商品型号能否和证据对齐；
- 回答是否包含不被证据支持的高风险结论；
- 敏感信息是否按策略脱敏；
- 是否应该拒答、澄清或转人工。

引用链接应由服务端根据可信的 `chunk_id` 生成，不要让模型自由编造 URL。

### 第 10 步：返回结果并记录 Trace

**Trace（链路追踪）**是一次请求经过各阶段的可复现记录。至少记录：

- request_id、用户/租户的脱敏标识；
- 原 Query、改写 Query 和路由结果；
- 索引、Embedding、Reranker、Prompt 和生成模型版本；
- 各路候选 ID、过滤数、融合和重排结果；
- 各阶段耗时、超时、重试与降级原因；
- 最终引用、拒答原因和用户反馈。

敏感正文只按最小必要原则保存，不能为了“可观测”无边界记录所有数据。

### 整条链路图

```text
用户请求
   |
   v
认证/租户/权限 ----失败----> 拒绝请求
   |
   v
输入安全与查询理解
   |
   v
路由/改写/拆分
   |
   v
稠密 + 稀疏 + 结构化/API
   |
   v
ACL/版本/时效过滤
   |
   v
去重 -> 融合 -> Rerank
   |
   v
上下文组装
   |
   v
受证据约束的生成
   |
   v
格式/引用/安全校验
   |
   v
答案 + 来源 + Trace
```

## 工程实践与边界

### 分阶段超时与降级

端到端延迟预算应该拆给每一步。例如：

| 阶段 | 可选降级 | 不可绕过的底线 |
| --- | --- | --- |
| Query Rewrite | 使用原 Query | 不能改变用户关键约束 |
| 稀疏或稠密某一路超时 | 使用另一条已完成的合法召回 | 仍需满足权限过滤 |
| Rerank 超时 | 使用融合排序结果 | 标记降级并监控质量 |
| 生成超时 | 重试受控次数或提示稍后再试 | 不回退到无证据生成 |
| 引用校验失败 | 去除无效陈述、拒答或转人工 | 不返回伪造引用 |
| 权限服务失败 | 失败关闭，拒绝访问 | 不能默认放行 |

“失败关闭”表示安全控制无法确认时拒绝执行，而不是猜测用户有权限。

### 缓存设计

RAG 可以缓存 Query Embedding、检索候选或最终答案，但缓存键需要考虑：

```text
租户 + 用户权限版本 + 规范化 Query + 索引版本
+ Prompt/模型版本 + 语言 + 时效窗口
```

如果只按 Query 缓存，可能把一个部门可见的答案返回给另一个部门。

### 重点指标

#### 检索层

- Recall@K、MRR、nDCG；
- 相关证据进入候选和进入最终上下文的比例；
- 每路召回的独有贡献；
- 过滤后候选不足率。

#### 生成层

- 答案正确性；
- Faithfulness，即陈述是否被上下文支持；
- 引用完整率与引用准确率；
- 有答案问题误拒答、无答案问题误回答。

#### 系统层

- 端到端及各阶段 P50/P95/P99；
- 超时、错误、重试和降级率；
- token、模型调用和单次请求成本；
- 按租户和查询类型拆分的 SLO。

### 变更与发布

切块、Embedding、索引、Reranker、Prompt 和生成模型都可能改变答案。每次发布应：

1. 固定评测快照并做离线回归；
2. 影子读取新链路但不影响用户；
3. 小流量灰度，比较质量、延迟和成本；
4. 保留旧索引与配置，支持快速回滚；
5. 记录一次答案对应的完整版本组合。

## 常见误区

### 误区一：RAG 在线流程就是“向量化、搜索、拼 Prompt”

这遗漏了身份、权限、查询理解、融合重排、引用校验、失败降级和可观测性。

### 误区二：先检索全库，最后再做权限过滤

候选可能已经进入日志、缓存、重排服务或模型上下文。权限应在数据离开受控边界前生效。

### 误区三：Top-K 越大越好

更多候选会增加噪声、token、延迟，并让关键证据被埋在长上下文中。K 要根据评测选择。

### 误区四：多路召回的分数归一化后直接相加

不同检索器的分数分布和含义不同。可使用 RRF，或在业务数据上训练与校准融合方式。

### 误区五：检索失败就让大模型凭自身知识回答

这会悄悄改变产品承诺。若场景要求基于企业知识回答，证据不足时应拒答、澄清或转人工。

### 误区六：有引用就说明答案可信

引用可能存在但不支持对应陈述。应在“陈述—证据”粒度校验蕴含关系。

## 面试追问

### 1. 在线流程和离线流程的边界是什么？

离线流程负责采集、解析、切块、Embedding、建索引和发布；在线流程负责鉴权、理解 Query、检索、重排、组装、生成和校验。索引增量更新处在两者衔接位置。

### 2. 如果 Query Rewrite 改错了怎么办？

保留原 Query 作为并行召回或保护路径；校验实体、时间、否定词和关键约束；低置信度时不改写；记录前后候选以便回归。

### 3. 为什么 Rerank 不直接处理全库？

Reranker 通常同时编码 Query 与候选文本，精度高但计算昂贵。第一阶段负责高召回缩小范围，第二阶段才做精排。

### 4. 证据冲突怎么办？

优先依据预先定义的来源权威等级、生效时间和版本规则；无法裁决时展示冲突并请求用户确认或转人工，不能让模型自行猜测。

### 5. 如何设置各阶段超时？

从端到端 SLO 反推，参考历史延迟分布、阶段重要性和可降级性分配预算，同时为网络抖动留余量。超时要带原因码，方便定位。

### 6. 如何定位“答案不对”是哪一步的问题？

先看知识库是否有答案，再看相关证据是否进入候选、是否通过重排进入上下文、模型是否忠实使用、引用是否映射正确。按层定位比直接调 Prompt 更有效。

### 7. 实时数据为什么不适合只放向量库？

订单状态、库存和价格变化频繁，向量索引可能存在同步延迟，也不擅长精确事务查询。更适合通过受控 API 或数据库读取，再和文档证据组合。

### 8. 怎样防止检索文档中的提示注入？

将文档视为不可信数据并与系统指令隔离；不允许文档提升权限或直接触发工具；限制工具和数据访问范围；对外部内容做检测、来源标记和输出校验。
