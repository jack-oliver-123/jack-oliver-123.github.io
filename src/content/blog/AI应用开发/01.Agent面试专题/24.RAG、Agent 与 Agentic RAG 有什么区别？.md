## 本篇目标

分清 RAG、Agent 和 Agentic RAG 三个经常混用的概念；理解基础 RAG 的离线索引与在线问答流程，以及 Agentic RAG 如何把检索变成可反复决策的动作；能够根据任务复杂度、风险、成本和可测试性做选型。

## 60 秒回答

RAG 是 Retrieval-Augmented Generation，也就是检索增强生成。它先从外部知识库找出与问题相关的内容，再把这些证据交给大模型生成答案。工程中最常见的是固定流程：查询改写、检索、重排、拼接上下文、生成答案。它解决的是“模型怎样使用外部知识”，不天然等于 Agent。

Agent 是围绕目标持续行动的任务执行系统。它根据当前状态和环境反馈选择下一步，可以调用搜索、数据库、邮件、代码执行等工具。Agent 可能使用 RAG，也可能完全不需要检索，例如预约会议或执行退款。

Agentic RAG 不是一项单一协议，而是一类带有 Agent 决策闭环的 RAG 系统。它会动态判断要不要检索、去哪里检索、如何拆分问题、是否改写查询、当前证据是否充分，以及应该继续搜索还是停止。基础 RAG 是“按固定流水线查一次再回答”，Agentic RAG 更像“围绕问题反复查证的研究员”。

选型上，单知识库 FAQ、政策问答优先基础 RAG；需要多跳推理、多数据源、查询纠错和证据核验时再考虑 Agentic RAG；若任务还要执行退款、发信等外部动作，则应把 RAG 作为 Agent 的一个知识工具，并为动作单独设置权限和审批。

一句话记忆：

> **RAG 重点是“把知识找来”，Agent 重点是“决定下一步做什么”，Agentic RAG 重点是“动态决定怎样找、找多少、怎样验证”。**

## 详细解析

### 先用查资料理解三者

假设用户问：

> “我去年买的海外版产品现在坏了，能不能在国内保修？”

**基础 RAG** 像一名按照固定流程工作的资料员：把问题拿去知识库搜索一次，取回最相关的几段保修政策，然后据此组织答案。

**Agent** 像一名可以办事的客服专员。它可能查询订单、询问序列号、读取政策、创建维修单，也可能在权限不足时转人工。检索只是它可以使用的一种能力。

**Agentic RAG** 像一名会自己调整调查方法的研究员。它先发现问题包含“购买地区、产品版本、购买时间、维修地区”四个条件，于是拆成多个子问题；如果内部知识库没有海外版条款，就换到地区政策库；如果检索结果相互冲突，就查生效日期和权威级别；证据仍不足时，它会向用户补问，而不是强行回答。

三者不是三个同级产品，更准确地说：

- RAG 是一种给生成模型补充外部知识的系统模式；
- Agent 是一种围绕目标进行决策和行动的系统形态；
- Agentic RAG 是两者的交叉区域，把更强的动态决策用于检索过程。

### 先认识 RAG 中的常用术语

| 术语 | 通俗解释 |
| --- | --- |
| 文档切分（Chunking） | 把长文档切成便于检索的小段 |
| Embedding | 把文字变成能表示语义位置的数字向量 |
| 向量检索 | 找到语义上相近的内容，不只匹配相同关键词 |
| 关键词检索 | 根据词语出现情况搜索，例如 BM25 一类方法 |
| 混合检索 | 同时利用关键词与向量检索，再合并结果 |
| Metadata | 文档的来源、时间、产品、地区、权限等附加信息 |
| Top-K | 取回得分最高的 K 条结果 |
| Reranker | 对初次召回的候选结果再次精细排序 |
| Grounding | 让答案建立在提供的证据上，而不是只靠模型记忆 |
| Citation | 在答案中标明证据来自哪里 |

Embedding 和向量数据库很常见，但它们不是 RAG 的定义本身。只用关键词搜索也可以构成 RAG；反过来，只把文档存进向量数据库但生成时不使用检索结果，也不能算完整的 RAG。

### 基础 RAG 其实有两个阶段

很多初学者只看到“提问 -> 检索 -> 回答”，忽略了前面的知识准备。一个完整 RAG 通常分为离线索引和在线查询两个阶段。

**阶段一：离线索引**

```text
原始文档
  ↓
解析与清洗
  ↓
按语义或结构切分
  ↓
补充来源、时间、权限等 Metadata
  ↓
建立关键词索引和 / 或向量索引
```

**阶段二：在线回答**

```text
用户问题
  ↓
必要时改写或补全查询
  ↓
按权限过滤并检索候选片段
  ↓
重排、去重和控制上下文长度
  ↓
把问题与证据交给大模型
  ↓
生成带依据的答案
```

原始 RAG 论文把预训练模型的参数称为“参数化记忆”，把可检索的外部索引称为“非参数化记忆”。今天工程语境中的 RAG 已经更宽泛，通常指任何“先从外部知识源取证，再辅助生成”的架构。

### 基础 RAG 的路径通常是固定的

最常见的基础 RAG 由开发者提前规定：每个问题都检索一次，每次取固定数量的片段，再调用一次模型回答。即使其中使用了大模型做查询改写或重排，只要路径和分支仍由代码预先规定，它依然主要是 Workflow。

例如：

```text
问题 -> 改写一次 -> 知识库检索 Top 10 -> 重排 Top 3 -> 生成
```

固定流程的优势很实际：

- 延迟和成本较稳定；
- 容易复现与测试；
- 权限边界清楚；
- 简单问题往往已经足够；
- 故障定位比开放式循环容易。

因此，Agentic RAG 并不天然比基础 RAG 更高级。若固定流程能可靠完成任务，引入自主循环只会增加复杂度。

### Agent 的范围比检索大得多

Agent 的核心是状态、动作和反馈闭环：

```text
观察目标与状态 -> 选择动作 -> 执行 -> 读取结果 -> 更新状态 -> 继续或停止
```

它的动作可能包括：

- 搜索知识库；
- 查询订单数据库；
- 调用计算器；
- 运行代码；
- 创建工单；
- 请求人工审批；
- 向用户补问。

所以两个判断都成立：

- 一个 Agent 可以把 RAG 当作工具；
- 一个 RAG 系统可以完全没有 Agent。

“用了向量数据库”不能证明系统是 Agent，“会调用搜索工具”也不能单独证明系统是 Agent。还要看它是否会根据检索反馈动态改变后续动作。

### Agentic RAG 到底“Agentic”在哪里

“Agentic RAG”目前更像行业中的总称，而不是字段和流程都统一的正式协议。不同系统的自主程度差异很大，通常会包含下面的一部分或全部能力。

#### 1. 判断是否需要检索

“把这句话翻译成英文”通常不需要查知识库；“今天的汇率是多少”需要外部新信息。系统可以先判断是否需要检索，避免每个请求都机械搜索。

#### 2. 拆分多跳问题

问题“某产品在政策更新前购买，现在是否仍享受旧条款”至少要查：

- 购买时间；
- 新旧政策生效时间；
- 是否有过渡条款；
- 产品和地区是否适用。

Agentic RAG 可以把问题拆成多个子查询，再合并证据。这里的“多跳”是指后一部分推理依赖前一部分结果，不是一条搜索就能直接得到答案。

#### 3. 选择数据源与检索方式

不同问题适合不同来源：

| 问题 | 更合适的数据源 |
| --- | --- |
| 公司制度 | 内部政策库 |
| 当前订单状态 | 事务数据库或订单 API |
| 产品技术说明 | 官方文档库 |
| 最新公开事件 | 受控网络搜索 |

系统还可以根据专有名词、编号和自然语言表达选择关键词检索、向量检索或混合检索。

#### 4. 改写和扩展查询

第一次没有找到，并不一定表示知识库没有答案。可能是用户用了简称、错别字或口语。Agentic RAG 可以根据失败原因：

- 补充产品全称；
- 加入时间或地区限制；
- 为缩写生成同义词；
- 把宽问题拆成更窄问题；
- 取消过度严格但不影响权限的内容过滤条件。

权限过滤不能因为“搜不到”而被取消。

#### 5. 评估检索结果质量

系统需要判断取回的内容是否：

- 与问题相关；
- 来源权威；
- 时间仍然有效；
- 彼此一致；
- 足以支撑结论；
- 允许当前用户访问。

Self-RAG 研究了按需检索以及对检索结果和生成内容进行自我反思；CRAG 研究了先评估检索质量，再按置信程度采取纠正动作。这些工作体现了“检索不是一次固定动作”的思路，但不必把每一种自适应 RAG 都叫作通用 Agent。

#### 6. 决定继续、补问还是停止

Agentic RAG 必须有停止条件，否则它可能不停换关键词和数据源。常见停止条件包括：

- 已获得足够且一致的权威证据；
- 必要字段缺失，需要向用户提问；
- 达到最大检索轮数、时间或费用；
- 权限不足；
- 所有允许数据源都没有可靠证据；
- 需要人工专家判断。

“没有足够证据”也是一种正确结果。可靠系统应该允许回答“不确定”或“需要补充信息”。

### 三者核心对比

| 对比项 | 基础 RAG | Agent | Agentic RAG |
| --- | --- | --- | --- |
| 核心目的 | 用外部知识辅助生成 | 围绕目标持续完成任务 | 动态规划、执行和验证检索 |
| 是否以检索为必要能力 | 是 | 否 | 是，或至少会判断是否需要 |
| 典型路径 | 开发者预设的固定流水线 | 根据状态动态选择动作 | 根据证据动态选择检索动作 |
| 检索次数 | 通常固定一次或固定轮数 | 不确定 | 可能零次、一次或多次 |
| 数据源 | 通常预先指定 | 可以使用多种工具 | 可在允许的数据源中动态路由 |
| 是否执行外部动作 | 通常不执行 | 可以 | 重点仍是检索，也可属于更大 Agent |
| 状态与反馈 | 较少，流程内状态为主 | 核心能力 | 记录查询、证据、缺口和预算 |
| 延迟与成本 | 相对稳定 | 变化较大 | 通常高于基础 RAG |
| 可复现性 | 较强 | 较弱 | 介于固定 RAG 与开放 Agent 之间 |
| 典型场景 | FAQ、政策问答 | 客服处理、自动化任务 | 复杂研究、多跳问答、跨源查证 |

### 同一个客服问题的三种实现

**基础 RAG：**

```text
用户问题
  -> 在售后政策库检索
  -> 取回最相关片段
  -> 生成答案并引用政策
```

**Agentic RAG：**

```text
用户问题
  -> 判断缺少购买地区
  -> 向用户补问
  -> 拆分“产品版本、购买时间、维修地区”
  -> 分别查询地区政策库和产品政策库
  -> 检查政策生效日期与冲突
  -> 证据充分后生成带引用的结论
```

**包含 RAG 的完整客服 Agent：**

```text
先用 Agentic RAG 判断保修资格
  -> 查询订单和设备序列号
  -> 用户确认后创建维修单
  -> 预约网点
  -> 返回工单编号
```

最后一种不仅“回答知识问题”，还改变了外部业务状态，因此必须增加工具权限、审批、幂等和审计。

### 什么时候选哪一种

可以先问五个问题：

1. 一次检索能否稳定找到答案？
2. 问题是否需要拆分成多个相互依赖的子问题？
3. 是否存在多个用途不同的数据源？
4. 检索失败后，能否用固定规则处理？
5. 任务只是回答，还是还要执行外部动作？

常见选择如下：

| 场景 | 推荐起点 | 原因 |
| --- | --- | --- |
| 单一知识库 FAQ | 基础 RAG | 路径稳定，容易测试且成本低 |
| 企业政策问答并带引用 | 基础 RAG + 重排 + 权限过滤 | 通常不需要开放式决策 |
| 跨多个文档库的复杂研究 | Agentic RAG | 需要拆题、路由、反复查证 |
| 检索质量经常波动 | 纠错型 / 自适应 RAG | 需要评估后改写或换源 |
| 处理退款、发信、建工单 | Agent，把 RAG 作为知识工具 | 除知识外还包含真实动作 |
| 纯创作或格式转换 | 普通模型调用 | 外部检索可能没有价值 |

原则仍然是从最简单的可靠方案开始，用真实评测证明复杂性有收益后再升级。

### RAG 为什么也会“有资料却答错”

RAG 不是消除幻觉的开关。失败通常发生在四个位置：

| 层次 | 典型失败 |
| --- | --- |
| 知识层 | 文档过期、缺失、重复、解析错误、权限标签错误 |
| 检索层 | 查询表达不匹配、切分不合理、相关片段没有召回 |
| 上下文层 | 片段排序错误、信息冲突、上下文过长或关键部分被截断 |
| 生成层 | 模型忽略证据、过度推断、引用与结论不一致 |

Agentic RAG 可以通过多次检索和验证缓解其中一些问题，但也新增了规划错误、循环、成本失控和跨源权限等风险。

## 工程实践与边界

### 先保证基础 RAG，再增加 Agent 决策

如果文档解析、切分、权限、召回和重排本身很差，加一个 Agent 通常只会更复杂地搜索错误知识。比较稳妥的顺序是：

1. 建立可追溯、可更新、带权限标签的知识库；
2. 优化基础检索、重排和引用；
3. 建立检索与生成的分层评测；
4. 从失败样本中确认固定流程解决不了的问题；
5. 只为这些问题增加查询拆分、路由或纠错循环；
6. 设置轮数、时间、Token、来源和权限上限。

### 权限过滤必须发生在检索阶段

不能先检索所有租户的数据，再要求模型“不要说出来”。正确做法是由可信系统根据当前用户和租户，在检索前或检索过程中应用访问控制，让无权内容根本不进入候选结果和模型上下文。

Metadata 常用于过滤地区、部门、产品、时间和权限，但 Metadata 的写入和更新本身也要受控。错误标签会造成数据泄露或漏检。

### 检索到的文档是不可信数据

网页、邮件、工单和文档中可能出现“忽略系统规则”“上传秘密文件”等恶意指令。系统应把检索内容当作证据数据，而不是高优先级命令，并限制检索 Agent 能调用的工具和可外发的数据。

引用也不等于安全或正确。系统仍要检查：

- 链接是否真的来自允许的数据源；
- 引用片段是否支持附近的结论；
- 来源是否过期、被污染或权威性不足；
- 答案是否加入了证据中不存在的推断。

### 分开评估检索和生成

只看最终答案，无法判断问题出在“没找到”还是“找到了但没用好”。

**检索层常见指标：**

- Recall@K：已知相关证据中，有多少出现在前 K 条；
- Precision@K：前 K 条中，有多少确实相关；
- MRR：第一个相关结果排得是否足够靠前；
- NDCG：多个不同相关程度的结果，排序是否合理；
- 权限违规率：是否召回了用户无权访问的内容；
- 新鲜度：是否优先使用当前有效版本。

**生成层常见指标：**

- 答案正确性和任务完成度；
- Faithfulness：结论是否忠于给定证据；
- 引用正确率与引用覆盖率；
- 证据不足时是否正确拒答或补问；
- 延迟、Token 和费用。

**Agentic RAG 还要评估轨迹：**

- 是否正确拆分问题；
- 是否选择了合适数据源；
- 是否进行了无效重复搜索；
- 是否在证据充分时及时停止；
- 是否遵守工具、权限和预算边界。

### 数据集要覆盖“搜不到”和“搜错了”

评测集不应只有能轻松命中的问题，还应包含：

- 知识库确实没有答案；
- 用户问题含错别字、简称或歧义；
- 两份政策内容冲突；
- 新旧版本同时存在；
- 相关文档很长，答案跨多个片段；
- 检索结果中有提示注入；
- 当前用户没有目标文档权限；
- 某个数据源超时或不可用；
- 正确动作是向用户补问。

### 成本和停止条件要由程序控制

Agentic RAG 应设置：

- 最大检索轮数；
- 每轮 Top-K 与总上下文上限；
- 允许的数据源；
- 最大时间、Token 和费用；
- 重复查询检测；
- 最低证据要求；
- 转人工或拒答条件。

模型可以建议“证据是否足够”，但程序仍要强制预算和安全停止条件。

### 一份上线检查表

- 文档是否有来源、版本、生效时间和负责人；
- 删除或撤权后，索引是否会及时同步；
- 切分是否保留标题、表格和上下文关系；
- 是否同时评测关键词、向量和混合检索；
- 是否有重排、去重和上下文长度控制；
- ACL 是否在检索前由可信系统执行；
- 是否把检索文本当作不可信数据；
- 答案是否能追溯到具体证据；
- 是否允许证据不足时拒答或补问；
- Agentic 循环是否有预算、停止和人工接管；
- 模型、提示、索引、文档和评测集是否版本化。

## 常见误区

- **“用了向量数据库就是 RAG”**：还需要在生成时检索并使用外部证据；RAG 也可以使用关键词或混合检索。
- **“RAG 就是 Agent”**：固定的检索再生成流程通常是 Workflow，没有动态决策闭环。
- **“会多查几次就是 Agentic RAG”**：关键是后续检索是否根据当前证据和状态动态决定，并且有明确停止条件。
- **“Agentic RAG 天然优于基础 RAG”**：简单任务中，它可能只增加延迟、成本和不确定性。
- **“检索到了正确文档，答案就可靠”**：模型仍可能忽略、误读或过度推断证据。
- **“有引用就没有幻觉”**：引用可能不支持结论，也可能来自过期或不可信来源。
- **“Top-K 越大越好”**：更多片段会增加噪声和上下文成本，甚至稀释真正证据。
- **“搜不到就取消过滤条件”**：内容条件可以调整，租户和权限过滤绝不能为了召回率而放宽。
- **“让模型自己判断停止就够了”**：还需要程序化的轮数、时间、费用和权限边界。

## 面试追问

> **面试官：** RAG、Agent 和 Agentic RAG 的本质区别是什么？
>
> **候选人：** RAG 解决外部知识怎样进入生成过程；Agent 解决系统怎样围绕目标根据反馈持续选择动作；Agentic RAG 是两者的交叉，把是否检索、查询拆分、数据源路由、结果评估和停止都放进动态决策闭环。三者不是简单的高低级关系。

> **面试官：** 普通 RAG 为什么通常算 Workflow？
>
> **候选人：** 因为查询改写、检索 Top-K、重排和生成等步骤通常由开发者预先固定。即使某些节点用了大模型，只要后续路径不是根据证据动态选择，控制权仍主要属于 Workflow。

> **面试官：** Agent 必须使用 RAG 吗？
>
> **候选人：** 不一定。预约会议、调用计算器或执行固定业务动作可能不需要知识检索。反过来，FAQ RAG 也可以没有 Agent。是否需要 RAG 取决于任务是否需要外部、私有或及时知识。

> **面试官：** 什么场景值得使用 Agentic RAG？
>
> **候选人：** 多跳问题、多个异构数据源、查询表达不稳定、检索结果需要纠错或核验，并且基础固定流程在真实评测中明显失败时值得使用。单库 FAQ 通常先用基础 RAG。

> **面试官：** Agentic RAG 常见的动态动作有哪些？
>
> **候选人：** 判断是否检索、拆分子问题、改写查询、选择数据源和检索方式、评估证据质量、处理冲突、继续搜索、向用户补问，以及在证据充分或预算耗尽时停止。

> **面试官：** 怎样评估一个 RAG 系统？
>
> **候选人：** 我会把知识、检索、上下文和生成分开。检索看 Recall@K、Precision@K、排序、权限和新鲜度；生成看正确性、忠实度、引用和拒答；Agentic RAG 再看数据源选择、重复搜索、停止、成本和安全轨迹。

> **面试官：** 怎样避免 RAG 跨租户泄露数据？
>
> **候选人：** 当前身份和租户由可信运行时提供，在检索阶段应用资源级 ACL，让无权文档不进入候选集和模型上下文。不能先全量检索，再依赖提示词要求模型保密。

> **面试官：** 检索不到结果时应该怎么办？
>
> **候选人：** 先区分查询表达问题、数据源故障、权限不足和知识确实缺失。可以在不改变权限的前提下改写查询、换允许的数据源或补问；仍无证据时明确拒答或转人工，不能凭模型记忆编造。

> **面试官：** Self-RAG 和 CRAG 与 Agentic RAG 有什么关系？
>
> **候选人：** 它们代表自适应检索的重要思路。Self-RAG研究按需检索和自我反思，CRAG研究先评估检索质量再触发纠正动作。它们可以被归入广义的 Agentic RAG 技术谱系，但不应因此把每个方法都等同于具备任意工具和完整目标闭环的通用 Agent。

> **面试官：** RAG 系统已经能回答，为什么还要保存引用？
>
> **候选人：** 引用让用户和评测系统能够核验来源、版本与证据，也便于定位是检索错还是生成错。但引用本身不保证结论成立，还要检查引用片段是否真正支持对应陈述。
