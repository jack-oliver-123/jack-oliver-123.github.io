## 本篇目标

学完这一篇，你应该能够：

- 区分“使用图数据库”和“完整 GraphRAG 方法”；
- 理解实体、关系、Claim、社区和社区摘要的作用；
- 解释为什么实体消歧不能只按名称相同就合并；
- 设计带证据血缘、版本、增量重算和回滚的图索引；
- 识别 GraphRAG 在抽取、社区计算、摘要、查询和权限上的成本；
- 判断什么问题值得使用 GraphRAG，什么问题普通 RAG 更合适。

## 60 秒回答

生产 GraphRAG 的难点不是把三元组写入图数据库，而是让自动抽取的图长期可信、可追溯和可更新。实体消歧要结合规范名、别名、类型、上下文、时间、邻接关系与外部业务主键；低置信实体宁可暂不合并或进入人工队列。每个实体、关系、Claim 和摘要都保存来源文档、原文位置、抽取模型、版本和置信信息，支持后续拆分、撤销和审计。

增量更新时，新增或删除一篇文档可能改变实体支持证据、边、社区结构和社区摘要，所以要维护反向 lineage，计算受影响子图，并在影响范围较小时局部重算，超过阈值时构建完整新版本。图、社区、摘要、向量和查询配置使用同一发布 ID，先影子评测、再灰度和版本切换，保留旧版本回滚。

成本要拆成文档解析、实体关系抽取、实体消歧、社区计算、摘要、Embedding、存储和在线查询分别测量。可以限制实体类型和高价值语料、缓存内容哈希、批处理、使用分层模型，并按查询路由：局部事实走普通 RAG，需要跨文档关系、主题演化和全局总结时才走 GraphRAG。ACL 需要沿来源传播，不能让混合权限文档生成一个无法拆分的共享摘要。

## 详细解析

### 先说人话：普通 RAG 找“几段资料”，GraphRAG 试图画“关系地图”

用户问：

> A 公司和 B 项目有什么关系？

普通向量检索可能找到同时出现“A 公司”和“B 项目”的几段文字。
但真正答案可能分散在三份文档：

1. A 公司投资了 C 公司；
2. C 公司负责 B 项目的供应链；
3. 某负责人同时担任两边的董事。

GraphRAG 会尝试把“公司、项目、人物、投资、供应”等对象和关系组织成图，再沿关系找到跨文档证据。

### 贯穿案例：差旅制度背后的组织关系

公司有以下资料：

- 《中国区差旅制度》由集团财务部发布；
- 上海研发中心适用中国区制度；
- 海外事业部有独立补充规则；
- “王明”既可能是上海研发中心员工，也可能是另一位财务审批人；
- 旧文档把“集团财务”和“财务共享中心”混用；
- 某会议纪要讨论了制度，但没有发布权限。

用户问：

> 上海研发中心的海外客户项目，住宿标准由哪个部门的哪套规则决定？

这可能需要理解：

- 组织隶属；
- 项目归属；
- 规则发布方；
- 补充规则覆盖关系；
- 文档生效时间；
- 人名和部门别名。

这类跨文档关系问题是 GraphRAG 更有机会体现价值的场景。

### 先认识七个基础术语

| 术语 | 通俗解释 | 示例 |
| --- | --- | --- |
| 实体（Entity） | 图中的对象 | 上海研发中心、王明 |
| 关系（Relationship） | 对象之间的连接 | 隶属于、负责、发布 |
| Claim | 带来源的事实陈述 | “集团财务部发布制度” |
| 实体消歧（Entity Resolution） | 判断不同提法是否是同一对象 | 集团财务 = 集团财务部？ |
| 社区（Community） | 关系更紧密的一组节点 | 中国区组织与制度群 |
| 社区摘要 | 对一个社区主要信息的总结 | 中国区差旅治理概览 |
| Provenance / Lineage | 每个结论来自哪里、如何派生 | 边来自哪份文档哪一段 |

### 有图数据库不等于 GraphRAG

#### 只用图数据库的 RAG

如果企业本来就有高质量业务图：

~~~text
员工 → 部门 → 地区 → 适用制度
~~~

RAG 可以把自然语言转成受控图查询，拿到结构化结果，再生成解释。这是图增强 RAG。

#### Microsoft GraphRAG 方法的典型思路

公开 GraphRAG 研究和实现包含：

~~~text
文档切分
→ 模型抽取实体、关系和 Claim
→ 实体消歧与图构建
→ 社区检测
→ 为社区生成报告或摘要
→ 建立文本与向量等索引
→ Local / Global 等查询方式
~~~

它不只是“把 chunk 存进 Neo4j”。面试中要说清使用的是哪种图方案。

### 图索引从哪里来

#### 1. 实体抽取

从文本识别：

- 人；
- 组织；
- 地点；
- 产品；
- 项目；
- 制度；
- 事件；
- 业务自定义类型。

实体类型不要无限扩张。类型越多，抽取 Prompt、校验、图结构和评测成本越高。先围绕业务问题定义 schema。

#### 2. 关系抽取

例如：

~~~text
上海研发中心 --隶属于--> 中国区
集团财务部 --发布--> 差旅制度
海外客户项目 --归属--> 海外事业部
补充规定 --覆盖--> 差旅制度部分条款
~~~

每条边都要指向原文证据，不能只保存模型生成的三元组。

#### 3. Claim 抽取

Claim 是更完整的事实陈述，可能包含：

- 主体；
- 行为或属性；
- 客体；
- 时间；
- 状态；
- 来源；
- 置信信息。

例如：

~~~text
主体：集团财务部
事实：发布 2025 版中国区差旅制度
生效时间：2025-01-01
来源：policy-2025.pdf 第 1 页
~~~

#### 4. 社区检测和摘要

社区检测把连接紧密的实体分组。GraphRAG 论文使用层级社区与社区报告，支持从局部文档事实上升到全局主题总结。

社区摘要本身是派生数据，也可能幻觉、过期或混入无权内容，所以要保存：

- 输入节点和边；
- 支持文档；
- 生成模型和 Prompt；
- 摘要版本；
- 权限域；
- 生成时间。

### 实体消歧为什么很难

#### 同名不一定是同一个实体

- 两个员工都叫王明；
- Apple 可能是公司，也可能是水果；
- “研发一部”在不同地区都存在；
- 同一项目名称可能被不同年份重复使用。

#### 不同名字可能是同一个实体

- 集团财务；
- 集团财务部；
- Group Finance；
- 财务总部；
- 内部系统 ID FIN-001。

如果该合并的没合并，图会碎成多个节点；如果不该合并的合并了，会制造虚假路径。

### 一条可靠的实体消歧流程

~~~text
抽取实体提及
→ 规范化名称
→ 生成候选实体
→ 根据类型、主键、上下文、时间和邻居评分
→ 高置信自动链接
→ 中等置信人工审核或暂缓合并
→ 低置信创建新实体
→ 保存决策证据和版本
~~~

#### 第一步：名称规范化

可以处理：

- 大小写；
- 全角半角；
- 标点和空格；
- 常见公司后缀；
- 语言和音译；
- 维护过的别名。

规范化只用于候选生成，不能直接证明两个实体相同。

#### 第二步：优先使用权威主键

如果实体来自业务系统，尽量使用：

- 员工 ID；
- 组织 ID；
- 客户 ID；
- 项目 ID；
- 法人统一代码；
- 产品 SKU。

稳定主键通常比模型猜测更可靠。

#### 第三步：结合上下文特征

| 特征 | 示例 |
| --- | --- |
| 类型 | 人、公司、项目是否一致 |
| 时间 | 两个任职时间是否冲突 |
| 地点 | 上海王明与北京王明 |
| 属性 | 邮箱、部门、职位 |
| 邻居 | 是否连接到相同组织和项目 |
| 来源 | 权威 HR 系统还是会议纪要 |

#### 第四步：保留不确定性

低置信时不要为了让图“更干净”强行合并。可以：

- 暂存两个节点；
- 记录 same_as_candidate；
- 进入人工队列；
- 查询时提示不确定；
- 后续证据增加后再判断。

#### 第五步：支持拆分与撤销

系统需要保存：

- 哪次运行做了合并；
- 使用哪些证据；
- 合并前节点；
- 影响的边、社区和摘要。

一旦发现误合并，能够拆分并重算派生数据。

### 关系也要做证据治理

一条边不只是：

~~~text
A --负责--> B
~~~

还应保存：

- 来源文档和原文；
- 页码或位置；
- 抽取模型与 Prompt；
- 抽取时间；
- 有效时间；
- 支持该关系的来源数量；
- 来源权威等级；
- 是否被人工确认；
- ACL。

多个来源支持同一关系时，可以记录多个 evidence，而不是复制多条语义相同的边。

### 删除一篇文档为什么会影响整张图

假设一份文档是“王明负责 B 项目”唯一证据。删除后：

1. 该 Claim 失去支持；
2. 对应关系可能删除；
3. 王明节点可能变成孤立节点；
4. 社区结构可能变化；
5. 社区摘要中的“王明负责 B 项目”失效；
6. 向量索引和查询缓存也要更新。

所以 GraphRAG 增量更新比普通 chunk upsert 更复杂。

### 增量更新需要反向 Lineage

至少能从源文档反向找到：

~~~text
源文档
→ 文本单元
→ 实体提及
→ 规范实体
→ 关系和 Claim
→ 社区
→ 社区摘要
→ Embedding
→ 查询缓存
~~~

没有 lineage，就难以证明删除完整，也无法计算局部重算范围。

### 新增、修改和删除怎么处理

#### 新增文档

1. 内容哈希去重；
2. 抽取实体和关系；
3. 与现有实体做消歧；
4. 新增证据支持；
5. 识别受影响社区；
6. 重算必要摘要和向量；
7. 进入待发布图版本。

#### 修改文档

不要直接追加新抽取结果。先比较旧版本：

- 哪些文本单元删除；
- 哪些实体提及变化；
- 哪些边失去或新增证据；
- 哪些 Claim 时间或状态改变；
- 哪些社区受影响。

#### 删除文档

- 标记源版本不可见；
- 撤销它提供的 evidence；
- 只有没有其他有效证据时才删除关系；
- 清理孤儿实体；
- 重算受影响社区和摘要；
- 失效向量与缓存；
- 保留必要审计记录。

### 局部重算还是全量重建

| 情况 | 更可能的策略 |
| --- | --- |
| 少量独立文档新增 | 局部抽取与社区更新 |
| 单篇文档修正 | 按 lineage 局部重算 |
| 一个高连接中心实体变化 | 扩大影响子图或全量重算 |
| 实体 schema 改变 | 全量重建 |
| 消歧模型或规则大改 | 影子全量重建 |
| 社区算法或参数改变 | 重新计算社区和摘要 |
| Embedding 模型改变 | 重建对应向量索引 |
| 大量删除或合并 | 评估后常选择全量重建 |

可以设置影响范围阈值：

- 受影响节点数；
- 受影响边比例；
- 触及社区数量；
- 摘要重算量；
- 预计成本。

超过阈值时，全量重建可能更简单、更可验证。

### 图版本怎么发布

不要让实体表是 v12、社区摘要是 v11、向量却是 v10。可以使用统一发布 ID：

~~~text
graph_release_2025_08_17_01
├─ entities
├─ relationships
├─ claims
├─ communities
├─ community_reports
├─ embeddings
└─ query_config
~~~

发布流程：

1. 构建影子版本；
2. 做结构、来源、权限和质量校验；
3. 跑实体消歧、关系和问答回归；
4. 用影子流量比较新旧结果；
5. 按租户或查询类型灰度；
6. 原子切换版本路由；
7. 保留旧版本到观察期结束。

### GraphRAG 成本花在哪里

#### 离线成本

| 阶段 | 主要成本 |
| --- | --- |
| 文档解析和切块 | CPU、OCR、存储 |
| 实体关系抽取 | 大量模型输入输出 token |
| Claim 抽取 | 额外模型调用 |
| 实体消歧 | 候选比较、模型或人工审核 |
| 社区检测 | 图计算 |
| 社区报告 | 社区内容的模型总结 |
| Embedding | 节点、文本和摘要向量化 |
| 图与向量存储 | 存储、备份和复制 |

#### 在线成本

- 查询理解；
- Local 或 Global 搜索；
- 图遍历；
- 社区报告选择；
- 多轮汇总；
- 最终生成；
- 可能的验证与引用。

全局问题可能要读取和汇总多个社区报告，成本与普通 top K RAG 的模式不同。

### 如何控制成本

#### 1. 只对高价值语料构图

不必把所有邮件、重复网页和低价值附件都做完整实体关系抽取。

#### 2. 限定业务 Schema

只抽取真正影响问题的实体类型、关系和 Claim，减少 token、噪声和消歧组合。

#### 3. 内容哈希与结果缓存

文档内容和抽取配置未变时，可复用已验证结果。缓存键要包含模型、Prompt 和 schema 版本。

#### 4. 分层使用模型

可用较轻模型做候选抽取，规则与强模型处理困难样本，但需要评测证明不会明显损害图质量。

#### 5. 批处理和并发控制

批量抽取、异步队列和配额管理可以提高吞吐，但要保留每份来源的 lineage 和错误隔离。

#### 6. 查询路由

~~~text
简单局部事实 → 普通 RAG
精确结构化关系 → 图查询
跨文档关系 → GraphRAG Local Search
全局主题与趋势 → GraphRAG Global Search
~~~

不让每个 FAQ 都支付图查询和多级总结成本。

### GraphRAG 的 Local 和 Global 思路

#### Local Search

从查询相关实体出发，结合邻居、关系、文本单元和社区信息回答局部问题。适合：

- 某人和某项目的关系；
- 一个实体的属性和证据；
- 局部多跳路径。

#### Global Search

利用不同层级社区报告，对整个语料的主题、趋势或整体问题进行汇总。适合：

- 整个数据集有哪些主要风险；
- 多个部门共同关注什么；
- 一个主题在全局如何演化。

如果用户只问某条制度金额，Global Search 通常没有必要。

### ACL 是 GraphRAG 的高风险难点

假设社区摘要同时由：

- 所有人可看的制度；
- 只有财务可看的预算；
- 只有 HR 可看的薪酬文档

共同生成。即使查询时过滤掉秘密节点，摘要文字里可能已经混入秘密事实。

可选策略：

- 在 ACL 同质的数据分区内构图和生成摘要；
- 按租户或权限域维护独立图；
- 不预生成跨权限摘要，在请求时仅对授权证据汇总；
- 为摘要记录完整来源和权限上界；
- 权限收紧时重算相关社区摘要；
- 缓存键绑定租户、ACL 和图版本。

“查询时过滤节点”无法从已经生成的混合摘要中可靠删除越权信息。

### GraphRAG 怎么评测

#### 图构建质量

- 实体识别 Precision、Recall、F1；
- 实体链接或消歧准确率；
- 关系抽取 Precision、Recall、F1；
- 边的证据支持率；
- 无来源节点和边比例；
- 重复实体和误合并率；
- 时间与状态字段正确率。

#### 社区与摘要

- 社区稳定性；
- 摘要事实支持率；
- 关键实体和关系覆盖；
- 过期信息比例；
- 权限混合违规；
- 摘要更新延迟。

#### 查询质量

- 局部关系问答；
- 多跳证据覆盖；
- 全局主题问答；
- 引用正确性；
- 与普通 RAG 的增益；
- 无答案与冲突处理。

#### 系统指标

- 每千文档抽取成本；
- 每次增量更新成本；
- 全量重建时间；
- 图和向量存储；
- P95/P99 查询延迟；
- 单位成功任务成本；
- 回滚和删除传播时间。

## 工程实践与边界

### 先从“可验证小图”开始

推荐：

1. 选一个跨文档关系明确的业务；
2. 定义有限实体与关系 schema；
3. 建一组实体、关系和问答金标；
4. 与强普通 RAG 基线比较；
5. 评估索引成本和增量更新；
6. 证明收益后再扩大语料。

不要先把全部企业文档构成一张巨图，再寻找使用场景。

### 对高影响实体做人工治理

例如公司、重大项目、核心产品等高连接节点，一次误合并会污染大量路径。可以：

- 人工确认权威主键；
- 审核高连接度节点；
- 监控异常合并；
- 为拆分操作提供工具；
- 对低置信新别名设置审批。

### 故障与降级

| 故障 | 处理 |
| --- | --- |
| 抽取模型不可用 | 暂停新版本发布，旧图继续服务 |
| 局部更新异常扩大 | 停止增量，转影子全量重建 |
| 社区摘要回归 | 回退旧摘要或路由普通 RAG |
| 图查询超时 | 使用向量检索和原文证据降级 |
| ACL 版本不一致 | fail closed，不使用混合摘要 |
| 新版本成本异常 | 停止灰度并保留旧路由 |

### 避免“图幻觉”

模型抽取的边不等于真实关系。每次答案应尽量回到原文证据：

~~~text
图用于发现候选关系与路径
→ 原文证据用于核验
→ 最终答案引用原文
~~~

不要只引用“图里有这条边”。

### 何时不适合 GraphRAG

- 问题主要是单文档局部事实；
- 数据量小，长上下文或普通 RAG 已足够；
- 关系 schema 无法定义，抽取质量不可评；
- 文档更新非常频繁，重算成本无法接受；
- 权限高度碎片化，社区摘要很难安全共享；
- 团队没有图版本、lineage 和回滚能力；
- 没有证据证明全局或关系问题带来业务价值。

### 技术边界

- 自动实体和关系抽取会产生误差；
- 社区检测结果受图结构和算法参数影响；
- 社区摘要是生成内容，也会遗漏或幻觉；
- 局部增量可能改变远处社区，影响范围不总是容易界定；
- GraphRAG 不替代原始文档和权限系统；
- 官方开源实现提供方法参考，生产级 SLA、权限和运维仍需团队设计与验证。

## 常见误区

- **“把三元组存进图数据库就是 GraphRAG”**：完整方法还可能包含消歧、社区、摘要和查询策略。
- **“实体名字一样就应该合并”**：同名人、公司和项目很常见。
- **“模型抽取的边就是事实”**：边必须有原文证据、版本和置信信息。
- **“新增文档只需要增加节点”**：边、社区、摘要和向量也可能变化。
- **“删除节点就完成删除”**：还要撤销证据、重算摘要并失效缓存。
- **“图越大，答案越好”**：噪声、误合并、权限和成本也会扩大。
- **“GraphRAG 应替换普通 RAG”**：二者更适合按查询类型组合。
- **“过滤无权节点后，社区摘要就是安全的”**：摘要可能已经混入越权事实。

## 面试追问

### 1. 实体消歧为什么不能只按字符串相似度？

同名可能是不同对象，别名也可能差异很大。要结合类型、权威主键、时间、属性、上下文和图邻居；低置信时暂不合并，并保留可拆分历史。

### 2. 如何验证实体合并质量？

建立同实体和不同实体金标集，测 Precision、Recall 和误合并；对高连接、高业务影响实体做人工抽检；监控合并后属性和时间冲突。

### 3. 删除一篇文档为什么会重算社区？

文档可能是某条边的唯一证据。边删除后图结构、社区划分和社区摘要都可能改变，所以要沿 lineage 计算影响并重算。

### 4. 怎么决定局部重算还是全量重建？

根据受影响节点、边、社区、摘要和预计成本设置阈值。schema、消歧规则、社区算法或模型大改时通常全量影子重建更容易验证。

### 5. GraphRAG 如何做权限隔离？

ACL 从来源传播到实体、边和摘要。优先在权限同质分区内构图或按租户隔离；混合权限社区不能预生成共享摘要，必要时请求时只基于授权证据汇总。

### 6. 如何降低索引成本？

限制高价值语料和业务 schema，按内容哈希复用抽取，批处理，分层使用模型，局部更新，并通过查询路由让简单问题走普通 RAG。

### 7. GraphRAG 适合什么问题？

跨文档实体关系、多跳路径、全局主题和数据集级总结更可能受益。单条制度金额、局部 FAQ 通常先用普通 RAG。

### 8. 如何证明 GraphRAG 值得上线？

与强普通 RAG 在同一真实查询集上比较关系、多跳和全局问题的任务成功与引用质量，同时报告构建、更新、存储、P95 延迟和单位成功成本。
