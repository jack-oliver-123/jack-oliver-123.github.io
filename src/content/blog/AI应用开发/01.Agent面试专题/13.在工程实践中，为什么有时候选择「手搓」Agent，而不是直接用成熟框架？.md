## 本篇目标

读完这篇，你应该能够：

- 分清模型 SDK、Agent 框架、业务编排代码分别解决什么问题。
- 理解“手搓 Agent”通常是手写薄编排层，而不是从零训练模型或重写所有基础设施。
- 知道框架的真实价值和可能代价，不做“框架都好”或“手写都好”的极端判断。
- 用需求、复杂度、可靠性、团队能力和总拥有成本做选型。
- 说出无论选哪条路都不能省略的生产级能力。

## 60 秒回答

选择手写还是框架，本质是控制权、交付速度和维护成本的取舍。成熟 SDK 或框架能快速提供模型调用、工具封装、状态图、持久化、Handoff 和追踪，适合原型、标准集成较多或复杂流程；但如果核心循环很小，业务需要精确控制状态、权限、重试和副作用，或者对依赖、部署、合规和升级稳定性有严格要求，手写一个薄编排层可能更透明、更容易审计。

“手搓”不等于所有东西从零造。我通常仍复用官方模型 SDK、数据库、队列和可观测组件，只把任务状态机、安全策略和业务事务掌握在自己的代码中。反过来，用框架也不应让业务代码到处依赖框架类型，可以通过窄适配层隔离。

我的选型原则是先画清运行时契约，再选能满足需求的最薄方案，并用原型验证。无论哪种方案，都要有超时取消、预算、状态持久化、幂等、权限、审计、评测和升级回归。不能因为框架代码少，就相信未知默认值；也不能因为手写代码看得见，就认为它天然可靠。

## 详细解析

### 先说人话：买整套厨房，还是按自己的菜单搭工作台

成熟 Agent 框架像一套已经装好的厨房：灶台、烤箱、收纳和水路都有标准位置，你可以很快开始做饭。若你的菜单与它的设计相符，会非常省时间。

手写薄编排层像根据少量固定菜品搭一个专用工作台：结构可能更简单，动作更透明；但水电、安全、清洁和维护都要自己负责。

问题不在“哪一种更高级”，而在：

- 你的业务到底需要多少能力；
- 框架抽象是否与业务匹配；
- 团队能否理解和维护它；
- 发生故障时，谁能快速定位和恢复。

### 先分清 SDK、框架和业务代码

初学者容易把它们统称为“框架”。实际上可以分为三层：

| 层次 | 主要作用 | 例子 |
| --- | --- | --- |
| 模型 SDK | 调用模型 API、流式输出、工具调用数据结构、错误处理 | 官方模型客户端 |
| Agent 框架/SDK | Agent 循环、Handoff、状态图、持久化、追踪、守卫 | OpenAI Agents SDK、LangGraph 等 |
| 业务编排 | 订单状态、审批规则、权限、事务、补偿、业务验收 | 你的项目代码 |

选择“手搓 Agent”通常不是不用任何库，而是：

> 复用稳定的底层组件，但自己实现很薄的 Agent 控制循环和业务状态机。

例如你没有必要自己实现 HTTP、数据库连接池和 OpenTelemetry；但“退款超时后到底能不能重试”应由业务代码明确决定，不能交给看不见的默认行为。

### 一个最小 Agent 到底有多少代码逻辑

先不考虑具体语言，一个最小受控循环是：

```text
加载任务状态
  ↓
在预算内调用模型
  ↓
模型选择：回答、请求工具、请求澄清或结束
  ↓
若请求工具：校验 schema、权限和风险
  ↓
执行工具并记录结果
  ↓
持久化状态
  ↓
继续下一轮或程序化停止
```

如果你的系统只有两个只读工具、三个明确状态和一个简单停止条件，专用代码可能很少。此时引入一个支持几十种抽象的框架，团队反而要额外理解框架概念、生命周期和默认重试。

但如果你需要复杂状态图、暂停恢复、多 Agent Handoff、人工审批、持久化和完整追踪，全部自己实现会迅速变成基础设施项目，成熟框架可能更合适。

### 贯穿案例：订单查询与退款 Agent

假设第一版需求只有：

1. 查询订单；
2. 读取退款政策；
3. 符合条件后提交人工审批；
4. 审批通过再退款。

#### 手写薄编排层的设计

```text
固定状态机：收集订单号 → 查询 → 判断 → 等审批 → 执行 → 验证
模型负责：理解用户表述、选择只读查询、生成解释
程序负责：身份、规则、审批、退款提交、幂等和恢复
```

这个流程的业务边界很清楚，核心写操作本来就不应由模型自由规划。手写状态机可能更直接。

#### 框架方案的设计

使用框架提供的图节点、检查点、人工中断、工具封装和 trace，再把每个业务步骤接入。若团队随后还要增加物流、换货、发票、多个专属 Agent 和大量连接器，框架的统一抽象会逐渐产生价值。

所以同一个项目在不同阶段可能做出不同选择。原型阶段用框架快速验证，稳定后保留框架；也可能发现只使用了很少能力，再逐步收缩为薄编排层。反过来，早期手写方案随着状态图变复杂，也可能迁移到成熟框架。

### 成熟框架能带来什么

#### 1. 更快验证想法

工具注册、模型循环、Handoff 和基础追踪已经封装，团队可以先验证“这个任务是否适合 Agent”，而不是先写基础设施。

#### 2. 统一执行模型

团队成员使用相同的节点、状态和事件概念，多个项目更容易形成规范。

#### 3. 现成的集成能力

框架可能提供模型、向量库、检查点、观测和文档加载连接器，减少重复适配。

#### 4. 持久化与可观测能力

复杂任务的暂停、恢复、trace 和调试 UI 都需要不少工程工作。若框架已有成熟实现，复用通常比重写划算。

#### 5. 社区经验与标准模式

常见问题可能已有文档、示例和修复，团队不必独自踩完所有坑。

### 框架可能带来什么代价

这里说“可能”，因为不同框架、版本和用法差异很大，不能一概而论。

#### 1. 抽象不匹配

框架希望一切都表示成图节点，但你的业务已有成熟状态机；或者框架默认 Agent 自主重试，而业务写操作要求严格人工确认。这时团队要绕着抽象写代码。

#### 2. 隐藏默认行为

默认重试、消息裁剪、工具异常转换、并发方式或状态合并规则如果没有被理解，会在生产中产生意外行为。

#### 3. 升级与迁移成本

快速演进的库可能改变 API、序列化格式或执行语义。业务代码如果到处直接依赖框架类型，升级会影响很大。

#### 4. 依赖与供应链

更多传递依赖意味着更大的部署体积、许可证检查和漏洞修复范围。是否构成问题要看团队环境，不能只看依赖数量。

#### 5. 调试需要理解两层逻辑

开发者既要理解业务流程，也要理解框架怎样调度、保存状态和处理错误。没有完整 trace 时，问题可能被埋在抽象层内。

### 什么时候更适合手写薄编排层

- 控制循环很小，状态和分支数量有限；
- 业务已经有可靠的 Workflow 或状态机，只需在局部调用模型；
- 需要精确控制消息顺序、Token、超时和错误语义；
- 写操作、事务和补偿必须完全由业务掌握；
- 对依赖、部署体积、离线环境或供应链有明确限制；
- 团队有能力长期维护测试、持久化和可观测；
- 框架抽象经原型验证后明显比业务本身更复杂。

### 什么时候更适合成熟框架

- 需要快速做 POC，先验证任务可行性；
- 有复杂状态图、循环、暂停恢复和人工审批；
- 需要多个 Agent、Handoff 或大量连接器；
- 团队需要统一开发模式和调试工具；
- 框架提供的检查点、追踪或评测能显著减少自研；
- 团队理解它的执行模型，并能承担版本升级。

### 最常见也最实用的方案：混合使用

很多生产系统并不是二选一，而是：

```text
业务自己掌握：身份、权限、状态真相、审批、事务、幂等、补偿
框架或组件提供：模型客户端、工具 schema、图调度、追踪、文档解析
```

关键是建立窄接口，避免业务逻辑被某个框架的数据类型渗透：

```text
ModelClient.generate(request) -> ModelDecision
ToolExecutor.execute(auth_context, ToolRequest) -> ToolResult
StateStore.load/save(task_id) -> TaskState
Tracer.record(event) -> void
```

切换实现时，业务层仍围绕自己的 `TaskState` 和 `ToolRequest` 工作。适配层不必抹平所有厂商差异，只要隔离真正需要稳定的核心契约。

### 一套更可靠的选型流程

#### 第一步：画运行时契约

先不选框架，写清：

- 状态有哪些；
- 模型能提出哪些动作；
- 工具有哪些输入输出；
- 谁做权限判断；
- 哪些步骤会产生副作用；
- 失败、重试、暂停和取消怎样定义；
- 任务何时成功、失败或转人工。

#### 第二步：做两个小型 Spike

`Spike` 是为了验证技术风险而写的短期原型。可以分别用薄编排和候选框架实现一条关键链路，不追求完整产品。

#### 第三步：用实际指标比较

- 开发关键功能需要多久；
- 失败轨迹能否看懂；
- 状态能否可靠恢复；
- 是否能精确控制写操作；
- P95 延迟和额外开销；
- 升级影响面；
- 团队上手与维护成本。

#### 第四步：选择最薄的满足方案

“最薄”不是代码行数最少，而是满足需求的同时，团队需要理解和维护的概念尽可能少。

## 工程实践与边界

### 无论选哪条路都不能省的能力

- 输入输出 schema 与校验；
- 身份、授权和最小权限；
- 超时、取消和预算传播；
- 任务状态持久化与检查点；
- 写操作幂等与外部状态核对；
- 可审计的工具调用和 trace；
- 敏感日志脱敏；
- 程序化停止与人工接管；
- 离线评测、回归用例和故障演练。

框架提供其中一些能力时，也要验证配置和边界；手写时则要明确补齐。

### 使用框架时的边界

1. **锁定依赖版本**：不要让生产环境自动漂移到未经验证的新版本。
2. **阅读变更日志**：重点看状态序列化、重试、工具调用和回调行为。
3. **审查默认重试**：只读操作可按策略重试；写操作必须幂等或显式关闭。
4. **隔离框架类型**：业务实体不要到处继承框架类。
5. **做升级回放**：用黄金任务、安全用例和历史失败轨迹比较升级前后。
6. **治理供应链**：记录许可证、软件物料清单和漏洞修复策略。

### 手写时的边界

1. 不要只实现“while 循环 + 模型调用”，忽略持久化和停止条件。
2. 不要吞掉工具异常，要保留明确错误类型和可恢复语义。
3. 取消、超时和预算必须贯穿模型与工具调用。
4. 状态迁移要有测试，尤其是崩溃恢复和重复消息。
5. 追踪事件要带任务、轮次、工具调用和关联 ID。
6. 不要把“我看得懂代码”误认为“代码已经可靠”。

### 五个典型失败场景

#### 场景一：框架自动重试导致重复退款

退款请求已经成功，但响应超时；框架把异常当普通网络错误自动重试。

处理：审查重试策略；写操作携带幂等键；不确定时查询真实状态，不能盲目重放。

#### 场景二：框架升级后无法恢复旧任务

新版本改变状态序列化格式，运行一半的任务无法读取。

处理：状态 schema 自己版本化；升级前用生产副本做迁移演练；必要时双读或保留旧 Worker 完成存量任务。

#### 场景三：手写循环停不下来

模型连续调用搜索工具，每轮都认为还需要更多资料。

处理：程序限制最大步数、时间、Token、费用和工具次数；定义完成与转人工条件。

#### 场景四：为了“去锁定”写出一个巨型通用层

团队试图统一所有模型的每个参数，适配层比任何框架都复杂。

处理：只抽象真正稳定的核心契约；厂商特有能力允许通过受控扩展暴露，不追求最低公分母。

#### 场景五：POC 直接上线

原型使用进程内存、管理员密钥和完整日志，没有租户隔离与恢复机制。

处理：上线前做生产差距清单；补齐权限、存储、审计、预算、评测和故障演练，不能把演示代码等同于生产系统。

### 如何量化总拥有成本

不要只比较“框架 20 行、手写 100 行”。应考虑：

- 初次开发时间；
- 每次新增能力的修改范围；
- 平均故障定位与修复时间；
- 升级、迁移和回滚成本；
- 额外延迟与基础设施费用；
- 安全审计和合规成本；
- 团队培训、人员变动后的维护风险；
- 自研组件长期修复和测试成本。

代码少不一定成本低，代码多也不一定可控。关键是系统行为能否被团队理解、验证和持续维护。

## 常见误区

- **“手搓就是不用任何 SDK”**：通常只是自己掌握控制循环和业务编排，底层稳定组件仍应复用。
- **“手写天然性能更好”**：大多数延迟在模型和外部 I/O，必须测量后判断。
- **“框架总是难调试”**：成熟框架可能有优秀 trace；关键是团队是否理解其执行模型。
- **“框架成熟就可以信任默认配置”**：默认重试、状态和日志策略未必符合你的风险边界。
- **“用了抽象层就完全没有供应商锁定”**：模型能力和语义有差异，只能降低替换成本，不能消除差异。
- **“核心逻辑手写就没有升级风险”**：模型、官方 SDK、协议和外部服务仍会变化。
- **“POC 能跑就是生产可用”**：生产还需要权限、恢复、审计、评测和故障处理。

## 面试追问

> **面试官：** 为什么有时要手搓 Agent？
>
> **候选人：** 当控制循环很小，且业务需要精确掌握状态、权限、重试和副作用时，薄编排层可能比通用抽象更透明。但要把自研测试和维护成本一起计算。

> **面试官：** 手搓是不是所有代码都自己写？
>
> **候选人：** 不是。我会复用官方模型 SDK、数据库、队列和观测组件，主要自己掌握任务状态机、安全策略和业务事务。

> **面试官：** 如何量化是否该移除框架？
>
> **候选人：** 比较关键功能开发时间、故障定位时间、额外延迟、升级迁移成本、依赖风险和自维护成本，而不是只看代码行数。

> **面试官：** 手写最容易漏掉什么？
>
> **候选人：** 超时与取消传播、状态持久化、工具调用关联、写操作幂等、程序化停止、敏感日志处理和崩溃恢复测试。

> **面试官：** 如何避免框架锁定？
>
> **候选人：** 用自己的窄接口隔离模型、工具、状态和追踪，业务实体不依赖大量框架类型；同时保留契约测试和替换演练，但不强行抹平所有厂商差异。

> **面试官：** 什么情况下你会坚定选择框架？
>
> **候选人：** 当项目需要复杂状态图、暂停恢复、多 Agent Handoff 和完整追踪，而框架已提供经验证的实现，复用收益明显高于团队自研维护成本时。
