把模型的候选 Token 想成一家食堂的菜：

- Temperature 决定你有多愿意尝试冷门菜；
- Top-k 规定只看销量最高的前 $k$ 道；
- Top-p 规定从最热门开始取，直到这些菜合计覆盖足够多的选择概率。

它们都不创造新知识。

它们只是改变模型已经给出的候选分布，再从中抽样。

网上常见“温度 0.7、Top-p 0.9 最佳”的说法缺少前提。

同一组参数对写诗可能保守，对财务抽取又可能过于随机。

## 本篇目标

读完本篇，你应该能够：

- 区分 logits、概率、采样和确定性选择；
- 逐符号解释 Temperature 公式；
- 手算 Temperature 变大或变小时的影响；
- 解释 Top-k 的固定候选集合；
- 解释 Top-p 的动态累计概率集合；
- 说明三者组合时的处理顺序和交互；
- 解释为什么低温不等于事实正确；
- 针对抽取、问答、创作、代码和推理设计参数实验；
- 回答“Temperature=0 到底是什么意思”等追问。

核心结论：

> Temperature 调整概率分布的尖锐程度，Top-k 和 Top-p 截断候选集合；不存在跨模型、跨任务通用的最佳值。

## 60 秒回答

Temperature、Top-k 和 Top-p 都用于控制随机采样。

Temperature 用 $T$ 缩放 logits，再做 softmax。$T<1$ 会让高概率 Token 更占优势，输出更稳定；$T>1$ 会拉平分布，增加多样性和低概率 Token 被选中的机会。它通常不改变 logits 的排序，只改变概率差距。

Top-k 每步只保留概率最高的 $k$ 个 Token，再对它们重新归一化采样。Top-p，也叫 nucleus sampling，会从高到低累加概率，保留累计概率达到阈值 $p$ 的最小候选集合，因此集合大小会随上下文变化。分布很确定时集合很小，分布平坦时集合会变大。

工程中没有固定最佳值。我会先按任务选低随机性或受控采样，再固定模型、提示和评测集，做小范围参数实验。事实抽取和结构化输出优先低随机性并配合约束与校验；普通对话使用中等受控采样；创作可适当放宽；代码和推理可生成多个候选后用测试或验证器筛选。低温只能降低随机性，不能保证正确和不幻觉。

## 详细解析

### 1. 参数作用在生成阶段

模型每一步先输出全词表 logits：

~~~text
Token A → 2.0
Token B → 1.0
Token C → 0.0
~~~

这些是相对分数，不是概率。

生成器通常经历：

~~~text
原始 logits
  ↓
应用 Temperature
  ↓
应用 Top-k / Top-p 等过滤
  ↓
重新归一化为概率
  ↓
采样下一个 Token
~~~

不同框架对处理器顺序可能有差异。

阅读当前版本文档比凭经验猜测更可靠。

### 2. 先复习 softmax

softmax 公式：

$$
p_i
=
\frac{\exp(z_i)}
{\sum_{j=1}^{V}\exp(z_j)}
$$

其中：

- $V$ 是词表大小；
- $z_i$ 是第 $i$ 个 Token 的 logit；
- $p_i$ 是归一化概率；
- 所有 $p_i$ 之和等于 1。

对 logits `[2, 1, 0]` 做 softmax，概率约为：

~~~text
A：0.665
B：0.245
C：0.090
~~~

模型明显更偏好 A，但 B、C 仍有机会。

### 3. Temperature 的公式

加入温度 $T$ 后：

$$
p_i(T)
=
\frac{\exp(z_i/T)}
{\sum_{j=1}^{V}\exp(z_j/T)}
$$

逐个解释：

- $z_i$ 是原始 logit；
- $T$ 是大于 0 的温度；
- 先用 $z_i/T$ 缩放分数；
- 再做 softmax；
- $p_i(T)$ 是缩放后的概率。

关键不是“给概率直接乘温度”。

Temperature 通常作用在 logits 上。

### 4. 低温为什么更确定

仍用 logits `[2, 1, 0]`。

若 $T=0.5$，缩放后为：

~~~text
[4, 2, 0]
~~~

softmax 约为：

~~~text
A：0.867
B：0.117
C：0.016
~~~

最高项 A 从 0.665 提升到 0.867。

概率差距被放大。

所以低温让抽样更集中在高分 Token。

### 5. 高温为什么更多样

若 $T=2$，缩放后为：

~~~text
[1, 0.5, 0]
~~~

softmax 约为：

~~~text
A：0.506
B：0.307
C：0.186
~~~

分布更平坦。

B、C 被选中的机会变大。

高温没有提高 Token 的真实性。

它只是让模型更愿意探索原本低概率的选择。

### 6. Temperature 的极限直觉

当 $T\to0^+$：

- 最大 logit 的概率趋近 1；
- 行为接近 argmax；
- 若最大项并列，仍需实现决定如何处理。

当 $T\to\infty$：

- logits 差异被压小；
- 分布趋向均匀；
- 大量低质量 Token 获得机会。

工程 API 中 `temperature=0` 往往被特殊解释为贪心或近似确定性。

数学公式本身不能直接除以 0。

因此必须查看供应商的字段语义。

### 7. Temperature 通常不改变排名

只要 $T>0$：

~~~text
如果 z_A > z_B
那么 z_A / T > z_B / T
~~~

所以 Token 排名不变。

变化的是概率间距。

Top-k 则会直接删除排名靠后的候选。

### 8. 什么是 Top-k

Top-k 每一步保留 logit 或概率最高的 $k$ 个 Token。

其他 Token 被设为不可采样，剩余概率重新归一化。

假设：

| Token | 原概率 |
| --- | ---: |
| A | 0.40 |
| B | 0.30 |
| C | 0.15 |
| D | 0.10 |
| E | 0.05 |

设置 `top_k=3` 后只保留 A、B、C。

它们原概率和为：

$$
0.40+0.30+0.15=0.85
$$

重新归一化：

| Token | 新概率 |
| --- | ---: |
| A | $0.40/0.85\approx0.471$ |
| B | $0.30/0.85\approx0.353$ |
| C | $0.15/0.85\approx0.176$ |

D 和 E 的概率变成 0。

### 9. Top-k 的优点与缺点

优点：

- 容易理解；
- 直接排除长尾候选；
- 每步候选数量上限固定；
- 有助于避免极低概率 Token。

缺点：

- 不适应分布形状；
- 分布很尖锐时，$k$ 可能太大；
- 分布很平坦时，$k$ 可能太小；
- 不同模型词表和校准不同，不能机械复用 $k$。

### 10. Top-k 的两个极端

如果 `top_k=1`：

- 只保留最高概率 Token；
- 行为接近贪心；
- Temperature 基本失去采样意义。

如果 `top_k` 大于等于词表大小：

- 不删除候选；
- 相当于未启用 Top-k。

一些 API 用 `0` 表示关闭 Top-k。

这不是统一标准。

### 11. 什么是 Top-p

Top-p 也叫 nucleus sampling。

步骤是：

1. 按概率从高到低排序；
2. 从最高概率开始累加；
3. 找到累计概率达到阈值 $p$ 的最小集合；
4. 删除集合外 Token；
5. 对集合内概率重新归一化并采样。

### 12. Top-p 手算例子

仍用：

| Token | 概率 | 累计概率 |
| --- | ---: | ---: |
| A | 0.40 | 0.40 |
| B | 0.30 | 0.70 |
| C | 0.15 | 0.85 |
| D | 0.10 | 0.95 |
| E | 0.05 | 1.00 |

若 `top_p=0.80`：

- A+B 累计 0.70，还不够；
- 加 C 后累计 0.85，达到阈值；
- 保留 A、B、C；
- 删除 D、E。

重新归一化结果与上一个 Top-k=3 例子相同。

但下一步不一定仍保留三个 Token。

### 13. Top-p 为什么是动态集合

场景一，模型非常确定：

~~~text
A：0.92
B：0.03
C：0.02
……
~~~

`top_p=0.9` 时，仅 A 就已达到阈值。

场景二，模型很不确定：

~~~text
A：0.15
B：0.13
C：0.12
D：0.10
……
~~~

同样 `top_p=0.9` 可能需要保留很多 Token。

这正是 Top-p 相比固定 Top-k 的主要直觉优势。

### 14. 边界 Token 是否保留

Top-p 通常需要保留那个让累计概率首次达到或超过阈值的 Token。

否则累计质量可能低于设定值。

不同实现还可能规定至少保留一个或若干 Token。

因此不要自己用几行排序代码就假设与推理引擎完全一致。

### 15. 三者的本质差别

| 参数 | 改变什么 | 候选数量 |
| --- | --- | --- |
| Temperature | 概率分布尖锐程度 | 通常不直接改变 |
| Top-k | 按排名截断 | 固定上限 $k$ |
| Top-p | 按累计概率截断 | 随上下文动态变化 |

它们都发生在单步生成。

下一步模型分布改变，过滤结果也会改变。

### 16. 同时使用时会怎样

概念上可能是：

~~~text
logits
  ↓ Temperature
缩放后的分布
  ↓ Top-k
保留前 k 个
  ↓ Top-p
从剩余候选取 nucleus
  ↓ sampling
~~~

但真实框架的 logits processors / warpers 顺序应以代码和文档为准。

同时设置较严格的 Top-k 和 Top-p，最终集合是多重过滤后的更小集合。

这可能过度收缩多样性。

### 17. 一个组合例子

假设温度缩放后概率为：

~~~text
A 0.45
B 0.25
C 0.15
D 0.08
E 0.07
~~~

先用 `top_k=4`：

- 删除 E；
- A、B、C、D 重新归一化。

再用 `top_p=0.8`：

- 可能只留下 A、B、C；
- D 再次被删除。

因此“Top-k=4”不保证最终总有四个候选。

### 18. 为什么参数不能独立理解

高温 + 很小 Top-k：

- 只在少数候选间拉平；
- 不会探索词表长尾。

低温 + 高 Top-p：

- 高概率 Token 可能已经占据大部分质量；
- nucleus 仍可能很小。

Top-k=1 + 任意正温度：

- 最终只有一个候选；
- 采样退化为选它。

参数效果由组合决定。

### 19. 熵帮助理解随机性

分布熵为：

$$
H(p)
=
-\sum_i p_i\log p_i
$$

- 分布集中时，熵较低；
- 分布平坦时，熵较高；
- 降低温度通常降低熵；
- 提高温度通常提高熵。

熵描述分布不确定性。

它不直接等于文本质量或创造力。

### 20. 参数与幻觉的关系

宽松采样会增加选择低概率错误 Token 的机会。

但幻觉不只来自采样。

即使 `temperature=0`：

- 模型也可能缺少知识；
- prompt 可能误导；
- 上下文可能冲突；
- 最高概率答案也可能是错的。

所以：

$$
\text{低随机性}\neq\text{事实保证}
$$

### 21. 与 repetition penalty 的区别

Temperature、Top-k、Top-p 根据当前分布整体控制选择。

repetition penalty 会针对已经出现过的 Token 修改 logits。

它们可以组合，但目标不同。

惩罚过高可能导致：

- 专有名词被改写；
- 代码变量不一致；
- 列表字段缺失；
- 语言变得不自然。

### 22. 与结构化输出的区别

Top-k 和 Top-p 只能提高或降低某些 Token 被选中的机会。

它们不能保证：

- JSON 括号配对；
- 枚举值合法；
- 必填字段存在；
- 数字满足业务范围。

严格结构应使用约束解码或供应商的 Structured Outputs，再做业务校验。

## 工程实践与边界

### 1. 不提供“万能最佳值”

参数效果依赖：

- 模型与版本；
- 预训练和对齐方式；
- Tokenizer；
- 任务；
- prompt；
- 输出长度；
- 推理引擎；
- 是否量化；
- 评测指标。

所以面试中更好的回答是说明实验方法，而不是背一个数字。

### 2. 场景起点表

下面仅用于建立实验起点，不是承诺值：

| 场景 | 随机性倾向 | 还必须做什么 |
| --- | --- | --- |
| 分类、字段抽取 | 很低 | 枚举或 Schema 约束 |
| 事实问答 | 低到中 | 检索、引用、事实评测 |
| 普通对话 | 中等 | 质量与安全评测 |
| 营销文案 | 中到较高 | 品牌规则与人工审核 |
| 诗歌创作 | 较高 | 多样性与主观评审 |
| 代码补全 | 低温或多样采样 | 编译和测试 |
| 数学推理 | 多候选采样 | 验证器或投票 |

### 3. 三组概念配置

稳定抽取起点：

~~~yaml
do_sample: false
max_new_tokens: 128
schema_constraint: enabled
~~~

普通对话实验起点：

~~~yaml
do_sample: true
temperature: 0.7
top_p: 0.9
top_k: disabled
max_new_tokens: 512
~~~

多候选创作实验起点：

~~~yaml
do_sample: true
temperature: 1.0
top_p: 0.95
num_return_sequences: 4
~~~

这些值只用于开始实验。

不要原样当成生产结论。

### 4. 为什么通常先少调参数

若同时改变：

- Temperature；
- Top-k；
- Top-p；
- repetition penalty；
- prompt；

结果变好后无法判断原因。

建议：

1. 先固定 prompt 与模型；
2. 选择贪心或单一采样基线；
3. 一次改变一个主要因素；
4. 对随机策略重复生成；
5. 比较均值、方差和失败样本。

### 5. 一个简单参数网格

例如只比较：

~~~text
temperature ∈ {0.2, 0.7, 1.0}
top_p      ∈ {0.8, 0.95, 1.0}
~~~

每个组合对每条评测样本生成多次。

但网格数量会快速增长：

$$
3\times3=9
$$

如果再加 4 个 Top-k，变成：

$$
3\times3\times4=36
$$

应根据预算先做粗筛，再在候选区域细调。

### 6. 评测不能只看一条输出

随机采样需要报告分布：

- 平均正确率；
- 最差分位；
- 格式失败率；
- 幻觉率；
- 重复率；
- 多样性；
- 平均长度；
- 延迟与成本。

只展示一个“灵感很好”的样本是选择性汇报。

### 7. 多样性的基本测量

可以观察：

- 不同输出的唯一比例；
- distinct-1 / distinct-2；
- 语义相似度分布；
- 人工对“内容是否真正不同”的判断。

distinct-n 高也可能只是胡乱用词。

它要和质量指标一起看。

### 8. 事实任务的防线

事实任务不要只靠低温：

~~~text
检索或工具提供证据
  ↓
模型基于证据生成
  ↓
引用或字段校验
  ↓
无法确认时允许拒答
~~~

参数只是最后一层控制。

### 9. 参数要进入版本和日志

每次请求至少可追踪：

- 模型 ID 与版本；
- 解码参数；
- seed；
- prompt 模板版本；
- 输出 Token 数；
- stop 原因；
- 延迟；
- 评测或反馈结果。

否则线上质量变化很难定位。

### 10. 供应商 API 边界

不同 API 可能：

- 不提供 Top-k；
- 限制 Temperature 范围；
- 将 0 特殊处理；
- 建议不要同时激进调整 Temperature 与 Top-p；
- 在服务端使用未公开采样细节；
- 模型升级后改变默认值。

调用前查看对应模型的当前官方文档。

### 11. 自托管引擎边界

自托管时还要确认：

- logits processor 应用顺序；
- 浮点与量化造成的 logit 差异；
- batch 调度对随机数流的影响；
- 不同 worker 的 seed 管理；
- 流式取消是否释放资源；
- 参数是否被兼容 API 忽略。

“接口返回 200”不代表参数真正生效。

### 12. 推荐的选型表达

面试中可以按这个顺序：

1. 说明三个参数的数学作用；
2. 说明不存在通用最佳值；
3. 按任务给实验起点；
4. 说明需要独立验证与结构约束；
5. 给出评测和可观测性方案。

这比背诵几个数值更像真实工程经验。

## 常见误区

- **把 Temperature 当概率乘数**：它通常除 logits 后再做 softmax。
- **认为 Temperature 改变 Token 排名**：正温度缩放通常保持 logit 排名，只改变差距。
- **认为 Temperature=0 可以直接代入公式**：数学上会除零，API 通常特殊处理为贪心。
- **认为 Top-k=50 每一步都会保留 50 个 Token**：若再叠加 Top-p，最终集合可能更小。
- **认为 Top-p 保留固定数量候选**：它保留累计概率达到阈值的动态集合。
- **认为 Top-p=0.9 会删除概率低于 0.1 的 Token**：它看累计概率，不是单 Token 阈值。
- **认为低温等于没有幻觉**：最高概率答案也可能是错的。
- **认为高温等于创造力**：它也可能增加乱码、跑题和事实错误。
- **同时把 Top-k、Top-p 调得很小**：可能过度收缩分布，输出僵化。
- **把一套参数复制到所有模型**：概率校准、词表和对齐方式不同。
- **只测一次随机输出**：无法估计质量方差。
- **忽略服务端默认值和版本**：默认参数变化会造成隐性回归。

## 面试追问

### 问：Temperature 为什么小于 1 会让分布更尖锐

**答：** logits 除以小于 1 的正数后，数值差距被放大；softmax 的指数运算进一步放大差异，因此高 logit Token 占据更多概率质量。

### 问：Temperature 会改变概率最高的 Token 吗

**答：** 对正温度的单纯缩放，一般不会改变 logits 排名，所以最高项不变。但后续过滤、惩罚、量化误差或并列处理可能影响最终选择。

### 问：Top-k 和 Top-p 最大区别是什么

**答：** Top-k 固定保留排名前 $k$ 个候选；Top-p 保留累计概率达到阈值的最小集合，集合大小随每一步分布变化。

### 问：Top-p=0.9 的集合怎样确定

**答：** 先按概率降序，从最高项开始累加，直到首次达到或超过 0.9；保留这批 Token，屏蔽其余候选，再重新归一化采样。
