你在聊天窗口里问大模型：

> “帮我查一下明天从北京到上海最早的高铁，并预订一张二等座。”

普通大模型只能根据已有知识生成文字。

它不知道明天真实的车次，也不能凭空访问铁路系统，更不能替你完成支付。

要让它真正完成任务，应用必须给它接上查询车次、读取乘客信息、创建订单等外部能力。

这就是“工具调用”要解决的问题。

第一次学习时，很多人会被一串相似的词绕晕：

- Function Calling 是不是模型直接执行函数？
- Tool Calling 与 Function Calling 是一回事吗？
- MCP 是不是 Function Calling 的升级版？
- MCP Server 为什么不等于传统后端服务？
- Tool、Resource、Prompt、Skill 分别是什么？
- WebSocket、SSE、WebRTC 又为什么会出现在面试题里？
- 模型返回了合法 JSON，为什么仍然不能直接执行？

本专题会从零开始拆开这些问题。

每篇先给一段可以直接用于面试的“60 秒回答”，再解释直觉、完整流程、工程边界、失败分支和常见追问。

读完以后，你不只要会背定义，还应该能够画出一条真实调用链，并说明每一层由谁负责。

## 先记住最重要的一句话

> **大模型通常只负责表达“想调用哪个工具、传什么参数”；宿主应用负责校验、授权、执行和回传结果。**

这里的“宿主应用”，就是承载模型、会话、用户权限和工具执行循环的程序。

它可能是聊天网站、桌面客户端、IDE、客服系统，也可能是一个后端 Agent 服务。

把这条边界记牢，后面大部分概念都会变清楚。

模型生成：

~~~text
工具：search_train
参数：{
  "from": "北京",
  "to": "上海",
  "date": "明天"
}
~~~

宿主应用才会继续做这些事：

1. 把“明天”转换为准确日期；
2. 检查出发地和目的地是否合法；
3. 确认当前用户是否有查询权限；
4. 调用真实车次服务；
5. 处理超时、限流和错误；
6. 把查询结果交回模型；
7. 让模型基于真实结果组织答案。

所以：

~~~text
模型提出调用意图
        ↓
应用验证并执行
        ↓
工具返回真实结果
        ↓
模型继续回答或提出下一次调用
~~~

这才是完整的工具调用闭环。

## 贯穿全系列的案例

本专题主要使用“企业差旅助手”作为统一案例。

用户可能提出：

> “下周三去上海见客户，帮我找上午到达的航班，再申请一张不超过 1500 元的机票。”

系统可能接入这些工具：

| 工具 | 作用 | 风险等级 |
| --- | --- | --- |
| search_flights | 查询航班 | 只读、低风险 |
| get_travel_policy | 查询差旅制度 | 只读、可能涉及权限 |
| get_employee_profile | 读取员工信息 | 涉及个人数据 |
| create_travel_request | 创建出差申请 | 写操作 |
| book_flight | 预订机票 | 高风险、有费用 |
| cancel_booking | 取消订单 | 高风险、可能产生退改费 |

这条业务链会反复帮助我们理解：

- 模型怎样从用户语言中选择工具；
- Schema 怎样描述参数；
- 为什么查询与预订要采用不同安全策略；
- MCP 怎样让多个 AI 应用复用同一套差旅能力；
- 并行调用何时安全，何时会破坏依赖；
- 工具很多时怎样发现、路由和动态加载；
- 怎样评估模型选工具和填参数的准确率。

## 核心心智模型：把系统分成四层

不要把所有东西都叫“工具调用”。

一个生产系统通常至少可以拆成四层：

| 层次 | 要回答的问题 | 典型技术 |
| --- | --- | --- |
| 模型决策层 | 现在要不要调用？调用哪个？参数是什么？ | Function Calling、结构化输出、提示与训练 |
| 宿主编排层 | 怎样校验、授权、重试、记录和继续循环？ | Agent Runtime、Workflow、策略引擎 |
| 能力接入层 | 外部能力怎样发现、描述、连接和调用？ | MCP、内部工具注册表、API Adapter |
| 真实执行层 | 谁访问数据库、网络、文件或业务系统？ | 函数、微服务、SaaS API、沙箱 |

Function Calling 主要位于“模型决策层与宿主编排层的接口”。

MCP 主要位于“宿主应用与能力提供方的接口”。

二者不在同一层，因此既不是同义词，也不是简单替代关系。

最常见的组合是：

~~~text
用户
  ↓
Host（承载模型和策略）
  ↓  模型通过 Function Calling 表达调用意图
Host 校验调用
  ↓  MCP Client 发送 tools/call
MCP Server
  ↓
真实业务 API / 数据库 / 本地程序
~~~

模型甚至可以完全不知道底层使用了 MCP。

Host 只要把 MCP Tool 转换成模型接口能够理解的工具定义即可。

反过来，确定性 Workflow 也可以直接调用 MCP Server，而不经过模型的 Function Calling。

## 零基础术语表

| 术语 | 先这样理解 | 必须注意的边界 |
| --- | --- | --- |
| LLM | 能理解和生成语言的模型 | 本身不自动拥有外部系统权限 |
| Tool | 可被系统调用的一项外部能力 | 可以是函数、API 或协议服务 |
| Function Calling | 模型输出工具名和结构化参数的机制 | 通常不负责真实执行 |
| Tool Calling | 更宽泛的称呼 | 不同厂商字段与格式可能不同 |
| Schema | 描述输入结构和约束的规则 | 结构合法不等于业务合法 |
| JSON Schema | 描述 JSON 数据结构的标准 | 供应商不一定支持全部关键字 |
| Arguments | 一次工具调用的参数 | 必须视为不可信输入 |
| Tool Result | 工具执行后的返回值 | 也可能包含恶意或敏感内容 |
| Host | 承载模型、用户和安全策略的应用 | 是最终信任边界 |
| Runtime | 驱动模型与工具循环的运行程序 | 负责状态、错误和停止条件 |
| MCP | 模型上下文协议 | 解决应用与能力提供方的互操作 |
| MCP Client | Host 内连接一个 MCP Server 的协议组件 | 不等于整个 Host |
| MCP Server | 通过 MCP 暴露能力的程序 | 不一定是远程服务器 |
| Tool（MCP） | MCP Server 暴露的可执行能力 | 仍需 Host 授权和用户同意 |
| Resource | 通过 URI 暴露的上下文数据 | 与“只读 Tool”语义不同 |
| Prompt | Server 提供的可复用消息模板 | 不等于 Host 的全部系统提示 |
| Sampling | Server 请求 Client 侧模型生成 | 模型凭据留在 Host/Client |
| Elicitation | Server 请求用户补充信息 | 不能绕过用户同意 |
| Roots | Client 声明可见文件系统根目录 | 是范围提示，不是万能沙箱 |
| stdio | 用标准输入输出传输消息 | 常用于本地子进程 |
| Streamable HTTP | MCP 的标准远程传输 | 仍需认证、授权和来源校验 |
| SSE | 服务器向客户端推送事件的 HTTP 机制 | 通常是单向下行 |
| WebSocket | 持久的全双工通信通道 | 需要额外治理连接状态 |
| WebRTC | 面向实时音视频和数据的通信技术 | 适合低延迟媒体，不是通用 API 替代品 |
| 幂等 | 重复请求不会重复产生副作用 | 写操作重试的关键 |
| 补偿 | 用反向动作处理已发生的副作用 | 不等于数据库事务回滚 |
| 提示注入 | 外部内容诱导模型违背原规则 | 工具结果同样可能携带攻击 |
| Tool Routing | 从大量工具中选出候选集合 | 不等于最终决定执行 |
| Eval | 用数据评估系统行为 | 要分开测选择、参数、执行与最终答案 |

## 你需要哪些前置知识

本专题按零基础编写，不要求学过机器学习。

你只需要先知道：

- API 可以理解为软件之间约定好的办事窗口；
- JSON 是一种键值结构的数据格式；
- 函数接收参数并返回结果；
- HTTP 是常见网络通信方式；
- 身份认证回答“你是谁”，授权回答“你能做什么”。

如果这些词仍然陌生，也可以直接开始。

文章会在第一次出现术语时解释，并用具体例子重复建立直觉。

不需要先背某家模型供应商的字段名。

面试更重要的是说清跨系统都成立的职责边界。

## 分阶段学习路线

### 第一阶段：看懂最小闭环

先读 01—03。

目标是回答：

- Function Calling 究竟输出什么；
- 模型为什么能生成工具名和参数；
- 训练数据怎样教会模型“调用”和“不调用”。

学完后，你应该能独立写出：

~~~text
工具定义 → 模型调用意图 → 参数校验 → 执行 → 工具结果 → 最终回答
~~~

### 第二阶段：分清 Function Calling 与 MCP

学习 04—08。

目标是回答：

- MCP 解决什么集成问题；
- Host、Client、Server 分别负责什么；
- Tools、Resources、Prompts 怎样区分；
- 为什么 MCP 与 Function Calling 可以组合；
- 模型是否支持工具调用，与应用是否支持 MCP 为什么是两件事。

### 第三阶段：理解 Skill 与 Agent 协作

学习 09—12。

目标是分清：

- Tool 提供“能做什么”；
- Skill 提供“怎样做好一类任务”的知识与流程；
- MCP 提供能力接入协议；
- A2A 处理 Agent 与 Agent 之间的协作。

### 第四阶段：理解通信与基础设施

学习 13—16。

目标是能根据场景比较：

- stdio 与 Streamable HTTP；
- SSE 与 WebSocket；
- WebRTC 与 WebSocket；
- 模型网关与业务工具网关。

不要只背“谁是单向、谁是双向”，还要说延迟、代理兼容、断线恢复、会话状态和运维成本。

### 第五阶段：掌握可靠执行

学习 17—23。

目标是能设计：

- 严格、清晰、可演化的工具 Schema；
- 超时、有限重试、幂等和补偿；
- 最小权限执行环境；
- MCP OAuth；
- 生命周期与能力协商；
- Sampling、Elicitation、Roots 的安全边界。

### 第六阶段：处理规模化问题

学习 24—25。

目标是回答两个生产系统绕不开的问题：

- 工具数量从 5 个增长到 500 个后，如何做发现、路由与动态加载；
- 如何用离线评测、线上指标和完整 Trace 证明系统可靠。

## 怎样读每篇文章

建议按四遍法学习。

第一遍，只读开头场景和“60 秒回答”。

先建立直觉，不要急着记协议字段。

第二遍，顺着“详细解析”画出流程。

每一步都问自己三个问题：

1. 输入是什么？
2. 谁负责处理？
3. 失败后会发生什么？

第三遍，读“工程实践与边界”。

把“能跑通 Demo”和“能在生产使用”分开。

第四遍，遮住“面试追问”的答案，用自己的语言回答。

如果回答中只有名词，没有责任主体、失败分支和取舍依据，就还没有真正掌握。

## 面试答案怎样背

不要逐字背整篇文章。

每道题只背一个五段式骨架：

1. **一句定义**：它是什么；
2. **职责边界**：谁做什么，谁不做什么；
3. **完整流程**：输入怎样变成结果；
4. **工程风险**：失败、安全和可观测性；
5. **选型条件**：何时适用，何时不适用。

例如，回答 Function Calling：

> Function Calling 是模型表达工具调用意图的结构化输出机制。应用先提供工具定义，模型返回工具名和参数；宿主应用再做校验、授权和真实执行，并把结果关联到调用交回模型。模型本身通常不直接访问外部系统。生产上还要处理超时、幂等、审批、审计和工具结果中的提示注入。

这段话不是唯一标准答案。

真正需要背的是它的结构：

~~~text
定义 → 模型职责 → 宿主职责 → 闭环 → 工程边界
~~~

## 没做过真实项目怎样回答

不要虚构生产经验。

可以明确区分“亲手验证过”和“生产环境需要补充验证”。

例如：

> “我做过最小工具调用闭环，验证了工具定义、参数校验、执行回传和错误分支。没有在大规模生产流量下验证过。若上线，我会补充离线调用集、影子流量、超时与重试策略、幂等键、权限矩阵、完整 Trace 和灰度回滚。”

这种回答比编造 QPS、成功率或故障案例更可信。

面试官通常不是要求每个人都运营过超大规模系统，而是判断你是否知道缺少哪些证据。

## 通用答题模板

遇到任何工具调用题，可以按下面顺序回答：

### 第一步：确定讨论层级

先判断问题在问：

- 模型怎样表达调用；
- Host 怎样编排；
- 外部能力怎样接入；
- 工具代码怎样执行；
- 还是多个 Agent 怎样协作。

层级错了，后面很容易把不同概念混在一起。

### 第二步：给出责任主体

不要只说“系统会执行”。

要明确：

- 模型做什么；
- Host 做什么；
- Client 做什么；
- Server 做什么；
- 真实业务系统做什么。

### 第三步：说明成功路径

从工具定义开始，一直说到最终回答。

### 第四步：补充失败路径

至少覆盖：

- 参数不合法；
- 用户无权限；
- 工具超时；
- 写操作结果不确定；
- 工具返回恶意内容；
- 模型重复调用；
- 上下文或工具列表过大。

### 第五步：给出评测方式

不要只说“效果更好”。

可以测：

- 是否应该调用；
- 工具选择是否正确；
- 参数是否完整、合法；
- 执行成功率和延迟；
- 最终答案是否忠于工具结果；
- 高风险操作是否被正确拦截。

## 推荐练习

只读文章很难真正掌握。

建议实现一个最小差旅助手：

1. 定义一个只读的航班查询工具；
2. 故意输入缺少日期的问题；
3. 让系统先追问，而不是猜日期；
4. 给工具制造一次超时；
5. 验证系统不会无限重试；
6. 加入一个创建申请的写工具；
7. 为写操作增加确认和幂等键；
8. 把同一工具封装成 MCP Server；
9. 分别用确定性 Workflow 和模型决策触发它；
10. 记录完整 Trace，比较两条链路。

完成这些步骤后，你会拥有一段真实、可解释的面试经历。
