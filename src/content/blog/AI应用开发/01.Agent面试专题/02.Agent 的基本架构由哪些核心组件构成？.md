## 本篇目标

看懂一个 Agent 系统里有哪些角色，以及这些角色为什么不能全部交给大模型。读完后，你应该能从零画出一张基本架构图。

## 60 秒回答

我不会死记“Agent 有四大组件”或“六大组件”，而会按职责拆分。一个生产级 Agent 通常包括六层：模型与指令负责理解目标和提出决策；编排运行时维护执行循环、状态转移和停止条件；工具与执行环境连接数据库、搜索和业务接口；状态与记忆保存当前进度和可复用历史；安全层负责身份、权限、参数校验和人工审批；可观测与评测层负责追踪、回放和发布门禁。

最关键的边界是：模型只提出下一步动作，运行时才有权校验并执行。规划不一定是单独的服务，记忆也不等于向量数据库。组件数量可以随业务变化，但目标、状态、执行、安全和停止这些职责不能缺失。

## 详细解析

### 先说人话：把 Agent 想成一家公司

如果只把大模型接上几个接口，就像让一位聪明的新员工拿着公司所有密码直接工作：他可能能力很强，但不知道审批制度，也可能因为理解错误做出危险操作。

完整的 Agent 更像一家公司：

| 公司里的角色 | Agent 中的组件 | 主要职责 |
| --- | --- | --- |
| 员工的大脑 | 模型 | 理解问题、判断下一步 |
| 员工手册 | 指令 | 说明目标、规则和输出要求 |
| 项目经理 | 编排运行时 | 安排步骤、维护进度、决定何时停止 |
| 员工的双手 | 工具 | 查询数据或执行真实动作 |
| 工作笔记 | 状态与记忆 | 保存任务进度、事实和历史 |
| 门禁与审批制度 | 安全层 | 控制身份、权限和高风险操作 |
| 监控与质检部门 | 可观测与评测 | 记录过程、发现问题、判断能否上线 |

这张表不是要求你背七个名词，而是帮助你理解：**模型只是 Agent 系统中的决策组件，不是整个系统。**

### 用一个退款 Agent 看完整流程

假设用户说：“我的耳机坏了，帮我退款。”

一个基本流程可能是：

1. **模型理解目标**：用户想退款，但当前缺少订单号。
2. **编排器保存状态**：任务目标是退款，当前步骤是等待订单号。
3. **Agent 向用户提问**：请提供订单号。
4. **工具查询订单**：运行时校验用户身份后，调用订单接口。
5. **工具返回结果**：订单存在，金额 299 元，但已经超过自动退款期限。
6. **模型提出动作**：根据售后政策，建议申请人工特批。
7. **安全层拦截高风险动作**：退款属于写操作，需要有权限的客服审批。
8. **审批通过后执行**：运行时调用退款接口，并使用幂等键防止重复退款。
9. **可观测层记录过程**：保存每一步的动作、结果、耗时和结束原因。
10. **程序判断结束**：确认退款状态成功后回复用户并停止。

如果没有编排器，系统不知道现在进行到哪一步；没有安全层，模型可能越权退款；没有状态，进程重启后可能重复执行；没有评测和追踪，出错后也难以复现。

### 组件一：模型与指令

#### 模型负责什么

模型接收用户目标、系统规则、当前状态、相关资料和工具说明，然后生成：

- 给用户看的自然语言；
- 一个结构化动作，例如“查询订单”；
- 一个完成信号，例如“信息已足够，可以结束”；
- 一个澄清请求，例如“需要用户补充订单号”。

模型擅长理解自然语言、处理模糊信息和进行判断，但它的输出具有概率性。同样的问题在不同上下文中可能得到不同结果，所以模型输出不能直接当作已经授权的业务命令。

#### 指令负责什么

指令相当于岗位说明书，它应告诉模型：

- 当前扮演什么角色；
- 最终目标是什么；
- 可以使用哪些工具；
- 哪些规则不能违反；
- 信息不足时应该怎么办；
- 输出需要什么格式；
- 什么情况下必须转人工。

指令不能代替真正的权限控制。即使系统提示写着“不要退款超过 500 元”，运行时仍然要用代码再次校验金额。

#### 是否只能使用一个模型

不一定。系统可以让能力强的模型负责复杂判断，让更快、更便宜的模型负责分类或信息抽取，这叫模型路由。但路由结果也要经过评测，不能简单地把“输入短”等同于“任务容易”。

### 组件二：编排运行时

编排运行时是最容易被低估的部分。它通常负责：

1. 创建唯一的任务 ID；
2. 读取目标和当前状态；
3. 组装本轮模型需要的上下文；
4. 调用模型并解析结构化结果；
5. 检查动作是否符合格式、权限和政策；
6. 调用工具并处理超时或错误；
7. 把结果写回状态；
8. 判断继续、重试、暂停、转人工还是结束；
9. 强制执行步数、时间和费用预算。

所谓 ReAct、Plan-and-Execute 或固定 Workflow，本质上都是编排层采用的不同控制策略。

可以把它记成：

> 模型负责“建议做什么”，编排运行时负责“现在能不能做、怎样做，以及做完以后怎么办”。

### 组件三：工具与执行环境

Tool 是 Agent 可以调用的外部能力，例如：

- 查询订单；
- 搜索知识库；
- 读取文件；
- 发送邮件；
- 修改工单；
- 执行退款。

一个生产工具不能只有名字，还需要清晰的契约：

| 契约内容 | 要解决的问题 |
| --- | --- |
| 工具名称和用途 | 模型什么时候应该调用它 |
| 输入 Schema | 需要哪些字段、字段是什么类型 |
| 输出 Schema | 成功后会返回什么 |
| 错误类型 | 参数错、权限错、超时分别如何表示 |
| 副作用 | 是否会发信、付款、删除或修改数据 |
| 权限范围 | 谁可以操作哪些资源 |
| 超时与重试 | 什么错误可以安全重试 |

执行环境负责真正的鉴权、限流、超时、并发控制、幂等和沙箱。模型上下文中只应该看到必要的工具描述，不应该看到生产密钥。

MCP 可以帮助不同应用用统一方式发现并连接工具、资源和提示，但它是连接协议，不会自动替业务系统做好权限和安全治理。

### 组件四：状态与记忆

初学者很容易把状态、上下文和记忆混成一件事：

- **上下文**：本次调用时临时提供给模型的信息；
- **运行状态**：当前任务必须精确保存的进度；
- **长期记忆**：跨任务可能再次使用的信息。

退款 Agent 的运行状态可能包括：

- 任务 ID；
- 用户 ID 和订单 ID；
- 当前步骤；
- 已完成的工具调用；
- 退款是否已提交；
- 审批状态；
- 剩余预算；
- 最终结束原因。

这些数据要求精确，适合关系数据库或键值存储，不能依赖向量相似度“猜”出来。文档片段和历史案例才更适合语义检索。

为什么不能只把所有内容塞进 Prompt？因为 Prompt 不是可靠数据库：它有长度和成本限制，进程重启后无法精确恢复，并发任务还可能相互覆盖。

### 组件五：安全与治理

安全层至少要回答四个问题：

1. **谁在操作**：真实用户和租户身份是什么；
2. **可以操作什么**：当前身份拥有哪些权限；
3. **这个参数合法吗**：订单是否属于用户，金额是否超限；
4. **是否需要人审**：付款、退款、删除和对外发送是否需要审批。

系统提示、模型自我检查和安全分类器都可以提供帮助，但真正的安全边界必须在模型之外。策略服务不可用、身份信息缺失或审批过期时，高风险操作应默认拒绝。

### 组件六：可观测与评测

这两个概念有关，但用途不同：

- **可观测性**回答“这一次运行发生了什么”；
- **评测**回答“这个版本整体表现是否足够好”。

一次 Trace 可以串联：

> 用户请求 -> 模型调用 -> 工具调用 -> 审批 -> 状态变化 -> 最终结果。

常见记录包括动作名称、脱敏后的参数摘要、工具延迟、错误类型、Token、费用、模型版本和结束原因。评测则用固定数据集和线上抽样检查任务成功率、轨迹质量、成本、延迟和安全性。

不需要记录或公开模型的私有思维链。为了排错，记录可观察的计划、动作、结果和简短理由已经足够。

### 一张基础架构图应该画什么

面试时可以先画出下面的关系：

> 用户目标 -> 编排运行时 -> 模型决策 -> 安全校验 -> 工具执行 -> 状态更新 -> 再次决策或结束

旁边再补三条横向能力：

- 状态与记忆为每一轮提供信息；
- 可观测性记录整个执行轨迹；
- 预算、权限和人工审批约束所有动作。

比起画很多框，更重要的是说明数据怎么流动、谁拥有控制权、失败后从哪里恢复。

## 工程实践与边界

- **做租户隔离**：消息、任务状态和长期记忆都带可信的租户 ID 与用户 ID，并在数据库层执行隔离，不能只依赖 Prompt 中的用户名。
- **凭据不进入模型**：由运行时按工具和请求临时注入短期凭据，模型只看到不敏感的能力说明。
- **工具结果也是不可信输入**：限制返回大小，校验类型，标记来源；网页或邮件中的文字不能升级成系统指令。
- **区分状态提交和副作用**：退款、发信等动作前后写检查点；恢复时先查询动作是否已经发生，不能简单重放。
- **处理个人信息**：为 PII 设置数据分类、最小化、保留期、删除接口和访问审计，不能因为“以后可能有用”就永久保存。
- **设计失败路径**：架构图中除了成功，还要有超时、重试、暂停、取消、降级和人工接管。

## 常见误区

- **“Agent 固定由四个组件组成”**：不同系统拆法不同，关键是职责完整且边界清楚。
- **“规划器必须是独立模型”**：简单任务可以由同一个模型在循环中完成局部规划。
- **“记忆就是向量数据库”**：精确状态、权限和订单事实不应该依赖相似度检索。
- **“安全规则写进 Prompt 就够了”**：Prompt 不是授权边界，高风险动作必须由运行时代码校验。
- **“接入 MCP 就自动安全”**：MCP 解决互操作，资源授权和业务政策仍由应用负责。
- **“只画正常路径就算架构完整”**：没有失败、恢复和停止条件的图只是演示流程。

## 面试追问

> **面试官：** 哪个组件最容易被低估？
>
> **候选人：** 编排运行时。模型和工具最显眼，但真正负责预算、权限、重试、状态迁移、暂停恢复和停止条件的是运行时，它决定 Agent 能否从演示走向生产。

> **面试官：** 为什么模型不能直接执行工具？
>
> **候选人：** 模型输出具有不确定性，也可能受外部注入影响。它可以提出动作，但宿主程序必须重新校验身份、权限、参数和副作用，再决定是否执行。

> **面试官：** 规划和记忆能否都放进 Prompt？
>
> **候选人：** 小任务可以临时这样做，但长任务会遇到成本、精确恢复和并发一致性问题。关键状态应外置并结构化，只把当前步骤需要的信息投影进上下文。

> **面试官：** 如何替换模型而不重写整个 Agent？
>
> **候选人：** 在模型适配层统一消息、结构化输出、工具调用和错误类型，业务编排只依赖自有接口；替换后再用契约测试和回归评测验证行为差异。

> **面试官：** 如何判断架构设计是否完整？
>
> **候选人：** 除了模型和工具，我会检查目标、状态、权限、失败处理、预算、停止条件、人工接管和可观测性是否都有明确责任方。
