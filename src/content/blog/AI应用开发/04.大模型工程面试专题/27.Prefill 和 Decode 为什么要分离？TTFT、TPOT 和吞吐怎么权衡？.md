想象一家餐厅。

新客人一次点了二十道菜，后厨需要集中备料，这像处理一段很长的 Prompt。

另一桌已经开始上菜，只希望每隔几十秒稳定上一道，这像模型逐 Token 生成。

如果同一组厨师正在连续上菜时，突然被一张超长订单占满，老客人会感觉上菜卡顿。

如果只照顾老客人的逐道上菜，新客人的第一道菜又要等很久。

大模型服务中的 Prefill 与 Decode 也存在类似干扰。

将两阶段分离，是解决这种干扰的一种架构手段，但它会新增 KV Cache 传输、资源匹配和调度复杂度。

## 本篇目标

学完本篇，你应该能够：

1. 把一次大模型请求拆成排队、Prefill、Decode 和后处理；
2. 解释 Prefill 与 Decode 的输入输出和资源特征；
3. 逐个符号解释 TTFT、TPOT、ITL、端到端延迟和吞吐；
4. 说明为什么长 Prefill 会干扰正在进行的 Decode；
5. 区分 Chunked Prefill 与 Prefill/Decode 分离；
6. 画出分离部署的数据流和 KV Cache 传输过程；
7. 理解 Goodput 和 SLO attainment 为什么比裸吞吐更接近生产目标；
8. 估算 KV 传输是否会抵消分离收益；
9. 设计 Prefill 与 Decode 独立扩缩容和路由；
10. 判断什么规模与负载值得采用分离架构。

先记住一句话：

> Prefill 负责一次性理解输入并生成首个 Token 所需状态，Decode 负责逐步生成后续 Token。分离的价值是隔离两种负载并独立扩缩容；代价是必须可靠、快速地把 KV Cache 和请求状态从 Prefill 侧交给 Decode 侧。

## 60 秒回答

一次 LLM 推理通常分成 Prefill 和 Decode。Prefill 并行处理全部输入 Token，计算每层 KV Cache，并产生首个输出分布；它对长 Prompt 计算量大，矩阵较大，在不少配置下更偏计算密集。Decode 每步只处理一个新 Token，并读取不断增长的 KV Cache，在很多服务配置下更容易受显存带宽和 KV 容量限制。

两阶段放在同一批次时，长 Prefill 可能阻塞或干扰正在 Decode 的请求，让首 Token 和 Token 间隔互相牵制。Prefill/Decode 分离让一组实例专门做 Prefill，生成 KV 后传给 Decode 实例；两边可以采用不同 GPU、并行策略、批处理和扩缩容目标。

核心指标不能只看吞吐。TTFT 是请求到首个 Token 的时间，受排队和 Prefill 主导；TPOT 是首 Token 之后平均每个 Token 的时间；ITL 看相邻 Token 间隔；端到端延迟还与输出长度有关。Goodput 则只统计同时满足 TTFT 和 TPOT 等 SLO 的请求或 Token。

分离并非免费。KV 传输可能很大，还会增加网络、背压、故障恢复和跨节点调度。工程上我会先用连续批处理与 Chunked Prefill 建立基线，再根据真实输入输出长度、到达率和 SLO 模拟与压测；只有阶段干扰明显、负载规模足够且 KV 传输可被网络隐藏时，才考虑分离。

## 详细解析

### 一、一次请求到底经历哪些阶段

可以把端到端路径画成：

```text
客户端发送请求
      |
      v
网关、鉴权、排队
      |
      v
Prefill：处理全部输入 Token
      |
      v
生成首个输出 Token
      |
      v
Decode：逐个生成后续 Token
      |
      v
流式传输、后处理、结束
```

总时间可以粗略写为：

$$
T_{e2e}=T_{gateway}+T_{queue}+T_{prefill}+T_{decode}+T_{post}
$$

其中：

- $T_{e2e}$：端到端完成时间；
- $T_{gateway}$：网关、鉴权和请求解析时间；
- $T_{queue}$：等待调度的时间；
- $T_{prefill}$：处理输入并产生首 Token 所需计算；
- $T_{decode}$：生成后续输出的时间；
- $T_{post}$：后处理和最后传输时间。

不同请求的主导项可能不同。

长文档只输出一个分类标签时，Prefill 占比高。

短 Prompt 生成长篇文章时，Decode 占比高。

### 二、什么是 Prefill

Prefill 也常叫 Prompt Processing。

输入是一整段 Token：

```text
x1, x2, x3, ..., xS
```

模型用因果掩码并行计算各位置隐藏状态，并为每层产生 K/V。

结果包括：

- 输入前缀的 KV Cache；
- 最后位置的 Logit；
- 用于采样首个输出 Token 的分布。

虽然注意力有因果关系，但已知的 Prompt Token 可以在矩阵运算中并行处理。

### 三、什么是 Decode

Decode 在每一步只处理一个新的 Token。

第 $t$ 步：

1. 输入上一步采样的 Token；
2. 计算它的新 K/V；
3. Query 读取历史 KV Cache；
4. 产生下一个 Token 分布；
5. 采样并追加；
6. 直到停止条件满足。

```text
Token 1 -> Token 2 -> Token 3 -> ...
```

这条链具有串行依赖。

单个请求无法像 Prefill 那样同时生成所有未来 Token。

### 四、两阶段的资源特征为什么不同

#### Prefill 的常见特征

- 一次处理很多输入 Token；
- 矩阵乘法规模较大；
- 长输入会产生明显计算峰值；
- 产出大块 KV Cache；
- 直接影响首 Token 等待。

在不少模型和硬件组合上，Prefill 更容易接近计算吞吐瓶颈。

#### Decode 的常见特征

- 每个序列每步只有一个新 Token；
- 不断读取模型权重和历史 KV；
- 每步计算量较小但步骤多；
- KV Cache 随序列长度增长；
- 直接影响流式输出的平滑程度。

在不少批量配置上，Decode 更容易受内存带宽或 KV 容量约束。

这些是常见趋势，不是对所有架构和负载的硬性定律。

批量、量化、MoE、上下文长度、并行策略和硬件都会改变瓶颈。

### 五、TTFT 是什么

TTFT 全称 Time To First Token。

定义为：

$$
TTFT=t_{first\_token}-t_{request\_sent}
$$

其中：

- $t_{request\_sent}$：请求发出或服务端接收时间；
- $t_{first\_token}$：首个输出 Token 可用的时间。

TTFT 通常包含：

```text
网络 + 网关 + 排队 + Prefill + 首次采样 + 首包传输
```

报告 TTFT 时必须说明测量边界。

客户端计时和服务端计时会不同。

### 六、TPOT 是什么

TPOT 全称 Time Per Output Token。

一种常见定义是：

$$
TPOT=\frac{T_{e2e}-TTFT}{N_{out}-1}
$$

其中：

- $T_{e2e}$：从请求到最后 Token 的总时间；
- $TTFT$：首 Token 时间；
- $N_{out}$：输出 Token 数；
- 减 1：首 Token 已计入 TTFT，剩余有 $N_{out}-1$ 个间隔。

这个定义要求 $N_{out}>1$。如果请求只产生一个输出 Token，就没有“首 Token 之后的间隔”，该请求的 TPOT 应记为不适用，而不是除以 0；聚合指标也要明确如何处理这类请求。

假设：

```text
TTFT = 1.2 秒
完整耗时 = 5.2 秒
输出 = 101 Token
```

则：

$$
TPOT=\frac{5.2-1.2}{100}=0.04\text{ 秒/Token}
$$

也就是平均每个后续 Token 40 毫秒。

有些系统把 TPOT 定义为 Decode 总时间除以全部输出 Token。

比较数据前要确认定义。

### 七、ITL 是什么

ITL 全称 Inter-Token Latency，即相邻 Token 的间隔。

第 $i$ 个间隔：

$$
ITL_i=t_i-t_{i-1}
$$

其中：

- $t_i$：第 $i$ 个 Token 到达时间；
- $t_{i-1}$：前一个 Token 到达时间。

TPOT 是平均值，可能掩盖卡顿。

例如：

```text
大多数 Token：20 ms
每隔一段突然停顿：500 ms
```

平均 TPOT 可能看起来尚可，但用户会明显感觉输出不顺。

因此还应看 ITL 的 P95/P99 和时间序列。

### 八、吞吐是什么

吞吐至少有两种口径。

#### 1. 请求吞吐

$$
Throughput_{req}=\frac{N_{completed}}{\Delta t}
$$

单位可以是请求/秒。

#### 2. Token 吞吐

$$
Throughput_{tok}=\frac{N_{input}+N_{output}}{\Delta t}
$$

也可以分开报告输入 Token/s 和输出 Token/s。

一个长 Prompt 请求和一个短 Prompt 请求，工作量差异很大。

所以只看请求/秒可能误导。

### 九、Goodput 为什么更接近生产目标

吞吐会把超时或严重违反 SLO 的完成请求也算进去。

Goodput 只计算满足服务目标的有效工作。

例如：

$$
Goodput=\frac{N_{completed\ and\ SLO\ met}}{\Delta t}
$$

其中：

- $N_{completed\ and\ SLO\ met}$：完成且满足 TTFT、TPOT 等约束的请求数；
- $\Delta t$：统计时间窗口。

假设系统每秒完成 100 个请求，但只有 60 个满足 TTFT < 2 秒和 TPOT < 50 ms：

```text
吞吐 = 100 请求/秒
Goodput = 60 请求/秒
```

为了追求高吞吐把批量做得过大，可能让 Goodput 下降。

### 十、为什么 Prefill 会干扰 Decode

设一批正在 Decode 的请求需要每几十毫秒运行一轮。

此时插入一个 50,000 Token 的 Prefill。

如果内核长时间占据 GPU：

- Decode 请求无法按时调度；
- ITL 出现尖峰；
- TPOT 和尾延迟变差；
- 流式输出突然停顿。

反过来，如果调度器总优先照顾 Decode：

- 新请求的 Prefill 被推迟；
- TTFT 上升；
- 长 Prompt 可能饥饿。

这就是阶段干扰。

### 十一、第一种缓解：连续批处理

静态批处理等待整批请求一起结束，长输出会拖住短请求。

Continuous Batching 允许在每个迭代：

- 已结束请求离开；
- 新请求进入；
- 活跃序列继续 Decode。

它显著提高服务利用率，是现代 LLM Serving 的基础。

但如果 Prefill 工作块仍很大，阶段干扰仍可能存在。

### 十二、第二种缓解：Chunked Prefill

Chunked Prefill 把长 Prompt 分成小块：

```text
50,000 Token Prefill
  -> chunk 1
  -> 插入 Decode
  -> chunk 2
  -> 插入 Decode
  -> ...
```

优点：

- 避免一个超长 Prefill 长时间独占设备；
- 可以与 Decode 交错或混合批处理；
- 不需要跨节点转移 KV；
- 架构改动相对小。

代价：

- 一个请求 Prefill 被切成多轮，TTFT 可能增加；
- 调度器更复杂；
- Chunk 太小会增加内核和调度开销；
- Chunk 太大又难以缓解干扰。

Sarathi-Serve 等工作研究了通过分块 Prefill 改善吞吐—延迟权衡。

### 十三、第三种方案：Prefill/Decode 分离

分离架构把两阶段放到不同工作者：

```text
              +----------------+
请求 --------> | Prefill Worker |
              +----------------+
                       |
                       | KV Cache + 请求状态
                       v
              +----------------+
流式输出 <---- | Decode Worker  |
              +----------------+
```

Prefill Worker：

- 接收 Prompt；
- 完成输入前向计算；
- 生成 KV Cache；
- 选择或通知 Decode Worker；
- 把 KV 和元数据传过去。

Decode Worker：

- 接收 KV；
- 继续生成 Token；
- 执行停止条件和采样；
- 流式返回客户端；
- 请求结束后释放状态。

### 十四、分离为什么有机会更好

#### 1. 性能隔离

长 Prefill 不再直接占用 Decode Worker 的执行槽。

Decode 的 ITL 更容易稳定。

#### 2. 独立扩缩容

Prompt 突然变长时，可以增加 Prefill 容量。

输出变长时，可以增加 Decode 容量。

#### 3. 不同硬件与并行策略

Prefill 可能偏好高算力和较大的张量并行。

Decode 可能更重视显存容量、带宽和高效连续批处理。

分离后可以分别优化，但是否值得使用异构硬件需实测。

#### 4. 分别控制 SLO

Prefill 调度器关注 TTFT。

Decode 调度器关注 TPOT/ITL。

相比在单队列里平衡两个目标，控制面可能更清晰。

### 十五、分离的最大代价：KV Cache 传输

Prefill 生成的不是几个字节，而是每层、每个 Token 的 K/V 张量。

KV 大小近似：

$$
M_{KV}\approx2LBSH_{kv}D_hb
$$

符号含义：

- $L$：层数；
- $B$：序列批量；
- $S$：输入 Token 数；
- $H_{kv}$：KV Head 数；
- $D_h$：Head 维度；
- $b$：每元素字节；
- 2：K 和 V 两份。

跨节点传输时间粗略为：

$$
T_{transfer}\approx\frac{M_{KV}}{BW_{effective}}+T_{setup}
$$

其中：

- $BW_{effective}$：考虑协议和争用后的有效带宽；
- $T_{setup}$：连接、注册、同步等固定开销。

如果传输 4 GiB KV，有效带宽 40 GiB/s，忽略固定开销：

$$
T_{transfer}\approx\frac{4}{40}=0.1\text{ 秒}
$$

100 ms 对某些离线任务可以接受，对严格实时 SLO 可能很大。

### 十六、如何隐藏 KV 传输时间

常见思路：

- 分块计算后流水式传输；
- 使用高速互联和 RDMA；
- 让 Decode Worker 提前预留空间；
- 计算与传输重叠；
- KV 压缩或低精度传输，但需验证质量；
- 根据数据局部性选择 Decode Worker；
- 命中远端 KV 缓存时跳过部分重算。

“隐藏”不表示网络没有成本，只表示成本与计算并行，减少关键路径暴露。

### 十七、路由器如何配对 Prefill 与 Decode

路由要考虑：

- Decode Worker 的空闲 KV 容量；
- 当前活跃序列数；
- 预计输出长度；
- 网络拓扑和带宽；
- 模型、Adapter 与版本；
- 租户和数据位置；
- SLO 优先级；
- Worker 健康状态。

只选“队列最短”的节点可能把 KV 传到很远的位置，增加总时间。

更好的成本函数可以是：

$$
Cost_j=w_qQ_j+w_tT_{transfer,j}+w_mM_j+w_sSLO_j
$$

其中：

- $j$：候选 Decode Worker；
- $Q_j$：预计排队时间；
- $T_{transfer,j}$：KV 传输时间；
- $M_j$：显存压力；
- $SLO_j$：违反 SLO 的风险估计；
- $w_*$：业务设置的权重。

### 十八、两阶段资源比例怎么定

粗略需求量可以从工作量估算。

设每秒到达 $\lambda$ 个请求：

$$
W_{prefill}=\lambda\cdot E[N_{in}]
$$

$$
W_{decode}=\lambda\cdot E[N_{out}]
$$

其中：

- $E[N_{in}]$：平均输入 Token；
- $E[N_{out}]$：平均输出 Token；
- $W_{prefill}$：每秒输入 Token 工作量；
- $W_{decode}$：每秒输出 Token 工作量。

但输入 Token/s 和输出 Token/s 不能直接按一比一换算成同等 GPU 数量，因为两阶段每 Token 的计算与带宽特征不同。

应使用目标模型和硬件上的实测服务率：

$$
n_p\ge\frac{W_{prefill}}{\mu_p\rho_p}
$$

$$
n_d\ge\frac{W_{decode}}{\mu_d\rho_d}
$$

其中：

- $n_p,n_d$：Prefill 与 Decode 实例数量；
- $\mu_p,\mu_d$：单实例实测处理能力；
- $\rho_p,\rho_d$：目标安全利用率，小于 1，为突发和尾延迟留余量。

### 十九、背压和故障怎么处理

可能出现：

1. Prefill 很快，Decode 堆积；
2. Decode 有空闲，但 Prefill 队列很长；
3. KV 传输中断；
4. Prefill 完成后目标 Decode Worker 故障；
5. 客户端取消，但 KV 仍在传输；
6. 版本不匹配，Decode 无法读取 KV。

需要设计：

- 有界队列；
- 请求取消传播；
- KV 生命周期和引用计数；
- 传输校验；
- Worker 预留与超时；
- 失败后重算或重新路由；
- 版本协商；
- 防止重复流式输出的请求状态机。

### 二十、什么时候分离可能值得

通常更值得评估：

- 服务规模大，资源利用差异能产生明显收益；
- 输入和输出长度分布差异大；
- 长 Prefill 明显造成 Decode 卡顿；
- TTFT 和 TPOT 都有严格 SLO；
- 有足够快的节点互联；
- 团队能维护分布式 KV 和复杂调度；
- 工作负载相对稳定，可做容量建模。

通常先不要急于分离：

- 流量很小；
- 单机或单节点已经满足 SLO；
- Prompt 和输出都很短；
- 网络较慢；
- 团队尚未做好连续批处理、Paged KV 和基础监控；
- 运维复杂度比节省的 GPU 更昂贵。

## 工程实践与边界

### 一、先优化同机基线

顺序通常可以是：

1. 正确使用 KV Cache；
2. 使用连续批处理；
3. 做合理的 Paged KV 管理；
4. 调整最大批量和调度；
5. 评估 Chunked Prefill；
6. 确认阶段干扰仍是主要瓶颈；
7. 再评估 PD 分离。

这样能判断收益来自哪里。

### 二、压测要使用到达率，而不只是固定并发

固定并发适合观察饱和吞吐。

生产流量常有随机到达和突发。

应模拟：

- 平均到达率；
- 峰值突发；
- 输入长度分布；
- 输出长度分布；
- 优先级；
- 客户端取消；
- 缓存命中。

### 三、分开报告不同切片

至少按以下切片：

```text
短输入/短输出
短输入/长输出
长输入/短输出
长输入/长输出
```

平均值可能掩盖长输入对 TTFT 和长输出对 Decode 容量的影响。

### 四、SLO 要定义在用户边界

例如：

```text
TTFT P95 < 2 秒
ITL P95 < 80 ms
请求成功率 > 99.9%
```

服务端内核时间不等于用户感知时间。

网关、网络、排队和流式协议也要纳入。

### 五、成本要算网络与冗余

PD 分离的成本不只有 GPU：

- 高速网卡与交换网络；
- KV 传输带宽；
- 缓冲和远端存储；
- 冗余容量；
- 控制面；
- 可观测性；
- 故障演练和运维人力。

### 六、保证状态版本一致

KV 生产者和消费者必须匹配：

- 模型权重；
- Tokenizer；
- Adapter；
- KV 数据类型；
- 张量布局；
- 并行分片；
- 引擎协议版本。

元数据中应携带不可歧义的版本标识和校验。

### 七、取消要贯穿整条链路

用户断开后：

- 网关停止等待；
- Prefill 任务取消；
- 正在传输的 KV 释放；
- Decode 不再占用批次；
- 远端缓存引用减少。

否则大量“幽灵请求”会继续消耗 GPU。

### 八、用 Trace 定位阶段问题

一个请求的 Trace 至少包含：

```text
gateway_start
queue_enter / queue_leave
prefill_start / prefill_end
kv_transfer_start / kv_transfer_end
decode_admit
first_token
each_token_or_aggregated_itl
request_finish / cancel / error
```

只有分阶段时间，才能判断 TTFT 高在排队、Prefill 还是 KV 传输。

## 常见误区

### 误区一：Prefill 生成 Prompt 的所有答案 Token

Prefill 处理已知输入并产生首个输出分布；后续答案仍由 Decode 逐步生成。

### 误区二：Decode 不需要计算，只读缓存

每步仍要运行模型层、计算新 Q/K/V、访问历史 KV 并产生 Logit。

### 误区三：TTFT 就等于 Prefill 内核时间

TTFT 还包括网络、网关、排队、采样和首包传输。

### 误区四：TPOT 低，用户体验就肯定好

平均 TPOT 可能掩盖很差的 TTFT 或偶发 ITL 卡顿。

### 误区五：吞吐越高，服务越好

如果大部分请求违反延迟 SLO，裸吞吐没有体现有效服务能力。

### 误区六：PD 分离可以消除所有干扰

它隔离计算阶段，却新增网络、队列、KV 容量和上下游背压。

### 误区七：KV 传输只是几个 Token ID

传输的是各层 K/V 张量，长 Prompt 时数据量可能很大。

### 误区八：Prefill 和 Decode 实例按请求数一比一配置

比例取决于输入输出 Token 分布、单实例服务率和 SLO。

### 误区九：Chunked Prefill 和 PD 分离是同一个方案

前者在调度中切分 Prefill，后者把阶段放到不同工作者并传递 KV。

### 误区十：分离架构越复杂越先进

是否采用应由真实瓶颈、规模、网络和总拥有成本决定。

## 面试追问

### 追问一：Prefill 为什么更影响 TTFT？

首 Token 必须等输入 Prompt 被处理并产生首个分布，长 Prompt 的 Prefill 和排队会直接进入 TTFT。

### 追问二：Decode 为什么更影响 TPOT？

首 Token 之后的每个输出都依赖一次新的 Decode 步，调度间隔和 KV 访问决定 Token 流动速度。

### 追问三：PD 分离后 KV 怎么传？

可以通过 GPU 直连、RDMA、CPU 中转或远端 KV 系统传输，具体取决于硬件和框架；要带版本、布局和生命周期元数据。

### 追问四：如何判断 KV 传输值不值得？

比较“隔离和独立优化节省的排队/计算时间”与“传输、等待、额外资源和故障成本”，并以 Goodput 和总成本做端到端压测。

### 延伸问题与回答要点

| 问题 | 回答要点 |
| --- | --- |
| Chunk 大小怎么选 | 扫描不同输入和并发，平衡 Prefill 效率、Decode 卡顿和调度开销 |
| Goodput 怎么定义 | 先定义 TTFT、TPOT、错误率等 SLO，再只统计满足全部门槛的完成工作 |
| 如何独立扩容 | Prefill 看到达率与输入 Token，Decode 看输出 Token、活跃序列和 KV 容量 |
| 为什么看 P99 | 长 Prompt、排队和网络争用集中在尾部，平均值无法反映最差用户 |
| 推测解码放哪边 | 主要作用于 Decode，可能改变 Decode 资源和调度，需要与 PD 架构联合压测 |
| Prefix Cache 有何影响 | 命中会减少 Prefill 工作量，改变两阶段容量比例，也可能引入远端 KV 加载 |
