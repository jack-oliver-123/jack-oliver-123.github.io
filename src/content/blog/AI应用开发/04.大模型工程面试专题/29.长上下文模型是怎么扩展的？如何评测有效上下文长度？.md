一个模型接口写着“支持超长上下文”。

你把一本书放进去，在最后问：“第 37 页提到的合同期限是多少？”

模型也许答对。

然后你把同一个事实放到全文中间，周围加入十个相似日期，它却答错了。

这说明两个概念不同：

- **标称上下文窗口**：接口能接收多少 Token；
- **有效上下文长度**：在目标任务和位置上，模型还能可靠利用多少 Token。

长上下文不是只把一个配置项从 8K 改成 128K。

它同时涉及位置表示、训练数据、注意力计算、KV Cache、推理服务和评测方法。

## 本篇目标

学完本篇，你应该能够：

1. 解释 Token、上下文窗口和训练长度的区别；
2. 说明为什么模型超出训练长度后可能退化；
3. 理解 RoPE 如何把位置信息加入 Query 和 Key；
4. 解释 Position Interpolation、YaRN 和 LongRoPE 的基本思路；
5. 区分位置外推、继续训练、稀疏注意力、记忆压缩和 RAG；
6. 分析标准注意力的计算与 KV Cache 成本；
7. 解释为什么 Needle-in-a-Haystack 只能测部分能力；
8. 用 RULER、LongBench、InfiniteBench 和 Lost in the Middle 设计多维评测；
9. 区分“能输入”“能定位”“能推理”“能在预算内服务”；
10. 在面试中给出长上下文选型和生产边界。

先记住一句话：

> 上下文窗口是输入上限，不是记忆质量保证。长上下文扩展要让位置表示、训练分布、注意力与服务系统共同适配；有效长度必须在真实任务、不同位置、不同干扰强度和真实成本下测量。

## 60 秒回答

长上下文扩展通常有四类手段。

第一类是位置编码扩展。对 RoPE 模型，可以缩放位置、调整频率或做非均匀插值，让更长位置映射到模型熟悉的相位范围。Position Interpolation、YaRN、LongRoPE 都属于这一方向，但具体缩放和训练策略不同。

第二类是在长序列上继续预训练或微调，让模型学习跨长距离使用信息。只改 RoPE 参数而没有长数据适配，模型可能能运行，却不擅长利用远处内容。

第三类是降低计算和内存，例如滑动窗口、稀疏注意力、分块、压缩记忆或线性注意力。它们改变哪些 Token 可以直接交互，可能牺牲部分全局信息。

第四类是系统级方案，例如 Prefix Cache、KV 分层、并行和 RAG。RAG 通过先检索再输入减少上下文，不等于扩展模型原生窗口，但常是更经济的替代。

评测不能只做单针检索或看 Perplexity。我会按长度、信息位置、针的数量、干扰项、任务类型和语言分层，测精确检索、多跳推理、聚合、代码和真实文档任务，同时记录输入 Token、TTFT、显存、吞吐与成本。某长度只有在关键任务达到质量门槛并满足 SLO 时，才算项目的有效上下文。

## 详细解析

### 一、什么是上下文窗口

模型一次前向或连续生成能直接条件化的 Token 总数，通常称为上下文窗口。

它包括：

```text
系统提示词
+ 对话历史
+ 检索文档
+ 工具定义
+ 用户输入
+ 已生成输出
```

如果窗口上限为 $W$，输入 Token 为 $N_{in}$，计划最多输出 $N_{out}$，通常需要：

$$
N_{in}+N_{out}\le W
$$

还要为模板、特殊 Token 和工具结果留出空间。

不能把全部窗口都塞满输入，然后期待模型继续生成很长答案。

### 二、字符、字数和 Token 不等价

上下文长度通常按 Token 计，不按字符或汉字计。

同一句内容在不同 Tokenizer 下可能得到不同 Token 数。

影响因素包括：

- 语言；
- 空格与标点；
- 代码和数字；
- 特殊 Token；
- Tokenizer 词表；
- 图片或音频对应的视觉/音频 Token。

因此评测必须用目标模型的实际 Tokenizer 计数。

“100 页 PDF”不是可复现的上下文长度单位。

### 三、训练长度、支持长度和有效长度

#### 1. 训练长度

模型预训练或后训练时实际看到的序列长度分布。

即使最大训练长度很长，大部分样本也可能较短。

#### 2. 支持长度

模型配置和推理框架允许接受的最大 Token 数。

超过后可能被拒绝或截断。

#### 3. 有效长度

在某个任务、位置、干扰和质量门槛下仍能可靠使用的长度。

可以写成业务定义：

$$
L_{effective}=\max\{L:Q(L,s)\ge Q_{min},\ SLO(L)\text{ 满足}\}
$$

其中：

- $L$：测试上下文长度；
- $s$：任务与位置等切片；
- $Q(L,s)$：该长度和切片上的质量；
- $Q_{min}$：业务最低质量门槛；
- $SLO(L)$：该长度下的延迟、成本和可用性要求。

有效长度不是模型的单一常数。

它会随任务和标准变化。

### 四、为什么 Transformer 需要位置编码

纯注意力根据内容相似度计算，不天然知道顺序。

下面两句词相同、顺序不同：

```text
小王帮助小李
小李帮助小王
```

位置表示帮助模型区分“谁在前、谁在后”和 Token 距离。

常见方法包括：

- 基于位置索引的可学习位置嵌入；
- Sinusoidal 位置编码；
- RoPE；
- ALiBi；
- 相对位置偏置。

很多现代 Decoder 模型使用 RoPE，因此长上下文扩展经常围绕 RoPE 展开。

### 五、RoPE 的核心直觉

RoPE 全称 Rotary Position Embedding。

它不是简单把位置向量加到词向量，而是按位置旋转 Query 和 Key 的二维分量。

对二维向量，旋转角度 $m\theta$ 的矩阵为：

$$
R(m\theta)=
\begin{bmatrix}
\cos(m\theta)&-\sin(m\theta)\\
\sin(m\theta)&\cos(m\theta)
\end{bmatrix}
$$

其中：

- $m$：Token 位置；
- $\theta$：该维度对的基础角频率；
- $R$：旋转矩阵。

位置 $m$ 的 Query 变为：

$$
q_m'=R(m\theta)q_m
$$

位置 $n$ 的 Key 变为：

$$
k_n'=R(n\theta)k_n
$$

两者点积包含相对位置 $m-n$ 的信息。

不同维度对使用不同频率：

$$
\theta_i=\text{base}^{-2i/d}
$$

其中：

- $i$：第几个二维维度对；
- $d$：Head 维度；
- `base`：RoPE 基数超参数。

低频维度变化慢，适合表示较长距离；高频维度变化快，提供细粒度位置区分。

### 六、为什么直接超出训练长度可能退化

假设训练主要看到位置 0 到 4095。

推理突然使用位置 100,000：

- 旋转相位进入训练中未充分覆盖区域；
- 高频维度发生大量周期旋转；
- 注意力分数分布可能变化；
- 模型未学会在如此远距离进行检索和组合；
- 长序列带来更多干扰项。

即使数学函数能计算，模型行为也可能不稳定。

“公式可以外推”和“神经网络学会使用外推位置”不是同一件事。

### 七、Position Interpolation 的思路

Position Interpolation，简称 PI，把长位置压缩到训练过的范围。

假设原训练长度为 $L_0$，目标长度为 $L_1$，缩放因子：

$$
s=\frac{L_1}{L_0}
$$

把新位置 $m$ 映射为：

$$
m'=\frac{m}{s}
$$

例子：

```text
原长度 L0 = 4096
目标长度 L1 = 32768
s = 8
新位置 24000 -> 插值位置 3000
```

这样 RoPE 看到的相位范围回到训练区间附近。

优点：

- 简单；
- 通常只需少量长上下文继续训练；
- 避免直接把所有位置外推到未见范围。

边界：

- 所有位置被压缩，近距离位置分辨率可能下降；
- 缩放很大时局部 Token 角度更接近；
- 仍需要训练让模型适应新映射；
- 只处理位置，不解决注意力和 KV 的系统成本。

### 八、NTK-aware 类缩放的直觉

不同 RoPE 频率承担不同尺度的信息。

统一把所有位置除以同一个数，可能过度压缩高频或低频维度。

NTK-aware 类方法通过调整 RoPE base 或按频率设计缩放，让不同频率以不同方式扩展。

直觉是：

```text
高频维度：保留局部位置分辨率
低频维度：承担更远距离变化
```

“NTK-aware”在不同代码库中可能对应不同公式和实现。

使用时要以模型发布说明和训练配置为准，不能只复制一个 `rope_scaling` 字段。

### 九、YaRN 做了什么

YaRN 全称 Yet another RoPE extensioN method。

它在频率维度上结合插值与外推思想，并引入注意力幅度或温度相关的调整，以改善长上下文扩展效率和短上下文表现。

可以用直觉理解：

- 对某些频率更偏向保留原始行为；
- 对另一些频率更多做插值；
- 在过渡区平滑混合；
- 通过较少长序列训练适配目标窗口。

YaRN 不是只改一个最大长度数字。

它包含位置频率处理、训练长度与超参数组合。

### 十、LongRoPE 做了什么

LongRoPE 研究非均匀位置插值。

核心观察是：

- 不同 RoPE 维度对扩展的敏感度不同；
- 不同位置区间也可能需要不同处理；
- 统一缩放未必是最合适的方案。

它搜索或优化每个维度的缩放因子，并通过渐进扩展等策略把上下文推向更长范围。

代价是：

- 配置更复杂；
- 需要搜索和训练资源；
- 结果与具体模型强相关；
- 长度声明仍需任务评测验证。

### 十一、ALiBi 为什么也能外推

ALiBi 不把位置向量加进表示，而是在注意力分数上加入与距离成比例的线性偏置。

简化形式：

$$
score_{ij}=\frac{q_i k_j^\top}{\sqrt{d}}-m_h(i-j)
$$

其中：

- $i$：Query 位置；
- $j$：Key 位置，因果注意力下通常 $j\le i$；
- $m_h$：第 $h$ 个 Head 的斜率；
- 距离越远，偏置通常越负。

ALiBi 的论文展示了“Train Short, Test Long”的外推能力。

它也会对远距离内容施加衰减偏好，具体任务效果需验证。

已有 RoPE 模型不能只在部署时无训练地切换成 ALiBi。

### 十二、只改位置编码为什么不够

长上下文能力还依赖训练数据。

模型需要看到并学习：

- 在远处找到信息；
- 跨章节合并证据；
- 抵抗相似干扰项；
- 在全文中计数和聚合；
- 理解长代码依赖；
- 在不同位置保持指令优先级。

如果继续训练数据只是把短文随机拼接成长文，模型可能主要学到局部语言建模，而不是长程任务。

长训练数据的结构和任务设计很重要。

### 十三、长序列继续训练的代价

标准全注意力的注意力分数矩阵大小与序列长度平方相关：

$$
M_{attn}\propto S^2
$$

其中 $S$ 是序列长度。

如果长度从 8K 增加到 32K，即增加 4 倍，朴素注意力矩阵元素数增加约：

$$
4^2=16\text{ 倍}
$$

FlashAttention 等方法减少中间矩阵的显存读写和物化，但标准全注意力的运算规模仍与 $S^2$ 相关。

训练还要保存激活和梯度，因此长序列成本很高。

### 十四、推理阶段的成本

#### Prefill

标准全注意力 Prefill 的注意力计算随输入长度近似二次增长，其他线性层随长度线性增长。

实际总瓶颈取决于模型宽度、硬件和内核。

#### Decode

有 KV Cache 后，每个新 Token 不重算历史 K/V，但要读取历史缓存。

每步注意力访问量随当前上下文长度近似线性增长。

#### KV Cache

KV 容量近似：

$$
M_{KV}\approx2LBSH_{kv}D_hb
$$

序列长度 $S$ 增加 8 倍，KV 原始容量也增加约 8 倍。

所以长上下文会同时影响 TTFT、TPOT、并发和显存。

### 十五、稀疏与滑动窗口注意力

降低长序列成本的一类方法，是不让每个 Token 关注所有历史 Token。

#### 滑动窗口

每个 Token 只关注最近 $w$ 个 Token：

```text
当前位置 i -> [i-w, ..., i]
```

注意力规模从全局 $O(S^2)$ 降为近似 $O(Sw)$。

优点：局部建模高效。

局限：超出窗口的信息不能直接被当前层访问，需要通过多层传播、全局 Token 或额外记忆机制。

#### 块稀疏

只让特定 block 之间连接，例如局部块加少量全局块。

需要专门稀疏内核，否则理论稀疏不一定转化为实际加速。

#### 全局 Token

让少量特殊位置能看到全局，并作为信息中转。

它依赖模型训练学会使用这些通道。

### 十六、压缩记忆与递归处理

另一类方法把旧上下文压缩：

```text
原始历史
  -> 摘要/记忆 Token/状态
  -> 与最近原文一起供模型使用
```

优点：内存可控。

风险：

- 压缩会丢细节；
- 早期摘要错误会持续传播；
- 很难回答需要原文精确措辞的问题；
- 记忆更新和冲突处理复杂。

这类系统的“可记住很久”不等于原生全注意力窗口同样长。

### 十七、RAG 与长上下文是什么关系

RAG 先检索相关片段，再把少量片段放进 Prompt。

```text
海量文档
  -> 检索 Top-K
  -> 只输入相关证据
  -> 生成答案
```

优点：

- 不必每次读取全部文档；
- 数据可以更新；
- 更容易引用来源；
- Prompt 与 KV 成本较低。

局限：

- 检索可能漏掉证据；
- 跨很多片段聚合困难；
- 需要索引、切块和重排；
- 不适合所有全局分析任务。

长上下文和 RAG 可以组合：先检索候选章节，再用较长窗口做综合。

### 十八、最基础评测：Perplexity

语言模型在序列上的平均负对数似然：

$$
L=-\frac{1}{N}\sum_{t=1}^{N}\log p(x_t\mid x_{<t})
$$

困惑度：

$$
PPL=\exp(L)
$$

其中：

- $N$：评测 Token 数；
- $x_t$：第 $t$ 个真实 Token；
- $p(x_t\mid x_{<t})$：模型给真实 Token 的概率；
- $L$：平均负对数似然；
- $PPL$：Perplexity。

较低 PPL 通常表示对该文本分布预测更好。

但 PPL 不直接回答：

- 能否找到远处的一句话；
- 能否跨文档推理；
- 能否抵抗干扰；
- 能否正确引用证据。

所以它只能作为一个信号。

### 十九、Needle-in-a-Haystack 测什么

经典 NIAH 流程：

1. 准备很长的无关文本 Haystack；
2. 在某个位置插入一条关键事实 Needle；
3. 在末尾提问；
4. 检查模型能否找回该事实。

例如：

```text
针：“特别通行码是 739184。”
问题：“特别通行码是什么？”
```

评测矩阵通常改变：

- 上下文长度；
- Needle 深度位置；
- 针的措辞；
- 背景文本。

结果可画成长宽热力图。

### 二十、为什么单针测试远远不够

单针通常是精确、孤立、无冲突的事实。

它可能被表面词匹配轻易找到。

真实任务还需要：

- 多个 Needle；
- 相似但错误的干扰项；
- 对所有目标聚合；
- 目标与问题不同措辞；
- 目标分散在不同文档；
- 多跳因果或时间推理；
- 识别没有答案；
- 长代码依赖。

模型通过 NIAH 只说明它在该设置下能做基础长程检索。

### 二十一、Lost in the Middle 告诉了什么

“Lost in the Middle”研究发现，一些语言模型对信息位置敏感：

- 关键信息在开头时表现较好；
- 在结尾时也可能较好；
- 放在长上下文中间时明显下降。

这说明评测不能只把答案放在固定位置。

至少测试：

```text
开头 0%-10%
前中 25%
中间 50%
后中 75%
结尾 90%-100%
```

还要随机化，避免模型记住模板位置。

### 二十二、RULER 如何提高难度

RULER 在 Needle 类任务之外加入：

- 多针追踪；
- 变量跟踪；
- 聚合；
- 问答；
- 不同干扰规模。

它强调标称上下文与实际任务有效长度之间的差异。

使用时应查看每个任务和长度的分数，而不是只看汇总数字。

### 二十三、LongBench 与 InfiniteBench

LongBench 提供双语、多任务长上下文评测，覆盖：

- 单文档问答；
- 多文档问答；
- 摘要；
- Few-shot 学习；
- 合成任务；
- 代码。

InfiniteBench 把评测扩展到超过 100K Token 的场景，覆盖长对话、书籍、代码和检索等任务。

这类基准比单针丰富，但仍不能替代业务文档、语言和风险要求。

### 二十四、一套有效长度评测矩阵

可以设计以下维度：

| 维度 | 示例档位 |
| --- | --- |
| 长度 | 4K、16K、32K、64K、目标上限 |
| 位置 | 开头、25%、50%、75%、结尾 |
| Needle 数 | 1、3、10、随长度增长 |
| 干扰 | 无干扰、同类数字、矛盾说法、过期版本 |
| 任务 | 检索、聚合、多跳、摘要、代码、拒答 |
| 语言 | 中文、英文、混合、代码 |
| 输出 | 短答案、引用、长综合 |
| 负载 | 单请求、目标并发、峰值突发 |

每个格子记录：

- 任务分数；
- 引用和证据正确率；
- TTFT；
- TPOT；
- 峰值显存；
- OOM 和截断；
- 每次成功任务成本。

### 二十五、避免评测本身泄漏答案

常见问题：

- Needle 使用固定模板；
- 关键事实词汇很罕见，容易被定位；
- 问题与原文完全同词；
- 背景内容与任务无关；
- 答案总在同一位置；
- 测试样本出现在训练或公开演示中。

改进：

- 使用多种模板；
- 做同义改写；
- 加入高相似干扰；
- 随机位置；
- 保留私有生成种子；
- 对答案和上下文做语义去重；
- 人工检查样本是否真的需要长程信息。

### 二十六、一个任务可能不需要完整长上下文

如果问题只与一段政策有关，输入整本知识库可能比 RAG 更差：

- Prefill 更慢；
- 成本更高；
- 干扰更多；
- 关键证据更容易被淹没；
- KV Cache 更大；
- 并发下降。

“上下文能放下”不是“应该全部放进去”。

需要比较：

```text
全量长上下文
vs
RAG Top-K
vs
层级检索 + 长上下文综合
vs
预先结构化工具查询
```

### 二十七、长上下文与 Prompt Injection

长文档中可能包含：

- 恶意指令；
- 伪造系统提示；
- 数据外泄请求；
- 看似可信的过期规则。

上下文越长，非可信内容的攻击面越大。

必须区分：

- 系统指令；
- 用户指令；
- 检索到的非可信数据；
- 工具返回。

权限和数据外发控制应在模型外执行。

长上下文能力不能当作安全隔离。

## 工程实践与边界

### 一、先验证模型发布配置

记录：

- 权重版本；
- Tokenizer；
- 原训练长度；
- 目标上下文；
- RoPE scaling 类型和参数；
- 是否需要继续训练；
- 推理框架最大长度；
- KV 数据类型；
- 并行和缓存策略。

不要随意把 `max_position_embeddings` 改大就上线。

### 二、短上下文回归同样重要

位置插值和继续训练可能影响短任务。

要同时测：

- 短问答；
- 普通对话；
- 代码补全；
- 格式遵循；
- 原生产回归集。

长能力提升不应以关键短任务明显退化为代价，除非业务明确接受。

### 三、控制 Prompt 预算

可以按优先级分配：

```text
系统与安全规则：保留
当前用户请求：保留
关键工具 Schema：按需加载
检索证据：重排后截取
历史对话：摘要或裁剪
低价值日志：删除
输出预算：预留
```

截断必须知道删了什么。

不要从字符串末尾机械切割而破坏消息和文档边界。

### 四、做长度感知路由

路由器可考虑：

- Token 数；
- 任务是否需要全局阅读；
- 是否能检索缩短；
- 模型有效长度；
- 当前 KV 容量；
- 延迟和费用预算；
- 是否有专门长上下文集群。

超长请求可进入独立队列，避免阻塞普通 Decode。

### 五、监控长度分布和截断

线上记录：

- 输入/输出 Token 分位数；
- 被截断请求；
- 超长拒绝；
- TTFT 与长度关系；
- TPOT 与上下文长度关系；
- KV OOM；
- Prefix Cache 命中；
- 长请求成功率；
- 每长度桶成本。

### 六、引用比“说对答案”更可诊断

对文档问答，要求返回：

- 文档 ID；
- 页码或片段 ID；
- 支持结论的短引用；
- 无证据时明确说明。

评测可以分别检查检索证据和最终结论。

只看答案字符串，无法判断模型是从上下文找到还是碰巧猜中。

### 七、容量规划要考虑 KV

权重能装进 GPU，不代表目标长上下文并发能装下。

估算：

$$
B_{max}\lesssim\frac{M_{free}}{M_{KV\ per\ sequence}}
$$

其中：

- $B_{max}$：粗略可并发序列数；
- $M_{free}$：权重和工作区后剩余显存；
- $M_{KV\ per\ sequence}$：一条目标长度序列的 KV。

实际还要留碎片、临时张量和安全余量。

### 八、长上下文失败要分类

失败可能来自：

- Tokenizer 或截断；
- 位置扩展配置错误；
- 模型定位失败；
- 检索干扰；
- 多跳推理失败；
- 输出预算不足；
- KV OOM；
- 排队超时；
- 文档解析丢失结构。

不同失败需要不同修复，不能都归因于“模型上下文不够长”。

## 常见误区

### 误区一：上下文窗口等于模型记忆容量

窗口只是可见 Token 范围，有效利用能力随位置、任务和干扰变化。

### 误区二：修改最大长度配置就完成扩展

位置表示、长数据训练、推理内核和 KV 容量都需要兼容。

### 误区三：通过单针测试就证明长程推理强

单针主要测简单定位，不覆盖多针、聚合、多跳和真实文档。

### 误区四：PPL 稳定就代表所有长任务稳定

语言预测与远距检索、引用和推理不是同一个指标。

### 误区五：FlashAttention 把全注意力计算变成线性

它显著优化内存访问和中间存储，但标准全注意力的理论计算规模仍随序列长度平方增长。

### 误区六：长窗口可以替代 RAG

全量输入与检索各有边界，成本、新鲜度、干扰和召回要联合评测。

### 误区七：RAG 就是扩展原生上下文

RAG 是外部检索后选择输入，模型原生窗口上限并未因此改变。

### 误区八：位置插值只影响长文本

频率和位置映射变化也可能影响短上下文，需要回归。

### 误区九：能跑到目标长度就能生产使用

还要满足任务质量、TTFT、TPOT、显存、并发和成本要求。

### 误区十：所有失败都是位置编码问题

训练覆盖、注意力结构、服务截断、文档解析和评测设计都可能是原因。

## 面试追问

### 追问一：Position Interpolation 为什么能扩展 RoPE？

它把目标长位置按比例压回模型训练过的相位范围，再通过少量长序列训练让模型适应；代价是位置被压缩，局部分辨率可能下降。

### 追问二：标称 128K，如何判断有效长度？

按长度、位置、干扰和任务建立矩阵，在质量与 SLO 同时达到门槛的最大长度作为该场景的有效长度，而不是直接采用接口上限。

### 追问三：为什么信息放中间更容易丢？

一些模型存在首因、近因和注意力位置偏好，长上下文中间信息可能获得较弱利用；具体幅度需用位置扫描实测。

### 追问四：长上下文和 RAG 怎么选？

需要全局比较、跨章节聚合且文档规模可控时长上下文有价值；知识库巨大、更新频繁或只需少量证据时，RAG 往往更经济。也可以先检索再长上下文综合。

### 延伸问题与回答要点

| 问题 | 回答要点 |
| --- | --- |
| RoPE 为什么有多个频率 | 不同二维维度对覆盖不同位置尺度，兼顾局部与长距离表示 |
| YaRN 和 PI 区别 | PI 统一压缩位置；YaRN 对不同频率采用混合策略并配合注意力调整与训练 |
| 如何测多跳能力 | 把多个必要证据放在远距离和不同位置，设置相似干扰，要求引用每一步证据 |
| 为什么 KV 是瓶颈 | KV 随层数、批量、长度和 KV Head 数线性增长，挤压可并发序列数 |
| 滑动窗口会丢什么 | 窗口外内容无法直接访问，依赖层间传播、全局 Token 或额外记忆 |
| 如何避免长请求拖垮服务 | 长度感知路由、单独队列、Chunked Prefill、配额、超时和容量预留 |
