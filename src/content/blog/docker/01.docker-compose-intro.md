## 为什么用 Docker Compose

用 `docker run` 启动单个容器没问题，但当参数越来越多（端口、环境变量、数据卷、网络……），命令会变成几百字符的长串，维护困难且容易出错。

Docker Compose 用一个 `docker-compose.yml` 文件声明式地描述服务配置，一条命令启停所有相关容器。优势：

- **可读性好**：YAML 格式，每个配置项一行；
- **可版本控制**：提交到 Git，团队共享同一套环境；
- **多容器编排**：定义依赖关系、共享网络、统一管理生命周期。

## 安装

Docker Desktop（Windows/Mac）自带 Docker Compose。Linux 使用 `docker-compose-plugin`：

```bash
# 如果安装 Docker 时已包含 docker-compose-plugin，直接可用
docker compose version
```

> 注意：新版推荐使用 `docker compose`（空格，V2 命令），旧版 `docker-compose`（连字符，V1）仍然兼容但已不再更新。本系列统一使用 V2 写法。

## 基本结构

一个最小的 `docker-compose.yml`：

```yaml
services:
  redis:
    image: redis:7.2
    container_name: redis-local
    restart: always
    ports:
      - "6379:6379"
    volumes:
      - ./data:/data
    environment:
      - TZ=Asia/Shanghai
```

等价于：

```bash
docker run -d --name redis-local --restart always -p 6379:6379 -v ./data:/data -e TZ=Asia/Shanghai redis:7.2
```

Compose 版本更清晰，且支持添加 healthcheck、资源限制等高级配置。

## 核心配置项

```yaml
services:
  app-name:                    # 服务名（自定义）
    image: nginx:1.26          # 使用的镜像
    container_name: my-nginx   # 容器名（可选）
    restart: unless-stopped    # 重启策略
    ports:
      - "8080:80"              # 宿主机端口:容器端口
    volumes:
      - ./html:/usr/share/nginx/html   # 挂载目录
      - ./conf:/etc/nginx/conf.d       # 挂载配置
    environment:
      - TZ=Asia/Shanghai       # 环境变量
    healthcheck:               # 健康检查
      test: ["CMD-SHELL", "curl -fs http://localhost/ || exit 1"]
      interval: 10s
      timeout: 3s
      retries: 3
    deploy:                    # 资源限制（Docker Desktop 支持）
      resources:
        limits:
          memory: 512M
          cpus: '1.0'
    depends_on:                # 依赖关系
      db:
        condition: service_healthy
    networks:
      - app-net                # 加入自定义网络

networks:
  app-net:
    driver: bridge
```

## 常用命令

所有命令在 `docker-compose.yml` 所在目录执行：

```bash
# 启动所有服务（后台运行）
docker compose up -d

# 查看运行状态
docker compose ps

# 查看日志
docker compose logs -f

# 查看单个服务日志
docker compose logs -f redis

# 停止所有服务
docker compose down

# 停止并删除数据卷（慎用，数据会丢失）
docker compose down -v

# 重启单个服务
docker compose restart redis

# 重新构建并启动
docker compose up -d --build
```

## 多容器编排示例

Elasticsearch + Kibana 的组合演示了 `depends_on` + `healthcheck` 的配合：

```yaml
services:
  elasticsearch:
    image: elasticsearch:7.17.14
    environment:
      - discovery.type=single-node
      - ES_JAVA_OPTS=-Xms512m -Xmx512m
    ports:
      - "9200:9200"
    healthcheck:
      test: ["CMD-SHELL", "curl -fs http://localhost:9200/_cluster/health || exit 1"]
      interval: 10s
      timeout: 5s
      retries: 5

  kibana:
    image: kibana:7.17.14
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
    ports:
      - "5601:5601"
    depends_on:
      elasticsearch:
        condition: service_healthy
```

这样 Kibana 会等 ES 完全就绪后才启动，避免连接失败。

## 日志管理

长时间运行的容器日志会不断增长。建议在 Compose 中配置日志轮转：

```yaml
services:
  app:
    image: nginx:1.26
    logging:
      driver: json-file
      options:
        max-size: "50m"
        max-file: "3"
```

这样单个日志文件最大 50MB，保留最近 3 个文件。

## 常见问题

### 端口冲突

报错 `port is already allocated` 说明宿主机端口被占用。修改 `ports` 中的宿主机端口（冒号左边）。

### volume 权限问题

Linux 下挂载的目录属于 root，容器内应用可能无权写入。解决方案：

```bash
# 方法 1：提前设置权限
chmod 777 ./data

# 方法 2：在 environment 中指定用户 ID
environment:
  - PUID=1000
  - PGID=1000
```

### 容器间通信

同一个 `docker-compose.yml` 中的服务默认在同一网络中，可以用**服务名**作为主机名互相访问。例如上面的 Kibana 用 `http://elasticsearch:9200` 连接 ES。
