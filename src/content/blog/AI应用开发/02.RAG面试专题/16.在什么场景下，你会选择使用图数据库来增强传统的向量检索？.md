## 本篇目标

学完这一篇，你应该能够：

1. 解释向量检索擅长“找相似内容”，图检索擅长“沿明确关系找路径”。
2. 区分图数据库、知识图谱和 GraphRAG，不把三个概念混为一谈。
3. 判断哪些多实体、多跳、依赖、影响分析问题适合图增强，哪些不值得。
4. 讲清实体抽取、实体消歧、关系来源、时态、图查询、证据回溯与增量更新。
5. 设计向量 + 图 + 原文证据的组合检索，并评测路径和答案质量。

## 60 秒回答

> 当问题的核心是明确的实体关系、路径约束或跨文档聚合，而不是只找语义相似段落时，我会考虑图数据库增强。例如“某供应商故障会影响哪些产品和客户”“某员工参与的项目依赖哪些系统”“两种药物通过哪些机制产生相互作用”。向量检索能找到相关描述，但不天然执行指定方向、跳数和关系类型的遍历。
>
> 典型实现是先从文档抽取实体、关系、来源、有效时间和置信度，做实体消歧后写入知识图；在线通过实体链接定位种子节点，沿受限关系和跳数扩展，再取节点或边关联的原始文档证据，与 BM25/向量候选融合并 Rerank。最终答案引用原文或权威记录，而不是把模型抽取出的边直接当成无条件事实。
>
> 我不会把图当作默认升级。若问题主要是单文档事实、关系稀疏、实体消歧差、更新频繁或维护成本高，混合检索加 Rerank 更简单。选型时要比较实体链接准确率、路径 Recall/Precision、答案质量、P95、图更新延迟和成本，并重点治理 ACL、来源追踪、错误边传播和删除。

## 详细解析

### 先说人话：向量在找“像什么”，图在找“和谁有什么关系”

假设有三段文档：

```text
文档 A：供应商“云芯”生产芯片 C7。
文档 B：产品“星云 X2”使用芯片 C7。
文档 C：客户“远山公司”采购了星云 X2。
```

用户问：

```text
云芯停产会影响哪些客户？
```

答案需要连接三段关系：

```text
云芯 --生产--> C7 --用于--> 星云 X2 --被采购--> 远山公司
```

向量检索可能分别找出这些段落，但不保证完整连接路径。图把实体和关系显式保存，能按方向和类型遍历。

### 三个概念先分清

#### 图数据库

图数据库是存储和查询节点、边、属性的数据库技术。

```text
(供应商)-[生产]->(芯片)-[用于]->(产品)
```

它提供图遍历、路径查询和图算法，但数据库里有什么取决于你的数据建模和写入质量。

#### 知识图谱

知识图谱是对实体、关系和属性的知识表示。它可以存于图数据库，也可以存在其他系统中。

知识图谱需要考虑：

- Schema/本体；
- 实体唯一身份；
- 关系语义与方向；
- 来源、时态和置信度；
- 冲突与版本。

#### GraphRAG

GraphRAG 是利用图结构增强检索和生成的一类方法。它可能使用知识图、社区发现、社区摘要、局部实体检索和全局主题检索。

因此：

```text
图数据库 != 知识图谱 != GraphRAG
```

三者有关联，但不是同一个概念。

### 向量检索为什么不擅长确定性关系查询

向量相似度回答的是：

```text
哪段文字和 Query 在表示空间里接近？
```

它不天然保证：

- 必须沿 `生产 -> 用于 -> 采购` 三种边；
- 只走两跳或三跳；
- 方向不能反；
- 列出全部上游依赖；
- 对同名实体使用正确身份；
- 在某个时间点关系有效。

图查询可以显式表达这些约束，但前提是图中的实体和边正确、完整、及时。

### 哪些场景适合图增强

#### 1. 多跳关系问题

```text
负责 X2 的团队还维护哪些被支付系统依赖的服务？
```

需要团队—项目—服务—依赖路径。

#### 2. 影响分析与依赖传播

- 某供应商停产影响哪些产品、订单和客户；
- 某服务故障影响哪些上游业务；
- 某法规变更关联哪些流程和合同。

#### 3. 组织、权限与所有权

- 谁负责某系统；
- 某人通过哪些角色拥有资源权限；
- 跨部门项目有哪些审批链。

权限结果应来自权威系统，图可用于查询和解释，不能代替最终授权判断。

#### 4. 推荐与关联发现

用户—商品—行为—属性形成图时，可发现多层关联。此类系统还需防止反馈偏差和隐私滥用。

#### 5. 全局主题与跨文档汇总

当问题问“整个语料中的主要风险主题及其关联实体”，GraphRAG 的社区发现和摘要可能比逐块向量检索更合适。

#### 6. 需要可解释路径

图能展示：

```text
结论来自哪条路径、每条边由什么来源支持
```

不过“路径可展示”不等于“路径事实正确”，来源仍需核验。

### 哪些场景通常不值得上图

- 问题主要是单份文档中的局部事实；
- 实体关系很少，几乎没有多跳查询；
- 文档变化极快，图更新无法跟上；
- 实体抽取和消歧准确率较低；
- 团队没有 Schema、图查询和数据治理能力；
- 混合检索 + Query 分解 + Rerank 已达到目标；
- 只为演示“关系网络”，没有明确业务指标。

图方案的长期成本常在实体治理和更新，而不是把 Neo4j 等数据库启动起来。

### 一条完整的离线图构建链路

```text
原始文档
  -> 解析与切块
  -> 实体/关系/属性抽取
  -> 实体规范化与消歧
  -> Schema 和规则校验
  -> 写入节点/边/来源/时间
  -> 社区或摘要（可选）
  -> 质量评测
  -> 发布图版本
```

### 第一步：设计 Schema

以供应链为例：

```text
节点：Supplier、Component、Product、Customer、Document
关系：PRODUCES、USED_BY、PURCHASED、SUPPORTED_BY
属性：source_id、effective_from、effective_to、confidence、tenant_id
```

Schema 过严会漏掉新关系，过松会让关系含义混乱。应从高价值 Query 反推最小 Schema。

### 第二步：实体与关系抽取

可以结合：

- 规则和词典；
- 命名实体识别模型；
- 大模型结构化抽取；
- 权威主数据/API；
- 人工审核高风险关系。

模型输出应经过 schema 校验，例如：

```json
{
  "subject": "云芯",
  "relation": "PRODUCES",
  "object": "C7",
  "source_document": "doc_a",
  "evidence_span": "云芯生产芯片 C7",
  "effective_from": "2026-01-01",
  "confidence": 0.94
}
```

`confidence` 只是抽取信号，不能替代来源和人工规则。

### 第三步：实体消歧

**实体消歧/实体链接**解决“文本中的名称到底指哪个真实实体”。

典型问题：

- “苹果”是公司还是水果；
- “X2”在不同产品线中是否同一型号；
- “云芯科技”和“云芯”是否同一供应商；
- 公司更名后旧名称如何映射；
- 不同租户可能都有名为“销售部”的节点。

常见方法：

1. 规范化名称和别名表；
2. 结合实体类型、上下文和唯一标识；
3. 使用向量候选 + 规则/模型精排；
4. 低置信度创建待审核实体，不强行合并；
5. 合并操作可回滚并保留审计记录。

错误合并两个实体会让大量路径被污染，通常比漏合并更危险。

### 第四步：为边记录来源、时态和置信度

不要只存：

```text
云芯 -> 生产 -> C7
```

还要存：

- 哪份原始文档或权威系统支持；
- 支持的具体文本位置；
- 何时生效和失效；
- 哪个抽取器/规则版本产生；
- 是否人工确认；
- 适用于哪个租户和权限范围；
- 是否存在冲突来源。

这样才能回答“2025 年当时的供应关系”和“现在的关系”这两个不同问题。

### 在线检索模式一：向量先找种子，图再扩展

```text
Query -> 向量检索定位“云芯/C7”相关文档或实体
      -> 实体链接
      -> 沿指定关系扩展 2~3 跳
      -> 获取路径关联原文
      -> Rerank
```

适合用户没有使用规范实体名称的自然语言问题。

### 在线检索模式二：实体先定位，图后取证

```text
Query -> 抽取“云芯”与“受影响客户”关系意图
      -> 实体链接到 supplier_001
      -> 图查询 PRODUCES/USED_BY/PURCHASED 路径
      -> 取每条边的原始证据
      -> 组织答案
```

适合实体明确、路径规则稳定的问题。

### 在线检索模式三：图与文本并行

```text
           -> 图路径候选 --------|
Query -----|                      |-> 去重/融合 -> Rerank
           -> BM25/向量文档候选 --|
```

图提供结构化关系，文本检索补充说明和细节。最终上下文同时包含路径和原文证据。

### 在线检索模式四：局部与全局 GraphRAG

#### 局部检索

从 Query 中的实体出发，取邻居、关系、相关文本和社区信息。适合“某实体与谁有关”。

#### 全局检索

利用社区摘要回答“整个语料库有哪些主要主题和风险”。它需要聚合多个社区的部分答案，再生成总体回答。

全局摘要由模型生成，必须保留到社区、实体、关系和原文的追溯路径。

### 控制图遍历爆炸

高连接节点可能迅速扩展出成千上万条路径。可限制：

- 允许的关系类型和方向；
- 最大跳数；
- 每层最大节点数；
- 节点/边置信度和有效时间；
- 与 Query 的相关性；
- 重复路径和环检测；
- 查询时间和总预算。

“多跳”不是跳数越多越好。路径越长，错误和无关关系传播的机会也越多。

### 贯穿案例完整回答

问题：

```text
云芯停产会影响哪些客户？
```

流程：

1. 实体链接确认“云芯”是供应商 `supplier_001`；
2. 查询有效的 `PRODUCES` 边，找到组件 C7；
3. 查询 `USED_BY`，找到产品 X2；
4. 查询权威订单数据中的 `PURCHASED`，找到受影响客户；
5. 过滤当前租户和用户可见客户；
6. 为每条路径取原始合同、BOM 和订单证据；
7. 对路径按影响状态、时间和来源权威性排序；
8. 回答中展示影响链和不确定项；
9. 若供应关系已过期或订单状态未知，明确说明。

## 工程实践与边界

### 图与文本双版本治理

文档更新时可能发生：

- 文本索引已经是新版本，图仍是旧关系；
- 边已删除，社区摘要还包含旧结论；
- 节点合并后引用仍指向旧 ID。

建议：

1. 使用稳定 source_id 和实体 ID；
2. 图、文本索引和摘要记录发布版本；
3. 新版本在 staging 中完成抽取和校验；
4. 原子切换兼容版本组合；
5. 删除时反向找到派生边和摘要；
6. 保留回滚和审计记录。

### ACL 与多租户

图遍历可能通过一条公开边进入敏感节点。每一步扩展都要执行权限约束，不能先遍历全图再在答案中隐藏。

需要测试：

- 跨租户同名实体；
- 节点可见但某些边不可见；
- 权限撤销后的缓存和摘要；
- 社区摘要是否混入其他租户内容；
- 图查询超时或授权服务失败时是否安全拒绝。

### 评测指标

#### 抽取与图质量

- 实体抽取 Precision/Recall/F1；
- 实体链接准确率；
- 关系类型、方向、时间和来源准确率；
- 冲突率、孤立节点率、重复实体率；
- 图更新与删除传播延迟。

#### 检索质量

- Seed Entity Recall：种子实体是否找对；
- Path Recall/Precision：所需路径是否找到、返回路径是否相关；
- 多跳答案所需证据覆盖率；
- 图路的独有相关证据与 Oracle Union Recall；
- 局部/全局 Query 分项质量。

#### 端到端

- 答案正确性和 Faithfulness；
- 路径与原文引用准确率；
- P95/P99、展开节点数和查询成本；
- 误合并导致的错误传播；
- 越权路径和敏感信息泄漏率。

### 选型决策表

| 判断问题 | 倾向图增强 | 倾向普通混合检索 |
| --- | --- | --- |
| 答案依赖明确关系和路径吗 | 是 | 否 |
| 查询经常跨多份文档聚合吗 | 是 | 很少 |
| 实体有稳定 ID 和主数据吗 | 是 | 否 |
| 关系抽取和消歧能达到业务要求吗 | 是 | 尚不能 |
| 图更新能跟上业务时效吗 | 是 | 否 |
| 路径解释有业务价值吗 | 是 | 不明显 |
| 混合检索 + 分解已满足指标吗 | 否 | 是 |

## 常见误区

### 误区一：图数据库能让大模型获得可靠推理

图提供结构化查询和路径，不保证抽取的边真实，也不保证模型对路径的解释正确。

### 误区二：多跳问题都应该用图

少量可预测子问题可能用 Query 分解和普通检索更简单。图适合关系被反复查询且可维护的场景。

### 误区三：实体抽取准确率高就够了

实体消歧、关系方向、时间、来源和删除传播同样会影响路径正确性。

### 误区四：GraphRAG 就是把图节点做 Embedding

图增强还包括图构建、社区发现、局部/全局查询、摘要和来源回溯。向量只是其中可能的一部分。

### 误区五：图中的边可以直接当作最终证据

自动抽取边可能出错。高风险答案应引用支持该边的原始文档或权威结构化记录。

### 误区六：先把全图遍历出来，再做权限过滤

越权内容可能已经进入内存、日志、缓存或模型。权限应参与每一步图查询。

## 面试追问

### 1. 图检索和向量检索怎样组合？

可以向量先定位种子实体再沿图扩展，也可以实体链接后图查询再取文本证据，或图与文本并行召回后融合、Rerank。

### 2. 实体消歧为什么重要？

同名不同实体若被错误合并，会产生大量虚假路径；同一实体若未合并，则路径被切断。它直接影响多跳查询质量。

### 3. 如何处理关系冲突？

保留每条边的来源、有效时间、权威等级和置信度；按业务规则选择；无法裁决时展示冲突，不静默覆盖。

### 4. 文档删除后图怎么更新？

通过 source_id 找到由该文档支持的边和摘要，移除该来源；若边还有其他可靠来源则保留并重算置信度，否则删除，再发布一致的新版本。

### 5. 如何避免图遍历爆炸？

限制关系白名单、方向、跳数、每层节点、时间窗口和总预算；按 Query 相关性剪枝；检测环和重复路径。

### 6. 为什么最终还要取原文？

原文提供可核验依据和细节。图边是结构化派生结果，可能缺条件、例外和上下文，不能替代来源证据。

### 7. 全局 GraphRAG 适合什么问题？

适合询问整个语料库主要主题、群体、趋势和跨文档模式的问题，不适合只查一个实体的精确事实。

### 8. 怎样证明上图值得？

建立关系型 Query 集，对比混合检索、Query 分解和图增强的路径 Recall、答案质量、P95、更新延迟与总成本，确认图路提供了独有价值。
