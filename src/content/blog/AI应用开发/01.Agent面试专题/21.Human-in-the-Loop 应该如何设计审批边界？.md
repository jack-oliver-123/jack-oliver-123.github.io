## 本篇目标

学完后，你应该能判断哪些动作可以自动执行、哪些必须人工审批，解释为什么审批必须绑定最终动作参数，并能设计包含暂停、审批、恢复、执行和审计的完整流程。

## 60 秒回答

Human-in-the-Loop 不是在流程最后统一加一个“确认”按钮，而是根据影响范围、可逆性、数据敏感度、判断置信度和异常程度划分审批边界。低风险动作可以自动执行，中等风险动作可以执行后抽样复核，高风险动作需要执行前审批，违反政策或超出系统能力的动作应直接禁止自动化。

涉及资金、不可逆删除、对外发送或发布、生产配置、权限提升、敏感数据访问，以及低置信但高影响的判断，通常应在执行前人工确认。审批要绑定不可变动作快照，包括工具、目标资源、关键参数、数据范围、理由、预期影响、有效期和审批人权限，并计算动作哈希；只要金额、收件人或资源等关键参数改变，原审批立即失效。

系统要把任务持久化到暂停点，恢复时重新检查身份、权限、审批有效期、动作哈希、资源版本和外部状态，避免审批后环境已经变化。人工审批是额外门禁，不能替代最小权限、参数校验、幂等、审计和补偿机制。

## 详细解析

### 先说人话：不是所有事情都让实习生来问，也不是所有事情都让他自己做

假设你带一个新实习生：

- 查公开资料，不需要每次请示；
- 给内部文档加标签，做错了容易撤销，可以抽查；
- 给客户发送正式邮件，需要发出前确认；
- 把公司数据库密码发给别人，无论谁点确认都不应该允许。

Human-in-the-Loop（常缩写为 **HITL**）就是在自动化流程中安排人类参与判断或批准。但重点不是“有没有人点按钮”，而是：**人应该在哪个风险边界介入，看到什么信息，批准的到底是哪一个动作。**

### 为什么不能让所有动作都审批

每个动作都找人确认，看起来最安全，实际会带来三个问题：

1. 流程很慢，自动化失去价值；
2. 人每天看到大量低价值弹窗，会形成审批疲劳；
3. 机械点击“同意”后，真正的高风险请求反而不容易被发现。

因此目标不是“审批越多越好”，而是把有限的人类注意力放到高影响、难恢复、信息不完整的决策上。

### 先认识六个术语

| 术语 | 通俗解释 |
| --- | --- |
| HITL | 自动流程在某些节点暂停，让人判断、补充或批准 |
| 审批边界 | 从自动执行转为必须人工确认的风险分界线 |
| 动作快照 | 准备执行的工具、资源和参数的固定版本 |
| 动作哈希 | 根据动作快照计算的指纹，用于发现参数被修改 |
| TOCTOU | 检查时和真正使用时之间，资源或条件发生了变化 |
| 补偿 | 已执行动作无法直接撤销时，用另一个业务动作纠正 |

### 贯穿案例：客服 Agent 发送退款通知

Agent 准备完成两件事：

1. 给订单 `order-123` 退款 800 元；
2. 向客户 `customer@example.com` 发送通知。

系统生成审批快照：

```json
{
  "approval_id": "approval-001",
  "tool": "create_refund",
  "resource": "order-123",
  "amount": 800,
  "currency": "CNY",
  "recipient": "customer@example.com",
  "reason": "服务无法交付",
  "expires_at": "2026-07-15T15:00:00+08:00",
  "action_hash": "sha256:..."
}
```

审批人看到真实金额、订单、收件人、理由和影响，确认后系统记录审批。执行前，运行时重新计算动作哈希：

- 参数完全相同且审批仍有效：可以继续；
- Agent 把金额改成 1,800 元：哈希变化，重新审批；
- 收件人变成其他邮箱：哈希变化，重新审批；
- 订单已经由别的流程退款：外部状态变化，停止并对账；
- 审批人的退款权限已被撤销：拒绝执行。

这说明“批准一个计划”和“批准最终动作”不是一回事。

### 如何给风险分级

可以综合五个维度：

| 维度 | 低风险例子 | 高风险例子 |
| --- | --- | --- |
| 影响范围 | 只影响草稿 | 影响大量客户或生产系统 |
| 可逆性 | 一键撤销的内部标签 | 不可追回的付款或公开发布 |
| 数据敏感度 | 公开资料 | 身份、医疗、财务或密钥数据 |
| 判断置信度 | 规则明确、证据一致 | 信息冲突、模型不确定 |
| 异常程度 | 常规金额和频率 | 金额突增、异常收件域、夜间批量操作 |

不要只用模型的“置信度”决定是否审批。模型可能对错误答案也很自信，高影响动作即使置信度高，也仍应受固定政策约束。

### 四类控制方式

| 风险级别 | 例子 | 建议控制 |
| --- | --- | --- |
| 低 | 读取公开资料、生成草稿 | 自动执行并记录 |
| 中 | 更新可回滚的内部标签 | 自动执行、抽样复核或异常时审批 |
| 高 | 对外发信、退款、修改生产配置 | 执行前审批 |
| 禁止 | 获取明文密钥、绕过访问控制 | 直接拒绝，不向 Agent 提供能力 |

“禁止”非常重要。Human-in-the-Loop 不代表一个本来违反安全政策的动作，只要有人点一下就能变合法。

### 哪些场景通常需要执行前审批

- 付款、退款、转账和购买；
- 删除、覆盖和不可逆的数据修改；
- 对外发邮件、发消息、发布内容或签署文件；
- 修改生产配置、部署和权限；
- 读取、导出或外发敏感数据；
- 代表用户作出法律、医疗或财务高影响决定；
- 模型证据不足、来源冲突或出现异常行为；
- 批量操作或影响范围超过阈值。

具体边界由业务、法规和组织风险偏好共同决定，不能只抄一张通用表。

### 审批界面应该展示什么

审批人要看见真实后果，而不是模型的一句“这个操作是安全的”。界面应展示：

- 将调用哪个工具；
- 操作哪个真实资源；
- 金额、收件人、文件、权限等关键参数；
- 与当前版本相比的差异；
- 数据来自哪里，证据是否可信；
- 为什么触发审批；
- 是否可撤销，失败后如何补偿；
- 审批范围和有效期；
- 批量操作的总量、上限和样本预览。

确认按钮附近不能用模糊文案，例如“继续优化”。应直接写明“退款 800 元”或“向 12 位客户发送邮件”。

### 一条完整的 HITL 流程

```text
Agent 提议高风险动作
   ↓
运行时先做身份、参数和政策校验
   ↓
生成不可变动作快照与 action_hash
   ↓
持久化任务并进入 waiting_for_approval
   ↓
向有权限的审批人展示真实影响
   ↓
批准 / 拒绝 / 要求修改 / 到期
   ↓
恢复时重新校验权限、哈希、版本和外部状态
   ↓
执行动作
   ↓
验证结果并分别记录审批、执行和结果审计
```

先做基本政策校验再送审，可以避免把明显不合法的动作推给人，减少审批噪音。

### 为什么暂停和恢复必须持久化

审批可能几小时后才完成，服务期间可能重启。暂停点至少要保存：

- `run_id`、当前节点和状态版本；
- 动作快照与哈希；
- 需要什么角色审批；
- 已有审批、拒绝和修改意见；
- 有效期、预算与截止时间；
- 相关证据引用；
- 外部动作是否已经发出。

不能只把审批消息发到群里，然后让内存中的线程一直等待。

### TOCTOU：批准时正确，执行时可能已经变了

TOCTOU 是 “Time of Check to Time of Use”，即检查到使用之间的竞态。

例如审批人批准“以 100 元购买商品 A”，十分钟后执行时价格涨到 500 元；或者审批后订单已经被退款。恢复时必须重新读取关键资源版本和状态。变化超过批准范围时，原审批失效。

### 人也会犯错

人工也会犯错。高风险场景可以增加：

- **职责分离**：提出者不能批准自己的动作；
- **双人审批**：两位有权限的人分别确认；
- **专业复核**：法律、医疗、财务由相应专业人员判断；
- **金额与范围上限**：即使批准也不能无限扩大；
- **抽样审计**：复查审批质量和误操作。

## 工程实践与边界

### 降低审批疲劳

- 风险分级，不把低风险动作全部推给人；
- 合并同类、边界明确的小动作，但显示总范围和上限；
- 提供关键差异和证据，不让审批人自己翻完整日志；
- 优先修复经常触发的无价值误报；
- 监控审批量、等待时间、通过率、拒绝率和撤销率；
- 不使用默认选中的“同意”按钮或诱导性界面。

高通过率未必代表规则准确，也可能代表审批人已经机械点击，需要结合复核结果判断。

### 拒绝后怎么处理

拒绝不是把 Agent 放回原步骤，让它换一种说法继续请求。系统应保存拒绝原因和动作指纹，限制同一动作重复提交。只有关键事实确实变化或用户提供新信息时，才允许生成新快照再次申请。

### 常见失败场景及处理

| 失败场景 | 后果 | 处理方式 |
| --- | --- | --- |
| 审批后参数被修改 | 人批的不是实际动作 | 动作哈希变化即失效 |
| 审批等待期间价格变化 | 实际影响超出批准范围 | 恢复时校验资源版本，重新审批 |
| 审批人没有对应权限 | 门禁形同虚设 | 审批时和执行时都校验角色 |
| 大量低风险请求 | 审批疲劳 | 风险分级、抽样和规则优化 |
| 拒绝后反复重提 | 诱导绕过人审 | 保存拒绝指纹、限制重提 |
| 审批超时后自动执行 | 高风险动作失去门禁 | 保持暂停、升级或取消 |
| 执行成功但回复超时 | 可能重复操作 | 幂等键、外部状态对账 |

### 审计和补偿

审批记录、工具执行记录和最终结果应分开保存，才能回答：谁批准了什么、系统实际执行了什么、外部结果是什么。不要通过删除日志来“撤销”动作；业务上无法直接撤销时，应执行明确补偿，例如冲正退款或发送更正通知，并保留完整审计链。

## 常见误区

- **“有人点击就安全”**：信息不完整、按钮含糊和审批疲劳都会让门禁失效。
- **“审批计划等于审批动作”**：最终工具和参数变化后必须重新审批。
- **“只有模型低置信时才需要人审”**：高置信但高影响的操作仍然需要固定门禁。
- **“让所有动作都审批最稳妥”**：过量审批会降低效率并制造机械点击。
- **“人工审批可以替代权限控制”**：最小权限、参数校验、幂等和审计仍然必须存在。
- **“审批超时可以默认同意”**：高风险动作应保持暂停、升级或取消。

## 面试追问

> **面试官：哪些动作需要人工审批？**
>
> **候选人：** 重点看影响、可逆性、数据敏感度、置信度和异常度。资金、不可逆删除、对外发布、生产权限和敏感数据操作通常需要执行前审批。

> **面试官：审批后 Agent 改了收件人怎么办？**
>
> **候选人：** 收件人是关键参数，动作哈希变化后原审批立即失效。系统应展示新的最终参数并重新审批，不能只信模型说修改很小。

> **面试官：为什么审批恢复时还要重新校验？**
>
> **候选人：** 审批等待期间，权限、价格、资源状态和代码版本都可能变化。恢复时要防 TOCTOU，确认实际动作仍在原批准范围内。

> **面试官：如何降低审批疲劳？**
>
> **候选人：** 按风险分级，低风险自动化或抽样；审批界面突出差异、证据和真实后果；持续分析高频误报并优化规则，同时保留高风险硬门禁。

> **面试官：人工长时间不响应怎么办？**
>
> **候选人：** 任务持久化暂停，设置到期、提醒和升级路径；到期后取消或继续等待，不能自动执行高风险动作。

> **面试官：审批能否替代幂等？**
>
> **候选人：** 不能。审批解决“是否允许执行”，幂等解决“同一动作重试是否重复生效”，两者处理的是不同风险。
