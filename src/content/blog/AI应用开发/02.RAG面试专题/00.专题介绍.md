很多人第一次接触 RAG，会同时遇到 Chunk、Embedding、向量数据库、召回、重排等十几个新词。每个词单独看似乎都能理解，连成一套系统后却容易发懵：

- 为什么大模型已经学过很多知识，还要去检索文档？
- 文档为什么不能整篇塞给大模型，而要先切成很多块？
- Embedding 为什么能找到“意思相近”的内容？
- 检索到了正确文档，为什么最终答案仍然可能出错？
- 面试官问“你做过什么优化”时，应该从哪一层回答？

这套专题就是为解决这些问题而写。它不要求你提前学过机器学习，也不会只给一段需要死记硬背的结论。每篇都会先给出一段可以直接用于面试的“60 秒回答”，再用同一个业务案例解释原理、工程取舍、常见故障、评测指标和追问方式。

本专题共 26 道题。按顺序读完后，你不仅应该能说出 RAG 是什么，还应该能够从数据进入知识库开始，一直讲到查询、召回、重排、生成、评测、更新、安全与线上运维。

## 先用一句话认识 RAG

RAG 是 Retrieval-Augmented Generation 的缩写，中文通常译为“检索增强生成”。

先不要被英文吓到，它做的事情很像一场允许查资料的开卷考试：

1. 用户提出问题；
2. 系统先从可信资料中寻找与问题有关的内容；
3. 再把问题和找到的资料一起交给大模型；
4. 大模型依据这些资料组织答案。

可以把它记成：

> **先查资料，再组织答案。**

大模型负责理解和表达，检索系统负责把本次回答需要的外部知识找回来。两者组合，才构成完整的 RAG 链路。

## 贯穿全系列的案例

为了避免每篇都换一个陌生业务，本专题主要使用“企业员工制度助手”作为贯穿案例。

公司有员工手册、报销制度、请假制度、差旅标准和各部门补充通知。员工可能会问：

> “我去上海出差，住宿费每晚最多能报多少？”

一个可用的 RAG 系统需要完成以下工作：

- 预先收集并解析公司的制度文件；
- 识别文件版本、适用部门、生效时间和访问权限；
- 把长文档切成适合检索的片段；
- 为片段建立关键词索引和向量索引；
- 收到问题后确认提问者身份与部门；
- 找到当前仍然生效的差旅规定；
- 把相关证据交给大模型；
- 生成答案，并给出来源和适用条件；
- 证据不足或制度冲突时说明不确定，而不是编造金额。

后面的 Chunking、Embedding、向量数据库、Query Rewrite、多路召回、重排和评测，都是在解决这条链路中的具体问题。

## 零基础术语表

下面这些词会在专题中反复出现。第一次阅读只需要建立直觉，不必立刻背下所有定义。

| 术语 | 中文直觉 | 在员工制度案例中的含义 |
| --- | --- | --- |
| LLM | 大语言模型 | 读取问题和证据，并组织自然语言答案 |
| RAG | 检索增强生成 | 先查制度，再依据制度回答 |
| 知识库 | 允许系统查询的一组资料 | 员工手册、报销制度和通知 |
| 文档 | 一份完整资料 | 一份《2026 年差旅管理办法》 |
| Corpus | 语料库或文档集合 | 公司全部可检索制度 |
| Parser | 解析器 | 从 PDF、Word 或网页中提取标题、正文和表格 |
| Chunk | 文档切分后的可检索片段 | “上海住宿标准”所在的一小段 |
| Token | 模型处理文本时使用的基本单位 | 中文字符、词片段或标点经过分词后的单位 |
| Query | 查询 | 用户输入的“上海出差能报多少住宿费” |
| Embedding | 向量表示 | 把问题和制度片段变成可比较的数字坐标 |
| Vector | 向量 | 一串用于表达语义特征的数字 |
| Index | 索引 | 帮助系统更快找到候选片段的数据结构 |
| Retrieval | 检索或召回 | 从大量片段中找出可能相关的候选 |
| Top-K | 取分数最高的 K 个结果 | 先找回最相关的 20 个片段 |
| Rerank | 重排 | 用更精细的模型把 20 个候选重新排序 |
| Context | 上下文 | 最终提供给大模型的制度证据 |
| Grounding | 有依据地回答 | 让答案受到已检索证据约束 |
| Hallucination | 幻觉 | 模型生成了资料中没有依据的内容 |
| Metadata | 元数据 | 文档版本、部门、时间、来源、权限等信息 |
| ACL | 访问控制列表 | 规定哪些用户可以检索哪些文档 |
| Offline | 离线链路 | 提前处理文档、生成向量并构建索引 |
| Online | 在线链路 | 用户提问后实时检索并生成答案 |
| Recall | 召回 | 正确证据有没有进入候选集合 |
| Precision | 精确 | 召回结果中有多少真正相关 |
| Latency | 延迟 | 从用户提问到系统返回答案所需的时间 |

## 建立一张完整心智地图

学习 RAG 时，最容易犯的错误是只盯着向量数据库。实际上，一套生产 RAG 至少包含六层：

| 层次 | 核心问题 | 常见故障 |
| --- | --- | --- |
| 数据源 | 权威资料在哪里 | 文件缺失、版本过期、来源不可信 |
| 数据处理 | 怎样把资料变成可检索内容 | PDF 解析错乱、表格丢失、语义被切断 |
| 索引 | 怎样保存并快速找到内容 | 向量不兼容、索引未更新、过滤失效 |
| 检索 | 怎样找到充分且相关的证据 | 正确证据没召回、候选重复、排序错误 |
| 生成 | 怎样依据证据回答 | 忽略证据、错误归纳、引用错位 |
| 治理与评测 | 怎样知道系统可靠 | 无评测集、无法追踪、权限泄露、不能回滚 |

定位问题时也要按层次检查。最终答案错误，不等于“大模型不够强”；也可能是原始文档错误、解析失败、切块不合理、权限过滤过早删除了证据、召回漏掉了答案，或者重排把正确片段排到了后面。

## 学习路线

### 第一阶段：建立基本边界

先学习 01–03，弄清三个问题：

- RAG 的完整流程是什么；
- 它具体解决哪些问题，又不能解决哪些问题；
- RAG、提示词和微调应该怎样分工。

完成这一阶段后，你应该能够用“开卷考试”类比讲清 RAG，而不是只背一句“减少幻觉”。

### 第二阶段：理解知识库怎样建成

学习 04–09，重点掌握：

- 原始文档、Chunk、向量和元数据分别怎样保存；
- Chunk 为什么没有通用固定长度；
- 怎样缓解语义被切断；
- Embedding 在数学上和工程上分别是什么；
- 向量数据库如何索引、过滤、更新和压测。

完成这一阶段后，你应该能够从一份 PDF 开始，讲出它怎样变成可检索的知识。

### 第三阶段：理解在线检索怎样优化

学习 10–16，重点掌握：

- 用户问题怎样流过在线链路；
- 关键词检索和向量检索各自擅长什么；
- 为什么需要 Query Rewrite、多路召回、融合和重排；
- 复杂问题为什么可能需要查询分解、迭代检索或图检索。

完成这一阶段后，你应该能够针对“搜不到”“搜不准”“资料冲突”分别提出方案。

### 第四阶段：掌握生产治理

学习 17–20，重点掌握：

- 怎样降低没有证据支持的回答；
- 怎样分层评测检索与生成；
- 文档更新、删除和索引发布怎样保持一致；
- 线上 RAG 最常见的真实难点是什么。

完成这一阶段后，你应该能够回答监控、灰度、回滚和故障定位问题。

### 第五阶段：准备进阶追问

学习 21–26，理解 Agentic RAG、多模态文档、权限与提示注入、Late Interaction、GraphRAG 和长上下文选型。这些题不适合只背定义，关键是说清新增收益是否值得新增成本。

## 每篇文章应该怎样学

建议用四遍学习法：

1. **第一遍只读“先说人话”**：先建立直觉，不纠结公式和产品参数。
2. **第二遍顺着案例走一遍**：尝试说明每个步骤的输入、输出和失败方式。
3. **第三遍背“60 秒回答”**：背结构和关键词，不需要逐字背诵。
4. **第四遍回答“面试追问”**：遮住答案，用自己的语言说出依据与边界。

如果你没有真实 RAG 项目，也不要虚构数据。可以诚实地说：

> “我没有在这个规模下做过生产验证，但我会固定数据、查询、过滤条件和硬件，对比召回率、延迟、吞吐、成本与恢复能力，再依据业务 SLO 选型。”

这种回答比随口编造 QPS 或准确率更可信。

## 面试答题通用模板

遇到 RAG 工程题，可以按以下顺序组织：

1. **先下定义**：一句话说明讨论对象是什么。
2. **再讲目标**：它解决的是哪类问题。
3. **拆解流程**：输入、关键步骤和输出分别是什么。
4. **说明选型**：依据数据、负载、质量和风险做取舍。
5. **补充失败场景**：指出方案在哪些情况下会失效。
6. **给出评测指标**：说明怎样证明改动有效。
7. **落到边界**：权限、时效、成本、延迟和回滚怎样处理。

面试官真正关心的通常不是你能说出多少名词，而是你能否把一个指标变化追溯到具体环节。
