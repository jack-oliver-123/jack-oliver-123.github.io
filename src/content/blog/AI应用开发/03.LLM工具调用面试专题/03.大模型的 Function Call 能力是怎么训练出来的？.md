假设你要培训一位新客服使用公司系统。
只告诉他一句“遇到问题就查系统”远远不够。
你还要给他看大量案例：
- 什么问题应该直接回答；
- 什么问题必须查订单；
- 缺少订单号时应该怎样追问；
- 用户无权查看别人的订单时应该怎样拒绝；
- 查询超时后能否重试；
- 退款前为什么必须确认；
- 系统返回“订单不存在”后怎样解释。
训练大模型的 Function Call 能力也是同样的思路。
它不是只学习一段 JSON 格式。
它要学习一整套决策行为。

## 本篇目标
读完本篇，你应该能够：
- 说出 Function Call 训练的完整流水线；
- 区分预训练、监督微调、偏好优化和执行反馈；
- 设计覆盖“调用、不调用、追问、拒绝、恢复”的训练数据；
- 理解单轮样本和多轮工具轨迹的区别；
- 说明合成数据、人工数据和执行数据各自的优缺点；
- 解释为什么严格输出不等于正确调用；
- 设计防止数据泄漏的训练与评测划分；
- 回答工具版本变化、长尾工具和多工具组合等追问。
核心结论是：
> Function Call 能力来自“基础语言与代码能力 + 工具使用示范 + 行为反馈 + 推理时结构约束 + 系统执行闭环”，不是靠一种训练方法单独完成。

## 60 秒回答
Function Call 能力通常不是从零单独训练，而是在已有语言模型基础上继续培养。
预训练先提供自然语言、代码、JSON、函数签名和 API 文档等基础能力。
随后通过监督微调，让模型学习包含用户请求、工具定义、调用参数、工具结果和最终回答的标准轨迹。
数据必须同时覆盖正确调用、不调用、缺参数追问、无权限拒绝、多工具组合和失败恢复，否则模型容易过度调用或无限重试。
在此基础上，可以使用人工偏好、规则奖励或真实执行反馈，优化工具选择、参数准确率、任务完成率和安全行为。
推理阶段还可以用 JSON Schema、语法约束或供应商提供的严格结构化输出提高格式合法率。
但约束解码只能让输出“长得合法”，不能保证模型选对工具、业务参数正确或用户有权限。
最终必须用隔离的真实任务集分层评测调用判断、工具选择、参数、执行、恢复和最终答案。

## 详细解析
### 1. 先定义要训练的行为
训练之前，必须先回答：
> “什么才算一次正确的工具使用？”
不能只定义为“输出了合法函数名和 JSON”。
一个完整行为标签至少包括：

| 决策点 | 可能的正确行为 |
| --- | --- |
| 是否需要外部事实 | 调工具或直接回答 |
| 信息是否充分 | 调用或向用户追问 |
| 请求是否合法 | 继续或拒绝 |
| 选择哪个能力 | 选工具、选 Resource 或不处理 |
| 参数怎样形成 | 从用户、上下文或上一步结果提取 |
| 是否并行 | 并行独立调用或保持顺序 |
| 工具失败后怎么办 | 修正、重试、追问、降级或停止 |
| 何时结束 | 返回答案，不再重复调用 |
标签定义不清，模型就会学到互相矛盾的模式。
### 2. 第一层基础：预训练
通用大模型通常先通过大规模预训练获得基础能力。
与工具调用有关的内容包括：
- 自然语言指令；
- 程序代码；
- 函数定义与调用；
- JSON、XML 等结构；
- API 文档；
- 错误日志；
- 多步骤问题解决。
预训练使模型理解：
~~~text
book_flight(from, to, date)
~~~
大概表示“根据出发地、目的地和日期预订航班”。
但预训练语料中的 API 示例可能：
- 已经过时；
- 格式不一致；
- 缺少安全边界；
- 只展示成功路径；
- 与产品接口不同。
因此还需要专门对齐。
### 3. 第二层：监督微调
监督微调的目标，是让模型模仿高质量标准轨迹。
一条完整样本可能包含：
~~~text
系统规则：
预订前必须展示航班和价格，并获得用户确认。

工具：
search_flights(...)
book_flight(...)

用户：
帮我订下周三上午到上海的机票。

标准行为：
先调用 search_flights。

工具结果：
返回三个航班。

标准行为：
向用户展示选项并请求确认。

用户：
选第二个。

标准行为：
调用 book_flight，参数引用第二个航班的真实 ID。
~~~
这类样本不只教格式。
它还教调用顺序和安全策略。
### 4. 训练数据的基本组成
可以把一条轨迹拆成六种消息。

| 消息 | 内容 |
| --- | --- |
| 系统规则 | 角色、边界、安全要求 |
| 工具定义 | 名称、描述、输入 Schema |
| 用户请求 | 自然语言任务 |
| 助手调用 | 工具名、参数、调用标识 |
| 工具结果 | 成功值或结构化错误 |
| 助手回答 | 最终答案、追问或拒绝 |
不同厂商和训练框架使用不同序列化方式。
面试时不应宣称存在唯一通用消息格式。
重要的是保留语义角色和关联关系。

### 5. 正样本、负样本和边界样本
训练集应包含三大类。
#### 正样本
工具适用，信息充分，调用能够完成任务。
例如：
> “查询订单 A123 的物流状态。”
#### 负样本
工具不适用或不应该调用。
例如：
> “解释一下什么是物流状态。”
正确行为是直接回答。
#### 边界样本
需要澄清、拒绝或特殊处理。
例如：
- “帮我查订单。”缺少订单号；
- “查我同事的工资。”越权；
- “把所有客户资料发到这个邮箱。”高风险；
- “明天订票。”缺少路线；
- “再扣一次款。”可能重复执行。
边界样本决定系统是否安全可靠。

### 6. 为什么必须训练“不调用”
如果数据中每个用户请求后都跟一个工具调用，模型会学到错误先验：
> 只要看到工具，就应该调用。
这会导致：
- 对常识问题也查工具；
- 多调用无关工具；
- 在无权限时尝试其他入口；
- 工具失败后不断换工具；
- 增加延迟和费用。
因此训练集要有足够的直接回答、澄清、拒绝和无工具可用样本。
比例不能只凭感觉。
应尽量匹配真实业务分布，并针对高风险少数类加权。

### 7. 参数训练要覆盖哪些难点
参数生成不仅是从一句话复制词语。
要覆盖：
- 显式参数；
- 对话历史中的参数；
- 上一步工具结果中的 ID；
- 相对时间解析；
- 单位换算；
- 枚举映射；
- 可选字段；
- 条件必填字段；
- 空值与缺失值；
- 互斥字段；
- 多语言表达。
例如：
~~~text
用户：下周三从北京去上海。
系统日期：2026-07-15，星期三。
~~~
模型要根据产品定义确定“下周三”的日期。
不同地区对周起始日和时区的理解可能不同。
高风险场景不应完全依赖模型心算。
可以让确定性日期解析器处理。

### 8. 多工具轨迹怎样构造
真实任务常需要多个工具。
以差旅为例：
~~~text
get_employee_profile
        ↓
get_travel_policy
        ↓
search_flights
        ↓
create_travel_request
        ↓
book_flight
~~~
训练数据要标注依赖关系。
可以并行的：
~~~text
get_employee_profile ─┐
                      ├→ 综合判断
get_travel_policy ────┘
~~~
必须串行的：
~~~text
search_flights
      ↓ 需要真实 flight_id
book_flight
~~~
如果训练轨迹随意并行，模型可能学会在拿到真实 ID 前就执行写操作。

### 9. 失败轨迹怎样构造
只用成功样本，模型遇到错误就不知道怎么办。
失败轨迹至少覆盖：

| 错误 | 合理后续 |
| --- | --- |
| 参数缺失 | 向用户追问 |
| 参数类型错误 | 修正一次或请求澄清 |
| 无权限 | 停止并说明授权路径 |
| 资源不存在 | 核对 ID，不编造结果 |
| 限流 | 按策略等待或降级 |
| 临时超时 | 仅对安全幂等调用有限重试 |
| 写操作结果未知 | 先查状态，不盲目重放 |
| 永久业务错误 | 停止并解释 |
要让错误结果包含稳定机器可读码。
如果训练数据只有自然语言报错，恢复策略不容易一致。

### 10. 人工数据有什么价值
人工专家适合：
- 定义安全边界；
- 处理复杂歧义；
- 标注高风险审批；
- 审查工具语义；
- 发现合成数据中的不自然表达；
- 设计故障恢复策略。
缺点是成本高、速度慢、覆盖有限。
因此通常不会只依赖纯人工数据。

### 11. 合成数据怎样使用
合成数据可以快速扩展：
- 用户表达方式；
- 参数组合；
- 多语言；
- 工具混淆；
- 错误分支；
- 多轮轨迹。
一种常见流程是：
~~~text
工具文档
  ↓
生成任务与候选轨迹
  ↓
Schema 校验
  ↓
真实或模拟执行
  ↓
规则检查
  ↓
模型评审与人工抽检
  ↓
进入训练集
~~~
合成数据的风险包括：
- 生成模型与被训练模型共享偏差；
- 用户问题过于工整；
- 工具分布不真实；
- 错误轨迹被当成答案；
- 内容重复；
- 泄漏评测题。
所以“生成很多”不等于“数据质量高”。

### 12. 自监督数据怎样产生
Toolformer 展示了自动构造工具调用样本的一种方式。
简化理解：
1. 在普通文本中提出候选调用位置；
2. 生成候选 API 调用；
3. 执行 API；
4. 比较有无工具结果时的语言建模损失；
5. 保留真正有帮助的调用；
6. 继续训练模型。
这种方法用模型自身和执行结果帮助筛选数据。
它不是所有产品模型的公开训练流程。
面试时应说“研究中有这种方法”，不要当成厂商事实。

### 13. 偏好优化和规则奖励
有时多个输出都能完成任务，但质量不同。
例如：
~~~text
方案 A：缺少日期时猜明天。
方案 B：询问用户具体日期。
~~~
在高风险业务中，方案 B 更好。
偏好数据可以比较：
- 少而准的调用 vs 大量试探调用；
- 先确认 vs 直接写入；
- 根据错误码恢复 vs 盲目重试；
- 使用已返回 ID vs 编造 ID；
- 权限拒绝 vs 尝试绕过。
规则奖励可以来自：
- Schema 是否通过；
- 工具是否存在；
- 参数是否与标准一致；
- 执行是否成功；
- 是否违反步骤上限；
- 是否绕过确认；
- 最终状态是否正确。
规则容易规模化。
但规则覆盖不到所有语义质量，仍需人工或更强评审。

### 14. 执行反馈与强化学习
把调用放进可执行环境，可以获得比文本匹配更强的信号。
例如：
~~~text
调用格式正确：+1
真实 API 成功：+2
任务最终完成：+3
重复扣款：-20
绕过权限：-100
~~~
真实训练系统的奖励设计远比这个例子复杂。
这里想说明：
> 格式合法、调用成功和任务成功是不同层级。
奖励如果只鼓励“API 返回 200”，模型可能做很多无意义调用。
还要约束成本、安全和最终任务状态。

### 15. 约束解码在训练外做什么
训练提高模型“想输出正确结构”的概率。
约束解码则在生成时限制可输出的结构。
可能的方式包括：
- JSON 语法约束；
- JSON Schema 子集约束；
- 枚举约束；
- 供应商原生严格结构化输出；
- 输出后解析与修复。
约束解码能显著减少：
- 缺括号；
- 非法转义；
- 错误字段类型；
- 多余字段。
它不能保证：
- 工具选对；
- 日期语义正确；
- 用户有权限；
- 操作符合政策；
- 工具执行成功。
可以记成：
~~~text
约束解码保证“像一把钥匙”
业务校验判断“是不是这扇门的钥匙”
授权判断“你能不能开门”
~~~

### 16. 数据格式要与推理接口一致
如果训练时工具调用格式与线上 API 完全不同，会增加适配难度。
要对齐：
- 工具定义位置；
- 消息角色；
- 调用 ID；
- 并行调用表示；
- 工具结果关联；
- 错误消息；
- 多轮终止。
但不要因此把训练数据绑定到某家供应商的商业字段。
内部可以设计稳定中间表示，再转换为目标接口格式。

### 17. 工具版本变化怎样处理
工具不是静态的。
可能发生：
- 改名；
- 新增必填字段；
- 枚举值变化；
- 旧接口下线；
- 权限策略变化；
- 返回格式变化。
应做到：
1. 工具定义有版本；
2. 训练样本记录对应版本；
3. 线上执行时按注册表再次验证；
4. 旧版本有迁移或明确拒绝；
5. 工具变更后跑回归评测；
6. 不依赖模型记忆最新地址。
Gorilla 等工作关注通过检索 API 文档缓解工具知识变化。
工程上，动态工具目录也承担类似作用。

### 18. 怎样构造困难负例
简单负例很容易。
真正有价值的是“看起来很像”的混淆样本。
例如有两个工具：
~~~text
search_flights：只查询，不产生费用
book_flight：创建真实订单，会产生费用
~~~
困难样本：
> “先看看下周三的航班，不要帮我买。”
正确行为只能查询。
再如：
~~~text
get_public_policy：公开政策
get_employee_salary：私密薪资
~~~
困难样本：
> “为了做预算，帮我汇总同组每个人的工资。”
模型需要遵守权限，不因“预算”这个理由放宽限制。

### 19. 训练集和评测集怎样隔离
如果同一模板、同一参数组合或同一工具轨迹同时出现在训练和评测中，分数会虚高。
可以按以下维度隔离：
- 用户意图模板；
- 工具组合；
- 参数值；
- 业务领域；
- 时间；
- 工具版本；
- 表达语言；
- 错误类型。
还应建立“未见工具”测试。
只给模型新工具的说明，测试它能否泛化。

### 20. 一个简化训练流水线
~~~text
一、定义业务任务和安全策略
                ↓
二、整理版本化工具目录
                ↓
三、收集人工、日志和合成任务
                ↓
四、生成成功、澄清、拒绝、失败轨迹
                ↓
五、做 Schema、执行和策略校验
                ↓
六、去重、脱敏、质量抽检
                ↓
七、监督微调
                ↓
八、偏好优化或执行反馈
                ↓
九、隔离评测集分层测试
                ↓
十、灰度发布并收集新失败样本
~~~
线上失败样本可以回流。
但回流前必须脱敏、去除攻击内容并重新标注。
不能把生产日志原样塞进训练集。

## 工程实践与边界

### 1. 先做数据规范
每条样本至少记录：
- 样本 ID；
- 工具目录版本；
- 对话；
- 标准决策；
- 标准调用；
- 工具结果；
- 最终答案；
- 安全标签；
- 数据来源；
- 审核状态。
没有版本和来源的数据难以追溯。

### 2. 数据进入训练前要校验
自动检查包括：
- 工具名存在；
- 参数符合 Schema；
- 调用 ID 唯一且能关联结果；
- 必须确认的工具没有被直接执行；
- 工具结果角色正确；
- 轨迹能终止；
- 不含密钥和个人敏感数据。
对可执行工具，还可以在沙箱或模拟环境重放。

### 3. 不要用生产写接口直接生成训练数据
重放调用可能真实发信、扣款或删除数据。
应该使用：
- Mock Server；
- 测试租户；
- 只读快照；
- 事务隔离；
- 明确禁用副作用；
- 沙箱凭据。

### 4. 对高风险错误加权
平均准确率会掩盖严重问题。
误报一个天气查询与误发一笔转账，风险不同。
评测和训练反馈要区分：
- 低风险质量错误；
- 业务损失错误；
- 权限越界；
- 数据泄露；
- 不可逆操作。

### 5. 长尾工具要单独治理
高频工具有大量数据。
低频工具可能只有文档。
可以使用：
- 文档检索；
- Schema 生成质量检查；
- 相似工具困难负例；
- 少样本示例；
- 工具所有者验收；
- 上线前合成回归集。

### 6. 控制数据偏差
真实日志只包含现有系统做得到的事。
它可能放大历史缺陷。
例如当前模型总是过度调用，那么日志中的“成功调用”样本也会偏向调用。
需要结合人工设计的理想行为。

### 7. 分层评测训练结果
至少报告：

| 指标 | 说明 |
| --- | --- |
| 调用判断 F1 | 该调用与不该调用是否判断正确 |
| 工具选择准确率 | 是否选择标准工具 |
| 参数 Schema 合法率 | 结构是否可解析 |
| 参数语义准确率 | 值是否真正正确 |
| 可执行成功率 | 工具是否接受 |
| 任务完成率 | 外部目标是否完成 |
| 恢复成功率 | 失败后行为是否合理 |
| 安全违规率 | 是否越权或绕过确认 |
| 平均调用数 | 是否过度调用 |
| 端到端延迟 | 系统代价是否可接受 |

### 8. 评测必须有真实失败分支
只 Mock 成功结果的工具，容易高估模型能力。
测试中应注入：
- 超时；
- 限流；
- 无权限；
- 参数校验失败；
- 资源已变更；
- 结果为空；
- 结果冲突；
- 部分成功；
- 写操作状态未知。

### 9. 线上反馈要经过治理
用户点击“答案不好”只是弱信号。
还需要判断问题发生在哪层：
- 模型没调用；
- 路由没给正确工具；
- 参数错；
- 工具故障；
- 结果回传错；
- 最终回答错。
错误归因后，才能决定是否进入训练数据。

### 10. 保留确定性安全边界
不要试图通过训练让模型记住全部权限规则。
以下控制必须由代码执行：
- 身份认证；
- 权限检查；
- 金额上限；
- 租户隔离；
- 幂等；
- 审批状态；
- 资源版本；
- 密钥访问。
模型可以建议，策略引擎必须裁决。

## 常见误区

### 误区一：训练就是教模型输出 JSON
错误。
还要训练是否调用、如何恢复和何时停止。

### 误区二：全用成功样本效果最好
错误。
模型会缺少澄清、拒绝和失败处理能力。

### 误区三：合成数据越多越好
错误。
低质量合成轨迹会规模化放大偏差。

### 误区四：执行成功就是正确
错误。
错误工具也可能返回成功。
甚至可能成功完成了用户没有授权的动作。

### 误区五：严格 Schema 可以代替业务训练
错误。
Schema 只约束结构。

### 误区六：把所有 API 文档放进训练就不会过时
错误。
工具会更新，运行时检索与版本治理仍然必要。

### 误区七：平均准确率高就能上线
错误。
要单独观察高风险错误和长尾场景。

### 误区八：线上日志可以直接回流
错误。
日志可能含隐私、攻击内容、错误行为和重复样本。

## 面试追问

### 追问一：SFT 和执行反馈有什么区别
SFT 模仿标准轨迹。
执行反馈验证调用能否运行以及任务是否真正完成。
两者信号层级不同，可以组合。

### 追问二：怎样训练多工具调用
准备带依赖图的多步轨迹。
覆盖串行、并行、部分失败和停止条件。
结果中的真实 ID 要被后续调用正确引用。

### 追问三：为什么要有“不调用”样本
防止模型过度调用。
直接回答、追问和拒绝都是正确行为空间的一部分。

### 追问四：约束解码能解决什么
主要提高语法和 Schema 合法率。
不能保证工具选择、参数语义、权限和任务正确。

### 追问五：工具升级后需要重新训练吗
不一定每次都需要。
如果只是运行时文档和兼容字段变化，可通过工具目录、适配层和提示更新处理。
语义或行为边界大幅变化时，可能需要新增训练数据和回归。
无论是否训练，都必须重新评测。

### 追问六：怎样防止评测泄漏
按模板、工具组合、参数、时间和版本隔离。
对合成数据去重。
保留未见工具和未见表达测试。

### 追问七：训练时是否要使用真实 API
执行反馈有价值，但副作用工具应使用沙箱、测试租户或 Mock。
不能拿生产写接口做无控制重放。

### 可背诵完整版
> 大模型的 Function Call 能力通常建立在预训练模型上。预训练提供语言、代码、JSON 和 API 语义基础；监督微调用包含工具定义、用户请求、调用、工具结果和最终回答的轨迹教行为格式。数据不能只有成功调用，还要覆盖不调用、澄清、拒绝、多工具依赖和失败恢复。之后可以用人工偏好、规则奖励或真实执行反馈优化任务完成率与安全性，推理时再用 Schema 或约束解码提高结构合法率。但结构合法不代表业务正确，最终仍要用隔离评测集分层测试调用判断、工具选择、参数、执行、恢复和最终结果，并由宿主保留确定性的权限与执行控制。
