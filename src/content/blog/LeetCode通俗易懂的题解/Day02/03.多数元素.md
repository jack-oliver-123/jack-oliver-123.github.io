## 题目入口

题目链接：[多数元素](https://leetcode.cn/problems/majority-element/)

## 一句话读懂题目

给你一个数组 `nums`，找出其中出现次数超过 `⌊n/2⌋` 次的元素。

题目保证这个元素一定存在。

比如：

```text
nums = [3, 2, 3]
```

`3` 出现了 2 次，超过 `3/2 = 1` 次，返回：

```text
3
```

再比如：

```text
nums = [2, 2, 1, 1, 1, 2, 2]
```

`2` 出现了 4 次，超过 `7/2 = 3` 次，返回：

```text
2
```

注意"超过一半"这个条件非常强：整个数组里最多只可能有一个这样的元素。

## 先用生活例子理解

想象一场村长选举，村里有 `n` 个人投票，每人投一票。

题目告诉你：一定有一个候选人拿到了超过半数的票。

怎么快速找出这个人？

**投票对消法**：

两个支持不同候选人的村民站出来，互相"抵消"，两人同时离场。

因为赢家的票数超过一半，无论怎么抵消，最后留在场上的一定是赢家。

这就是 Boyer-Moore 投票算法的核心思想：

> 不同的元素互相抵消，剩下的就是多数元素。

## 方法一：Boyer-Moore 投票算法（推荐）

这是时间 `O(n)`、空间 `O(1)` 的最优解。

维护两个变量：

```text
candidate：当前候选人
count：候选人的票数
```

遍历数组：

```text
如果 count == 0
    换一个候选人（candidate = 当前元素）
如果当前元素 == candidate
    count++
否则
    count--（抵消一票）
```

遍历结束后，`candidate` 就是多数元素。

为什么这样做是对的？

因为多数元素的数量超过 `n/2`，即使每个多数元素都被一个非多数元素抵消掉，多数元素还是有剩余。最后活下来的一定是它。

## 手动走一遍例子

```text
nums = [2, 2, 1, 1, 1, 2, 2]
```

| 步骤 | 当前元素 | candidate | count | 操作 |
| --- | --- | --- | --- | --- |
| 初始 | - | - | `0` | - |
| 1 | `2` | `2` | `1` | count 为 0，换候选人 |
| 2 | `2` | `2` | `2` | 相同，count++ |
| 3 | `1` | `2` | `1` | 不同，count-- |
| 4 | `1` | `2` | `0` | 不同，count-- |
| 5 | `1` | `1` | `1` | count 为 0，换候选人 |
| 6 | `2` | `1` | `0` | 不同，count-- |
| 7 | `2` | `2` | `1` | count 为 0，换候选人 |

最终 `candidate = 2`，就是答案。

关键观察：

- 第 4 步 `count` 变成 0，说明前 4 个元素里 `2` 和 `1` 各出现 2 次，互相抵消干净了；
- 第 6 步 `count` 又变成 0，前面的又抵消了；
- 最后剩下的 `2` 就是多数元素。

## Java 代码

先记住一句话：

> 不同就抵消，相同就加票，最后剩下的就是多数。

### 写法一：Boyer-Moore 投票（推荐）

```java
class Solution {
    public int majorityElement(int[] nums) {
        int candidate = 0;
        int count = 0;

        for (int num : nums) {
            if (count == 0) {
                candidate = num;
            }

            if (num == candidate) {
                count++;
            } else {
                count--;
            }
        }

        return candidate;
    }
}
```

### 写法二：HashMap 计数

```java
class Solution {
    public int majorityElement(int[] nums) {
        Map<Integer, Integer> map = new HashMap<>();

        for (int num : nums) {
            map.put(num, map.getOrDefault(num, 0) + 1);

            if (map.get(num) > nums.length / 2) {
                return num;
            }
        }

        return -1;
    }
}
```

### 写法三：排序取中位数

```java
class Solution {
    public int majorityElement(int[] nums) {
        Arrays.sort(nums);
        return nums[nums.length / 2];
    }
}
```

## 代码逐行拆解（投票算法）

```java
int candidate = 0;
int count = 0;
```

`candidate` 是当前的候选人，`count` 是它的净票数。

初始 `count = 0` 表示还没有候选人。

```java
for (int num : nums) {
```

遍历数组中的每一个元素（每一张选票）。

```java
if (count == 0) {
    candidate = num;
}
```

如果当前净票数为 0，说明之前的候选人被完全抵消了。

换当前元素作为新的候选人。

```java
if (num == candidate) {
    count++;
} else {
    count--;
}
```

如果当前元素和候选人相同，票数加一；不同则抵消一票。

```java
return candidate;
```

遍历结束，`candidate` 就是多数元素。

题目保证多数元素一定存在，所以不需要二次验证。

## 方法二详解：排序取中位数

如果把数组排序，多数元素出现超过 `n/2` 次，它一定会覆盖数组的中间位置。

无论多数元素是最小值、最大值还是中间值：

```text
[1, 1, 1, 2, 3]   -> 中间是 1 ✓
[1, 2, 2, 2, 2]   -> 中间是 2 ✓
[1, 2, 2, 2, 3]   -> 中间是 2 ✓
```

所以排序后直接返回 `nums[n/2]` 就行。

这个方法代码最短，但时间复杂度是 `O(n log n)`。

## 三种方法怎么选

| | 投票算法 | HashMap | 排序 |
| --- | --- | --- | --- |
| 时间复杂度 | `O(n)` | `O(n)` | `O(n log n)` |
| 空间复杂度 | `O(1)` | `O(n)` | `O(log n)` |
| 代码量 | 短 | 中等 | 最短 |
| 理解难度 | 需要理解抵消 | 简单 | 简单 |

面试时推荐投票算法：时间最优、空间最优、代码不长。

排序法适合快速写出来拿分，HashMap 适合作为第二种方法补充。

## 复杂度分析

### 方法一：Boyer-Moore 投票

时间复杂度：`O(n)`。

遍历一次数组。

空间复杂度：`O(1)`。

只用了两个变量。

### 方法二：HashMap

时间复杂度：`O(n)`。

遍历一次数组，每次 `put` 和 `get` 都是 `O(1)`。

空间复杂度：`O(n)`。

最坏情况 HashMap 存了 `n` 个不同的 key。

### 方法三：排序

时间复杂度：`O(n log n)`。

排序决定复杂度。

空间复杂度：`O(log n)`。

Java `Arrays.sort` 的递归栈开销。

## 常见坑

### 1. 以为需要二次验证

有些版本的题目不保证多数元素一定存在，那种情况需要第二次遍历确认 `candidate` 的出现次数是否真的超过 `n/2`。

但 LeetCode 169 这道题明确说了"多数元素一定存在"，所以不需要验证，直接返回。

### 2. count == 0 时忘记更新 candidate

如果 `count` 变成 0 后不换候选人，下一个元素的比较就会出错。

`count == 0` 意味着"之前选的人被完全抵消了，得重新选一个"。

### 3. 混淆"出现次数最多"和"出现超过一半"

这道题不是找"出现次数最多的元素"（众数），而是找"出现次数超过 n/2 的元素"。

投票算法只在"超过一半"的条件下保证正确。如果只是"出现最多但不到一半"，投票算法可能给出错误答案。

### 4. 排序法取错下标

排序后应该取 `nums[nums.length / 2]`，不是 `nums[(nums.length - 1) / 2]`。

比如长度为 5 的数组，`5/2 = 2`，取下标 2（第 3 个元素）。

## 面试时怎么讲

### 30 秒版

> 用 Boyer-Moore 投票算法。维护一个候选人和计数器，遍历数组：计数为 0 时换候选人，当前元素和候选人相同就加一，不同就减一。因为多数元素超过半数，抵消完后一定是它存活。时间 `O(n)`，空间 `O(1)`。

### 面试官可能追问

**为什么投票算法是正确的？**

多数元素出现超过 `n/2` 次，其他所有元素加起来不到 `n/2` 次。

每次抵消都是一个多数元素和一个非多数元素配对消掉（最坏情况）。即使每次都消耗一个多数元素，它的数量也比其他所有元素的总和多，所以最后一定剩下。

**如果不保证多数元素存在呢？**

需要第二次遍历验证。投票结束后再数一遍 `candidate` 的出现次数，如果不超过 `n/2` 就返回 -1 或抛异常。

**能不能用分治法？**

可以。把数组分成两半，分别求左半边和右半边的多数元素。如果相同就直接返回，不同就数一下两个候选人在整个区间里的出现次数，取多的那个。时间 `O(n log n)`，空间 `O(log n)`。

**投票算法能推广到出现超过 n/3 的元素吗？**

可以。出现超过 `n/3` 的元素最多有 2 个，维护两个候选人和两个计数器就行。这是 LeetCode 229 题。

**为什么排序后取中间一定对？**

多数元素出现超过 `n/2` 次。排序后，它在数组中连续占据超过一半的位置。无论它从哪里开始，一定会覆盖到 `n/2` 这个中间下标。

## 最后总结

多数元素这道题，核心洞察是：

> 出现超过一半的元素，无论怎么和其他元素抵消，都不可能被完全消灭。

投票算法模板：

```text
candidate = 0, count = 0
遍历数组
    count 为 0 -> 换候选人
    和候选人相同 -> count++
    和候选人不同 -> count--
返回 candidate
```

刷题时可以用下面几个边界用例自检：

```text
nums = [3, 2, 3] -> 3
nums = [2, 2, 1, 1, 1, 2, 2] -> 2
nums = [1] -> 1（单元素）
nums = [1, 1, 1, 1] -> 1（全相同）
nums = [6, 5, 5] -> 5
```

最后把这句话记住：

> 超过半数就是绝对优势。不同就抵消，抵消到最后，站着的那个就是赢家。
