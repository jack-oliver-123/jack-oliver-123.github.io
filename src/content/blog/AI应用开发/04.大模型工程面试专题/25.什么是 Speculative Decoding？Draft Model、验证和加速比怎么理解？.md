假设一位资深编辑正在逐字写一篇文章。

他判断准确，但每写一个字都要重新翻阅大量资料，所以速度不快。

现在安排一位速度很快的助理先草拟几个字，再让资深编辑一次检查这一小段：

- 草稿正确，就整段接受；
- 某个位置错误，就从那里修正；
- 然后继续下一段。

如果助理经常猜对，资深编辑一次检查就能确认多个字，整体速度会提高。

如果助理总是猜错，检查和草拟反而增加额外工作。

Speculative Decoding，中文常译为**推测解码**，就是这个思路。

## 本篇目标

学完本篇，你应该能够：

1. 解释大模型逐 Token 解码为什么容易受内存带宽限制；
2. 说清 Draft Model 和 Target Model 各自负责什么；
3. 描述“草拟—并行验证—接受或修正”的完整流程；
4. 逐个符号解释经典推测采样的接受概率；
5. 解释为什么拒绝后不能简单从目标分布重新采样；
6. 估算接受率、草稿长度和加速比的关系；
7. 区分双模型、同模型多头和特征级推测方案；
8. 判断什么负载适合推测解码，什么负载可能不值得；
9. 说明“无损分布保持”与工程实现限制之间的边界。

先记住一句话：

> 推测解码不是让小模型替代大模型，而是让便宜的草稿先提出多个候选，再由目标模型并行验证；能否加速主要取决于草稿接受率、草稿成本、验证效率和服务负载。

## 60 秒回答

自回归大模型通常一次只生成一个 Token，每一步都要读取大量权重。Speculative Decoding 先让较快的 Draft Model 连续生成 $\gamma$ 个草稿 Token，再让 Target Model 用一次并行前向计算得到这几个位置的概率并验证。

经典推测采样对第 $i$ 个草稿 Token 的接受概率是 $\min(1,p_i/q_i)$，其中 $p_i$ 是目标模型概率，$q_i$ 是草稿模型概率。若拒绝，不能随意重采样，而要从归一化后的正残差 $\max(0,p-q)$ 中采样。按论文算法实现时，最终输出分布与直接从目标模型采样保持一致，因此加速来自计算调度，而不是用草稿答案降低质量标准。

加速效果不只看 Draft 有多小。Draft 太弱会降低接受率，太强又会消耗过多计算。草稿长度太短，验证摊销不足；太长，后半段常因前面出错而浪费。高温采样、跨语言切换、代码长标识符、目标与 Draft Tokenizer 不兼容，都可能影响收益。

工程上我会在真实输入、输出、并发和采样参数下，对比关闭和开启推测解码的 TPOT、端到端延迟、吞吐、P95/P99、接受长度、GPU 利用率和输出一致性，并按负载动态调整 Draft 长度。它主要优化 Decode，不会消除 Prefill 成本。

## 详细解析

### 一、为什么普通解码一次只走一步

大语言模型通常按下面的方式生成：

```text
输入 Prompt
  -> 预测第 1 个新 Token
  -> 把它加入上下文
  -> 预测第 2 个新 Token
  -> 把它加入上下文
  -> ...
```

第 $t$ 个 Token 依赖前面已生成的所有 Token：

$$
p(y_t\mid x,y_1,...,y_{t-1})
$$

其中：

- $x$：输入 Prompt；
- $y_t$：第 $t$ 个新 Token；
- $y_1,...,y_{t-1}$：此前已生成 Token；
- $p$：目标模型给出的条件分布。

在不知道 $y_t$ 前，无法直接确定 $y_{t+1}$ 的真实上下文。

所以普通自回归解码具有串行依赖。

### 二、为什么 Decode 经常没有吃满计算单元

Prefill 阶段一次处理整个输入，矩阵通常较大，较容易利用 GPU 并行计算。

Decode 阶段每个序列每步只有一个新 Token。

即使有 KV Cache，模型每步仍需要读取大量权重，并对少量 Token 做计算。

在不少硬件和批量配置下，Decode 更容易受内存带宽限制，而不是纯算力限制。

可以用算术强度理解：

$$
I=\frac{\text{FLOPs}}{\text{Bytes moved}}
$$

其中：

- $I$：算术强度；
- FLOPs：浮点运算次数；
- Bytes moved：从显存等层级搬运的数据量。

如果每搬运一大批权重只用于一个 Token，算术强度较低。

如果一次用同一批权重检查多个 Token，计算单元有机会得到更充分利用。

推测解码正是尝试把“逐个检查”变成“小批量检查”。

### 三、两个模型分别做什么

#### 1. Target Model

Target Model，也叫目标模型，是用户原本要使用的大模型。

它决定最终输出应服从什么概率分布。

目标模型不能被 Draft 的错误直接绕过。

#### 2. Draft Model

Draft Model，也叫草稿模型或提议模型，负责快速预测未来几个 Token。

它通常：

- 参数更少；
- 单步更快；
- 与目标模型语言和领域接近；
- 最好使用兼容的 Tokenizer 或词表；
- 不直接决定最终输出。

好的 Draft 不是“越小越好”，而是**单位成本能提出足够多会被接受的 Token**。

### 四、最小流程：先草拟，再验证

假设当前前缀是：

```text
北京是中国的
```

Draft 连续提出 4 个 Token：

```text
首 / 都 / ， / 也
```

目标模型一次前向计算可以得到这些位置对应的概率：

```text
前缀 -> “首”的分布
前缀+首 -> “都”的分布
前缀+首+都 -> “，”的分布
前缀+首+都+， -> “也”的分布
```

虽然这些位置有因果依赖，但 Draft 已经给出了完整候选前缀。

Target 可以用带因果掩码的并行计算评估整段候选，而不必像生成时那样等待每个位置的采样结果。

接着从左到右验证：

- 如果“首”被接受，继续检查“都”；
- 如果“都”被接受，继续检查“，”；
- 如果“，”被拒绝，在这个位置按校正分布采样；
- “，”之后的草稿全部丢弃，因为它们依赖了被拒绝的前缀。

### 五、贪心场景的直觉版验证

如果使用确定性的贪心解码，可以做简单理解：

1. Draft 提出多个 Token；
2. Target 计算每个位置的最高概率 Token；
3. 从左往右比较；
4. 接受最长相同前缀；
5. 在首个不同位置采用 Target 的 Token。

例如：

```text
Draft：A B C D
Target：A B X ...
```

本轮接受 `A B`，然后输出 Target 的 `X`。

`C D` 被丢弃。

但采样解码不能只做“相同就收，不同就换”这么简单，因为要保持目标概率分布。

### 六、经典推测采样的接受概率

设在某个位置：

- Draft 分布为 $q(x)$；
- Target 分布为 $p(x)$；
- Draft 实际采样到 Token $x$。

经典算法以如下概率接受 $x$：

$$
a(x)=\min\left(1,\frac{p(x)}{q(x)}\right)
$$

逐个符号解释：

- $a(x)$：草稿 Token $x$ 被接受的概率；
- $p(x)$：目标模型给 $x$ 的概率；
- $q(x)$：草稿模型给 $x$ 的概率；
- $p(x)/q(x)$：目标相对草稿对该 Token 的支持程度；
- $\min(1,\cdot)$：接受概率不能超过 1。

#### 情况一：Target 比 Draft 更支持这个 Token

假设：

```text
p(x)=0.30
q(x)=0.20
```

则：

$$
a(x)=\min(1,1.5)=1
$$

该 Token 被接受。

#### 情况二：Draft 对 Token 过于乐观

假设：

```text
p(x)=0.10
q(x)=0.25
```

则：

$$
a(x)=\min(1,0.4)=0.4
$$

系统再采样一个 0 到 1 的随机数决定是否接受。

直觉是：Draft 对这个 Token 分配得太多，必须拒绝其中一部分质量，才能校正回 Target 分布。

### 七、拒绝后为什么要从“正残差”采样

如果草稿 Token 被拒绝，不能直接再从 $p$ 采样。

因为前面的接受步骤已经保留了 Draft 分布中的一部分概率质量。

为保持最终边缘分布为 $p$，经典算法从以下分布采样替代 Token：

$$
p'(x)=\operatorname{norm}\left(\max(0,p(x)-q(x))\right)
$$

其中：

- $p'(x)$：拒绝后的校正分布；
- $p(x)-q(x)$：目标概率减去草稿概率；
- $\max(0,\cdot)$：只保留正差值；
- $\operatorname{norm}$：将所有正值除以它们的总和，使概率和为 1。

假设词表只有 A、B、C：

| Token | $p$ | $q$ | $\max(0,p-q)$ |
| --- | ---: | ---: | ---: |
| A | 0.50 | 0.30 | 0.20 |
| B | 0.30 | 0.40 | 0 |
| C | 0.20 | 0.30 | 0 |

正残差只有 A，因此拒绝后会选择 A。

更一般的情况下，会对多个正残差重新归一化。

这个校正步骤是经典无损推测采样保持目标分布的关键。

### 八、“保持目标分布”是什么意思

按经典论文的接受和残差采样规则，在数学假设与实现正确的前提下，最终生成 Token 的分布与直接从 Target 采样一致。

它不要求 Draft 与 Target 输出一样。

Draft 只影响效率，不应改变目标分布。

这与下面的近似做法不同：

- 直接相信小模型；
- 只检查 top-1 是否相同；
- 放宽阈值接受近似 Token；
- 改变采样参数；
- 使用有损量化或近似验证导致分布变化。

工程框架可能同时提供“无损”和“启发式”模式。

阅读文档时要确认具体算法，而不是看到 `speculative` 就假设行为相同。

### 九、一轮能前进多少 Token

设：

- 草稿长度为 $\gamma$；
- 每个草稿 Token 的平均接受概率近似为 $\alpha$；
- 为方便推导，先假设各位置接受概率相同。

要接受第 1 个，需要概率 $\alpha$。

要接受前 2 个，需要概率 $\alpha^2$。

要接受前 $i$ 个，需要概率 $\alpha^i$。

一轮期望前进 Token 数可以粗略写成：

$$
E[L]=\sum_{i=0}^{\gamma}\alpha^i
=\frac{1-\alpha^{\gamma+1}}{1-\alpha},\quad \alpha\neq1
$$

其中：

- $E[L]$：一轮期望确认的 Token 数；
- $\gamma$：Draft 一次提出的 Token 数；
- $\alpha$：近似平均接受率；
- $i=0$ 的 1：即使首个草稿被拒，也会由 Target 校正生成一个 Token。

当 $\alpha=1$ 时，理想情况下：

$$
E[L]=\gamma+1
$$

因为所有草稿都接受后，还可以从 Target 的下一个位置得到一个额外 Token。

实际接受率会随位置、上下文和采样参数变化，这个公式只是建立直觉。

### 十、一个数值例子

假设草稿长度 $\gamma=4$，平均接受率 $\alpha=0.8$：

$$
E[L]=1+0.8+0.8^2+0.8^3+0.8^4
$$

$$
=1+0.8+0.64+0.512+0.4096=3.3616
$$

也就是说，一轮 Target 验证平均前进约 3.36 个 Token。

普通解码一轮通常前进 1 个 Token。

但这不代表加速比就是 3.36 倍，因为还要付出：

- Draft 连续生成 4 步的时间；
- Target 验证 4 个位置的时间；
- 概率校正与调度开销；
- 可能增加的显存和通信。

### 十一、如何粗略理解加速比

把普通 Target 单 Token 步骤成本归一化为 1。

假设：

- Draft 每一步相对成本为 $c_d$；
- 生成 $\gamma$ 个草稿成本约为 $\gamma c_d$；
- Target 批量验证成本为 $c_v(\gamma)$；
- 一轮期望前进 $E[L]$ 个 Token。

粗略吞吐加速可以写成：

$$
S\approx\frac{E[L]}{\gamma c_d+c_v(\gamma)+c_o}
$$

其中：

- $S$：相对普通单步解码的近似加速；
- $c_o$：调度、采样和数据搬运等额外开销。

只有 $S>1$，方案才产生正向加速。

注意，$c_v(\gamma)$ 通常不是常数，也不等于 $\gamma$ 次普通 Decode。

它取决于硬件、批量、模型、内核和序列长度。

真实结果必须实测。

### 十二、Draft 为什么不能太弱或太强

#### Draft 太弱

- 接受率低；
- 经常第一个 Token 就被拒绝；
- 后续草稿全部浪费；
- Target 验证和 Draft 计算成为额外负担。

#### Draft 太强

- 接受率可能高；
- 但 Draft 自己也很慢；
- 可能需要大量显存；
- 与 Target 同时驻留会降低可用 KV Cache 或批量容量。

所以选 Draft 是一个优化问题：

```text
高接受率
  与
低草稿成本
  与
低额外显存
之间取平衡
```

### 十三、草稿长度 $\gamma$ 怎么选

草稿太短：

- 每次只验证一两个 Token；
- Target 并行验证收益不足；
- 固定调度开销占比大。

草稿太长：

- 越靠后的草稿依赖越多未经验证的 Token；
- 前面一旦拒绝，后面全部作废；
- Target 验证张量更大；
- Draft 串行生成时间增加。

可根据最近接受长度动态调整：

```text
连续多轮全部接受 -> 适当增加 gamma
连续多轮早期拒绝 -> 降低 gamma
接近延迟或显存上限 -> 降低 gamma
```

动态算法仍需要稳定边界，避免每轮剧烈振荡。

### 十四、温度和任务会影响接受率

在较低温度或贪心解码中，Draft 和 Target 的高概率 Token 可能更一致。

在高温采样中，候选更分散，接受率可能下降。

但不能据此推断所有模型和任务都遵循同一幅度。

接受率还与以下因素相关：

- 目标和 Draft 的训练语料；
- 对齐与后训练差异；
- 语言与领域；
- 代码、数学还是自然语言；
- 上下文长度；
- 重复性；
- Tokenizer；
- Top-k、Top-p 和其他采样处理顺序。

采样过滤应该在算法规定的一致分布上进行，否则接受公式可能失效。

### 十五、Tokenizer 为什么是工程难点

如果 Draft 和 Target 使用相同词表与 Tokenizer，Token 可以直接对齐。

如果不同：

```text
Draft Token："unbelievable"
Target Token："un" + "believ" + "able"
```

一对一位置验证不再直接成立。

系统需要重分词、字符级对齐或专门算法。

这会增加复杂度，并可能降低可接受草稿长度。

因此很多双模型实现要求 Tokenizer 兼容。

### 十六、除了双模型，还有哪些变体

#### 1. Medusa：多个解码头

Medusa 在主模型上增加多个头，分别预测未来位置的候选。

优点是不用常驻一个完整独立 Draft 模型。

它会构建候选树，再由主模型验证。

代价是需要训练额外头，并管理树候选和验证内核。

#### 2. EAGLE：从特征预测未来

EAGLE 系列从隐藏特征等信号预测未来 Token，试图提高草稿准确率并降低草稿成本。

它不只是用一个普通小语言模型逐 Token 猜测。

具体可用性取决于框架、模型支持和训练资产。

#### 3. Prompt Lookup Decoding

如果输出经常复制输入中的片段，例如摘要、编辑或代码补全，可以从 Prompt 中查找重复 n-gram 作为候选。

它不需要独立神经 Draft，但只适合高复制场景。

#### 4. N-gram 或缓存草稿

系统也可以从历史上下文、缓存或短语表提出候选。

候选来源不同，验证原则仍然是由 Target 审核。

#### 5. 多分支 SpecInfer

一些方法用多个 Draft 或树结构一次提出多条候选路径，让 Target 并行验证更多分支。

这提高候选覆盖，也增加树构建、显存和验证复杂度。

### 十七、推测解码主要优化哪个阶段

它主要针对生成阶段的串行 Decode。

一个请求的端到端时间可以粗略拆为：

$$
T_{e2e}=T_{queue}+T_{prefill}+T_{decode}+T_{other}
$$

其中：

- $T_{queue}$：排队时间；
- $T_{prefill}$：处理输入的时间；
- $T_{decode}$：生成输出的时间；
- $T_{other}$：网络、调度和后处理时间。

如果输入极长、输出只有几个 Token，$T_{prefill}$ 占主导，优化 Decode 的总收益有限。

如果输入较短、输出很长，推测解码更可能显著影响端到端时间。

### 十八、延迟和吞吐不是同一个目标

在单请求低并发下，推测解码可能减少 TPOT。

在高并发连续批处理下，Target 原本已经通过批量获得较高硬件利用率。

加入 Draft 后可能：

- 占用额外 GPU；
- 让不同请求每轮接受长度不一致；
- 增加调度碎片；
- 减少可容纳的批量；
- 改变吞吐与延迟的平衡。

因此单请求加速不代表服务总吞吐同比例提升。

### 十九、一个完整验证周期

设当前前缀为 $h$，草稿长度为 3。

#### 第 1 步：Draft 生成

```text
x1 ~ q(. | h)
x2 ~ q(. | h, x1)
x3 ~ q(. | h, x1, x2)
```

符号 `~` 表示“从该分布采样”。

#### 第 2 步：Target 并行评分

Target 计算：

```text
p(. | h)
p(. | h, x1)
p(. | h, x1, x2)
p(. | h, x1, x2, x3)
```

最后一个分布用于所有草稿接受时生成额外 Token。

#### 第 3 步：从左到右验证

对每个 $x_i$ 计算：

$$
\min(1,p(x_i)/q(x_i))
$$

一旦拒绝，采样校正 Token，并结束本轮。

#### 第 4 步：提交结果

只把已经验证的最长前缀和一个 Target 校正或额外 Token 加入正式输出。

未验证草稿不能泄漏到用户结果或 KV 状态。

## 工程实践与边界

### 一、先建立不开启推测的基线

基线要固定：

- Target 模型和权重；
- 精度与量化；
- Prompt 和输出长度；
- Temperature、Top-p、Top-k；
- 并发和到达率；
- 硬件与推理框架版本。

再对比开启推测后的变化。

### 二、至少监控这些指标

| 指标 | 作用 |
| --- | --- |
| Draft tokens proposed | 一共提出多少草稿 |
| Draft tokens accepted | 一共接受多少草稿 |
| Acceptance rate | Draft 与 Target 匹配程度 |
| Accepted length histogram | 每轮在第几个位置被拒绝 |
| Draft latency | 草稿生成成本 |
| Verification latency | Target 批量验证成本 |
| TTFT | 是否影响首 Token |
| TPOT / ITL | 生成流畅度是否改善 |
| E2E P95/P99 | 尾延迟是否恶化 |
| Throughput / goodput | 服务承载和 SLO 达成 |
| Output distribution tests | 是否保持预期采样行为 |

只看接受率不足以判断收益。

### 三、按请求选择是否启用

可以根据：

- 预计输出长度；
- 当前并发；
- Draft 与目标的领域匹配；
- 当前 GPU 余量；
- 采样温度；
- 历史接受率；
- 延迟 SLO；

动态决定是否启用及草稿长度。

但策略本身应稳定、可观测、可回滚。

### 四、不要忽略双模型显存

Draft 权重、KV Cache 和中间张量都会占显存。

如果为了放下 Draft 而减少 Target 的批量或 KV 容量，总吞吐可能下降。

可以考虑：

- Draft 放另一块设备；
- 使用同模型多头方案；
- 对 Draft 做量化；
- 共享可共享的组件；
- 按负载关闭 Draft。

跨设备还会引入通信开销。

### 五、对分布保持做回归测试

经典算法是随机算法，不能要求两次运行逐 Token 相同。

可以在固定随机种子和支持条件下做实现一致性检查，也可以对大量样本比较：

- Token 频率；
- 输出质量；
- 长度分布；
- 停止原因；
- 安全策略命中。

如果使用近似变体，应明确质量允许范围，而不是称为数学等价。

### 六、流式输出要只提交已验证 Token

Draft Token 只是候选。

在 Target 接受前不能流给客户端，否则拒绝后就无法可靠收回。

这要求服务端区分：

```text
草稿缓冲区
已验证输出区
客户端已发送区
```

### 七、模型升级会改变接受率

Target、Draft、Tokenizer、量化方式或采样内核升级后，都要重新压测。

即使接口没变，两个模型的概率分布接近程度也可能变化。

### 八、框架支持仍在快速演进

不同推理框架支持的 Draft 类型、采样算法、并行方式和模型组合不同。

应查目标版本官方文档，并在自己的硬件和负载上验证，避免照搬论文中的理想加速倍数。

## 常见误区

### 误区一：最终答案由 Draft Model 决定

经典方案中，Draft 只提出候选，Target 的验证和校正决定最终分布。

### 误区二：Draft 越小越快，效果就越好

Draft 太弱会导致接受率低，额外开销可能超过收益。

### 误区三：接受率 80% 就等于加速 80%

加速还取决于草稿长度、Draft 成本、Target 验证成本、批量和调度。

### 误区四：一次验证多个 Token 等于提前知道未来

Draft 已经提供了候选前缀，Target 只是并行评分这些已知候选位置。

### 误区五：拒绝后直接从 Target 普通分布采样

经典采样算法需要正残差校正，否则会改变目标分布。

### 误区六：推测解码会减少 Prefill

它主要优化 Decode，长 Prompt 的 Prefill 工作仍然存在。

### 误区七：单请求更快就代表集群吞吐更高

Draft 资源、连续批处理和请求分歧会改变服务级结果。

### 误区八：所有 Speculative 方案都是无损的

算法和实现不同，有些启发式方案会改变分布或质量。

### 误区九：Draft Token 可以先流给用户

未经验证的 Token 可能被拒绝，不能作为正式输出提交。

### 误区十：论文加速倍数可以直接用于容量规划

模型、硬件、输入输出长度、并发和软件栈都会改变结果。

## 面试追问

### 追问一：为什么 Target 能一次检查多个 Token？

因为 Draft 已经给出了整段候选前缀，Target 可以像训练或 Prefill 那样用因果掩码并行计算这些位置的 Logit。

### 追问二：草稿在第 2 个 Token 被拒绝，后面的怎么办？

第 2 个位置由校正分布生成替代 Token；后续草稿依赖了错误前缀，因此全部丢弃。

### 追问三：接受概率为什么是 $\min(1,p/q)$？

它保留 Draft 分布中不超过 Target 允许的那部分概率质量；Draft 过量的部分通过拒绝和残差采样校正。

### 追问四：如何选 Draft Model？

在真实任务上联合测接受率、Draft 延迟、额外显存、Tokenizer 兼容、Target 验证效率和服务吞吐，选择净收益最大的组合。

### 延伸问题与回答要点

| 问题 | 回答要点 |
| --- | --- |
| 草稿长度怎么调 | 做多档扫描，并根据近期接受长度、负载和显存动态调整 |
| 高温采样还能用吗 | 数学算法可以支持采样，但分布差异可能降低接受率，必须保证过滤和校正顺序一致 |
| 能跨供应商 API 做吗 | 通常需要访问每个位置的目标概率并控制解码循环，普通黑盒文本 API 往往不够 |
| Medusa 和双模型区别 | Medusa 用主模型附加多头提出未来候选，减少独立 Draft，但需额外训练和树验证 |
| 为什么输出越长越可能受益 | Decode 占端到端时间的比例更高，优化空间更大 |
| 如何确认没有质量退化 | 核对算法模式，做分布与任务回归，并分别报告精确和近似变体 |
