## 题目入口

题目链接：[删除有序数组中的重复项](https://leetcode.cn/problems/remove-duplicates-from-sorted-array/description/)

![image-20260707211452200](https://gitee.com/mu-mo_1_0/images/raw/master/image-20260707211452200.png)

## 一句话读懂题目

给你一个已经按非递减顺序排好序的数组 `nums`，你要把重复的数字删掉，让每个数字只出现一次。

最后返回去重后数组的长度 `k`。

比如：

```text
nums = [1, 1, 2]
```

去重后应该变成：

```text
[1, 2]
```

所以返回：

```text
2
```

注意，这道题有一个非常重要的要求：

> 不能新建一个额外数组来保存答案，需要用 `O(1)` 额外空间在原数组 `nums` 上直接修改。

`O(1)` 额外空间可以简单理解成：不管数组有多长，我们额外使用的变量数量都差不多，比如只用几个指针变量。

也就是说，LeetCode 最后只会检查：

```text
nums 的前 k 个元素是否是去重后的结果
```

后面的元素是什么，不重要。

比如返回 `k = 2` 时，只要数组前两个位置是：

```text
[1, 2]
```

后面即使还是旧数据，也不会影响答案。

## 先用生活例子理解

可以把这道题想象成：一群人按身高从矮到高排队。

因为队伍已经排好序了，所以相同身高的人一定会挨在一起。

比如：

```text
[1, 1, 2, 2, 3]
```

重复的 `1` 都在一起，重复的 `2` 也都在一起。

这时我们只需要做一件事：

> 每遇到一个和前面保留下来的数字不一样的新数字，就把它放到答案区的下一个位置。

所以这道题真正利用的是“数组已经有序”这个条件。

如果数组没有排序，重复数字可能散落在各处，处理方式就会不一样。

## 题目里的“原地修改”是什么意思

很多同学第一次做这道题，会想直接创建一个新数组：

```java
int[] result = new int[nums.length];
```

然后把不重复的数字放进去。

但是题目不允许这样做。

它要求我们直接改 `nums` 本身。

比如原数组是：

```text
nums = [1, 1, 2]
```

我们要把它改成类似这样：

```text
nums = [1, 2, 2]
```

然后返回：

```text
k = 2
```

这里最后一个 `2` 是旧数据，不用管。

因为题目只看前 `k` 个元素：

```text
nums[0] = 1
nums[1] = 2
```

这就是“原地修改”的意思：不用额外数组，而是在原数组前面整理出答案。

## 核心思路：快慢指针

这道题最经典的做法是双指针，也叫快慢指针。

我们可以准备两个指针：

```text
slow：指向去重后数组的最后一个位置
fast：负责从前往后扫描整个数组
```

可以把它理解成：

- `slow` 管“答案区”；
- `fast` 管“侦察区”。

刚开始，第一个数字一定要保留，所以：

```text
slow = 0
```

然后让 `fast` 从 `1` 开始往后走。

每次比较：

```text
nums[fast] 和 nums[slow]
```

如果它们相等，说明 `nums[fast]` 是重复数字，跳过。

如果它们不相等，说明遇到了一个新数字，就把 `slow` 往后移动一格，然后把这个新数字放到 `nums[slow]`。

核心逻辑就是：

```text
如果 nums[fast] != nums[slow]
    slow++
    nums[slow] = nums[fast]
```

最后答案长度就是：

```text
slow + 1
```

因为 `slow` 是下标，下标从 `0` 开始。

## 手动走一遍例子

以这个数组为例：

```text
nums = [1, 1, 2, 2, 3]
```

刚开始：

```text
slow = 0
fast = 1
答案区：[1]
```

### 第一步：fast 看到第二个 1

比较：

```text
nums[fast] = 1
nums[slow] = 1
```

两个数相等，说明是重复的 `1`，跳过。

```text
slow 还是 0
```

### 第二步：fast 看到 2

比较：

```text
nums[fast] = 2
nums[slow] = 1
```

不相等，说明 `2` 是新数字。

把 `slow` 往后移动一格：

```text
slow = 1
```

再把 `2` 放到 `nums[slow]`：

```text
nums = [1, 2, 2, 2, 3]
```

此时答案区是：

```text
[1, 2]
```

### 第三步：fast 看到第二个 2

比较：

```text
nums[fast] = 2
nums[slow] = 2
```

相等，说明是重复的 `2`，跳过。

### 第四步：fast 看到 3

比较：

```text
nums[fast] = 3
nums[slow] = 2
```

不相等，说明 `3` 是新数字。

`slow` 往后移动一格：

```text
slow = 2
```

把 `3` 放到 `nums[slow]`：

```text
nums = [1, 2, 3, 2, 3]
```

最后返回：

```text
slow + 1 = 3
```

前 `3` 个元素就是去重后的结果：

```text
[1, 2, 3]
```

后面的 `[2, 3]` 不用管。

如果把整个过程压缩成一张表，会更直观：

| fast 位置 | nums[fast] | nums[slow] | 是否新数字 | 操作后 slow | 数组前半部分 |
| --- | --- | --- | --- | --- | --- |
| 初始 | - | `1` | - | `0` | `[1]` |
| `1` | `1` | `1` | 否 | `0` | `[1]` |
| `2` | `2` | `1` | 是 | `1` | `[1, 2]` |
| `3` | `2` | `2` | 否 | `1` | `[1, 2]` |
| `4` | `3` | `2` | 是 | `2` | `[1, 2, 3]` |

这张表里最重要的是最后一列：

> 无论后面的旧数据是什么，数组前半部分一直保持为已经去重的结果。

## Java 代码

先记住一句话：

> 快指针负责找新数字，慢指针负责维护答案区。

```java
class Solution {
    public int removeDuplicates(int[] nums) {
        int slow = 0;

        for (int fast = 1; fast < nums.length; fast++) {
            if (nums[fast] != nums[slow]) {
                slow++;
                nums[slow] = nums[fast];
            }
        }

        return slow + 1;
    }
}
```

## 代码逐行拆解

```java
int slow = 0;
```

第一个元素一定是去重后数组的一部分，所以让 `slow` 从 `0` 开始。

`slow` 指向的是当前答案区最后一个元素的位置。

```java
for (int fast = 1; fast < nums.length; fast++) {
```

`fast` 从第二个元素开始扫描。

因为第一个元素已经默认保留，不需要和自己比较。

```java
if (nums[fast] != nums[slow]) {
```

如果当前扫描到的数字和答案区最后一个数字不同，说明遇到了一个新数字。

注意，这里能这么比较，是因为数组已经有序。

```java
slow++;
```

答案区要多放一个新数字，所以 `slow` 往后移动一格。

```java
nums[slow] = nums[fast];
```

把新数字放到答案区的末尾。

```java
return slow + 1;
```

`slow` 是最后一个有效元素的下标。

如果最后一个有效元素下标是 `2`，那有效长度就是 `3`。

所以返回 `slow + 1`。

## 为什么不用处理空数组

这道题的约束里说明：

```text
1 <= nums.length <= 3 * 10^4
```

也就是说，数组长度至少是 `1`。

所以代码里可以直接写：

```java
int slow = 0;
```

不用额外判断：

```java
if (nums.length == 0) return 0;
```

当然，如果是在真实业务代码里，输入可能为空，那就要根据业务要求补上空数组判断。

## 为什么数组有序很关键

这道题之所以可以只比较 `nums[fast]` 和 `nums[slow]`，是因为数组已经排好序。

比如：

```text
[1, 1, 2, 2, 3]
```

相同数字一定连续出现。

当 `slow` 指向答案区最后一个数字时，只要当前数字和它不一样，就说明当前数字一定是新的。

如果数组是这样的：

```text
[1, 2, 1, 3]
```

就不能用这套逻辑了。

因为第二个 `1` 没有和第一个 `1` 挨在一起，只比较最后一个保留数字会漏掉重复。

所以面试时一定要说清楚：

> 本题利用了数组有序这个前提。

## 再提升一点：循环不变量

如果只是刷题，理解快慢指针就够了。

如果是面试，想讲得更稳，可以补一句“循环不变量”。

这道题的循环不变量是：

> 在每一次循环开始前，`nums[0...slow]` 始终是已经处理过的元素里，去重后的有序结果。

换句话说，`slow` 前面的区域永远是干净的答案区。

当 `nums[fast] == nums[slow]` 时，当前数字重复，跳过不会破坏答案区。

当 `nums[fast] != nums[slow]` 时，当前数字是新数字，把它放到 `slow + 1` 的位置，答案区就安全地扩大了一格。

这也解释了为什么覆盖是安全的：

> 我们只关心前 `slow + 1` 个有效元素，后面的旧值本来就不属于最终答案。

## 复杂度分析

时间复杂度：`O(n)`。

因为 `fast` 从前到后只遍历了一次数组。

空间复杂度：`O(1)`。

因为只用了 `slow` 和 `fast` 两个变量，没有使用额外数组。

这也是题目要求的原地修改。

## 常见坑

### 1. 返回的是长度，不是数组

题目让你返回的是 `k`，也就是去重后数组的长度。

不要写成返回数组。

LeetCode 会根据你返回的 `k`，检查 `nums` 的前 `k` 个元素。

### 2. 后面的元素不用管

比如最后数组变成：

```text
[1, 2, 3, 2, 3]
```

返回 `3` 就可以。

因为前 `3` 个元素是：

```text
[1, 2, 3]
```

后面的 `2, 3` 不会被检查。

### 3. 不要一遇到重复就删除元素

有些同学会想：遇到重复元素，就把它从数组里删掉。

但是 Java 的普通数组长度是固定的，不能像 `ArrayList` 那样直接删除中间元素。

这道题的正确思路不是“真的删除”，而是：

> 把不重复的元素覆盖到数组前面。

### 4. `slow` 的含义不要搞混

`slow` 不是去重后数组的长度。

它是去重后数组最后一个元素的下标。

所以最后返回的是：

```java
slow + 1
```

不是：

```java
slow
```

## 面试时怎么讲

### 30 秒版

可以这样表达：

> 因为数组已经有序，所以重复元素一定相邻。我用快慢指针来原地去重，`slow` 指向去重后数组的最后一个元素，`fast` 从前往后扫描。当 `nums[fast]` 和 `nums[slow]` 不同时，说明遇到了一个新数字，就先让 `slow` 后移，再把 `nums[fast]` 覆盖到 `nums[slow]`。遍历结束后，前 `slow + 1` 个元素就是去重结果，返回 `slow + 1`。时间复杂度是 `O(n)`，空间复杂度是 `O(1)`。

### 面试官可能追问

**为什么可以只和 `nums[slow]` 比较？**

因为数组是有序的，相同元素一定连续出现。`nums[slow]` 是当前答案区最后一个保留的数字，如果当前数字和它不同，就说明当前数字是新的。

**为什么不是返回 `slow`？**

因为 `slow` 是下标，不是长度。下标从 `0` 开始，所以有效长度是 `slow + 1`。

**为什么不需要新数组？**

题目要求原地修改。我们用 `slow` 维护原数组前面的答案区，遇到新数字就覆盖到答案区末尾，所以不需要额外数组。

**如果所有数字都一样怎么办？**

比如：

```text
[1, 1, 1]
```

`slow` 一直停在 `0`，最后返回 `1`，前一个元素 `[1]` 就是答案。

**如果没有重复数字怎么办？**

比如：

```text
[1, 2, 3]
```

每次 `nums[fast] != nums[slow]` 都成立，`slow` 会一路往后走，最后返回 `3`，原数组本身就是答案。

**为什么覆盖原数组不会丢失还没看的数据？**

因为 `fast` 一直在前面扫描，`slow` 只在答案区末尾移动，并且永远不会超过 `fast`。

当我们写 `nums[slow] = nums[fast]` 时，写入位置一定是已经扫描过、可以整理的位置，不会覆盖还没扫描到的未知元素。

**如果题目变成“每个元素最多保留两次”怎么办？**

这就是 LeetCode 80 的思路。

仍然可以用快慢指针，只是判断条件会从“和答案区最后一个元素比较”，变成“和答案区倒数第二个元素比较”。

简单记忆：最多保留几次，就回头看答案区里第几个位置。

**如果数组没有排序，还能这么做吗？**

不能直接这么做。

因为无序数组里重复元素不一定相邻，只比较 `nums[fast]` 和 `nums[slow]` 会漏掉重复。

无序去重通常要借助哈希表，或者先排序再处理，但排序会改变原始顺序。

## 最后总结

删除有序数组中的重复项，表面上是在“删除重复数字”，实际上考的是：

> 如何在原数组前面维护一个干净的答案区。

快慢指针的分工很清楚：

```text
fast：一路往前找新数字
slow：记录答案区最后一个位置
```

只要遇到和答案区最后一个数字不同的元素，就把它放进答案区。

这道题可以记成一个很常用的模板：

```text
先保留第一个元素
fast 从第二个元素开始扫描
如果当前元素应该保留
    slow 后移
    把当前元素覆盖到 nums[slow]
最后返回 slow + 1
```

套到本题里，“当前元素应该保留”的条件就是：

```java
nums[fast] != nums[slow]
```

刷题时可以用下面几个边界用例自检：

```text
[1]              -> 返回 1，前 1 个元素是 [1]
[1, 1, 1]       -> 返回 1，前 1 个元素是 [1]
[1, 2, 3]       -> 返回 3，前 3 个元素是 [1, 2, 3]
[0, 0, 1, 1, 2] -> 返回 3，前 3 个元素是 [0, 1, 2]
```

以后看到这类题目，如果它要求“原地修改数组”，并且只需要保留一部分元素，就可以优先想到快慢指针。

最后再把本题的核心句背下来：

> `fast` 负责向前探索，`slow` 负责守住答案区；遇到新数字，就把它覆盖到答案区的下一个位置。
