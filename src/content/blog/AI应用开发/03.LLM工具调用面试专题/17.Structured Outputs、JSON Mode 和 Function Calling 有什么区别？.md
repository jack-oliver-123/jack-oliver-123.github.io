## 本篇目标

学完这一篇，你应该能够：

- 区分“合法 JSON”“符合 JSON Schema”和“请求调用工具”；
- 解释 JSON Mode、Structured Outputs、Function Calling 分别约束什么；
- 理解严格结构仍可能拒绝、截断、填错事实或产生越权参数；
- 为结构化抽取、UI 数据和外部动作选择合适机制；
- 设计“状态检查—结构校验—业务校验—授权—执行”的安全流程；
- 在面试中避免把结构正确误说成业务正确。

## 60 秒回答

JSON Mode 解决的是语法层：让模型响应成为可解析的 JSON，但不保证必填字段、类型、枚举或嵌套结构符合你的 Schema。Structured Outputs 解决的是结构契约：在供应商支持的 JSON Schema 子集内，让模型输出匹配给定 Schema；但仍要处理拒绝、截断和服务错误，也不保证字段里的事实正确。

Function Calling 解决的是交互意图：应用把工具定义给模型，模型返回工具名和参数，表达“希望宿主调用这个工具”。宿主拥有最终执行权，可以拒绝、请求确认或修改流程。Function Calling 的参数也可以使用严格 Schema，但它不等于 Structured Outputs 的普通结构化回答，更不会自动执行函数。

选型上，兼容旧路径且只需合法 JSON 时用 JSON Mode；抽取、分类或给 UI 返回固定对象时优先 Structured Outputs；需要查询数据库、发送消息或修改业务状态时用 Function Calling。三者都不能替代服务端 Schema 复验、业务规则、主体授权、资源状态、幂等和审计。

## 详细解析

### 先说人话：信封、表格和办事申请不是一回事

想象三种东西：

1. **JSON Mode** 像要求“必须装进一个能打开的信封”；
2. **Structured Outputs** 像要求“必须填写指定格式的表格”；
3. **Function Calling** 像提交“请柜台替我办理某项业务”的申请单。

信封能打开，不代表里面字段齐全；表格填完整，不代表内容真实；办事申请符合格式，也不代表申请人有权限或柜台已经执行。

### 贯穿案例：退款客服助手

用户说：

> 订单 O-2025-42 的耳机坏了，帮我退 299 元。

系统可能有三种需求。

#### 需求一：抽取用户表达

只想得到：

~~~json
{
  "order_id": "O-2025-42",
  "reason": "商品损坏",
  "requested_amount": 299
}
~~~

这是结构化信息抽取，更适合 Structured Outputs。

#### 需求二：返回 UI 卡片

希望模型产生：

~~~json
{
  "title": "退款申请摘要",
  "risk_level": "medium",
  "needs_confirmation": true,
  "display_items": [
    "订单：O-2025-42",
    "金额：299 元"
  ]
}
~~~

它仍是结构化回答，不需要自动访问外部系统。

#### 需求三：真正查询和退款

模型可能先请求：

~~~json
{
  "name": "get_order",
  "arguments": {
    "order_id": "O-2025-42"
  }
}
~~~

查询后再请求：

~~~json
{
  "name": "create_refund",
  "arguments": {
    "order_id": "O-2025-42",
    "amount": 299,
    "reason_code": "DAMAGED"
  }
}
~~~

这属于 Function Calling。应用要验证当前用户拥有该订单、金额不超过可退余额、退款状态允许，并在执行前确认。

### 最低前置概念：JSON 与 JSON Schema

#### JSON 是数据格式

合法 JSON 示例：

~~~json
{
  "name": "create_refund",
  "amount": "非常多",
  "admin": true
}
~~~

它语法合法，却可能完全不符合业务。

#### JSON Schema 是结构契约

可以规定：

- 对象有哪些属性；
- 属性类型；
- 哪些字段必填；
- 字符串允许哪些枚举；
- 数字范围；
- 是否允许额外字段；
- 嵌套对象和数组结构。

示例：

~~~json
{
  "type": "object",
  "properties": {
    "order_id": {
      "type": "string",
      "pattern": "^O-[0-9-]+$"
    },
    "amount": {
      "type": "number",
      "minimum": 0
    },
    "reason_code": {
      "type": "string",
      "enum": ["DAMAGED", "WRONG_ITEM", "OTHER"]
    }
  },
  "required": ["order_id", "amount", "reason_code"],
  "additionalProperties": false
}
~~~

即使通过这个 Schema，amount 也可能超过真实支付金额；这需要业务校验。

### JSON Mode：保证目标是有效 JSON

JSON Mode 的核心是把模型响应限制为 JSON 对象格式。它适合：

- 旧接口兼容；
- 结构非常简单；
- 应用已有完善验证和修复；
- 当前模型或接口不支持 Structured Outputs；
- 只要求先解决 Markdown 包裹、逗号等语法问题。

#### JSON Mode 不保证什么

给定期望：

~~~json
{
  "order_id": "string",
  "amount": "number"
}
~~~

模型仍可能返回：

~~~json
{
  "order": "O-2025-42",
  "amount": "299元",
  "extra_note": "已自动退款"
}
~~~

它是合法 JSON，但：

- 字段名错；
- amount 类型错；
- 多了字段；
- “已自动退款”可能是虚构状态。

#### JSON Mode 还要注意

- 提示中要明确要求 JSON；
- 仍要检测输出未完成或被截断；
- 仍要执行 JSON 解析和 Schema 校验；
- 不要通过字符串拼接“修好”后直接执行；
- 自行处理重试和错误反馈；
- 不要把 JSON 里的命令、URL、SQL 当可信输入。

### Structured Outputs：约束输出匹配 Schema

Structured Outputs 让模型在平台支持的 JSON Schema 子集内生成指定结构。

适合：

- 信息抽取；
- 分类；
- UI 卡片；
- 工作流决策对象；
- 评测标签；
- 报告中的固定字段；
- 稳定的服务间数据契约。

#### 它比 JSON Mode 多了什么

| 能力 | JSON Mode | Structured Outputs |
| --- | --- | --- |
| 输出可解析 JSON | 是 | 是 |
| 必填字段 | 不保证 | 在支持契约内约束 |
| 字段类型 | 不保证 | 在支持契约内约束 |
| enum | 不保证 | 在支持契约内约束 |
| 额外字段 | 不保证 | 可按支持方式限制 |
| 结构嵌套 | 需自行提示 | 可按支持子集声明 |

#### 它仍然不保证什么

1. **事实正确**：可以严格返回错误金额；
2. **资源存在**：订单 ID 可能不存在；
3. **当前用户有权限**：可以生成别人的订单号；
4. **业务状态允许**：已退款订单仍可能被要求退款；
5. **请求完整成功**：可能拒绝、截断、取消或服务错误；
6. **完整 JSON Schema 全部支持**：平台通常声明支持子集。

### Function Calling：模型表达工具调用意图

应用向模型提供工具定义：

~~~json
{
  "type": "function",
  "name": "get_order",
  "description": "查询当前用户可访问的订单。只在回答需要实时订单状态时使用。",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string"
      }
    },
    "required": ["order_id"],
    "additionalProperties": false
  },
  "strict": true
}
~~~

模型返回工具调用后，应用：

1. 解析工具名和参数；
2. 校验 Schema；
3. 从会话得到当前用户和租户；
4. 校验该用户能否访问订单；
5. 执行查询；
6. 把结构化结果与调用 ID 返回给模型；
7. 模型继续回答或提出下一次调用。

#### Function Calling 不会自动执行

“Calling”容易让初学者误以为模型直接调用了本地函数。实际上：

~~~text
模型：生成工具调用意图
宿主应用：决定是否执行
工具服务：真正产生读写效果
~~~

宿主可以：

- 拒绝；
- 要求用户补充；
- 请求人工审批；
- 修改为安全的查询；
- 选择不同实现；
- 超时或熔断；
- 返回结构化错误。

### Strict Function Calling 与 Structured Outputs 的关系

两者都可以使用 Schema 约束模型输出，但用途不同：

- Structured Outputs 的产物通常就是应用要消费的结构化回答；
- Strict Function Calling 的产物是一次工具调用请求的参数；
- 工具执行结果还要回到模型或工作流；
- Function Calling 增加了授权、执行、副作用、重试和幂等问题。

可以这样记：

~~~text
我要一个对象 → Structured Outputs
我要模型建议调用一个外部能力 → Function Calling
我只想兼容地拿到合法 JSON → JSON Mode
~~~

### 三者对比总表

| 维度 | JSON Mode | Structured Outputs | Function Calling |
| --- | --- | --- | --- |
| 主要目标 | JSON 语法 | 响应结构契约 | 表达工具调用意图 |
| 输出内容 | JSON 文本/对象 | 符合支持 Schema 的对象 | 工具名、参数和调用 ID |
| 是否访问外部系统 | 否 | 否 | 由宿主决定 |
| 是否可有严格参数 | 自行校验 | 是 | strict 开启时可约束 |
| 是否保证事实 | 否 | 否 | 否 |
| 是否保证授权 | 否 | 否 | 否 |
| 典型用途 | 兼容旧 JSON 流程 | 抽取、分类、UI 数据 | 查询、搜索、业务动作 |
| 主要失败 | 字段漂移、类型错误 | 拒绝、截断、内容错误 | 误选工具、参数或执行失败 |

### 一条安全的结构化响应流程

~~~text
发送请求
→ 检查 HTTP / API 状态
→ 检查模型是否拒绝
→ 检查响应是否完成
→ 解析 JSON
→ 本地 Schema 再校验
→ 业务规则校验
→ 作为数据使用
~~~

即使供应商承诺严格结构，本地再校验仍有价值：

- 防 SDK 或版本差异；
- 防中间层错误转换；
- 保护后续服务；
- 记录明确失败；
- 让接口契约可测试。

### 一条安全的工具调用流程

~~~text
模型返回调用意图
→ 检查调用状态与 ID
→ 允许名单检查
→ Schema 校验
→ 规范化参数
→ 从可信会话注入用户和租户
→ 资源级授权
→ 业务状态与不变量检查
→ 高风险动作确认
→ 幂等执行
→ 结构化结果或错误
→ 与调用 ID 关联返回模型
~~~

### 失败分支要显式设计

#### 模型拒绝

某些请求因安全策略被拒绝。拒绝响应不一定符合目标业务对象，应用要检查专用状态，而不是强行解析。

#### 输出截断

达到输出上限、连接中断或请求取消时，JSON 可能不完整。Structured Outputs 也不能把一个未完成响应变成完整业务对象。

#### Schema 不兼容

供应商支持的关键字子集不同。跨模型系统要：

- 选择共同子集；
- 在适配层转换；
- 为每个目标做契约测试；
- 不假设 JSON Schema 2020-12 的全部关键字都受模型约束支持。

#### 工具不存在或版本变化

模型缓存了旧工具名、网关转换丢字段、工具在会话中被撤销，都可能失败。应用应返回明确的不可重试或需重新规划错误。

#### 参数结构正确但业务错误

例如 amount 是合法正数，但：

- 大于订单支付金额；
- 币种不匹配；
- 已超过退款期；
- 当前用户不是订单所有者；
- 订单已经退款。

这些都只能由业务系统判断。

### Schema 版本如何治理

Schema 是接口契约，应保存：

- 名称；
- 版本；
- 字段语义；
- 单位和时区；
- required；
- enum；
- 默认行为；
- 兼容策略；
- 对应消费端版本。

#### 常见破坏性变更

- 新增必填字段；
- 删除 enum 值；
- 改字段类型；
- 金额从元改为分；
- 日期从本地时间改为 UTC；
- 同名字段改变业务含义。

结构上仍可解析，不代表语义兼容。

### 伪代码：三层校验

~~~python
response = call_model(schema=refund_request_schema)

if response.status != "completed":
    return handle_incomplete_or_refusal(response)

data = response.structured_data

json_schema_validate(refund_request_schema, data)

order = order_service.get_for_user(
    trusted_user_id=current_user.id,
    order_id=data["order_id"],
)

if data["amount"] > order.refundable_amount:
    raise BusinessRuleError("amount_exceeds_refundable_balance")

return data
~~~

注意 current_user.id 来自可信会话，而不是让模型填写。

## 工程实践与边界

### 怎么选

| 需求 | 推荐起点 |
| --- | --- |
| 只想让旧系统拿到合法 JSON | JSON Mode + 本地校验 |
| 从文本提取固定字段 | Structured Outputs |
| 为前端生成固定卡片 | Structured Outputs |
| 决定工作流下一状态 | Structured Outputs + 业务状态机校验 |
| 查询实时订单 | Function Calling |
| 发送邮件、退款、删除文件 | Function Calling + 授权与审批 |
| 既要查数据又要输出卡片 | Function Calling 取数，再 Structured Outputs 组织结果 |

### Structured Outputs 的评测

至少统计：

- 请求完成率；
- 拒绝率；
- 截断率；
- 本地 Schema 校验失败率；
- 字段事实准确率；
- enum 混淆矩阵；
- 不同文本长度和语言切片；
- 单次响应延迟与 token。

结构成功率很高，也可能字段事实准确率很低。

### Function Calling 的评测

- 应调用时的调用率；
- 不应调用时的拒调率；
- 工具选择准确率；
- 参数 Schema 通过率；
- 业务校验通过率；
- 越权请求拦截率；
- 工具执行成功率；
- 重复副作用数；
- 端到端任务成功率。

### 本地校验失败怎么处理

可以把简化、非敏感的结构化错误返回模型一次，让它修正：

~~~json
{
  "error": "INVALID_ARGUMENT",
  "field": "reason_code",
  "allowed_values": ["DAMAGED", "WRONG_ITEM", "OTHER"]
}
~~~

要限制修正次数。权限拒绝、危险参数和业务冲突不能靠模型反复猜到通过。

### 安全边界

无论哪种机制，以下值都视为不可信：

- URL；
- 文件路径；
- SQL；
- shell 命令；
- 资源 ID；
- 用户和租户 ID；
- 金额和币种；
- 邮件收件人；
- 权限与角色；
- 回调地址。

Schema 约束“像什么”，授权和业务规则决定“能不能做”。

### 兼容与回退

如果目标不支持 Structured Outputs：

1. 可以降级到 JSON Mode；
2. 保留本地 Schema 校验；
3. 对失败做有限修复；
4. 降级路径单独评测；
5. 高风险工具参数不因降级而放宽校验。

### 数据隐私

Schema 名称、描述和枚举也会发送给模型服务，可能暴露内部业务结构。应：

- 只发送任务需要字段；
- 不在描述中放密钥或内部秘密；
- 按数据分类选择模型服务；
- 日志脱敏；
- 对工具返回做最小化。

## 常见误区

- **“JSON Mode 保证符合 Schema”**：它只解决 JSON 语法，不保证字段契约。
- **“Structured Outputs 保证事实正确”**：结构可以正确，值仍可能错误。
- **“Function Calling 会自动执行函数”**：模型只输出意图，宿主控制执行。
- **“strict 开启后不需要服务端校验”**：授权、金额、归属和资源状态仍要检查。
- **“Function Calling 与 Structured Outputs 完全无关”**：严格工具参数也使用结构约束，但用途不同。
- **“完整 JSON Schema 2020-12 都能用于模型约束”**：供应商通常只支持声明的子集。
- **“解析失败就把字符串修一修执行”**：可能把恶意或错误文本变成危险参数。
- **“结构化就不需要处理拒绝和截断”**：响应状态必须先于对象消费。

## 面试追问

### 1. JSON Mode 和 Structured Outputs 最核心的区别是什么？

JSON Mode 保证目标是合法 JSON；Structured Outputs 进一步按支持的 JSON Schema 子集约束字段、类型、必填和枚举。

### 2. Structured Outputs 能替代 Function Calling 吗？

不能一概替代。前者通常返回结构化答案；后者表达希望宿主调用外部能力。需要读写外部系统时仍要工具调用和执行闭环。

### 3. Function Calling 为什么不能自动执行？

模型参数是不可信输入，执行还涉及当前用户、资源归属、业务状态、副作用和审批。宿主必须保留最终控制权。

### 4. strict 模式后为什么还要本地 Schema 校验？

本地校验形成服务边界，防止 SDK、适配层、版本或截断问题，也让失败可测试。更重要的是，Schema 之外仍有业务和授权校验。

### 5. 结构正确但事实错，怎么处理？

用权威数据源、检索证据、规则或人工审核验证关键字段。不要把 Schema 通过率当作事实准确率。

### 6. 什么时候仍会用 JSON Mode？

目标接口不支持 Structured Outputs、结构很简单、已有成熟本地验证，或作为明确评测过的兼容回退时。

### 7. 如何处理模型拒绝？

先检查响应状态和拒绝字段，不强行解析为业务 Schema；向用户给出安全说明，必要时调整合法请求或转人工。

### 8. 三种机制能否一起用？

可以。例如 Function Calling 查询订单，工具结果返回后，再用 Structured Outputs 生成固定 UI 对象；不支持严格结构的回退路径可能使用 JSON Mode。
