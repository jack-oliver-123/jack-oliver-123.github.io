## 本篇目标

学完这一篇，你应该能够：

1. 解释 Function Calling、Agent Skill 和 MCP 为什么不是三种同层方案。
2. 说清模型、Host、MCP Client、MCP Server 与 Skill 在一次调用中的位置。
3. 说明 Function Calling 只表达调用意图，不会替应用执行函数。
4. 画出三者组合后的完整工具调用闭环。
5. 判断什么时候只用其中一个，什么时候组合，什么时候必须用确定性工作流。
6. 回答结构化输出、工具 Schema、权限、幂等和提示注入等进阶问题。

## 60 秒回答

> 这三个概念不在同一层。Function Calling 是模型 API 与 Host 之间的交互机制：Host 给模型工具定义，模型返回工具名和结构化参数，真正执行仍由 Host 完成。Agent Skill 是以 `SKILL.md` 为入口的任务知识包，用指令、脚本、参考资料和模板告诉 Agent 怎样完成一类任务。MCP 是 Host 内的 Client 与 MCP Server 之间的运行时协议，标准化外部 Tools、Resources 和 Prompts 的发现与调用。
>
> 可以把它们串成一条链：退款 Skill 规定核验、查询、审批、执行和通知流程；模型通过 Function Calling 表达“调用 get_order”；Host 校验后通过 MCP Client 调用订单 Server，再把结果回传模型。这里 Skill 管方法，Function Calling 管模型的动作表达，MCP 管外部能力接入。
>
> 三者都不自动提供业务安全。Host 与服务端仍要过滤可见工具、校验参数和权限、要求高风险确认、设置超时、幂等、补偿和审计。固定强事务流程不应完全交给模型编排；可以让 Skill 做解释、Function Calling 做受控决策、MCP 做标准接入，但业务代码必须守住不变量。

## 详细解析

### 先说人话：操作手册、点单条和后厨接口

想象一家餐厅：

- **Skill** 像服务员的操作手册：怎样接待、确认忌口、处理退菜；
- **Function Calling** 像服务员填写的结构化点单条：菜名、数量、备注；
- **MCP** 像前台与不同后厨之间统一的接单和回执规范；
- **Tool 实现** 才是真正做菜的厨房设备和人员。

服务员写下“宫保鸡丁一份”不代表菜已经做好。

同样，模型返回函数名和参数，不代表函数已经执行。

操作手册也不能替代后厨，统一接口也不会自动决定先确认过敏信息还是先下单。

### 三个概念各自回答什么问题

```text
Skill：这类任务应该怎样完成？

Function Calling：模型这一刻建议调用什么，参数是什么？

MCP：Host 怎样发现、连接并调用能力提供方？
```

这是最值得背下来的三句话。

### 什么是 Function Calling

**Function Calling**，也常被称为 Tool Calling，是一种模型 API 交互模式。

应用把工具描述传给模型：

```json
{
  "name": "get_weather",
  "description": "查询指定城市的当前天气",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "城市名称"
      }
    },
    "required": ["city"]
  }
}
```

用户问：

```text
杭州现在需要带伞吗？
```

模型可能返回结构化意图：

```json
{
  "name": "get_weather",
  "arguments": {
    "city": "杭州"
  }
}
```

然后由应用：

1. 解析调用；
2. 校验参数；
3. 检查权限；
4. 执行真实工具；
5. 把结果送回模型；
6. 请求模型生成最终回答。

### Function Calling 不会替你执行函数

这是最高频误区。

模型产生的是：

```text
调用建议 + 结构化参数
```

不是：

```text
已经执行的业务结果
```

如果模型输出：

```json
{
  "name": "transfer_money",
  "arguments": {
    "to": "B",
    "amount": 1000
  }
}
```

应用仍必须判断：

- 当前用户是谁；
- 是否有转账权限；
- 余额是否足够；
- 收款人是否可信；
- 是否需要二次确认；
- 幂等键是什么；
- 如何审计。

模型输出不能直接当作授权凭证。

### Function Calling 是一种能力模式，不是一套完全统一的跨平台协议

不同模型 API 对以下细节可能不同：

- 工具定义字段；
- 支持的 JSON Schema 子集；
- 工具名称限制；
- 单次或并行调用；
- 调用 ID；
- 流式参数事件；
- 结果回传消息；
- 强制调用或禁止调用的配置；
- 严格结构约束；
- 错误和结束原因。

因此 Host 常需要模型适配层。

不能把某一家 API 的字段直接当成所有 Function Calling 的通用网络标准。

### 什么是 Agent Skill

本文中的 Agent Skill 指 Agent Skills 文件规范。

它是一个目录，至少包含：

```text
skill-name/
└── SKILL.md
```

还可以附带：

```text
scripts/
references/
assets/
```

Skill 的核心价值是把任务方法和资源复用起来。

例如一个发布 Skill 可以说明：

```text
1. 读取版本规范。
2. 检查工作区和测试结果。
3. 生成变更说明。
4. 创建候选版本。
5. 等待审批。
6. 发布后检查监控。
7. 异常时停止并回滚。
```

它可以指导 Agent 使用工具，却不定义模型 API 的工具调用事件，也不定义远程 Server 的传输。

### 什么是 MCP

MCP 是 Host 中的 Client 与 MCP Server 之间的协议。

它基于 JSON-RPC 消息，并定义：

- 初始化和版本协商；
- 能力协商；
- Tools；
- Resources；
- Prompts；
- 客户端和服务端的可选能力；
- 传输；
- 请求、响应、通知和错误语义。

一个 MCP Server 可以暴露：

```text
tools/list
tools/call
resources/list
resources/read
prompts/list
prompts/get
```

具体可用方法取决于协商出的能力。

### 三者的完整对比

| 维度 | Function Calling | Agent Skill | MCP |
| --- | --- | --- | --- |
| 所在层 | 模型 API 交互层 | 任务知识层 | 能力接入协议层 |
| 核心问题 | 模型建议调用什么 | 任务应该怎样做 | Host 怎样连接和调用能力 |
| 主要双方 | 模型与 Host | Host/Agent 与技能包 | MCP Client 与 Server |
| 主要载体 | 工具定义、调用事件、结果消息 | `SKILL.md` 与目录资源 | JSON-RPC 消息与传输 |
| 是否定义网络连接 | 否，依赖模型 API | 否 | 是 |
| 是否执行真实操作 | 否，由 Host 执行 | 不一定，脚本需 Host 执行 | Server 接到调用后执行或转发 |
| 是否描述完整业务流程 | 通常不描述 | 可以 | 通常不描述 |
| 是否携带参考资料 | 只通过上下文传入 | 可以 | Resources 可运行时读取 |
| 主要兼容问题 | API 格式和 Schema 子集 | Host 的发现与加载实现 | 协议版本、能力与传输 |
| 主要安全问题 | 误调用、参数、结果注入 | 内容与脚本供应链 | Server 信任、授权、会话和工具执行 |

### 三者组合后的完整链路

以“帮我退款”为例。

#### 阶段一：Skill 提供任务方法

Host 根据用户请求命中 `refund-review` Skill。

正文规定：

```text
先核验订单和政策；
参数不全时询问；
创建退款前必须展示金额并确认；
失败后查询状态，不能盲目重试。
```

#### 阶段二：Host 获取 MCP Tools

Host 通过 MCP Client 从 Server 获取：

```text
get_order
get_refund_policy
create_refund
get_refund_status
```

#### 阶段三：Host 转换工具定义

Host 把候选 MCP Tools 转换成当前模型 API 接受的 Function Calling 定义。

这一步可能需要：

- 工具名映射；
- JSON Schema 降级；
- 描述裁剪；
- 权限过滤；
- 命名空间处理。

#### 阶段四：模型表达调用意图

模型返回：

```json
{
  "name": "get_order",
  "arguments": {
    "order_id": "A1001"
  }
}
```

#### 阶段五：Host 执行安全检查

```text
工具是否在本回合允许列表
  -> 参数 Schema 是否合法
  -> 当前身份能否读取该订单
  -> 订单是否属于当前租户
  -> 是否触发限流
```

#### 阶段六：MCP 调用

Host 通过对应 MCP Client 发出 `tools/call`。

MCP Server 调用订单系统并返回结果。

#### 阶段七：工具结果回传模型

Host 对结果做校验、脱敏和裁剪，再使用模型 API 规定的工具结果消息回传。

#### 阶段八：继续循环或结束

模型可能继续请求政策工具，或者发现缺少用户确认。

最终执行写操作前，Host 的确定性状态机再次检查确认和幂等键。

完整链路可以记为：

```text
Skill 指导方法
      ↓
模型 Function Calling 表达动作
      ↓
Host 校验和路由
      ↓
MCP 调用外部能力
      ↓
Host 回传结果
      ↓
模型继续判断
```

### 三者并非必须一起出现

#### 只有 Function Calling

应用内有一个本地函数：

```text
calculator(expression)
```

Host 直接执行，不经过 MCP，也不需要 Skill。

#### 只有 Skill

写作 Skill 指导 Agent 按公司风格改稿，所需信息都在本地文件和上下文中。

没有外部工具，也就不需要 Function Calling 或 MCP。

#### 只有 MCP

用户点击固定按钮“读取当前项目状态”。

程序直接调用 MCP Resource 或 Tool，不需要模型选择，也不需要 Skill。

#### Function Calling + MCP

模型自由选择数据库、搜索或文件工具，Host 通过 MCP 调用。

任务简单时无需独立 Skill。

#### Skill + 普通工具

Skill 指导 Agent 使用应用内置函数，工具没有通过 MCP 暴露。

#### Skill + MCP，不使用模型 Function Calling

Skill 可以供人类或确定性工作流参考，程序按固定步骤调用 MCP。

也可以由模型生成计划，但关键工具由工作流状态机决定。

### Function Calling 与结构化输出有什么区别

结构化输出通常要求模型返回符合给定 Schema 的数据，例如：

```json
{
  "category": "refund",
  "urgency": "high"
}
```

它主要回答：

```text
返回数据长什么样？
```

Function Calling 还承载动作语义：

```text
模型是否建议调用工具？
调用哪个工具？
调用 ID 是什么？
工具结果如何回到同一次交互？
```

可以用结构化输出自建动作协议，但需要自己补齐调用状态、结果关联、并行、取消和错误语义。

### Tool Schema 在三者间怎样流动

常见路径：

```text
MCP Server 的 inputSchema
        ↓ Host 转换
模型 API 的 Function Calling Schema
        ↓ 模型生成
arguments
        ↓ Host 校验
MCP tools/call arguments
```

需要注意：

- 转换可能有损；
- 模型 API 不一定支持完整 JSON Schema；
- `description` 对工具选择很重要；
- Schema 合法不代表业务合法；
- Server 必须重新校验，不可信任 Host 或模型。

### Skill 如何引用 Tool

差的写法：

```text
调用一个合适的退款工具处理。
```

更好的写法：

```text
先调用只读订单查询能力。
如果订单不存在或不属于当前用户，停止。
确认政策和金额后，向用户展示摘要。
只有收到明确确认并通过 Host 审批，才能请求创建退款。
写操作状态不明时查询状态，禁止直接重放。
```

Skill 应描述语义和边界，不应假设展示名称会长期保持不变。

Host 可以维护稳定的能力别名到具体 MCP Tool 的映射。

### 谁负责工具选择

不一定总是模型。

可以有三种模式：

| 模式 | 选择者 | 适用场景 |
| --- | --- | --- |
| 模型自主 | 模型通过 Function Calling 选择 | 开放、低风险、需要语义判断 |
| 规则路由 | 业务代码 | 意图明确、固定操作 |
| 混合模式 | 规则先缩小集合，模型再选择 | 工具多、需要安全与灵活性平衡 |

高风险系统常用混合模式：

```text
代码决定允许哪些动作，模型只在允许集合内选择。
```

### 一次工具循环的状态机

生产系统不应只写一个无限 `while`。

可以显式定义：

```text
WAITING_FOR_MODEL
  -> TOOL_CALL_PROPOSED
  -> VALIDATING
  -> WAITING_FOR_CONFIRMATION
  -> EXECUTING
  -> TOOL_RESULT_READY
  -> WAITING_FOR_MODEL
  -> COMPLETED
```

异常状态包括：

```text
REJECTED
TIMED_OUT
CANCELED
UNKNOWN_SIDE_EFFECT
ESCALATED
```

Skill 可以解释流程，但 Host 状态机负责强制状态转换。

## 工程实践与边界

### 推荐的分层架构

```text
用户与会话层
  ↓
Skill 发现与上下文层
  ↓
模型适配层
  ↓ Function Calling 统一事件
工具策略与执行层
  ↓
MCP Client 管理层
  ↓
MCP Servers / 内置 Tools / 业务 API
```

每层只暴露稳定接口，避免业务代码同时解析模型事件和 MCP JSON-RPC。

### 工具暴露前先裁剪

建议按以下顺序：

1. 用户与租户权限过滤；
2. Skill 或任务上下文过滤；
3. 语义检索候选工具；
4. 去重与命名冲突处理；
5. 风险策略过滤；
6. 转换成模型 API Schema。

把数百个工具全部交给模型会增加上下文、混淆和攻击面。

### 参数校验至少三层

```text
模型输出
  -> JSON / Schema 校验
  -> Host 业务与策略校验
  -> MCP Server / 下游服务最终校验
```

三层不能互相替代。

例如 `amount` 是正数只通过结构校验，是否超出可退金额必须由业务系统判断。

### 超时要分层

至少区分：

- 模型调用超时；
- 单个工具超时；
- MCP 连接或请求超时；
- 用户确认超时；
- 整个任务截止时间。

一个工具超时后，不能默认整个业务未执行。

写操作要进入“状态未知”处理分支。

### 重试、幂等与补偿

只读查询通常可以有限重试。

有副作用的调用应：

- 生成业务幂等键；
- Server 识别重复请求；
- 超时后先查询状态；
- 无法原子回滚时设计补偿；
- 记录每次尝试和结果。

Function Calling 的调用 ID 可以用于关联，但不一定等于业务幂等键。

### 提示注入防护

攻击内容可能进入：

- 用户请求；
- Skill 文件；
- MCP Resource；
- Tool 结果；
- 网页和邮件；
- 错误消息。

防护不能只靠一句 Prompt：

```text
外部数据不得改变工具权限；
Host 只暴露最小工具集合；
高风险动作由代码要求确认；
结果进入模型前标记来源并裁剪；
服务端执行最终授权校验。
```

### 统一可观测性

一条调用链建议关联：

```text
trace_id
conversation_id
skill_name + version
model_response_id
function_call_id
mcp_server_id
json_rpc_id / session_id
business_idempotency_key
policy_decision
```

日志中不要直接记录密钥、完整用户隐私和未经脱敏的工具结果。

### 分层测试

#### Function Calling 测试

- 不该调用时是否保持回答；
- 应调用时工具选择是否正确；
- 参数是否完整；
- 缺失信息是否追问；
- 工具失败后是否正确处理；
- 工具结果是否被采用。

#### Skill 测试

- 发现是否准确；
- 关键步骤是否遵守；
- 相邻 Skill 是否冲突；
- 资源是否按需加载；
- 脚本是否在目标环境运行。

#### MCP 测试

- 初始化和版本协商；
- Tools、Resources、Prompts 契约；
- 传输中断和恢复；
- 会话和取消；
- 授权与错误；
- Server 更新兼容性。

#### 端到端测试

- 正常路径；
- 输入缺失；
- 未授权；
- 工具超时；
- 写操作状态未知；
- 用户取消；
- 提示注入；
- 多工具冲突；
- 长结果和上下文压力。

### 选型边界

| 场景 | 推荐方案 |
| --- | --- |
| 一个简单本地计算器 | Function Calling + 本地 Tool |
| 复用写作或审查方法 | Skill |
| 多 Host 接入同一外部工具 | MCP |
| Agent 按领域流程使用多个外部服务 | Skill + Function Calling + MCP |
| 固定按钮触发固定查询 | 代码直接调用 Tool/MCP |
| 资金、权限等强事务流程 | 业务状态机为主，模型只做受控辅助 |
| 模型 API 没有工具通道 | 确定性路由，或谨慎使用严格结构化适配 |

## 常见误区

### 误区一：三者都是工具调用协议

错误。Function Calling 是模型交互模式，Skill 是任务知识包，MCP 才是 Client 与 Server 的能力接入协议。

### 误区二：模型返回 Function Call 后函数已经执行

没有。Host 必须校验、授权和执行，再把结果回传。

### 误区三：MCP 取代 Function Calling

MCP 不规定模型怎样表达调用意图。Host 仍可使用 Function Calling、规则或工作流决定何时调用 MCP。

### 误区四：Skill 取代 MCP

Skill 的指令不能替代远程连接、能力协商和工具实现。

### 误区五：MCP 自动让任何模型会用工具

MCP 把工具接到 Host，模型 API 和模型行为仍需适配和评测。

### 误区六：Schema 合法就可以执行

还要进行业务规则、身份、租户、额度、审批和幂等校验。

### 误区七：Skill 中写了“先确认”就足够安全

模型可能漏掉该步骤。高风险确认要由 Host 状态机强制。

### 误区八：工具结果比用户输入可信

工具结果也可能被污染或包含提示注入，只能把它当带来源的数据。

### 误区九：所有写操作失败都可以重试

连接失败时业务可能已经完成。先用幂等键和状态查询确定结果，再决定重试或补偿。

## 面试追问

### 追问一：一句话区分三者

> Skill 管任务方法，Function Calling 管模型如何表达工具调用意图，MCP 管 Host 如何标准化接入外部能力。

### 追问二：Function Calling 必须经过 MCP 吗

不一定。

Host 可以直接执行进程内函数、HTTP API 或数据库封装；MCP 只是标准化能力接入的一种方式。

### 追问三：使用 MCP 必须有大模型吗

不需要。

普通应用或确定性工作流也可以通过 MCP Client 列出、读取和调用能力。

### 追问四：Skill 必须依赖工具吗

不需要。

纯写作、分析或格式规范 Skill 可能只需要指令、参考资料和模板。

### 追问五：为什么调用 ID 不能直接当业务幂等键

调用 ID 主要关联一次模型工具调用；模型重试、会话恢复或不同 Host 可能生成新 ID。业务幂等键要围绕“同一业务动作”稳定生成并由服务端识别。

### 追问六：工具列表很多时怎么办

先按身份和任务过滤，再做能力检索和去重，只把小而相关的候选集合交给模型。Skill 可提供任务语境，但不能绕过权限过滤。

### 追问七：结构化输出能模拟 Function Calling 吗

可以模拟基本工具名和参数，但 Host 要自己定义调用与文本区分、调用 ID、结果回传、多轮、并行、取消和错误语义。生产复杂度通常更高。

### 追问八：在哪里做最终权限校验

Host 要在调用前做策略校验，MCP Server 和下游业务系统还必须根据真实身份再次授权。最终数据和副作用的所有者不能信任模型输入。

### 追问九：三者升级时如何回归

分别固定 Skill、模型适配器、MCP 协议和 Server 版本，运行分层契约测试，再跑包含正常、失败、安全和状态恢复的端到端任务集。

### 可直接背诵的完整答案

> Function Calling、Skill 和 MCP 不在同一层。Function Calling 是模型 API 与 Host 的交互模式，模型根据工具 Schema 返回工具名和参数，真正执行由 Host 完成；Agent Skill 是以 `SKILL.md` 为入口的任务知识包，封装步骤、脚本、资料和模板；MCP 是 Host 中 Client 与 Server 的运行时协议，标准化 Tools、Resources、Prompts 等外部能力的发现和调用。
>
> 三者可以串起来：退款 Skill 规定核验和审批流程，模型用 Function Calling 选择订单工具，Host 校验后通过 MCP 调用订单 Server，再把结果回传模型。但它们都不自动解决安全问题。Host 和服务端仍要做工具裁剪、Schema 与业务校验、身份授权、高风险确认、超时、幂等、补偿和审计。简单本地工具可以只用 Function Calling，复用任务方法可以只用 Skill，标准化外部能力可以只用 MCP；强事务流程必须由确定性业务状态机守住不变量。
