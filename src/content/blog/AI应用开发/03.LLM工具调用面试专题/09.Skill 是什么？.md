## 本篇目标

学完这一篇，你应该能够：

1. 发现“Skill 是什么”这道题有歧义，并先确认面试官指哪一种 Skill。
2. 解释 Agent Skills 开放规范中的目录、`SKILL.md`、元数据和渐进式加载。
3. 区分“平台内置技能”与遵循 Agent Skills 规范的可移植技能包。
4. 区分 Agent Skills 文件规范与 A2A `AgentSkill` 能力描述对象。
5. 说明 Skill、Prompt、Tool、代码库和模型能力之间的关系。
6. 给出 Skill 的设计、触发、执行、安全、版本与评测方法。

## 60 秒回答

> “Skill”在 Agent 领域不是只有一个定义，回答前要先确认语境。第一种是 Agent Skills 开放文件规范：一个 Skill 是可分发的目录，至少包含带 YAML frontmatter 的 `SKILL.md`；`name` 和 `description` 用于发现，Markdown 正文提供指令，还可以附带 `scripts/`、`references/`、`assets/` 等资源。Host 通常先读取简短元数据，命中任务后再加载正文和需要的文件，这叫渐进式披露。
>
> 第二种是某个平台泛称的“技能”，可能只是内置工具、工作流、Prompt 模板或权限包，不一定兼容上述目录规范。第三种是 A2A 协议 Agent Card 里的 `AgentSkill`，它只是远程 Agent 的能力描述对象，包含 id、name、description、tags，以及可选示例、输入输出媒体类型和安全要求；它不是 `SKILL.md` 目录，也不会自动携带实现代码。
>
> 所以我会把 Skill 理解为“让 Agent 更稳定完成某类任务的可发现能力封装”，但落到工程上必须明确采用哪套规范、由谁加载、依赖哪些工具、如何授权和执行。安装 Skill 不会修改模型权重，也不会天然获得权限；其中的脚本和外部资料都要按不可信代码与数据治理。

## 详细解析

### 先说人话：Skill 像“岗位操作手册”，但这个词有三种常见用法

假设公司新来一名客服。

他会读写、会推理，就像大模型具备通用语言能力。

但“如何处理退款”还需要公司自己的知识：

- 先核对什么信息；
- 哪些订单可以退款；
- 调用哪个系统；
- 超过多少金额需要审批；
- 失败后如何回滚；
- 最终怎样向用户解释。

把这套知识整理成可复用说明，可以称为一个 Skill。

但在不同项目中，“Skill”可能分别指：

| 语境 | Skill 是什么 | 是否有统一文件格式 |
| --- | --- | --- |
| Agent Skills 规范 | 以 `SKILL.md` 为入口的目录包 | 有 |
| 某个平台的 Skills 功能 | 平台自行定义的能力、工具或工作流 | 不一定 |
| A2A `AgentSkill` | Agent Card 中描述远程 Agent 能力的对象 | 有协议字段，但不是目录包 |
| 日常语言 | 模型或 Agent 擅长的能力 | 没有 |

面试中如果直接把这四种含义混在一起，后续与 MCP、A2A 的比较都会出错。

### 第一种：Agent Skills 文件规范

Agent Skills 规范定义了一种开放、轻量的技能打包方式。

最小结构是：

```text
refund-review/
└── SKILL.md
```

更完整的结构可以是：

```text
refund-review/
├── SKILL.md
├── scripts/
│   ├── validate_order.py
│   └── calculate_refund.py
├── references/
│   ├── refund-policy.md
│   └── regional-rules.md
└── assets/
    ├── approval-template.md
    └── response-example.json
```

规范要求至少存在 `SKILL.md`。

其他目录是可选的，也可以存在额外文件或目录。

### `SKILL.md` 由什么组成

`SKILL.md` 包含两部分：

1. YAML frontmatter；
2. Markdown 正文。

示意如下：

```markdown
---
name: refund-review
description: 审核电商退款申请；当用户要求退款、取消订单或查询退款资格时使用。
license: Apache-2.0
compatibility: 需要订单查询工具和 Python 运行环境
metadata:
  owner: commerce-team
  version: "1.3.0"
---

# 退款审核

先确认订单号和当前用户身份，再读取退款政策……
```

规范中的必填 frontmatter 字段是：

- `name`；
- `description`。

还定义了若干可选字段，例如：

- `license`：许可信息；
- `compatibility`：环境或产品兼容要求；
- `metadata`：额外键值元数据；
- `allowed-tools`：实验性的预授权工具列表。

注意：不同 Host 对可选字段的支持程度可能不同，不能看到字段就假设所有运行时都执行相同语义。

### `name` 和 `description` 为什么非常重要

它们通常不是给最终用户看的装饰，而是 Skill 的“检索入口”。

Host 可能先只把这些短元数据放进上下文：

```text
name: refund-review
description: 审核退款申请；当用户要求退款、取消订单或查询退款资格时使用。
```

模型或路由器根据当前任务判断是否加载正文。

一个差的描述：

```text
description: 帮助处理业务。
```

问题是：

- 范围太宽；
- 不知道什么时候触发；
- 容易和其他 Skill 冲突；
- 无法评测漏触发和误触发。

一个更好的描述：

```text
description: 审核电商订单退款资格并计算可退金额；当用户提出退款、撤销交易或询问退款政策时使用，不用于换货与售后维修。
```

它同时说明：

- 做什么；
- 何时使用；
- 何时不要使用。

### 什么是渐进式披露

**渐进式披露**是指按任务需要逐步加载信息，而不是开局把所有内容塞进上下文。

可以分成三层：

```text
第一层：元数据
  name + description
        ↓ 命中任务
第二层：SKILL.md 正文
  步骤 + 边界 + 文件导航
        ↓ 确有需要
第三层：脚本、参考资料和资产
  scripts/ + references/ + assets/
```

这样做的价值包括：

- 减少上下文占用；
- 降低无关指令干扰；
- 允许一个 Skill 携带深度资料；
- 更容易维护清晰入口；
- 降低每次调用的成本。

但要注意，渐进式披露是 Host 和 Agent 的加载行为，不是把文件放在目录里就自动发生。

### 四类内容分别放在哪里

#### 1. `SKILL.md` 正文

适合放：

- 任务目标；
- 适用与禁用场景；
- 需要判断的流程；
- 关键安全规则；
- 资源导航；
- 完成标准。

正文应足够短，让 Agent 快速建立工作模型。

#### 2. `scripts/`

适合放：

- 重复执行的确定性转换；
- 格式校验；
- 批量处理；
- 精确计算；
- 可自动验证的步骤。

例如退款金额计算最好由代码完成，而不是让模型心算。

#### 3. `references/`

适合放：

- API 文档；
- 业务政策；
- 数据字典；
- 领域术语；
- 长篇操作说明。

正文应明确“什么情况下读取哪份资料”，避免 Agent 无目的地加载所有文件。

#### 4. `assets/`

适合放：

- 输出模板；
- 项目骨架；
- 示例配置；
- 样式资源；
- 需要复制或加工的成品材料。

参考资料主要供理解，资产主要供产出，两者不要混用。

### Skill 的运行流程

一个典型 Host 可能这样处理：

```text
发现 Skill
  -> 读取并验证元数据
  -> 建立名称和描述索引
  -> 收到用户任务
  -> 选择候选 Skill
  -> 加载 SKILL.md
  -> 按指令读取必要资源
  -> 调用现有工具或运行脚本
  -> 校验结果
  -> 返回用户
```

规范定义文件格式，但以下行为仍由 Host 决定：

- 从哪里发现 Skill；
- 由规则、模型还是检索器选择；
- 正文怎样进入上下文；
- 脚本能否执行；
- 能访问哪些文件和网络；
- 工具权限如何授予；
- 冲突指令如何处理；
- 运行后如何审计。

### Skill 不是只保存一段 Prompt

Prompt 通常是一段传给模型的指令或上下文。

规范化 Skill 可以同时包含：

```text
指令
+ 确定性脚本
+ 领域文档
+ 输出模板
+ 元数据
+ 资源导航
```

所以可以说 Prompt 是 Skill 的一部分，但不能把二者完全等同。

反过来，一段很短、没有复用价值的 Prompt，也不一定需要打包成 Skill。

### Skill 不是 Tool

Tool 是可被调用的执行能力，例如：

```text
get_order(order_id)
create_refund(order_id, amount)
send_email(to, subject, body)
```

Skill 更像告诉 Agent：

```text
处理退款时，先查订单，再判断资格，超过限额时请求审批，最后创建退款并通知用户。
```

对比如下：

| 维度 | Skill | Tool |
| --- | --- | --- |
| 解决的问题 | 怎样完成一类任务 | 执行一个具体动作 |
| 核心形态 | 指令与资源包 | 函数、API 或命令 |
| 是否会执行代码 | 不一定 | 通常会 |
| 是否可跨多个工具 | 可以 | 单个工具一般职责明确 |
| 是否自动获得权限 | 否 | 也否，调用仍需授权 |

Skill 可以指导 Agent 使用 Tool，Tool 不必知道是谁指导了它。

### Skill 不是模型能力升级

安装 Skill 不会修改模型权重。

它提供的是推理时上下文、资源和程序，不是重新训练。

因此它可能改善：

- 流程一致性；
- 领域术语理解；
- 工具使用顺序；
- 输出格式；
- 可复用性。

但不能保证：

- 模型能完整遵守所有指令；
- 模型突然获得原本没有的视觉或语音接口；
- 脚本能在当前系统正常运行；
- 工具自动拥有用户权限；
- 所有任务都能正确完成。

### 第二种：平台所说的“技能”

很多产品也使用 Skill 这个名称，但内部含义可能完全不同：

- 一个内置函数；
- 一组 API 权限；
- 固定工作流；
- Prompt 模板；
- 插件；
- 远程 Agent；
- 一套 UI 操作。

判断平台 Skill 时，应查四件事：

1. 有没有公开格式；
2. 能不能导出和迁移；
3. 谁负责发现与触发；
4. 执行权限和运行环境是什么。

如果平台没有声明兼容 Agent Skills 规范，就不能因为名字相同而假设其目录和字段相同。

### 第三种：A2A 协议中的 `AgentSkill`

A2A 是 Agent 与 Agent 协作协议。

远程 Agent 会通过 **Agent Card** 描述自身。

Agent Card 中可以列出 `AgentSkill` 对象。A2A 1.0 定义的核心字段包括：

| 字段 | 是否必需 | 含义 |
| --- | --- | --- |
| `id` | 是 | 能力的唯一标识 |
| `name` | 是 | 人类可读名称 |
| `description` | 是 | 能力的详细说明 |
| `tags` | 是 | 能力关键词 |
| `examples` | 否 | 示例提示或场景 |
| `inputModes` | 否 | 该能力支持的输入媒体类型 |
| `outputModes` | 否 | 该能力支持的输出媒体类型 |
| `securityRequirements` | 否 | 该能力要求的安全方案 |

示意：

```json
{
  "id": "route-planning",
  "name": "路线规划",
  "description": "根据起点、终点和偏好规划路线",
  "tags": ["地图", "路线", "出行"],
  "examples": ["规划一条避开收费站的路线"],
  "inputModes": ["text/plain", "application/json"],
  "outputModes": ["application/json"]
}
```

这个对象的作用是让另一个 Agent 发现和理解远程能力。

它不是：

- `SKILL.md` 文件；
- 可下载的目录包；
- 自动执行的脚本；
- 授权凭证；
- 完整实现代码。

可以把关系记成：

```text
Agent Skills 目录：把“如何做”打包给 Host
A2A AgentSkill：在 Agent Card 中声明“我能做什么”
```

### 三种 Skill 的对照

| 维度 | Agent Skills 规范 | 平台 Skill | A2A `AgentSkill` |
| --- | --- | --- | --- |
| 目标 | 分发任务知识与资源 | 取决于平台 | 描述远程 Agent 能力 |
| 主要载体 | 目录和 `SKILL.md` | 平台自定义 | Agent Card 字段 |
| 是否携带脚本 | 可以 | 可能 | 不携带实现 |
| 是否定义远程调用 | 否 | 可能 | 由 A2A 其他对象和绑定完成 |
| 是否可移植 | 规范兼容时较强 | 取决于平台 | 协议兼容时可发现 |
| 是否自动获得权限 | 否 | 否 | 否 |
| 核心问题 | 怎样完成任务 | 平台提供什么能力 | 远程 Agent 声明会什么 |

### 一个端到端案例

用户要求：

```text
审核这笔大额退款，并让合规 Agent 给出风险意见。
```

系统可能同时使用三种概念：

1. 本地加载 `refund-review` Agent Skills 目录；
2. Skill 正文要求先调用订单查询 Tool；
3. Host 根据 A2A Agent Card 找到具有 `refund-risk-review` AgentSkill 的远程合规 Agent；
4. Host 通过 A2A 发送任务；
5. 合规 Agent 内部可能再使用自己的 Skills 和 MCP Tools；
6. 本地 Agent 汇总结果并请求人工批准。

名字都叫 Skill，但它们位于完全不同的层。

### Skill 与代码工作流如何分工

不是所有流程都适合写在自然语言里。

| 内容 | 更适合放哪里 |
| --- | --- |
| “先了解用户真实诉求” | Skill 指令 |
| 金额精确计算 | 确定性代码 |
| 数据库查询 | Tool 或 MCP Server |
| 固定事务顺序 | 工作流引擎或业务代码 |
| 领域政策说明 | `references/` |
| 输出报告模板 | `assets/` |
| 何时升级人工 | Skill + 代码策略共同约束 |

经验原则：

```text
需要判断和解释的内容，Skill 可以指导；
必须精确、可审计、强事务的内容，交给代码保证。
```

## 工程实践与边界

### 从一个窄任务开始设计

差的 Skill：

```text
name: business-helper
description: 处理公司的所有业务任务。
```

好的 Skill 通常有清晰边界：

```text
name: refund-eligibility-review
description: 判断已支付电商订单是否符合退款政策并生成审核意见；不创建实际退款，不处理换货。
```

范围越清晰，越容易触发、授权、测试和维护。

### 正文推荐包含的内容

可以采用以下骨架：

```text
任务目标
适用场景
不适用场景
所需输入
执行步骤
何时读取哪些资料
可调用的工具与边界
失败和升级人工条件
结果验收标准
安全与隐私要求
```

不要只写“请认真完成任务”这种无法验证的要求。

### 把 Skill 当成代码供应链治理

Skill 虽以 Markdown 为入口，却可能：

- 指导 Agent 读取敏感文件；
- 引导调用高权限工具；
- 附带可执行脚本；
- 引用外部内容；
- 修改输出或安全判断；
- 在升级时改变行为。

安装第三方 Skill 前应检查：

1. 来源与维护者；
2. 许可证；
3. 所有脚本；
4. 外部下载与网络访问；
5. 文件与密钥读取；
6. 引用资源中的提示注入；
7. 版本差异；
8. 与本地高优先级规则的冲突。

### 脚本在沙箱中运行

根据任务授予最小权限：

- 只读或限定目录写入；
- 默认关闭不必要网络；
- 不把密钥放进参数或日志；
- 限制进程、CPU、内存和时长；
- 固定依赖版本；
- 对产物做格式与内容校验；
- 高风险副作用要求审批。

`allowed-tools` 之类的声明也不能替代运行时授权。

### 版本管理

一个 Skill 的行为可能由多个部分共同决定：

```text
SKILL.md
+ scripts/
+ references/
+ assets/
+ Host 版本
+ 模型与工具版本
```

版本升级时要：

- 固定来源和版本；
- 查看所有文件差异；
- 记录兼容条件；
- 对关键任务做回归；
- 保留回滚能力；
- 将执行版本写入审计日志。

仅给 `SKILL.md` 写一个版本号，不能代表整条运行链完全可复现。

### 评测不能只看最终答案

至少分层评估：

| 维度 | 示例指标 |
| --- | --- |
| 发现 | 应触发率、误触发率、漏触发率 |
| 指令遵循 | 关键步骤完成率、越界率 |
| 工具使用 | 工具选择准确率、参数错误率 |
| 脚本执行 | 成功率、超时率、结果一致性 |
| 安全 | 越权率、确认绕过率、注入成功率 |
| 任务结果 | 正确率、人工接受率、返工率 |
| 成本 | 上下文 token、延迟、工具调用数 |

测试集要包含：

- 明确应该触发；
- 明确不应该触发；
- 与相邻 Skill 容易混淆；
- 输入缺失；
- 工具失败；
- 恶意外部文档；
- 高风险请求；
- 多 Skill 冲突。

### 处理多个 Skill 的冲突

不要简单把所有正文拼接在一起。

可以采用：

1. 先按元数据召回少量候选；
2. 检查权限和依赖是否满足；
3. 根据任务选主 Skill；
4. 仅在职责互补时加载辅助 Skill；
5. 高优先级系统和安全规则始终覆盖 Skill；
6. 冲突无法判定时询问用户或转人工。

### 什么时候不需要 Skill

以下场景可能无需单独打包：

- 一次性、非常短的提示；
- 完全固定且已由代码可靠实现的流程；
- 只有一个简单函数调用；
- 没有复用、分发和维护需求；
- Host 根本不支持发现或加载该格式。

Skill 是工程工具，不是所有 Prompt 都必须套上的名词。

## 常见误区

### 误区一：所有叫 Skill 的东西都遵循同一规范

错误。必须明确是 Agent Skills 目录、平台自定义功能，还是 A2A `AgentSkill`。

### 误区二：Skill 就是一段 Prompt

不完整。Agent Skills 规范允许组合指令、脚本、参考资料、模板和元数据。

### 误区三：A2A `AgentSkill` 可以下载并执行

错误。它是 Agent Card 中的能力描述，不是实现包。

### 误区四：安装 Skill 会让模型重新训练

不会。它在运行时提供上下文和资源，不修改模型权重。

### 误区五：有 `scripts/` 就应该自动执行所有脚本

不应该。正文应明确何时使用，Host 还要进行来源审查、授权和沙箱控制。

### 误区六：`description` 写得越宽，覆盖越全面

描述过宽会制造误触发和冲突。应同时写清能力、触发条件与边界。

### 误区七：Markdown 没有安全风险

Markdown 可以包含恶意指令，Skill 还可能附带代码和外部引用，必须按供应链内容治理。

### 误区八：Skill 规定了操作步骤，就能替代服务端权限

自然语言指令不能充当授权系统。Tool 和业务服务仍需校验身份、权限、业务规则和审批状态。

### 误区九：渐进式披露由规范自动执行

规范提供适合渐进加载的结构，真正的发现、读取和上下文管理由 Host 实现。

## 面试追问

### 追问一：面试官只问“Skill 是什么”，怎样开场最好

可以回答：

> Skill 这个词有多种语义。您是想问以 `SKILL.md` 为入口的 Agent Skills 文件规范，还是某个平台的技能功能，或者 A2A Agent Card 里的 `AgentSkill` 描述对象？如果是 Agent Skills 规范，它是可发现、可按需加载的任务知识与资源目录。

这既纠正歧义，也自然引出标准答案。

### 追问二：Agent Skills 的最小合法结构是什么

一个目录，至少包含 `SKILL.md`。

`SKILL.md` 需要 YAML frontmatter 和 Markdown 正文，其中 `name` 与 `description` 是必填字段。

### 追问三：渐进式披露怎样节省上下文

Host 先索引所有 Skill 的名称和描述；只有命中当前任务时才加载正文；正文再按需要引用脚本、资料或资产。这样不必把整个技能库一次性放进模型上下文。

### 追问四：Skill 与 Tool 最核心的区别是什么

Skill 主要表达“如何完成一类任务”，Tool 主要提供“执行一个具体动作”的接口。Skill 可以组织多个 Tool，但本身不一定执行代码。

### 追问五：A2A `AgentSkill` 与 `SKILL.md` 能否一一映射

不一定。

一个远程 AgentSkill 背后可能组合多个内部 Skills、Tools 和工作流；一个本地 `SKILL.md` 也可能指导调用多个远程 Agent。两者粒度和目的不同。

### 追问六：怎样降低 Skill 误触发

缩小职责范围，在 `description` 中写明正向触发和排除条件，准备相邻任务反例，使用候选检索加二次判断，并评测误触发率与漏触发率。

### 追问七：为什么精确计算要放脚本

脚本可复现、可测试，适合算术、格式转换和批处理；模型更适合理解意图和处理例外。最终仍要校验脚本输入、运行权限和输出。

### 追问八：Skill 如何与 MCP 组合

Skill 可以写明完成任务的步骤，并指导 Agent 调用由 MCP Server 暴露的 Tools 或读取 Resources。Skill 不负责建立 MCP 连接，MCP 也不负责定义完整业务流程。

### 追问九：怎样审查第三方 Skill

检查来源、许可、全部指令和脚本、外部依赖、网络与文件访问、密钥处理、工具权限、提示注入风险和版本差异，然后在受限环境用安全测试集验证。

### 可直接背诵的完整答案

> Skill 在 Agent 领域是多义词。若指 Agent Skills 开放规范，它是一个可分发目录，至少包含带 YAML frontmatter 的 `SKILL.md`，其中 `name` 和 `description` 用于发现，正文提供操作指令，还可以按需带 `scripts/`、`references/` 和 `assets/`。Host 通常先加载短元数据，命中任务后再读取正文与具体文件，这叫渐进式披露。
>
> 但平台自称的 Skill 可能只是平台私有的工具、工作流或 Prompt，不一定兼容该规范；A2A 的 `AgentSkill` 又只是 Agent Card 中描述远程 Agent 能力的元数据对象，不是目录和实现代码。Skill 也不等于 Tool：Skill 主要说明怎样完成一类任务，Tool 执行具体动作。安装 Skill 不会修改模型权重或自动授予权限，脚本、资料和工具调用仍要经过来源审查、沙箱、授权、确认、版本管理和分层评测。
