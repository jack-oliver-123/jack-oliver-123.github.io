# 18. 如何设计高质量的工具 Schema？

## Overview

介绍工具 Schema 的设计原则，覆盖命名、描述、参数约束、版本治理和副作用边界

## Learning Objectives

The viewer will understand:

1. 复述“如何设计高质量的工具 Schema？”的关键流程与控制点
2. 说明关键工程边界、失败模式与验证方法

---

## Source Content (Verbatim)

👔面试官：工具 Schema 能通过校验，就算设计好吗？

🙋我：不算。它还要让模型容易选对、让服务端安全校验，并保持版本演进。

👔面试官：工具描述应该写实现细节吗？

🙋我：应写使用条件、输入语义和副作用，内部实现只在影响选择时说明。

## 60 秒回答

高质量工具 Schema 有四个目标：工具之间容易区分，参数含义明确，结构约束尽量收紧，副作用和前置条件可见。名称使用稳定的动词加对象；description 说明何时用、何时不用；参数用明确类型、enum、required 和边界，避免让模型填写可由服务端推导的身份或租户字段。

Schema 是不可信输入的第一层校验，不是安全边界。服务端还要做授权、业务不变量和资源状态检查。Schema 应版本化，并通过契约测试保持模型定义、MCP inputSchema 与执行器一致。

## 详细解析

### 工具边界先于字段设计

一个工具只承担一个清晰动作。read_order 与 cancel_order 应分开，因为权限、副作用和确认策略不同。不要设计 execute_action 这类名称，再用自由文本 action 参数承载所有行为，这会削弱模型选择和授权审计。

名称和描述要形成互斥边界：

- 名称稳定、简短，使用领域术语
- 描述写明适用条件、输出和副作用
- 相似工具明确差异，不靠提示猜测
- 高风险动作在描述中标注需要确认

### 参数约束

优先使用对象参数并显式列出 required。状态、类型等有限集合使用 enum。字符串要说明格式和单位，时间要明确时区。严格模式可用时，拒绝未声明字段。

身份、租户、权限和价格等可信数据应由服务端从会话推导，不让模型传入。模型可以提供候选资源 ID，但服务端必须核对其归属。

### 输出与错误

MCP Tool 可以声明 outputSchema。无论协议是否支持，都应稳定定义成功数据、可重试错误、需要用户输入和权限拒绝。错误信息要给模型足够的恢复线索，但不能泄露堆栈、密钥或内部路径。

## 工程实践与边界

用真实任务做选择评测，而不只做 Schema 单测。统计相似工具混淆、缺字段、错误 enum 和无效资源 ID。Schema 变更要跑旧会话与旧调用回放。

写操作必须接收或生成幂等键，并由执行器校验授权、截止时间和资源版本。重试策略、补偿和审计属于工具运行契约，不应只写在自然语言描述中。

## 常见误区

- **description 越长越好**：冗余会挤占上下文并掩盖决策边界
- **所有业务字段都让模型填写**：可信字段应由服务端派生
- **一个万能工具更灵活**：它会降低选择、授权和观测精度
- **Schema 兼容等于语义兼容**：同字段改单位或含义也会破坏调用

## 面试追问

**追问：工具数量太多怎么办？**

先按权限过滤，再按任务检索候选工具，只把小而相关的集合交给模型。

**追问：怎样验证描述是否有效？**

为相似工具建立正负样本和边界样本，测工具选择混淆矩阵，并检查无需调用时的拒调率。
