一本纸质词典不可能收录世界上每个句子，但只收单个笔画又会让查词太慢。分词器面对的是同一个折中：

- 词表太大，每个完整词都有编号，模型参数和长尾词问题会增加；
- 切得太碎，一个句子变成很长的序列，训练和推理成本会上升；
- 切分粒度合适，常见片段用一个 Token，生僻内容仍能拆开表示。

Tokenizer 就是模型文字世界的“编码规则”。它不是无关紧要的预处理脚本，而是模型输入输出协议的一部分。

## 本篇目标

读完本篇，你应该能够：

- 区分字符、词、子词、字节、Token 和 Token ID；
- 解释 Tokenizer 的训练、编码、解码是三件不同的事；
- 从例子理解 BPE 的合并过程；
- 说明 WordPiece 的目标与常见贪心编码；
- 理解 Unigram 从大候选词表中删减的思路；
- 解释 Byte-level BPE、Byte Fallback 和纯字节模型的区别；
- 说明词表大小对参数量、序列长度和多语言公平性的影响；
- 处理特殊 Token、Unicode、归一化和版本兼容问题。

核心结论：

> Tokenizer 把原始文本按固定规范转换成 Token ID，也把生成的 ID 解码回文本。BPE、WordPiece 和 Unigram 都在词表大小与序列长度之间做子词折中，但训练目标和切分算法不同。Tokenizer 一旦与模型共同训练，就成为模型协议，不能在部署时随意替换。

## 60 秒回答

Tokenizer 是文本与模型离散词表之间的编码器。典型流程包括文本规范化、预切分、子词切分、映射 Token ID、插入特殊 Token；解码时再把 ID 映射回 Token 并按规则还原字符串。

BPE 从较小符号集合开始，反复合并语料中高频相邻符号，直到达到词表或合并次数目标。WordPiece 也通过子词词表表示文本，经典描述通常选择能提高语言模型似然的合并，编码时常做最长匹配。Unigram 则先建立较大的候选子词集合，为子词分配概率，再迭代删除对语料似然影响较小的候选；一句话可能有多种切法，训练和编码可用动态规划。

字节级方案先把文本编码成 UTF-8 字节，使任意文本原则上都能表示，避免真正的未知字符，但可能让某些语言或罕见字符产生更多 Token。

工程上，Tokenizer 会直接影响上下文长度、训练成本、推理费用、代码和多语言效果。模型权重学习的是特定 Token ID 的含义，因此不能只替换词表文件；必须固定版本，并对规范化、特殊 Token、模板、截断和 encode-decode 一致性做测试。

## 详细解析

### 1. 为什么模型不能直接读取字符串

神经网络接收数值张量。Tokenizer 将：

```text
今天学习 LLM
```

转换为类似：

```text
Token：['今天', '学习', ' L', 'LM']
ID：[4812, 932, 417, 10921]
```

再通过嵌入矩阵：

$$
\mathbf{e}_t=E[x_t]
$$

其中：

- $x_t$：第 $t$ 个 Token ID；
- $E$：大小通常为 $V\times d$ 的嵌入矩阵；
- $V$：词表大小；
- $d$：隐藏或嵌入维度；
- $\mathbf{e}_t$：Token 向量。

同一个 ID 在不同模型词表中可能代表完全不同的字符串。

### 2. 六个基本术语

| 术语 | 含义 | 示例 |
| --- | --- | --- |
| 字符 | 书写系统中的字符概念 | `中`、`a` |
| Unicode Code Point | Unicode 分配的码点 | `中` 是 U+4E2D |
| 字节 | 编码后的 0–255 数值 | UTF-8 中一个汉字常占 3 字节 |
| 词 | 语言学或任务中的词单元 | `自然语言` |
| 子词 | 比词小、比字符灵活的片段 | `自然`、`语言` |
| Token | Tokenizer 最终产生的离散单元 | 可能是词、子词、字符或字节映射 |

Token 不等于单词，也不等于汉字。空格、换行、标点甚至字节片段都可能成为 Token。

### 3. Tokenizer 的三个阶段不要混淆

#### 训练 Tokenizer

从代表性语料学习：

- 规范化规则；
- 预切分方式；
- 词表；
- BPE 合并表或 Unigram 概率；
- 特殊 Token。

这通常在模型预训练前完成。

#### 编码

使用已经冻结的规则把新文本转换为 ID：

$$
\text{text}\rightarrow[x_1,x_2,\ldots,x_T]
$$

#### 解码

把 ID 序列转换回可显示字符串：

$$
[x_1,x_2,\ldots,x_T]\rightarrow\text{text}
$$

训练一次 Tokenizer 与每次请求编码不是同一个过程。

### 4. 一条完整编码流水线

概念上可能有：

```text
原始字符串
  ↓ Unicode 规范化
规范字符串
  ↓ Pre-tokenization
初步片段
  ↓ 子词模型
Token 字符串
  ↓ 词表查找
Token ID
  ↓ 模板处理
加入 BOS/EOS/角色标记
```

不同实现可能合并步骤或不做某一步，不能假设所有 Tokenizer 都先按空格切词。

### 5. 规范化为什么重要

视觉相同或相近的文本，底层 Unicode 序列可能不同。例如某些带重音字符可以表示为：

- 一个预组合 Code Point；
- 基础字母加组合附加符号。

Unicode NFC/NFKC 等规范化可以减少变体，但也可能改变语义细节，如兼容字符、全角形式或特殊符号。

工程选择应由数据和任务决定：

- 搜索可能希望合并更多表面差异；
- 代码模型需要谨慎保留空格和符号；
- 安全系统要防止同形异义字符绕过；
- 法律或取证文本可能要求精确保留原文。

### 6. Pre-tokenization 是什么

预切分在子词算法前划定候选边界，例如：

```text
"Hello, world!"
→ ["Hello", ",", " world", "!"]
```

常见规则可能依据空格、标点、数字或正则表达式。预切分会影响后续能否跨边界合并。

因此“都是 BPE”的两个 Tokenizer，也可能因为规范化和预切分不同而产生完全不同的 Token。

### 7. 为什么使用子词

#### 完整词词表的问题

若每个词一个 Token：

- 词形变化很多；
- 人名、网址、代码标识符层出不穷；
- 词表会很大；
- 未登录词无法表示。

#### 纯字符的问题

若每个字符一个 Token：

- 序列更长；
- 注意力和 Decode 步数增加；
- 常见词无法作为稳定整体表示。

子词折中：

```text
常见词：playing → [playing]
少见词：replaying → [re, play, ing]
```

### 8. BPE 的训练直觉

BPE 最初是一种数据压缩方法，后来用于子词分割。简化训练过程：

1. 从字符或字节符号开始；
2. 统计所有相邻符号对频率；
3. 合并最高频符号对；
4. 更新语料表示；
5. 重复到词表大小或合并次数达到目标。

假设语料词频：

```text
low      × 5
lower    × 2
newest   × 6
widest   × 3
```

初始可表示为字符加词尾标记：

```text
l o w </w>
l o w e r </w>
n e w e s t </w>
w i d e s t </w>
```

如果 `e s` 最常见，先合并：

```text
e + s → es
```

后续可能继续：

```text
es + t → est
l + o → lo
lo + w → low
```

最终得到常见片段词表。

### 9. BPE 编码怎样做

编码新文本时，从基础符号开始，按照已学合并规则及其优先级应用合并。

概念伪代码：

```python
pieces = split_to_base_symbols(text)

while True:
    pair = highest_priority_merge_present(pieces)
    if pair is None:
        break
    pieces = merge_all_occurrences(pieces, pair)

ids = [vocab[piece] for piece in pieces]
```

真实高性能实现会使用更有效的数据结构，不会每轮扫描所有组合。

### 10. BPE 的优点与局限

优点：

- 规则相对直观；
- 高频片段变成较短 Token 序列；
- 可从字符或字节保证覆盖；
- 训练和编码生态成熟。

局限：

- 贪心合并不直接优化所有下游任务；
- 切分由频率和预处理决定，未必符合语言学词素；
- 语料分布偏差会导致不同语言 Token 效率差异；
- 词表和合并表必须严格配套。

### 11. WordPiece 与 BPE 有什么不同

WordPiece 也构建子词词表。公开论文和常见实现对原始训练细节的描述不如 BPE 统一，经典解释通常是选择能改进训练数据语言模型似然的候选合并，而不只是原始频次最高。

编码阶段常使用最长匹配优先：

```text
unaffable
→ un + ##aff + ##able
```

`##` 是某些实现表示“不是词开头”的约定，不是 WordPiece 必须使用的通用字符。

简化最长匹配：

```python
start = 0
while start < len(word):
    choose_longest_vocab_piece_starting_at(start)
    start = end_of_piece
```

如果某个位置找不到可用片段，传统实现可能输出 `[UNK]`。是否如此取决于基础覆盖和具体实现。

### 12. Unigram Language Model 的思路

Unigram 不从小词表不断合并，而通常从一个较大的候选子词集合开始，再逐步删除不重要候选。

它假设一个切分 $s=(t_1,\ldots,t_n)$ 的概率近似为：

$$
P(s)=\prod_{i=1}^{n}P(t_i)
$$

其中：

- $t_i$：切分中的第 $i$ 个子词；
- $P(t_i)$：该子词概率；
- 乘积：Unigram 独立假设下整条切分概率。

一段文本可能有多种切法：

```text
unigram
→ [uni, gram]
→ [un, i, gram]
→ [u, ni, g, ram]
```

文本概率要考虑可能切分：

$$
P(x)=\sum_{s\in S(x)}P(s)
$$

其中 $S(x)$ 是文本 $x$ 的所有允许切分集合。训练常用 EM 类方法估计子词概率，并删除对整体似然影响较小的候选。

编码时可以用 Viterbi 动态规划找最高概率切分，也可按概率采样不同切分用于正则化。

### 13. SentencePiece 是什么

SentencePiece 是一个官方开源文本 Tokenizer 工具包，可以训练 BPE 或 Unigram 模型。它把输入视为原始 Unicode 字符串，不要求先由语言特定分词器按空格切词，并用特殊符号表达空格边界。

因此：

> SentencePiece 是工具与实现框架，BPE/Unigram 是其中可选模型；二者不在同一个分类层级。

### 14. Byte-level BPE 为什么能覆盖任意文本

Unicode 字符串可以编码成 UTF-8 字节，每个字节是 0–255。以字节作为基础符号，理论上任何合法字节序列都能拆开表示。

```text
字符 → UTF-8 字节 → 可显示映射 → BPE 合并
```

优点：

- 基础词表只需覆盖 256 个字节值；
- 不会因为新 Unicode 字符而真正无词可分；
- 对代码、emoji、混合语言更稳健。

代价：

- 非 ASCII 字符可能先变成多个字节；
- 若训练语料对某语言覆盖不足，常见字符序列未被充分合并，Token 数较多；
- 字节映射后的 Token 字符串未必直观可读。

### 15. Byte Fallback 与 Byte-level BPE 不完全相同

- Byte-level BPE：从字节基础符号开始训练 BPE；
- Byte Fallback：主要使用字符/子词词表，遇到无法表示内容时退回字节 Token；
- 纯字节模型：模型直接以字节为基本序列单位，可能不做子词合并。

它们都利用字节保证覆盖，但正常路径和序列长度不同。

### 16. 词表大小怎样影响模型

嵌入参数量近似为：

$$
P_{embed}=Vd
$$

其中：

- $V$：词表大小；
- $d$：隐藏维度。

若输出层不与输入嵌入共享，输出投影还可能有约 $dV$ 个参数。

大词表：

- 常见片段可用更少 Token；
- 序列缩短；
- 嵌入与输出层参数、计算可能增加；
- 长尾 Token 学习样本较少。

小词表：

- 嵌入参数少；
- 覆盖容易；
- 文本可能切得更碎；
- 注意力长度和生成步数增加。

没有对所有语言和模型都最优的固定词表大小。

### 17. Tokenizer 为什么影响多语言公平性

如果训练语料主要来自某种语言，合并规则会优先照顾该语言高频片段。表达相同信息时，不同语言可能产生不同 Token 数：

```text
语言 A：一句话 12 Token
语言 B：同义句 28 Token
```

这会影响：

- 有效上下文容量；
- API 按 Token 计费成本；
- 生成步数和延迟；
- 训练中每条文本占用的序列预算；
- 模型见到的语义信息密度。

因此多语言评测应同时统计任务质量和 Token 膨胀率。

### 18. 特殊 Token 有什么作用

常见特殊 Token：

| 类型 | 作用 |
| --- | --- |
| BOS | 序列开始 |
| EOS | 序列结束 |
| PAD | 批处理补齐 |
| UNK | 无法表示的内容 |
| MASK | 掩码语言建模 |
| 角色 Token | system/user/assistant 边界 |
| 工具 Token | 工具调用或结果边界 |

特殊 Token 不只是字符串标签。模型要在训练中见过它们的用法，推理模板也必须使用正确 ID。

### 19. 聊天模板也是协议的一部分

同样的对话可能序列化为：

```text
<|system|>你是客服<|end|>
<|user|>我要退款<|end|>
<|assistant|>
```

若模型训练时使用这种模板，部署时改成随意的：

```text
系统：你是客服
用户：我要退款
助手：
```

可能导致能力下降。模板决定角色边界、EOS 位置和生成起点，需和 Tokenizer、模型版本一起管理。

### 20. Encode 与 Decode 能完全可逆吗

未必。如果规范化、空白清理或未知 Token 丢失信息：

$$
decode(encode(text))\ne text
$$

例如 NFKC 可能把兼容字符转换成标准形式。某些 Decoder 还会处理前导空格或无效字节序列。

需要测试：

```python
samples = [
    "中文 English 123",
    "a  b\n\tc",
    "👩🏽‍💻",
    "e\u0301",
    "def f(x):\n    return x + 1",
]

for text in samples:
    ids = tokenizer.encode(text)
    restored = tokenizer.decode(ids)
    inspect(text, ids, restored)
```

不要默认所有样本都必须字节级完全相同，应根据 Tokenizer 规范定义预期。

### 21. Token 计数为什么会出错

常见原因：

- 客户端与服务端 Tokenizer 版本不同；
- 忘记聊天模板和特殊 Token；
- 截断发生在模板应用前后不同阶段；
- 工具定义、图片标记或系统消息也占上下文；
- 服务端采用不同模型或词表；
- Unicode 规范化不一致。

因此预算最好使用目标模型官方 Tokenizer，并为服务端保留安全余量。

## 工程实践与边界

### 1. Tokenizer 必须版本化

至少固定：

- 词表文件校验值；
- 合并表或模型文件；
- 规范化规则；
- Pre-tokenizer 配置；
- 特殊 Token 及其 ID；
- 聊天模板；
- 最大长度与截断方向；
- Tokenizer 库版本。

### 2. 不能只扩词表而不处理模型权重

新增 Token 会改变 $V$，必须扩展输入嵌入和通常的输出投影，并为新行初始化、训练和评测。若重排旧 Token ID，旧权重语义会整体错位，通常造成严重故障。

### 3. 训练 Tokenizer 的语料要代表部署分布

覆盖：

- 各目标语言及比例；
- 代码、数学、JSON 等结构文本；
- 专业术语；
- emoji 和特殊符号；
- 用户真实空白和拼写噪声；
- 需要保留的大小写与规范化差异。

同时做隐私、版权和敏感数据治理。

### 4. 评估不能只看词表大小

建议指标：

- 每字符 Token 数；
- 每字节 Token 数；
- 各语言 Token 膨胀率；
- 未知 Token 比例；
- 关键术语切分稳定性；
- encode/decode 速度；
- 下游模型质量；
- 上下文与推理成本。

### 5. 截断要理解业务方向

```text
左截断：保留最近对话，可能丢系统开头
右截断：保留开头，可能丢最新用户请求
```

聊天系统通常不能简单左截断到固定 Token 数，因为系统指令和完整消息边界需要保留。更稳妥的方法是按消息裁剪、摘要或检索。

### 6. 安全边界不能只按字符串匹配

恶意文本可能利用：

- Unicode 同形字符；
- 零宽字符；
- 组合字符；
- 大小写和全半角；
- Token 边界拆分；
- 特殊 Token 注入。

安全检测要明确在哪个规范化阶段运行，并避免让用户原文被错误解释为受信任的特殊 Token。

### 7. 缓存键要使用一致规范

若语义缓存、Prefix Cache 或去重系统在不同规范化层计算键，视觉相同文本可能缓存未命中，反之不同文本也可能被错误合并。应记录原始文本和规范文本的职责边界。

### 8. 迁移模型时重新计算成本

两个模型的上下文长度都标为 32K，不表示同一文档占用相同 Token。更换模型时要重新测：

- Prompt Token 数；
- 最大可放文档量；
- API 费用；
- 输出长度；
- 截断和模板行为。

## 常见误区

### 误区一：一个 Token 就是一个单词

错误。Token 可以是子词、汉字片段、空格、标点、字节或特殊标记。

### 误区二：中文一个字固定等于一个 Token

错误。切分取决于具体词表，常见多字词可能合并，罕见字符也可能拆成字节。

### 误区三：BPE、WordPiece、SentencePiece 是同一层概念

错误。BPE 和 WordPiece 是子词方法；SentencePiece 是可实现 BPE、Unigram 等模型的工具包。

### 误区四：字节级 Tokenizer 每个字符只占一个 Token

错误。一个 Unicode 字符的 UTF-8 编码可能有多个字节，是否被 BPE 合并取决于词表。

### 误区五：Tokenizer 不影响模型能力

错误。它影响序列长度、信息边界、嵌入学习、多语言效率和训练预算。

### 误区六：换一个更好的 Tokenizer 可以直接提升旧模型

错误。旧模型权重绑定原 Token ID。替换后即使文本看起来相同，向量语义也不匹配。

### 误区七：Decode 总能恢复原始字符串

错误。规范化、未知 Token 和无效字节处理可能造成不可逆变化。

### 误区八：Token 数可以用字符数除以固定常数估算

只可作非常粗略预算。语言、代码、空格和词表不同会产生很大差异，正式计费和截断必须调用目标 Tokenizer。

## 面试追问

### 问：BPE 和 Unigram 最核心的区别是什么

答：BPE 通常从小符号集合出发，逐步合并高频相邻单元；Unigram 通常从大候选集合出发，基于概率模型迭代删除候选，并可考虑一句话的多种切分。

### 问：为什么词表不能无限增大

答：更大词表可能缩短序列，但会增加嵌入和输出层参数/计算，并产生训练样本很少的长尾 Token。它是参数、序列长度、覆盖和数据效率之间的折中。

### 问：为什么 Byte-level BPE 几乎没有 UNK

答：任意 Unicode 文本都可先转成 UTF-8 字节，而 256 个字节值可以作为基础符号，所以即使没有高层合并，仍能退回字节序列表示。

### 问：模型更换 Tokenizer 会发生什么

答：Token ID 与嵌入行的对应关系改变，模型看到的向量语义错位；特殊 Token 和聊天模板也可能失效。除非同步转换权重并重新训练，否则通常不能直接替换。

### 延伸追问速答

| 追问 | 回答要点 |
| --- | --- |
| Tokenizer 是模型吗 | 它通常是确定性编码组件，也可能包含学习得到的词表或概率，但不是负责生成语义答案的 LLM |
| 为什么代码 Tokenizer 要重视空格 | 缩进、换行和符号直接影响代码语法与序列效率 |
| 怎样比较两个 Tokenizer | 在同一代表性语料上比较 Token 数、语言差异、速度、可逆性与下游质量 |
| 特殊 Token 可以由用户直接输入吗 | 宿主系统应区分普通文本和受信任控制 Token，避免角色边界被伪造 |
