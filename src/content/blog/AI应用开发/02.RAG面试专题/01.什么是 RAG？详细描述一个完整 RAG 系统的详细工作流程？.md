## 本篇目标

读完本篇，你应该能够：

- 用一句人话解释 RAG，而不是只会展开英文缩写；
- 区分离线建库与在线问答两条链路；
- 说清文档、Chunk、Embedding、索引、召回、重排和生成之间的关系；
- 根据中间结果定位一次错误来自数据、检索还是生成；
- 在面试中完整回答“一个 RAG 系统怎样工作”。

## 60 秒回答

RAG 是 Retrieval-Augmented Generation，也就是检索增强生成。它的核心思想是：大模型回答之前，系统先从外部知识源中检索与当前问题相关的证据，再把问题和证据一起交给大模型生成答案。它把模型参数中的通用能力与可更新、可追溯、可控制权限的外部知识结合起来。

完整 RAG 分为离线和在线两条链路。离线链路负责采集文档、解析清洗、按语义切块、补充来源与权限等元数据、生成 Embedding，并建立向量或关键词索引；在线链路负责用户鉴权、理解或改写查询、多路召回、权限与时效过滤、融合重排、上下文组装、模型生成、引用校验以及日志和评测。

RAG 不等于向量数据库，也不能保证没有幻觉。生产效果取决于数据质量、切块、召回、重排、上下文和模型是否忠实使用证据。证据不足时，系统应该澄清、拒答或转人工，而不是让模型猜答案。

## 详细解析

### 先说人话：RAG 像一场开卷考试

想象一名参加开卷考试的学生。

如果不允许查资料，他只能依靠记忆作答。学生知识丰富，但可能没学过公司昨天更新的制度，也可能把两个相似规定记混。

如果允许查资料，他可以先在资料库中找到相关章节，再结合题目组织答案。资料找对了，不代表答案就不会错；他仍可能漏看限制条件、理解错原文，或者引用了过期版本。但是，与完全依赖记忆相比，答案有机会使用最新资料，也更容易说明依据来自哪里。

在这个类比中：

| RAG 组件 | 开卷考试类比 |
| --- | --- |
| 大模型 | 负责理解题目和组织答案的学生 |
| 知识库 | 允许查阅的书和资料 |
| 检索器 | 帮助快速翻到相关页的目录与搜索系统 |
| Chunk | 从长资料中切出的可检索片段 |
| 上下文 | 本次真正放到学生桌上的几页证据 |
| 引用 | 答案对应的书名、章节和页码 |

因此，可以先记住最简单的公式：

> **RAG = 检索外部证据 + 基于证据生成答案。**

### R、A、G 分别做什么

RAG 的三个单词对应三个动作：

1. **Retrieval，检索**：根据用户问题，从知识库找出可能相关的内容。
2. **Augmented，增强**：把检索到的证据加入本次模型输入。
3. **Generation，生成**：大模型结合问题与证据，组织最终答案。

“增强”不代表修改模型参数。它只是让模型在本次请求中多看到一些外部信息。请求结束后，这些文档不会自动写入模型参数。

### 为什么需要外部知识

大模型参数中保存的是训练过程中学习到的统计规律，而不是一个可以按文档 ID 精确查询、修改和删除的业务数据库。企业应用还经常面对以下情况：

- 公司制度没有出现在公开训练数据中；
- 价格、库存、政策和组织信息经常变化；
- 不同员工只能查看自己有权限的资料；
- 回答需要展示出处，便于用户核对；
- 某份错误文档需要立即下线或回滚。

这些问题适合由外部知识系统管理。大模型继续负责语言理解、归纳和表达，RAG 负责在请求时把所需知识送进上下文。

### 贯穿案例：回答差旅住宿标准

假设员工小李问：

> “下周去上海出差，普通员工住宿费每晚最多能报多少？”

系统中存在以下资料：

- 2024 年旧版差旅制度；
- 2026 年新版差旅制度；
- 销售部门的补充标准；
- 只有财务人员能查看的内部审核指引；
- 一份尚未生效的制度草案。

这不是简单地找一句包含“上海”和“住宿”的文本。系统还要判断：

- 小李是不是普通员工；
- 2026 年新版是否已经生效；
- 销售部门补充标准是否适用于小李；
- 草案能不能作为正式依据；
- 财务内部指引是否允许被小李检索；
- 找到的金额是否带有旺季、职级或审批等限制条件。

完整 RAG 就是在离线和在线两个阶段解决这些问题。

### 离线链路：提前把资料整理成可检索知识

离线并不是“断网”，而是指这些工作通常在用户提问之前完成。

#### 第一步：接入权威数据源

系统从文档平台、对象存储、业务数据库、内部 Wiki 或受控 API 获取资料。每份资料至少要记录：

- 来源系统和原始地址；
- 文档 ID；
- 版本号或内容哈希；
- 创建、修改、生效和失效时间；
- 所属租户、部门和访问权限；
- 采集时间与同步状态。

这里的关键是确定“谁才是权威来源”。向量数据库通常只是用于检索的派生副本，不应该反过来成为业务事实的唯一来源。

#### 第二步：解析文档

解析器把 PDF、Word、HTML、Markdown、表格或扫描图片转换成机器可处理的结构。

解析不只是提取文字，还要尽量保留：

- 标题和章节层级；
- 段落顺序；
- 列表编号；
- 表头与单元格关系；
- 图片说明；
- 页码和原文位置。

如果一个双栏 PDF 被解析成左右两栏交叉的句子，后面的 Embedding 再强也很难补救。因此，解析质量是 RAG 的第一道质量关。

#### 第三步：清洗与规范化

常见处理包括去除重复页眉页脚、统一编码、识别语言、清理不可见字符、处理重复文档和标记低质量 OCR。

清洗不能简单删除所有“重复内容”。例如每页都有的保密声明通常可以去掉，但多个制度中看似相同的适用范围可能有不同法律含义，需要保留。

#### 第四步：切分 Chunk

Chunk 是为了检索而从长文档中切出的内容单元。

一整本员工手册过长，直接把整本书作为一个结果，既不利于精确匹配，也浪费模型上下文。系统通常按标题、段落、句子、表格或代码结构切分，再根据模型输入限制和业务问题调整长度。

每个 Chunk 应保留父文档和章节关系。例如：

| 字段 | 示例 |
| --- | --- |
| document_id | travel-policy-2026 |
| chunk_id | travel-policy-2026-section-4-2 |
| section_path | 差旅标准 / 国内差旅 / 住宿 |
| content | 上海普通员工住宿标准及例外条件 |
| version | 2026-v1 |
| valid_from | 2026-01-01 |
| acl | 全体正式员工 |

#### 第五步：生成 Embedding

Embedding 模型把查询或文本片段映射成一串数字，也就是向量。训练良好的检索模型会让语义相关的查询与文档在向量空间中更接近。

例如，“上海出差住酒店能报多少”和“上海地区住宿费报销上限”没有使用完全相同的词，但向量检索仍有机会判断它们语义相关。

Embedding 不是加密，也不能还原全部原文；它是为相似性比较设计的表示。

#### 第六步：建立索引

系统把 Chunk、向量和元数据写入检索系统。常见索引包括：

- **稠密向量索引**：根据语义相似度寻找内容；
- **关键词或稀疏索引**：根据词项、编号和精确短语寻找内容；
- **结构化索引**：根据部门、日期、文档类型和权限过滤。

生产系统经常同时使用多种索引，因为向量检索擅长语义，关键词检索更擅长精确编号、专有名词和罕见词。

#### 第七步：离线验收并发布索引

新索引不能构建完成后直接全量替换。通常要：

1. 检查文档数量、Chunk 数量与失败率；
2. 使用固定查询集测试正确证据能否被召回；
3. 比较新旧索引的质量、延迟和资源；
4. 给索引分配独立版本；
5. 灰度切换读流量；
6. 保留旧版本以便回滚。

离线链路可以概括为：

> 数据接入 -> 解析 -> 清洗 -> 切块 -> 补充元数据 -> 生成向量 -> 建立索引 -> 评测 -> 发布。

### 在线链路：用户提问后实时查证并回答

#### 第一步：认证、鉴权与租户隔离

系统先确认“谁在问”。用户身份、租户、部门和角色会影响哪些文档允许进入检索结果。

权限过滤应尽可能发生在检索阶段。只在生成答案后删除敏感文字太晚了，因为越权内容已经进入模型、日志或缓存。

#### 第二步：理解查询

用户问题可能包含错别字、省略、指代和多轮上下文。例如用户先问“上海标准是多少”，接着问“那北京呢”。第二个问题需要结合上一轮才能还原成完整查询。

系统可以执行：

- 拼写规范化；
- 指代消解；
- 意图分类；
- 查询改写；
- 复杂问题分解；
- 提取时间、部门和地区等过滤条件。

改写必须保留原始意图。错误改写可能让后面每一层都检索错方向，所以应同时记录原查询和改写查询。

#### 第三步：召回候选

召回的目标不是马上选出唯一答案，而是尽量让正确证据进入候选集合。

常见做法是同时运行：

- 向量检索；
- BM25 等关键词检索；
- 结构化数据库查询；
- 必要时的图检索。

每一路可能返回一组候选 Chunk。Top-K 表示取排名最靠前的 K 个，但 K 不是越大越好：太小可能漏证据，太大会增加噪声、延迟和后续成本。

#### 第四步：权限、版本与时效过滤

候选结果要继续检查：

- 用户是否拥有访问权限；
- 文档是否已经生效；
- 文档是否过期或被撤回；
- 是否属于正确租户和部门；
- 是否为重复或低质量内容。

过滤条件应该作为可测试的系统规则，而不是交给大模型自行猜测。

#### 第五步：融合与重排

多路检索的分数通常不可直接比较。系统可以使用倒数排名融合等方法合并候选，再让重排模型同时读取“问题 + 候选文本”，给出更精细的相关性排序。

可以把两阶段检索理解为：

- 召回阶段像从图书馆快速找出几十本可能相关的书；
- 重排阶段像逐本看目录，把最可能包含答案的几本放到前面。

重排更准确但计算更贵，所以通常只处理召回得到的小候选集。

#### 第六步：组装上下文

系统从重排结果中选取证据，并完成：

- 去重；
- 合并相邻片段；
- 恢复标题路径；
- 控制 Token 预算；
- 标记来源编号；
- 处理互相冲突的版本；
- 按重要程度安排位置。

上下文不是把 Top-K 原样拼接。重复片段会浪费空间，过多无关材料还可能干扰模型。

#### 第七步：让大模型生成答案

模型输入通常包含：

- 系统规则；
- 用户问题；
- 可用证据；
- 引用格式要求；
- 证据不足时的处理规则。

例如要求模型只依据提供的制度回答，明确适用范围，标注引用；如果资料冲突或不足，则说明缺少什么信息。

这里要注意：提示词只能约束模型行为，不能代替权限校验，也不能把没有召回的事实凭空补回来。

#### 第八步：输出校验

生成后可以检查：

- 答案中的金额和日期是否出现在证据中；
- 每条引用是否真的支持相应陈述；
- 是否泄露敏感信息；
- 输出格式是否符合接口契约；
- 高风险结论是否需要人工审核。

对于付款、医疗、法律或人事决定等高风险场景，RAG 输出不应自动等同于最终业务决定。

#### 第九步：记录、反馈和监控

为了复现问题，系统通常记录：

- 原始查询和规范化查询；
- 用户权限摘要；
- 每一路召回的候选 ID 与分数；
- 过滤和重排结果；
- 最终上下文的 Chunk ID；
- 模型、提示词、Embedding 和索引版本；
- 延迟、Token、成本与错误码；
- 用户反馈和人工标注。

日志需要脱敏，并遵守保留期限。可观测不等于把完整私密文档全部写进日志。

在线链路可以概括为：

> 身份确认 -> 查询理解 -> 多路召回 -> 权限与时效过滤 -> 融合重排 -> 上下文组装 -> 模型生成 -> 引用与安全校验 -> 监控反馈。

### 原始 RAG 论文与工程 RAG 的区别

Lewis 等人的原始 RAG 论文把预训练生成模型的参数化知识，与可检索的非参数化知识结合起来，并在开放域问答等知识密集型任务上进行研究。

生产工程在这个思想上增加了很多论文原型之外的组件：

- 文档解析和版本管理；
- ACL 与租户隔离；
- 混合检索与重排；
- 缓存、超时和降级；
- 引用校验；
- 离线评测与线上监控；
- 灰度发布和索引回滚。

因此，面试中既要知道核心论文思想，也要说明真实系统不是“一个向量库加一次模型调用”。

### RAG 为什么仍然会答错

一次错误可能沿着链路逐层传播：

| 层次 | 失败例子 | 结果 |
| --- | --- | --- |
| 数据源 | 新制度没有同步 | 知识库根本没有正确答案 |
| 解析 | 表格表头与金额错位 | 检索到的证据含义错误 |
| 切块 | 金额和适用条件被分开 | 模型只看到一半规则 |
| 召回 | 正确 Chunk 没进入 Top-K | 后续无法使用正确证据 |
| 过滤 | 把有效新版误判为未生效 | 候选只剩旧制度 |
| 重排 | 正确证据排到预算之外 | 上下文没有答案 |
| 生成 | 模型忽略否定词 | 证据正确但答案错误 |
| 引用 | 来源映射错位 | 用户无法核验答案 |

这就是为什么排查 RAG 不能只查看最终文本。

## 工程实践与边界

### 分层评测

生产评测至少分成三层：

| 层次 | 典型指标 | 回答的问题 |
| --- | --- | --- |
| 数据与索引 | 解析成功率、同步新鲜度、孤儿 Chunk 数 | 知识是否正确进入系统 |
| 检索 | Recall@K、MRR、nDCG、重复率 | 正确证据是否被找回并排在前面 |
| 生成 | 答案正确性、忠实度、引用正确率、拒答质量 | 模型是否正确使用证据 |
| 系统 | P50/P95/P99 延迟、可用率、吞吐、单次成本 | 系统是否满足线上约束 |

Recall@K 可以先理解为：在应该找到正确证据的查询中，有多少查询的正确证据进入了前 K 个候选。

### 必须保存版本

至少记录以下版本：

- 原始文档版本；
- 解析器与切块配置版本；
- Embedding 模型版本；
- 索引版本；
- 重排模型版本；
- 生成模型和提示词版本。

否则同一个问题今天和昨天答案不同，你很难判断是哪一层发生了变化。

### 设计降级策略

常见降级方式包括：

- 向量服务超时时退到关键词检索；
- 重排服务失败时使用融合后的排名；
- 证据不足时请求用户补充部门或日期；
- 高风险或冲突问题转人工；
- 新索引异常时切回旧索引。

降级不应悄悄把“基于公司制度回答”变成“依靠模型记忆猜测”。如果证据不可用，应显式降低置信度或拒答。

### 明确四个边界

1. **RAG 不是事实正确性的保证**：源文档可能错误，模型也可能曲解证据。
2. **RAG 不是权限系统**：ACL 需要由可信程序在检索和数据层执行。
3. **RAG 不是工具执行系统**：查询制度适合 RAG，真正提交报销仍需业务 API 或 Workflow。
4. **RAG 不是所有任务都需要**：纯改写、翻译、固定规则计算等任务可能无需检索。

### 一份上线前检查清单

- 权威数据源和同步责任人是否明确；
- 解析失败是否可见并能重试；
- 文档删除后派生 Chunk 与缓存是否同步删除；
- ACL 是否在检索前或检索中生效；
- 是否有覆盖真实问题的评测集；
- 是否同时观察检索和生成指标；
- 证据不足、冲突和超时时怎样处理；
- 新索引是否可灰度和回滚；
- 日志是否脱敏；
- 用户能否查看并核对引用。

## 常见误区

- **“RAG 就是向量数据库”**：向量数据库只承担索引和检索的一部分，文档解析、权限、重排、生成、引用和评测同样重要。
- **“把 Top-K 结果塞给大模型就完成了”**：候选可能重复、过期、越权或相互冲突，还需要过滤、重排和上下文组装。
- **“用了 RAG 就不会产生幻觉”**：模型可能忽略或误解证据，源文档也可能错误。RAG 只能改善部分知识缺失问题。
- **“Chunk 越多，答案越完整”**：更多候选会占用上下文并引入噪声，应该按评测和 Token 预算选取。
- **“相似度最高的就是答案”**：相似度衡量的是表示空间中的接近程度，不等于事实正确、权限允许或版本有效。
- **“索引更新后立即处处生效”**：构建、复制、缓存和读别名切换都可能带来延迟，需要定义新鲜度目标。
- **“引用了文档就代表结论有依据”**：引用必须真正支持对应陈述，不能只引用一篇主题相近的材料。

## 面试追问

1. **为什么权限过滤要尽量放在检索阶段？**

   因为越权内容一旦进入重排器、模型、缓存或日志，就已经扩大了泄露面。生成后再过滤无法撤销前面发生的数据暴露。

2. **检索到了正确证据，但答案还是错，怎样排查？**

   先确认正确证据是否进入最终上下文，再检查是否被截断、位置是否过后、是否与其他版本冲突、提示规则是否清楚，最后判断模型是否错误归纳。不要直接扩大 Top-K。

3. **RAG 和搜索引擎有什么区别？**

   搜索引擎主要返回文档或链接；RAG 还会让生成模型基于检索证据合成答案，因此新增了忠实度、引用和生成安全等风险。

4. **为什么同时需要离线和在线评测？**

   离线评测便于用固定样本快速比较版本；线上指标能反映真实流量、延迟、失败率和用户行为。两者观察的范围不同，需要共同使用。

5. **为什么不把所有制度一次性放进长上下文？**

   全量上下文会增加延迟和成本，也可能超过输入上限。无关信息和中间位置的信息还可能被模型忽略。是否使用长上下文应通过目标数据评测，而不是只看模型宣称的最大窗口。

6. **一个最小可用 RAG 包含什么？**

   至少需要可靠数据源、可追溯 Chunk、一个检索方法、上下文组装、受证据约束的生成，以及能判断正确证据是否被召回的评测集。生产环境还要增加权限、版本、监控和回滚。
