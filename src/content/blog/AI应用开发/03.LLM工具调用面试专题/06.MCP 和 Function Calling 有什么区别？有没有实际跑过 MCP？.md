想象一家餐厅。
顾客对服务员说：
> “我想点一份不辣的牛肉面。”
这句话表达了顾客的调用意图。
服务员再按照餐厅内部约定，把订单送到后厨。
前者类似模型通过 Function Calling 告诉 Host：
> “调用 order_noodles，参数是 beef、not_spicy。”
后者类似 Host 通过 MCP Client 与能力提供方交互。
顾客怎样表达点单，与服务员怎样把订单送到后厨，不是同一个接口。
这就是理解 MCP 与 Function Calling 的起点。

## 本篇目标
读完本篇，你应该能够：
- 从层级上区分 MCP 与 Function Calling；
- 比较两者的参与者、核心对象、发现、传输和生命周期；
- 解释两者怎样组合；
- 说明没有 Function Calling 时能否使用 MCP；
- 说明没有 MCP 时能否使用 Function Calling；
- 描述一个最小 MCP Server 与 Client 的验证过程；
- 讲清 initialize、initialized、tools/list 和 tools/call；
- 覆盖非法参数、超时、崩溃和关闭等失败分支；
- 在没有生产经验时给出诚实而专业的面试回答。

## 60 秒回答
Function Calling 和 MCP 解决不同层级的问题。
Function Calling 位于模型与 Host 之间，解决模型怎样表达“要调用哪个工具、参数是什么”。
模型通常只生成结构化调用意图，真实函数由 Host 校验和执行。
MCP 位于 Host 与外部能力提供方之间，解决应用怎样发现、连接和调用 Tools、Resources、Prompts，并定义 Host、Client、Server、生命周期、能力协商和标准传输。
两者可以组合：Host 通过 MCP tools/list 获取工具，把允许的工具转换成模型接口的定义；模型通过 Function Calling 选择工具；Host 校验后再通过 MCP tools/call 调用 Server。
两者也可以单独使用：Host 可以直接执行内部函数而不使用 MCP，确定性 Workflow 也可以直接调用 MCP 而不让模型选择。
如果面试官问是否实际跑过，应讲清初始化、工具列举、调用、错误处理和连接关闭，并说明亲自验证到什么范围，不要把最小 Demo 说成生产落地。

## 详细解析

### 1. 最核心的层级差异
~~~text
模型
  ↓ Function Calling
Host
  ↓ MCP
MCP Server
  ↓ 内部调用
真实 API / 数据库 / 本地程序
~~~
Function Calling 的接口两端是：
- 模型；
- Host。
MCP 的接口两端是：
- MCP Client；
- MCP Server。
Host 同时出现在两条链路中。
它负责把两套表示连接起来。

### 2. 详细对比表

| 维度 | Function Calling | MCP |
| --- | --- | --- |
| 主要目标 | 模型表达调用意图 | Host 接入外部上下文和能力 |
| 接口两端 | 模型 API 与 Host | MCP Client 与 MCP Server |
| 主要输入 | 工具定义、对话 | JSON-RPC 请求、能力与参数 |
| 主要输出 | 工具名、参数、调用标识 | 响应、错误、通知、能力结果 |
| 工具发现 | 通常由应用自行维护 | 规范定义列举语义 |
| 生命周期 | 由模型 API/应用决定 | 规范定义初始化与协商 |
| 传输 | 由模型供应商 API 决定 | 标准支持 stdio、Streamable HTTP |
| 上下文原语 | 主要围绕模型工具 | Tools、Resources、Prompts 等 |
| 真实执行 | Host 或其后端 | Server 执行；Host 仍控制策略 |
| 模型是否必需 | 是，概念本身面向模型输出 | 协议调用可由确定性程序驱动 |
| 安全 | 应用必须实现 | Host 与 Server 都必须实现 |
| 是否通用字段标准 | 各供应商实现可能不同 | MCP 规范定义协议消息 |

### 3. Function Calling 不负责什么
Function Calling 通常不负责：
- 工具进程怎样启动；
- 远程 Server 在哪里；
- 怎样列出外部能力；
- 怎样协商协议版本；
- stdio 还是 HTTP；
- Server 如何暴露 Resource；
- 如何订阅能力变化；
- 多个 Host 如何复用同一能力。
这些正是 MCP 更关注的问题。

### 4. MCP 不负责什么
MCP 不负责：
- 模型是否会选对工具；
- 模型怎样训练；
- 某家模型 API 的工具字段；
- Prompt 怎样让模型更稳定；
- 业务操作是否应该执行；
- 用户是否批准高风险动作；
- 写操作怎样保证幂等；
- 失败后怎样补偿；
- 最终答案怎样生成。
这些需要模型能力、Host 编排和业务系统共同完成。

### 5. 两者怎样组合
完整组合流程：
~~~text
一、Host 建立 MCP 连接
二、Client 与 Server 初始化并协商能力
三、Client 请求 tools/list
四、Host 按用户和任务筛选工具
五、Host 转换成模型供应商的工具格式
六、模型生成 Function Calling 意图
七、Host 校验工具名、参数、权限和确认状态
八、Client 请求 tools/call
九、Server 执行真实能力
十、Host 把结果关联到模型调用
十一、模型生成回答或继续调用
~~~
注意第六步和第八步不是同一个调用。
第六步发生在模型接口。
第八步发生在 MCP 协议。

### 6. Host 需要做格式映射
MCP Tool 定义与某家模型 API 的工具定义不一定字段完全相同。
Host 通常需要转换：
~~~typescript
function toModelTool(mcpTool: McpTool): ModelToolDefinition {
  return {
    name: mcpTool.name,
    description: mcpTool.description,
    inputSchema: mcpTool.inputSchema,
  };
}
~~~
这只是概念伪代码。
具体 API 字段要依据供应商文档。
映射时要处理：
- 不支持的 Schema 关键字；
- 名称长度或字符约束；
- 输出 Schema；
- 工具注解；
- 工具数量上限；
- 描述长度；
- 版本信息；
- 风险元数据。

### 7. 模型调用 ID 与 MCP 请求 ID
假设模型返回：
~~~text
tool call id = call_abc
tool = search_flights
~~~
Host 发送 MCP 请求：
~~~text
JSON-RPC id = 42
method = tools/call
~~~
这两个 ID 用途不同。
- call_abc 关联模型的调用意图与工具结果；
- 42 关联 MCP 请求与响应。
Host 的 Trace 应记录映射：
~~~text
trace_id
 ├─ model_tool_call_id = call_abc
 ├─ mcp_request_id = 42
 └─ business_request_id = travel-781
~~~
不能直接假设它们可以互换。

### 8. 不用 MCP 的 Function Calling
最简单的应用可以直接调用内部函数。
~~~text
模型生成 search_weather
        ↓
Host 查本地注册表
        ↓
直接执行 TypeScript/Python 函数
        ↓
回传结果
~~~
适合：
- 工具只服务一个应用；
- 同一团队维护；
- 不需要协议化发现；
- 需要低延迟进程内调用；
- 工具数量少且稳定。
所以 Function Calling 不依赖 MCP。

### 9. 不用 Function Calling 的 MCP
确定性程序也可以调用 MCP。
例如：
~~~text
每晚 02:00 定时任务
        ↓
MCP Client 初始化
        ↓
读取 report://daily Resource
        ↓
写入归档
~~~
工具选择由固定代码决定。
模型没有参与。
所以 MCP 也不依赖原生 Function Calling。

### 10. 最小 MCP 验证目标
“跑过 MCP”不能只证明 Server 进程能启动。
最小验证应覆盖：
1. 建立传输；
2. initialize；
3. notifications/initialized；
4. tools/list；
5. 合法 tools/call；
6. 非法参数；
7. 未知工具；
8. 连接关闭。
更完整还应覆盖：
- 超时；
- 取消；
- Server 崩溃；
- 重连；
- 列表变化；
- 并发；
- 授权；
- 日志。

### 11. 设计一个只读 Demo 工具
先选择低风险、确定性的工具。
例如：
~~~text
工具名：add
用途：计算两个数字之和
输入：a 和 b，都是 number
输出：a + b
~~~
为什么先用加法：
- 没有网络依赖；
- 没有副作用；
- 结果容易验证；
- 适合排除业务干扰；
- 便于故意测试非法参数。

### 12. Server 侧伪代码
下面是概念伪代码，不绑定某个 SDK 版本。
~~~typescript
const server = createMcpServer({
  name: "math-demo",
  version: "1.0.0",
});

server.registerTool({
  name: "add",
  description: "返回两个数字之和",
  inputSchema: {
    type: "object",
    properties: {
      a: { type: "number" },
      b: { type: "number" },
    },
    required: ["a", "b"],
    additionalProperties: false,
  },
  handler: async ({ a, b }) => ({
    content: [{ type: "text", text: String(a + b) }],
  }),
});

await server.connect(createStdioTransport());
~~~
实际代码应以当前官方 SDK 文档为准。

### 13. Client 侧验证伪代码
~~~typescript
const client = createMcpClient({
  name: "demo-client",
  version: "1.0.0",
});

await client.connect(
  createStdioClientTransport({
    command: "node",
    args: ["server.js"],
  })
);

const tools = await client.listTools();
assert(tools.some(tool => tool.name === "add"));

const result = await client.callTool({
  name: "add",
  arguments: { a: 2, b: 3 },
});

assert(readText(result) === "5");

await client.close();
~~~
SDK 通常会封装 initialize。
为了面试理解，仍然要知道底层生命周期存在。

### 14. 协议层消息顺序
不依赖 SDK 的概念顺序：
~~~text
Client → initialize
Server → initialize result
Client → notifications/initialized

Client → tools/list
Server → tools/list result

Client → tools/call(add, {a: 2, b: 3})
Server → tools/call result("5")

Client → close transport
Server → clean up
~~~
如果跳过初始化直接 tools/list，符合规范的实现应拒绝或视为协议错误。

### 15. 合法参数测试
测试：
~~~json
{
  "name": "add",
  "arguments": {
    "a": 2,
    "b": 3
  }
}
~~~
期望：
- 调用成功；
- 结果是 5；
- 日志有 request ID；
- 响应能正确关联。

### 16. 非法参数测试
测试：
~~~json
{
  "name": "add",
  "arguments": {
    "a": "two",
    "b": 3
  }
}
~~~
期望：
- Server 或 SDK 的输入校验拒绝；
- 不进入真实 handler；
- 返回可识别错误；
- 不崩溃；
- 日志不泄露内部堆栈。

### 17. 未知工具测试
测试：
~~~json
{
  "name": "subtract",
  "arguments": {
    "a": 2,
    "b": 3
  }
}
~~~
期望：
- 明确拒绝未知工具；
- 不回退到任意代码执行；
- 不让模型动态决定函数名后直接反射调用。

### 18. 超时测试
再增加一个故意等待的工具。
~~~typescript
server.registerTool({
  name: "slow_echo",
  inputSchema: {/* ... */},
  handler: async ({ text }) => {
    await delay(10_000);
    return text;
  },
});
~~~
Client 设置更短超时。
检查：
- Host 是否能取消等待；
- Server 是否感知取消；
- 子进程是否残留；
- 后续调用是否仍可用；
- 日志是否区分超时与业务错误。

### 19. Server 崩溃测试
让 Server 在请求中退出。
检查：
- Client 是否识别传输断开；
- 正在等待的请求是否失败；
- 是否有限重连；
- 是否重新 initialize；
- 是否刷新工具列表；
- 是否避免重复写操作。
对于只读 add 工具，重试风险低。
对于预订工具，不能直接照搬重试策略。

### 20. 用 MCP Inspector 做交互验证
官方 MCP Inspector 可以帮助检查 Server。
常见用途：
- 查看连接；
- 查看工具列表；
- 检查输入 Schema；
- 手动调用工具；
- 读取 Resource；
- 获取 Prompt；
- 观察错误。
Inspector 适合开发阶段。
它不代表系统已经完成生产安全验收。

### 21. 再接上模型
协议验证通过后，再接模型。
流程：
~~~text
tools/list
  ↓
Host 过滤并转换工具
  ↓
模型产生 Function Calling
  ↓
Host 校验
  ↓
tools/call
  ↓
结果关联到模型 tool call ID
  ↓
模型输出最终答案
~~~
这样排查更清楚。
先证明 MCP 链路正确，再评估模型选择。
不要把两层同时接上后，只看最终一句回答。

### 22. 怎样回答“有没有实际跑过”
如果只做过 Demo，可以这样回答：
> 我用官方 SDK 做过一个本地 stdio 的最小 Server，暴露只读 add 工具，并用 Client/Inspector 验证了 initialize、tools/list 和 tools/call。我还测试了非法参数、未知工具、超时和 Server 退出。之后把 MCP Tool 映射成模型工具，模型生成调用后由 Host 再通过 MCP 执行。我验证的是最小闭环，没有把它说成远程多租户生产部署；生产还需要 OAuth、资源级授权、限流、审计、幂等和高可用。
如果没有做过，可以这样回答：
> 我目前没有亲手跑过，不会虚构。但我理解验证路径：先用只读工具完成 initialize、initialized、tools/list、tools/call 和关闭，再注入非法参数、超时与 Server 崩溃；协议闭环通过后，才接模型做 Function Calling 映射，并用完整 Trace 关联模型调用 ID、MCP request ID 和业务请求。
诚实不会自动扣分。
含糊吹嘘才容易在追问中暴露。

## 工程实践与边界

### 1. 分开测试两层
测试矩阵：

| 层 | 测什么 |
| --- | --- |
| Function Calling | 调用判断、工具选择、参数 |
| MCP | 初始化、列举、调用、错误、传输 |
| Host 映射 | Schema 转换、ID 关联、权限 |
| 业务执行 | 真实结果、幂等、事务 |
| 端到端 | 任务完成与最终答案 |

### 2. 先使用只读工具
开发早期不要用真实扣款或删除接口验证。
优先：
- 加法；
- Echo；
- 查询测试数据；
- 读取临时目录；
- Mock 航班。
确认安全策略后再逐步增加副作用。

### 3. 工具定义映射要保留语义
检查：
- required 是否保留；
- additionalProperties 是否处理；
- enum 是否丢失；
- 描述是否截断；
- 不支持关键字怎样降级；
- 输出结构怎样传回；
- 工具版本怎样追踪。

### 4. 双层校验
Host 在发起 MCP 调用前校验一次。
Server 收到调用后再校验一次。
原因：
- Host 防止模型或用户恶意输入；
- Server 不能信任任何 Client；
- 两边可能版本不同；
- Server 还要做业务授权。

### 5. 双层授权
Host 控制：
- 当前用户是否可看到工具；
- 是否需要确认；
- 是否允许把数据传给 Server。
Server 控制：
- 凭据是否有效；
- 当前主体能否访问具体资源；
- 是否跨租户；
- 是否满足业务规则。

### 6. 设计超时预算
例如总预算 8 秒：
~~~text
模型选工具：2 秒
MCP 与业务执行：4 秒
模型总结：1.5 秒
缓冲：0.5 秒
~~~
实际值要压测。
不能让每层都独立使用整个总预算。

### 7. 写操作使用业务幂等键
不要把 MCP request ID 当成天然业务幂等键。
重连后新的 request ID 可能对应同一业务动作。
幂等键应该由稳定业务语义产生或由 Host 明确保存。

### 8. Trace 贯穿多层
建议记录：
~~~text
trace_id
user_id
model_response_id
model_tool_call_id
mcp_server_id
mcp_request_id
tool_name
tool_version
business_request_id
idempotency_key
authorization_decision
duration
status
~~~
敏感参数要脱敏。

### 9. 生产还要验证什么
最小 Demo 之外：
- 远程认证；
- OAuth；
- 资源级授权；
- 多租户隔离；
- 限流；
- 会话恢复；
- 负载均衡；
- 灰度升级；
- Server 列表变化；
- 提示注入；
- 数据出境；
- 审计保留；
- 灾难恢复。

### 10. 不把 MCP 当成业务事务协议
MCP 调用成功只表示协议和工具返回成功。
复杂业务事务仍由领域服务处理。
例如：
~~~text
创建申请成功
预订机票失败
~~~
是否撤销申请、保留待处理还是人工介入，是业务补偿策略。
MCP 不自动决定。

## 常见误区

### 误区一：MCP 是 Function Calling 2.0
错误。
二者位于不同接口。

### 误区二：接上 MCP 后模型就会使用工具
错误。
还要把工具映射给模型，并评估模型决策。

### 误区三：支持 Function Calling 的模型天然支持 MCP
错误。
Host 必须实现 MCP Client 和映射循环。

### 误区四：没有 Function Calling 就不能用 MCP
错误。
确定性程序可以直接调用。

### 误区五：tools/call 成功就证明生产可用
错误。
还缺授权、故障、安全、性能和治理验证。

### 误区六：MCP request ID 可以解决重复下单
错误。
业务幂等需要稳定幂等键与服务端去重。

### 误区七：Inspector 测通就算完整测试
错误。
Inspector 主要是开发与调试工具。

### 误区八：所有 Schema 可以无损映射到任何模型 API
错误。
供应商支持的 JSON Schema 子集和限制可能不同。

## 面试追问

### 追问一：谁真正执行工具
MCP Server 执行它暴露的 Tool 逻辑，可能再调用真实业务 API。
Host 负责是否允许发起调用。

### 追问二：模型知道自己在用 MCP 吗
不一定。
模型通常只看到被转换后的工具定义。

### 追问三：为什么先测 MCP 再接模型
这样能把协议问题与模型决策问题分离。
最终答案错时更容易定位。

### 追问四：怎样测试非法参数
构造缺字段、错类型、额外字段和越界值。
验证 Server 拒绝且 handler 不执行。

### 追问五：Server 崩溃后能自动重试吗
要看操作是否幂等、结果是否确定和预算。
写操作结果未知时应先查询状态。

### 追问六：MCP Tool 能映射到任何供应商吗
概念上可以适配。
具体要处理工具字段、Schema 子集、并行调用和结果消息差异。

### 追问七：如何证明不是只会背协议
讲清一个错误定位过程。
例如 stdout 混入日志导致 JSON-RPC 解析失败，或者 Schema 中字符串与数字不一致导致 tools/call 被拒绝。

### 可背诵完整版
> Function Calling 解决模型怎样向 Host 表达工具调用意图，MCP 解决 Host 怎样通过 Client 发现、连接和调用外部 Server。两者位于不同接口，可以组合：Host 用 tools/list 获取工具并映射给模型，模型产生调用后，Host 校验再发 tools/call。它们也可以独立使用。实际验证 MCP 时，我会先用只读工具跑通 initialize、initialized、tools/list、tools/call 和关闭，再测试非法参数、未知工具、超时、Server 崩溃和重连；之后才接模型，并用 Trace 关联模型 tool call ID、MCP request ID 和业务请求。Demo 跑通不等于生产可用，远程授权、幂等、审计、隔离和高可用仍需单独完成。
