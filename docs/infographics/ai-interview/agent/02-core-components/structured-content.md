# 2. Agent 的基本架构由哪些核心组件构成？

## Overview

系统拆解 AI Agent 的模型、工具、记忆、规划和执行环境等核心组件，说明各模块在完整 Agent 架构中的职责分工

## Learning Objectives

The viewer will understand:

1. 说明“Agent 的基本架构由哪些核心组件构成？”的组成部分及其关系
2. 说明关键工程边界、失败模式与验证方法

## Source Content (Verbatim)

## 60 秒回答

我会把 Agent 分成六个职责，而不是死记“四大组件”：模型与指令负责理解和决策；编排器维护循环与停止条件；工具层连接外部能力；状态与记忆保存当前进度和可检索历史；安全层负责身份、权限、输入输出校验与人工审批；可观测和评测层负责追踪、回放与质量门禁。

规划不一定是独立服务，它可能只是编排器中的一次模型调用。关键不是组件数量，而是每个职责有清晰边界，尤其要把“模型提出动作”和“运行时授权并执行动作”分开。

## 详细解析

### 模型与指令

模型读取系统指令、用户目标、相关上下文和工具描述，输出文本或结构化动作。模型路由可以按任务难度选择不同能力和成本的模型，但路由本身也要评测，不能只凭长度判断。

### 编排与状态机

编排器维护任务 ID、当前步骤、预算、重试次数和结束原因。它解析模型输出、调用策略检查、调度工具并写回结果。ReAct、Plan-and-Execute 或固定 Workflow 都属于这一层的控制策略。

### 工具与执行环境

工具定义应包含明确名称、用途、输入模式和错误语义。执行环境负责鉴权、超时、并发限制、幂等和沙箱；模型不能直接持有生产凭据。模型上下文协议（MCP）可以标准化上下文和工具连接，但不会替系统完成授权与安全治理。

### 状态与记忆

运行状态保存当前任务必须精确恢复的信息；长期记忆保存跨任务可能有价值的信息。两者不能混为一个向量库。订单状态等结构化事实适合关系或键值存储，文档片段才适合语义检索。

### 安全、可观测与评测

安全层在执行前做权限与政策判断，在执行后做输出过滤和审计。可观测层记录每步输入摘要、动作、工具延迟、错误和成本；评测层用固定数据集和线上抽样判断任务成功、轨迹质量和安全性。

## 工程实践与边界

- 给消息、任务状态和长期记忆都带上 tenant_id 与 user_id，并在数据库层执行隔离。
- 凭据由运行时按工具和请求临时注入，模型上下文只看到不敏感的能力描述。
- 对工具响应做大小限制、类型校验和不可信内容标记，避免外部文本反向控制 Agent。
- 将任务检查点与副作用提交分开；恢复时先确认外部动作是否已经发生。
- 个人可识别信息（PII）设置数据分类、保留期、删除接口和访问审计，不因“记忆有用”而永久保存。

## 常见误区

- **把规划器当成必需的独立组件**：简单 Agent 可以由同一模型在循环内规划。
- **把向量数据库当成全部记忆**：精确状态、权限和事务数据不应依赖相似度检索。
- **认为接入 MCP 就获得安全性**：协议提供互操作能力，授权策略仍由实现方负责。
- **只画正常路径**：没有失败、重试、暂停、取消和人工接管的架构图不完整。

## 面试追问

> **面试官：** 哪个组件最容易被低估？
>
> **候选人：** 编排运行时。模型和工具容易被看到，但真正保证预算、权限、重试、状态转移和停止的是运行时。

> **面试官：** 规划和记忆能否都放进 Prompt？
>
> **候选人：** 小任务可以，但长任务会遇到上下文成本、精确恢复和并发一致性问题。关键状态应外置并结构化，按需投影进上下文。

> **面试官：** 如何替换模型而不重写系统？
>
> **候选人：** 在模型适配层统一消息、结构化输出、工具调用和错误类型，并用契约测试与回归评测验证行为差异。
