## 本篇目标

学完这一篇，你应该能够：

1. 纠正“所有 AI 对话都应该用 WebRTC”这个前提。
2. 解释 WebRTC 不是一根普通网络管道，而是一套实时音视频协议与浏览器 API 栈。
3. 说清 ICE、STUN、TURN、RTP/RTCP、DTLS-SRTP、SDP 与 DataChannel 的作用。
4. 比较 WebRTC 与 WebSocket 在可靠性、顺序、媒体时钟、抖动、拥塞和二进制消息上的差异。
5. 画出浏览器实时语音 Agent 的典型端到端链路。
6. 根据文本、音频、视频、实时性和部署环境选择 WebRTC、WebSocket、SSE 或普通 HTTP。
7. 处理麦克风权限、NAT、TURN、网络切换、打断、重复工具调用和隐私合规。

## 60 秒回答

> WebRTC 不是所有 AI 对话都必需。纯文本生成通常用 SSE，频繁双向文本或控制事件可用 WebSocket，异步音频也可以上传文件。WebRTC 的价值主要出现在浏览器或移动端的实时双向音视频：它不只提供传输，还整合了媒体采集、编解码、时间戳、抖动缓冲、拥塞控制、丢包处理、回声消除、ICE 连通性检查和加密媒体传输。
>
> WebSocket 更像可靠、有序的双向消息管道，适合文本、JSON、工具状态和必须完整到达的控制消息。如果直接用它传实时音频，应用要自己处理分片、时间戳、编解码、抖动、丢包和播放时钟；底层重传还可能让过期音频阻塞后续音频。WebRTC 的媒体路径更强调“按时播放”，通常允许通过丢弃、掩盖、码率调整等方式换取连续低延迟。
>
> WebRTC 也不表示固定采用 P2P 或仅使用 UDP。它通过 ICE 选择可达路径，可能直连，也可能经 TURN 中继，AI 场景常连接到服务端媒体网关。SDP 信令需要另用 HTTP 或 WebSocket。工程上我会同时设计媒体通道和业务事件通道，监控端到端时延、抖动、丢包、RTT、TURN 使用率与打断成功率，并确保工具调用有独立状态机和幂等控制，不能因语音重连或打断而重复执行副作用。

## 详细解析

### 先说人话：WebSocket 是快递传送带，WebRTC 是实时直播系统

WebSocket 擅长把一条条消息可靠、有序地送到对面：

```text
消息 1 -> 消息 2 -> 消息 3
```

如果消息 2 丢了，可靠传输会努力重传，保持顺序和完整。

对于转账指令、工具状态和 JSON 消息，这通常是好事。

实时语音的目标不同。

20 毫秒前的一小段音频如果迟到 2 秒，再完整送到也没有多大播放价值，反而会挡住后续声音。

WebRTC 更像完整直播系统：

- 采集麦克风；
- 编码音频；
- 给媒体加时间戳；
- 选择网络路径；
- 根据拥塞调整发送；
- 接收后用抖动缓冲平滑播放；
- 对少量丢包做掩盖或恢复；
- 加密媒体；
- 处理回声和设备切换。

所以二者的核心差异不只是“一个 UDP、一个 TCP”。

### 先纠正题目：不是所有 AI 对话都要 WebRTC

不同“AI 对话”可能是：

| 场景 | 更常见的起点 |
| --- | --- |
| 网页文本生成 | SSE |
| 双向文本聊天与协作 | WebSocket |
| 上传录音后异步转写 | HTTP 文件上传 + 任务查询 |
| 按住说话、松开上传一段 | HTTP 或 WebSocket |
| 浏览器实时双向语音 | WebRTC |
| 实时视频理解 | WebRTC |
| 服务到服务、受控网络中的音频流 | WebSocket、gRPC 或专业媒体协议均可评估 |

协议选型必须从实时性、媒体类型和网络环境出发。

### 什么是 WebRTC

WebRTC 全称 **Web Real-Time Communication**。

它不是单一协议，而是一组浏览器 API 与 IETF 实时通信协议的组合。

常见组成：

```text
媒体采集：getUserMedia
连接对象：RTCPeerConnection
能力协商：SDP offer / answer
路径发现：ICE
公网地址发现：STUN
中继：TURN
媒体传输：RTP / RTCP
媒体加密：DTLS-SRTP 等规定机制
数据通道：SCTP over DTLS
```

初学者不需要一次背下所有缩写，但要知道 WebRTC 已经帮应用处理了大量媒体工程。

### 什么是 WebSocket

WebSocket 是双向消息协议。

浏览器建立连接后可以发送：

- 文本消息；
- 二进制消息。

它提供帧、消息、Ping/Pong 与关闭语义。

但它不直接提供：

- 麦克风采集；
- 音频编码选择；
- 媒体时间戳；
- 抖动缓冲；
- 回声消除；
- 丢包掩盖；
- 自适应码率；
- NAT 穿透；
- 媒体设备切换。

这些需要应用或额外媒体库承担。

### WebRTC 建连的第一步：信令

WebRTC 需要双方交换：

- SDP offer；
- SDP answer；
- ICE candidates。

这个交换过程叫 **信令**。

重要的是：WebRTC 标准没有规定你必须用哪种信令服务。

可以使用：

- HTTP；
- WebSocket；
- A2A 或其他应用协议；
- 已有消息系统。

因此常见架构是：

```text
HTTP / WebSocket：交换 WebRTC 信令
WebRTC：传输实时媒体和可选 DataChannel 数据
```

说“用了 WebRTC 就不需要 WebSocket”通常也不准确。

### SDP 是什么

SDP，即 Session Description Protocol，用来描述会话能力和媒体参数。

它可以表达：

- 音频或视频媒体；
- 编解码器；
- 媒体方向；
- 网络候选信息引用；
- 安全与传输协商所需属性；
- 数据通道相关参数。

SDP 不是媒体本身，也不是把语音装进一个大 JSON。

生产实现通常交给 WebRTC 库处理，不要用字符串拼接随意修改。

### ICE、STUN 与 TURN

#### 为什么需要 ICE

浏览器和服务端常位于 NAT、防火墙或企业代理之后。

本机看到的私网地址并不一定能被对方访问。

**ICE**，即 Interactive Connectivity Establishment，会：

1. 收集候选地址；
2. 与对方交换候选；
3. 对候选对执行连通性检查；
4. 选择可用路径；
5. 在网络变化时进行必要恢复。

#### STUN 做什么

STUN 帮助端点发现经过 NAT 后的公网映射地址，并参与连通性检查。

它不是媒体中继，也不能保证穿透所有防火墙。

#### TURN 做什么

直连不可达时，TURN Server 可以中继流量：

```text
Client -> TURN -> AI Media Server
```

TURN 提升连通性，但会带来：

- 额外网络跳数；
- 带宽成本；
- 中继容量压力；
- 更高延迟；
- 运维和凭证治理。

没有 TURN 的 WebRTC 演示，可能在开发网络正常，却在企业网络或移动网络大量失败。

### WebRTC 不等于 P2P

“PeerConnection”容易让人以为媒体必须在两个浏览器间直连。

实际路径可能是：

```text
浏览器 <-> 浏览器
浏览器 <-> SFU / 媒体服务器
浏览器 <-> AI 实时网关
浏览器 <-> TURN <-> AI 实时网关
```

在 AI 语音场景，对端经常是服务端媒体网关，而不是另一个用户浏览器。

所以更准确的说法是：

```text
WebRTC 建立 PeerConnection，并通过 ICE 选择路径；
路径可能直连，也可能经过服务端或 TURN 中继。
```

### WebRTC 不等于只用 UDP

实时媒体通常优先选择低延迟的 UDP 路径。

但实际网络可能迫使 ICE 使用其他可达候选或 TURN 中继，TURN 也可能通过 TCP/TLS 等方式提供回退。

面试中不要简单回答：

```text
WebRTC = UDP
WebSocket = TCP
```

应该说：

> WebSocket 通常提供可靠有序消息流；WebRTC 的路径由 ICE 选择，媒体设计倾向低延迟并具备实时媒体控制，实际是否直连、是否 UDP 取决于网络和中继。

### RTP 与 RTCP

WebRTC 媒体通常使用 RTP/RTCP 体系。

#### RTP

RTP 为媒体提供：

- 序列号；
- 时间戳；
- 负载类型；
- 来源标识；
- 媒体包传输语义。

接收端可以根据序列号发现丢包，根据时间戳安排播放。

#### RTCP

RTCP 提供控制和反馈信息，例如：

- 丢包；
- 往返时延；
- 抖动；
- 接收质量；
- 发送与接收统计。

发送端和拥塞控制算法可以据此调整码率和发送行为。

### 为什么“按时到达”比“全部到达”更重要

实时语音每个媒体包都有播放截止时间。

如果一个包已经迟到：

- 重传后也赶不上播放；
- 等它会增加整体延迟；
- 后续正常音频被阻塞；
- 用户感觉机器人卡顿。

实时媒体系统可能使用：

- 抖动缓冲；
- 丢包隐藏；
- 选择性重传；
- 前向纠错；
- 码率调整；
- 关键帧请求；
- 过期包丢弃。

目标是让用户听起来连续，而不是保证每个历史字节最终到达。

### WebSocket 的可靠有序语义

WebSocket 的经典传输建立在可靠有序字节流之上。

优点：

- 消息按顺序交付；
- 丢失数据会重传；
- 编程模型简单；
- 适合控制、状态和结构化消息。

对实时音频的挑战：

- 丢包重传会让后续数据等待；
- 过期音频仍占带宽；
- 应用要自己定义音频时间戳和播放队列；
- 应用要自己做丢包、抖动和码率处理；
- 音频与控制消息可能互相阻塞。

这不表示 WebSocket 不能传音频，只表示媒体工程责任转移给应用。

### 抖动缓冲是什么

网络包不会严格按固定间隔到达。

例如发送端每 20 毫秒发一个音频包，接收端可能在：

```text
18ms、25ms、14ms、31ms……
```

收到它们。

**抖动缓冲**先积累少量媒体，再按稳定节奏播放。

缓冲过小：

- 容易来不及播放；
- 断音增加。

缓冲过大：

- 更平滑；
- 但对话延迟上升。

WebRTC 实现会动态处理这一取舍。

### 拥塞控制

当网络可用带宽下降时，系统不能继续无上限发送。

WebRTC 会根据反馈调整：

- 码率；
- 分辨率；
- 帧率；
- 重传策略；
- 发送节奏。

WebSocket 应用若直接传音频，至少也要监控：

- 发送缓冲；
- 往返时延；
- 接收积压；
- 音频播放延迟；
- 丢包或断线。

否则慢网络会让延迟越积越大。

### 媒体加密

WebRTC 要求安全媒体传输，并通过规定的密钥协商与 SRTP 机制保护媒体。

WebSocket 应使用 `wss`，即 TLS 加密的 WebSocket。

但传输加密不等于端到端只有用户能解密。

在服务端 AI 场景中，媒体网关通常需要解码音频才能执行：

- 语音识别；
- 语音活动检测；
- 模型推理；
- 文本转语音。

所以要准确说明信任边界和数据处理方。

### DataChannel 是什么

WebRTC 还提供 `RTCDataChannel`。

它基于 SCTP over DTLS，可以配置：

- 有序或无序；
- 可靠重传；
- 最大重传次数；
- 最大存活时间。

适合传：

- 实时会话事件；
- 字幕增量；
- 打断指令；
- 光标或状态；
- 与媒体同一 PeerConnection 的辅助数据。

DataChannel 不是 RTP 媒体轨道。

也不表示所有业务事件都必须搬到 WebRTC；很多系统仍用独立 WebSocket 或 HTTP 处理工具调用和控制面。

### WebRTC 与 WebSocket 核心对比

| 维度 | WebRTC 媒体 | WebSocket |
| --- | --- | --- |
| 主要目标 | 低延迟实时音视频 | 双向应用消息 |
| 典型数据 | 音频、视频、可选 DataChannel | 文本、JSON、二进制 |
| 传输倾向 | ICE 选择路径，媒体偏实时 | 可靠有序消息流 |
| 迟到数据 | 可按媒体策略丢弃或掩盖 | 通常等待可靠有序交付 |
| 媒体时间戳 | RTP 提供 | 应用自行定义 |
| 抖动缓冲 | 实现内置 | 应用自行实现 |
| 拥塞控制 | 媒体栈提供 | 应用和底层传输共同处理 |
| 编解码 | 协商并由媒体栈处理 | 应用自行选择和处理 |
| 回声消除 | 浏览器媒体链常提供 | 不提供 |
| NAT 穿透 | ICE/STUN/TURN | 通常通过标准 Web 入口连接服务端 |
| 信令 | 需要额外通道 | 自身就是消息通道 |
| 实现复杂度 | 媒体栈复杂，但能力完整 | 基础简单，自己做媒体则复杂 |

### AI 实时语音的端到端链路

典型架构：

```text
浏览器麦克风
  -> WebRTC 音频轨
  -> AI Media Gateway
  -> VAD / ASR
  -> 对话编排与 LLM
  -> Tools
  -> TTS
  -> WebRTC 音频轨
  -> 浏览器扬声器
```

旁路事件可能走：

```text
DataChannel / WebSocket / SSE
  - 转写增量
  - 工具调用状态
  - 引用
  - 错误
  - 用户确认
  - 会话结束
```

### VAD 与回合检测

**VAD** 是 Voice Activity Detection，即语音活动检测。

它判断用户何时开始和停止说话。

实时 Agent 需要决定：

- 用户是否正在讲话；
- 多久静音视为一句结束；
- 环境噪声是否误触发；
- 模型何时开始回答；
- 用户再次讲话时是否打断。

阈值太激进会抢话，太保守会让响应很慢。

### Barge-in：用户打断 AI

**Barge-in** 指 AI 正在说话时，用户开始讲话并打断它。

完整流程可能是：

```text
检测用户开始说话
  -> 停止或淡出当前 TTS 播放
  -> 取消尚未生成的音频
  -> 保留已经确认的会话状态
  -> 继续接收用户语音
  -> 更新对话回合
```

最容易出错的是工具调用。

如果 AI 已经提交退款，用户打断不能让系统因为“上一轮被取消”再次提交。

媒体取消与业务副作用必须分开建模。

### 混合架构往往最好

一个成熟实时 Agent 不必只选一种协议。

例如：

```text
WebRTC：上行/下行音频
DataChannel：低延迟字幕和打断事件
HTTP：会话创建、鉴权、文件上传
WebSocket：复杂双向业务事件
普通 API/MCP：后端工具调用
对象存储：录音与大文件
```

选型按数据特性分离，比强迫所有内容走一条连接更清晰。

### WebRTC 的常见失败路径

#### 1. 麦克风权限被拒绝

系统应：

- 解释为什么需要权限；
- 提供重新授权指引；
- 允许切换文本输入；
- 不把拒绝解释成网络失败。

#### 2. ICE 连接失败

可能原因：

- 企业防火墙阻止 UDP；
- TURN 不可达；
- TURN 凭证过期；
- 候选交换失败；
- 服务端端口或证书配置错误。

#### 3. 连接成功但没有声音

检查：

- Track 是否发送；
- transceiver 方向；
- SDP 编解码协商；
- 浏览器自动播放策略；
- 音频设备选择；
- 服务端解码与回传；
- 是否被静音。

#### 4. 声音断续

检查：

- 丢包；
- 抖动；
- RTT；
- 可用带宽；
- CPU 编解码压力；
- TURN 路径；
- 抖动缓冲和播放状态。

#### 5. 延迟越来越大

可能是：

- 服务端排队；
- ASR/LLM/TTS 流水线阻塞；
- 播放队列未丢弃过期音频；
- WebSocket 自建音频无背压；
- 网络拥塞；
- 生成速度慢于播放需求。

#### 6. Wi-Fi 切换移动网络后断开

网络接口变化可能需要 ICE 恢复或重新协商。

应用应保存会话业务状态，让媒体重连不等于重建所有工具任务。

### WebSocket 传音频何时仍合理

WebSocket 不是不能传音频。

以下条件下可能合理：

- 服务到服务；
- 网络环境受控；
- 音频是短片段而非连续通话；
- 已有成熟分片、编解码和播放实现；
- 不需要浏览器 NAT 穿透；
- 实时性要求没有互动通话那么高；
- 团队更重视接入简单和可观测性。

要明确自己承担了哪些媒体栈责任。

### WebRTC 何时不值得

- 只有文字；
- 用户上传一段录音等待结果；
- 单向播放预先生成的音频；
- 没有低延迟打断需求；
- Client 环境不支持 WebRTC；
- 企业网络和合规环境使 TURN 成本过高；
- 团队没有媒体运维能力，而业务价值尚未验证。

可以先用简单方案验证产品，再决定是否投入实时媒体工程。

## 工程实践与边界

### 建连流程

一个简化的浏览器到 AI Gateway 流程：

```text
1. Client 通过 HTTPS 登录并创建实时会话
2. Server 返回短期会话凭证和 ICE 配置
3. Client 请求麦克风权限
4. Client 创建 RTCPeerConnection
5. 添加音频 Track
6. 创建 SDP offer
7. 通过信令通道发送 offer
8. Server 返回 SDP answer
9. 双方交换 ICE candidates
10. ICE 选择候选对并建立安全连接
11. 开始媒体与事件交互
12. 结束时停止 Track、关闭 PeerConnection 和业务会话
```

信令 API 要认证，短期凭证要绑定用户、租户和会话。

### 会话状态不要只存在 PeerConnection 中

持久化：

- 业务 `conversation_id`；
- 当前用户和租户；
- 已确认的消息；
- 工具调用与副作用状态；
- 媒体连接代次；
- 录音与保留策略；
- 取消和结束状态。

PeerConnection 是媒体连接，不是业务数据库。

### 工具调用独立状态机

建议状态：

```text
PROPOSED
  -> VALIDATING
  -> WAITING_FOR_CONFIRMATION
  -> EXECUTING
  -> SUCCEEDED | FAILED | UNKNOWN
```

每次调用使用：

- 唯一调用 ID；
- 业务幂等键；
- 绑定的用户确认；
- 参数快照；
- 审计记录。

语音打断、媒体断线和 ICE 重启不能自动重置该状态机。

### 延迟预算

端到端语音延迟由多段组成：

```text
采集和编码
+ 上行网络
+ VAD / ASR
+ 编排与模型首 token
+ 工具等待（如有）
+ TTS 首音频
+ 下行网络
+ 抖动缓冲和播放
```

只优化 WebRTC 连接不能消除模型排队或 TTS 延迟。

应为每一段建立 trace 和时间戳。

### 核心指标

网络与媒体：

- ICE 建连成功率；
- 建连耗时；
- 候选对类型；
- TURN 使用率；
- RTT；
- jitter；
- packet loss；
- 可用/发送/接收码率；
- 音频隐藏样本或丢弃情况；
- ICE restart 次数。

对话体验：

- 用户开口到识别增量延迟；
- 句末到模型开始响应；
- 句末到首音频；
- 端到端首音延迟；
- 打断检测延迟；
- 打断成功率；
- 误抢话率；
- 会话恢复率。

业务：

- 工具成功率；
- 重复副作用率；
- 确认绕过率；
- 人工升级率；
- 每分钟媒体与 TURN 成本。

### TURN 容量与安全

TURN Server 需要：

- 短期凭证；
- 限制可中继目标和端口；
- 用户或租户配额；
- 带宽与并发监控；
- 防滥用和 DDoS；
- 多地域部署；
- 容量压测；
- 凭证轮换。

开放匿名 TURN 会迅速成为公共代理和成本风险。

### 浏览器权限与用户体验

请求麦克风前：

- 说明用途；
- 由明确用户手势触发；
- 展示正在采集的状态；
- 提供静音和结束按钮；
- 允许选择输入输出设备；
- 权限拒绝时提供文本降级；
- 不在用户不知情时持续录音。

### 录音、转写与合规

实时语音可能涉及：

- 生物特征或个人身份信息；
- 通话录音同意；
- 地域数据驻留；
- 未成年人保护；
- 医疗、金融等敏感内容；
- 数据保留和删除；
- 模型训练使用限制。

需要明确区分：

```text
仅实时处理
临时缓冲
长期保存录音
保存转写
保存模型和工具日志
```

默认最小化收集和保留。

### 安全边界

- 信令 API 认证授权；
- ICE/TURN 使用短期凭证；
- 校验 SDP 和消息大小；
- 限制会话时长、码率和并发；
- 防止跨租户会话绑定；
- 工具事件单独授权；
- 转写文本按不可信输入处理；
- 远程语音可能包含提示注入；
- 日志与录音脱敏；
- 会话结束后撤销临时凭证。

语音只是另一种输入通道，不能拥有比文本更高的默认权限。

### 网络切换与恢复

恢复策略：

```text
检测连接状态异常
  -> 短时间等待瞬时恢复
  -> 必要时 ICE restart / 重新协商
  -> 仍失败则重建 PeerConnection
  -> 使用原业务 conversation_id 恢复
  -> 不重复已确认工具动作
  -> 返回最新文本与任务快照
```

媒体可以重新连接，业务状态必须连续。

### 降级路线

```text
WebRTC UDP 直达
  -> WebRTC 经 TURN UDP
  -> WebRTC 经 TURN TCP/TLS 等可达方式
  -> WebSocket/HTTP 音频片段模式
  -> 录音上传异步处理
  -> 纯文本
```

具体路线取决于产品和网络，不应无条件启用所有回退。

### 测试矩阵

至少覆盖：

- 家庭 Wi-Fi；
- 企业网络；
- 4G/5G；
- VPN；
- UDP 被阻断；
- 高丢包、高抖动、高 RTT；
- 网络切换；
- TURN 凭证过期；
- 麦克风拒绝；
- 无可用输入设备；
- 浏览器后台；
- 用户频繁打断；
- 双方同时说话；
- 长会话；
- 工具执行中媒体断线；
- 滚动发布和媒体 Server 排空。

### 选型决策表

| 需求 | 建议 |
| --- | --- |
| 纯文本 token 流 | SSE |
| 高频双向文本/控制 | WebSocket |
| 实时双向语音和打断 | WebRTC |
| 实时视频 | WebRTC |
| 短录音异步识别 | HTTP 上传 |
| 受控网络服务间音频 | 评估 WebSocket、gRPC 或媒体协议 |
| 实时媒体 + 复杂业务事件 | WebRTC + WebSocket/HTTP 混合 |
| 需要离线持久消费 | 消息队列，不由 WebRTC/WS 替代 |

## 常见误区

### 误区一：所有 AI 对话都应该用 WebRTC

错误。只有实时双向媒体和低延迟互动需求足够强时，WebRTC 的复杂度才值得。

### 误区二：WebRTC 就是 P2P

错误。路径可能经过媒体服务器或 TURN，AI 场景常直接连接服务端。

### 误区三：WebRTC 只使用 UDP

不准确。ICE 选择实际可达路径，可能使用中继和回退。

### 误区四：WebSocket 不能传音频

可以传二进制音频，但应用要承担更多分片、时间戳、抖动、播放和拥塞处理。

### 误区五：WebRTC 有了就不需要信令

错误。SDP 和 ICE candidates 仍要通过 HTTP、WebSocket 等信令通道交换。

### 误区六：WebRTC 丢包，所以不可靠

实时媒体的“可靠”是维持可理解、连续和低延迟的播放，而不是每个过期包最终到达。它有反馈、恢复和抗丢包机制。

### 误区七：连接恢复后可以重放上一轮所有事件

媒体可以恢复，业务副作用不能盲目重放。工具调用必须依赖独立状态与幂等键。

### 误区八：使用 DTLS-SRTP 就是端到端只有用户可见

在服务端 AI 场景，媒体通常在 AI 服务端解密处理。必须准确说明数据处理方和信任边界。

### 误区九：延迟高就是网络协议问题

ASR、模型排队、工具调用、TTS 和播放缓冲都可能是瓶颈，要做分段追踪。

## 面试追问

### 追问一：WebRTC 与 WebSocket 最本质的区别

> WebRTC 是面向实时媒体的完整通信栈，内置媒体时钟、抖动、拥塞、丢包和 NAT 穿透能力；WebSocket 是可靠有序的双向消息通道，媒体语义要由应用自行实现。

### 追问二：为什么实时音频不追求每个包最终到达

音频包有播放截止时间。过期后重传价值很低，等待它还会阻塞后续音频；系统更愿意掩盖少量缺失并保持连续低延迟。

### 追问三：WebRTC 为什么还需要 WebSocket

WebRTC 不规定信令通道，SDP 和 ICE 信息可以通过 WebSocket 交换；复杂业务事件也可能继续使用 WebSocket。两者可以组合。

### 追问四：STUN 和 TURN 有什么区别

STUN 主要帮助发现 NAT 映射并参与连通性检查；TURN 在直连不可达时实际中继流量，连通性更强但增加成本和延迟。

### 追问五：DataChannel 能否替代 WebSocket

有时可以承载同会话的低延迟数据，但它依赖 PeerConnection 和 WebRTC 建连。纯业务消息系统可能仍以 WebSocket 更简单，选型取决于生命周期、可靠性配置和基础设施。

### 追问六：AI 语音打断怎样避免重复工具调用

媒体播放、模型生成和工具执行使用独立状态机。工具调用绑定唯一 ID、参数快照、确认和业务幂等键；打断只取消可取消的生成与播放，不自动重置已经提交的副作用。

### 追问七：如何判断 WebRTC 质量

看建连成功率、ICE 候选对、TURN 使用率、RTT、抖动、丢包、码率、首音延迟、打断延迟和恢复率，并把 ASR、模型、工具、TTS 各段耗时关联起来。

### 追问八：企业网络下 WebRTC 失败怎么办

准备可靠 TURN 服务与短期凭证，支持可行的 UDP/TCP/TLS 候选，监控候选对和失败阶段；仍失败时降级到音频片段上传或文本，不要无限重连。

### 追问九：什么时候仍选择 WebSocket 传音频

服务间受控网络、短音频片段、实时要求适中，而且团队已有成熟音频分片、编解码、流控和播放实现时，WebSocket 可能更简单。

### 可直接背诵的完整答案

> WebRTC 不是所有 AI 对话都必需。纯文本通常 SSE 就够，频繁双向事件可以用 WebSocket，异步语音可以上传文件。WebRTC 适合浏览器或移动端的实时双向音视频，因为它是一套完整媒体栈：包含采集、编解码、RTP/RTCP 时间戳和反馈、抖动缓冲、拥塞控制、丢包处理、回声消除、ICE/STUN/TURN 连通性和加密媒体传输。
>
> WebSocket 更像可靠有序的双向消息管道，适合文本、JSON、工具状态和控制消息；直接传音频时，应用要自己处理分片、播放时钟、抖动、拥塞和过期数据，重传还可能导致队头阻塞。WebRTC 也不等于 P2P 或只走 UDP，它可能连接 AI 媒体网关或经 TURN 中继，而且 SDP 与 ICE 信令仍需要 HTTP 或 WebSocket。生产系统常采用混合架构，并把媒体连接与业务状态分离；语音打断或重连不能重复工具副作用，必须使用独立调用状态机、确认和幂等键。
