## 本篇目标

学完这一篇，你应该能够：

- 解释为什么知识库更新不只是向量数据库执行一次 upsert；
- 区分新增、修改、删除、权限变更和模型升级的处理方式；
- 理解 CDC、幂等、内容哈希、lineage、索引别名等基础术语；
- 设计一条可追踪、可重试、可灰度、可回滚的增量更新流水线；
- 说清什么时候局部更新，什么时候应该重建完整新索引。

## 60 秒回答

RAG 持续更新的核心不是“定时重新向量化”，而是把权威数据源的变化可靠地传播到所有派生数据。我的链路通常是：通过 CDC、Webhook 或定时扫描捕获新增、修改、删除和权限变更；给事件分配稳定文档 ID、源版本和幂等键；重新解析受影响文档，生成带版本的 chunk、Embedding、稀疏索引和元数据；写入待发布的新版本，再做数量、抽样内容、ACL 和检索回归验证。

正文、向量、关键词索引、图关系、缓存和权限往往跨多个系统，不能假设一次双写会自然一致。我会使用 Outbox 或可重放事件流，让各消费者幂等更新，并持续对账。删除要通过 tombstone 或删除事件传播到全部副本，权限收紧通常比普通内容更新具有更高优先级。

发布时采用影子索引、灰度和索引别名切换，保留旧版本用于回滚。监控源变更到可检索的 P50/P95 新鲜度、积压、失败和死信、重复与孤儿 chunk、删除传播延迟及版本不一致。切块规则或 Embedding 模型变化时，通常重建完整新索引，避免新旧向量空间混用。

## 详细解析

### 先说人话：更新知识库像更新连锁超市的价格

总部把一瓶水从 3 元改成 2.5 元，事情并不是“在总部表格里改一下”就结束。还要确保：

- 所有门店价签都更新；
- 收银系统价格更新；
- 线上商城和缓存更新；
- 旧促销规则不再生效；
- 更新失败的门店能够补发；
- 出错后可以查出哪家门店仍在使用旧价格。

RAG 的源文档、解析文本、chunk、向量、关键词索引、缓存和摘要，就像多个门店和系统。只更新其中一个，用户仍可能看到旧答案。

### 贯穿案例：差旅制度从 450 元更新到 500 元

假设公司发布 2025 版差旅制度：

- 深圳普通员工酒店上限从 450 元改为 500 元；
- 旧版制度从 2025 年 1 月 1 日起失效；
- 新制度只允许本公司员工查看；
- 原 PDF 的第 8 页表格是答案来源。

系统至少要完成：

1. 发现源文件发生变化；
2. 识别这是新版本，不是重复上传；
3. 解析第 8 页并重建受影响 chunk；
4. 生成与当前模型匹配的新向量；
5. 更新关键词索引、元数据和 ACL；
6. 让旧版 450 元不再参与当前检索；
7. 清除可能缓存的旧答案；
8. 验证问题“深圳住宿上限是多少”现在返回 500 元；
9. 如果新解析器把表格读错，能够快速切回旧索引。

这说明更新是一条端到端数据流水线，不是单一数据库操作。

### 先认识八个关键术语

| 术语 | 通俗解释 |
| --- | --- |
| 权威源（Source of Truth） | 最终认定哪份数据正确的原始系统 |
| CDC | Change Data Capture，捕获数据库新增、修改和删除 |
| Webhook | 源系统发生事件后主动通知下游 |
| 幂等（Idempotency） | 同一事件处理多次，最终结果与处理一次相同 |
| Upsert | 有记录就更新，没有就新增 |
| Tombstone | 表示“这条数据已删除”的删除标记或事件 |
| Lineage | 从源文档到 chunk、向量、摘要的派生关系 |
| 索引别名 | 应用使用稳定名称，背后可切换到不同索引版本 |

其中最容易被面试官追问的是幂等。消息系统常采用“至少投递一次”，同一个变更可能被重复消费。如果没有幂等设计，知识库会产生重复 chunk 或重复关系。

### 先定义持续更新的五个目标

| 目标 | 要回答的问题 |
| --- | --- |
| 新鲜度 | 源更新后多久可以被检索 |
| 完整性 | 所有变化是否都传播成功 |
| 一致性 | 正文、向量、权限、缓存是否指向同一版本 |
| 可追溯 | 一个答案能否追到源版本和处理链路 |
| 可恢复 | 出错后能否重试、修复、重放或回滚 |

项目需要根据业务风险定义目标。新闻知识库可能要求分钟级更新；内部制度允许小时级，但权限收紧和删除可能要求更快传播。

### 一条完整的增量更新流水线

~~~text
权威数据源
  ↓
捕获新增、修改、删除、ACL 变更
  ↓
事件队列或变更日志
  ↓
去重与版本检查
  ↓
下载并校验原始文档
  ↓
解析、清洗、切块
  ↓
生成 Embedding、稀疏索引和派生关系
  ↓
写入待发布版本
  ↓
数量、权限、内容和检索回归验证
  ↓
灰度或别名切换
  ↓
缓存失效、监控与对账
~~~

下面逐步拆开。

### 第一步：捕获变化

常见方式有三种：

| 方式 | 适合场景 | 优点 | 风险 |
| --- | --- | --- | --- |
| CDC | 数据库是权威源 | 能捕获行级新增、修改、删除 | 需要维护连接器和事件顺序 |
| Webhook | SaaS 或文档平台支持事件 | 延迟低，不必频繁扫描 | 可能丢失或重复，要能补偿 |
| 定时扫描 | 源系统能力有限 | 实现简单 | 新鲜度较低，扫描成本高 |

可靠系统通常还会保留定期全量对账，即使实时事件链路漏了一条，也能后续发现。

### 第二步：设计稳定 ID 与版本

不要只把随机 chunk ID 当身份。建议至少保留：

| 字段 | 作用 |
| --- | --- |
| source_id | 源系统中的稳定文档 ID |
| source_version | 源记录版本、更新时间或序列号 |
| document_version | RAG 侧发布版本 |
| chunk_id | 文档内稳定片段身份 |
| content_hash | 判断内容是否真实变化 |
| parser_version | 使用哪个解析器 |
| chunker_version | 使用哪套切块策略 |
| embedding_version | 使用哪个模型和维度 |
| acl_version | 权限规则版本 |

内容哈希可以跳过重复计算，但“正文没变”不等于“什么都不用更新”。权限、有效时间、标题、解析器或 Embedding 模型都可能变化。

### 第三步：让处理具备幂等性

假设“文档版本 12”事件被投递三次，系统不应生成三套相同 chunk。

一种常见思路是：

~~~text
幂等键 = source_id + source_version + pipeline_version
~~~

消费者处理前检查：

1. 该幂等键是否已经成功完成；
2. 当前事件是否比已处理版本更新；
3. 如果旧事件晚到，是否应忽略；
4. 上一次是否部分失败，需要从哪个阶段恢复。

状态不能在任务真正完成前就标为成功，否则中途崩溃会留下“看似处理过、实际不完整”的数据。

### 第四步：按变更类型处理

#### 新增文档

创建原始文档记录，解析、切块、向量化，写入待发布索引并验证。

#### 修改正文

可以按文档整体重建，也可以比较 chunk 内容哈希，只重算变化片段。
局部更新节省计算，但必须正确处理：

- chunk 边界变化；
- 被删除的旧 chunk；
- 前后文变化导致的上下文窗口变化；
- 文档级摘要和图关系变化。

#### 修改元数据

标题、时间、分类或权威等级变化，可能不需要重算向量，但要更新过滤字段和排序信号。具体取决于元数据是否参与向量文本。

#### 权限变化

权限收紧要优先处理。旧 ACL 未传播前，用户仍可能检索到不该看到的内容。
除了索引，还要失效：

- 检索缓存；
- 答案缓存；
- 预生成摘要；
- 页面预览；
- 离线导出。

#### 删除文档

删除不能只删源数据库的一行。要沿 lineage 清理：

- 原始文件副本；
- 解析文本；
- chunk；
- 向量和关键词索引；
- 图节点与关系；
- 摘要和推荐结果；
- 缓存；
- 受保留策略管理的日志和备份。

若物理删除需要时间，可以先写 tombstone，让查询层立即不可见，再异步完成各副本清理。

### 为什么“直接双写”容易出问题

假设应用先写文档数据库，再写向量数据库：

~~~text
写文档数据库成功
→ 写向量数据库时超时
→ 两边版本不一致
~~~

超时也不代表写入失败，客户端重试又可能生成重复记录。跨系统一般没有一个共同事务。

更常见的可靠方案是 Transactional Outbox：

1. 业务数据和待发送事件写入同一个本地事务；
2. CDC 或发布器把事件送到消息系统；
3. 向量、关键词、缓存等消费者分别处理；
4. 每个消费者使用幂等键；
5. 系统持续对账并能从事件日志重放。

它提供的是最终一致的工程手段，不是让所有系统在同一毫秒完成更新。

### 第五步：决定增量更新还是全量重建

| 变化 | 常见策略 | 原因 |
| --- | --- | --- |
| 新增少量文档 | 增量插入 | 影响范围小 |
| 单篇正文修改 | 文档级重建或局部增量 | 可通过 lineage 定位 |
| ACL 或元数据修改 | 更新字段并清缓存 | 通常无需重算全部向量 |
| 删除文档 | tombstone 加异步清理 | 先保证查询不可见 |
| 切块规则改变 | 重建新索引 | chunk 身份和边界变化 |
| Embedding 模型改变 | 重建新索引 | 新旧向量空间不可直接混用 |
| 向量维度改变 | 重建新索引 | 索引结构不兼容 |
| 大规模解析器升级 | 影子全量重建 | 影响范围难局部判断 |

全量重建并不等于停机。可以在后台构建新版本，验证通过后切换。

### 第六步：影子索引、灰度和原子切换

推荐把在线版本与构建版本分开：

~~~text
rag_index_v41  ← 当前线上
rag_index_v42  ← 后台构建和验证
rag_current    ← 应用访问的稳定别名
~~~

新版本完成后：

1. 检查文档数、chunk 数、向量维度和 ACL；
2. 抽样比对源文本与解析结果；
3. 跑冻结评测集和安全集；
4. 使用影子请求比较新旧检索结果；
5. 对少量租户或流量灰度；
6. 把别名切到新版本；
7. 保留旧版本到观察窗口结束。

切换时要让一次请求固定使用同一发布版本，避免检索使用新索引、引用页面却来自旧版本。

### 第七步：缓存失效

缓存键至少要考虑：

~~~text
租户 + 用户权限域 + 规范化查询 + 数据版本 + 检索配置版本
~~~

源文档变化后，可以：

- 直接切换全局数据版本，让旧缓存不再命中；
- 按文档 lineage 找到受影响查询并精准失效；
- 对高新鲜度业务设置较短 TTL；
- 对权限收紧和删除执行主动失效。

只依赖 TTL 会留下一个可见旧数据的窗口，是否可接受要由业务和合规要求决定。

### 第八步：监控“源到答案”的新鲜度

不要只看同步任务是否成功。建议监控：

| 指标 | 含义 |
| --- | --- |
| 变更捕获延迟 | 源变化到事件产生的时间 |
| 队列积压 | 尚未处理的事件数量和最老年龄 |
| 处理成功率 | 解析、Embedding、写索引各阶段结果 |
| 死信数量 | 多次失败后需人工处理的事件 |
| 新鲜度 P50/P95/P99 | 源变化到可检索的端到端时间 |
| 版本差异 | 源、正文、向量、ACL 的版本是否一致 |
| 孤儿 chunk | 找不到有效源文档的派生数据 |
| 删除传播延迟 | 删除或权限收紧到全链路不可见的时间 |

可以设置合成查询：制度发布后自动询问已知问题，确认新答案和引用真正上线。

## 工程实践与边界

### 故障场景与处理

| 故障 | 不安全做法 | 更合理的处理 |
| --- | --- | --- |
| Webhook 重复 | 每次都插入 | 幂等键去重 |
| 事件乱序 | 后到的直接覆盖 | 比较源版本，拒绝旧事件 |
| OCR 服务超时 | 标记整条更新成功 | 重试、死信，并保持旧版本在线 |
| 向量写入超时 | 立即盲目重试 | 查询写入状态，使用确定性 ID |
| 部分 chunk 失败 | 发布已成功部分 | 整个文档版本不发布或明确降级 |
| 新解析器质量下降 | 原地覆盖 | 影子索引评测后切换 |
| 权限收紧失败 | 等下次定时任务 | 高优先级重试、查询侧阻断并告警 |

### Lineage 为什么重要

如果第 8 页被删除，系统需要知道它影响了：

- 哪些 chunk；
- 哪些向量；
- 哪个文档摘要；
- 哪些图节点和关系；
- 哪些缓存；
- 哪些评测样本。

没有 lineage，只能大范围扫描或全量重建，删除和故障修复也难以证明完整。

### 大规模更新时的限流

新模型上线可能触发数百万 chunk 重算。要为在线查询保留资源：

- 批处理控制并发；
- 队列按租户和优先级调度；
- 权限与删除事件优先；
- 对 Embedding API 做退避和配额监控；
- 构建任务与在线查询隔离资源池；
- 超过预算时暂停构建，而不是影响线上 SLO。

### 多租户边界

- 每个事件携带可信租户身份，不能从文档正文猜租户；
- 消费者按租户授权写入对应分区或集合；
- 缓存和索引版本不跨权限域复用；
- 对账和运维工具也必须遵守租户隔离；
- 灰度可按租户进行，但不能让同一请求混用不同权限版本。

### 备份不等于回滚

回滚通常是切回一个仍可查询的已知好索引版本，速度应较快。
备份用于灾难恢复，恢复时间更长，且可能缺少切换后发生的新事件。设计时两者都需要，但职责不同。

### 系统边界

- 跨多个存储通常只能做到经过设计的最终一致；
- 删除传播到备份和审计日志要遵守独立保留规则；
- 有些向量引擎的删除或更新是异步可见，需测量实际延迟；
- 局部重建是否安全取决于 chunk、摘要和图关系的依赖范围；
- 新鲜度越高，计算、队列和运维成本通常也越高。

## 常见误区

- **“增量更新就是向量库 upsert”**：正文、稀疏索引、ACL、图关系和缓存也要更新。
- **“内容哈希没变就无需处理”**：权限、有效期、解析器和模型版本可能已经变化。
- **“定时全量跑一遍最可靠”**：规模大时成本和延迟高，运行中失败也需要恢复与对账。
- **“双写两个数据库就一致了”**：任一写入失败、超时或重试都可能让两边分叉。
- **“删除源文件就结束了”**：派生索引、摘要、缓存和预览仍可能返回内容。
- **“Embedding 升级可以逐条替换旧向量”**：查询向量与文档向量应属于兼容空间，通常要成套重建和切换。
- **“索引构建成功就可以上线”**：还需要内容、权限、回归、性能和安全验证。
- **“TTL 到期后权限自然会正确”**：权限收紧期间仍可能泄露，通常需要主动阻断和失效。

## 面试追问

### 1. 怎么保证同一事件重复消费不会产生重复数据？

使用由 source_id、source_version 和 pipeline_version 组成的幂等键，chunk 使用确定性 ID；消费者先检查已处理版本，写入用 upsert，并只在全部阶段成功后记录完成状态。

### 2. 如何处理事件乱序？

每条事件携带源版本或单调序列。消费者只接受比当前版本更新的事件，旧事件直接忽略；无法保证顺序时，定期与权威源做版本和哈希对账。

### 3. Embedding 模型升级为什么通常要全量重建？

不同模型生成的向量空间、维度和相似度分布可能不同。查询使用新模型而文档仍是旧模型时，分数没有可靠可比性，所以通常构建完整新索引，再同时切换查询编码器和索引。

### 4. 删除怎样做到及时又可靠？

先写 tombstone 或禁用状态，让查询层立即过滤；再通过事件异步删除向量、关键词索引、图关系、缓存和其他副本；最后对账并监控删除传播延迟。

### 5. 为什么使用 Outbox？

它把业务变更和“待发布事件”放在同一本地事务中，避免业务数据已经提交但事件丢失。下游再通过可重放事件流和幂等消费者更新各派生系统。

### 6. 怎样发现有文档漏更新？

定期从权威源做文档数、版本、内容哈希和 ACL 对账；检查队列积压、死信和孤儿 chunk；再通过合成查询验证关键新内容是否真正可检索。

### 7. 回滚后，回滚期间的新写入怎么办？

变更日志继续保留，不因切回旧索引而停止捕获。修复新版本后，从已记录的检查点重放这些事件，再验证并重新灰度。

### 8. 你如何定义知识新鲜度 SLO？

以“权威源提交变更”到“授权用户能够检索到正确版本”的端到端时间计算，报告 P50、P95、P99。删除和权限收紧可单独设置更严格的 SLO。
