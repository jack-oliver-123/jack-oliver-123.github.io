## 本篇目标

学会用“谁提供能力、谁规定流程、谁决定下一步”区分 Tool、Workflow 和 Agent。读完后，你应该能识别那些“看起来很智能、其实只是固定流程”的系统。

## 60 秒回答

Tool 是一个有明确输入、输出和权限边界的能力单元，例如查询订单、搜索资料或发送邮件。它负责执行一项能力，不负责决定整个任务目标。Workflow 是开发者预先定义的控制流，可以包含普通代码、模型调用、Tool，甚至 Agent 节点。Agent 则把一部分“下一步做什么”的控制权交给模型，让它根据当前状态和工具结果动态选择动作。

三者不是三选一，也不是谁包含谁那么简单。最有效的区分维度是控制权：Tool 等待调用；Workflow 的主要路径由代码或流程图定义；Agent 的局部路径由模型在运行时选择。生产系统通常用 Workflow 固定高风险主流程，在少数信息不完备的节点使用 Agent，再通过 Tool 与外部系统交互。

## 详细解析

### 先用一个厨房类比

可以把完成一顿晚餐想成一个系统：

- **Tool 像厨房工具**：菜刀、烤箱和搅拌机各自提供一种能力；
- **Workflow 像菜谱**：先切菜，再腌制，然后烘烤，步骤已经写好；
- **Agent 像厨师**：发现没有某种食材后，判断是换菜、找替代品还是询问客人。

菜刀不会决定今晚做什么；菜谱即使有“如果没有牛奶就用水”的分支，仍然是提前写好的；厨师才会根据现场情况在允许范围内改变下一步。

技术系统也是一样：**能力、流程和决策权是三个不同维度。**

### 用客服场景看三者怎样配合

用户说：“我买的鞋尺码不合适，想换货。”

系统可能拥有三个 Tool：

1. 查询订单；
2. 查询库存；
3. 创建换货单。

如果开发者规定：

> 查询订单 -> 检查是否在 7 天内 -> 查询同款库存 -> 有库存就创建换货单 -> 没库存就退款

这是一条 Workflow。里面可以调用三个 Tool，也可以让大模型生成回复，但主要路径已经由开发者写好。

如果用户的问题可能非常多样，例如订单号缺失、商品已经使用、政策存在例外、库存服务暂时不可用，系统需要根据每一步结果决定提问、换工具、修改计划或转人工，这一局部就具有 Agent 特征。

一个常见组合是：

> Workflow 管住主流程 -> Agent 处理模糊判断 -> Tool 完成真实操作。

### Tool：只提供能力，不拥有业务目标

Tool 可以是：

- 一段普通函数；
- 一个数据库查询接口；
- 一个搜索服务；
- 一个第三方 API；
- 一个浏览器或计算机操作能力；
- 甚至是对另一个模型或 Agent 的封装。

因此，把 Tool 统一视为确定性组件并不准确。天气查询通常比较确定，搜索排序可能变化，包装另一个模型的 Tool 更具有概率性。

Tool 的核心不是实现形式，而是**契约**：

| 契约 | 示例 |
| --- | --- |
| 名称 | 查询订单 |
| 用途 | 根据订单号读取订单，不修改数据 |
| 输入 | 订单号、可信用户身份 |
| 输出 | 订单状态、金额、商品信息 |
| 错误 | 参数错误、订单不存在、权限不足、服务超时 |
| 副作用 | 无，属于只读操作 |
| 权限 | 用户只能查看自己的订单 |

再看“执行退款”工具：

- 它是写操作；
- 会产生资金副作用；
- 超时后不能盲目重试；
- 可能需要人工审批；
- 应使用幂等键防止重复退款。

所以生产 Tool 不只是一个函数声明，还包括鉴权、校验、错误语义、审计和恢复策略。

### Workflow：开发者规定控制流

Workflow 的核心是**路径边界可预期**。它可以是直线，也可以有条件、循环、并行和失败分支。

例如：

> 如果金额小于 100 元，自动退款；如果在 100 到 500 元之间，转客服审批；如果超过 500 元，转主管审批。

虽然它会根据金额走不同分支，但所有合法路径都由开发者提前定义，所以仍然是 Workflow。

Workflow 也可以使用大模型：

- 大模型判断工单类型；
- 大模型从邮件里抽取订单号；
- 大模型生成客服回复；
- 大模型给结果打分。

关键不是“有没有 LLM”，而是模型是否只完成一个受限节点。一次模型分类后进入开发者预设的三个分支，通常仍是 Workflow。

常见 Workflow 形式包括：

- 提示链：一个步骤的输出进入下一个步骤；
- 路由：先分类，再进入不同流程；
- 并行：多个独立任务同时执行后汇总；
- 编排者—执行者：中心节点拆任务，多个执行者处理；
- 评估者—优化者：生成、评价、修改，直到通过或达到上限。

这些模式会在下一篇进一步展开。

### Agent：在受限范围内动态选择路径

Agent 的核心是模型参与控制循环：

1. 读取用户目标和当前状态；
2. 判断下一步需要回答、提问、调用工具还是结束；
3. 运行时检查并执行动作；
4. 接收新的观察结果；
5. 根据结果重新决策。

如果订单查询返回“订单号不存在”，Agent 可能询问用户确认；如果库存接口超时，它可能进行有限重试；如果发现商品属于特殊品类，它可能检索另一份政策。

这种灵活性解决了“路径难以穷举”的问题，但也带来：

- 可能选择错误工具；
- 可能重复动作；
- 可能偏离目标；
- 可能无限循环；
- 成本和延迟更难预测；
- 测试空间更大。

因此，Agent 的动态选择不是“想做什么就做什么”，而是在运行时允许的工具、权限、步数和政策范围内决策。

### 一张表看清三者

| 维度 | Tool | Workflow | Agent |
| --- | --- | --- | --- |
| 主要职责 | 执行单项能力 | 编排已知流程 | 探索未完全预定义的路径 |
| 谁决定下一步 | 调用方 | 开发者定义的代码或图 | 模型与外层策略共同控制 |
| 是否能包含 LLM | 可以 | 可以 | 通常以 LLM 为决策组件 |
| 是否需要调用外部系统 | 不要求 | 不要求 | 不要求 |
| 典型风险 | 参数、权限和副作用 | 分支遗漏和状态错误 | 漂移、循环、成本和安全 |
| 主要测试 | 契约与集成测试 | 路径和状态迁移测试 | 任务、轨迹和安全评测 |

### 什么是 Agentic Workflow

实际项目很少完全位于两个极端。所谓 Agentic Workflow，通常是：

- 外层由固定状态机约束；
- 高风险步骤和失败出口由代码规定；
- 某些节点允许模型动态选择工具或计划。

例如售后主流程固定为“身份验证 -> 问题诊断 -> 解决方案 -> 用户确认 -> 执行”，但“问题诊断”内部由 Agent 根据对话和工具结果灵活处理。

这种混合方式既保留 Workflow 的可控性，也使用 Agent 处理局部不确定性。

### 如何选：先问任务路径能否枚举

可以按下面的顺序判断：

1. **一次模型调用能完成吗？** 能就不要增加循环。
2. **步骤和分支能可靠列出来吗？** 能就优先 Workflow。
3. **只有某个节点存在模糊判断吗？** 只在该节点使用 Agent。
4. **环境反馈会不断改变后续路径吗？** 才考虑更完整的 Agent。
5. **新增自主性是否真的提升成功率？** 用评测数据验证。

Agent 不是 Workflow 的升级版。确定流程使用 Agent，往往只是让系统更贵、更慢、更难排错。

## 工程实践与边界

- **Tool 定义要清晰**：参数尽量少，名称避免重叠，明确输入输出、错误和副作用。
- **读写能力分开**：查询订单和修改订单使用不同工具，便于授权、审批和审计。
- **Workflow 定义失败语义**：每个节点设置超时、重试和退出；并行节点说明部分失败时怎样汇合。
- **Agent 只看必要工具**：工具越多、描述越相似，模型越容易选错，也会占用更多上下文。
- **外部结果不等于指令**：网页、邮件、文档和工具输出都是不可信数据，不能覆盖系统政策。
- **分开两类日志**：模型决策轨迹用于调试，业务审计记录用于证明谁在什么权限下执行了什么动作；敏感参数要脱敏。
- **用轨迹收敛系统**：观察线上 Agent 的常见路径，把稳定重复的部分逐步固化为 Workflow。

## 常见误区

- **“Workflow 就是多个 Agent 串联”**：Workflow 的节点可以全是普通代码，也可以只包含一次模型调用。
- **“有分支就不是 Workflow”**：条件和循环也可以由开发者预先定义，关键看路径边界由谁控制。
- **“Tool 只是函数声明”**：生产 Tool 还包含鉴权、执行、错误契约、副作用说明和审计。
- **“固定流程不能使用大模型”**：大模型可以作为分类、抽取、生成或评价节点。
- **“会调用多个 Tool 就是 Agent”**：如果顺序固定，它仍然是 Workflow。
- **“Agent 比 Workflow 高级”**：两者解决不同问题，确定流程通常更适合可预测的 Workflow。

## 面试追问

> **面试官：** LLM 路由节点算 Agent 吗？
>
> **候选人：** 不一定。如果模型只在开发者预设的有限分支中做一次分类，整体仍然是 Workflow；如果它持续根据环境结果选择工具、改变步骤并判断停止，才更接近 Agent。

> **面试官：** 一个 Workflow 能否包含 Agent？
>
> **候选人：** 可以。外层 Workflow 可以固定身份验证、审批和提交等主流程，只把信息不完整的诊断节点交给 Agent 动态处理。

> **面试官：** 为什么 Tool 必须标注副作用？
>
> **候选人：** 调度器需要据此决定能否自动重试、是否要求审批、失败后能否安全重放，以及日志和权限应该采用什么级别。

> **面试官：** 如何从 Agent 原型演进到生产？
>
> **候选人：** 先追踪真实轨迹，找出长期稳定的重复路径，把它们固化成 Workflow；只保留确实需要动态判断的节点，并给所有写操作增加权限、幂等和审批。

> **面试官：** 如何证明某个节点需要 Agent？
>
> **候选人：** 用同一评测集比较固定规则、单次模型调用和 Agent 三种方案。如果 Agent 没有显著改善任务成功率，却增加了延迟、成本和安全风险，就不值得引入。
