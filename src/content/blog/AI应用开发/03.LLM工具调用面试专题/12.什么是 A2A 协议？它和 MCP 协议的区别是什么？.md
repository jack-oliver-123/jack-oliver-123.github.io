## 本篇目标

学完这一篇，你应该能够：

1. 解释 A2A 解决的是 Agent 与 Agent 之间的协作，而不是简单的工具调用。
2. 说清 A2A 1.0 中 Agent Card、Message、Task、Artifact 的职责。
3. 区分 `contextId`、`taskId` 与 `messageId`。
4. 解释阻塞、立即返回、流式订阅、轮询和推送通知如何支持长任务。
5. 说明 `INPUT_REQUIRED`、`AUTH_REQUIRED`、终态与任务恢复的语义。
6. 比较 A2A 与 MCP，并说明它们为什么互补而不是互相替代。
7. 给出发现、授权、幂等、多租户和提示注入的工程边界。

## 60 秒回答

> A2A，也就是 Agent2Agent Protocol，用来标准化相互独立的 Agent 之间如何发现能力、发送消息、委派任务、交换产物并跟踪长任务。A2A 1.0 的核心对象包括：用于发现能力和接口的 Agent Card；承载交互内容的 Message；具有状态和生命周期的 Task；承载任务正式结果的 Artifact。Message 主要用于沟通，任务结果应该放在 Artifact 中。
>
> 调用方先获取 Agent Card，确认技能、输入输出媒体类型、协议接口与安全要求，再发送 Message。简单交互可以直接返回 Message；需要跟踪的工作会返回 Task。Task 可以处于 SUBMITTED、WORKING、INPUT_REQUIRED、AUTH_REQUIRED、COMPLETED、FAILED、CANCELED 或 REJECTED 等状态。长任务可选择默认阻塞等待、`return_immediately` 后轮询、订阅流式更新，或配置推送通知。
>
> A2A 与 MCP 互补：A2A 面向 Agent 与 Agent，抽象粒度是能力、消息、任务和产物；MCP 面向 Host/Client 与能力 Server，抽象粒度是 Tools、Resources、Prompts 等上下文和操作。采购 Agent 可以通过 A2A 委派合规审核，合规 Agent 内部再通过 MCP 查询政策和调用风险工具。二者都不自动建立信任，仍要做服务身份校验、逐请求授权、租户隔离、幂等、回调安全和结果验证。

## 详细解析

### 先说人话：A2A 像“部门之间派工”，MCP 像“员工使用办公设备”

假设公司里有三个独立部门：

- 采购部门；
- 合规部门；
- 财务部门。

采购部门不能只调用一个 `check()` 函数就假装完成跨部门协作。

它需要知道：

- 合规部门能处理什么；
- 应该把哪些资料交过去；
- 任务是否已经接收；
- 是否还缺用户输入或认证；
- 当前处理到哪一步；
- 最终报告在哪里；
- 任务失败、拒绝或取消时怎么办。

这就是 A2A 重点解决的问题。

而合规部门内部可能需要：

- 读取政策文档；
- 查询供应商数据库；
- 调用风险评分 API。

这些内部工具和上下文可以通过 MCP 接入。

最短记忆法：

```text
A2A：把工作交给另一个 Agent，并跟踪任务与产物。
MCP：让一个 Host 连接并使用外部工具和上下文。
```

### A2A 是什么

A2A 是面向 Agent 互操作的开放协议。

它不要求双方：

- 使用同一家模型；
- 使用同一种 Agent 框架；
- 共享内部 Prompt；
- 暴露思维过程；
- 共享内部 Tools；
- 部署在同一组织。

协议定义的是外部协作契约，而不是对方 Agent 的内部实现。

### A2A 1.0 的四个核心概念

```text
Agent Card：我是谁、我会什么、去哪里调用我
Message：我们在沟通什么
Task：这项工作当前是什么状态
Artifact：这项工作的正式产物是什么
```

下面逐个拆开。

### Agent Card：远程 Agent 的“能力名片”

**Agent Card** 是用于发现 Agent 的元数据文档。

常见发现位置是：

```text
https://agent.company.test/.well-known/agent-card.json
```

Agent Card 可以描述：

- Agent 名称和说明；
- 提供者信息；
- `supportedInterfaces` 中的目标 URL、协议绑定和协议版本；
- 支持的能力，例如流式或推送通知；
- 默认输入输出媒体类型；
- `AgentSkill` 能力列表；
- 安全方案与安全要求；
- 可选扩展；
- 是否支持获取扩展 Agent Card。

一个简化示意：

```json
{
  "name": "合同审核 Agent",
  "description": "审核合同条款并生成风险清单",
  "supportedInterfaces": [
    {
      "url": "https://agent.company.test/a2a/v1",
      "protocolBinding": "JSONRPC",
      "protocolVersion": "1.0"
    }
  ],
  "version": "1.0.0",
  "capabilities": {
    "streaming": true,
    "pushNotifications": true
  },
  "defaultInputModes": ["text/plain", "application/pdf"],
  "defaultOutputModes": ["text/markdown", "application/json"],
  "skills": [
    {
      "id": "contract-risk-review",
      "name": "合同风险审核",
      "description": "识别合同中的付款、责任与终止风险",
      "tags": ["合同", "合规", "风险"]
    }
  ]
}
```

这只是帮助理解的简化结构，生产实现要以当前规范和生成的 Schema 为准。

### Agent Card 不是授权凭证

能下载到一张 Agent Card，只说明发现了一个自我声明的服务。

它不证明：

- 服务可信；
- 发布者就是其声称的组织；
- 当前调用者有权使用所有 Skill；
- 返回结果正确；
- Card 中的 URL 不会被篡改。

生产环境需要 HTTPS、服务身份校验、信任目录、签名或其他供应链控制，并按 Card 声明的安全方案获取凭证。

### `AgentSkill` 是能力描述，不是 Skill 文件包

A2A 的 `AgentSkill` 描述远程 Agent 的一项能力。

核心字段包括：

- `id`；
- `name`；
- `description`；
- `tags`；
- 可选 `examples`；
- 可选 `inputModes`；
- 可选 `outputModes`；
- 可选 `securityRequirements`。

它用于发现和选择远程 Agent 能力。

它不是 Agent Skills 规范中的 `SKILL.md` 目录，也不携带实现代码。

### Message：一次沟通内容

**Message** 是 Client Agent 与 Server Agent 之间的一次消息。

重要字段包括：

- `messageId`：由消息创建者生成的唯一标识；
- `contextId`：消息所属的交互上下文；
- `taskId`：可选，消息关联的任务；
- `role`：消息角色；
- `parts`：消息的内容部分；
- 可选扩展、引用任务和元数据。

`parts` 允许承载不同媒体或结构化内容，而不只是纯文本。

例如：

```text
Message
  ├── 文本：请审核附件合同
  ├── 文件：contract.pdf
  └── 结构化数据：合同编号、地区、截止日期
```

### Task：可跟踪的工作单元

**Task** 是 A2A 中可跟踪工作的核心对象。

它包含：

- `id`；
- 可选 `contextId`；
- 当前 `status`；
- 可选 `artifacts`；
- 可选 `history`；
- 可选 `metadata`。

任务与一次普通请求的最大区别是：它有持续状态。

远程 Agent 可以先确认接单，再异步处理，期间要求补充信息，最后产生多个 Artifact。

### Artifact：正式任务产物

**Artifact** 是任务生成的结果对象。

例如合同审核任务可以产生：

```text
Artifact A：风险摘要 Markdown
Artifact B：结构化风险 JSON
Artifact C：带批注的合同文件
```

规范明确区分 Message 与 Artifact：

- Message 主要用于沟通、追问和状态说明；
- Artifact 用于交付任务结果。

不要只在一条状态 Message 中塞最终报告，然后让调用方从自然语言里猜哪里是正式结果。

### Message、Task、Artifact 的关系

```text
Client Message
  “请审核这份合同”
        ↓
Task
  id = T100
  status = WORKING
        ↓
Server Message
  “缺少适用地区，请补充”
        ↓
Task
  status = INPUT_REQUIRED
        ↓
Client Message
  “适用地区为中国大陆”
        ↓
Task
  status = WORKING
        ↓
Artifacts
  风险报告 + 结构化清单
        ↓
Task
  status = COMPLETED
```

### `messageId`、`taskId`、`contextId` 不要混淆

| 标识 | 粒度 | 作用 |
| --- | --- | --- |
| `messageId` | 一条消息 | 消息唯一标识，由消息创建者生成 |
| `taskId` | 一项工作 | 查询、订阅、取消或继续同一任务 |
| `contextId` | 一组相关交互 | 关联同一会话或业务上下文中的多个任务和消息 |

一个 Context 中可以有多个 Task，一个 Task 可以有多条 Message 和多个 Artifact。

不能把 `contextId` 当授权边界，也不能只凭知道 `taskId` 就允许读取任务。

### Task 状态机

A2A 1.0 定义的 TaskState 包括：

| 状态 | 含义 | 类型 |
| --- | --- | --- |
| `TASK_STATE_UNSPECIFIED` | 未知或未确定 | 异常/未知 |
| `TASK_STATE_SUBMITTED` | 已提交并确认 | 进行中 |
| `TASK_STATE_WORKING` | 正在处理 | 进行中 |
| `TASK_STATE_INPUT_REQUIRED` | 需要额外用户输入 | 中断态 |
| `TASK_STATE_AUTH_REQUIRED` | 需要额外认证 | 中断态 |
| `TASK_STATE_COMPLETED` | 成功完成 | 终态 |
| `TASK_STATE_FAILED` | 失败结束 | 终态 |
| `TASK_STATE_CANCELED` | 已取消 | 终态 |
| `TASK_STATE_REJECTED` | Agent 拒绝执行 | 终态 |

**中断态**不是失败。

它表示任务暂时不能继续，但可以在满足条件后恢复。

### `INPUT_REQUIRED` 怎样继续

例如 Agent 返回：

```text
Task T100
state = INPUT_REQUIRED
status.message = “请提供合同适用地区。”
```

Client 应：

1. 向用户展示明确问题；
2. 收集真实输入；
3. 发送新的 Message；
4. 使用同一个 `taskId`；
5. 保持匹配的 `contextId`；
6. 继续观察任务状态。

不能让模型自行编造“用户已同意”或“地区是中国”。

### `AUTH_REQUIRED` 怎样继续

`AUTH_REQUIRED` 表示继续任务需要认证。

正确做法通常是：

- 暂停任务；
- 按 Agent 声明的安全方案在协议外或规定流程中完成认证；
- 不把长期凭证放进普通 Message；
- 完成后发送关联同一任务的后续消息或请求；
- 服务端重新检查授权。

“Agent 说需要认证”不等于可以把用户密码直接发给它。

### 简单交互与任务交互

并非每条 Message 都会创建 Task。

远程 Agent 可以直接返回 Message，例如：

```text
Client：你支持哪些合同格式？
Agent：支持 PDF、纯文本和结构化 JSON。
```

需要持续处理、状态跟踪或产物交付时，更适合返回 Task。

调用方必须能处理两种响应，而不是假设任何请求都有 `taskId`。

### 长任务的四种获取方式

#### 1. 默认阻塞等待

对于非流式 Send Message，默认阻塞语义会等待任务到达：

- 终态；或
- `INPUT_REQUIRED`、`AUTH_REQUIRED` 这类中断态。

适合预计很快完成且基础设施超时允许的任务。

#### 2. `return_immediately`

调用方请求立即返回后，服务端可以返回仍在进行中的 Task。

调用方随后负责：

- 轮询 Get Task；
- Subscribe to Task；
- 或接收已配置的推送通知。

适合长时间运行、调用方不想一直占用原请求的场景。

#### 3. 流式更新

流式操作可以返回不同类型的 `StreamResponse`：

- Task；
- Message；
- `TaskStatusUpdateEvent`；
- `TaskArtifactUpdateEvent`。

一次 StreamResponse 只包含其中一种。

调用方可以逐步展示状态和 Artifact 增量。

#### 4. 推送通知

长任务还可以配置回调，让 Agent 主动把更新推送到 Client 的 webhook。

适合：

- 任务持续数分钟或数小时；
- Client 不适合保持长连接；
- 需要跨服务异步处理。

推送通知必须考虑回调身份、SSRF、防重放、重试、幂等和签名验证。

### 流断开不等于任务结束

流式连接只是“更新传递通道”，Task 才是业务工作状态。

连接断开可能表示：

- 网络抖动；
- 网关超时；
- Client 重启；
- Server 主动关闭流；
- 任务已经完成；
- 任务仍在后台运行。

恢复时应基于 `taskId` 查询或重新订阅，而不是重新创建同一业务任务。

### Message 不是关键结果的可靠投递存储

A2A 规范指出，并非所有 Message 都保证持久化到 Task history。

流式断开后，调用方也可能错过状态消息。

因此：

- 正式结果放 Artifact；
- 关键业务状态由 Task 表达；
- 不能依赖一条临时 Message 作为唯一证据；
- 调用方应能通过 Get Task 恢复最终状态和产物。

### A2A 与 MCP 的核心区别

| 维度 | A2A | MCP |
| --- | --- | --- |
| 主要交互双方 | Client Agent 与 Remote Agent | Host 内 MCP Client 与 MCP Server |
| 抽象粒度 | Agent 能力、消息、任务、产物 | Tools、Resources、Prompts 等能力 |
| 发现方式 | Agent Card | 初始化、能力协商与列表方法 |
| 状态模型 | 完整 Task 生命周期 | 请求/响应、通知，另有可选 Tasks 能力 |
| 长任务协作 | Task、流式、轮询、推送通知 | 取决于具体能力与规范特性 |
| 输入输出 | 多 Part Message、Artifact、媒体类型 | 协议内容块、工具参数与能力数据 |
| 内部实现是否暴露 | 不要求 | Server 也隐藏下游实现 |
| 典型场景 | 委派研究、审核、规划、跨组织协作 | 查文件、搜数据、调用业务 API |
| 是否互斥 | 否 | 否 |

### 为什么 A2A 不是“更高级的 MCP”

它们的边界不同。

调用一个天气 Tool 通常不需要创建一个有生命周期的 Agent Task。

委派一项三小时的合同审核，也不适合只暴露成一个无法跟踪状态的简单函数。

选择取决于你需要的是：

```text
一个能力调用，还是一个有自主处理和产物交付的协作方？
```

### A2A 与 MCP 的组合架构

```text
采购 Agent
  │
  │ A2A：委派“供应商风险审核”任务
  ▼
合规 Agent
  ├── MCP Client -> 政策 Server
  ├── MCP Client -> 供应商数据库 Server
  └── MCP Client -> 风险评分 Server
```

合规 Agent 可以：

1. 通过 A2A Agent Card 声明审核能力；
2. 接收采购 Agent 的 Message；
3. 创建 Task；
4. 内部通过 MCP 获取资料和调用工具；
5. 流式发送状态和 Artifact 更新；
6. 最终交付风险报告 Artifact。

采购 Agent 不需要知道合规 Agent 内部用了哪些模型和 Tools。

### A2A AgentSkill 与 MCP Tool 的区别

| 维度 | A2A `AgentSkill` | MCP Tool |
| --- | --- | --- |
| 含义 | Agent 声明的一项高层能力 | 可调用的具体操作 |
| 载体 | Agent Card 元数据 | MCP Server 工具定义 |
| 调用结果 | 可能形成 Task 和多个 Artifact | 返回工具调用结果 |
| 自主性 | 远程 Agent 可自行规划和追问 | Tool 通常执行明确操作 |
| 典型粒度 | “审核合同风险” | “读取第 12 条条款” |
| 是否一一对应 | 否 | 否 |

一个 AgentSkill 可能内部调用几十个 MCP Tools。

一个 MCP Tool 也可能被多个 AgentSkill 复用。

## 工程实践与边界

### 发现后先验证，不要直接信任

接入 Agent Card 时检查：

- URL 是否来自可信目录；
- 是否使用 HTTPS；
- 域名与证书是否符合预期；
- 协议版本是否兼容；
- 声明的接口与绑定是否受支持；
- 安全方案是否符合企业策略；
- Skill、输入输出模式是否匹配；
- Card 是否缓存过期或发生未审查变更。

公开 Card 只放适合公开的能力信息。

敏感能力可以使用需要认证的扩展 Agent Card。

### 每个请求都做授权

服务端不能因为 Client 知道 `taskId` 就返回任务。

Get、List、Cancel、Subscribe 和推送配置都应：

- 验证调用者身份；
- 限制到当前租户或组织；
- 检查对特定任务的权限；
- 在查询数据库前应用授权范围；
- 避免通过错误差异泄露任务是否存在。

### 任务幂等

网络超时时，调用方可能不知道 Send Message 是否已经创建 Task。

应设计：

- 稳定业务请求键；
- Client 生成的唯一 `messageId`；
- Server 去重策略；
- 重试前查询或恢复；
- 下游副作用幂等。

不能仅靠新的 `taskId` 判断这是新业务。

### 任务状态落库

长任务至少持久化：

- `taskId`；
- `contextId`；
- 当前状态与时间；
- 已接收 Message；
- Artifact 元数据与存储位置；
- 调用者和租户；
- 幂等键；
- 授权与策略决策；
- 下游副作用状态。

这样流断开或服务重启后才能恢复。

### Artifact 验证

`COMPLETED` 只表示远程 Agent 宣称任务成功。

调用方仍要验证：

- Artifact 是否存在；
- 媒体类型是否符合约定；
- 文件大小与哈希；
- 结构化结果是否通过 Schema；
- 是否满足业务不变量；
- 是否含恶意文件或提示注入；
- 是否来自正确任务和租户。

### 推送通知安全

Client 提供 webhook 时要防：

- SSRF；
- 内网地址探测；
- DNS 重绑定；
- 未签名伪造回调；
- 回放攻击；
- 重复投递；
- 大响应或慢响应拖垮发送方。

常见控制：

```text
URL 白名单或验证
  + HTTPS
  + 回调认证/签名
  + 时间戳和随机数
  + 事件 ID 去重
  + 超时与指数退避
  + 出站网络限制
```

### 人在回路

遇到 `INPUT_REQUIRED` 或高风险决策时：

- 向用户展示具体缺失信息；
- 不代替用户同意；
- 不在普通 Message 里传长期密钥；
- 将确认绑定到具体动作、参数和有效期；
- 参数变化后重新确认。

### 提示注入与数据边界

来自远程 Agent 的 Message 和 Artifact 都是不可信内容。

它们不能：

- 修改本地系统指令；
- 自动增加本地 Tool 权限；
- 要求泄露其他 Task 的数据；
- 绕过用户确认；
- 让本地 Agent 执行未授权脚本。

把内容作为数据解析，动作由本地策略决定。

### 可观测性

建议关联：

```text
trace_id
client_agent_id
remote_agent_id
agent_card_version
message_id
context_id
task_id
artifact_id
stream_event_id
push_delivery_id
tenant_id
policy_decision
```

同时记录状态转换耗时、断线恢复次数、重复通知和下游工具调用摘要。

### A2A 选型边界

适合使用 A2A：

- 对方是具有独立能力和生命周期的 Agent；
- 任务可能需要追问或长时间运行；
- 结果可能包含多个 Artifact；
- 需要跨组织或跨框架互操作；
- 不希望暴露对方内部工具编排。

不必使用 A2A：

- 只是调用一个简单确定性函数；
- 标准 REST API 已稳定且语义很简单；
- 没有 Agent 自主协作需求；
- 所有步骤都在同一进程内；
- 只需通过 MCP 访问一个 Tool 或 Resource。

### A2A 与 MCP 选型表

| 需求 | 更适合 |
| --- | --- |
| 查询数据库记录 | MCP Tool/Resource |
| 读取文件或知识库 | MCP Resource/Tool |
| 委派多步骤研究 | A2A Task |
| 跨组织合同审核 | A2A Task |
| 远程 Agent 内部访问业务 API | MCP |
| 调用方跟踪远程产物 | A2A |
| 固定程序调用一个 API | 普通 API 或 MCP |
| 远程 Agent 协作且内部需工具 | A2A + MCP |

## 常见误区

### 误区一：A2A 是用来替代 MCP 的

错误。A2A 管 Agent 级协作，MCP 管上下文与工具接入，可以组合。

### 误区二：A2A 只是 Agent 之间发 HTTP 请求

普通 HTTP 只是传输基础。A2A 还定义 Agent Card、Message、Task、Artifact、状态、流式与协议绑定语义。

### 误区三：每条 Message 都会创建 Task

不一定。简单交互可以直接返回 Message；需要跟踪的工作才形成 Task。

### 误区四：Task 完成后的答案放在 Message 就行

正式任务输出应该放 Artifact。Message 主要用于交流和状态说明。

### 误区五：流断开表示任务失败

流只是更新通道。应使用 `taskId` 查询或重新订阅真实任务状态。

### 误区六：Agent Card 是可信身份和授权证明

它是发现元数据。仍需验证服务身份并执行认证授权。

### 误区七：知道 `taskId` 就可以读取任务

错误。每个操作都要按调用者和租户授权，标识符不是权限。

### 误区八：`INPUT_REQUIRED` 可以让本地模型自动补值

如果需要用户事实或同意，必须向真实用户收集，不能让模型编造。

### 误区九：`COMPLETED` 等于业务结果正确

仍要验证 Artifact、业务不变量、来源、安全与下游副作用。

## 面试追问

### 追问一：一句话解释 A2A

> A2A 是 Agent 间互操作协议，用 Agent Card 发现能力，用 Message 沟通，用 Task 跟踪工作状态，用 Artifact 交付正式结果。

### 追问二：A2A 与 MCP 最本质的区别是什么

> A2A 把对方视为能自主处理任务的 Agent，MCP 把对方视为向 Host 暴露工具和上下文能力的 Server。

### 追问三：Message 和 Artifact 为什么要分开

Message 用于交互、追问和状态说明，不保证所有内容持久化；Artifact 是任务正式产物，便于结构化获取、校验和恢复。

### 追问四：长任务断线后怎样恢复

保存 `taskId` 和授权上下文，通过 Get Task 查询最新状态和 Artifact，或重新 Subscribe to Task；消费更新要去重，不能重新创建相同副作用任务。

### 追问五：`INPUT_REQUIRED` 和 `AUTH_REQUIRED` 是失败吗

不是，它们是中断态。调用方提供真实输入或完成认证后，可以围绕同一 `taskId` 继续交互。

### 追问六：阻塞模式与 `return_immediately` 怎样选择

短任务可默认阻塞到终态或中断态；长任务宜立即返回 Task，再通过轮询、订阅或推送获取更新。还要考虑网关超时、资源占用和用户体验。

### 追问七：A2A AgentSkill 与 Agent Skills 规范是什么关系

名称相似但不是同一对象。A2A AgentSkill 是 Agent Card 中的能力描述；Agent Skills 是以 `SKILL.md` 为入口的指令与资源目录。一个远程能力背后可以使用多个本地 Skills。

### 追问八：如何防止推送通知被伪造或重放

使用 HTTPS、回调认证或签名，验证时间戳和随机数，按事件 ID 去重，限制来源与出站目标，并让通知触发的是状态查询而非直接高风险执行。

### 追问九：为什么普通 REST API 有时已经够用

如果调用只是一个稳定、确定、短时的函数，没有 Agent 能力发现、任务状态、追问、流式和 Artifact 需求，普通 API 更简单。协议应解决真实复杂度，而不是为了名字而引入。

### 可直接背诵的完整答案

> A2A 是 Agent 与 Agent 之间的互操作协议。A2A 1.0 用 Agent Card 描述远程 Agent 的能力、接口和安全要求，用 Message 进行交互，用 Task 表达有生命周期的工作，用 Artifact 交付正式结果。Task 可以在 SUBMITTED、WORKING、INPUT_REQUIRED、AUTH_REQUIRED 和多个终态之间转换；长任务可以默认阻塞等待，也可以立即返回后轮询、流式订阅或使用推送通知。流断开不代表任务结束，应该通过 `taskId` 恢复状态。
>
> 它与 MCP 互补而非替代。A2A 面向 Agent 级委派，抽象是能力、消息、任务和产物；MCP 面向 Host/Client 与 Server，抽象是 Tools、Resources、Prompts 等上下文和操作。采购 Agent 可以用 A2A 委派合规审核，合规 Agent 内部再用 MCP 查询政策和风险工具。工程上还必须验证 Agent Card 和服务身份、对每个任务操作授权、隔离租户、保证幂等、保护回调，并验证远程 Artifact，不能把协议互通误当成业务信任。
